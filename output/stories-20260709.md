## Show HN: Getting GLM 5.2 running on my slow computer

- 原文链接: [Show HN: Getting GLM 5.2 running on my slow computer](https://github.com/JustVugg/colibri)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48842459)

This post is about running a huge AI language model, GLM-5.2 with 744 billion parameters, on a regular computer with just 25 GB of RAM using a tool called colibrì. The tool is written in pure C, has no extra dependencies, and streams most of the model from disk as needed.

The main idea is that only a part of the model needs to be in memory at any time. The "dense" part (like attention and embeddings) uses about 10 GB of RAM. The much larger "expert" parts, which are split into over 21,000 pieces, stay on your disk, taking up about 370 GB, and are loaded into RAM only when needed. The system uses a smart cache to keep the most-used experts handy, and the rest are streamed from disk using your computer’s operating system cache.

This tool avoids using GPUs or Python at runtime. Instead, it’s just a single C file for the engine plus some small helper files. It has features like fast integer math, compressed caches, and tries to minimize memory use. The tool also includes its own tokenizer, reads how much memory is available, and prevents your computer from running out of memory.

When you first start it, there’s a heavy load on your SSD, as it reads a lot of data for every new answer. The project warns this could wear out cheaper drives over time. There’s an option to download a pre-converted model, so you don’t need to convert it yourself. To get started, you just build the code, download or convert the model, and run the chat command; Python is only needed for the one-time conversion.

The tool gets faster the more you use it, as it learns which experts you need most and pins them in RAM. If you have a stronger machine, you can use more RAM or faster disks to speed things up. The developer shares benchmarks from their own laptop and from others, showing that bigger RAM and faster SSDs make a big difference in how quick the model responds.

There’s a request for help from the community: the developer wants to know how much accuracy is lost by using compressed (int4) weights instead of full precision, but can’t measure it well on their own hardware. They ask people with better machines to run tests and share results.

In the comments, many people are impressed that such a large model can run on a normal computer. Several users note that the heavy disk use is clever, but they worry about SSD wear and slow response times. Some think this is a smart way to democratize access to big models, making them available to more people without expensive GPUs. Others talk about the technical challenges, like managing memory and streaming data efficiently, and suggest ways to improve disk and RAM handling.

A few users ask about possible uses, such as running private AI assistants or doing research without cloud costs. Some share their own attempts at running large models, comparing notes on performance and hardware. There’s talk about further optimizations, like mixed precision or using even more aggressive quantization. Some warn that for real-time or production use, the speed might not be enough unless you have very fast disks or lots of RAM.

Overall, the discussion is positive, with people excited about the technical achievement. There’s a lot of interest in seeing how this idea can be extended, and several users offer to help test or contribute hardware benchmarks to move the project forward.

## EU Parliament greenlights Chat Control 1.0

- 原文链接: [EU Parliament greenlights Chat Control 1.0](https://www.patrick-breyer.de/en/eu-parliament-greenlights-chat-control-1-0-breyer-our-children-lose-out/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48843923)

The article is about a new EU rule called “Chat Control 1.0,” which allows tech companies to scan people’s private messages to look for child abuse, even if there is no reason to suspect those people. The European Parliament voted on this law, and most members voted against it, but not enough to officially reject it, so the law passed and will stay until 2028.

A small exception was made for encrypted messages, but this does not change much, since these were not scanned before. Some politicians wanted scanning only for people under investigation by the courts, but this rule did not pass. Patrick Breyer, a civil rights activist, says this law is bad for democracy and does not really protect children. He thinks real child protection laws might not happen now, because governments are sticking with scanning everyone’s messages. Breyer says scanning everyone is like trying to clean a floor while water is still running—it does not solve the real problem. He also says these scans make too many false alarms, waste police time, and rarely help find new crimes or rescue children.

Now, US tech companies can scan private messages on apps like Instagram, Discord, Gmail, and iCloud, without a warrant. Public posts and cloud files were already being scanned, and end-to-end encrypted messages (like WhatsApp) are still not scanned. European-made messaging apps never had this scanning. The article explains that since more messages are encrypted, abuse reports have dropped by half since 2022. Most useful reports come from public posts, not private chats, and many alerts do not show real crimes. Sometimes, investigations even target minors, not abusers. Most of the reported material is already known to police. There is no proof that scanning everyone’s messages leads to more arrests or rescues.

Commenters on Hacker News have many opinions. Some people worry this law hurts privacy and could be abused by companies or governments. They say it is wrong to scan everyone’s messages just in case there is a crime. Others agree with Patrick Breyer that the law does not really help children and just creates more work for police. Some point out that if criminals use encrypted apps, they will not be caught by this scanning anyway. A few support the law, saying child safety is important and tech companies need to do more to stop abuse. Some commenters think the EU’s process is undemocratic, since the law passed even though most members voted against it. Others believe there are better ways to protect children, like stronger policing and better reporting tools, instead of mass scanning. Some survivors of abuse say privacy is important for victims, and mass scanning could actually hurt them. There are also worries about false alarms and the risk of innocent people being investigated. Overall, most commenters do not think this law will fix the real problems, and many want a more targeted, careful approach.

## Buried Apple feature turns an iPhone into the perfect kids' dumb phone

- 原文链接: [Buried Apple feature turns an iPhone into the perfect kids' dumb phone](https://www.wired.com/story/this-buried-apple-feature-turns-an-iphone-into-the-perfect-kids-dumb-phone/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48803004)

The article explains how an iPhone can become a simple, safe phone for kids by using a hidden Apple feature called Assistive Access. This feature is found in the Accessibility settings and was first made for people with cognitive disabilities.

With Assistive Access, parents can pick only a few apps for the child to use, like Calls, Messages, Maps, Camera, Photos, and Music. The interface changes to show big, easy-to-tap tiles, making it simple for kids. Importantly, parents can block web browsers like Safari, so the child cannot access the internet or open links sent in messages. This solves the problem of children finding ways around normal restrictions. Parents can also control who the child can contact, and even set up which functions are available in each app, like turning off the camera’s selfie mode.

The setup is easy: choose which apps to allow, set a special passcode, and triple-click the side or home button to switch in and out of Assistive Access. The child cannot bypass these restrictions unless they know the passcode. The phone cannot be turned off in this mode, and if an app crashes, only the parent can reset it by leaving Assistive Access.

This setup means parents don’t need to buy a new “dumb phone” or pay for third-party apps to lock down the iPhone. It uses an old iPhone, so there is no extra cost. The phone can still be tracked if lost, and the child gets important features like navigation and music. The article also mentions some downsides: Assistive Access can run slowly, voicemail is not available, and there was a bug where the Messages app froze when searching for emojis.

Commenters shared different views. Some were excited about this discovery and thanked the author for sharing an easy, free way to make a child’s phone safe. A few said they wish Apple advertised this feature more, as many parents do not know about it. Others wondered why Apple does not make a special mode just for kids, instead of hiding the tool in accessibility menus. Some pointed out that the feature is not well known, even by Apple Store employees, and they hoped Apple would train staff better. A few users discussed the small problems, like the phone running slow or the lack of voicemail, but said these were minor compared to the benefits. Others said they would try this with older relatives who need simple phones too. Some worried about losing an expensive iPhone but agreed that tracking helps. A few people debated if this solution is better than buying a cheap basic phone, with most agreeing that being able to track and control the phone is worth it.

## Distributed system is slower than a laptop

- 原文链接: [Distributed system is slower than a laptop](https://codegood.co/writing/your-distributed-system-is-slower-than-a-laptop)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48818192)

This article says many companies use big, complex distributed systems when a single good laptop or server would be faster and much cheaper. The author explains how a simple laptop beat large clusters in real benchmark tests.

The article starts with a famous study where researchers ran big data jobs on both a large cluster and a single laptop. The laptop finished faster, even for very large tasks like analyzing Twitter’s follower graph or running PageRank. The study made a metric called COST: how many cores does a distributed system need to beat a single thread? For many systems, the answer was “never.”

The author gives a real-world example: a SaaS company handles 2 billion events a day with Kafka and Flink clusters, 128 CPUs, and several engineers. This costs about $1.4 million a year. But all this work—2 billion events, 1 terabyte a day—can be done easily by one modern server at a tiny fraction of the cost (about $57,000 a year). Most of the cluster’s effort is spent just moving data around and dealing with complexity, not solving the actual business problem.

The reason for this waste is history. Years ago, computers were weak, so splitting up big jobs was needed. Now, hardware is much stronger, but people still use distributed systems by default. Most companies don’t have enough data to need such complex setups. Some big companies, like Amazon and Segment, saved lots of money by switching from microservices to simpler systems.

So why do people keep choosing distributed systems? The article says it’s because frameworks advertise themselves using their own benchmarks, engineers like building complex things for their resumes, cloud vendors make more money selling more machines, and companies worry too much about “scaling” before it’s needed. The author lists four honest reasons to distribute: the data doesn’t fit on one machine, you need high uptime, your users are far apart in the world, or your company is very large. Most teams don’t meet these reasons.

The author’s advice: before building a distributed system, always test if a single-machine setup can do the job. This only takes a week and can save huge amounts of money. Teams who do this learn more about their real needs and often find the simple way is better.

In the comment section, many readers agree with the article and share their own stories. Some say they have seen companies spend millions on clusters when a single server was enough. Others point out that simple systems are easier to maintain and debug.

A few commenters argue that sometimes distributed systems are still needed, especially for high reliability or special cases. Some say that using standard tools makes hiring and onboarding easier—more engineers know Kafka and Flink than custom single-machine setups.

Several people note that “resume-driven development” is real: engineers want to work on exciting new tech, even if it’s overkill. Some mention that cloud vendors push complex architectures because it makes them more money.

There are also worries about “premature optimization”—some say it’s risky to build everything simple and then scramble to scale later if the company grows fast. Others reply that it’s easy to scale if you actually reach that point, but most never do.

A few comments mention that modern laptops and servers really are very powerful, but not everyone knows how to use them well. Some people admit they didn’t believe single-threaded solutions could be that fast until they tried it.

Finally, several readers say that measuring and testing should always be part of system design. They agree with the article: checking the single-machine baseline is cheap and smart, but often skipped.

## No leap second will be introduced at the end of December 2026

- 原文链接: [No leap second will be introduced at the end of December 2026](https://datacenter.iers.org/data/latestVersion/bulletinC.txt)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48846281)

The article says there will be no leap second added at the end of December 2026. Leap seconds are sometimes added to keep our clocks (UTC) in sync with the Earth's rotation.

The group that manages time, the International Earth Rotation and Reference Systems Service (IERS), checks if a leap second is needed every six months. They use measurements to see the difference between atomic time (TAI) and solar time (UT1). Right now, since January 1, 2017, UTC is 37 seconds behind TAI. If the Earth's rotation slows down or speeds up, a leap second can be added or removed. But for now, this update says everything is stable, and no changes are needed. The next possible date for a change will be checked again in six months, as usual. This helps computers, networks, and people who use precise time to plan.

In the comments, some people are happy there is no leap second because it makes programming easier. Leap seconds can cause bugs and problems in software, especially for things that need exact time, like databases or distributed systems. Others say the leap second is an old solution, and we should maybe stop using them in the future. Some users point out that leap seconds are rare, but when they happen, they can cause big issues for big companies like Google or for global systems. A few commenters discuss possible new ways to handle time, like letting UTC drift and fixing it with bigger changes less often. Others remind us that leap seconds are about keeping time linked to the Sun, which still matters for some jobs. Some people share stories about past leap seconds causing outages or confusion. There are also jokes about how hard it is to handle time in code. Finally, a few users remind everyone that, even without leap seconds for now, this could change in the future if the Earth's rotation changes.

## A road to Lisp: Why Lisp

- 原文链接: [A road to Lisp: Why Lisp](https://scotto.me/blog/2026-07-09-why-lisp/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48845209)

This article is about why Lisp is a special programming language and what makes it different from others. The author talks about how Lisp looks strange at first, with lots of parentheses and unusual syntax, but says that learning Lisp can change how you think about programming.

The main idea is that Lisp is more powerful and flexible than most languages. The author explains that Lisp lets you write code that can change the language itself, thanks to something called macros. Macros in Lisp are not like macros in C or Rust; they let you create new programming tools and control how code is run. For example, if Lisp does not have a “while” loop, you can add it yourself with a macro. The author shows how macros keep code unevaluated until needed, which is different from normal functions.

Another important point is that Lisp uses lists for everything—both data and code. This is called “code as data” or homoiconicity. It means you can easily write programs that write other programs, making Lisp very good for creating new language features and tools for your own needs.

The article also talks about Lisp’s interactive style. Instead of always stopping to recompile and rerun your program, you can keep a Lisp process running and change your code live. This is called REPL-driven development. You can test and update everything on the fly, which is like evolving your program while you work. This kind of workflow was in Lisp long before “hot-reloading” became popular in other languages like JavaScript.

Lisp also makes it easy to build software that users can extend. Because you can create your own domain-specific languages (DSLs) inside Lisp, you can let users write plugins or customize your app using the same tools you use. The author gives examples, like a CMS where users can write templates with Lisp, or math software where people use Lisp-like code to draw graphs. Big software like Emacs and AutoCAD use their own versions of Lisp to let users automate and extend them.

Finally, the author says Lisp never became the most popular language, and maybe it never will. But its mix of features—extensibility, interactive development, and code-as-data—make it powerful and unique. Even if you don’t use Lisp daily, learning it can make you a better programmer.

In the comments, some people say they tried Lisp and found the syntax hard at first, but later understood its beauty. Others say macros are both powerful and dangerous, because you can create confusing code if you’re not careful. A few readers mention that while Lisp can do amazing things, it’s not always the best choice for every project, especially in big teams or companies that want “mainstream” languages.

Some users talk about their love for the REPL and how it makes fixing bugs and testing ideas much faster. Others share stories about how Emacs or AutoLISP helped them automate tasks that would be hard in other languages. There are also comments from people who think Lisp’s “everything is a list” design is elegant, while others say it can make code hard to read for beginners.

A few commenters point out that many new languages have borrowed ideas from Lisp, like macros or REPLs, but don’t go as far with code-as-data or extensibility. Some people wish more programmers would give Lisp a chance, while others are fine using it as a secret tool for special problems. At the end, there’s agreement that even if Lisp is not for everyone, learning it can open your mind to new programming ideas.
