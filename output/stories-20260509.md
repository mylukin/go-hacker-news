## FreeBSD: Local Privilege Escalation via Execve()

- 原文链接: [FreeBSD: Local Privilege Escalation via Execve()](https://www.freebsd.org/security/advisories/FreeBSD-SA-26:13.exec.asc)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48077971)

A new bug in FreeBSD lets someone on the system get superuser rights using the execve() function. This bug affects all supported versions of FreeBSD and is tracked as CVE-2026-7270.

The problem is with how execve(), a system call that starts new programs, handles its arguments. Because of a mistake in the code, data can overflow from one buffer to another. This means someone without admin rights could trick the system and get root access. There is no simple workaround—systems must be updated and rebooted to fix the bug. The FreeBSD team has provided updates for all the main versions. Users can update using the pkg tool, the freebsd-update command, or by applying a patch and rebuilding the system. The advisory lists all the correct git commit hashes for people who update from source. The bug was reported by Ryan from Calif.io and fixed quickly by the FreeBSD team.

In the comment section, some people are surprised such a basic bug still happens in core system code. Others thank the FreeBSD team for fixing it fast and giving clear update steps. A few users point out that buffer overflows are a common cause of security holes in all operating systems, not just FreeBSD. Some ask if OpenBSD or Linux has the same bug, but others say it seems to be FreeBSD-specific. There are users who say this shows why automatic updates and reboot reminders are important for servers. Some discuss how hard it is for small teams to keep up with patching and testing after updates. Others share tips on using tools to check for CVEs on their servers. A few people worry about how long it takes some users to update production systems even when a serious bug like this comes out. Finally, many agree that open security advisories and fast patches are signs of a healthy project.

## I’ve banned query strings

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48076173)

This article is about why Chris Morgan stopped allowing query strings—like ?utm_source=example—in URLs on his website. He does not like when people add tracking data or marketing tags to his links, especially without his permission.

Chris says if someone wants to know where a visitor came from, they should look at the “Referer” header, not add extra tags to the URL. He feels adding tracking codes can invade user privacy and is not respectful. He points out that UTM parameters are often used for tracking by marketers, but he thinks only the website owner should decide to use them. He also mentions some past use of query strings for things like cache-busting, but he is okay breaking those old links, as they are not needed anymore. Now, he blocks any request to his site that has an unknown query string. He even invites readers to try it by adding a query string to his URL to see what happens. Chris says it’s his site, so he can do as he likes—and encourages others to do what they want with their own sites. He explains that he set up this rule in his Caddy web server configuration.

In the comments, some people agree with Chris, saying they also dislike tracking codes and find them messy or annoying. Others say query strings are sometimes useful—not just for tracking, but also for things like filtering, searches, or sharing state in web apps. A few mention that blocking all query strings could break real features, like when other sites or services depend on them. Some users warn that this might cause confusion or problems for people linking to the site. Others think it is fine for a small personal website, but would be risky for a big or public-facing site. One commenter points out that tracking is often more about analytics than privacy invasion, but agrees it should be the site owner’s choice. Some users like the idea of keeping URLs clean and simple. Overall, people respect Chris’s right to set his own rules, even if they would do things differently.

## Show HN: I made a Clojure-like language in Go, boots in 7ms

- 原文链接: [Show HN: I made a Clojure-like language in Go, boots in 7ms](https://github.com/nooga/let-go)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48076815)

This article is about “let-go,” a new programming language that looks and feels like Clojure but is written in Go. The main idea is to have a fast, small, and easy-to-use Clojure-like language that starts in just 7 milliseconds and can be used for building standalone apps, web apps (with WASM), or as a scripting layer inside Go programs.

The creator explains why let-go exists: you can create single-file binaries, run apps in browsers, enjoy super-fast startup, and have a much smaller memory footprint compared to tools like Babashka or the Java JDK. Let-go includes many standard libraries out-of-the-box, such as HTTP servers, JSON, core.async channels, and more. You can use Go code directly from let-go, and vice versa, making it easy to mix both languages in the same project. The language tries to be as close to Clojure as possible, passing over 95% of a big Clojure test suite, but it's not a full drop-in replacement—some edge features are missing or work differently.

Let-go supports Babashka pods, so you can connect to databases or other services easily. It has special features for Go interop, like mapping Go structs to let-go records and calling Go functions. Benchmarks show that let-go is much faster and uses less memory than similar tools, especially for short-lived tasks or small apps. The article lists some missing features, like no support for certain Clojure concurrency types and some differences in how numbers or regex work compared to original Clojure.

Real projects have already been built with let-go, like games or container runtimes. You can try it online in your browser, install it via Homebrew, download a binary, or build from source if you have Go installed. Running, compiling, and distributing let-go apps is simple: you can make bytecode files, standalone executables, or even full web apps with one command. There’s also a built-in nREPL server for connecting with popular editors like Emacs or VS Code. Let-go can be embedded in Go applications, making it useful as a scripting engine.

In the Hacker News comments, many people were excited by how fast let-go starts up and how small the binary is. Some praised the clean Go integration and thought it could be useful for scripting or serverless tasks. Others pointed out the impressive Clojure compatibility and liked that you can build web apps with WASM so easily. A few users were curious about the missing features, asking if things like Clojure’s STM or certain macros might be added later. Some commenters compared let-go to Babashka or Joker, discussing trade-offs like startup time versus language features. A couple of people worried about long-term maintenance or how closely let-go could track Clojure changes in the future. Others just liked playing with the online REPL and said it was fun to see a new Clojure dialect with fresh ideas. Overall, the mood was supportive, with many developers saying they wanted to try using let-go for small tools or as an embedded scripting language in Go projects.
