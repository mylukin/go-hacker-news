## Elixir v1.20: Now a gradually typed language

- 原文链接: [Elixir v1.20: Now a gradually typed language](https://elixir-lang.org/blog/2026/06/03/elixir-v1-20-0-released/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48388324)

Elixir just released version 1.20, and now it can check types in your code step by step, even if you don’t write type annotations. This means Elixir finds more bugs and dead code before you run your program.

The new type system is “gradual.” It uses a special type called `dynamic()`. This type lets Elixir check types without being too strict. For example, if a variable could be an integer or a string, Elixir won’t complain unless you do something impossible, like using both as a map. Elixir only warns you if it’s sure something will crash during running the code.

Elixir’s type checking is based on set theory, using unions, intersections, and negations. This helps the system be smart about what types variables can be. If you write a function that checks if a variable is a list or an integer, Elixir can figure that out and narrow down the type. It also understands guards and pattern matching, so it can catch mistakes like missing fields in maps or using the wrong data shape.

The `dynamic()` type is different from the usual `any()` in other languages. It doesn’t just say “anything goes.” Instead, Elixir tries to follow how the type changes as you use it, making the system better at finding real problems but not false alarms.

This release also made Elixir faster to compile, especially on computers with many cores. There’s a new option to change how modules are defined, which can help big projects build even quicker. The team has done a lot of research to make sure the type system works well and is easy to use.

For the future, Elixir wants to add more advanced types, like recursive and parametric types, but only if they work fast and well. The team is still working on those features.

From the Hacker News comments, many people are excited that Elixir is getting types, saying it helps find bugs early and makes code safer. Some users are surprised that Elixir can do type checking without extra annotations, and they like that it won’t flood you with useless warnings. Others are worried it might make Elixir less simple or slower, but the article shows that compile times are actually faster now.

A few commenters point out that gradual typing is tricky and can sometimes miss bugs, but most agree that only reporting “verified bugs” is a good balance. Some developers ask how this compares to types in languages like TypeScript or Python, and they like that Elixir’s system is more precise and has fewer false positives.

Some users are interested in when they can start writing their own type annotations, while others are happy the system works automatically for now. There are questions about how this will affect existing Elixir codebases, and people are glad the changes don’t break old code.

Overall, most commenters are positive, calling this a big step for Elixir and saying it might attract more developers. Some want to wait and see how the new features work in real projects, but they appreciate the focus on both safety and developer happiness.

## Gemma 4 12B: A unified, encoder-free multimodal model

- 原文链接: [Gemma 4 12B: A unified, encoder-free multimodal model](https://blog.google/innovation-and-ai/technology/developers-tools/introducing-gemma-4-12b/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48385906)

Gemma 4 12B is a new AI model from Google DeepMind that can understand images, audio, and text, and it works on normal laptops. The model is special because it does not need separate encoders for vision or audio—everything goes straight into the main language model.

The article explains that Gemma 4 12B is between Google’s earlier small and large models, offering strong performance but with less memory use. You can run it on a laptop with 16GB memory, which means developers and users do not need big servers. It is the first mid-size Gemma model to handle audio directly, making it useful for voice and sound tasks.

Gemma 4 12B’s architecture is new. Unlike older models that use separate encoders to process pictures and audio, this model uses a simple way: for images, it just does a quick calculation to turn the picture into something the model can use; for audio, it skips the encoder and puts the sound into the model as if it were text. This makes the model faster and uses less memory.

The article says Gemma 4 12B can do hard reasoning tasks, almost as well as Google’s much larger 26B model. It can also predict several words at once, which helps it answer faster. The model is open source, free for anyone to use, and works with many tools popular among developers. There are guides, sample apps, and a “Skills Repository” to help people build new AI agents using Gemma 4 12B.

In the Hacker News comments, many people think the encoder-free design is a clever way to save memory and speed up the model. Some are excited that it is so easy to run on a normal laptop, not just in the cloud. Others ask if skipping separate encoders will make the model miss details in pictures or sounds. A few users compare Gemma 4 12B to other models, like Meta’s Llama or OpenAI’s models, and wonder which is better for real tasks.

Some developers like that it is open source and works with many tools, making it easy to try and change. Others worry about the quality of audio and vision tasks, since direct projection might not be as good as special encoders. There are also comments about the Apache 2.0 license, which people like because it allows commercial use.

A few users share that they have already downloaded the model and found it runs well on their laptops. Some want more benchmarks and real-world tests. Others hope the model will be used for helpful apps, like speech-to-text or smart assistants, and are happy Google supports open development. Overall, the comments show both excitement and some careful questions about how the new design works in practice.

## Artificial intelligence is not conscious – Ted Chiang

This article talks about whether artificial intelligence, like Claude from Anthropic, is truly conscious. The author, Ted Chiang, looks at how people sometimes treat AI as if it has feelings or thoughts like a person.

Chiang shares examples from Anthropic, an AI company. They wrote a long document called Claude’s “constitution,” which talks about Claude’s values and behavior. The document is written as if Claude is the one reading it. It even says things like “Claude may have some functional version of emotions or feelings.” The CEO of Anthropic has said that maybe AI could be conscious one day. One of the company’s philosophers even said she wants Claude to be “very happy” and worries about Claude feeling anxious if people are mean to it online.

Chiang thinks this is strange. He says that writing to Claude as if it can feel or worry is not based on reality. He believes large language models like Claude do not have real feelings or thoughts. They work by predicting words and sentences, not by understanding or feeling anything. Chiang warns that treating AI as if it is conscious can lead to problems. For example, it may make people think AI deserves rights or special treatment when it really does not.

He explains that anthropomorphism—treating non-human things as if they are human—can make people confused about what AI really is. He says that, right now, AI does not have the same kind of mind as a person. It just follows rules and patterns from data. Chiang believes we should be careful not to give AI moral status or think it can truly be happy or sad.

Others point out that even if AI acts like it has feelings, it is only copying patterns from training data.

However, some commenters have a different view. They say that as AI gets better, it might become hard to know if it is truly conscious or just acting like it is. A few think that someday, AI could develop something like feelings, especially if it becomes more advanced. There are also people who say we should keep an open mind—they think we do not fully understand consciousness, so we cannot be sure about AI’s limits.

Some comments joke about people worrying too much about AI feelings. Others share stories about people getting attached to chatbots or virtual pets, showing how easy it is to imagine machines have feelings. A few people talk about the ethics of treating AI well, just in case, while others say this is a waste of time. ## Pwnd Blaster: Hacking your PC using your speaker without ever touching it

- 原文链接: [Pwnd Blaster: Hacking your PC using your speaker without ever touching it](https://blog.nns.ee/2026/06/03/katana-badusb/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48382310)

This article explains how the author found a way to hack a Creative Sound Blaster Katana V2X speaker using only Bluetooth, without ever touching the device. The speaker is usually connected to a PC over USB, and it also has Bluetooth for controlling settings from a phone.

The author started by looking at how the speaker’s firmware works and found that it uses a simple protocol (CTP) for communication. To make changes or update the firmware over USB, you need to pass a challenge-response authentication. But the key for this is always the same and can be found in the app’s files. Firmware updates are just ZIP-like files, and there is no real protection—just a simple checksum that is easy to change.

When the author checked the Bluetooth functions, they found that anybody nearby can connect to the speaker over Bluetooth, without pairing, and send CTP commands. This means anyone can read info, change settings, and even upload new firmware over Bluetooth. The process is a bit slow, but it works.

The speaker has a microphone, so an attacker could make custom firmware to secretly listen to conversations. But the bigger risk is that the speaker is seen by the PC as a trusted USB device. With a firmware change, the speaker can pretend to be a keyboard and type commands on the connected computer, just like a “Rubber Ducky” attack. This could let an attacker run any command on your PC.

The author found technical ways to make this work, like changing the USB descriptors to add keyboard functions and reusing code already in the firmware. They were able to make the speaker type “echo pwned” on the PC without any physical access.

Creative, the company, was very slow to respond and does not consider this a security problem, so there is no official fix. The author made their own patch to block CTP over Bluetooth, but this breaks the mobile app.

People in the comments had a lot to say. Some were surprised that a speaker could be used to attack a PC like this, and many thought this was a serious security problem. Others pointed out that Bluetooth devices often have weak security, so this is not a new type of issue. Some users were frustrated at Creative for not caring about the security risk, saying companies should take these reports more seriously. A few commenters liked that the author shared technical details and even made a patch, but some worried about users having to apply unofficial fixes themselves. There were also suggestions that people should turn off Bluetooth or avoid connecting untrusted devices to their computers. Some people discussed the bigger problem of hardware companies ignoring security researchers, and a few joked that “smart” devices keep making new security problems. Others asked if this kind of attack could be used on other USB speakers or devices, and some were curious about the author’s reverse engineering skills. Overall, the comments showed a mix of fear, frustration, and respect for the technical work.

## Gooey: A GPU-accelerated UI framework for Zig

- 原文链接: [Gooey: A GPU-accelerated UI framework for Zig](https://github.com/duanebester/gooey)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48386725)

Gooey is a new UI framework for building fast, GPU-powered apps in Zig. It works on macOS (Metal), Linux (Vulkan/Wayland), and, in the future, the web (using WebGPU/WASM).

The framework is designed to be both immediate and retained mode, making it easy to update the UI and keep state in sync. Gooey uses the GPU for rendering, so everything is smooth and quick. It supports anti-aliasing, animations, custom shaders, and many modern UI features. The API is declarative—meaning you describe what you want the UI to look like, and Gooey handles the details. You get built-in components like buttons, text inputs, checkboxes, modals, tooltips, lists, tables, images, and more. The layout system is similar to flexbox, so arranging things is simple. Gooey also supports themes (light and dark), accessibility (like screen readers), and international text input. You can build apps with zero external Zig dependencies; it only needs system libraries.

The framework separates "state" (your app’s data and logic) from "layout" (how things look). There are clear patterns for updating state, handling events, and running background tasks without blocking the UI. For example, Gooey uses Zig’s new async features to do network or file work on a separate thread and then sends results back to the UI. Handlers can get arguments (if they’re small enough), and you can defer actions to avoid event handler problems (like when opening a file dialog). There is built-in support for form validation, animations, keybindings, and logging.

On Linux, Gooey uses Wayland and Vulkan for speed and modern features—no X11 fallback. Text is rendered with FreeType and HarfBuzz, so it looks like other Linux apps. Clipboard, file dialogs, HiDPI, and input methods are all supported. On macOS, you get liquid glass effects and native file dialogs. The web path is blocked for now due to Zig issues, but the code is ready for when support returns.

The Hacker News comments highlight several points. Some users are excited that Zig now has a real, modern UI toolkit, and praise Gooey’s careful design and documentation. Others compare it to frameworks in other languages, like Rust’s Druid or GPUI, and note similarities. A few are impressed by the zero-dependency approach and the focus on accessibility. There’s discussion about Zig’s async model and how Gooey leverages it for background tasks. Some users point out that Wayland-only support is good for the future but limits adoption right now for Linux users still on X11. Developers like the idea of combining immediate and retained mode, as it can make building UIs easier and more testable. There are questions about the state of the WASM/WebGPU build and hopes that Zig’s upstream fixes will allow web apps soon. A few wish for Windows support, while others respect the focus on quality for just macOS and Linux. Some ask about performance, memory use, and how Gooey handles large lists or tables—others point to the virtual list examples as proof that it can scale. Finally, users recognize the challenge of making a cross-platform UI toolkit and thank the author for open-sourcing and documenting the project so well.

## Journey to JPEG XL: open-source experiments shaped the future of image coding

- 原文链接: [Journey to JPEG XL: open-source experiments shaped the future of image coding](https://opensource.googleblog.com/2026/06/journey-to-jpeg-xl-how-open-source-experiments-shaped-the-future-of-image-coding.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48390688)

This article is about how JPEG XL, a new image format, was created using many open-source ideas and experiments. It explains the long process of improving how images are stored and shared on the internet.

The article starts by saying that old formats like JPEG worked well for years, but new screens and cameras needed better image quality and smaller file sizes. The team first worked on improving old formats instead of making something new right away. They tried new ideas in projects like WebP Lossless, Brotli, Guetzli, and Brunsli. WebP Lossless used a smart way to choose how to shrink image data, and Brotli used similar ideas for compressing files. Guetzli used a tool called Butteraugli to make JPEG images smaller without losing visible quality, but it was slow. Brunsli allowed people to make JPEG files smaller without any loss at all.

They learned from these projects and listened to people who needed high-quality images, like online stores. In 2017, they combined their best tools into a new open-source project called PIK. Later, they joined ideas with another project called FUIF. Together, these became the base for JPEG XL, which could compress images much more than before and still look good. JPEG XL uses special tricks like VarDCT to make images smaller and keep quality high.

JPEG XL is now used in many areas: digital cameras (like Apple ProRAW), medical images (DICOM), and documents (PDF, EPUB). Big companies like Apple, Adobe, Microsoft, and Linux support it. Some web browsers, like Safari, support JPEG XL, while others like Chrome and Firefox are still testing it. There are also new tools and chips being made for JPEG XL, so it can work fast even on small devices.

In the end, the article says that building JPEG XL took many years, lots of experiments, and help from open-source communities. The team hopes JPEG XL will be used for many years, just like JPEG was.

In the comments, some people are excited about JPEG XL’s better quality and smaller file sizes. They like that it is open-source and works well for both photos and important data like medical images. Others are worried about slow support from big browsers like Chrome and Firefox, making it hard for websites to use JPEG XL right now. A few users say that switching image formats can be risky for long-term storage, and they hope JPEG XL will still be supported in the future.

Some developers mention that JPEG XL’s tools are easy to use and the documentation is good. Others say it can be hard to convince companies to change from old JPEG files because the old format is everywhere. Some people ask if JPEG XL is better than other new formats like AVIF. A few reply that JPEG XL is faster and has better quality in some cases, but AVIF is already supported in more browsers.

Overall, the comments show hope that JPEG XL will become popular, but also worry about slow adoption and competition with other formats. Many agree that open-source projects helped JPEG XL become a strong new option for images.
