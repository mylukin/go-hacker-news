## Let your Coding Agent debug the browser session with Chrome DevTools MCP

This article is about a new feature in Chrome DevTools that lets coding agents help debug live browser sessions using the MCP server. Now, you can let an AI or script debug your current Chrome session—no need to sign in again or start a new browser.

The main news is that coding agents can reuse your active browser session. For example, if you are logged into a site, your coding agent can access that session to debug a problem behind a login wall. You can also let your agent join an ongoing debugging session in Chrome DevTools. If you spot a broken network request or a bad element, you can select it and then ask your coding agent to help. This makes it easy to switch between doing things yourself and letting an agent help.

The new setup works by adding an “autoConnect” option to the MCP server. This connects it to any running Chrome window, but you must first enable remote debugging in Chrome (at chrome://inspect/#remote-debugging). Each time an agent wants to connect, Chrome will show a pop-up asking for your permission. While the agent is connected, you will see a warning banner that Chrome is being controlled by software.

To get started, you enable remote debugging in Chrome version 144 or newer, then run the MCP server with the right command. You might need to use the beta channel for now. Once set up, you can use tools like gemini-cli to ask your agent to, for example, check a site’s performance. The agent will open the site and collect data, all using your current browser session.

This helps people who want to combine manual debugging with automated help. For example, you might find a bug, look at it with DevTools, and then let your agent investigate further or suggest a fix. The Chrome team says this is just the start—they plan to give agents access to even more DevTools panels in the future.

In the Hacker News comments, some users are excited about the improvement. They say it will save time, especially for tricky bugs that only appear after login. Others like the extra security steps—requiring user permission before an agent connects. Some users wonder about privacy, asking if agents could access sensitive data. There are questions about how safe it is to let an agent control your browser, and if the warning banner is enough. A few people are happy that automation and manual debugging can now work together, while others worry this could lead to new types of browser attacks if not handled carefully. ## The 49MB Web Page

- 原文链接: [The 49MB Web Page](https://thatshubham.com/blog/news-audit)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47390945)

The article looks at how a single New York Times web page can be 49 megabytes in size and why modern news sites are so heavy and hard to use. The author compares this to old software like Windows 95 and MP3 music files, explaining that one news page now is bigger than an entire operating system or a whole music album.

The main reason for this size is ads and tracking scripts. When you open a news article, your browser loads hundreds of things—mostly ads and trackers, not the news itself. These scripts run ad auctions, collect your data, and send it out to many companies, all before you finish reading the headline. This makes your device slower and can even make it hot or drain the battery, especially on phones.

The author explains how news websites put up many barriers before you see the content. There are cookie banners, pop-ups asking you to subscribe, notification prompts, and ads everywhere. Sometimes, you have to close two or more windows just to read the article. Only a small part of the screen shows the news; the rest is filled with ads and banners. The text can also jump around when ads load, so you lose your place while reading.

Other tricks include auto-playing video ads that follow you as you scroll, sticky headers that take up space, and close buttons that are hard to find or press. Some sites even block part of the article until you click "Continue Reading," letting them show more ads. All these things make the site hard to use and annoying for readers.

The author says these choices come from business needs. Publishers want to make money from ads and keep you on the page longer, but this hurts the reader's experience. He suggests better ways: don’t show pop-ups right away, combine banners, and let people read before asking them to sign up. He also says to reserve space for ads so the layout doesn’t shift, and use light, simple versions of sites—like text.npr.org—when possible.

In the comments, many people agree that news sites are heavy and frustrating. Some say they use ad blockers or even avoid these sites completely. Others remember when websites were fast and simple, and wish things could go back to that. A few point out that not only big news sites, but many modern websites, are too large and slow because of bloated frameworks and unnecessary scripts.

One commenter says publishers feel trapped: they need ad money, but these ads drive users away. Another person explains that advertisers want more data and "engagement," which leads to more tracking. Some say there should be more rules against these dark patterns, or at least real penalties for breaking web standards and hurting the user experience.

A few users mention they use RSS feeds or "reader" modes to get just the text. Some suggest supporting sites like NPR’s text-only page or using browsers with built-in blockers. Others share that they have given up on many news sites and get news from newsletters, forums, or social media instead.

Overall, commenters agree the web is much worse for readers now, and many wish publishers would respect their users more. Some hope that if enough people stop visiting these sites, things might change, but others are less sure—business pressures are strong, and change comes slowly.

## //go:fix inline and the source-level inliner

- 原文链接: [//go:fix inline and the source-level inliner](https://go.dev/blog/inliner)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47339463)

This article talks about a new feature in Go 1.26 called the source-level inliner, which helps update Go code by inlining function calls directly in the source code. The main point is that Go’s new `go fix` tool, with the `//go:fix inline` directive, lets package authors mark old functions so calls to them can be swapped out with their newer replacements in user code.

The inliner works by replacing a function call with the body of the function, putting the arguments in place of the parameters. This is different from what the compiler usually does, because these changes happen in your source code, not just while building. For example, if a function is deprecated and now just calls another function, adding `//go:fix inline` will let the tool update all code to use the new function directly. This can help big codebases switch away from old APIs.

The inliner is careful to keep your program working as before. It checks for many tricky cases. For instance, if a parameter is used more than once, it avoids copying values too much. If the function arguments have side effects, like changing variables, the inliner keeps their order safe. It also makes sure variable names are not mixed up by accident. Sometimes, it cannot inline code if it would change how things work, for example with deferred functions or tricky constants.

The tool can also update types and constants, not just functions. If you used an old constant or type, `go fix` can swap it for the new one. The article gives examples, like moving from an old math package to a new one just by adding the right annotation and running the tool.

Behind the scenes, the inliner has to solve many hard problems, like making sure substitutions don’t cause compile errors or break code style. It tries to make the changes as neat as possible, but sometimes a human will still need to clean up.

In the comment section, some people are excited about this new tool and say it will make big refactors much easier, especially in large codebases. Others share worries about automatic changes, saying they might make the code messier or hide problems. A few users compare this to similar tools in other languages, like Java or C++, and are glad to see Go catch up. One person points out that inlining at source level can help remove old dependencies over time, which keeps projects cleaner. However, some warn that the tool might not always know the best style, so code review is still important. Another common point is the balance between safety and tidiness—some want the tool to be more aggressive, while others prefer it to be careful and conservative. A few users ask if this could be expanded for more complex refactoring in the future. Overall, people see this as a helpful step for Go, but with the reminder that no tool is perfect and some manual changes will always be needed.

## Separating the Wayland compositor and window manager

- 原文链接: [Separating the Wayland compositor and window manager](https://isaacfreund.com/blog/river-window-management/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47388137)

This article talks about a new way to design Wayland compositors and window managers. The new release of “river” splits the compositor and window manager into two programs instead of one.

The old way in Wayland was to have one big program do everything: showing windows, handling input, and arranging windows. This made it hard to write a new window manager because you had to write the whole compositor too. The new “river” system changes this by letting the window manager be a separate program. Now, many different window managers can work with river.

The river protocol lets window managers control things like window position, key shortcuts, and window focus. The compositor part (river) takes care of drawing everything on the screen smoothly and quickly. The article explains that, in X11, splitting display server and compositor caused extra delays and input problems. Wayland fixed this by combining them, but also put the window manager in the same program, which was not needed.

With river, the protocol is designed to avoid extra delays or extra messages between programs, so everything stays fast. There is an idea called “frame perfection.” This means when you move or open windows, you never see gaps or windows in the wrong place, even for a moment. River groups updates together to keep everything in sync.

The protocol splits window state into two parts: one for managing windows (like their size and focus), and one for how they are drawn. Updates are done in batches, so the window manager is only active when needed, making things efficient.

Making the compositor and window manager separate makes it much easier to write a new window manager. You can focus only on how you want windows to work, not on the hard parts of drawing to the screen. Crashes in the window manager do not break everything—you can restart it easily. Now, you can even write a window manager in slower, easier languages, and it will not affect performance.

There are already 15 window managers that work with river, so more choice is coming, like in X11. But river does not support things like 3D desktops or wild window effects yet. The project is still growing, and more features will come, but it is already good for daily use.

In the comments, some people like this new design because it lowers the barrier for making custom window managers. They think it will help bring more creativity to Wayland, just like X11 had many window managers. Others worry about possible limits, like missing features for special use cases or fancy effects. Some users are excited to try river, while some are waiting for better documentation or more mature tools.

A few comments ask about security, wondering if splitting these parts could open new risks. Others say that debugging will be easier now, since you do not lose your session if the window manager crashes. Some people like the idea of being able to write a window manager in languages like Python or JavaScript. There’s also talk about how this could make switching window managers on the fly much easier.

Some comments compare river to other projects like sway or wlroots. A few mention that while this idea is not totally new, river’s protocol is more formal and stable, which is good for developers. Overall, the community is hopeful that this new approach will make Wayland more flexible and fun to use.

## What makes Intel Optane stand out (2023)

- 原文链接: [What makes Intel Optane stand out (2023)](https://blog.zuthof.nl/2023/06/02/what-makes-intel-optane-stand-out/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47388141)

Intel Optane is a special kind of SSD that came out in 2017 for professional and consumer use. The article explains why Optane is different from normal SSDs and what makes it important for tech people.

Optane uses a technology called 3D XPoint, which is not like the common NAND flash found in most SSDs. Because of this, Optane drives have very low latency, high durability, and very good write consistency. The article compares two Optane models: the P4800X and P5800X. These drives can handle many more writes per day than regular SSDs, making them great for systems that need to write a lot of data all the time. Optane also has power-loss protection, which keeps your data safe if the power goes out. Another big point is performance: Optane drives can read and write data much faster and more consistently than normal NAND SSDs, even when the drive is under heavy use. Unlike NAND SSDs, which can slow down when they get full or busy, Optane keeps its speed. This makes it useful for things like caching, databases, and places where fast, steady storage is needed. The article also says that Intel stopped making new Optane tech in 2022, but you can still buy these drives for now. The author tested Optane drives and found them to be very good for high-performance tasks. Even though NAND SSDs are getting better and cheaper, Optane still stands out for special uses.

In the comments, one reader says this topic is important because it shows where hardware is going in the future. They believe fast, direct storage like Optane will become normal in IT soon. The reader also praises the article for explaining everything clearly. The author replies and thanks the reader for the feedback. On Hacker News, some people are sad that Intel ended Optane, saying it was a unique and useful technology. Others think Optane was too expensive and had low capacity, so it was hard for most people to use it. Some users say they loved Optane for special jobs, like ZFS or databases, while others report it was hard to justify the cost for common uses. A few wonder if new memory technologies, like CXL, will replace what Optane offered. Overall, people agree that Optane was different and had strong points, but it did not fit every need and market forces made it hard to keep going.

## C++26: The Oxford Variadic Comma

- 原文链接: [C++26: The Oxford Variadic Comma](https://www.sandordargo.com/blog/2026/03/11/cpp26-oxford-variadic-comma)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47332593)

C++26 will soon require a comma before the ellipsis (`...`) in variadic function parameters, making code like `void foo(int...);` deprecated in favor of `void foo(int, ...);`. This change makes C++ code match C, clears up confusion, and prepares the language for future updates.

The article explains that C++ has allowed both `void foo(int, ...);` and `void foo(int...);`, but only the first is valid in C. The version without the comma comes from old C++ and is now seen as confusing, especially after C++11 added parameter packs, which also use `...`. Many developers mix up ellipsis parameters and parameter packs because of the similar syntax. For example, `template<class Ts> void f(Ts...);` does not declare a parameter pack, but it looks similar to one. There are also strange cases like `void h(auto......);`—six dots—which are legal but very unclear.

C++26 will only deprecate the comma-less version, not remove it. Tools can easily update old code by adding a comma. The change will not break current code but will warn developers to update. The article says this cleanup helps future C++ features, since the old syntax blocks new uses. The main goal is to make the language clearer and more like C.

In the comments, some people are happy about the change, saying it reduces confusion and makes the language easier for new programmers. Others find it funny that such an old, small syntax rule stuck around so long. A few are surprised that `void foo(int...);` was ever legal, since they never saw it used in real code. Some worry about tools or old code that might break, but most agree that automated tools can fix this easily. One commenter points out the benefit for mixed C/C++ projects, as code will now look and work the same in both languages. Another person says it is a good example of how C++ sometimes cleans up old mistakes, even if slowly. A few developers joke about the “Oxford comma” name, saying it’s a clever way to describe the fix. Overall, the community seems to like the change and sees it as a small but useful improvement.

This article is about teaching a humanoid robot to play tennis using imperfect human motion data. The main idea is to help robots learn how to move and hit tennis balls like people, even if the training data is not perfect.

The researchers built a system called LATENT. Instead of using full, perfect recordings of people playing tennis, they used small pieces of human motion—just short parts that show basic tennis moves. These pieces are easier to collect and still give helpful hints about human movement. The robot learns from these pieces and then puts them together to make full tennis actions. The system trains the robot to hit incoming balls and send them to certain spots on the court while moving in a way that looks natural, like a real athlete.

The team also designed ways to make sure the skills learned in simulation work on a real robot. They tested their method on a Unitree G1 humanoid robot. The robot could rally with human players, hitting the ball back and forth several times without missing. It even played against different people and used good footwork to move around the court. The researchers also tried letting the robot play against itself in a simulation.
