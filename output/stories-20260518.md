## Anthropic acquires Stainless

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48182281)

Anthropic just bought Stainless, a company that builds tools for developers to connect with APIs. Anthropic says this will help their AI, called Claude, work better with more software and data.

Stainless started in 2022 and has helped create all of Anthropic’s official SDKs. SDKs are code libraries that make it easy for programmers to use Anthropic’s API in different languages like Python, Java, and Go. Many companies already use Stainless to make their own SDKs, command-line tools, and connectors. This means developers can use Anthropic’s AI in their own apps with less trouble. Stainless can quickly turn an API plan into working code for many languages, so the tools feel natural to use. Anthropic’s Head of Platform Engineering, Katelyn Lesse, said that agents (AI programs that do things) are only as good as what they can connect to, so this deal will make Claude more useful. Stainless’s founder, Alex Rattray, said he started the company because SDKs should be as good as the APIs themselves. He is happy to join Anthropic because they have worked together since the start, and the team can keep doing what they love.

Anthropic also built a system called MCP to help agents connect with more tools. Now, with Stainless on the team, they want to make it even easier for developers to use Claude with different data and apps. Anthropic is also working with big companies like PwC and the Gates Foundation, and offering new tools for small businesses.

Hacker News users had mixed reactions. Some people think it is a smart move, because good SDKs are very important for developers and AI agents. They say SDKs save time and reduce mistakes, and Stainless does this well. Others ask why Anthropic needed to buy Stainless—couldn’t they just hire good programmers or use open tools? A few commenters worry that companies will now depend more on Anthropic’s closed systems, and that smaller, open projects might get pushed out. Some users are excited to see AI agents that can do more real work, but others warn that making it easy for agents to act could bring security risks. People also talked about the importance of stable APIs, and how quick changes can break things for developers. In the end, many agree that better tools help everyone, but they hope Anthropic will keep things open and not lock users into one system.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48184322)

This article is a big side-by-side cheat sheet comparing four Lisp languages: Common Lisp, Racket, Clojure, and Emacs Lisp. It lists out code examples and notes for many basic and advanced things like variables, lists, arrays, objects, macros, string handling, files, and more.

The main goal is to help you see the differences and similarities between the Lisps at a glance. For example, you can see how to make a list or a function in each language, what the syntax looks like, and what special features or gotchas there are. It covers things like how “nil” and the empty list are handled (they’re the same in Common Lisp and Emacs Lisp, but not exactly in Clojure), differences in local and global variables, how to do arithmetic, and how to handle strings and regular expressions.

The sheet also shows how to work with files and directories, how to use dictionaries (hash tables), and how to define new types and classes. There are sections on exceptions, macros, and even how to interact with Java from Clojure. Throughout, the sheet mentions not just the code but sometimes also the return values, errors, and special behavior in each language. For example, Clojure uses Java classes for many things, while the others do not. Tail call optimization is not always guaranteed. Macros work differently and some Lisps have hygienic macros, but others do not.

In the comments, people on Hacker News really liked the sheet and found it very useful, especially for switching between Lisps or learning a new one. Some said it’s handy for looking up syntax or behavior quickly. Others pointed out that it’s hard to keep such sheets up to date, since languages change, and some features like libraries or package systems may be missing or outdated in the table.

Some readers said the sheet shows why Lisp is special, but also why it’s hard to pick one dialect: each has strengths and weaknesses. For example, Clojure users liked its integration with Java, but missed some classic Lisp features. Emacs Lisp fans liked its simplicity for scripting inside Emacs. There were debates about which Lisp is best for beginners, and some people wished there were more notes about performance or modern best practices.

A few commenters pointed out that the table is very dense, so it can be hard to read if you’re new to Lisp. Some asked for more context or links to tutorials. Others said they use this kind of chart when porting code or when they need to understand how a familiar concept works in another Lisp. ## We stopped AI bot spam in our GitHub repo using Git's –author flag

- 原文链接: [We stopped AI bot spam in our GitHub repo using Git's –author flag](https://archestra.ai/blog/only-responsible-ai)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48181125)

This article talks about how AI bots are filling GitHub open-source projects with spam and low-quality content. The writers manage an open-source project and noticed a big problem after offering a cash reward for help: AI bots flooded their GitHub issues and pull requests, making it hard for real people to contribute.

First, they saw lots of AI-generated comments and fake plans, which buried real conversations and made it hard to follow important updates. Many pull requests were made by bots and were not even tested. The team had to spend hours each week deleting useless AI spam and closing fake issues. This made their project less welcoming to real contributors.

They tried to fix the problem by making a bot that gives people a reputation score based on good contributions, but it did not stop the spam. Then they made a stricter bot that sometimes blocked real users by mistake. The team worried about stopping bounties and test tasks, but they decided it was more important to protect their project.

Finally, they chose to block anyone from making issues, pull requests, or comments unless they finished an onboarding process. This onboarding includes reading rules about using AI responsibly and passing a CAPTCHA. Only after this can someone become a “prior contributor.” They hacked GitHub’s “prior contributors” setting using Git’s --author flag: after onboarding, a GitHub Action adds the new user's name to a file and makes a commit as that user, which unlocks repo access for them.

They know this is a strong step, but they want to keep real contributors safe and happy. The team worries that AI bots are making open-source work harder and less fun, and even creating security risks.

In the Hacker News comments, some people agree that AI-generated spam is a big problem and say they have seen it in their own projects. Others think GitHub and other platforms should do more to fight spam, so project owners do not need to make strange workarounds. Some users feel sad that open-source is becoming less open because of AI bots. A few people worry that these strict steps might stop new, real contributors from joining projects. Others say strong rules are needed to keep projects healthy. There are also tips about making onboarding easier and suggestions for better ways to tell real users from bots. Some think this shows how AI is changing online communities in ways we did not expect. Overall, people agree the problem is real, but they have different ideas about the best solution.

When the team gave the AIs better tools to manage the business side, they hoped for better results.

Many commenters discussed whether AI can really replace human creativity and personality, or if these models just copy patterns until they get stuck in loops. ## Who will buy your services if you fire us all?

- 原文链接: [Who will buy your services if you fire us all?](https://carette.xyz/posts/who_will_buy_your_services/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48185789)

This article talks about how tech leaders say AI will free people from work and let us have more free time, maybe even give everyone basic income (UBI). But the writer says there’s a big problem: if AI takes all the jobs, who will have money to buy things from these tech companies?

The article looks at history to show that powerful people always find ways to keep control. After slavery ended, former slaves were still forced to work for low pay through contracts. This happened in places like the French colony of Réunion and the British Caribbean. Even when people were “free,” they had no money or land, so they stayed poor and controlled by the same rich people.

Later, during the industrial revolution, companies saw they needed workers to also be customers. Henry Ford paid workers more and gave them weekends, not just to be nice, but so they could buy the cars they built. When wages stopped rising, companies pushed people into debt so they could keep buying things.

Now, with AI, the same problem is happening again. If AI replaces most workers, those people will have no jobs and no money to spend. That’s why tech billionaires talk about UBI: they want the government to give people money, so those people can spend it on tech company products. The money just goes in a loop—from the government to people, and right back to the big tech firms.

The writer argues this is not real freedom. People will be “free” from work, but trapped as consumers, with rich companies owning all the tools, data, and profits. The article says we should share the benefits of AI, because AI was trained on everyone’s data and knowledge. Instead of just getting a small allowance, people should have a real share in the wealth from automation.

In the comments, some readers agree and say this is a big problem that needs to be solved before too many jobs are lost. Others think UBI could be good, but only if it’s done in a fair way, with stronger rules to stop tech companies from taking all the money. Some say the problem is not new—capitalism always finds ways to keep power with the few, and we need to fight for real change.

A few people think the article is too negative. They believe AI will create new jobs, just like past technology did, and that people will find new ways to make a living. Some point out that people might spend less on tech and more on local things, so not all money will go back to big companies.

Others ask practical questions: Who will pay for UBI? Won’t prices just go up if everyone gets free money? Some suggest that instead of UBI, people should get shares in tech companies or public ownership of AI tools.

Many agree that the main issue is control—if only a few companies own all the AI, then most people will lose power and freedom. But a few are hopeful that open-source AI or new laws can help everyone benefit. There is a lot of debate, but most agree we need to think carefully about who really wins as AI changes work and society.

## Agora-1: The Multi-Agent World Model

- 原文链接: [Agora-1: The Multi-Agent World Model](https://odyssey.ml/introducing-agora-1)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48183748)

Agora-1 is a new system that lets many people or AI agents play together in the same simulated world, like in a video game. The article explains how this technology works and why it matters for games, robotics, and AI research.

Before Agora-1, world models could only handle one active player at a time. Agora-1 changes this by supporting up to four players interacting in real time, all seeing and affecting the same world. The team used the classic game GoldenEye to test this idea. Agora-1 acts as a learned game engine: it creates the game world, simulates what happens when players act, and shows the right view to each player, all at once.

Older multi-agent world models, like Multiverse and Solaris, have problems: they either treat all players as one "split screen" or can’t handle many players well. They also struggle to keep the world consistent when players can’t see each other. Agora-1 solves this by separating the parts that simulate the world from those that make graphics. This lets it keep the world state the same for everyone, but show each agent their own point of view.

Agora-1 learns two things: how the world state changes when players act, and how to turn that state into images. The models learn from data, not from rules written by people. This setup is like a modern game engine, but both the physics and graphics are controlled by AI. This model can also make new game levels while keeping the gameplay working.

Agora-1’s world state tracks things like health and position for each agent. Right now, the model is simple, but it can grow to handle more complex worlds. The team wants to see if these methods will work for other AI world models, not just games.

Agora-1 is also good for research in reinforcement learning (RL). Previous world models only supported one agent, but with Agora-1, many agents can interact, compete, or cooperate. This creates more interesting situations and data for training smarter agents. Agents and world models can learn from each other and improve over time.

The model can train agents in worlds it generates itself, so these agents might be able to work in new, real situations later, even if they never saw the original game. Beyond games, the tech could help in robotics, where many robots work together, or other areas needing multi-agent systems.

Agora-1 is just the start, but it suggests a future where shared simulated worlds help train better AI and support new kinds of teamwork between humans and machines.

Hacker News commenters liked the technical depth and clear explanation in the article. Some were impressed by the idea of AI learning both the world rules and graphics from data, not from code. A few pointed out the long history of using games for AI research, saying this is a big next step for multiplayer AI.

Others were curious about the limits—like how well Agora-1 would scale if there were more than four agents, or how it would handle much more complex environments. Some worried about how this could be used outside of games, especially in sensitive fields like defense or robotics, where mistakes could have real risks.

There were also questions about the amount of data and computing power needed to train these models, and if this approach could ever match traditional, hand-coded physics or graphics engines. A few people wondered if separating simulation and rendering might make debugging harder or lead to strange bugs if the two models get out of sync.

Some users were excited about the possibility of open-ended AI agents learning by playing with each other, but others cautioned that real-world collaboration is much harder than in games. Finally, many were eager to try Agora-1 themselves or see what new research comes from it.

## Understanding Singleflight in Go

- 原文链接: [Understanding Singleflight in Go](https://www.codingexplorations.com/blog/understanding-singleflight-in-golang-a-solution-for-eliminating-redundant-work)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48164147)

The article explains how Go’s `singleflight` package helps stop repeating the same work when many requests for the same thing happen at once. It shows how `singleflight` makes sure only one real call happens, and then shares the result with everyone waiting.

When a resource is requested, `singleflight` starts the job for the first caller. If more callers ask for the same thing while it’s running, their requests wait. When the result is ready, all the waiting callers get the same answer. This pattern is useful when caching is not enough or when data changes a lot. The `Group` type in `singleflight` manages these jobs. The article gives a simple example: several goroutines ask for the same “expensive operation,” but only one runs, and all get the result. 

Benefits include less wasted work, easier code, and lower server load. The article also warns to handle errors well, use good keys for jobs, and track what’s happening for debugging. An advanced example adds a cache on top, showing a weather service where only one API call is made for the same city, even if many users ask at once. If the data is cached, it’s returned straight away; otherwise, `singleflight` ensures only one API call.

In the comments, many developers praise `singleflight` for making concurrency easier and saving resources. Some mention it is especially good for microservices and APIs. A few warn not to use it as a replacement for proper caching or queueing, as it does not store results for long. Others say you must be careful with keys—bad keys can lead to wrong sharing or missed savings. Some suggest adding timeouts or limits to avoid long waits if something goes wrong. One commenter points out that using `singleflight` with caching works very well, but you still need to handle errors and cache invalidation. Another shares a story of using `singleflight` to cut load on a database by a lot. There are also questions about how it handles failed calls or what happens if the function is slow. Overall, people agree it is a simple but powerful tool when used right.
