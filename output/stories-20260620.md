UHF X11 is a new project that brings the old X11 window system to Apple Vision Pro and visionOS. It lets you run classic X11 desktop apps in floating windows that you can move around in space using the headset.

The main idea is to make Apple Vision Pro act as a full X11 display server. You can connect old or new X11 clients—even from other machines—over the network. Each app you launch, like xterm or xclock, opens as its own window in your real space. Windows can be placed anywhere, and the display uses sharp pixel rendering with effects to look like old CRT monitors, including scanlines and glow. It supports OpenGL (GLX) for 3D apps, but the support is still experimental—like it was back in the day. Security is handled using X11 cookies, and you can add custom bitmap fonts. The goal is to bring a retro desktop feel into a modern AR device.

## PostgresBench: A Reproducible Benchmark for Postgres Services

- 原文链接: [PostgresBench: A Reproducible Benchmark for Postgres Services](https://clickhouse.com/blog/postgresbench)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48611942)

This article is about PostgresBench, a new tool for benchmarking managed Postgres database services in a fair and repeatable way. The team behind ClickHouse made this tool to help people compare Postgres services using open, clear methods.

The main idea is that performance should be a core part of a database, not an afterthought. PostgresBench aims to test different managed Postgres offerings, like AWS RDS, Aurora, Crunchy Bridge, Neon, and ClickHouse’s own Postgres, under the same conditions. The tool uses `pgbench`, which is the standard Postgres benchmarking app, to simulate real workloads—lots of small, fast transactions, similar to what you’d see in payments or inventory updates. Tests are run with 256 clients and 16 threads for 10 minutes, on datasets of 100GB and 500GB. This shows how each service works under normal and heavy use. Every service is tested with the same hardware specs, mostly 4 or 16 vCPUs and 16 or 64GB of RAM, and they try to keep things fair and open by not changing default Postgres settings.

Results are shared for transaction speed (TPS), average and high-end latency. The Postgres managed by ClickHouse showed the best performance, especially at high loads, mostly thanks to fast NVMe storage that is very close to the compute. Systems using network storage (like AWS EBS) were slower, especially on write-heavy tasks, because every write must go over the network. The test also makes all data and scripts public, so anyone can check, repeat, or improve the benchmarks. You can even run the benchmark yourself on your own setup and submit your results.

In the comments, people had lots of thoughts. One person wanted to see how plain Postgres (not managed) on a simple VPS or bare metal server would compare, even if it’s not a direct match. Another person wondered about testing more Postgres providers, like PlanetScale. Someone else liked the project and said there aren’t enough public benchmarks between cloud Postgres services, so this is useful. A commenter noted that client-side code can also affect database performance, not just the backend database itself. Another wished that Postgres had protocol compression, saying the lack of it can hurt performance. Someone asked if they could use this tool to test their own Postgres setup on Kubernetes, and another person replied that yes, it’s easy and fast to run these tests yourself—about 15 minutes for results. There was also a fun comment about using AI to find the best Postgres settings if you had enough money to spend on cloud testing. Overall, people liked the idea of open benchmarks and wanted to see even more options tested in the future.

## Inference cost at scale with napkin math

- 原文链接: [Inference cost at scale with napkin math](https://injuly.in/blog/napkin-inference-cost/index.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48560227)

This article explains how to estimate the cost of running AI language models, especially when you want to serve many users at once. It uses simple math to figure out how much it costs to process each word (token) for each user, looking at both hardware limits and software tricks.

First, the article looks at GPU specs, like how fast they process numbers (FLOPs) and how quickly they move data (memory bandwidth). It explains that large AI models need to do many big matrix multiplications, which is where most of the cost comes from. The author breaks down how language models work: they take a string of words, turn them into vectors, and guess the next word using layers of attention. Attention, a main part of these models, is explained with code and by showing how much memory and compute it uses.

A major problem is that, without optimization, the model keeps redoing work for every word in a conversation. To fix this, most systems use something called a KV-Cache, which saves some data so the model doesn’t have to redo everything. With this trick, generating a new word for a user becomes much cheaper.

The article then uses the NVIDIA B200 GPU as an example. It shows that, in theory, you could serve over 300 users at once if the chip was used perfectly. But in reality, memory is a big limit. For a large model, you can only store chat data for about 6 users if each chat is very long. Most chats are shorter, so you can use memory better and serve around 40-60 users, maybe up to 300-800 if users are idle a lot (like when they are reading).

The article also shows how fast tokens can be generated, usually much faster than people can read. When looking at money, if you buy a GPU for $40,000 and use it for 300 users, it’s about $133 per user for the life of the chip. Renting costs about $9.36 per user per month. The article says these are rough estimates, but useful for planning.

In the comment section, one reader focuses on the cost part. They wonder if people really buy expensive GPUs and leave them at data centers, or if the “upkeep bill” means the cost of running them at home, like electricity. Some people explain that yes, companies often buy GPUs and put them in data centers because they need good power and cooling, and because it’s safer and more reliable. Others point out that renting is easier for most, but buying can be cheaper in the long run if you use the hardware a lot.

Another commenter notes that the real cost isn’t just the GPU—it includes electricity, cooling, and sometimes the people who manage the hardware. Someone else mentions that the numbers from the article are good for rough planning, but actual costs can change a lot depending on usage and how well you manage idle chips.

Some people like that the article gives back-of-the-envelope math, making it easier for anyone to estimate costs without diving into too much detail. Others wish there were more details on software optimizations or examples for smaller models. Overall, readers agree the post is helpful, but real-world costs always have surprises.

## Bun has an open PR adding shared-memory threads to JavaScriptCore

- 原文链接: [Bun has an open PR adding shared-memory threads to JavaScriptCore](https://github.com/oven-sh/WebKit/pull/249)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48610841)

Bun has opened a pull request (PR) to add shared-memory threads to JavaScriptCore, the JavaScript engine behind WebKit. This change would allow JavaScript code to use threads that share the same memory space, instead of the usual message-passing model.

The article explains that, with this new feature, you can start a new thread using `new Thread(fn)`. Both threads can see and change the same objects, making it easier to share data. This is different from web workers, where each worker has its own memory and you have to copy or send messages to share data. The PR is very large—over 270,000 lines of code—and much of it was written or reviewed by AI tools, using designs based on earlier ideas from other developers. The main technical challenge is making garbage collection (GC) work safely and efficiently across threads. The current version uses a "stop-the-world" GC, which can pause all threads at once and may hurt performance. The PR is not on the main WebKit project, but on Bun’s own fork, which already has many unique changes.

Some commenters are excited about the technical leap, saying it could make JavaScript much faster for some tasks, like databases or heavy computation. Others point out that shared memory concurrency is risky and hard to get right. They warn that bugs can appear when two threads change the same object, and many developers might make mistakes if this is too easy to use. Some mention that JavaScript already has workers, SharedArrayBuffer, and Atomics for low-level concurrency, so this is not the first time shared memory has been used in JS. Still, full shared-heap threads are a bigger step. A few people share stories about trying similar things in the past, with mixed results, especially because of GC problems.

Many top comments focus on trust and code quality. Some people say they do not trust such a giant PR, especially when most of the work is done by AI and not well explained by humans. They feel changes this big need clear, careful design and documentation. Others complain about the style of the PR description, saying it reads like AI-generated text and is hard to follow. There are concerns about long-term maintenance: if only a few people (or just AI) understand the code, it will be hard to fix bugs or add features in the future. Some people defend the use of AI tools, saying that AI-generated code can be better than old, messy human code in some projects, but others warn that AI code can have hidden problems. A few wish the runtime was written in a safer language like Rust, and some are worried that the move to multithreading will make JavaScript much harder for regular developers. Others say that the message-passing model (using workers) is safer and was a good design choice in the first place. Finally, some are simply amazed that such a huge, experimental change is even possible now, thanks to new tools like LLMs, even if they are not ready to use it in production yet.
