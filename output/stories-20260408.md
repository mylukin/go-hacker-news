## I ported Mac OS X to the Nintendo Wii

- 原文链接: [I ported Mac OS X to the Nintendo Wii](https://bryankeller.github.io/2026/04/08/porting-mac-os-x-nintendo-wii.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47691730)

This article is about a developer who made Mac OS X 10.0 (Cheetah) run on the Nintendo Wii. The writer explains the whole process, from checking if it’s possible, to writing code, fixing problems, and finally getting the Mac OS X desktop to work on the Wii.

First, the writer checked if the Wii’s hardware was similar enough to old Macs. Both use PowerPC CPUs, which was a good start. The Wii has less RAM than a real Mac from that time, but it’s enough for Mac OS X Cheetah. The Wii’s hardware (like USB ports, SD card, and video output) needed new drivers, but nothing seemed impossible.

Next, the writer looked at how Mac OS X boots on a real Mac. Macs use something called Open Firmware and a bootloader named BootX, but the Wii doesn’t have these. Instead of copying them, the writer decided to make a new, simple bootloader just for the Wii. This bootloader starts the hardware, loads the Mac OS X kernel from the SD card, and gives the kernel the needed information.

To know if the kernel was running, the writer used a trick: patching the kernel to turn on a light on the Wii. That way, he could see how far the OS got before crashing. When the kernel asked for information about the hardware (the device tree), he made a custom, hard-coded version for the Wii.

The next steps were about patching the Mac OS X kernel to support the Wii’s unique memory layout and writing drivers for the SD card, video output (framebuffer), and USB support. Each driver had its own challenges, like fixing color problems (because the Wii uses a different color model for video) and dealing with USB code that expected Mac hardware, not Wii hardware.

For USB, the writer had to find old source code, make his own fake PCI (even though the Wii doesn’t have PCI), and change how the code handles data byte order. After lots of searching and some help from people on IRC, he finally got the USB keyboard and mouse to work.

The writer also improved the bootloader so users can choose which partition to boot and made it possible to load drivers from disk instead of needing to change the kernel every time. This made the system more flexible. In the end, the Wii could boot Mac OS X, show the desktop, and let users interact using the keyboard and mouse.

## USB for Software Developers: An introduction to writing userspace USB drivers

- 原文链接: [USB for Software Developers: An introduction to writing userspace USB drivers](https://werwolv.net/posts/usb_for_sw_devs/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47695012)

This article explains how software developers can write USB drivers without needing to mess with kernel code. The author shows that you can make userspace drivers using tools like libusb, and it’s not as hard as it may sound.

First, the article uses an Android phone in bootloader mode as an example device. This is a good choice because it’s common and its protocol is simple and well-documented. When you plug in a USB device, the computer asks the device for information, like its Vendor ID (VID) and Product ID (PID), to decide which driver to use. On Linux, you can use the `lsusb` tool to see these details. If the device is “vendor specific,” like the Android phone in bootloader mode, the OS usually doesn’t load a driver, which lets you experiment easily.

The article explains that you can use libusb to write a userspace application that does what kernel drivers do. With libusb, you can detect when your device is plugged in, and you don’t need to write any kernel code. The author shows code to detect the device and send it a standard “GET_STATUS” request, getting a response from the device. The article also explains USB descriptors—these are bits of data the device sends to describe itself, like what features it supports and what drivers it wants. You can get all this info with libusb or the `lsusb` tool.

The article covers USB endpoints, which are like ports. There are different types: control (for setup and basic info), bulk (for lots of data), interrupt (for fast, small messages like from a keyboard), and isochronous (for things like audio or video). Endpoints can be IN (data to the computer) or OUT (data to the device). The author explains that the host (computer) always starts the conversation; the device never sends data unless asked.

Finally, the article shows how to talk to the phone’s Fastboot protocol. This is a simple protocol where you send a command string, and the device replies with a status and data. The example code sends a “getvar:version” message and gets back “OKAY0.4,” which matches the docs. The main point is that you can write a USB driver in userspace with only basic tools and some patience.

In the comments, many readers say this is a great, clear guide for beginners. People like that it breaks down USB concepts simply, and they appreciate seeing real code examples. Some developers share their own struggles with USB, saying that learning the basics helped them solve tricky problems. Others mention that USB protocols can get much harder with more complex devices, like cameras or printers, but the core ideas stay the same.

A few people point out that libusb works well on Linux but can be harder to use on Windows. They suggest using tools like Zadig to fix driver issues on Windows computers. There’s also talk about how the USB specification is very long and can be confusing, but guides like this help a lot. Some readers wish there were more examples for other types of devices, while others recommend learning by experimenting with cheap USB gadgets.

One commenter says writing userspace drivers is much safer than kernel drivers because bugs won’t crash the whole system. Another person reminds everyone to be careful when sending commands, because it’s easy to accidentally mess up a device if you don’t know what you’re doing. Overall, most readers agree this article makes USB driver development less scary and encourages more people to try it.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47687273)

This article shares five Git commands the author uses before reading any code in a new project. The goal is to understand where problems might be hiding and how the team works, just by looking at the code history.

First, the author runs a command to find the 20 files that changed the most in the last year. Files with lots of changes can mean trouble, especially if no one wants to work on them. The author explains that a file with high churn and many bugs is a big risk. Next, the author checks who made the most commits. If one person did most of the work and is now gone, it’s a problem. The author also looks at how many people are still active, since often the people who wrote the code aren’t the ones maintaining it. The next command searches for files with many bug-related commits. This helps find “bug hotspots”—files that break often but never really get fixed. The author warns that this only works well if commit messages are clear.

Another command shows how many commits happen each month. If activity drops suddenly, it could mean a key team member left. A slow, steady drop means the project might be dying. Spikes in commits can show the team batches work for releases, instead of shipping often. Finally, the author checks for emergency fixes and reverts. Too many of these means the team is often in crisis mode, or doesn’t trust their deployment process.

In the comments, many people liked these tips and said they use similar commands. Some users noted that these commands don’t work as well if the team squashes commits or writes poor commit messages. Others said that high-churn files aren’t always bad—they can just mean the project is active. Some people shared extra commands, like checking for large files or finding old, forgotten code. A few pointed out that not every project’s history tells the full story, especially after big rewrites or migrations. Some readers liked the focus on team activity, not just code. Others warned that numbers can be misleading without context. One person said these commands are a good “smoke test” before going deeper. Another mentioned that these checks helped them spot risky parts of a project before trouble started. Overall, readers thought the approach was practical, but said it should be just the first step in understanding a new codebase.

## Understanding the Kalman filter with a simple radar example

- 原文链接: [Understanding the Kalman filter with a simple radar example](https://kalmanfilter.net)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47693153)

The article explains the Kalman Filter, a tool used to predict and estimate things—like tracking an airplane with radar—even when there is noise or uncertainty in the data. It uses a simple radar example to show how the filter works step by step.

The Kalman Filter helps you guess the real position and speed of an airplane, even when your radar measurements are noisy. First, you start with an initial measurement (like the plane is at 10,000 meters, going 200 meters per second), and you also know how uncertain this measurement is. Using a model (for example, the plane keeps moving at the same speed), you predict where the plane will be after some time, but your prediction will also have some uncertainty. Sometimes, things like wind (called "process noise") make your prediction less certain.

When the radar measures the airplane again, the new reading might also be noisy, and maybe even less reliable than the first one. Instead of just trusting either the prediction or the new measurement, the Kalman Filter mixes both using a special weight (the Kalman Gain). This weight depends on how uncertain each estimate is: if the new measurement is less certain, you trust it less, and vice versa. The filter updates its best guess about the plane's position and speed, and also updates how sure it is about that guess. This process repeats: predict, then update with new data, over and over.

The article also covers the math behind the filter, such as using vectors and matrices to handle position and speed together, and how to calculate and update uncertainties. It shows that, after updating, your guess is usually better than either your prediction or the new measurement alone. The article uses clear examples, explains all the math, and gives links to more detailed resources and a book for those who want to learn more.

From the comment section, many people liked how the article makes things easier to understand, especially compared to other resources that use too much math. Some readers said this kind of example helps them finally "get" what the Kalman Filter does. Others pointed out that the filter is very useful in real life, like for robots, GPS, or finance, but warned it can be tricky to tune the settings, like the noise values, in practice.

A few commenters shared stories about using Kalman Filters in their own projects—one used it for a drone, another for tracking moving objects in games. Some people argued that, while the examples are good for learning, real-world systems often have more complicated problems, like measurements that don’t match the model, or sudden changes in movement. One person mentioned that the Extended Kalman Filter or Unscented Kalman Filter can help with these harder cases.

Some readers wished the article had more about what to do when measurements are really bad (outliers) or missing, but others pointed out that the linked book covers this in more detail. A few comments debated if Kalman Filters are always the best choice, since newer methods like particle filters might work better for some problems, but most agreed Kalman Filters are still very popular because they are simple and fast. Overall, people liked the hands-on style, and many said they planned to share the article with coworkers or students.

Swift now works in more code editors, including Cursor, VSCodium, AWS Kiro, and Google Antigravity. This happens because Swift’s official extension is now in the Open VSX Registry, and these editors can use VS Code extensions.

The Swift team says you can use Swift in many IDEs, like VS Code, Xcode, Neovim, and Emacs. Swift uses the Language Server Protocol (LSP), which helps editors understand the Swift code you write. The new extension gives you features like code completion, refactoring, debugging, and a test explorer. It also supports DocC, which helps you write and read documentation inside your editor. Now, if you use an editor that works with Open VSX, you can easily find and install the Swift extension. Agentic IDEs like Cursor and Antigravity will even install Swift automatically for you.

To get started, open your editor’s Extensions panel, search for “Swift,” and install it. For Cursor, there is a special guide to help you set things up, including AI workflow skills. The team encourages developers to try the extension in their favorite editor and send feedback.

## Pgit: I Imported the Linux Kernel into PostgreSQL

- 原文链接: [Pgit: I Imported the Linux Kernel into PostgreSQL](https://oseifert.ch/blog/linux-kernel-pgit)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47648538)

This article is about importing the full history of the Linux kernel into a PostgreSQL database using a tool called pgit. The author wanted to see if a SQL database could handle the huge amount of data and let people query code history with SQL.

The Linux kernel is one of the largest projects in the world, with over 1.4 million commits, 24 million file versions, and 38,000 contributors. The import took about 2 hours on a powerful server with a lot of RAM and SSD storage. The server was tuned for speed, with changes to the CPU, memory, and disk settings. The author used pgit, which stores all git data in PostgreSQL tables. It uses delta compression, which means only changes are stored, not full copies. The final database was about 2.7 GB of data, compared to about 1.95 GB for git's most aggressive compression. The author notes that git is still a bit smaller, but pgit lets you run SQL queries on code history, which opens new ways to analyze the project.

With the data in PostgreSQL, the author could easily answer questions like: how many people contributed only once (about 14,000); which files often change together; who merges the most patches; and which companies sponsor the most work. For example, Intel, Red Hat, and kernel.org are top contributors. Most commits touch just one file. The author also found fun facts, like which commit is blamed for the most bugs (the initial import), and how often developers use swear words in commit messages (very rarely).

The article includes technical details about server setup, PostgreSQL settings, and how pgit compresses and organizes data. The author shares example SQL queries for common questions, and shows that even big queries run fast. The project also inspired others to build new tools on top of the same compression library.

In the Hacker News comments, many people were impressed by the scale of the import. Some liked the idea of using SQL to explore git history, saying it’s useful for research or company audits. ## Veracrypt project update

- 原文链接: [Veracrypt project update](https://sourceforge.net/p/veracrypt/discussion/general/thread/9620d7a4b3/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47686549)

The VeraCrypt project cannot release Windows updates because Microsoft closed the developer’s account used to sign drivers and the bootloader. The project lead, Mounir Idrassi, did not get any warning or explanation, and all attempts to contact Microsoft have failed.

Without a valid signature, most Windows users cannot install new VeraCrypt versions, especially with Secure Boot enabled. Updates for Linux and macOS are still possible, but most VeraCrypt users are on Windows, so this is a big problem. The last signed Windows version uses an old certificate that will soon expire, making things worse. People are worried that unsigned or self-built versions will be hard to use or install, especially for users who are not technical. Some suggest that even mounting encrypted file containers may require turning off Secure Boot, which is not practical for many.

Several comments offer help and advice. Some users think the account was closed by mistake and suggest trying all official recovery links, posting the story on social media, or contacting Microsoft support agents directly. Others say to reach out to well-known people in the security and open-source world, or even try to get help from the media or friendly lawmakers, since VeraCrypt is important for privacy.

One commenter had a similar issue and fixed it by proving their domain ownership to Microsoft, suggesting the problem could be with automatic validation, not a real ban. Another person offers to share Mounir’s contact info inside Microsoft to help. Some worry the account was closed because the software can be used for privacy, which is sometimes viewed with suspicion. People also discuss possible workarounds, like making a version of VeraCrypt that does not need signatures, or limiting features for now.

The community is supportive, sharing tips and contacts, and hoping the problem is a technical or paperwork issue, not a policy change against encryption tools. Some remain worried about the growing control large companies have over important security projects. Others are hopeful that with enough noise and the right contacts, the problem can be solved and VeraCrypt can continue to protect user data.
