For iPhone users, though, there’s no need to install extra Google apps to pass the same check, so the rule only hurts people who choose not to use Google on Android.

This is a problem for privacy-focused users who avoid Google software because of data tracking. Now, these users are treated as “suspicious” just because they want more privacy. The article warns web developers that by adding this new reCAPTCHA, they are locking out people who care the most about privacy, even if it’s a small group.

Others argue that most users won’t care since they already use Google apps anyway, and that Google has the right to protect its tools from bots however it wants.

Some think this is a sign that the web is becoming less open and more controlled by big tech companies. A few suggest websites should stop using Google reCAPTCHA and pick other options that respect privacy better. Others point out that captchas in general are getting harder for real people and easier for bots, so maybe a new solution is needed. There are also people who say the change makes sense for security, but wish Google would be more open about it and allow for exceptions. Overall, most commenters see this as bad news for privacy and for users who want more control over their phones.

## AI is breaking two vulnerability cultures

- 原文链接: [AI is breaking two vulnerability cultures](https://www.jefftk.com/p/ai-is-breaking-two-vulnerability-cultures)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48066524)

This article talks about how AI is changing the way people handle security bugs in software. The main story is about a recent Linux vulnerability called "Copy Fail," and how the usual process for fixing it was quickly made public.

The article explains there are two main cultures for handling security bugs. The first is "coordinated disclosure." Here, if someone finds a bug, they tell the software team in private and wait for a fix before telling the world. This usually takes up to 90 days. The second is the "bugs are bugs" way, popular with Linux. In this culture, people just fix the bug as fast as possible in public, without drawing much attention, hoping attackers do not notice before users update.

With AI, both cultures are having problems. Now, AIs can look at software changes and quickly spot security fixes. This means hiding fixes in normal updates is much harder. Also, long waiting periods (embargoes) are not safe anymore, because AIs and many hackers can find the same bugs fast. In the "Copy Fail" case, two people reported the same problem within nine hours. The article says maybe very short embargoes (few hours or days) are the best way forward, since both hackers and defenders use AI now. The author ran a test using Gemini, ChatGPT, and Claude, showing that AI can spot security patches quickly—even without full context.

In the comments, some people say public disclosure is better, because it pushes companies to fix things fast and helps users stay safe. Others worry that if bugs are shown too soon, attackers will use them before people can patch their systems. A few think that embargoes are now almost useless, since AI makes both finding and attacking bugs much faster. Some suggest using AI for defense, like real-time patching or faster updates. Other commenters say Linux’s open style worked before, but now anyone (including bad actors) can use AI to spot important fixes. There’s debate on whether closed or open handling is safer now, and if we need new rules for AI times. Some feel the basic idea of “security through obscurity” is dead, and software needs to be safer by design. A few point out that coordinated disclosure still has value, but only if the response is very fast. Overall, readers agree the world of software security is changing fast, but there is no easy answer yet.

## You gave me a u32. I gave you root. (io_uring ZCRX freelist LPE)

- 原文链接: [You gave me a u32. I gave you root. (io_uring ZCRX freelist LPE)](https://ze3tar.github.io/post-zcrx.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48067734)

This article explains a new security bug in Linux kernels 6.15 to 6.19, found in the zero-copy receive (ZCRX) feature of io_uring. The bug lets a user with special network rights (CAP_NET_ADMIN) get full root access by carefully using network hardware and kernel memory.

The problem happens because the kernel does not check the upper limit when it adds items to a memory list called a “freelist.” Two different parts of the kernel can return network buffers at the same time, which can push the counter too high and cause the system to write 4 bytes outside the end of an allocated memory area (an out-of-bounds write). This can change nearby kernel memory, which is normally not allowed.

The value written is a small number (0 to N-1), but by choosing how much memory to use, an attacker can control where this write happens and what value is used. The article shows how to line up this freelist next to a message queue object (msg_msg) in memory. By overwriting the start of this object, the attacker can change pointers and trick the kernel into reading or writing more data than normal.

With this trick, the attacker can read secret kernel addresses (bypassing KASLR security) and then overwrite the modprobe_path, a setting that controls which program the kernel runs as root when it sees an unknown network type. The attacker writes a script path to this setting, then asks the kernel to run it, giving root access.

The author explains how to set up the bug using special network cards, how to time the attack, and how each step works. There are code samples and a clear breakdown of each stage, from the memory bug to full root access. The bug is fixed in commit 770594e, but at the time of writing, the fix was not in main Linux branches.

In the comments, many readers are impressed with the technical depth and creativity of the exploit. Some say it is a reminder of how small memory bugs can give big security problems, especially in complex kernel features. Others discuss how the bug is not in most Linux systems yet, since ZCRX is new and only used with special network cards. A few readers note that needing CAP_NET_ADMIN limits how many users can use the bug, but others warn that in cloud or container setups, this right is often given to users or processes that should not have it.

Some commenters mention that the fix is simple—a single bounds check—and wonder why it was missed. Others point out that complex kernel systems like io_uring are hard to review and test, especially with new hardware. A few discuss the history of similar bugs in other freelist memory systems and call for more careful code reviews.

There is also debate about whether the steps needed to exploit the bug are “practical” or only possible for skilled attackers. Some think most real-world systems are safe unless they use specific hardware and kernel options, while others are more cautious, saying that attackers are creative and could adapt this method to future kernel versions or other parts of the system.

Overall, the comments agree this is a serious, well-explained bug, showing the risks of unchecked memory operations in the kernel, and the need for fast security updates when bugs like this are found.

## Non-determinism is an issue with patching CVEs

- 原文链接: [Non-determinism is an issue with patching CVEs](https://flox.dev/blog/achieving-rapid-cve-remediation-in-an-era-of-escalating-vulnerabilities/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48068947)

This article talks about how fixing security problems in software packages (called CVEs) is getting harder because AI can now find many more vulnerabilities. In the past, checking for these problems was slow and manual, but now the number of issues is exploding.

The main problem is that most companies do not have a clear record of every software package and version they use. Package managers like apt, npm, or pip can install different versions depending on the system, the time, and other factors. This makes it hard to know if you are using a risky package. The article says that using “lockfiles” helps, but only a little, because they do not cover the whole environment.

The author suggests using tools like Nix and Flox. Nix is a package manager that makes every build the same every time (this is called “determinism”). Flox uses Nix but makes it easier for teams to use. With these tools, you can group environments that use the same set of packages. Instead of checking every server or app (which could be hundreds), you check only the unique sets of packages, which might be much fewer. This saves a lot of work.

Nix does this by building packages in a special way. It records every input, and the results are stored in a way that is easy to track and verify. This makes it simple to find out which environments have a risky package and to fix all of them at once.

The article also talks about non-determinism. With most package managers, you can get different results every time you install. This makes it very hard to track problems. Nix avoids this by being strict: the same input always gives the same output.

Another issue is that as AI gets better, both good and bad actors can use it to find and attack new vulnerabilities faster. So, good tools and clear records are even more important.

Now, let’s look at what people said in the comments. Some users agree that non-determinism is a big problem. They say that it is very hard to know what is really running in production, especially when systems update themselves in the background. Others think that Nix and Flox are good ideas but worry that they are hard to learn and slow to use.

A few commenters mention that big companies already use custom solutions for this, but smaller teams may struggle. Some people point out that disk space and slow builds are less of a problem now, so using tools like Nix makes more sense than before. Others are skeptical and say that changing package managers is a big risk and takes a lot of effort.

One commenter warns that even with good tools, you still need to check for things added outside the system, like manual changes or runtime downloads. Another user brings up that lockfiles are helpful, but they miss system libraries and other parts outside the main package manager.

In general, the comment section shows that people like the idea of more control and tracking, but they are worried about the cost and complexity. Some think AI will make things better, while others think it will just make attacks faster. There is agreement that the old way does not scale and that new tools are needed, but not everyone is sure what the best path is.

## Serving a website on a Raspberry Pi Zero running in RAM

- 原文链接: [Serving a website on a Raspberry Pi Zero running in RAM](https://btxx.org/posts/memory/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48064312)

This article is about running a public website on a Raspberry Pi Zero, using only its RAM for everything and no disk. The writer explains how they set up their website to be fast, simple, and reliable on very cheap hardware.

They use a Raspberry Pi Zero with 512MB memory, booting Alpine Linux in diskless mode. The microSD card is only for starting up; after that, the system runs completely in RAM. This means the system is fast and, once started, you can even remove the SD card. To save changes or files, they use the `lbu` tool to copy data to the SD card when needed. For serving the website, they use `darkhttpd`, a very lightweight web server, but suggest `nginx` as a more flexible option. The site itself is just static files, kept small to fit in memory.

Because the Pi Zero is weak and can’t handle HTTPS (TLS) well, they send traffic through a cheap VPS using HAProxy. The VPS does the heavy work like handling TLS and uses `socat` to forward HTTP traffic back to the Pi at home. The writer describes how to set up port forwarding, dynamic DNS (with DuckDNS), and keeping backups by cloning the SD card. The whole setup is very low-cost—about $4 per year for the VPS. They explain each step, from preparing the SD card, installing Alpine, setting up the web server, and connecting everything together.

In the comments, many readers say this project is fun and clever. Some people like that it uses so little hardware, and that everything fits in RAM. Others point out that running diskless systems can be risky if you forget to save changes before a reboot. Some think it’s a great learning project, but not practical for bigger sites or heavy traffic. A few suggest using other hardware, like old laptops or different single-board computers, for more power or reliability. Some warn about the risks of exposing home devices to the internet, and recommend being careful with security. Others discuss the pros and cons of using a VPS or a CDN like Cloudflare for the front-end. Some like that the setup avoids big cloud providers and tracking, matching the spirit of self-hosting. A few point out that with careful setup, this method could work for other small projects, like file sharing or IoT dashboards. Overall, people are impressed by how much can be done with so little, and appreciate the clear guide.

## An Introduction to Meshtastic

- 原文链接: [An Introduction to Meshtastic](https://meshtastic.org/docs/introduction/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48061566)

Meshtastic is a project that helps people use cheap LoRa radios to talk with each other over long distances, even if there is no phone or internet service. The project is open source and run by a group of volunteers.

Meshtastic radios use a special radio technology called LoRa. This makes them work over many kilometers, even in places without cell towers. Anyone can use these radios; you do not need a license like with ham radios. The devices create a mesh network, which means each radio passes messages along to others. This way, even people far away can get messages if there are other radios between them. You do not need a phone to use Meshtastic, but you can connect one phone at a time to a radio if you want to send messages from your phone. The messages are encrypted, so only your group can read them. The radios also have good battery life and can send GPS locations if you want. The project is hosted on GitHub, and volunteers write and care for the software. There are guides and a friendly community to help new users.

People on Hacker News liked that Meshtastic is open source and does not need any central server or internet connection. Some commenters said it is useful for outdoor adventures, during disasters, or in remote places. A few people shared stories about using Meshtastic for hiking, camping, or even at big events where phones do not work well. Others talked about the technical side, like how LoRa works and what real-world range you can expect. Some users mentioned that you need to place radios carefully to get the best distance, and that trees or hills can make the range shorter. There were questions about how secure the messages are and if the encryption is strong enough. A few people wanted more details about battery life and how easy it is to set up the radios. Some worried about the cost and the need to buy special hardware, but others pointed out that Meshtastic radios are cheaper than satellite phones and do not have monthly fees. A few suggested ways to improve the project or add new features, like better group messaging or more user-friendly apps. Overall, most commenters thought Meshtastic was a clever idea with lots of potential for people who need off-grid communication.

## All means are fair except solving the problem

- 原文链接: [All means are fair except solving the problem](https://yosefk.com/blog/all-means-are-fair-except-solving-the-problem.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48037655)

This article talks about a software engineer who tries to help by printing warnings when his code is used the wrong way, but this causes trouble for other scripts that expect a certain output. Instead of fixing the misuse, people suggest workarounds to avoid changing the scripts. The main problem is that the warning messages change the program’s output, which breaks automation that looks for specific lines like “yay, done.” When warnings appear after this line, scripts think something went wrong, even if everything is fine.

The author explains that, rather than fixing the root cause (the code misuse), people prefer to suggest other solutions. For example, some suggest printing “yay, done” again at the end, even if a warning was shown. Others say to hide the warnings by default or write them to a separate file. There are debates about the best technical tricks to hide or move the warning messages, but nobody wants to fix the actual places where the code is used wrongly.

The author finds this funny but also sad. He says it shows a common problem in tech: people often prefer complicated workarounds instead of solving the main issue. He mentions Hyrum’s Law, which says that any behavior in software will eventually be depended on by someone. This means even strange or wrong things in your program might be important for someone else’s workflow, making it hard to change anything without breaking something else.

In the comments, some people agree that this happens a lot at big companies. They say that changing old workflows is risky because many teams depend on them, and nobody wants to be responsible if something breaks. Others think warning messages should be handled better so they don’t break scripts, but they still wish people would fix the real problems instead of hiding them.

A few commenters point out that sometimes, it is just too hard or takes too long to fix the real issue, especially if many teams are involved. Some share their own stories about similar situations where small changes caused big problems for unexpected reasons. Others talk about how hard it can be to keep software clean and simple when everyone depends on tiny details.

Some readers think that printing extra messages is not a good idea because it makes automation fragile. A few suggest that better communication between teams could help, so everyone understands why changes are needed. One person jokes that in software, the real rule is: “Don’t touch anything or someone will be upset.” Overall, the comments show that while everyone sees the problem, most have gotten used to working around it instead of fixing it.
