## Show HN: Kage – Shadow any website to a single binary for offline viewing

- 原文链接: [Show HN: Kage – Shadow any website to a single binary for offline viewing](https://github.com/tamnd/kage)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48529990)

This article introduces Kage, a tool that lets you save any website for offline viewing by removing all JavaScript. With Kage, you can copy a website as it looks after loading, keeping CSS, images, and fonts, but deleting scripts so nothing runs in the background.

Kage uses a real browser (Chrome or Chromium) to open the webpage, waits until everything settles, then takes a snapshot of what you see. It then deletes all scripts and saves local copies of all assets—so the files are safe to open later, even years after the website might have changed or disappeared. You don’t need to worry about trackers or broken links. The saved pages are just plain HTML and assets, ready to open in any browser.

To use Kage, you can install it using Go, download a ready-made binary, or run it in a container. The main command is `kage clone <url>`, which saves the site to your computer. You can preview the copy offline with `kage serve`, or use `kage pack` to make a single file archive (ZIM) or a self-contained executable that serves the site. Kage also supports options for how many pages to save, how deep to follow links, or to include subdomains. It politely reads rules from `robots.txt` and can resume a failed download.

The ZIM format is popular for offline content, so you can use your saved file in other apps like Kiwix. Making a self-contained binary lets you give the site to anyone, and they can run it without installing anything else. If you build Kage with a special option, the saved site can even open in its own window, not just a browser tab.

Kage works by using Chrome to render the page, stripping scripts, and saving everything else to disk with paths rewritten so all links and assets work offline. The project is open source and easy to build or test yourself.

In the comments, many people are excited about Kage, saying it solves the problem of "Save As" not working well for modern websites. Some like that it removes tracking and scripts, making offline copies safer and more private. A few users compare it to other tools like HTTrack, wget, or SingleFile, and discuss the pros and cons.

One commenter points out that saving JavaScript-heavy sites is tricky, but Kage’s use of a real browser helps capture the finished state. Others wonder if it can handle interactive or login-required pages. Some users wish Kage could keep full search features in the ZIM files, like Kiwix does, but understand that’s hard to implement.

A few worry about the size of the self-contained binary files, but others say the trade-off is worth it for easy sharing. People like that Kage is open source, and some hope it keeps being updated. There are also technical suggestions, like adding support for more browsers or improving asset handling. Overall, the reaction is positive, with many users planning to try Kage for archiving important web content.

## Zinnia: A modular 64-bit Unix-like kernel written in Rust

- 原文链接: [Zinnia: A modular 64-bit Unix-like kernel written in Rust](https://zinnia-os.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48532705)

Zinnia is a new operating system kernel, like Linux, but written almost fully in the Rust programming language. The main goal is to make a safe, modern, and modular Unix-like system that uses as little unsafe code as possible.

Zinnia supports many POSIX system calls, which means it can run a lot of software that works on Linux or BSD systems. It also includes some special features found in modern systems, like epoll and timerfd, which help with advanced input and timing tasks. Zinnia can run a desktop using Wayland or X11, so you can have a familiar graphical interface.

Most device drivers in Zinnia are not built-in; instead, they are loaded as separate modules when the system starts. These modules are written in Rust and use the ELF file format, just like programs on Linux. Zinnia uses the Limine bootloader, so it can boot on any computer that supports UEFI, which is most new computers.

The project started in 2024 as a way for the creator to learn more about how operating systems work. Over time, it gained more features, and now Zinnia can run on many real computers that use x86_64 processors. Support for ARM and RISC-V computers is planned for the future, but not yet available. The creator welcomes help and fixes from other developers.

In the comments, some users are excited to see a kernel written in Rust, as Rust is known for safety and preventing bugs common in C code. Others point out that making a new operating system is very hard, and it will take a long time to catch up to Linux or BSD in features and stability. A few people ask about performance, since Rust’s safety checks can sometimes make things slower, but others note that this is a good trade-off for safer code.

Some users like the choice to load drivers as modules, because it makes the system flexible and easier to update. Others wonder how much software will run on Zinnia, since many programs expect Linux-specific features. A few think it is great for learning and could be useful for research or small projects, even if it is not ready for daily use yet.

There are also comments about the need for good documentation and clear ways to contribute. Some people offer to help test on different hardware. Overall, the community seems supportive, interested in seeing how Zinnia will grow, and happy that new ideas are being tried in the world of operating systems.

## Yserver: A modern X11 server written in Rust

- 原文链接: [Yserver: A modern X11 server written in Rust](https://github.com/joske/yserver)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48531394)

Yserver is a new X11 server for Linux, written from scratch in Rust. The main goal is to make a modern, practical X11 server that works with current desktops and apps, but without supporting old features that most people no longer use.

The creator does not want to copy Xorg exactly, but instead create something cleaner and simpler. Yserver drops old features like support for multiple screens, non-TrueColor visuals, indirect GLX, the DDX driver ABI, and endian-swapped clients. It aims to still run full desktop environments like MATE, XFCE, and Cinnamon, as well as window managers like FVWM3 and Window Maker. The server supports many X11 extensions, including GLX, Composite, RandR, XInput, and more. GLX_EXT_texture_from_pixmap is implemented and works on AMD, Intel, Asahi, and Qualcomm GPUs, but not on Nvidia's proprietary drivers.

Yserver has been tested on various hardware: AMD, Intel, Nvidia, Qualcomm, Apple Silicon (with Asahi Linux), and virtual machines. The project uses modern Linux graphics (DRM/KMS) and works with seat management via libseat. To build and run yserver, you need Rust and some system libraries, and there are install instructions for Arch, Ubuntu, and Alpine Linux.

Yserver can run as the display server for LightDM, making it easy to use with graphical logins. You can also start it directly from the console with a command, and there are some handy keyboard shortcuts for taking screenshots or dumping screen images. The project checks its protocol support by running the X.Org test suite (xts5) and rendercheck. Yserver is open source under the MIT license.

In the Hacker News comments, many users are excited to see a fresh take on X11, especially one written in Rust. Some think this could help make X11 safer, since Rust helps avoid memory bugs. Others are happy to see a modern codebase without old, hard-to-maintain features. However, some users point out that X11’s future is limited because Wayland is becoming more popular. A few ask if it makes sense to put effort into X11 instead of Wayland.

Some commenters worry about driver support, especially with Nvidia cards, since the project says Nvidia’s proprietary driver does not work. Others mention that dropping legacy features might break compatibility with some old apps, but many agree that dropping unused code can make maintenance easier. There are also helpful suggestions about testing and contributing, and some users share their own experiences writing X11 code or trying out new display servers. Overall, the community shows a mix of excitement, curiosity, and cautious optimism for yserver’s future.

## Chaosnet (1981)

- 原文链接: [Chaosnet (1981)](https://tumbleweed.nu/r/lm-3/uv/amber.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48531449)

Chaosnet was a local area network built at MIT’s AI Lab to connect computers close to each other, like in the same building. The network had no central controller—each computer could talk to others directly over a shared cable.

The system was designed for Lisp machines, with every user having their "own" computer, but sharing files and resources like printers over the network. Chaosnet was fast and simple: it used coaxial TV cable as its main wire, and each computer had special hardware to connect to this cable. Computers sent "packets" (small chunks of data) to each other, and each packet had a sender, receiver, and a check value to catch errors. If two computers tried to send at the same time, their messages could crash into each other—a "collision." The system used clever tricks to avoid or quickly recover from collisions, like assigning time slots for when each could send.

The software part of Chaosnet handled how programs on different computers talked, making sure messages arrived correctly and in order. It had simple ways to start and stop conversations (called "connections"), keep track of who’s talking to whom, and make sure packets didn’t get lost or doubled. The network could connect many types of computers, using "bridges" to link separate cables and "routing tables" to find the best way to deliver messages. The design avoided things it didn’t need, like long-distance links or complex security features. Instead, it focused on speed, reliability, and easy repairs. Chaosnet even let programs ask for services (like files or time) from any computer, using a system of requests and answers similar to today’s client-server setups.

In the Hacker News comments, many people praise Chaosnet’s simplicity and its influence on later networks, such as Ethernet and TCP/IP. Some remember using Chaosnet in the 1980s and note how quick and easy it was to get started. Others compare it to modern protocols, pointing out that Chaosnet avoided much complexity by only worrying about local connections. Some say the lack of security wouldn’t work today, but at the time, it helped keep things fast and easy to fix. A few commenters share stories about debugging with the network’s blinking lights and test buttons, or about how Chaosnet could recover from problems without crashing everything. There’s also debate about whether its collision-avoidance ideas could work in bigger, more crowded networks, and some point out that Chaosnet’s design is a reminder that sometimes simple tools are best for small, focused problems. Overall, people seem to agree that Chaosnet was an important step in networking, and its lessons can still help us build better systems today.

Each meeting session is saved as audio and markdown files in Finder, so you fully control your data. You can move, back up, or delete these files at any time. The app can also connect to your calendar to automatically name recordings and remind you before meetings. Trace identifies different speakers and marks who said what. It asks for audio recording permissions, but it never accesses your screen or sends data anywhere.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48489636)

This article talks about segmented type, which is a style of writing numbers and letters using straight lines, like you see on digital clocks or train station signs. The writer shares a list of numbers and mentions spotting this typeface on trains in Spain.

Segmented type is made by lighting up certain lines to form each character. It is popular in places like digital watches, calculators, and display boards because it is easy for machines to show. Each number or letter is broken down into simple segments—usually straight lines—that turn on or off. For example, the number “8” uses all the segments, while “1” uses just two. This method is good for places where you need to show information quickly and clearly, even from far away or in bad lighting. Segmented type is also strong and can last a long time outdoors or in busy places like train stations. It is not the most beautiful way to write, but it is very practical and easy to read. The article also seems to encourage readers to try typing with segments themselves.

In the comments, some people say they feel nostalgic when they see segmented type, because it reminds them of old electronics from their childhood. Others point out that while segmented type is simple, it can be hard to show all the letters of the alphabet, so it works better for numbers. A few readers enjoy the design and think it gives a fun, retro look to signs and devices. Some users note that today’s screens can show much more detailed fonts, so segmented type is used less often now. ## Caddy compatibility for zeroserve: 3x throughput and 70% lower latency

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48527145)

This article is about zeroserve, a fast HTTPS server that can now use Caddy configuration files (Caddyfiles). Zeroserve reads a Caddyfile, turns it into eBPF code, then into machine code, and runs it in a special fast event loop.

The main idea is that zeroserve can now work like Caddy, but it is much faster. The author shares a table to compare four servers: zeroserve with two compilers, Caddy, and nginx. Zeroserve (with clang) reaches almost 39,000 requests per second, while Caddy only handles about 12,500. Zeroserve’s median response time (p50) is 1.45ms, while Caddy’s is 4.74ms. The slowest 1% (p99) for zeroserve is also much faster than Caddy. Zeroserve also uses less memory than Caddy.

You can try zeroserve by downloading it, making it executable, and running it with your own Caddyfile. Zeroserve can run custom eBPF code, so you can do advanced things like signing requests for AWS S3 with a plugin. The article shows an example of a Caddyfile that uses zeroserve to reverse proxy an S3 bucket, including authentication with AWS SigV4.

In the Hacker News comments, some people are impressed by the performance results and the technical work. They think using eBPF in userspace is interesting and clever. A few users point out that for most small websites, these speed gains may not matter much. Some worry about security, since running custom eBPF could add risks.
