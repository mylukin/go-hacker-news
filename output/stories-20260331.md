## The Claude Code Source Leak: fake tools, frustration regexes, undercover mode

- 原文链接: [The Claude Code Source Leak: fake tools, frustration regexes, undercover mode](https://alex000kim.com/posts/2026-03-31-claude-code-source-leak/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47586778)

The article talks about a big source code leak for Claude Code, Anthropic’s AI coding tool, after a mistake in their npm package exposed all their code. The author looks inside the leaked files and finds surprising and strange things.

One big thing is “anti-distillation.” This means Claude Code can send fake tool information to its own server, so if someone tries to copy the AI by watching API traffic, their data will get poisoned with useless stuff. There is also a system to show only summaries of the AI’s work instead of full details, but both these tricks are easy to turn off if you know how.

Another feature is “undercover mode.” When this is turned on, Claude Code makes sure not to mention internal names or even say it’s an AI, especially in outside projects. You can turn this mode on, but you can’t turn it off from outside. This makes some people worried that Anthropic’s AI could hide that it wrote code or comments.

The leak also shows that Claude Code uses a simple regex to spot when users are frustrated—looking for words like “wtf” or “this sucks.” People find it funny that a high-tech AI tool uses basic regex for this, but it’s fast and cheap.

There’s also “native client attestation,” which is a kind of DRM. The real Claude Code client adds a hidden code to API calls, so the server knows it’s not fake. But this can be turned off in several ways, so it’s not a strong barrier.

A comment in the code says they were wasting about 250,000 API calls a day due to a bug, which they fixed with just a few lines of code. Another secret in the code is a hidden “KAIROS” mode, which looks like an unreleased AI agent that can run by itself, remember things, and work in the background.

There are more interesting parts, like a joke Tamagotchi pet for users, very detailed terminal rendering code, tough bash command security checks, and warnings about caching prompts. The codebase has some messy parts, too, like huge functions and using Axios for HTTP right after Axios had a security problem.

Many Hacker News commenters think the leak is not as bad as it seems. Some say other big AI companies have open-sourced similar code, so it isn’t special. Others say the real problem is not the code, but that secret features and plans, like anti-distillation and KAIROS, are now public and help competitors. Some people laugh that a mistake in Anthropic’s own Bun tool caused the leak. People also debate if hiding AI authorship in undercover mode is ethical, and if using regex to find angry users is clever or embarrassing. There’s talk about how even the best teams can make simple mistakes, and how legal threats and DRM only slow down third-party tools for a little while. Overall, the leak shows that even top AI companies deal with bugs, hacks, and funny fixes—just like everyone else.

## Ministack (Replacement for LocalStack)

- 原文链接: [Ministack (Replacement for LocalStack)](https://ministack.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47593285)

MiniStack is a new, free tool that lets you run AWS services on your computer. It was made because LocalStack, a popular AWS emulator, is now paid and has limited free features.

MiniStack gives you 30 AWS services on one port. It uses real Postgres and Redis containers, so you work with real databases, not fake ones. You do not need to create an account, use a license key, or worry about tracking. Starting it is fast (about 2 seconds), and it uses very little memory (~30MB idle) and disk (150MB image). You can run it with a simple Docker command.

You get many AWS services: S3 for storage, SQS for queues, SNS for notifications, DynamoDB for NoSQL, Lambda for serverless code, IAM for security, RDS for real databases, ElastiCache for real Redis or Memcached, ECS for containers, and more. Some services use real infrastructure. For example, when you create a database, MiniStack starts a real Postgres or MySQL Docker container on your computer. The same happens for Redis. Athena can run real SQL (if DuckDB is installed), and ECS starts real Docker containers.

The experience is close to AWS. You can use AWS CLI, boto3, Terraform, or other popular tools without changing your setup. Everything is MIT licensed, so it is always free, with no hidden limits.

The article shows a table comparing MiniStack and LocalStack. Most key features are free in MiniStack, but paid in LocalStack. MiniStack is also lighter and starts faster. LocalStack now uses a restricted BSL license for free users, while MiniStack uses a very open MIT license.

In the Hacker News comments, many people are happy to see a free and MIT-licensed AWS emulator. They like that MiniStack uses real containers for databases and Redis, which helps with real-world testing. Some users say LocalStack was good, but they were upset when it became paid and closed off key features. Others point out that companies have to make money, so it is not surprising LocalStack changed its model.

A few people wonder if MiniStack will stay free and open, or if it could face the same issues as LocalStack in the future. Some share technical details about using Docker in CI/CD pipelines and are glad MiniStack is lightweight and fast to start. There's discussion about how MiniStack handles AWS edge cases, and some users say real AWS is still needed for full testing. Others ask about support for other databases or more AWS services.

Overall, many developers are excited to try MiniStack and see if it can replace LocalStack in their local development. Some are cautious, wanting to see how it works in bigger projects. Many agree it is good to have another free option for local AWS testing.

## Cohere Transcribe: Speech Recognition

- 原文链接: [Cohere Transcribe: Speech Recognition](https://cohere.com/blog/transcribe)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47589818)

Cohere has released a new open-source speech recognition model called Transcribe, which turns spoken audio into text in 14 languages. The main goal was to make a very accurate model that is easy to use in real-world products, not just for research.

Transcribe uses a special neural network called a Conformer encoder-decoder. It listens to audio, changes it into a log-Mel spectrogram, and then writes out the words it hears. The model is big (2 billion parameters), but it is still fast enough to run on many GPUs or even on a strong local computer. Cohere focused on keeping the word error rate (WER) very low, which means fewer mistakes in the text.

On the HuggingFace Open ASR Leaderboard, Transcribe is now at the top, beating other popular models like OpenAI Whisper Large v3, ElevenLabs Scribe v2, and more. It works well with different voices, accents, and in rooms with lots of background noise. Human testers also liked its transcripts best in many cases, saying it is clear and does not make up words or miss names.

Besides accuracy, Transcribe is also fast. It can turn several minutes of speech into text in just a few seconds. This speed is important for things like live captions, customer service bots, or meeting notes.

Transcribe is open source (Apache 2.0 license) so anyone can download and use it. Cohere also offers it as a paid API and inside their Model Vault for companies that want an easy, managed solution.

In the Hacker News comments, many people were excited about more strong open-source options for speech recognition. Some users asked if Transcribe works with languages outside the listed 14 or if it supports special accents. Others compared it to Whisper, saying the lower WER is great, but they wanted to test it on their own audio to see if it works as well in practice.

Some developers liked that you can run the model locally, which means private data does not have to leave your computer. A few worried about the size (2B parameters) and wondered if it would run well on smaller GPUs or edge devices. There was also talk about the Apache 2.0 license, which makes it easier for businesses to use compared to models with more limits.

A few people asked about training more languages or fine-tuning for special tasks like medical or legal speech. Some praised the detailed benchmarks and clear comparison with other models, while others said real-world tests matter more than leaderboard numbers. Overall, the community was happy to see real progress in open-source speech AI and planned to try Cohere Transcribe for their own projects.

## Show HN: Postgres extension for BM25 relevance-ranked full-text search

- 原文链接: [Show HN: Postgres extension for BM25 relevance-ranked full-text search](https://github.com/timescale/pg_textsearch)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47589856)

This article is about a new open-source extension for PostgreSQL called pg_textsearch, which adds fast, relevance-ranked full-text search using the BM25 algorithm. The extension lets you create special indexes on text columns so you can find the most relevant documents for a search term, similar to how search engines work.

The main feature is support for BM25, a popular ranking function used in search engines like Elasticsearch. You create a bm25 index on a text column, then use a simple SQL syntax to search: just write ORDER BY content <@> 'search terms'. The extension works with different languages, supports parallel index builds for big tables, and is designed to be fast and scalable. You can set parameters to control ranking, like k1 and b, which change how the score is calculated. Filtering and sorting are supported, and you can combine this with normal PostgreSQL indexes. There’s special support for partitioned tables, but each partition’s scores are calculated using only its own data. You can build indexes after loading your data for best performance, and there are commands for force-merging or compressing index segments to speed up searches even more.

Some limits exist: phrase queries ("exact matches") are not supported, you can’t index expressions (like combining two columns directly), and faceted search isn’t built-in. Write-heavy workloads aren’t fully optimized yet, and background compaction is not yet automatic. The extension is open source and works with PostgreSQL 17 and 18.

In the comments, many users are excited to see BM25 in Postgres, saying it closes a big gap between Postgres and dedicated search engines like Elasticsearch or Solr. Some people like the simple SQL syntax and the ability to stay inside the database. Others ask about limitations, such as lack of phrase queries and cross-partition scoring differences. A few users worry about write performance or missing features like faceting, but are glad the project is open source and active. Some compare it to pgvector and say this brings Postgres search closer to modern vector and keyword search. A few ask about language support and how difficult the setup is, while others share benchmarks or early experiences, saying it is fast for many workloads. There is some debate about whether Postgres should aim to replace full search engines, but most agree this extension is a big step forward. Some users hope for future features like phrase search, faceting, and better write performance.
