## How's Linear so fast? A technical breakdown

- 原文链接: [How's Linear so fast? A technical breakdown](https://performance.dev/how-is-linear-so-fast-a-technical-breakdown)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48437609)

This article talks about why Linear, a project management tool, feels so fast compared to most web apps. The author explains the technical choices and small details that make Linear’s speed possible.

Linear makes the user’s browser the main database. When you change something, it updates instantly in the browser, not waiting for a server response. The server gets the update in the background. This means there are no spinners or waiting for most actions. The app uses IndexedDB in the browser and MobX for fast, fine-grained updates. If the server rejects a change, the app can roll it back, but this rarely happens.

The frontend is built with React and TypeScript, and the backend uses Node.js and Postgres. Linear uses simple tools, avoiding complex frameworks or server-side rendering. They have changed their build system several times (Parcel, Rollup, Vite, Rolldown) to reduce the amount of JavaScript and CSS sent to users. They drop support for old browsers, use code splitting, and preload important files so the app loads quickly.

After the first load, a service worker downloads and caches the rest of the app in the background. This lets users use Linear even when offline. Linear also splits vendor files so updating one library doesn’t force users to redownload everything. Fonts are loaded in a smart way to avoid flashes or double downloads.

When starting the app, the CSS for the loading screen is inlined, so users see something immediately. Authentication is done after the UI is shown—if there’s local data, it’s rendered first, and only later does the app check if the login is still valid.

Linear’s sync engine is built on three ideas: (1) Data is already in the browser when the app starts; (2) Mutations are made locally first, then synced; (3) Updates are very granular, so only the part of the UI that changed is re-rendered.

Design is also a big part of the speed. Most actions have keyboard shortcuts or can be done from a command palette, making the app feel even faster to use. Animations are kept short and only affect properties that don’t slow down the browser, like transforms and opacity, not layout.

In the comment section, many people praise Linear’s speed and attention to detail. Some say more apps should use the “local-first” model, where the browser is trusted first, and changes are pushed to the server later. Others mention the risks: building a custom sync engine is hard, and handling data conflicts or offline errors can be tricky. Some worry about the size of the JavaScript bundle (over 20 MB), but others respond that code splitting and caching make this less of a problem.

A few comment that not all apps need this level of complexity, and tools like SWR or Tanstack Query can give similar “optimistic” updates for simple apps. Some users like that Linear avoids server-side rendering, keeping the architecture simple and easy to understand. People also point out how much design choices, like keyboard shortcuts and fast command palettes, add to the sense of speed.

Some wish more web apps would preload assets and cache aggressively, as Linear does, especially for tools people use every day. A few users share concerns about browser storage limits, IndexedDB quirks, or the extra work required to handle offline and sync properly. Overall, readers agree that Linear’s speed is not from one trick, but from many careful, well-executed decisions over years.

## If LLMs Have Human-Like Attributes, Then So Does Age of Empires II

- 原文链接: [If LLMs Have Human-Like Attributes, Then So Does Age of Empires II](https://arxiv.org/abs/2605.31514)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48437568)

This article asks if we say large language models (LLMs) have human-like qualities, could we also say the same about the video game Age of Empires II. The authors look at how people often give LLMs traits like morality or understanding, but maybe this is not special to LLMs.

They trained a simple neural network to play Age of Empires II and found that if we look hard enough, we might see “human” qualities in almost anything complex, not just in LLMs. The main argument is that giving LLMs human-like traits is not based on unique proof. Instead, it depends on how we choose to see their actions. For example, a game, a city, or even LEGO blocks could be seen as having “human” traits if they are complex enough and we want to see them that way.

The article says that unless we have clear ways to measure these traits, it is easy to make mistakes or go in circles. If we just say LLMs have or do not have these traits, without good tests, we learn nothing new. So, the writers suggest a “null” way to think about LLMs. Instead of assuming they are special, we should start by thinking they are not unique, and then try to prove otherwise with real evidence.

They also show that Age of Empires II is Turing-complete, which means it can, in theory, do any computation a computer can. This supports their idea: if LLMs get special treatment for being complex, why not a game that is just as complex?

In the comments, some people agree with the article. They say humans are quick to see patterns and “personality” in things, even when it is not there. Others remind us that LLMs are trained on human language, so it is natural for them to act a bit like us. Some think that calling LLMs “human-like” is mostly a shortcut—helpful for talking about them, but not true in a deep way.

A few users point out that even if Age of Empires II is Turing-complete, it does not mean it can really understand or talk like a person. They believe LLMs are different, because they are made to use language in a human way. There is also worry that if we treat LLMs as “just programs,” we miss real differences in how they work compared to games or other software.

Some comments mention that the idea of “anthropomorphism” (seeing things as human) is very old. People have always done it with animals, objects, and now with computers. One person says that the real question is not if LLMs are human-like, but why we want to see them that way. Some think it is dangerous, because it could lead to mistakes or trust in the wrong places.

Finally, a few users joke that maybe someday Age of Empires II will be seen as a person too, if we keep going down this path. Others say the article is a good reminder to be careful before giving too much credit to AI or any complex system.

## Powering up a module from the IBM 604: an electronic calculator from 1948

- 原文链接: [Powering up a module from the IBM 604: an electronic calculator from 1948](https://www.righto.com/2026/06/ibm-604-thyraton-tube-module.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48436819)

This article is about the IBM 604, an early electronic calculator from 1948, and how one of its modules with a special tube was powered up. The IBM 604 was not a full computer but a programmable calculator that used vacuum tubes instead of mechanical parts, making it much faster and more powerful for the time.

The IBM 604 was about the size of two refrigerators and was affordable to businesses and universities. It could perform up to 60 math operations in under a second and process about 100 punch cards per minute. People programmed it by plugging wires into plugboards, not by writing code. This plugboard system let users set up different calculations by changing how the wires were plugged.

One big innovation was the pluggable tube module. Each module had a vacuum tube and its needed parts inside a small box with pins. If a module broke, you could pull it out and swap in a new one, making repairs much easier. This was different from earlier machines, where fixing things took more work.

The article focuses on a special type of tube in the IBM 604 called a thyratron. This tube acted like a high-power electronic switch. It could turn on a big current to power things like the machine’s card punch. The thyratron had a little xenon gas inside, which made it work differently from normal vacuum tubes. When you sent a small signal to the grid, the gas would let lots of electricity flow and keep flowing until you cut the power.

The writer even tested a thyratron module by using it to turn a light bulb on and off. Pressing one button started the current, lighting the bulb. The current kept flowing until another button cut the power and turned off the bulb.

These pluggable modules and thyratron tubes helped make the IBM 604 reliable, easier to fix, and mass-produce. The success of the IBM 604 led to more advanced IBM computers in the 1950s, like the IBM 650, which became a real programmable computer.

In the comments, many readers were impressed by how much engineering went into the IBM 604 and its modules. Some said they loved seeing old technology still working and enjoyed the deep technical look at the tube module. Others talked about how smart the pluggable module idea was, making fixing machines much easier for repair people.

A few commenters shared their own stories about working with vacuum tube equipment or seeing it in museums. Some compared the thyratron tube to modern electronics, noting the similarity to silicon-controlled rectifiers (SCRs). There was also some talk about how power-hungry and heavy these machines were, showing how far computers have come.

People discussed the IBM 604’s place in tech history, saying it was an important step between mechanical calculators and true computers. Some felt the plugboard programming was clever but limited, while others admired its speed for the time. One commenter noted the practical use of the 604 in places like Los Alamos for science work.

Overall, readers respected the careful work needed to keep such old parts running and enjoyed learning about the clever ideas from early computing days. Some hoped to see more about restoring and testing other old computer modules in the future.

## What is the purpose of the lost+found folder in Linux and Unix? (2014)

- 原文链接: [What is the purpose of the lost+found folder in Linux and Unix? (2014)](https://unix.stackexchange.com/questions/18154/what-is-the-purpose-of-the-lostfound-folder-in-linux-and-unix)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48409474)

The article explains why there is a folder called `lost+found` at the root of Linux and Unix systems. This folder is there to help when the computer shuts down the wrong way or if the file system gets damaged. When you run a tool called `fsck` (which checks and fixes file system problems), it may find files or data blocks that are not linked anywhere, but still take up space.

If `fsck` finds such lost files, it puts them into the `lost+found` folder. These files once had names and places, but that information is gone now. The folder acts like a “lost and found” for the file system. Usually, these files are there because the system crashed, or because of a power cut, or sometimes due to bugs in software or hardware. Sometimes, the files were about to be deleted, but the system stopped before it finished. Most of the time, these files are not needed, but sometimes they might have important information, even if the names are missing or the files are broken.

Each partition (or disk area) has its own `lost+found` folder. The folder is special because it is created with some extra space for `fsck` to use. If you delete the folder by mistake, you should use the `mklost+found` command to make a new one, not just `mkdir`. After big problems, you might want to check this folder to see if there is anything you want to keep, but most people never need to look in it.

In the comments, people share different views. Some say the folder is mostly empty and you never need to worry about it. Others point out it only appears on certain file systems like ext2, ext3, and ext4, but not all Unix systems. Some say it can be annoying because it often gives “permission denied” errors when you search files as a normal user. A few users remember seeing files in `lost+found` only once or twice after a crash, and sometimes the files are just junk. But some are thankful the folder is there, because it can save important files if something goes wrong. Others wish it was easier to hide or handle, especially when using tools like `find`. There’s also some debate about which systems actually have this folder, but most agree it’s a useful safety net, even if it is rarely needed.

## My automated doubt development process

- 原文链接: [My automated doubt development process](https://www.alexself.dev/blog/automated-doubt)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48437305)

This article talks about how the author builds trust in AI-assisted software development by using many automated checks during the process. The author lost trust in AI tools that tried to do too much, so they created a system to check, question, and review every step.

The main idea is to use different "subagents," which are AI helpers, each focused on a special part of the code or design. The process starts with writing a detailed plan (spec) for a new feature. The author asks AI to create this plan, then uses agents with names like "Assumption Excavator" and "Documentation Validator" to find problems or missing parts. If the project is big, more agents like "Gap Analyzer" and "Ambiguity Mapper" join in to look for things that could go wrong or are not clear.

Each agent gives feedback, such as missing details or possible issues with future updates. The author keeps updating the plan with these findings until it feels complete. Next, the writing of code begins, always following the improved plan. During this, the author avoids letting agents write code directly because past mistakes made them careful. Only one main AI instance writes the code, while others review.

After coding, more agents check the work. These include "Code Validator," "Type Safety Validator," and "Security Analyst." They look for bugs, security problems, and unclear code. This step often finds many issues, which the author then fixes and re-checks with the agents. This repeats until the code meets the author's quality standards.

Before release, a “Ship” workflow runs. It includes agents to check everything again—like code quality, public interface, security, and even “Anxiety Reader,” which looks for risky parts that could cause trouble in production. Only when everything is good does the author release the software.

The article ends with thoughts about quality: everyone, including the AI agents, has their own idea of what “good” means. The author says it is important to keep checking and improving until the project feels right. But this process uses a lot of AI resources (“tokens”), so it may be too much for small projects, or not enough for very big ones.

In the comments, some people really like this careful, step-by-step process. They say it helps catch mistakes early and makes them feel more confident in using AI. Others worry about the cost and time it takes, especially for small teams or solo developers. A few developers explain they just want simple checks and think this process is too complex for most cases.

One commenter points out that using many agents could be hard to manage, and asks for tips to keep things simple. Another suggests that trust in AI will get better over time and maybe someday we won’t need so many checks. Someone else shares that they also use AI for reviews but find the feedback sometimes too generic or repetitive.

Some people are interested in trying out the “Assumption Excavator” agent, saying hidden assumptions are a big problem in their own projects. Others say this style feels like “pair programming with a robot,” which they like. There is also a comment about the value of having many perspectives—human or AI—when building important software.

A few comments raise concerns about relying too much on AI or getting “false confidence” from automated checks. Some ask if these agents are available for other AI models or only Claude. One person wonders if this approach could slow down teams under tight deadlines. Finally, several agree that the idea of “never finished, only released” is true in software, and they appreciate the author’s honesty about the trade-offs.
