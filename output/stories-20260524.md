## DeepSeek reasonix, DeepSeek native coding agent with high caching and low cost

- 原文链接: [DeepSeek reasonix, DeepSeek native coding agent with high caching and low cost](https://esengine.github.io/DeepSeek-Reasonix/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48256953)

Reasonix is a new open-source coding agent that works in your terminal and connects directly to DeepSeek, an AI coding API. Its main feature is a special caching system that makes it much cheaper and faster to use for long coding sessions.

The agent uses DeepSeek’s “prefix cache,” which means most of your chat history does not need to be sent and paid for again—this saves you about 80% on token costs in long sessions. Reasonix is easy to install: you just need Node.js, and you don’t need to install anything globally. When you start it, you paste in your DeepSeek API key, then use your terminal as usual. The tool is designed to fit into your normal command-line workflow, so you keep using tools like git diff and ls.

The software is made to get the best from DeepSeek’s caching by never changing the order or content of messages. It always adds new messages at the end and keeps everything in a steady order. This makes sure the cache works well and keeps costs down. It’s terminal-first, not an IDE plugin, so the focus is on people who want a strong command-line workflow.

Reasonix has other features too: you can add new skills by dropping Markdown files into a folder, connect external tools using a simple protocol, and every session is saved so you can replay or audit it later. All settings and extensions are simple text files—no magic, no hidden stuff, and easy to track in git.

The roadmap shows they want to add things like a project scaffolder, cross-device sync, plugins, and better UI themes. They’ve made the design decision to only support DeepSeek because generic AI backends can’t keep the same kind of cache, making things more expensive and less stable. Reasonix is free (MIT license), but you pay for DeepSeek API usage.

Some users are excited about the high cache hit rate and the cost savings, saying it’s a smart way to handle long AI coding sessions. Others mention they would like support for other AI backends, but understand the technical reasons for focusing only on DeepSeek. A few users wish there was an easier way to try it without signing up for DeepSeek, but others point out the API is cheap and easy to get started with.

Some think the Markdown-based skills are a clever way to add features without needing a big plugin system. A few users wonder if the tool will support teams or multi-user sessions in the future. Others ask about running it with self-hosted DeepSeek servers, and are happy to see that is possible. Most agree the transparent, plain-text config is a good idea.

Some are curious about the real savings compared to other AI coding tools, asking for more real-world usage stats. There are suggestions for more documentation and example workflows. This article is about finding and bringing back an old DOS game called LAN-LOK, made at Palmer Station in Antarctica in 1991. LAN-LOK is special because it was created by researchers during the start of their first computer network, and it shows the real problems and jokes from that time.

LAN-LOK is a game where you try to "break" as many computers on the network as possible in five minutes, while an AI character called Evil Al tries to fix them. You choose which computer to attack by typing its real name, then pick how to break it—like sending too many print jobs, deleting files, or formatting the hard drive. The more damage you do, the more points you get, but Evil Al can take away your points by fixing things. If you format your own computer, you lose right away. The game uses real computer names from Palmer Station, and the humor comes from real-life network troubles that the researchers faced.

The article explains that LAN-LOK was passed around only in Antarctica and was not shared on the internet or in public archives, so it was almost lost. The author found a copy in 2025, confirmed its story with a real person from the game, and now you can play it in your browser or with DOSBox. The article also talks about other games with a similar "break and fix" style, like Wreck-It Ralph and Rampart, but says LAN-LOK was one of the first to use this idea.

To play LAN-LOK today, you have to use an emulator, because modern computers can’t run old DOS games. The article gives clear steps for downloading, setting up, and running the game with DOSBox. It also talks about maybe updating the game in the future using newer tools and AI.

This article talks about how working with Git can be tiring, especially when trying to make “good” commits for big features. The writer explains that commits should be small and focused, making code review easier, but in reality, commits often get messy and mixed up.

The author introduces Jujutsu (jj), a new version control tool. Jujutsu helps jump around commits and change things quickly, but it still takes effort to keep your commit history clean. Tools like jj absorb and jj squash help, but they are not perfect. Absorb sometimes puts changes in the wrong commit, and squash can create hard merge conflicts if commits are not very clean.

The writer shares a new way to fix messy commits. First, you make your ideal commit history, with each commit only having one type of change (like only UI or only database code). Then, you move each change into the right commit, using Jujutsu’s squash tool. You repeat this until all changes are in the right place, and the old messy commit is empty. This approach is like sorting a big pile of laundry: you make a mess first, then sort everything at the end.

This method is easier for big features, because you don’t have to keep your commits perfect as you work. You can make quick, messy commits and clean them up later, which saves effort and avoids “git fatigue.” The writer says this is better than splitting commits as you go, because it’s easier to sort changes in the end and avoid breaking things. However, the downside is that not every commit will compile, which can be a problem for some teams.

A few people say that messy commits are fine as long as you clean them up before sharing. Others like the idea of sorting changes at the end, but some still prefer to make careful commits as they work, to avoid confusion later. There are also comments about how tools like Jujutsu could change how teams work together, making version control less stressful. Some say that any tool that makes it easier to clean up history is a big help, especially on large projects. ## Memory has grown to nearly two-thirds of AI chip component costs

The article explains that memory, especially high-bandwidth memory (HBM), now makes up about 63% of the total cost to make AI chips, up from 52% just a year before. This study looked at chips from big companies like Nvidia, AMD, Google, and Amazon, using their production numbers to estimate costs.

The main point is that the cost of memory in AI chips is rising much faster than other parts. In one year, spending on HBM went from $12 billion to $32 billion. Other parts, like the logic section (the “brain” of the chip), stayed about the same at 13% of the total cost. Advanced packaging (how parts are put together) dropped from 19% to 15%, and smaller, extra parts dropped from 15% to 9%. The total amount spent on making these chips more than doubled, from $22 billion to $52 billion.

Memory is likely going to get even more expensive, because there are not enough HBM chips to go around and prices keep rising. Big cloud companies like Microsoft and Meta are planning to spend billions more on memory for their data centers in the next year. The article also says their cost numbers aren’t exact, because prices change depending on contracts, suppliers, and how many chips are made.

The study used public financial reports and supplier data to estimate costs for four main parts: memory, logic, packaging, and extra components. There is some uncertainty in the exact numbers, but the trend is clear—memory is taking up a bigger and bigger part of the cost.

## Using HTTP/2 Cleartext for a server in Go 1.24

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48195698)

This article explains how to set up a Go server to use HTTP/2 without encryption (called “h2c”), especially for Google Cloud Run. The author needed this because their app uses server-sent events (SSE), and Cloud Run does not notice when a client disconnects if HTTP/1.1 is used.

Before Go 1.24, using h2c was complex. You had to use extra packages like `golang.org/x/net/http2/h2c` and wrap your handler. You also needed to configure both the HTTP server and the HTTP/2 server, then link them. It took several lines of code and was not easy to read.

With Go 1.24 and newer, things are much simpler. You can now just set new protocol options directly on your server. No extra packages or handler wrapping are needed. You write fewer lines, and the meaning is clearer. Now, you set the server to allow HTTP/1 and unencrypted HTTP/2 directly.

To make sure this works before you deploy, you can run a curl command that tries to connect using HTTP/2 cleartext. If the server is set up correctly, it will work; if not, it will fail.

For deploying on Cloud Run, you must also set up the Cloud Run service to use h2c. You can do this in the Google Cloud console, but the article shows how to do it with Terraform. The configuration sets the right port, increases allowed timeouts for long-lived SSE connections, and sets the concurrency so Cloud Run can handle more open connections.

There is no need to change the load balancer settings. The system already upgrades connections as needed, and default timeouts are long enough for most cases.

Some readers were confused about when you need h2c, and the discussion helped them understand it is mostly for internal service-to-service use. ## Migrating from Go to Rust

- 原文链接: [Migrating from Go to Rust](https://corrode.dev/learn/migration-guides/go-to-rust/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48259808)

This article is about moving backend software projects from Go to Rust, focusing on why teams do it, what changes, and how to do it well. It talks about Go and Rust’s strengths, tools, and main differences, especially for backend services.

Go and Rust are both fast, typed languages used for backend work. Go is easy to learn and use, with a simple toolchain and very fast compile times. Rust has a stricter compiler, more powerful types, and checks for more mistakes at compile time. In Go, you often use tools and conventions to avoid bugs, but Rust builds these checks into the language, making many errors impossible to ignore.

One big reason teams move from Go to Rust is to avoid production bugs from `nil` pointers and data races. Go’s `nil` can cause crashes if forgotten, but Rust forces you to handle missing values with `Option<T>`. Go finds data races during tests, but Rust’s type system prevents most at compile time. Error handling in Go means lots of `if err != nil { ... }` code, while Rust uses the `Result<T, E>` type and the `?` operator to make errors clearer and easier to manage.

Go added generics recently, but they are not used much in its standard library. Rust’s generics are central and make code more flexible and type-safe. Rust also gives more predictable performance, with no garbage collector to pause your program, which is useful for high-performance or low-latency systems.

The migration isn’t easy. The Rust “borrow checker” makes sure your code never uses or shares memory in unsafe ways, which is new and hard for Go developers at first. Compile times in Rust are slower, and writing async code is more complex than in Go. The Rust library ecosystem is smaller in some backend areas, so you may have to build or update libraries yourself.

The best way to move from Go to Rust is step by step. Start by rewriting a single service, worker, or background task, instead of the whole system at once. Keep the same APIs, so clients don’t need to change. Don’t try to write Rust like Go—learn the Rust way. Use the compiler’s help, and invest in training.

Go is still good for many jobs, like Kubernetes tools, command-line apps, and simple API layers, especially when you need to move fast and don’t need all of Rust’s safety. Many companies end up using both languages: Go for simple or “boring” services, Rust for critical or high-performance parts.

In the comments, some people say Go is good enough for most backend tasks and like how simple and fast it is. Others think Rust’s strictness and safety are worth the pain, especially for large or important systems. Some warn that Rust’s learning curve is steep, and the borrow checker can be very frustrating for new users. A few say the Rust ecosystem still lacks tools or libraries for some backend needs, but many agree it is improving quickly. Several developers who have made the switch share that after a few months, Rust’s strict compiler feels like a helpful teammate, not an enemy. There’s also debate about error handling: some prefer Go’s explicit style, while others like Rust’s automatic checks. Many agree that both languages have their place, and the best choice depends on your team and your project’s needs.

## Constraint Decay: The Fragility of LLM Agents in Back End Code Generation

- 原文链接: [Constraint Decay: The Fragility of LLM Agents in Back End Code Generation](https://arxiv.org/abs/2605.06445)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48256912)

This article looks at how well large language models (LLMs) can write backend code when they have to follow strict rules. The authors tested LLMs on many backend code tasks, making sure the models had to use certain structures, like specific frameworks and database rules.

The study gave LLMs 80 new code projects and 20 feature additions, all with an “API contract” that the code had to follow. The experiments used eight different web frameworks, covering simple ones like Flask and more complex ones like Django and FastAPI. The authors checked the code in two ways: by running tests to see if the code worked, and by using static tools to check if the code followed the rules.

The main finding is that LLMs do well when there are only a few rules, but their performance gets much worse when more rules are added. This problem is called “constraint decay.” For simple tasks, LLMs pass most tests, but for tasks with strict structure, their test scores drop by about 30 points. Some weaker LLMs almost fail completely when many rules are present.

The authors also found that LLMs do better with frameworks that are simple and have clear instructions, like Flask. But when using frameworks that require following many hidden rules, like Django or FastAPI, the LLMs often make mistakes. The most common errors happen when dealing with databases—like writing bad queries or using the object-relational mapping (ORM) in the wrong way. The study says that getting both the code’s behavior and its structure right at the same time is still very hard for AI coding tools.

People on Hacker News had a lot to say about this. Some think the results are expected: LLMs are good at “guessing” code but not at following all the small rules needed for big projects. Others say this shows why human developers are still needed, especially for hard backend work.

A few commenters argue that maybe the prompts to the LLMs were not clear enough, or that the LLMs might do better if they could see more examples or get feedback while coding. Some people point out that frameworks like Django are hard even for beginners, so it’s not surprising that LLMs struggle.

There are also comments about how real-world projects often have hidden rules that are not in the code, making it even harder for AI to get things right. Some users suggest that LLMs could be improved by training them with more complex, multi-file projects, not just short code snippets. Others think that LLMs might be more useful for simple tasks, like writing small scripts or filling in boilerplate code.

A few are hopeful that these problems will be fixed as models get better. But many agree that we should not trust LLMs for big backend jobs just yet. Overall, people find the study useful because it shows both the power and the limits of AI coding tools today.
