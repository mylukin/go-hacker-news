First, let’s talk about how Apple and Google are using hardware-based attestation more and more. Hardware attestation is a way for your device to prove to apps and websites that it’s real and not changed. Apple has the App Attest API, and Google uses Play Integrity API. Both want app makers and web services to use these tools. Apple already added this to the web with Privacy Pass, and Google wants to do the same.

The main point is that Apple and Google are getting more control over which devices and apps can use their services. With hardware attestation, only devices that pass the checks can use some apps or websites. This blocks custom or open-source systems, like GrapheneOS or LineageOS, and stops rooted or modified phones. The article warns that this could hurt privacy and user choice. If more services require attestation, you may have to use only the official software and hardware from Apple or Google. This gives them more power in the tech world.

Many people worry this will make it harder to use alternative apps or devices. For example, your banking app might not work if you use a different OS. Smaller companies and hackers may not be able to build their own systems easily. The article warns this could create a monopoly, where only Apple and Google decide who can play.

In the comments, people say this is a big problem for freedom and privacy. Some think it’s bad if only big companies can decide which devices are trusted. Others say hardware attestation can stop fraud and cheating in games, and is good for security. But many point out it hurts people who want to change or fix their own devices, and could lock out millions of older phones. Some say we need rules to stop Apple and Google from getting too much control. There are worries that these changes could hurt the open-source movement on phones. Some suggest services should offer both options: with and without attestation. Others fear that soon, most online services will not work unless you use official Apple or Google devices.

Now, let’s talk about SSH security. The article explains how to stop man-in-the-middle attacks when you first connect to a new server using SSH on a VPS or cloud provider. The writer shares a script that uses cloud-init to give the server a temporary SSH host key, which is used only once to connect and set up the real SSH keys. This stops attackers from pretending to be your server during the first connection.

Using a long-term SSH key in cloud-init is risky, because if someone can see the cloud-init data, they can steal your key. By using a short-lived key, the risk is much lower. The script keeps the temporary key away from your normal SSH files and deletes it after use. It also uses OpenSSH’s key rotation to keep only the right keys in your known_hosts file.

In the comments, many people like this idea and say it solves a real security problem. Some say the usual way—just trusting the first SSH connection—is not safe, and this script is better. Others note that some cloud providers have their own solutions, but these only work with their own tools. A few worry about cloud-init leaks, but most agree that using a short-lived key reduces this risk a lot. Some discuss alternatives, but say they are hard to automate or too slow. Overall, the community thinks this technique is a helpful step to make SSH safer, especially on new or small cloud providers.

Next, we have an article about Lakebase architecture and how it helps Postgres write data up to five times faster. Lakebase separates compute and storage, allowing each part to do its best job. Traditional Postgres writes all changes to a log first and sometimes saves big chunks of data, called full page writes, to be safe. But full page writes make the log big and slow things down.

Lakebase changes this by separating compute and storage, so there’s no local disk to corrupt. This means Lakebase can turn off full page writes, cutting extra data and making writes faster. Lakebase’s storage layer also helps: when a page has changed a lot, it makes a new full copy itself, keeping reads fast.

Benchmarks show big gains, with write throughput up to 4.5 times better and much less data sent. Read speed and latency also improve. For some customers, data ingestion jumped from 17,000 to 62,000 rows per second. The rollout was smooth, with no downtime.

In the comments, some people are impressed by the technical solution. Others point out that separating compute and storage can have trade-offs. There are questions about cost and if small projects will see these gains. Some compare this to other cloud databases like Aurora or Neon. Some ask about data durability and recovery. Many agree that moving heavy work to background storage is a smart way to scale modern databases.

Now, let’s go back in time to the 8-bit era. The article talks about eight lesser-known microprocessors from the 1970s and early 1980s. These chips were not as famous as others but have interesting stories.

The Texas Instruments TMX-1795 was designed to be the first microprocessor but never made it to production. Its instruction set later showed up in the Intel 8008, which became important for the x86 family. The Mostek 5065 had features like a special memory page for the stack and fast interrupts, but was a dead end. The Intel 8085 was an update to the 8080 but lost to the Zilog Z80, which had more features.

The Signetics 2650 tried to be like a minicomputer, with vectored interrupts and its own call stack, but had limits and was late to market. The RCA 1802 was one of the first CMOS chips, used little power, and worked well in space, even flying on Voyager and Galileo. Its simple design was both good and bad, missing things like conditional branches.

The Electronic Arrays 9002 was cheap and simple, with only 64 bytes of RAM. The Intersil 6100 was a 12-bit processor based on the PDP-8, but did not fit well in the 8-bit world. The TMS 9900 was a 16-bit chip competing in the 8-bit market. It was slow and part of the TI 99/4 computer, which sold well but lost money for TI.

In the comments, people enjoyed learning about these old chips and shared memories. Some thought it was fun to see all the designs before the market picked a few winners. Others said even failed chips can teach us about design and business mistakes. There were stories about the RCA 1802’s use in space, programming on the TI 99/4A, and debates about which chips really fit the 8-bit era. Many said learning about both winners and losers helps us understand today’s computers. The comments showed a lot of respect for computer history and for the people who designed these chips.
