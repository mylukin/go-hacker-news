Hardware attestation is a way for devices to prove to apps and websites that they are real and not tampered with. Apple uses something called App Attest API, and Google uses Play Integrity API. Both companies want app makers and web services to use these tools. Apple already added this to the web with a thing called Privacy Pass, and Google wants to do the same.

The main idea is that these big companies are pushing for more control over which devices and apps can access services. If more services require attestation, users may have to use only the official software and hardware. This gives Apple and Google more power over the tech market.

People worry that this will make it harder to use alternative apps or devices. For example, you might not be able to use your phone’s banking app if you run a different OS. ## Stop MitM on the first SSH connection, on any VPS or cloud provider

- 原文链接: [Stop MitM on the first SSH connection, on any VPS or cloud provider](https://www.joachimschipper.nl/Stop%20MITM%20on%20the%20first%20SSH%20connection,%20on%20any%20VPS%20or%20cloud%20provider.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48061602)

This article talks about stopping man-in-the-middle (MitM) attacks when you first connect to a new server using SSH on a VPS or cloud provider. The writer shares a script that uses cloud-init to help make your first SSH connection safe, even on providers that do not give special security tools.

The main idea is to create a temporary SSH host key and inject it into the new server with cloud-init. You use this temporary key only once, just to safely connect and set up the real, long-term SSH host keys. The script is careful so the temporary key is never saved in your normal known_hosts file. This stops attackers from pretending to be your server during your first connection, which is a common risk if you just type "yes" when asked to trust an unknown host.

The article explains that putting a long-term SSH private key into cloud-init is risky. If an attacker can read the cloud-init data (from the VM itself, from the provider, or during transfer), they could steal your key. By using a short-lived temporary key, the risk is much lower. The script keeps this temporary key away from your computer's usual SSH files and deletes it after use.

The writer looks at different attack scenarios. If a hacker controls the network, they still cannot break in because they never see the key at the right time. If your computer is hacked but you never connect to the VM, the real SSH keys stay safe. If the attacker only hacks the VM or the cloud provider, your local machine is still protected, as the real key never leaves the VM.

The script also uses OpenSSH’s key rotation features, so only keys that the VM controls end up in your known_hosts file. This helps avoid problems if the host is compromised and tries to trick your SSH client later.

In the comments, many people like this idea and say it is a simple, clean solution for a real security problem. Some note that the usual way—just trusting the first SSH connection—is not very safe, and this script is better. Others say that some cloud providers already have their own ways to solve this, but those ways usually work only with their own tools.

A few users worry about cloud-init leaks, pointing out that any sensitive data sent with cloud-init could be read by attackers. But most agree that using a short-lived key reduces this risk a lot. Some suggest that it’s always good to double-check your known_hosts file after setup, just to be sure.

Others discuss alternatives, like manually copying the host key, but say these are hard to automate or too slow for many servers. One user points out that if you use passwords or agent forwarding, you are still at risk if you don’t check host keys. Some people ask for even easier tools or better integration into popular cloud platforms. Overall, the community thinks this technique is a helpful step to make SSH safer for everyone, especially when using new or small cloud providers.

## Lakebase architecture delivers faster Postgres writes

- 原文链接: [Lakebase architecture delivers faster Postgres writes](https://www.databricks.com/blog/how-lakebase-architecture-delivers-5x-faster-postgres-writes)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48064789)

This article explains how Lakebase architecture helps Postgres write data up to five times faster, which is a big deal for apps that need to handle lots of writes. The main idea is that Lakebase separates compute (the part that does the work) from storage (the part that keeps the data), letting each part focus on what it does best.

Traditional Postgres keeps your data safe using something called Write-Ahead Log (WAL). Every change is written to this log first, and Postgres sometimes makes "checkpoints" by saving chunks of data (pages) to disk. But if the server crashes while saving, it can corrupt a page. To stop this, Postgres normally writes the whole page to the log the first time it's changed after a checkpoint—this is called a "Full Page Write" (FPW). FPWs are safe, but they make the log much bigger and slow down the system, especially with lots of writes.

Lakebase changes this. Since compute and storage are separate and there’s no local disk to corrupt, the risk of a "torn page" goes away. This means Lakebase can turn off FPWs, cutting down on extra data and making writes much faster. But if you just turn off FPWs, reads could get slower because the system has to piece together lots of small changes to rebuild a page.

To fix this, Lakebase adds a smart storage system. When a page has changed a lot, the storage layer itself makes a new full copy of the page. This keeps reads fast without needing big logs from the compute side. As a result, Lakebase reduces network traffic by 94%, scales better, and improves both write and read speed.

Benchmarks show big gains: on a 32-core machine, write throughput is 4.5 times better, and in real production, WAL data drops from 30 MB/s to 1 MB/s. Read speed also improves, with much lower latency. For some customers, data ingestion jumped from 17,000 to 62,000 rows per second. The rollout was smooth, with no downtime for users.

People in the comments had different opinions. Some were impressed by the technical solution and liked how it solves an old Postgres problem. Others pointed out that separating compute and storage can have trade-offs, especially for workloads that need very fast, local storage. A few asked about the cost and if smaller projects would really see these gains. Some wondered how this compares to other cloud database solutions, like Aurora or Neon, that also separate compute and storage. There were questions about data durability and if this new method might make recovery harder in rare cases. A few users warned that while benchmarks look great, real-world performance always depends on the specific use case. Finally, many agreed that moving heavy work to background storage is a smart way to scale modern databases.

## Eight More 8-bit Era Microprocessors (2024)

- 原文链接: [Eight More 8-bit Era Microprocessors (2024)](https://thechipletter.substack.com/p/eight-more-8-bit-era-microprocessors)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48061556)

This article talks about eight lesser-known microprocessors from the 8-bit era, mostly from the 1970s and early 1980s. These chips were not as famous or successful as some others, but they are still interesting and have stories to tell.

First, the Texas Instruments TMX-1795 was designed to be the first microprocessor but never made it to production. Its instruction set later showed up in the Intel 8008, which became important for the x86 family we use today. The TMX-1795 also played a role in a patent lawsuit many years later. The Mostek 5065, made by Motorola and then licensed to Mostek, was a dead-end chip but had features that later became common, like using a special memory page for the stack and having fast interrupts. The Intel 8085 was an update to the successful 8080 but was not much better and lost to the Zilog Z80, which had more features.

The Signetics 2650 tried to be like a small minicomputer and came with vectored interrupts and its own call stack, but it was limited by how its memory was divided and arrived late to the market. The RCA 1802 was one of the first CMOS chips, meaning it used very little power and worked well in space, even flying on Voyager and Galileo probes. Its simple design was both a strength and a weakness, as it left out things like conditional branches.

The Electronic Arrays 9002 was cheap and simple, with only 64 bytes of RAM and limited memory. The Intersil 6100 was a 12-bit processor, based on the DEC PDP-8, but did not fit well in a world moving to 8 bits. Finally, the Texas Instruments TMS 9900 was actually a 16-bit chip but competed in the 8-bit market; it was slow and part of the TI 99/4 computer, which sold well but lost a lot of money for TI.

In the comments, people enjoyed learning about these old chips and shared memories of using or reading about them. Some thought it was fun to see how many different designs companies tried before the market settled on a few winners. Others pointed out that even failed chips can teach us a lot about design and business mistakes. One user mentioned the RCA 1802’s use in space and talked about how important radiation resistance is for those missions. Another person remembered programming on the TI 99/4A and said it was tricky but interesting. Some users debated whether the TMS 9900 should really count as an 8-bit era chip since it is 16-bit, but agreed its story fits the time. There were also comments about how the Z80 was so much better than the 8085, and how the 6100’s 12-bit design felt out of place. A few readers wished more of these chips had succeeded, as more competition might have led to more creative computers. Others noted how chip history is full of strange turns, lucky breaks, and missed chances. Some said that learning about both the winners and losers helps us understand why today’s computers work the way they do. Overall, the comments showed a lot of respect for computer history and for the people who designed these early microprocessors.
