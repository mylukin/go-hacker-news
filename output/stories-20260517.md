A Vermont engineer is bringing back pay phones by making them work with internet phone technology (VoIP) and offering them for free in rural areas. The project started when Patrick Schlott, who loves old gadgets, found old phone parts and wanted to help people who don’t have good cell service.

He uses a device called an analog telephone adapter (ATA) to connect the old pay phones to VoIP services, so calls can be made over the internet. The first phone was installed at a general store where cell service is poor, after the owners realized it could help people in emergencies. Now, seven phones are working in places like libraries, schools, and a town hall, with more planned for the future.

Schlott says using pay phones is important because they are strong, built for public use, and everyone knows what they are for—unlike a normal phone that might look private. The phones are set up for free calls anywhere in the US and Canada. Special codes let you reach help lines, directory assistance, and emergency services, with each phone’s location sent automatically if someone dials 911. He pays for the phones and service himself, sometimes with donations. Finding old pay phones is tricky and prices vary a lot, but the running cost is only a few dollars per phone each month. He even leaves the coin slots working for fun, hoping to inspire people who like to tinker with old tech.

Restoring the phones isn’t always easy—some are damaged, locked, or wired in strange ways, so he often has to reverse engineer them using old manuals and diagrams. But he enjoys learning how all the different models work.

## Prolog Coding Horror

- 原文链接: [Prolog Coding Horror](https://www.metalevel.at/prolog/horror)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48173268)

This article talks about common mistakes people make when writing Prolog code and why these mistakes are a problem. The writer explains that, in Prolog, it is easy to write code that looks fine but gives wrong answers or misses possible solutions.

One big mistake is using "impure" or "non-monotonic" constructs like the cut operator (!/0), if-then-else (->/2), and var/1. These can make your program skip some answers that should be there. The article suggests using pure data structures and constraints instead, which are safer and more logical.

Another common issue is using global state with predicates like assertz/1 and retract/1. This can lead to bugs that are hard to find because different parts of the program can change the state without warning. Instead, you should pass state around using arguments, which keeps everything clear.

Printing answers directly inside your program is also a problem. When you do this, you can't easily test your code or use it as a relation. The better way is to let Prolog's toplevel handle the output and keep your code pure and testable.

The article warns against sticking to old, low-level language features, especially for arithmetic. Using things like is/2 and > /2 makes Prolog harder to learn and teach. It's better to use modern constraints, which are more in line with the logic programming style.

The author shows a "horror factorial" example to highlight the problems. The code uses a cut and low-level arithmetic, so it only works in special cases and fails for general queries. It also makes it hard to debug and reason about the code. By switching to pure constraints and removing the cut, the code becomes more general and handles all queries correctly.

In the end, the writer suggests that Prolog programmers should "rebel" wisely—use modern, declarative features to make code better and more logical, not just different from the mainstream.

In the Hacker News comments, some people agree that beginners often make these mistakes in Prolog, especially with cuts and global state. Others point out that Prolog's syntax and old tutorials encourage these bad habits. A few commenters say that using constraints and pure style makes Prolog much easier and more fun, but it can be hard to learn at first.

Some users think that the cut operator is useful in some cases for speed, though most agree it should be used carefully. Others wish that more Prolog books would teach the pure, declarative style from the start. There is also discussion about how Prolog is different from other languages and why thinking in terms of logic and relations is important.

A few commenters share their own "horror stories" from past Prolog projects, saying they learned these lessons the hard way. One or two people defend some old practices, saying they are not always bad for small, quick programs. Overall, the comments show that Prolog is powerful, but you need to be careful to write good code.

## The occasional ECONNRESET

- 原文链接: [The occasional ECONNRESET](https://movq.de/blog/postings/2026-05-05/1/POSTING-en.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48170799)

This article is about a network error called `ECONNRESET` that sometimes happens between two services talking over TCP on the same machine. The writer sets up a test: one program acts as a server and sends a big amount of data (600,000 bytes) to a client, which reads the data. Most of the time, everything works fine, but sometimes the client gets `ECONNRESET`, meaning “connection reset by peer.”

To find the problem, the writer uses a special flag called `--spam` on the client. With this flag, the client sends some data to the server before reading. When using `--spam`, the error happens often: sometimes the client only gets part of the data before the connection is reset.

Looking at the server, it uses `sendto()` to send all the data and then closes the connection. The server doesn’t crash or show other errors. Tools like `strace` and `tcpdump` show that the server sends all the data, then closes the connection normally, but if the client sent extra data (because of `--spam`), the server never reads it. When the server closes the socket without reading all incoming data, the TCP stack sends a RST (reset) to the client. This causes the client to see `ECONNRESET`.

The same thing happens in real life with gunicorn (a Python web server) behind nginx (a reverse proxy). Sometimes, nginx gets `ECONNRESET` from gunicorn if gunicorn does not read the full HTTP body sent by nginx before closing the connection. The workaround is to make sure the Python app always reads the full HTTP body, even if it doesn’t need it.

The writer’s first guess is that closing a socket with unread data causes the RST. They mention RFC 1122, which says that if a program closes a socket while there is still data to be read, TCP should send a RST to signal data loss. The writer plans to confirm if this is the real cause and maybe report the bug to the right project.

In the comments, some people say this is a normal part of how TCP works: if you close a socket with unread data, you can get a RST. Others say they have seen this problem in production and it’s easy to forget to read all incoming data before closing. Some suggest always reading the request body in web apps to avoid these errors. A few point out that this can be a problem with large uploads, and you need to be careful not to run out of memory.

There are also comments about other ways to avoid this, like using `shutdown()` on the socket before closing, or handling partial reads in the application. Some users focus on how this can cause bugs that are hard to trace, especially under load or with timing differences. Others wonder if it should be fixed in gunicorn or in the web application code. There is some debate about whether this is a bug or just expected behavior from TCP.

Overall, people agree this is a tricky issue that is hard to notice until it causes real problems. Many say it is a good example of why low-level network details matter even in high-level web programming. There are different opinions on the best fix, but most agree that understanding and handling all data before closing sockets is important.

## Where OpenClaw Security Is Heading

- 原文链接: [Where OpenClaw Security Is Heading](https://openclaw.ai/blog/where-openclaw-security-is-heading)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48172952)

OpenClaw is working to make its AI personal assistant both powerful and safe for users. The article explains how OpenClaw handles sensitive actions like reading files, running commands, and connecting to the internet.

The main idea is that OpenClaw can do many things on your computer, but this comes with risks. The team wants to make sure these risks are managed and easy to understand. They talk about a library called `fs-safe`, which keeps files and plugins within safe boundaries, so plugins cannot accidentally touch files they shouldn’t. Instead of letting plugins make their own safety checks, `fs-safe` gives a standard way for everyone to follow. They are also moving certain data, like session details, into a database (SQLite) to reduce how often the system needs to touch the file system at all.

For network safety, OpenClaw uses something called Proxyline. This tool makes sure all network requests go through a proxy, so it’s easier to block dangerous addresses and watch where data is going. Even though Proxyline can’t block every possible network path, it covers most normal cases and makes tracking safer.

When it comes to plugins, ClawHub acts as the main authority for checking if a plugin is safe. They use scans, metadata, and even manual checks to decide if a plugin is clean or risky. If ClawHub marks a plugin as bad, OpenClaw will refuse to install it. The team wants to make it easy for users to see this trust information before installing anything.

OpenClaw also tries to reduce “prompt fatigue”—that feeling when users see so many security prompts that they start ignoring them. Instead, OpenClaw is working on smarter prompts, only asking users when it’s really important, and using better ways to check what commands are actually doing.

To catch bugs, they use static analysis tools like OpenGrep and CodeQL. These tools look for patterns in code that have caused problems before, so new mistakes can be found and fixed early.

People on Hacker News had many thoughts about these changes. Some are happy to see strong security features and clear boundaries, saying it makes them more likely to try OpenClaw. Others worry that even with these tools, a smart attacker might still find ways around the rules, especially if plugins can run shell commands. A few users like the idea of moving sensitive data to databases like SQLite, saying it’s a big step forward. Some commenters want even stricter sandboxing or better isolation, while others think too many checks might make plugins hard to write or slow down the system. There are also comments about prompt fatigue—many agree that too many pop-ups can make users careless, and like that OpenClaw is trying to fix this. A few people are curious about how well the network proxying works in real life, saying it sounds good on paper but might be hard to enforce everywhere. Some like the use of tools like OpenGrep for bug finding, but warn that if the tools are too noisy, developers might start ignoring them. Overall, most agree that OpenClaw is moving in a good direction, but say that keeping the right balance between power and safety will always be a challenge.

## Hindenburg’s Smoking Room

- 原文链接: [Hindenburg’s Smoking Room](https://www.airships.net/hindenburg-smoking-room/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48140556)

The article explains that the Hindenburg airship had a real smoking room, even though the whole ship was filled with flammable hydrogen gas. The smoking room was designed to be safe: it was pressurized so hydrogen could not enter, and it had a double-door airlock to separate it from the rest of the passenger area. Only one electric lighter was allowed inside, and matches or open flames were banned everywhere on the airship. A staff member watched the room at all times. The pressurization was partly for safety, but also to make passengers feel safe. The smoking room was on the lowest deck, so leaking hydrogen would have floated upward, away from it. Still, the real danger was fire—if any fire started, it could spread quickly to the hydrogen cells and destroy the ship. The smoking room was popular, not just for smoking, but also because the ship’s bar was there.

In the comments, one person said the British airship R101 also had a smoking room lined with asbestos, but it likely was not pressurized; another British airship, the R100, did not have a smoking room at all. Someone wondered if the rules changed when the Hindenburg’s sister ship switched to helium instead of hydrogen. Another commenter wanted more details about the smoking room’s look and decorations, like the color of the walls or special maps. There was also a question about how food was cooked if there were no open flames, and the author answered that the kitchen used electric stoves and ovens, even in 1936. Some readers were surprised to learn about the use of electric cooking back then. The mix of comments shows people are interested in both the technical safety steps and the lifestyle of airship passengers. Some think the smoking room was a risky idea, while others are impressed by the clever design and strict rules to prevent disaster.

## Show HN: Semble – Code search for agents that uses 98% fewer tokens than grep

- 原文链接: [Show HN: Semble – Code search for agents that uses 98% fewer tokens than grep](https://github.com/MinishLab/semble)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48169874)

This article is about Semble, a new tool for searching code that is much faster and uses a lot fewer tokens than the usual methods like grep. Semble is made for coding agents and helps them find only the code they need, not whole files, so it saves time and resources.

Semble works by splitting code into smart chunks and searching with both semantic and keyword-based methods. It uses a static model for code similarity and a BM25 method for keyword matches, then combines the results so you get the best matches. It is very fast: indexing an average project takes about 250 milliseconds, and searches take just 1.5 milliseconds. Everything runs on a normal computer CPU—no need for a GPU, cloud, or API keys. You can use it with many coding agents (like Claude Code, Codex, Cursor, and OpenCode) either as a server or from the command line. Semble returns only the exact code snippets that match your question or search term, which means agents don’t waste tokens reading a lot of extra code. You can search with plain language, like “How is authentication handled?”, and get only the relevant pieces back. It also has tools to find code similar to a certain spot in the file and shows you how many tokens you’ve saved by using it. The tool is open source, works with local and remote repositories, and you can even use it as a Python library for your own projects. Benchmarks show it's almost as accurate as very large transformer models but much, much faster and more efficient.

People in the comments are impressed by the speed and token savings. Some users like that it works locally and without a GPU or API keys, which makes it easy to use in many places. Others are happy that it returns just the right code snippets, not whole files, which is useful for coding agents with limited context windows. A few compare it to other tools, saying Semble seems much more efficient for agent use cases, especially when working with big codebases. Some users ask how well it handles different programming languages, and the author replies that it supports many popular ones. There are questions about how Semble splits code into chunks, and the answer points to the Chonkie library, which is designed for code-aware splitting. A few commenters wonder if this tool could also help human developers, not just agents, and others say it looks helpful for anyone who searches code often.
