## Agent Safehouse – macOS-native sandboxing for local agents

- 原文链接: [Agent Safehouse – macOS-native sandboxing for local agents](https://agent-safehouse.dev/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47301085)

Agent Safehouse is a tool for macOS that keeps AI agents safe by limiting what they can do on your computer. The main idea is to stop risky actions, like deleting files outside your project folder, even if an AI agent makes a mistake.

The article explains that many AI coding tools and agents have full access to your files, which can be dangerous. For example, if an agent runs a command like `rm -rf ~`, it could delete everything in your home folder. Agent Safehouse solves this by using the macOS kernel to block any file actions outside your project folder. It stops agents from reading or changing files like your SSH keys or files from other projects.

You install Safehouse by downloading a shell script—there’s no complicated setup. To use it, you just run your agent with the `safehouse` command, and it only has access to the files you allow. The default setting is “deny first,” meaning nothing is accessible unless you say so. For example, your project folder is allowed, but your SSH keys, AWS credentials, and other folders are denied.

Safehouse works with many popular AI agents, like Claude, Codex, Gemini, Aider, and more. The article shows examples of how Safehouse blocks reading private files and stops agents from seeing other folders. You can set up your shell so that every time you run an agent, it’s inside Safehouse automatically. If you ever want to run an agent without the sandbox, you can do that too.

In the Hacker News comments, many people liked the idea and thought it was a good step for safety, especially as AI agents get more powerful. Some developers said they worried about AI accidentally breaking things, so they liked having a tool to limit damage. Others pointed out that macOS sandboxing is hard to set up, so they were glad Safehouse is simple.

A few people wondered if Safehouse would slow down their workflow or cause problems for advanced users. Some said they already use virtual machines or Docker for similar safety, but liked that Safehouse works directly on macOS without extra tools. Others asked if something like this could be made for Linux.

Some commenters said it’s important to trust the tools you use, and worried that Safehouse could block useful things if not set up right. A few users shared ideas for making Safehouse even better, like having more control over what agents can do, or showing clear warnings. There was also talk about open-source and how Safehouse being under the Apache license is a good sign.

Overall, the community thought Agent Safehouse is a helpful, simple way to make running AI agents on your Mac safer.

This article looks at single board computers (SBCs) released in 2025, comparing 15 different boards from 8 manufacturers. The writer talks about boards that cost from $42 to $590, with different chips from companies like Rockchip, Broadcom, Qualcomm, MediaTek, Allwinner, StarFive, CIX, and Texas Instruments.

The budget boards (under $50) include BeagleBoard BeagleBone Green Eco (good for industry, but not fast), StarFive VisionFive 2 Lite (cheap way to try RISC-V, but slow), Arduino UNO Q (Arduino’s first SBC, using Qualcomm chip, but confusing to use), Orange Pi RV and RV2 (RISC-V boards, RV2 is faster, but software is still young), and Radxa Cubie A7A (the best value, with strong performance for the price). 

In the mid-range ($50–$100), the Radxa ROCK 4D offers good value with a Rockchip chip, while the Radxa Dragon Q6A uses a Qualcomm chip and is very fast for $70. The ArmSoM CM5 is a compute module version of the ROCK 4D, for special builds, and the Banana Pi R4 is built for network tasks, not just normal computing.

High-end boards ($100 and up) include the Raspberry Pi 500+ (strong, well-built, but not a huge change), ArmSoM AIM7 (for Jetson Nano replacements), and two new boards using the CIX P1 chip (Radxa Orion O6N and Orange Pi 6 Plus), which have the best performance. The most expensive, Radxa Fogwise Airbox Q900, is made for AI tasks and edge computing, but is not the best value for normal use.

A big problem this year is high RAM prices. Memory costs keep going up because factories focus on memory for AI, so some boards are now more expensive or hard to find. The writer says to always check the latest prices before buying.

Some main takeaways: the new CIX P1 chip is very fast and may become popular if the software gets better; Qualcomm is now serious about the SBC market; RISC-V is growing but still slower than ARM; Rockchip’s RK3576 is a solid mid-range choice, but needs better software support; and there are now more choices and prices than ever before. Raspberry Pi is making small changes, not big ones, and is waiting for better memory prices before making a big update.

## We should revisit literate programming in the agent era

- 原文链接: [We should revisit literate programming in the agent era](https://silly.business/blog/we-should-revisit-literate-programming-in-the-agent-era/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47300747)

This article talks about bringing back literate programming because new coding agents, like AI models, can help make it easier. Literate programming means mixing code and text so anyone can read your codebase like a story and understand it.

The author says he likes the idea, but in the past, it was hard work to keep both the code and the text updated. This extra work stopped most people from using literate programming. Some people do use it, like with Jupyter notebooks for data science, or with Emacs Org Mode, but these are special cases and not common for big projects.

With Org Mode, you write both code and notes together, and you can run code blocks and see the results right in the document. But for big software, it's a pain, because you have to keep exporting ("tangling") the code from the document, and sometimes things get out of sync.

The author says he used Org Mode for personal tasks, like testing commands and keeping notes. This was helpful because you get both the notes and the code together, with no extra work. Now, with AI coding agents (like Claude or Kimi), this gets even better: the agent can write the explanations, keep them updated, handle tangling, and never gets tired of making changes to both code and text.

So, the hard part—keeping everything in sync—is done by the agent. This means it could be practical to have large codebases that read like a story, with the code and explanation always matching. The author thinks this would make code easier to read and maybe improve code quality, because the intent is always written out next to the code.

He admits he hasn’t tried it on a big project yet, just on small tasks. Also, Org Mode is tied to Emacs, which not everyone uses. Markdown is more popular, but it doesn’t support all the features needed for literate programming, like metadata for code blocks.

In the Hacker News comments, some people are excited about the idea. They say that AI agents could finally make literate programming practical, since they are good at explaining things and don’t get bored. Others remember that literate programming was always a good learning tool, but too much work for daily coding.

A few people worry that code and comments will still go out of sync, even with agents, especially when many people are making changes at the same time. Some think Org Mode is too niche, and suggest that we need better, more universal tools for literate programming.

Some commenters share their own experiences: they liked Jupyter notebooks for mixing code and text, but found it hard to manage for big projects. Others ask if having so much prose will really help, or if it will just make code harder to change. There are also jokes about how “nobody reads comments anyway.”

Finally, there’s a debate about whether code should “tell a story,” or if it’s better to keep things simple and just write clear code. Some people are hopeful that AI will change the way we write—and read—code, but others are skeptical and want to see real examples before they believe it.

TrueNAS lets the writer create automatic backups, called snapshots, every hour, day, and week. If a file is deleted by mistake, it can be restored from an old snapshot. Besides file storage, the writer uses TrueNAS to run several apps. Scrutiny watches the health of all the drives and warns if one is failing. Backrest is used to make backups to Backblaze B2, a cloud storage service. Immich manages photos and videos, and works with mobile phones, so the writer doesn’t have to use Google Photos or iCloud. Mealie helps save and organize recipes from cooking websites. Ollama runs small AI models on the server’s GPU, so the writer can experiment with AI at home.

For remote access, the writer uses Tailscale, a VPN that lets them connect to their server from anywhere, without opening their home network to the public internet. Every device needs to use the Tailscale app and be approved to join the network.

Right now, the writer has to remember the IP address and port for each app, which is a bit annoying, so they hope to set up custom domain names for easier access soon.

## Pushing and Pulling: Three reactivity algorithms

- 原文链接: [Pushing and Pulling: Three reactivity algorithms](https://jonathan-frere.com/posts/reactivity-algorithms/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47293195)

This article talks about three ways to build reactive systems: push, pull, and push-pull reactivity. The author uses the example of a spreadsheet, where cells depend on each other, to show how updates should flow when data changes.

First, push-based reactivity means that when something changes, it tells everything that depends on it to update, sending updates “down” the chain. This is simple and works well for only updating what needs to change, but it can do extra work if nodes get updated more than once, and it's easy for things to get out of sync (called glitches) if you aren’t careful with update order.

Pull-based reactivity flips the direction: when something needs to update, it asks its dependencies for their latest value, pulling updates “up” the chain. This makes it easy to keep everything in sync and handle dynamic dependencies (like when formulas only depend on some values based on a condition), but it can be wasteful if it recalculates too much, and caching results is tricky.

The hybrid push-pull method tries to get the best of both. When an input changes, you mark all the things that depend on it as “dirty” (meaning they need to update). Then, only when the output is needed, you recalculate just those dirty nodes. This approach is efficient (each node is updated once), only updates what’s needed, avoids glitches, and handles dynamic relationships easily.

The author says push-pull is usually the best for big, complex systems, especially in web or GUI work, but points out that it can get complicated if updates take a long time or can’t all happen at once. These systems might then need to be split into state machines or other patterns.

Looking at the top comment on Hacker News, the user appreciates the article’s clear breakdown and mentions that many popular frameworks (like React, Svelte, and MobX) use these models or combinations of them. Another commenter points out that push-pull is similar to how spreadsheets and some dataflow languages work, and that the dirty-flag pattern is common in game engines too. Someone else highlights the real-world challenge of tracking dynamic dependencies, especially when conditions change often, and notes that even with careful design, glitches can sneak in if code observes state at the wrong time.

A few readers discuss performance tradeoffs. One person says that for small systems, pure push or pure pull might be simpler and “good enough,” but for bigger apps, push-pull really helps. Another user wonders about how these ideas fit with async data or distributed systems, since in those cases you can’t always guarantee everything updates at once.

Some developers mention the importance of tools and debugging: it’s easy to break things in reactive systems if you don’t know the dependencies, so good visualization and tracing tools are key. Finally, a reader shares that many bugs in their apps were due to accidental glitches, and they wish more frameworks made glitch-prevention easier by default.
