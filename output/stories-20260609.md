Alpine Linux 3.24.0 is out, bringing lots of new and updated software. This version updates big tools like GRUB, LLVM, Rust, GNOME, Go, KDE Plasma, Qt, Sway, and nginx to their latest versions.

A big change is that the setuptools package for Python removed the old `pkg_resources` module, which may break projects that still use it. The `qemu-binfmt` service is now deprecated, and users should use the new `binfmt.d` method instead. The installer is better now, with Limine boot loader support, IPv6, and easier setup for headless (no monitor) systems. COSMIC desktop from System76 is now in the community repository, and GTK+ 3.0 was moved out of the main repository. Old GTK 2, Qt5, and libsoup 2 packages are gone, as part of a cleanup.

To upgrade, users should run `apk upgrade --available` and GRUB users must reinstall GRUB after upgrading. Special care is needed for users with separate `/` and `/usr` filesystems.

Others worry about breakage from package removals, especially the missing `pkg_resources` for Python, since some projects still need it. ## Ultrafast machine learning on FPGAs via Kolmogorov-Arnold Networks

- 原文链接: [Ultrafast machine learning on FPGAs via Kolmogorov-Arnold Networks](https://aarushgupta.io/posts/kan-fpga/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48466277)

This article explains how a new type of neural network, called Kolmogorov-Arnold Networks (KANs), can make machine learning run very fast on FPGAs, which are special hardware chips. The author talks about his Master’s thesis work and two papers that show how KANs help speed up both inference and learning on these chips.

Most machine learning today uses GPUs, which are good for large data and many parallel tasks. But GPUs have delays and are not great for things that need very fast response, like in microseconds or less. FPGAs are chips that you can reprogram to do specific tasks very quickly. They use lookup tables (LUTs) to store simple operations, and you can design neural networks as direct digital logic on them.

The article explains that FPGAs work with bits, not real numbers, so numbers must be turned into bitstrings by a process called quantization. One method is fixed-point quantization, where numbers are represented with a set number of bits, which can cause small errors.

Neural networks on FPGAs can use LUTs to represent their functions, but if you try to use LUTs for big, multi-input functions, the number of possible inputs becomes too large. KANs solve this by using small, one-input (univariate) functions for each connection in the network. These functions are learned and stored in LUTs, which saves space and allows for easy changes.

KANs use special functions called B-splines that are local and only affect a small part of the input range. This makes it easy to update and keeps calculations fast and simple. During inference (when the model is used), all operations happen in parallel using LUTs and adder trees.

A big result is that this method is not only fast (nanoseconds for inference) but also uses less hardware. The author’s team got a 2700x speedup compared to earlier ways of running KANs on FPGAs.

The article also talks about online learning—updating the model while it runs, not just before deployment. With KANs, you can do both the forward and backward passes and update the network directly on the FPGA in less than a microsecond. This is possible because B-splines keep all values within a predictable range, so quantization errors are small and learning is stable.

In summary, KANs mapped to LUTs on FPGAs allow both ultrafast inference and real-time learning, which can help in areas where quick response is needed, like quantum control.

From the comments, many people are excited by the speed and efficiency, saying this could help machine learning in fields where every microsecond counts. Some ask about the limits: what kinds of problems can KANs solve well, and where might they not work? Others wonder about the cost and complexity of building FPGA-based systems compared to just using GPUs, since FPGAs are harder to program and debug.

A few readers point out that KANs, while efficient, may be less flexible than standard neural networks, especially for bigger or more complex data. Some ask if this approach could work for vision or language tasks, or if it's best for control and simple function approximation.

There are also technical questions about the training: how do you avoid errors with quantization, and does this method work for deep networks? Some commenters share their own FPGA experiences, noting it is tricky to balance speed, resource usage, and accuracy.

One commenter suggests this kind of work could be useful for edge devices or low-power applications, where there is no room for big GPUs. Others are interested in the open-source tools and whether the code and hardware designs will be made public.

Overall, the comment section shows interest in the technical details, possible uses, and practical challenges of making machine learning run this fast on custom hardware.

## Upcoming breaking changes for NPM v12

- 原文链接: [Upcoming breaking changes for NPM v12](https://github.blog/changelog/2026-06-09-upcoming-breaking-changes-for-npm-v12/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48467705)

NPM version 12 will soon change how installing packages works to make things safer. These changes will stop some automatic actions during “npm install” unless you say it’s okay.

The biggest change is that install scripts from dependencies will not run by default. Before, scripts like preinstall, install, or postinstall could run automatically. Now, you must approve each one. This also blocks scripts for native modules using node-gyp, unless you allow them. “Prepare” scripts from Git, file, or link dependencies are also blocked unless approved. To help, NPM gives you commands: you can run “npm approve-scripts --allow-scripts-pending” to see what scripts want to run. Then, you can approve the ones you trust using “npm approve-scripts” and deny others with “npm deny-scripts.” The list of your choices goes into your package.json file, and you should save this to your code repo.

Another big change is about Git dependencies. Now, NPM will not install packages directly from Git unless you use a special “--allow-git” flag. This helps close a security hole where Git dependencies could run code in bad ways. This rule also stops remote URL dependencies, like tarballs from the internet, unless you use “--allow-remote.” Local file and directory installs are not changing.

To get ready, you should upgrade to npm 11.16.0 or newer, install your packages as usual, and look at any warnings. Approve scripts you trust, and commit your changes. When you later upgrade to npm 12, only approved scripts will run.

People in the comments have many thoughts. Some like these changes for better security. They remember times when install scripts broke their setup or did something they didn’t expect. Others worry this will break builds, especially for tools that depend on install scripts, like native modules or some front-end libraries. Some say the new approve/deny system could be hard for beginners or big teams with many dependencies. A few think the process will be noisy at first but will make things better later. Some users want even more control, like better ways to see what scripts do before approving them. Others suggest that package authors should stop using install scripts unless really needed. There is also talk about how this change might affect continuous integration and deployment tools. Some say it’s good that npm gives warnings in older versions before the big switch. Finally, a few ask for help guides and better documentation, since these changes could confuse people at first.

## Test-case reducers are underappreciated debugging tools

- 原文链接: [Test-case reducers are underappreciated debugging tools](https://tratt.net/laurie/blog/2026/test_case_reducers_are_underappreciated_debugging_tools.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48459659)

This article talks about test-case reducers, which are tools that help make failing test cases smaller so bugs are easier to find. The author says these tools are not used enough, even though they make debugging much easier.

The main idea is simple: when a program fails with a big input, it’s hard to know what part caused the problem. If you can make the input smaller and still see the bug, it’s easier to understand what’s wrong. The author shows that you can do this by hand, but it’s slow and boring. Test-case reducers automate this process. You give the reducer your program, your failing input, and a small script called an “interestingness test” that checks if the bug is still there. The reducer then tries to remove parts of the input, running the test each time, and keeps only the changes that do not fix the bug. Even a simple reducer written in Python can cut down input size by a lot, sometimes over 90%. There are advanced tools like Shrink Ray that do this even better and faster. The article also explains that writing good interestingness tests is important and can be hard. If your test is wrong, the reducer might remove too much and the bug will disappear. The author also shows that reducers can help turn random bugs (that don’t always happen) into bugs that always happen, making them easier to fix. You can even ask reducers to prefer inputs that are faster to run or cause shorter logs, not just smaller files. The author shares tricks for making reducers work better, like running tests in parallel or restarting the reduction when a new, smaller input is found.

In the comments, many readers agree that test-case reducers are very useful, especially for debugging compilers and big systems. Some say they use tools like creduce or delta and could not live without them. Others say they did not know these tools worked for things other than code, like logs or text files. A few people warn that writing a good interestingness test is tricky and you have to be careful, or you might lose the bug. Some users wish these reducers were easier to use with other languages, not just C or C++. One commenter points out that sometimes reducing the input too much removes context and makes the bug harder to see. Another says that reducers are slow for big tests, but it’s worth the wait. Some readers are happy to see clear examples and say they plan to try reducers in their next debugging session. There is also a short debate about whether reducers are as useful for random bugs, but most agree that they help a lot. Finally, a few people share their own scripts or tips for writing better reducers and tests.

## Exif Smuggling

- 原文链接: [Exif Smuggling](https://github.com/signalblur/exifsmugglingpoc)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48467759)

This article is about a proof-of-concept attack called “Exif Smuggling,” which hides a dangerous file inside a photo’s metadata. The project shows how attackers can use both browser caching and Exif data (hidden information in images) to load malware without making clear network requests.

The attack works by putting a second, hidden file (like a DLL) inside the Exif section of a JPG image. When a user visits a phishing site, their browser saves this image in its cache. Later, a loader script (for example, written in PowerShell) can pull out this hidden file straight from the browser cache, so it never needs to download the dangerous file openly. This makes it much harder for security tools to spot or block the threat, since there is no direct download happening when the loader runs. The GitHub project includes scripts to help convert loader commands, hide the payload inside a JPG, and even build example phishing pages. The code uses Python and PowerShell, making it easy for attackers to try on different systems.

Exif Smuggling is a new step in cache smuggling attacks. It’s clever because it uses normal web features (image caching, photo metadata) in a way that security teams don’t expect. The linked blog post gives more details and shows why this method is hard to detect.

In the comments, many readers are surprised by how simple and effective this trick is. Some say it’s a reminder that browser caches can be a weak spot in security. Others wonder if antivirus tools will start scanning Exif data in images more closely now. A few point out that hiding data in images (steganography) is not new, but using browser cache this way is a smart twist. Some developers talk about ways to block this, like clearing browser caches or not trusting cached files. Others worry that normal users and even many IT teams won’t know about this risk. People also discuss how websites could stop serving images with strange Exif data, or scan uploads for hidden files. Overall, the discussion shows both respect for the cleverness of the attack and concern for how hard it might be to defend against.

## A giant star may have destroyed itself in one of the rarest explosions

- 原文链接: [A giant star may have destroyed itself in one of the rarest explosions](https://phys.org/news/2026-05-giant-star-destroyed-universe-rarest.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48451966)

A giant star may have exploded in a very rare way called a “pair-instability supernova.” Astronomers found this event, named SN 2023vbw, in a small galaxy about 1.3 billion light-years away.

This explosion was first thought to be a normal Type II supernova, but scientists noticed its brightness changed in a strange way. Instead of fading quickly, it kept getting brighter for about 190 days, then faded fast, and then stayed bright for a while. The explosion was much stronger than a normal supernova, with more energy and much more material thrown out into space.

The star that exploded was very big, maybe between 170 and 350 times the mass of our Sun. It was probably a “blue supergiant,” which is a very hot and massive type of star. The location of the explosion had few heavy elements, which matches the theory for this rare kind of supernova. The star may have formed when two stars joined together, creating a big, fast-spinning star that then exploded.

Scientists also saw signs that the material from the explosion hit a disk of gas around the star, which the star had lost before. This supports the idea of a merger. But there are still many questions, like whether big stars become red or blue before they die.

A pair-instability supernova happens when the core of a very big star gets so hot that it makes particles called electron-positron pairs. This takes away the pressure in the star’s center, and the star suddenly explodes with nothing left behind—not even a black hole.

Stars that do this are very rare, needing to be at least 140 times the mass of the Sun and in places with not much metal. SN 2023vbw fits these rules and is close enough for telescopes to keep watching. New telescopes will help find more of these explosions in the future.

In the Hacker News comments, some people were amazed by how huge these stars are, and how the explosion can leave nothing behind. Others talked about how hard it is to study such rare events, since they happen so far away and so few times. Some users discussed the science behind pair-instability supernovae, sharing links to past studies and asking if this changes what we know about how stars die.

A few people wondered if this event could help us learn more about the early universe, since big stars like this were more common back then. Others were excited that new telescopes might find more of these explosions soon, making it easier to understand how the biggest stars live and die.

Some comments were more skeptical, saying it’s hard to be sure this is really a pair-instability supernova, since we can’t see the star before it exploded. Others pointed out that computer models are still not perfect, and there is much to learn about how massive stars really explode. A few users just enjoyed reading about the strange and powerful things happening far out in space.
