First, Google has changed how reCAPTCHA works on Android. Now, if you use an Android phone without Google Play Services — for example, if you use a privacy-focused system like GrapheneOS — you may not be able to pass reCAPTCHA checks on many websites. Google’s new rule asks users it thinks are “suspicious” to scan a QR code, but you need Google Play Services to do that. If you do not have Play Services, you just fail the check. Google says this is part of its new “Cloud Fraud Defense” system to fight bots and AI. But the article argues this is really about making people use Google’s own software.

For iPhone users, there is no such rule, so this mostly hurts Android users who want privacy and avoid Google’s apps. These users are now treated as suspicious just because they want more privacy. The article warns web developers: if you add this new reCAPTCHA, you are locking out people who care about privacy, even if it is a small group.

In the Hacker News comments, many people are upset about this move. Some say it is unfair for Google to force Android users to use Google software just to access websites. Others say Google is using its power to control Android and push out privacy-minded users. Some people argue most users won’t care, since they already use Google apps, and that Google has a right to protect its systems from bots. But there is a strong feeling that the web is becoming less open and more controlled by big tech. Some suggest websites should stop using Google reCAPTCHA and choose more privacy-friendly options. Others point out that captchas in general are getting harder for real people and easier for bots, so maybe we need a new solution. Some say the change makes sense for security, but wish Google would be more open and allow exceptions. Most agree this is bad news for privacy and user choice.

Next, let’s talk about software security and AI. There is a story about a recent Linux bug, called “Copy Fail,” and how it shows the changing way people handle security bugs. There are two main ways: coordinated disclosure, where someone finds a bug, tells the team in private, and waits for a fix before telling the world; and the Linux way, where bugs are fixed quickly in public, hoping attackers do not notice before users update.

But now, with AI, both methods have problems. AI tools can spot security changes in code almost instantly, so hiding fixes in public is much harder. Also, long embargo times are not safe, because AIs and hackers can find the same bugs fast. In the “Copy Fail” case, two people reported the same problem within nine hours. The article suggests that very short embargoes, maybe just a few hours or days, might be the best way now.

The author even tested Gemini, ChatGPT, and Claude, and found that AI can spot security patches quickly, even without full context. In the comments, some people say public disclosure is better and helps users stay safe. Others worry that bugs shown too soon will be used by attackers before people can update. Many think embargoes are now almost useless, since AI makes everything so fast. Some suggest using AI for defense, like real-time patching. Others say Linux’s open style worked before, but now anyone — good or bad — can use AI to spot important fixes. There is debate about whether open or closed handling is safer, and if we need new rules for the AI era. Some feel “security through obscurity” is dead and software needs to be safer by design. Many agree security is changing fast and there are no easy answers.

Speaking of Linux security, there is also news about a new bug in Linux kernels 6.15 to 6.19, found in the zero-copy receive feature of io_uring. This bug lets users with special network rights, called CAP_NET_ADMIN, get full root access by carefully using network hardware and kernel memory. The bug happens because the kernel does not check the upper limit when adding items to a memory list. Two parts of the kernel can return network buffers at the same time, pushing a counter too high and causing an out-of-bounds memory write. An attacker can control where and what is written, and by overwriting certain memory, can read secret kernel addresses and change settings to gain root.

The article gives a step-by-step guide, with code and explanations. The bug is fixed in a recent commit, but at the time of writing, the fix was not in main Linux branches. In the comments, many people are impressed with the technical details and creativity of the exploit. Some say this is a reminder how small memory bugs can become big security risks, especially in complex kernel features. Others note that the bug is not in most Linux systems yet, since ZCRX is new and only used with special network cards. Some discuss that needing CAP_NET_ADMIN limits how many users can use the bug, but warn that in cloud or container setups, this right is often given more widely than it should be.

Some wonder why such a simple bug — a missed bounds check — was not caught. Others talk about how complex kernel systems are hard to review, especially with new hardware. There is debate on how practical the exploit is, but people agree it is a serious, well-explained bug, showing the need for careful code review and fast security updates.

Next, we look at a story about handling security bugs in software packages, called CVEs. The article says fixing these problems is getting harder because AI can now find many more vulnerabilities, and tracking what software and versions you use is very difficult. Most companies do not have a clear record of all their packages and versions. Package managers like apt, npm, or pip can install different versions depending on many things, making it hard to know if you are at risk.

The author suggests using tools like Nix and Flox. Nix makes builds the same every time and records every input, making it easy to track and fix problems. Flox uses Nix but makes it easier for teams. With these tools, you can check only the unique sets of packages instead of every server, saving time. The article also talks about non-determinism — with most package managers, you get different results every time you install, which makes tracking problems hard.

In the comments, some users agree non-determinism is a big problem and like Nix and Flox, but worry they are hard to learn and slow. Some say big companies have their own tools, but smaller teams may struggle. Others say that disk space and slow builds are less of a problem now, so tools like Nix make more sense. Some are skeptical and say changing package managers is risky and takes effort. Others warn that even with good tools, you still need to watch for changes made outside the system. Overall, people like the idea of more control, but worry about cost and complexity. There is agreement that the old way does not scale, and new tools are needed, but the best path is not clear.

Now, a creative project: running a public website on a Raspberry Pi Zero using only RAM and no disk. The writer uses a Pi Zero with 512MB memory, running Alpine Linux in diskless mode. The microSD card is just for starting; after that, everything runs in RAM, making the system fast. The website is just static files, kept small to fit in memory. Because the Pi is weak and cannot handle HTTPS well, traffic goes through a cheap VPS using HAProxy, which does the heavy work and forwards HTTP traffic to the Pi at home.

In the comments, many readers say this project is fun and clever. Some like how little hardware it uses and that everything fits in RAM. Others point out that diskless systems can be risky if you forget to save before reboot. Some think it is great for learning, but not practical for big sites. A few suggest using other hardware for more power or reliability. Some warn about exposing home devices to the internet and recommend being careful with security. Others discuss using a VPS or CDN for the front-end, or praise the project for avoiding big cloud providers and tracking. Some say this method could work for small projects like file sharing or IoT dashboards. Overall, people are impressed by how much can be done with so little and appreciate the clear guide.

Next, a story about Meshtastic, an open source project that uses cheap LoRa radios to let people talk over long distances without phone or internet. The radios work over many kilometers, create a mesh network, and do not need a license. You can connect a phone to a radio to send messages, and messages are encrypted. The project is on GitHub, with guides and a friendly community.

On Hacker News, people like that Meshtastic is open source and does not need a central server or internet. Some say it is useful for outdoor adventures, disasters, or remote places. A few share stories about using Meshtastic for hiking, camping, or big events. Others discuss technical details, like LoRa’s range and security. Some worry about cost and needing special hardware, but others say the radios are cheaper than satellite phones and have no monthly fees. Some suggest new features, like better group messaging or easier apps. Overall, most think Meshtastic is a clever idea with lots of potential for off-grid communication.
