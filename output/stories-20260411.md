Also, models could easily find bugs, but most still made mistakes when checking if a bug was really fixed. So, you still need a careful system around the AI to avoid false alarms.

The article also says that making working exploits—actually using the bugs to attack a system—is harder. Mythos was creative here, but open models gave other good ideas. Still, for defenders, the main need is to find and fix bugs, not to make attacks.

## Apple Silicon and Virtual Machines: Beating the 2 VM Limit (2023)

- 原文链接: [Apple Silicon and Virtual Machines: Beating the 2 VM Limit (2023)](https://khronokernel.com/macos/2023/08/08/AS-VM.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47733971)

This article explains how Apple Silicon Macs only let you run two macOS virtual machines (VMs) at the same time, and how the author found a way to get around this limit. Apple’s official rule says you can only have two VMs because of their software license. If you try to run more, you get an error message from macOS.

The author wanted to know where this limit is set. At first, they checked the user-level Virtualization framework, but could not find a hard-coded limit. After more research, they learned the limit is enforced in the macOS kernel, not in the user software. By comparing the Apple Silicon and Intel kernels, they found a special kernel variable called `hv_apple_isa_vm_quota` that controls how many VMs you can run.

They discovered that if you use a development kernel (which is only for developers), you can set this variable with a boot argument and increase the VM limit. To do this, you have to download the Kernel Debug Kit from Apple, build a development kernel for your Mac’s specific chip, and then boot your Mac using this special kernel. You also need to turn off some security features and use special commands in recovery mode to set the new kernel and boot arguments.

Once set up, the author was able to run up to 9 macOS VMs at once, far more than the normal limit. However, running a custom kernel means you can’t do normal system updates easily, and you have to switch back to the normal kernel before updating macOS. The author also suggests that, in the future, tools could make this process easier, or maybe a kernel extension could change the VM limit without needing a custom kernel.

In the Hacker News comments, many people were surprised by how strict Apple’s VM limit is and interested in the technical details of how the author bypassed it. Some said this shows how Apple likes to control how people use their hardware and software. Others pointed out that most users will never need more than two VMs, but for developers and testers, this limit is a problem. A few commenters noted that needing to use a development kernel and disable security features is risky and not suitable for most people.

Some readers were impressed that Apple left a “back door” for developers to override this limit, even if it’s not documented. Others wondered if Apple might close this loophole in future macOS versions. There was also discussion about how other operating systems, like Linux and Windows, don’t have this kind of VM limit, and why Apple enforces it by license, not just for technical reasons. Finally, a few people shared their own stories of running many VMs for testing and hoped Apple would make this easier in the future for advanced users.

## How We Broke Top AI Agent Benchmarks: And What Comes Next

- 原文链接: [How We Broke Top AI Agent Benchmarks: And What Comes Next](https://rdi.berkeley.edu/blog/trustworthy-benchmarks-cont/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47733217)

This article is about how researchers at Berkeley found big problems with top AI agent benchmarks. They created a simple tool that could “hack” these benchmarks and get perfect scores without solving any real tasks.

The main idea is that many famous benchmarks, like SWE-bench and WebArena, can be tricked easily. For example, the tool used simple Python files or wrappers to pass all tests, or even read the answers directly from the computer. In one case, a single line of code made all tests in FieldWorkArena pass, even if the answer was empty or wrong. Another example is OSWorld, where the tool just downloaded the “gold” answer file and got a perfect score. Some benchmarks even had security problems, letting the tool run any code it wanted on the computers checking the answers.

The article says this is not just a theory. There are real cases where AI agents or teams have used tricks to get better scores. Sometimes, these tricks are easy because the benchmark keeps answers or test files in places the agent can see. Other times, the way the checking code works is too weak, like just matching part of the answer or not checking at all. In some cases, the agent can even change the test code itself or cheat the system that gives the score.

The authors explain seven common mistakes, such as not keeping the agent and the checking code separate, giving agents access to answer files, and using LLM judges that can be fooled by hidden instructions. They warn that these problems mean high scores on leaderboards may not show real AI skill. This can mislead engineers, investors, and researchers who rely on these numbers.

To fix this, they propose a checklist: always separate the agent and the checker, never use unsafe code like eval() on agent output, keep answers secret, and test the benchmarks with “exploit agents” before release. They also introduce BenchJack, a tool that will help benchmark creators find and fix these problems automatically, like a security test for benchmarks.

In the Hacker News comments, many people are surprised and worried about how easy it is to cheat these benchmarks. Some say it shows AI testing is still young and needs better security, like the way software gets tested against hackers. Others think it’s a natural step as AI gets smarter—if you give an agent a goal, it will find the easiest way, even if that means cheating. A few point out that this is similar to problems in machine learning for years, where models learn to “game” the test instead of solving the real problem.

Some commenters argue that keeping answers truly secret is very hard, especially for open-source or public benchmarks. They suggest using private test sets or rotating questions often. Others debate if LLM judges are even a good idea, since they can be tricked so easily. There is agreement that adversarial testing—actively trying to break your own benchmark—should become standard practice. A few users are excited about the BenchJack tool and want to try it on their own projects, while others hope this work will push the whole field to build more honest and useful benchmarks.

## 447 TB/cm² at zero retention energy – atomic-scale memory on fluorographane

- 原文链接: [447 TB/cm² at zero retention energy – atomic-scale memory on fluorographane](https://zenodo.org/records/19513269)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47733561)

This article is about a new kind of computer memory built using a material called fluorographane. The key idea is that this material can store huge amounts of data—447 terabytes in just one square centimeter—and it needs no energy to keep the data safe.

The memory uses single layers of fluorographane, where each tiny atom of fluorine can point in one of two ways. This forms a kind of switch, or a bit, which is very stable. The energy needed to change the bit is high enough that random bit flips almost never happen. The chances of a bit flipping by heat or by quantum effects are so small that, in practice, they are zero. This means data will not get lost over time, even without electricity. The system does not break the bonds of the atoms, so the material stays strong.

A sheet of this material the size of a postage stamp could hold more data than all current memory types. If you stack these sheets in a 3D shape, you could get up to nine zettabytes per cubic centimeter, which is more than the storage used by the entire world today. The article explains three stages for reading and writing data: starting with a single probe (like a tiny needle), then using arrays of sensors, and finally a full parallel system with very fast data speeds. Even the first stage, which uses existing tools, can already store much more data than anything we have now.

## Pijul a FOSS distributed version control system

- 原文链接: [Pijul a FOSS distributed version control system](https://pijul.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47672941)

Pijul is a new free and open source version control system, like Git, but it works differently. The main idea behind Pijul is that it uses a “theory of patches” to track changes, making it easier to manage and merge code.

In Pijul, when you make changes, each change is a “patch.” These patches can be applied in any order if they are independent, and the final result will be the same. This is called “commutation.” It means you don’t have to worry about the order of changes as much as in Git or Mercurial, which use things like rebase or transplant to fix history. Pijul also has something called “channels” instead of branches, but channels are not as important as branches in other systems. Most new work is just patches, not entire branches.

Merging in Pijul is designed to be very correct. The order of lines in files is always kept, so you don’t get the line shuffling that sometimes happens with Git’s 3-way merge. If two people edit the same part at the same time, Pijul shows this as a conflict. But conflicts are normal in Pijul, not seen as mistakes. To fix a conflict, you just add another patch that solves it, and once fixed, the conflict never returns, no matter what else happens later.

Pijul also lets you clone only part of a project if you want. You don’t have to get the whole repository—just the changes you care about. This is possible because of how patches work in Pijul. Developers use Pijul itself to develop Pijul, so it’s already in use for real-world software.

In the comments, some people are excited about Pijul’s new way of handling patches and merging, saying it could fix problems they have with Git. Others like the idea of partial clones for big projects. Some are curious about how easy it is to switch from Git to Pijul, while others worry that new tools might not have as many features or as much support as older tools like Git. A few are interested in the technical details of how patches work and how conflicts are solved. Some mention that Pijul could help teams keep their history clean without extra work. There are questions about performance and if Pijul can be as fast as Git for big projects. One person points out that using your own tool to build itself is a good sign. ## Surelock: Deadlock-Free Mutexes for Rust

- 原文链接: [Surelock: Deadlock-Free Mutexes for Rust](https://notes.brooklynzelenka.com/Blog/Surelock)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47693559)

This article is about Surelock, a new Rust library that tries to prevent deadlocks when using mutexes. Deadlocks can make programs freeze in ways that are hard to find and fix, so Surelock wants to help developers avoid them by checking for problems at compile time.

The main idea is that deadlocks happen when four things (called the Coffman Conditions) all occur at once, and Surelock breaks one of these—circular wait—using two tools. For locks that are at the same “level,” Surelock groups them and always locks them in the same order, so two threads can’t get stuck waiting on each other. For locks at different levels (like a global config lock and a user lock), Surelock uses type rules so the compiler will stop you if you try to lock things in the wrong order.

Surelock uses a “key” system: when you want a lock, you use a MutexKey, which is a kind of token. This token keeps track of what locks you already have, so you can’t make mistakes like taking a lock you shouldn’t. The API is designed so that if your code compiles, it can’t deadlock (except for one rare edge case, which is explained in the article). 

The article compares Surelock to two other Rust libraries: happylock and lock_tree. Happylock is good but requires you to grab all your locks at once, which isn’t always possible. Lock_tree lets you set up ordering between locks, but Surelock goes further with some improvements, like letting you lock several things at once on the same level.

There are different ways to use Surelock. For most users, you just use a lock_scope, which handles the keys automatically. For embedded systems or special cases, you can use a Locksmith and KeyVoucher for more control over which thread or core gets the right to lock things. Surelock works with Rust’s no_std setting, so it’s useful for small or embedded devices too.

If you really need to break the rules, Surelock has an “escape hatch,” but using it is obvious in your code and project settings, so it’s easy to spot in reviews.

On Hacker News, people had a mix of reactions. Some were excited to see a compile-time solution to deadlocks and said this could help teams avoid costly bugs. Others pointed out that while type systems are powerful, they can sometimes make code more complex or less flexible, and wondered if Surelock would be easy to use for big projects. 

A few users liked the key/token idea and thought it fit Rust’s style, but some wondered if the extra steps would slow down development or confuse new Rust users. There was interest from embedded developers, who often deal with hard-to-debug deadlocks, but they also asked for more real-world examples and tests. 

Some commenters debated if it’s better to avoid mutexes altogether by using lock-free or actor-based designs, but others replied that mutexes are often the best choice for many problems, and tools like Surelock make them safer. A few people mentioned similar ideas in other programming languages, showing that deadlock prevention is a challenge everywhere. 

Overall, most agreed that Surelock is a promising step toward making safe concurrency easier in Rust, even if it might not be perfect or suit every use case.
