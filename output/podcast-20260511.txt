First, let’s talk about the malware found in the TanStack npm packages. Recently, some new versions of these popular JavaScript packages were discovered to be infected. The first sign came from the @tanstack/history package, but soon other related packages were checked as well. The attack works by hiding a special dependency in the package.json file, which tells npm to fetch a certain commit from GitHub. When you run “npm install,” this commit triggers a “prepare” script that runs a big, hidden JavaScript file called router_init.js.

This hidden code tries to steal credentials from the infected computer, looking for AWS, GCP, Kubernetes, npm, GitHub, and SSH keys. The stolen data is sent over an encrypted channel, making it hard to block or track. The code also tries to find what other npm packages the user maintains, so it can spread and infect even more packages.

The attacker used GitHub Actions OIDC to publish the infected code, showing that this is not just a leaked token problem—the whole publishing workflow may be compromised. Simply rotating tokens will not fix it; the workflow itself must be disabled. To check your packages, you can unpack them and look for strange optionalDependencies and the router_init.js file.

One commenter warned that the malware even installs a “dead man’s switch.” If you try to block the stolen GitHub token, the script may delete your files. It sets up a background service to watch if the stolen token still works; if it is revoked, it erases the user’s home directory, making recovery very hard.

People in the comments had many reactions. Some said this shows how dangerous the npm supply chain is and that we need better security. Others noted that install scripts are always risky and should be avoided. Some users want npm to block install scripts by default or at least show a strong warning. A few were shocked at the steath of the attack, hiding in a commit and spreading itself. Others asked for clear cleanup steps and ways to prevent future attacks. There were calls for more transparency from maintainers and registries, and reminders to always check dependencies, keep backups, and use strict security settings.

Next, we have TypedMemory, a new Java library that lets developers map Java records onto off-heap memory using the FFM API in Java 25 or newer. This makes it easier to work with memory outside the heap, keeping things strongly typed and safe. Instead of writing a lot of low-level code, you can use simple get and set methods to access structured data in native memory. It works well with Java’s FFM model, supports nested data and fixed-size arrays, and helps check memory layouts for native interop. It’s still experimental and only works with Java 25 or above.

In the comments, many developers are excited, seeing value for high-performance and native interop. Some say it brings Java closer to C for memory tasks. Others like that it makes the FFM API easier and less error-prone. A few point out the limitation that you need Java 25, which is still new. There are questions about arrays as fields, union types, and complex layouts. Some want to see benchmarks, while others warn that changes in Java could break the library. Overall, there’s excitement, but people know this is early and needs more testing.

Now to security news: Google says hackers used AI to find a new zero-day bug in software. This is the first time Google has seen AI help criminals find and use a secret bug. They didn’t say when the attack happened or what AI tool was used, but it was not Google’s own Gemini. The bug was in a popular open-source web tool. If the attack had worked, hackers could have bypassed two-factor login, but still needed real usernames and passwords. Google found and stopped the attack, and told the software maker to fix the bug.

Experts have worried for years that AI could help attackers find bugs faster. Now, that fear is real. Other companies, like Anthropic, have also built AI that can find many hidden bugs. Some AI tools are so strong that only trusted groups get access. AI has already been used by state-backed hackers, for example from China, to look for weak spots.

Google says signs of AI-written code include too much extra explanation, which people usually don’t add. Former NSA experts agree this is a kind of AI “fingerprint.” Google warns this is just the start, and the real problem may be even bigger. Some people think AI will help make software safer, but for now, there are many weak systems that AI can help break.

In the comments, some are worried, saying AI is now a tool for attackers. Others think this was bound to happen and are surprised it took this long. Some are skeptical and want details from Google. A few point out that both attackers and defenders will use AI, leading to an arms race. Some ask if open-source tools are now bigger targets since their code is public. Others say AI can help defenders too, so it’s not all bad. Some call for stronger security rules for AI tools. Many agree that as AI gets better, quick patching and teamwork will be needed to stay safe.

Let’s look at a clever personal tech project. Someone used AI tools to build a system to track what noises wake them up at night in a noisy city. They used cheap microphones, a Raspberry Pi, and data from a sleep watch to match noise events with sleep interruptions. The system records short audio clips only when loud noises happen while the person is in bed. All data—sleep stages, heart rate, noise, and sensor events—are shown in a web app to help see exactly when and why sleep was disturbed.

AI helped build the code quickly, but did not listen to the sounds; the human still checks the audio. The system found that doors, dishes, and street noise were common causes. This helped the writer use sound panels and better insulation, and talk to flatmates to solve the problem. The setup is private and only records when needed. Future features could include smarter notifications and automatic sound grouping.

In the comments, readers liked the DIY approach and how quickly AI helped. Some wanted this kind of tool in consumer products. Others worried about privacy, but liked that the data stayed local. A few asked for open-source code, but the writer said it’s too rough. Some shared their own sleep and noise stories. There was debate about letting AI write and run code with device access, but some said it’s fine for personal use. Some suggested using machine learning for sound classification, but others warned that could be hard. Many liked the “measure before you fix” idea and said AI makes small, useful projects possible for individuals.

Now, some promising news in medicine. UCLA scientists have found a new drug that helps mice recover from stroke, working like physical rehab but as a pill. Usually, people who have a stroke lose some movement and rehab only helps a bit, with no drugs to fix brain damage. The team found that certain brain cells lose their rhythm after a stroke, making movement hard. Rehab brings this rhythm back, helping recovery.

The researchers tested two new drugs. One, called DDL-920, helped mice recover movement almost as well as real rehab. This is the first time a drug has copied the effects of rehab in the brain. But so far, it’s only tested in mice, and more studies are needed before it can help people.

In the comments, some were hopeful, saying this could help many stroke patients, especially those who can’t do hard rehab. Others were careful, noting that mouse results don’t always work in humans. Some worried about side effects and costs. There was technical discussion about brain rhythms and recovery. Some shared stories about friends or family needing new treatments. Overall, most agreed this is a big step for stroke recovery research.

Now, a look back at software history. Nullsoft, the company behind Winamp, started as a small team making a fast, simple MP3 player. Winamp was popular for its design, speed, and fun skins and plugins. AOL bought Nullsoft in 1999, giving the team more resources at first, but over time, big company rules slowed things down. AOL bosses wanted to add ads and change the software, and some team members left. In the end, Winamp lost its special culture and place as the top music player.

Many commenters remember using Winamp and loving its design. Some say AOL ruined it by trying to make it like other big-company products. Others point out that streaming made it hard for any MP3 player to stay popular. Some shared funny stories about using Winamp at parties. Others wish it could come back or are happy with today’s apps. The story is seen as a classic case of a small, fun tech team losing its spark after being bought by a big company.

Next, a big attack on Canonical—makers of Ubuntu—happened in April 2026. Canonical’s main websites and services went down for almost a day because of a strong denial-of-service attack. The attackers used a paid service called Beamed, which advertises ways to get past Cloudflare’s protection and is itself hosted by Cloudflare. Canonical put its most important servers—archive.ubuntu.com and security.ubuntu.com—behind Cloudflare to keep them online. Other sites stayed down longer.

The article looks into the companies and networks behind the attack tools, finding links to old privacy and activism groups. The network changed hands and countries, but stayed connected to similar services. At the same time, Canonical updated their server certificates, likely to move to Cloudflare. No ransom was paid, and there is no sign of a payment or demand.

In the comments, some feel Cloudflare is not doing enough to stop bad actors from using its service, making it look like they profit from both attackers and victims. Others say Cloudflare can’t police all users without risking privacy. Some see this as a big internet problem, where companies that protect against attacks also help attackers hide. Others argue Canonical had no choice but to pay for Cloudflare’s help. Several compare the situation to old “protection rackets.” Some feel Cloudflare is caught in the middle, just a tool, and the real problem is the people running attack services. Others think Canonical should have been better prepared. Some warn that more bans could lead to too much control or censorship. Most agree the whole situation is troubling and the line between attacker and protector is unclear.

Now, onto something more fun: Ratty, a new terminal emulator that uses your computer’s graphics card to show 3D graphics right inside the terminal. The app even has a spinning rat as the cursor, adding a playful touch. Ratty lets you see 3D models and animations next to regular command-line output. This can help developers with visualizations, previews, or even games, all inside the terminal. Ratty is open source and available for download.

In the comments, many are excited about the creative features, especially the spinning rat cursor. Some see use cases for debugging, data visualization, or learning graphics programming. Others wonder if it’s more of a cool demo than a daily tool. A few worry about performance and compatibility, since using the GPU may drain laptop batteries or not work on older systems. Some point out this is a big step away from the simple, text-based UNIX philosophy. Others compare Ratty to old-school demos that pushed the limits of terminals. The community is split between those who see it as the future for developer tools and those who prefer classic simplicity.

Finally, Google has changed how you register for a new Gmail account. Now, you must scan a QR code and send a text message from your phone to Google, instead of just getting a code by SMS. This step is meant to stop fake accounts and make things more secure, but it also makes it harder for people who want to stay anonymous or use temporary phone numbers. The process blocks common tricks like using online SMS services, since Google now needs to see the message come from your real phone number.

Some commenters say this will only stop regular users, not skilled hackers or sellers of ready-made accounts. Others worry that privacy-focused users now have no easy way to get a Google account without sharing their real number. Most agree this is a big step against staying anonymous. Some hope for a workaround, but for now, making an anonymous Gmail account is much more difficult.
