## Anatomy of the .claude/ folder

- 原文链接: [Anatomy of the .claude/ folder](https://blog.dailydoseofds.com/p/anatomy-of-the-claude-folder)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47543139)

The article explains what the `.claude` folder is and why it matters for people using Claude Code, an AI tool for coding. Many users see this folder in their projects but do not know what each file inside does.

The `.claude` folder is like a control center for Claude. It stores rules, commands, and memory about your project. There are two main `.claude` folders: one in the project (shared by the team), and one in your home directory (just for you). The project folder goes into git and sets team-wide rules, while the home folder saves your personal preferences and history.

The most important file is `CLAUDE.md`. This file tells Claude how to behave in your project. You can set coding rules, commands, and style preferences here. You can also have personal rules in `CLAUDE.local.md`, which are ignored by git and only affect your own setup. For big teams, you can split rules into smaller files inside `.claude/rules/`, and even make some rules apply only to certain folders.

You can add custom slash commands with `.claude/commands/`. Each file here becomes a command you can run in Claude Code, and you can pass arguments to them. Personal commands go in your home folder, and team commands stay in the project.

Skills are like commands, but they run automatically when Claude sees a matching task. Each skill is a folder with instructions and extra files. You can also create agents—special roles that do focused tasks, like code review, with their own settings and rules.

`settings.json` in `.claude/` controls what Claude can and cannot do, like which shell commands are safe or blocked. There is also a personal override file, `settings.local.json`, for your own exceptions.

The article suggests starting simple: use `/init` to make a starter `CLAUDE.md`, set up allow/deny rules, and add your most-used commands. As your project grows, split rules into smaller files and add more tools as needed.

Commenters on Hacker News had mixed reactions. Some liked the clear structure and found the idea of project and personal configuration helpful. Others worried about the complexity growing too fast, especially for small teams. A few users wondered about security risks, like running commands or saving sensitive data. Some praised the idea of path-scoped rules, saying it makes big projects easier to manage. Others asked if the `.claude` folder could become a mess over time if not cleaned up. People also compared this setup to other AI tools (like Copilot), debating if giving so much control is good or if it makes things harder. Some said the guide helped them understand how to get more value from Claude, while others felt the setup is too much for simple projects.

The system shows you real-time output, errors, and lets you use over 48 electronic parts—like LEDs, sensors, and displays—by dragging them onto a canvas. You can run Velxio online, with Docker, or set it up manually on your own computer. It uses modern tech like React for the frontend, FastAPI for the backend, and emulation tools like QEMU and avr8js. Velxio is open-source for personal and educational use, but you need a paid license for closed-source or business projects.

## Telnyx package compromised on PyPI

- 原文链接: [Telnyx package compromised on PyPI](https://telnyx.com/resources/telnyx-python-sdk-supply-chain-security-notice-march-2026)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47540388)

The Telnyx Python SDK package on PyPI was hit by a supply chain attack on March 27, 2026. Two fake versions (4.87.1 and 4.87.2) with bad code were uploaded and stayed online for about six hours before removal.

The attack only affected the Python SDK on PyPI. The main Telnyx platform and APIs were safe. You were at risk if you installed or updated the telnyx package during that time, or if another package pulled it in as a dependency. If you used version 4.87.0 or older, or did not install the package that day, you are safe. The SDK itself is only a wrapper for public APIs; it has no special access to Telnyx systems.

If you want to check if you’re affected, run `pip show telnyx` in your terminal. If the version is 4.87.1 or 4.87.2, treat your system as compromised. The advice is to downgrade to 4.87.0, change all your secrets (API keys, passwords, cloud tokens, SSH keys), and check if your system connected to the attacker’s server (83.142.209.203:8080). Also, check your CI/CD pipelines or Docker builds if they might have used the bad versions.

The malicious versions were removed quickly. Telnyx is investigating how their publishing credentials were stolen. This is part of a bigger wave of attacks—other Python packages like Trivy and LiteLLM were also hit recently.

From the Hacker News comments, some people are worried about the growing number of supply chain attacks on PyPI and other package registries. Developers say it’s hard to trust open source packages now, especially with transitive dependencies. Many suggest pinning package versions and using tools to check for changes or known bad packages. Some blame weak security around publishing keys. Others point out that even big companies and security-focused tools have been compromised. A few users wish PyPI and NPM would add more checks or require two-factor authentication for uploads. There’s also talk about using private package mirrors, but some feel that’s too much work for small projects. Overall, many agree that supply chain risks are here to stay and everyone needs to stay alert.

## Installing a Let's Encrypt TLS certificate on a Brother printer with Certbot

- 原文链接: [Installing a Let's Encrypt TLS certificate on a Brother printer with Certbot](https://owltec.ca/Other/Installing+a+Let%27s+Encrypt+TLS+certificate+on+a+Brother+printer+automatically+with+Certbot+(%26+Cloudflare))
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47542644)

This article explains how to set up a real HTTPS certificate from Let’s Encrypt on a Brother printer using Certbot and Cloudflare. The writer wanted to make their printer’s web page use a trusted, safe connection, just like websites do, but without a lot of manual work.

They use Cloudflare to manage their DNS records, and NextDNS to help with local network names. Their homelab uses Tailscale, but the Brother printer can’t run Tailscale, so they use DNS tricks to make the printer reachable only from their network. They want to avoid using a reverse proxy for the printer because it adds complexity and could break if the proxy server is down. They also want to use IPP printing over HTTPS, not HTTP.

The setup needs a Brother printer, a computer (they use a Mac Mini) to run scripts, and DNS records managed by Cloudflare. Certbot is used to get certificates, and a tool called “Brother Cert” uploads the certificate to the printer. The process is automated using Cronicle, but you can use any scheduler or run scripts by hand.

The script does two things: First, it asks Let’s Encrypt for a certificate by proving control over the printer’s DNS name using Cloudflare. Brother printers need an RSA-2048 type certificate, so the script requests that. Second, it takes the new certificate, uses the Brother Cert tool to convert and upload it to the printer (switching to HTTP if the old certificate is expired), and then reboots the printer to make the new certificate active.

Credentials for Cloudflare and the printer are stored in separate files for safety and easy updates. The script includes steps for setting up the environment, installing needed software, and explains what variables to change for your own setup. The writer shares the whole Bash script, which includes checks, error messages, and details for both certificate creation and deployment.

They also show how to set up Cronicle to run the script every month, keeping the certificate up to date without manual work. The end result is a Brother printer with a real, valid HTTPS certificate, making the web portal safe to use from browsers and client devices.

In the Hacker News comments, many readers find the solution clever and practical. Some point out that printer security is often ignored, and using real certificates is a good idea. Others note that many printers make it hard to update certificates, so this script is helpful. Some worry about storing sensitive credentials on disk, though the article does warn not to share these files and gives tips on using permissions.

A few users ask if this method works for other printer brands or older Brother models; the answer depends on the printer’s firmware and support for custom certificates. Some suggest using internal certificate authorities instead of public ones, but the author wanted trusted certificates for all devices, not just ones with their own CA installed. There is also a discussion about the risks of exposing printers to the internet, with most agreeing it’s best to keep devices like printers only available inside the local network.

Overall, readers like the automation and clear steps, and some share how they might adapt the script for their own homelabs or office printers. A few mention that keeping printers secure is hard, so tools like this are very welcome.

## Explore the Hidden World of Sand

- 原文链接: [Explore the Hidden World of Sand](https://magnifiedsand.com/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47493677)

This article is about looking at sand grains under a microscope and discovering how different and beautiful each grain is. It shows that sand is not just tiny rocks, but a mix of minerals, shells, bits of marine life, and more, each telling a story about its origin.

Sand grains are between 0.02 and 2 millimeters in size. They form when rocks, shells, and sea life break down over a long time. Each grain is unique, like a snowflake, and there are about 5 sextillion grains of sand on Earth. The sand from each beach is different because of the local rocks, volcanic activity, and sea animals. For example, sand from volcanic places like Iceland is dark, while sand from tropical beaches can be white from shells and coral. The article gives examples from beaches all over the world—like black sand in Iceland, soft white sand from Zanzibar, or shell-filled sand in Malta. Some beaches have sand mostly from rocks, others from sea creatures, or a mix of both. There are also interactive maps to see sand samples in 3D.

The article also explains that sand shows the history of its area. Plate tectonics, volcanoes, and the sea all help make the sand we see. Some places have sand from very old rocks, others from new volcanic activity, and some from coral reefs. Looking at sand under a microscope shows all the colors and shapes that we can’t see with our eyes.

In the comments, many people are surprised by how beautiful and colorful sand grains are when seen up close. Some say they never thought sand could be so interesting or wonder why they didn’t learn this in school. Others enjoy seeing the differences between beaches around the world. A few users share their own stories about collecting sand on their travels and noticing the colors and textures.
