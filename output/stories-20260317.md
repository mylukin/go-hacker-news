## A Decade of Slug

- 原文链接: [A Decade of Slug](https://terathon.com/blog/decade-slug.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47416736)

Eric Lengyel wrote about the Slug Algorithm, a method for rendering fonts directly from Bézier curves on the GPU, which he first developed ten years ago. This algorithm has been used in many industries, from video games to medical equipment, and by big names like Activision, Adobe, and Ubisoft.

The main idea behind Slug is to render text and vector graphics straight from curve data, not from pre-made images or textures. This gives smooth, high-quality text at any size or angle and is fast enough for use in games. The algorithm avoids common problems, like floating-point errors that can cause glitches or missing pixels. Key parts, like the root eligibility check and winding number calculation, have stayed the same since 2017. Some features, like “band split optimization,” were removed because they made the code more complex and didn’t always improve performance. Supersampling, which helped with small text, was also dropped, as a new technique made it unnecessary. Multi-color emoji are now handled by stacking glyphs, which is faster and simpler.

The biggest recent upgrade is “dynamic dilation.” Before this, users had to guess how much to expand a glyph’s boundary to avoid visual errors. Now, the algorithm calculates the perfect amount automatically for each glyph, making the results cleaner and faster with no wasted GPU work. Eric even explained the math behind dynamic dilation, showing how the system moves each vertex just enough to cover half a pixel in the final image. This also involves using a Jacobian matrix to keep everything lined up correctly.

Eric announced that he is giving up his patent on the Slug algorithm, making it free for anyone to use forever. He has also shared reference code for the shaders on GitHub under the MIT license, so developers can use or learn from the actual code used in Slug.

In the comments, many people praised Eric for his work and for releasing the patent. Some said this will help open-source projects and small developers who could not afford a license before. Others were excited to try the Slug algorithm in their own projects, especially after seeing the clear, real-world benefits. A few users discussed the technical details, such as how dynamic dilation works and why rendering from curves is better than using texture maps. Some mentioned that this move could push other companies to release their patents too. There were also questions about performance on newer GPUs, and some users shared their own experiences with font rendering problems. A few people wondered if this will help improve emoji support and multi-language text in games and apps. Overall, most comments were very positive, saying this is a big win for the graphics and developer community.

## Python 3.15's JIT is now back on track

- 原文链接: [Python 3.15's JIT is now back on track](https://fidget-spinner.github.io/posts/jit-on-track.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47416486)

Python 3.15 will include a new Just-In-Time compiler (JIT) that is now showing clear speed improvements. The article explains that, after a long period of slow progress and doubts, the team reached their modest speed goals ahead of schedule, with around 11-12% faster performance on macOS and 5-6% on Linux.

Earlier JIT versions in Python 3.13 and 3.14 were often slower than the normal interpreter, and team funding was lost, making the project’s future uncertain. The main changes came from growing the team, breaking complex work into smaller tasks, and giving clear instructions for new contributors. The team also focused on making sure more people could work on all parts of the JIT, not just a few experts. A big technical improvement was switching to a trace recording approach, which means the JIT records and optimizes repeated code paths. This change was made by accident, but it paid off, especially after another lucky mistake led to a faster system called dual dispatch. Another useful optimization was removing extra steps (branches) for reference counting, which saved time and helped teach newcomers about the JIT. The team’s work was helped by daily performance tests on a few computers, which quickly showed if new changes made things better or worse. The author also thanks people outside the team, like PyPy contributors, for sharing knowledge.

One commenter is impressed by the focus on making the project open to new contributors, saying this helps the whole Python community. Another points out that luck, teamwork, and learning from others are often as important as technical skill. ## Get Shit Done: A Meta-Prompting, Context Engineering and Spec-Driven Dev System

- 原文链接: [Get Shit Done: A Meta-Prompting, Context Engineering and Spec-Driven Dev System](https://github.com/gsd-build/get-shit-done)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47417804)

This article is about a tool called "Get Shit Done" (GSD), which helps developers use AI tools like Claude Code, Copilot, and others to write and manage code projects. GSD is made for people who want to describe what they want and have the AI build it, without needing all the big-company planning steps.

GSD fights a problem called "context rot." This happens when the AI forgets or mixes up details as a project gets bigger. GSD keeps everything clear by using files like PROJECT.md and ROADMAP.md, and by making sure each development step has the right information. The workflow is broken into phases: you describe your project, answer questions, approve the plan, and let the AI work in clear steps. Each phase includes research, planning, execution, and testing. Tasks are grouped by dependencies, so independent work runs in parallel, and related tasks wait for each other.

You can run GSD in "quick mode" for fast, simple tasks, or go through the full process for big features. Every task gets its own git commit, making it easy to track and fix problems. The system is modular—you can add or remove phases, start new milestones, or adjust plans without starting over. GSD works with different AI models and lets you set how much automation you want. For security, you can block the AI from reading secret files.

There are many commands, like starting a new project, discussing a phase, planning, executing, and verifying work. GSD also helps with UI design, codebase mapping, debugging, and keeping notes or todos. Settings let you tweak the process, pick which AI models to use, and choose how git branches are handled. Installation is simple, and GSD works on Mac, Windows, and Linux.

They liked that it keeps things simple and doesn’t force people into enterprise-style processes. Some users mentioned they’ve tried similar tools but found GSD easier and more effective. Others wondered about security, asking if giving so much control to the AI could be risky, especially with sensitive files. A few people were concerned about the "skip permissions" mode, saying it could be dangerous if not used carefully.

## Edge.js: Run Node apps inside a WebAssembly sandbox

- 原文链接: [Edge.js: Run Node apps inside a WebAssembly sandbox](https://wasmer.io/posts/edgejs-safe-nodejs-using-wasm-sandbox)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47416081)

Edge.js is a new tool that lets you run Node.js apps inside a WebAssembly sandbox, making them safer and faster to start than using containers like Docker. The project aims to keep full Node.js compatibility, but adds strong sandboxing by isolating risky operations using WebAssembly and WASIX.

The main idea is to let developers run their current Node.js projects, including native modules, without any code changes. Edge.js does this by only sandboxing the unsafe parts—like system calls and native code—while letting JavaScript run at full speed in a regular JS engine. You can choose from different JS engines, like V8 or QuickJS. When you use the “--safe” mode, the app is strongly sandboxed.

Edge.js was made because containers are often too slow and heavy for edge computing, where you need to start many apps quickly. Earlier solutions like WinterJS had problems with speed and app compatibility. The Edge.js team tried several ways to make Node.js secure and fast on the edge, but settled on using the NAPI interface and WASIX for the best mix of speed and safety.

Edge.js is almost fully compatible with Node.js v24, even passing more tests than Bun or Deno for many core modules. It works with popular frameworks like Next.js and Astro out of the box. The performance is about 5–20% slower than native Node.js, or 30% slower when fully sandboxed, but the team is working to improve this. Startup times aren’t as fast as they want yet, but there’s a plan to fix this.

The project was built quickly with the help of AI tools like GPT and Codex, which made it possible for a small team to move fast. The blog encourages people to try Edge.js and give feedback.

In the comments, some users are excited about running Node.js securely without Docker, saying this could help with serverless scaling and multi-tenancy. Others like that Edge.js is almost 100% compatible with Node, which is rare for alternative runtimes. Some are interested in using it for running untrusted code, such as plugins or user scripts.

However, a few commenters worry about the actual security of the WebAssembly sandbox, asking if it’s truly safe against real-world attacks. Some think the performance hit is too big for some use cases, especially if startup time or throughput is critical. Others note that the project is young, so there may be bugs or missing features.

There are also comments comparing Edge.js to Bun and Deno. Some say Edge.js has better module compatibility, but Bun is faster for some tasks. A few people wish Edge.js had more documentation or real-world benchmarks. Some suggest that container-based isolation is still needed for very sensitive data.

Overall, the response is positive, with many users interested in trying Edge.js. People see it as a promising step for edge computing and safer Node.js, but want to see how it works in practice before fully trusting or switching to it.

## Toward automated verification of unreviewed AI-generated code

- 原文链接: [Toward automated verification of unreviewed AI-generated code](https://peterlavigne.com/writing/verifying-ai-generated-code)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47397367)

This article looks at how to trust AI-generated code without reading it line by line. The author tried using automated tests and checks to confirm the code is correct, instead of always reviewing it by hand.

He started with a simple coding problem and used a coding agent to generate a solution. Then, he used four main checks: property-based tests, mutation testing, making sure there are no side effects, and enforcing type checks and code style rules. Property-based testing means checking the code works with many random inputs, not just a few examples. Mutation testing changes parts of the code to see if the tests still catch errors. No side effects mean the code does not change things outside its own scope, like writing files by accident. Type and style checks are common in Python, but might not be needed in all languages.

The author found these checks made him trust the code, even without reading it. He also noticed that if the code passes all these tests, there is only a small chance that it is still wrong. He worried at first that the code would be hard to maintain or read, but now thinks it is okay to treat AI code like compiled code—something we do not read, but just run if it passes all checks. He admits that setting up these tests takes time, but thinks this work could get easier as tools get better.

The article also mentions related work. Formal verification is another way to check code, but it takes a lot of effort. Some projects use big test suites to check AI code, and sometimes AI models copy test data, but property-based tests help stop this. There are even teams building “Software Factories” where no humans review code at all.

In the comments, some readers agree that strong, automated tests can help trust code you did not write. They like property-based and mutation testing, saying these make it harder for bad code to slip through. Others worry that tests and checks can miss deep bugs, especially if the tests themselves are not good. A few say that code readability and maintainability still matter, because humans may need to change or debug the code later. Some point out that setting up all these checks is too much work for small projects, but may be worth it for large or safety-critical code. One reader says that even with many tests, you should still be careful about security risks from AI code. Another thinks that as AI gets better, more tools will appear to make all this easier and faster. A few are excited about the idea of “Software Factories,” but others feel worried about removing humans from the loop. Some also wonder if these methods can fully replace human review, or if they are just good enough for now.

## Node.js needs a virtual file system

- 原文链接: [Node.js needs a virtual file system](https://blog.platformatic.dev/why-nodejs-needs-a-virtual-file-system)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47413195)

This article talks about why Node.js should have a virtual file system (VFS). The main idea is that Node.js apps often need to work with files, but the current file system is not flexible enough for some use cases.

The author explains that with a virtual file system, developers could make files and folders in memory, not just on disk. This can help when you want to run code in a safer way, or when you want to test something without changing real files. For example, if you use tools like Webpack or Vite, they sometimes need to make fake files or read files that do not exist on the disk. Right now, they have to use tricks to do this, which is hard and sometimes slow.

A virtual file system could also help with security. If you run user code, you could limit what files it can see or change. This makes it harder for hackers to do something bad. The author says other systems, like .NET and some JavaScript runtimes, already have something like this.

The article gives examples of how a virtual file system could make testing and development faster. It could also help if you want to bundle code or run code in the browser, because you could use the same file logic everywhere. The author says adding a VFS to Node.js would not break old code. It could be an extra feature that you use only when you want.

In the comments, some people agree that a virtual file system would be useful, especially for testing and security. They say it would make it easier to sandbox code or to work with files in the cloud.

Other commenters think it might add too much complexity to Node.js. They worry that it could be hard to maintain or could confuse new developers.

Some users point out that you can already use in-memory file systems with libraries, so maybe this is not needed in Node.js core. Others reply that having it built-in would make things easier and more consistent.

A few people share stories about bugs or problems they had because Node.js does not support virtual file systems well. Some mention that other platforms do this better, and Node.js should learn from them.

Overall, the comment section shows that some developers really want this feature, while others are not sure if it is worth the cost. The discussion is lively, with real-world examples and different opinions about how Node.js should grow.
