Now, let’s talk about running Google’s Gemma 4 large language model on your own computer with LM Studio’s new command-line interface. You can avoid cloud AI problems like costs, privacy, and slow speeds by using a strong AI model locally. Gemma 4 uses a “mixture-of-experts” design: it has 26 billion parameters, but only 4 billion are active at a time. This gives high-quality results but does not need a huge computer. The author runs Gemma 4 on a MacBook Pro with 48 GB memory and gets about 51 tokens per second, which is fast for local use.

Google released several Gemma 4 models for different hardware. The 26B-A4B model is a good balance between speed and memory use. It does well on AI benchmarks, almost matching much larger models but using less memory and running faster. You also get features like vision (analyzing screenshots), long context windows (up to 256,000 tokens), and tool calling.

LM Studio version 0.4.0 brings a headless “daemon” so you can run everything from the terminal. You install LM Studio, start the daemon, download Gemma 4, and chat with it using commands. The CLI shows speed and memory use, and you can manage models easily. There are scripts and tips for planning memory use, especially on Macs, where the CPU and GPU share memory. You can control how long a model stays loaded, how many tasks run in parallel, and monitor everything with logs.

There are advanced features like flash attention (which saves memory) and speculative decoding (which speeds up dense models), but speculative decoding does not work well with Gemma 4’s MoE design yet. The desktop app is still good for vision features.

A highlight is the Anthropic-compatible API endpoint. This lets you use Claude Code (a coding assistant) with your local Gemma 4 model instead of Anthropic’s cloud. This keeps your code private and saves cloud costs. Some features are slower or missing compared to the cloud, but for most coding and review tasks, it works well.

Users discuss the mixture-of-experts model, saying it is smart to use only part of the model for each answer. Next, let’s talk about a fun tool called Caveman, a plugin for Claude Code and Codex that makes AI write in short, simple “caveman” style to save tokens. The main goal is to cut the number of words in AI responses by up to 75%, while keeping the important technical meaning. The article shows examples: normal AI answers are long and polite, but Caveman answers are short and direct. For example, instead of a full explanation, Caveman might say “New object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo.” This style removes filler words, pleasantries, and extra language, but keeps technical terms and code the same. Benchmarks show big savings, and on average, a 65% reduction across many coding tasks. Caveman only changes the output, not how the AI thinks, so technical accuracy stays the same. The plugin is easy to install and works with simple commands like “/caveman” or “talk like caveman.” You can turn it off at any time.

Science supports this style. A research paper found that short, direct answers can even make AI more accurate on some tasks. Using Caveman is not just about saving money; short answers are easier and faster to read, and they make the AI more fun.

Others say it helps focus on the real answer, without all the fluff. Some think it is good for technical work, but maybe not for customer service or teaching, where more polite language is needed. Now, let’s look at building a fast interpreter in Rust, using a new feature called the “become” keyword. The interpreter is for Uxn, a simple virtual CPU. The author wanted to make the interpreter as fast as possible, even faster than hand-written assembly. The Uxn CPU is basic: it has two small stacks, some RAM, and a program counter. Usually, you write an interpreter that reads each instruction, runs code for that instruction, and repeats. In assembly, the author made a “threaded code” interpreter, where each instruction jumps directly to the next one. This is fast, but the code is large and unsafe.

In Rust, the author tried the same idea, but the stack kept growing. The new “become” keyword in nightly Rust solves this: it replaces the current function with the next, so the stack does not grow. The author built a macro to reduce repeated code for each instruction.

Performance was very good, especially on ARM (Apple M1): the tail-call interpreter was faster than assembly and much faster than the basic VM. On x86, the tail-call interpreter was still faster than the basic VM, but not as fast as the hand-written assembly. The author found that the generated machine code was not as good as the assembly. Testing in WebAssembly, tail calls were much slower than the normal VM in browsers and in wasmtime.

In the comments, some readers were excited about the new “become” keyword and its uses. Others said that writing interpreters in this way is an old idea, known from languages like Forth or from threaded code in general. Some talked about the macro, calling it “ugly but effective,” and asked if Rust could make this easier in the future. Some people worried about using nightly Rust and unstable features in production. Others discussed the differences in performance between ARM and x86, and hope the Rust compiler and LLVM backend improve. Some noted that WebAssembly is not yet good at tail calls. Some shared experiences with similar interpreters in C or other languages. There were questions about the safety of this approach, but the author said the code is “safe Rust.” Others wondered if this style could help with other virtual machines or emulators. People liked the deep benchmarks and code examples, and some asked for more details about the Uxn system or how to make the x86 version faster. The overall mood was positive, with readers interested in how nightly Rust is evolving.

Finally, there is an article about building your own coding AI, called "nanocode," for about $200 using JAX and TPUs. Nanocode is inspired by Anthropic's Claude Code and Karpathy’s "nanochat," but focuses on agentic coding, meaning it can use tools and interact with files and code on your computer. The author explains how to use open datasets to train the model, and that nanocode is optimized for TPUs but should also run on NVIDIA GPUs. There are two main versions: a smaller one (477M parameters) and a bigger one (1.3B parameters), with training times and costs shared for each. Nanocode uses special tokens and templates so the model knows when a user, assistant, or tool is speaking, and can perform actions like reading, editing, or searching files by calling simple tools.

Much of the work is about teaching the AI to use these tools. The training process uses many coding examples and follows the “Constitutional AI” method, aligning the AI’s behavior with a written SOUL document. Training includes supervised fine-tuning with feedback loops and direct preference optimization. The model is tested and compared against GPT-2 and nanochat, showing that nanocode is strong at coding tasks. The author explains how to interact with nanocode using a command-line tool and encourages readers to modify the codebase and try new experiments.
