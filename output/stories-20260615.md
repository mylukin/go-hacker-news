## A backdoor in a LinkedIn job offer

- 原文链接: [A backdoor in a LinkedIn job offer](https://roman.pt/posts/linkedin-backdoor/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48546294)

A software engineer got a message on LinkedIn from someone claiming to be a recruiter for a crypto startup. The recruiter asked him to review a GitHub codebase and to look into a problem with outdated Node modules. 

The engineer felt something was strange, so instead of running the code on his computer, he used a safe, temporary server and special tools to only read the files. He quickly found a suspicious file in the codebase. Inside this file, the code quietly built a web address and then ran any code sent from a server at that address. This dangerous part was hidden between lots of comments and looked like a normal test file.

The backdoor was set up so that just running `npm install` (a basic step to set up many Node.js projects) would also run the harmful code. The recruiter’s request to "check out the deprecated Node modules" was really just a trick to get him to run `npm install` and trigger the backdoor.

The GitHub code was pretending to be written by a real developer (someone with a real LinkedIn and GitHub profile), but this developer had nothing to do with it. The recruiter’s LinkedIn profile also belonged to a real person—a well-known arts journalist with no tech background. When the engineer asked for help, the "recruiter" suddenly acted like an expert in Node.js, which was suspicious.

The engineer realized this scam could happen to anyone, especially if they were busy or not careful. He found that using a read-only tool made it much easier to spot the dangerous code. He reported the fake repo and recruiter profile, but they were still online when he wrote the article.

In the comments, many people said this kind of scam is becoming more common, and that attackers are getting smarter about hiding malicious code in open source projects. Some readers shared similar stories, saying they had also been sent suspicious job offers with code to review. Others pointed out that many developers would not notice the backdoor, especially if it looks like a normal part of the project. 

A few commenters discussed ways to protect themselves, like running code in a virtual machine or using read-only tools, just as the article’s author did. Some said it’s important to always check the commit history and be wary when a recruiter asks you to run code from a stranger. Others worried that reporting scams does not always remove them quickly, and that more needs to be done by GitHub and LinkedIn to catch these tricks. Finally, some people praised the author for sharing his story, saying it was a good reminder for everyone to stay careful online.

## Iroh 1.0

- 原文链接: [Iroh 1.0](https://www.iroh.computer/blog/v1)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48542480)

Iroh 1.0 is a new networking tool that lets you connect devices using keys instead of IP addresses. The team thinks this is a better and safer way for devices to find and talk to each other on the internet.

Instead of using IPs that can change, be blocked, or are hard to control, Iroh uses keys you create and manage. These keys stay the same, even as your device moves or switches networks. With Iroh, your device can always be reached, even behind firewalls or in private networks. The software is open source and already runs on millions of devices, being used for things like video streaming, secure chats, sending files, and even playing games.

Iroh 1.0 is the first stable version, after many updates and improvements. The team built it using open standards, and added support for advanced networking features like QUIC multipath and NAT traversal. This helps Iroh make fast, direct connections, even if devices are on different networks. You can connect to local devices without internet, use Iroh in browsers, and even add custom ways to connect, like Bluetooth, WiFi, or Tor.

The key is central in Iroh—it secures your connection, helps set up identity and permissions, and acts as the address to reach your device. This makes connections safer and easier. Most data sent with Iroh goes straight between devices, making connections faster and cheaper. Iroh now supports many programming languages, like Rust, Python, Node.js, Swift, and Kotlin, so developers can use it in many types of apps.

The Iroh team promises that version 1.0 will stay stable, and future updates won’t break connections between devices running this version. They also have a support plan for older versions and public relays to help devices connect when needed.

On Hacker News, many users liked the idea of using keys instead of IP addresses, saying it could fix many old problems with networking and NATs. Some were excited about using Iroh for peer-to-peer apps, file sharing, and games, since it could make these easier to build and more reliable. Others praised the support for many platforms and languages, which makes Iroh useful in more projects.

A few users had questions about security, asking if using keys makes it harder for attackers or if it could cause privacy problems. Some wondered how well Iroh works with firewalls and strict networks, or if it needs special relays all the time. There were also comments about performance, with users interested in how direct connections really are in real-world cases.

A couple of people compared Iroh to older projects like Tor or ZeroTier, thinking about what’s different or better here. Some shared hope that Iroh could help make the internet less centralized and more private. A few were curious about how easy it is to set up and if it’s ready for big, commercial use. Overall, the reaction was mostly positive, with many developers eager to try Iroh in their own projects.

This article explains how the writer built an AI-powered development setup at home to manage their own servers and services. They use a tool called OpenCode Web UI, which connects with Git to help make changes and updates easier and safer.

The writer runs many services in Docker containers. Before, updating these services took a lot of time: they had to read release notes, check for problems, run updates, and manually test everything. Now, with AI, they can quickly read summaries and get help adding health checks, saving hours of work. The writer first tried AI tools like Claude Code but found them limiting because of token and usage rules. So they switched to OpenCode, which is vendor-independent, easy to use, and has a good web interface—even on phones.

OpenCode runs on a virtual machine (VM) on their TrueNAS server. It has its own user and SSH keys for security. The AI can suggest and write code, but it cannot push changes directly to the live system. Instead, it creates pull requests (PRs), which the writer reviews and merges. This way, no unapproved code gets deployed, and the risk is low because the AI only operates inside its own VM.

The workflow is simple: plan new features or fixes with OpenCode, test and review changes, let OpenCode push the changes to a feature branch, open a PR, merge it after a manual check, and then let GitOps (like Arcane) handle deployment. The writer moved their Docker setups to git-backed storage, which works well with OpenCode. They can now manage updates or network changes from any device, including their phone. The only problem is with CI (Continuous Integration) feedback—Forgejo, their git server, does not easily share job logs with AI for fast debugging.

In the comments, many people liked the clear workflow and the mix of AI with manual checks. Some thought the extra security—like stopping the AI from direct deploys—was smart. Others worried that giving a coding AI root access, even in a VM, could be risky if the AI ever made a big mistake, but most agreed the setup was good for a home lab. A few readers asked about the tools, like how OpenCode compares to GitHub Copilot or other AI assistants. Some liked that the whole thing ran on self-hosted hardware and open tools, not big cloud services.

There were comments about the future, with readers saying this could be used at work, not just at home. Some people wanted more details on the hardware and network setup. Others pointed out the missing CI feedback as a real problem, suggesting possible workarounds or improvements. A few worried about the complexity of using AI for simple home tasks, but most felt the benefits were worth it. Finally, some readers were just surprised how much could now be automated at home, thanks to new AI tools.

## Factoring "short-sleeve" RSA keys with polynomials

- 原文链接: [Factoring "short-sleeve" RSA keys with polynomials](https://blog.trailofbits.com/2026/06/12/factoring-short-sleeve-rsa-keys-with-polynomials/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48503509)

This article explains a new way to break some weak RSA keys called “short-sleeve” keys by using polynomials. These weak keys appear when the random numbers used to make them are not truly random, so the bits are mostly zeros, creating a regular pattern.

The authors found hundreds of real-world RSA keys with these patterns by scanning public data. They identified two main patterns: one’s cause is unknown, but the other comes from a bug in the CompleteFTP software used for generating cryptographic keys between 2016 and 2023. The bug made big numbers with lots of zeros in regular places. This problem also affected DSA keys. The CompleteFTP team fixed the bug and made tools to help users check and replace bad keys.

To find these weak keys, the authors used the badkeys project, which collects and checks public cryptographic keys for problems. They noticed that in affected keys, the numbers (called moduli) have repeated zero blocks. By looking at these numbers as polynomials, factoring becomes much easier than normal—usually, factoring RSA numbers is very hard, but with these patterns, it’s simple.

The article explains how the software bug caused this problem. The code was supposed to fill big numbers with random data, but because of a mismatch, most bits were zero. The authors reverse-engineered older versions of the software to confirm this, and they found similar issues in DSA key generation. Later versions switched to standard libraries, which fixed the problem.

The article also describes how the number of affected servers grew while the bug was active, but then stopped growing after the fix, though many servers still use old, weak keys. The CompleteFTP team responded well, helping customers fix the problem and updating their own software.

The authors suggest there could be more keys like this in the wild, as similar bugs may exist elsewhere. They highlight how real-world cryptographic failures can inspire better attacks and, in turn, help make systems safer.

In the top Hacker News comments, readers had mixed reactions. Some praised the article’s clear explanation of a complex topic and liked the new attack idea. Others worried about how many systems might still use weak keys, especially if admins don’t regularly regenerate keys after software updates. A few pointed out that using trusted, standard libraries is safer than writing custom cryptography code, and this case proves it. Some commenters discussed the technical details, expressing interest in the polynomial factoring trick and asking for simple code samples. A few mentioned that software should warn users when keys are made with outdated or buggy code, not just after problems are found. There was concern about old or forgotten servers still running vulnerable keys, which could be a risk for years. Some people wondered if pattern 1 might be from another, still-unfixed bug. Others were glad the disclosure process was handled well by the company. Finally, a few shared stories about finding similar cryptographic bugs in their own work, saying this kind of research is important for the whole community.
