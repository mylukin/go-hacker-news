Our next story covers a new project called Manyana, which tries to make version control better by using something called CRDTs—Conflict-Free Replicated Data Types. The author explains that CRDTs are good because they always merge changes without errors, but they have not been used much in version control because of user experience problems.

With Manyana, if two people change the same part of a file, the system shows exactly what each person did—not just a big, confusing mix. For example, if one person deletes a function and another edits it, Manyana will show who did what, instead of making you compare two big blocks of code. This makes it easier to understand and solve conflicts. CRDTs also make sure that merges never fail, and the result is always the same no matter the order. This is helpful when many people work on the same file at once. The whole history is saved in the file structure, so you don’t need to look for a common ancestor when merging. Rebasing can keep all history, not just rewrite it, which makes tracking changes easier.

Manyana is just a demo for now. It’s written in Python, works on single files, and is open source. It does not yet have features like cherry-picking or local undo.

In the comments, some people are excited about using CRDTs in version control. They think this could fix old problems with Git—like confusing conflicts and merges. Others point out that CRDTs have been tried before but did not catch on, maybe because they are hard to understand or because developers are used to Git. Some users worry about performance with big projects or binary files, while others think the better conflict view is worth it. There are questions about how this will work for big teams and complex projects. Some say the demo is simple but shows the idea works. Others think better user tools are needed before people will switch from Git. Some hope Manyana will push existing tools to improve, while others are not sure if companies will want to switch. Most agree that version control needs better ways to show and solve conflicts, and Manyana is a good step forward.

Now, let’s go back in time to the story of RollerCoaster Tycoon, a famous game known for being fast and smooth, even on old computers from 1999. The main reason is that Chris Sawyer, the creator, wrote almost all of the game in Assembly, a low-level language that gives lots of control over how code runs.

The game’s code was carefully optimized. Different types of money in the game use different data sizes to save memory and make calculations faster. For example, the whole park’s value uses four bytes, while the price of a shop item uses only one byte. The code also uses bit shifting instead of normal multiplication and division, because it’s much faster for computers. For example, multiplying by four is done by shifting bits to the left, which is quicker than using multiply.

Chris Sawyer also designed the game to match what computers could do quickly. For example, guests in the park walk randomly and only use pathfinding when they really need it, like when they want to leave. If a guest can’t find the exit quickly, they just give up and complain, which shows up in the game as “can’t find exit” messages. Guests also don’t “see” each other, so there’s no need for a collision system. But guests still get unhappy if there are too many people nearby, so the player has a reason to build bigger paths. All these tricks mean the game can handle thousands of moving guests without slowing down.

Many of these optimizations were possible because Chris Sawyer was both the main programmer and the designer. He could change the game’s design to help the code run faster, something that is hard today with bigger teams.

In the comments, people praise Chris Sawyer’s skill and say this level of optimization is rare today. Some share stories about how well RollerCoaster Tycoon ran on their old computers, while newer games often struggle. Some point out that writing in Assembly is hard and takes a lot of time, and most modern games use high-level languages now. Others say modern compilers can do many optimizations, but there’s still value in designing games for performance from the start. Some mention OpenRCT2, a fan-made project that re-creates the game and gives more insight into how it was made. While some wonder if such deep optimization is needed today, most agree it’s inspiring to see what’s possible when someone really understands both code and design.

Next, we have a story about the future of code, even with new tools like AI. The article says that code is not dying—writing code is still very important. Using English to describe what software should do may sound clear, but in reality, it is often too vague for computers. AI can help turn English into code, but with bigger or more complex projects, problems and bugs appear quickly if you are not exact. The story of a “vibe-coded” app crashing shows that simple instructions are not enough for hard things like live collaboration in a text editor.

The main idea is that making software is hard because of hidden complexity. Humans can only think about a few things at once, so we use abstraction—creating simple ideas that hide details—to manage this. Good code uses good abstractions, and that’s why writing code is more than just “telling the computer what to do.” Even with future AI, we will still need good code and strong abstractions. The best use of AI is to help us create better code, not just more code. The author shares a story about using AI to solve a tough programming problem, showing tools can help, but clear, careful code is still needed.

In the comments, some agree that code will not die, because real software needs deep precision and understanding. “No-code” tools often fail on big or complex projects. Most agree that code and coding skill will stay important, and AI is more of a helper than a replacement.

Next up is an article about starting and running a systems reading group at Microsoft over five years. The author started the group soon after joining, wanting to share and learn about databases and storage systems with coworkers. Meetings were simple: everyone read a paper and then talked about it. Topics included B-trees, LSM-trees, and classic research papers. Over time, members suggested new papers, sometimes from other fields, and shared blog posts or talks in a chat channel. This informal sharing was as useful as the meetings.

The group didn’t stay focused only on databases. They explored memory, networking, and consensus algorithms. Later, they read from famous books or focused on big topics, which helped build deeper understanding and better discussions. The author’s advice: start small, stick to a regular schedule, and let topics grow naturally. A co-organizer helps when one person is busy. It’s important to make meetings friendly, even for people who didn’t finish the reading, by giving a short summary. You don’t need to be an expert—just willing to learn together.

The main benefits were learning new things and meeting other curious people. These connections sometimes helped with real work. The author encourages others to start a reading group without worrying about getting it perfect.

In the comments, many liked the idea and shared their own reading group experiences. Some said regular meetings are hard but important, and monthly works better than weekly. Overall, readers thought it was good, practical advice for anyone starting a group at work.

Let’s move to Project NOMAD, a free, open-source tool that lets you access a huge library of knowledge, AI, maps, and education tools offline. It’s made for people who want information when the internet isn’t available, like in emergencies or off-grid living.

NOMAD brings together resources like Wikipedia, medical guides, and Khan Academy, and makes them work without an internet connection. You can also run AI models locally, so you can chat with an assistant, write, or code offline. NOMAD uses strong hardware, so you get fast AI and big collections of information. It bundles open-source tools like Kiwix for offline Wikipedia, Ollama for AI, OpenStreetMap, and Kolibri for education. Everything is free, runs on any PC, and is easy to install with just two commands on Linux.

Unlike other offline products that are expensive or locked to certain hardware, NOMAD gives you full control and works on more powerful computers. You can choose what content to download, upgrade your setup, and make your own offline server. The project is funded by the community and has no paywalls.

In the comments, many people like the idea of offline access to knowledge, especially for remote areas or travel. Some say this could help in places with bad internet or censorship. A few point out that offline AI is exciting, even if models are not as powerful as online ones, but having privacy and control is a big plus.

Our next story is about running a huge AI model—397 billion parameters—on a normal MacBook Pro laptop using a tool called Flash-MoE. The idea is to make very large language models work fast on small machines, without needing a server.

The model, Qwen3.5-397B-A17B, would normally need a lot of memory and a strong computer. Flash-MoE makes it possible by streaming the model’s data from the SSD only when needed, instead of loading everything into RAM. It uses C and Apple’s Metal for speed—not Python or big frameworks. You get more than 4 tokens per second, which is fast enough for real use, including chat and tool calling.

Only a small part of the model is loaded at a time. The OS handles caching, so there’s no need for a custom memory cache. The project uses special GPU code for math, like matrix multiplies, and clever tricks like reordering math to use fused multiply-add instructions for more speed. The pipeline is designed to use the SSD and GPU efficiently, avoiding slowdowns. Many experiments were tried—some worked, like trusting the OS page cache and using BLAS for math; others didn’t, like compression or custom caching.

The main hardware is a MacBook Pro with an Apple M3 Max chip and 48GB memory. Only about 6GB of RAM is used for code and data, and the rest is for SSD caching. The model is 209GB on disk. The software is open source, written mostly in Objective-C and C.

In the comments, many people are excited to see such a large model running on a laptop, calling it impressive. Some ask about running it on other systems, but for now, it’s for Apple hardware and Metal. Others say this shows how much hardware and software have improved. There are questions about real-world use, battery life, and SSD wear, but most are happy to see open source code written in C and Metal, not just Python. Overall, people see this as a clever project that pushes the limits of laptops and big AI models.

Now let’s talk about using AI for science. One article tested if large language models can predict how fast coffee cools in a mug, using real math and a home experiment. The author asked several LLMs for an equation to estimate coffee temperature over time after pouring boiling water into a mug at room temperature.

The problem is hard, because many things affect cooling: heat moves between the water, cup, air, and table; water evaporates; the mug’s material matters; and more. Even humans can’t guess perfectly. The LLMs all used exponential decay equations, showing temperature dropping quickly at first, then more slowly. Some used two rates—a fast and a slow process—while others used just one. The author did a real experiment, measuring the coffee’s temperature every few seconds.

Comparing the LLM predictions with the real data, the models were all “OK” but not perfect. The best model was close, but still not fully correct. The real coffee cooled even faster at first and slower at the end than the models expected. The author jokes that their own guess was worse than the LLMs, so maybe the models are still useful.

In the comments, people are impressed that LLMs can do this much with little detail. Others point out that these problems are hard even for scientists, and exponential decay is the usual approach. Some think the LLMs did better than many humans, and liked that the author tested the predictions. There are questions about whether giving the models more details would help, or if LLMs could ever replace experiments. Some say the cost of using LLMs for this is still high, especially when some models charge money but don’t answer. Others joke about the “middle-school science project” style, and say it’s a good reminder that real-world physics is messy. Some mention that LLMs’ answers are limited by their training data and might not improve unless they see more real experiments. Others suggest combining LLMs with physics tools for better results. Overall, people are curious, impressed, and a bit skeptical about using AI for science problems.

Next, good news for .NET developers: MAUI, a framework for building cross-platform apps, can now run on Linux thanks to a new Avalonia backend. The Avalonia team released a preview that lets developers use Avalonia to bring .NET MAUI apps not only to Linux, but also to WebAssembly for web apps.

Getting started is easy: create a MAUI app, add the Avalonia.Controls.Maui.Desktop NuGet package, add the net11.0 target, and update your MauiBuilder. You don’t need a lot of custom code. Avalonia handles the platform-specific work, and you can extend the setup if you want more control.

The team improved Avalonia by adding new navigation APIs and controls, helping close the gap with MAUI’s features. Because Avalonia draws its own UI, your app will look the same everywhere. The team tested several apps, like MauiPlanets, a 2048 game, and larger samples. Most code worked as expected, with only small fixes needed for themes. They also tested WeatherTwentyOne, using Avalonia’s new open-source WebView.

If your app uses MAUI’s GraphicsView or SkiaSharp, it should work with Avalonia MAUI without changes. The team is working on more features, like MAUI Essentials and better WinUI support. The goal is to give developers choices: native controls for each platform, or Avalonia’s drawn controls for a consistent look.

In the comments, users are excited to see MAUI running on Linux—something often missing from cross-platform frameworks. Some think Avalonia is now the top option for desktop apps with .NET, since it brings real cross-platform support. Others still prefer native toolkits for the best experience, but say it’s good to have choices. Some are worried about performance and memory usage for drawn UIs, but for business apps, having the same UI everywhere is often more important. There are questions about how well complex apps will work, especially those with deep OS integration. Some are happy about the open-source direction and the easy migration path to Linux. Others hope the project keeps improving documentation and support. A few remember the long history of cross-platform .NET UI attempts, and hope this version will “stick.” Many are also excited about WebAssembly support for using .NET apps in browsers. Overall, Avalonia MAUI is seen as a big step forward for cross-platform .NET development.

Our last story today is about using Claude, an AI tool, to help test a mobile app called Zabriskie, built by a solo developer. The main problem was testing the same app across web, Android, and iOS, without writing different code for each or doing all the manual work alone.

The developer used Capacitor, a tool that wraps a web app into a mobile app so the same code runs everywhere. This makes building and updating fast, but testing is hard. Usual web testing tools don’t work, and native mobile tools can’t see inside the web part. So, the developer taught Claude to do mobile QA by driving the apps, taking screenshots, checking for problems, and making bug reports.

On Android, this was easy. The developer could connect to the Android WebView using Chrome DevTools Protocol (CDP), the same tool used by Puppeteer. With CDP, Claude could log in, move around, take screenshots, and spot issues quickly. The whole Android setup took only 90 minutes.

On iOS, the process was much harder—over 6 hours. The iOS Simulator blocks many ways of controlling the app. Typing special characters for emails didn’t work well. Native dialogs were hard to dismiss. Tapping buttons by guessing screen coordinates often failed. The developer had to write backend changes, dig into system databases, and use special tools to finally get reliable, automated testing working.

There was also a mistake with git, where Claude made changes in the wrong place, mixing unrelated code and breaking tests, causing extra work to fix and clean things up.

In the end, both Android and iOS can now be tested automatically each day, with screenshots and bug reports. The key lessons: use system protocols when you can, measure before acting, keep code changes isolated, and always run tests before pushing code.

Commenters praised the creative solutions, especially for iOS, saying the debugging stories were both funny and painful. Others pointed out that Apple’s closed approach makes automation much harder than Android, and wish Apple would offer better tools. Some liked the lesson about not mixing code changes and the reminder to always run tests. A few said that using cross-platform tools always brings pain, but it’s often the only way for solo developers. There were questions about whether these methods work on real devices or just simulators. Some hope Apple will make testing easier, or that third-party tools might help. Many said this showed why indie developers need to be creative and flexible, especially when tools don’t do what they want. Overall, readers agreed that mobile testing is hard, and this story showed both the challenges and the smart workarounds possible with enough patience.
