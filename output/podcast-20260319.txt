Next, let’s talk about Cockpit, a tool to manage Linux servers using a web browser. Cockpit gives you a simple interface for checking logs, handling storage, configuring networks, and starting or stopping containers. You can see terminal errors in the web interface and switch between the terminal and web tool easily. If you install Cockpit on more than one server, you can connect and jump between them using SSH. Cockpit works on many Linux systems and is open source, built with JavaScript, Python, and C. The project is active, with many contributors and regular updates.

In the Hacker News comments, users say Cockpit is very helpful, especially for people not comfortable with the command line. They like doing admin tasks from their browser and seeing everything in one place. Some use Cockpit at home or in small businesses. Others warn about security and say you should use strong passwords and firewalls. Some people prefer command-line tools for their flexibility, but agree Cockpit is good for beginners or for quick server checks. Users compare Cockpit to Webmin, noting Cockpit looks more modern and is easier to use. Some worry about giving a web interface root access, but others say Cockpit uses standard Linux permissions. Most agree it is a good tool for simple server management, but you should know its limits.

After waiting, you can allow unverified apps for a week or forever, but you must accept more warnings. Next, we have a story about the game “Return of the Obra Dinn” and its special graphic style called spherical mapped dithering in a 1bpp first-person game. The game uses only black and white pixels, like old computer screens. Dithering is used to show shades and textures by placing black and white dots in patterns. Spherical mapped dithering wraps these patterns smoothly around 3D shapes, making them look natural and detailed, even with simple graphics. The developer worked hard to keep everything clear and readable, testing different methods for shadows and lighting.

Now, Kitten TTS is a new open-source text-to-speech library that can run without a GPU and needs less than 25MB for its smallest model. There are three models: a 15 million parameter “nano” model, a 40 million “micro” model, and an 80 million “mini” model. All work well on CPUs and don’t need special hardware. Kitten TTS offers eight voices, adjustable speed, and makes audio at 24kHz. It is written in Python and easy to install with pip. There is a demo online and clear instructions for advanced uses. The project is open source under Apache 2.0, but the team also offers paid support.

On Hacker News, people are excited about the small, fast models and that you don’t need a GPU. Many want to use it on Raspberry Pi, robots, or toys. Some say the voices are not as natural as big cloud TTS services, but the trade-off is fair for size and speed. Some users hope for more languages and custom voice support. There are questions about how it compares to older TTS engines and if it can run on even smaller devices. Many are happy it is open source and not locked to a company or cloud. Overall, people see Kitten TTS as a big step for running speech AI on small devices and keeping control of your data.

Next, noq is a new open-source QUIC implementation in Rust, made by the n0 team after they needed more control than the Quinn library could give for their iroh project, especially for features like multipath and NAT traversal. noq’s main feature is full support for QUIC Multipath, letting one connection use several network paths at the same time for better speed and reliability. noq supports QUIC NAT traversal and has a feature called QUIC Address Discovery to help devices learn their public IP addresses more privately. It also adds better Qlog support for debugging. noq introduces a WeakConnectionHandle type to manage connections efficiently. It has been used in production with iroh since version 0.96 and has passed tests with other QUIC implementations.

In the comments, developers are excited about a strong, open-source QUIC in Rust, especially with multipath and NAT traversal. Some ask how noq compares to Quinn and appreciate the teams still share ideas. Others ask about performance, memory use, and security. Many like that noq is tested in production. There are questions about QUIC Multipath spec updates and the difficulty of NAT traversal. People are glad to see better debugging tools and ask about using noq for peer-to-peer projects. The community is very positive and hopes for more documentation and examples.

Now, an article from Oxide Computer Company shows how to look at network data from raw electrical signals with an oscilloscope up to seeing real UDP packets with Wireshark. The author connected an oscilloscope to a network switch, captured voltage changes, and wrote code to extract waveform data. They decoded the QSGMII link, which combines four network channels, and handled 8b/10b encoding. Using Python and Rust, they found the start of each packet, removed repeated bytes, and extracted real data like MAC addresses and IPv6 headers. Finally, they wrote the packets to a pcap file and viewed them in Wireshark, confirming they saw the expected UDP packets.

In the comments, people are impressed by this deep dive, from hardware signals to network packets. Some say this is a great example of debugging and understanding every layer. Others share their own stories using oscilloscopes or logic analyzers for network problems. Some suggest using existing tools, but agree that writing your own teaches you a lot. There are questions about the hardware and why certain choices were made. Some mention that seeing network packets at the physical layer helps them understand how errors can sneak in. Many appreciate the complexity and beauty of modern networking.

Next, we have a story about making language models much more data-efficient using NanoGPT Slowrun. The team trained an ensemble of large models and got the same results with only 100 million data points, instead of 1 billion. They used heavy regularization, looping layers, new activation functions, and joined models together. Ensembling helps models learn better from the same data, and "chain distillation" lets each model learn from the last one. Regularization prevents memorizing data, forcing the model to generalize. Looping layers help the model think harder about each input. Architecture changes like removing some attention parts and using SwiGLU activation also helped.

The main point is that these changes together made the model use data 10 times more efficiently. In the comments, people are excited but ask if this can work on bigger, real-world datasets. Some wonder if heavy regularization and looping would help in other settings. Others like ensembling and chain distillation, saying they are not used enough in big AI research. Some point out that using many models is expensive during usage, not just training. Others wish for more comparison with other tricks like data augmentation. Some say the paper is hard to follow, while others are happy to see new ideas, not just bigger models. Many think this could help smaller research groups do more with less data.

Then, there is an article about testing AI coding models on strange, rare languages called esolangs, like Brainfuck and Whitespace, instead of popular ones like Python. The authors made a benchmark of 80 problems in five esolangs, with almost no training data for the models. The best model, GPT-5.2, got only 4.2% correct, and none could solve Whitespace problems. Even giving the AI more hints, examples, or letting it try again did not help much. Only running the code with an interpreter after each try helped a little, but scores stayed very low.

In the comments, some people say this is a fair way to test if AIs truly understand programming, not just copy patterns. Others point out that almost nobody uses these languages, so maybe it’s not practical to expect good results. Some say solving esolang problems would prove real reasoning skills. A few worry that AI companies will just add esolang code to training data. Others think better tool use might help, and some share stories about writing Brainfuck or Whitespace themselves. Some say this shows AI is still far from true understanding, especially with new or strange problems.
