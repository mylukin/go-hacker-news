First, let’s talk about uv, a new Python package manager that’s making waves for its speed. Many think uv is fast just because it’s written in Rust, but the real reasons are deeper. Python packaging used to be slow because tools like pip had to run each package’s setup.py, sometimes executing any code, just to figure out dependencies. Over time, new standards like PEP 518 and PEP 621 allowed package info to be read from simple files. uv uses these new standards, skipping the risky and slow parts.

uv is also fast because it drops support for old features like the .egg format, pip.conf, and compiling bytecode. It always requires virtual environments, so there’s no need to check system permissions, and it uses stricter package rules to avoid wasting time on unnecessary backtracking. When searching for packages, uv stops at the first match.

Some of these tricks could help pip as well, such as using HTTP range requests, parallel downloads, and a smarter dependency resolver called PubGrub. Rust gives extra speed in some areas, like “zero-copy” reading and concurrency, but these are smaller gains compared to dropping legacy features and using modern standards. However, pip can’t just drop old features, since it must support many older packages, so it will likely stay slower for now.

In the comments, many developers agree that dropping legacy support is key to uv’s speed, not just Rust. Some note that pip’s compatibility is important for large companies and older projects. Others point out that ecosystems like Rust’s Cargo and npm have used static metadata for years. Some users worry uv’s strictness could make switching harder for some projects, but most are happy to see package management in Python finally improve.

Next, we have a fascinating piece about Lewis Carroll, the author of “Alice in Wonderland,” who also invented a way to find the determinant of a matrix—Dodgson condensation. This method shrinks the matrix step by step, taking each 2×2 square, computing its determinant, and repeating until only one number is left. The process involves dividing by certain numbers from the original matrix, but you need to avoid dividing by zero.

Dodgson’s method is especially nice with whole numbers, as it keeps everything as integers until the end. It’s faster than cofactor expansion and similar in speed to Gaussian elimination, but avoids fractions and is easy to parallelize. In the comments, people praise the method’s simplicity, though some point out problems if there are zeros in the matrix. Others say Gaussian elimination is still more common in big software. Many like how Dodgson’s method is easy to teach and wish textbooks included it more often. There’s also some admiration for Lewis Carroll’s talent as both a writer and mathematician.

Now to some unusual science news: researchers have found a mushroom called Lanmaoa asiatica, or Jian shou qing, in Yunnan, China. Eating this mushroom can cause people to see tiny, fairytale-like people, a rare hallucination known as Lilliputian Hallucinations. The mushroom is not related to “magic mushrooms,” and its active compound is still a mystery.

Stories about seeing little people after eating this mushroom come not only from China, but also Papua New Guinea and the Philippines, showing it’s more than just folklore. In China, many people know about the effect and even tell funny stories about measuring the tiny people in their soup. DNA tests show that dried packs of the mushroom online are often mixed with other, sometimes poisonous, species. Scientists are still searching for the chemical that causes the visions.

In the comments, readers are both amused and amazed. Some are curious about the safety of eating these mushrooms and hope for more research into the unknown compound. Others find it fascinating that stories about seeing “little people” after eating mushrooms appear in different cultures. There are warnings not to eat wild mushrooms without expert help, since many poisonous ones look similar. Some readers joke about seeing tiny people in their soup, while others are excited for more scientific studies.

Let’s shift to Kubernetes. An article explains that Kubernetes is more than just an orchestrator—it’s a system where you declare your desired infrastructure state, and Kubernetes works to keep your system matching that. It uses a type system, with resources like Pods, Deployments, and Services defined in YAML files. The real power comes from “reconciliation”: if something changes or fails, Kubernetes tries to bring things back to the declared state. GitOps is another layer, where the desired state is kept in Git, and a controller ensures the cluster matches what’s in the repository.

Comments agree that thinking in terms of “desired state” and reconciliation makes Kubernetes easier to understand. Some find the system too complex for small apps, while others say the type system is helpful in big projects. There’s discussion about the challenges of debugging, the pros and cons of GitOps, and warnings about making things too abstract. Some wish Kubernetes were simpler, but many agree that letting the system do the hard work helps avoid manual mistakes.

Next, a new desktop app lets you train and view 3D Gaussian splats from your own videos or photos on Windows, using Python and GPU power with CUDA and PyTorch. The app guides you through loading images, finding frames, training 3D splats with one of three methods, and then viewing or exporting the result. It wraps tools like COLMAP for camera positions and supports live previews and structured project folders.

In the comments, people are excited by the simple workflow and the quality of results, especially using just one or a few images. Some note that needing Windows and CUDA limits who can use it, and memory use can be heavy with some training methods. There are ideas for supporting Linux, web versions, and easier package management. Some praise the code structure and caching, while others suggest improvements for camera controls and export options.

Another article discusses why using Git as a database for package managers and other tools rarely works well as projects get bigger. Many package managers, like Cargo, Homebrew, and CocoaPods, started with Git because it’s easy and free, but they later switched to HTTP-based systems or real databases as their projects grew. Git becomes slow and hard to manage with huge numbers of files and large histories, and services like GitHub struggle with the load.

Most commenters agree that Git is not meant for this use and share their own stories of scaling pain. Some defend using Git for small projects or tracking changes, but many say that if your tool starts to look like a database, it’s better to use a real one from the start.

Moving on, there’s a story about an insulin pump, a medical device for people with diabetes, that runs the Linux kernel. The company making the pump does not follow the GPL license: they refuse to share their source code, even when asked. This breaks the rules and makes it harder for people to trust or fix their devices.

In the comments, people are upset that companies break open-source licenses, especially for devices people’s lives depend on. Some argue that open source can be risky for medical devices, but many believe users deserve the right to see and fix the code in their products. There’s also discussion about how hard it is to enforce the GPL, with some saying only big organizations have the resources for legal action.

On the security front, a new exploit called MongoBleed targets MongoDB servers by leaking server memory when handling certain compressed requests. The attack, tracked as CVE-2025-14847, can leak sensitive data like passwords or tokens, and is being compared to the famous Heartbleed bug. The exploit script automates the attack, and defenders are urged to update servers, use firewalls, and avoid exposing databases to the public internet.

Commenters are alarmed and discuss the importance of memory safety and input validation. Many warn that too many MongoDB servers are open to the internet without protection. Some thank the author for sharing details to help defenders, while others wonder how quickly MongoDB will release a fix.

Next, LearnixOS is a new project and book that teaches how to build your own operating system in Rust, step by step, with no external libraries. The book covers everything from boot loading and memory management to file systems, processes, and even running Doom. Readers are encouraged to ask questions, and the author explains every step clearly.

People in the comments are excited for a beginner-friendly approach to OS development in Rust. Some hope it will be simple enough for newcomers, while others want to see how deep it goes into hardware. There’s discussion on whether the book will cover real devices or just emulators, and some compare it to other learning OS projects.

Finally, the FFmpeg project has accused IloveRockchip of copying thousands of lines of FFmpeg code without following the open-source license. After waiting nearly two years for a fix, FFmpeg developers filed a DMCA takedown. Most on Hacker News support FFmpeg’s action, saying license rules matter and companies should respect open-source work. Some worry DMCA takedowns can be too strong, while others want GitHub to help warn about license problems. In general, the community agrees that copying code without following the license is unfair, and it’s good to see open-source developers stand up for their rights.
