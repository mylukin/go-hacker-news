## Speech Recognition and TTS in less than 500kb

- 原文链接: [Speech Recognition and TTS in less than 500kb](https://github.com/moonshine-ai/moonshine/tree/main/micro)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48911793)

This article is about Moonshine Micro, an open-source toolkit that lets small devices, like cheap microcontrollers, do speech recognition and text-to-speech with less than 500 KB of memory. The project aims to bring voice control to very low-cost hardware, using the Raspberry Pi RP2350 chip as an example.

Moonshine Micro includes three main parts: voice activity detection (VAD), speech-to-text (STT), and text-to-speech (TTS). Each part is built to use very little memory and computing power, so it can run on simple devices. The voice activity detection can run in about 36 KB of RAM and takes up about 89 KB of storage. The speech-to-text model uses around 346 KB of RAM and about 1.3 MB of storage. The text-to-speech part is larger, needing about 340 KB of RAM and storing a 1.8 MB voice pack, but all three can share the same memory. The full demo pipeline needs only about 468 KB of RAM, which fits nicely into the RP2350’s 520 KB limit. The software uses TensorFlow Lite Micro for its neural network computations. Moonshine Micro is open source under the MIT license, so anyone can use it or even build products with it. It comes with example projects, like setting up WiFi using voice commands on a microcontroller. The documentation is clear and gives step-by-step help for each part, such as custom word recognition or using the neural TTS.

In the comments, many people are impressed by how much has been squeezed into so little memory. Some remember when speech recognition needed huge servers, so running it on tiny chips feels amazing. Others ask about the quality of the speech and if it’s good enough for real products; a few say it’s not as natural as big cloud-based models, but it’s still quite clear and useful for simple tasks. There are questions about hardware support—some want to know if it works on other chips besides the RP2350, and the answer is that it should, as long as there is enough memory and the right processor. A few developers share ideas about using this for toys, home automation, or even robots. Some comment on the importance of everything being open-source, which means you’re not locked into a company or cloud service. Others point out that, because it’s all local, privacy is much better—your voice data doesn’t leave the device. A couple of people ask about training for other languages; the authors say it’s possible but may need more data and work. One user wonders about battery life, and another explains that since the models are small and efficient, power use should be low. Some warn that while this is great for commands and simple speech tasks, it won’t beat large cloud AI for accuracy or naturalness, but many agree this is a huge step for edge computing.

## Setting up your spare Mac for Claude Code to control, a step-by-step guide

- 原文链接: [Setting up your spare Mac for Claude Code to control, a step-by-step guide](https://ykdojo.github.io/claude-controls-mac/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48959392)

This article explains how to set up a spare Mac so that Claude Code, an AI agent, can fully control it. The goal is to safely let Claude Code handle risky or automated tasks by running it on a separate Mac, not your main computer.

The guide tells you to wipe your spare Mac and make a new, clean user account with no personal data and no Apple ID. You give this account admin rights, so Claude Code can do anything it needs. You turn on SSH (Remote Login) so your main Mac can connect to the spare one. You set up passwordless sudo, so Claude Code and your commands do not need a password for admin tasks. The article explains how to find your spare Mac’s network name (hostname) and set up passwordless SSH login. It also covers how to keep the Mac awake and stop it from sleeping, so it’s always ready.

There are instructions for syncing the clipboard between Macs over SSH, using a script, so you can copy and paste text or images easily. The guide shows how to install Claude Code on the spare Mac and suggests optional scripts to set up useful tools and settings for working with Claude Code. You learn how to log into Claude and GitHub for code-related tasks, and it recommends using a separate GitHub account for safety.

One technical part is getting computer-use features to work over SSH, since macOS locks down screen and input controls. The guide uses a tmux server in the GUI session, so Claude Code can control the desktop when you connect over SSH. You have to grant permissions (Screen Recording, Accessibility, Full Disk Access) by hand to tmux. There are tips for installing VPNs or other apps, and for controlling the Mac from your phone or using browser control with a Chrome extension. The article ends with steps to use Tailscale so you can reach your spare Mac from anywhere, not just your home network.

In the comments, many people like the idea of using a spare Mac as a safe, isolated environment for AI agents, especially with the risks of running powerful code on your main computer. Some users say the guide is very detailed and helpful, though a bit long. Others share that setting up passwordless sudo and SSH can be risky if you do not secure your network, and they advise using strong passwords or even disabling password login after setup.

A few commenters say that for some tasks, using a virtual machine or a container might be safer or easier than using a real spare Mac. However, others agree with the author that running in a container has limits, especially for apps that need a real Mac environment, like Unity or GUI programs. Some people ask about using older Macs, and others discuss power use and whether it’s worth keeping a spare Mac running all the time.

A few users mention that granting lots of permissions (like Full Disk Access and Accessibility) is always risky, so it’s good to use a truly empty machine. There are tips for scripting parts of the setup, and some people like the clipboard sync trick. Some ask about using similar setups with Linux or Windows. Overall, the main feeling is that this is a useful guide for people who want to explore agentic coding safely, but you need to be careful about security and privacy.

## Fable 5 vs. GPT-5.6 Sol on an NP-Hard Problem: Does /goal help?

- 原文链接: [Fable 5 vs. GPT-5.6 Sol on an NP-Hard Problem: Does /goal help?](https://charlesazam.com/blog/fable-5-gpt-5-6-sol-goal/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48956879)

This article is about testing two AI models, Fable 5 and GPT-5.6 Sol, on a very hard optimization problem called KIRO, which is about designing fiber networks for cities in France. The author wanted to see if using a special `/goal` mode in the models would help them solve the problem better.

The KIRO problem is tough because there are many ways to connect the network, with a huge number of possible solutions. The goal is to use the least cable length, while following rules about loops and branches. The author used his own old C++ code as a human baseline, and then tested both Fable 5 and GPT-5.6 Sol, with and without the `/goal` command, to compare their results.

Fable 5 did very well on this problem, giving the best solutions and being very consistent. GPT-5.6 Sol was not as strong and showed more varied results. The `/goal` mode is supposed to help the AI “try harder” or focus better, but in practice, it did not always help. Sometimes it made results better, but other times it made them worse. For both models, using `/goal` won more individual runs, but the average result was actually worse.

The article explains how `/goal` works differently in each system. For Claude Code (Fable), it uses a separate “evaluator” model that checks if the goal is reached. For Codex (GPT), the system keeps track of goals and lets the main model decide when it is done. These two approaches mean the models behave differently when using `/goal`.

In the tests, the best single result came from Fable 5 with `/goal`, but the safest and most stable setting was Fable 5 without `/goal`. The main message is that persistence, or trying longer, can sometimes help but can also make things worse if the AI is going down the wrong path. This is especially true in hard problems where more effort may just repeat a bad idea.

In the Hacker News comments, some readers praised the careful testing and the detailed benchmark setup. Others were surprised that a special “try harder” mode did not always help, and liked seeing real data on how these tools work. A few people discussed how the search space in the problem was huge, and how that makes optimization hard for both humans and AI. Some commenters pointed out that AI consistency is important, and Fable’s stable results are valuable in practice. Others worried about relying on closed systems like Claude, since you can’t see how they work inside.

A few readers shared their own stories working on similar optimization problems, saying that “more time” or “harder work” often makes little difference if the main approach is not right. Some asked about possible ways to make `/goal` smarter, or wondered if the results would be different on other types of tasks. One person noted that this kind of testing is important to guide how we use AI in real engineering work, not just code generation. Others liked that the full scripts and data are open for others to check or reproduce the results. Overall, the comments showed a mix of interest in the technical details, appreciation for transparency, and caution about over-trusting AI features without strong evidence.

The site says Elixir is easy to use, good for both small and big teams, and works well on anything from one computer to many around the world. Elixir is built on Erlang, so it's reliable and can handle trouble (like system errors) without crashing. The site explains that Elixir code is safe, easy to read, and simple to fix when something goes wrong. Thanks to its type system and its way of handling data, problems are easier to spot and fix.

For hardware projects, Nerves lets you build small, tough software for things like smart devices. AtomVM helps Elixir run on tiny computers. Numerical Elixir (Nx) lets you work with machine learning and data science. Broadway and Membrane help handle lots of data and media, like video or sound. Elixir uses the same system as Erlang, so it’s also good for things like IoT and big, spread-out systems. The site lists many open-source tools and encourages people to join the Elixir community.

Some commenters love the strong tools that come with Elixir, while others say they still prefer the older site’s style. There are also people who really like that Elixir is based on Erlang, giving it a long history of being reliable. A couple of users talk about using Nerves and AtomVM for tiny devices, saying it worked well. But a few are still not sure if Elixir can grow with very big teams or super huge projects.
