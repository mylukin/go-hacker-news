Hello everyone, this is the 2026-04-15 episode of Hacker News Daily Podcast. Today, we bring you stories about AI and security, new developer tools, privacy, open source, and the changing world of software and creativity.

First, let’s talk about Mythos, a new AI model from Anthropic made to find security problems in computer systems. Mythos is so strong at breaking into networks that Anthropic decided not to release it to the public. For now, only big software companies can use it. The AI Security Institute tested Mythos and found it much better than other AI models. In one test, Mythos finished a complex 32-step attack that usually takes a human 20 hours. It did this in 3 out of 10 tries, but each try used a huge number of computer tokens, costing about $12,500. The more tokens Mythos used, the more bugs it found, with no sign of running out of things to discover. This makes some people worry that security has become a game of “proof of work.” Now, to keep software safe, you must spend more money and computer power checking for bugs than an attacker will spend trying to break in. It is less about being clever and more about outspending the attacker.

Open source software is still important, but there are new risks. If a big company spends a lot to check an open source library, it can be safer than small private code. But open source is used by many people, so attackers also target it more. The article says that with powerful AI, the software development process might soon include three steps: build, review, and “harden” with expensive AI security checks.

In the comments, people say this is a big change for security. Some worry that only rich companies can afford good defenses. Others question if this “outspend the attacker” model will really work, since attackers only need to find one weakness. There are worries about rising costs, small teams being left behind, and open source projects needing more funding. Some believe this arms race will never end and perfect security is not possible, but hope better AI will at least make things harder for attackers.

Next, we have Keycard, a new tool for managing API keys on your own computer. Keycard’s main idea is to stop using .env files or password managers for developer secrets. Instead, it offers a local, encrypted vault. You can quickly save API keys from your clipboard, group them into profiles, and inject them as environment variables only for the process you run. Nothing is sent to the cloud, and everything is encrypted with strong methods. The tool works from the command line and is open source, with plans for paid “Pro” and “Team” versions in the future.

Commenters like the local-first, command-line approach, saying it helps avoid mistakes like checking secrets into Git. Some want support for Linux and Windows, since right now it’s only for macOS. There’s debate about local-only versus cloud sync—some value the safety of keeping everything on one device, others need cross-device syncing. Security experts ask about encryption and backups, and some point out that forgetting your password means losing access forever. Overall, people find Keycard promising, but want to see more platform support and maybe open standards.

Moving on, someone shared their experience building their own terminal pager, called "lore." Terminal pagers like `less` or `bat` help developers scroll and search big text files in the command line. The author builds terminal user interfaces and made a reusable “viewport” component in Go, handling scrolling, searching, Unicode, and more. Unicode is tricky, since some characters like emojis take up more space. The author explains how their pager handles these cases and offers search and selection features. "lore" is built on Bubble Tea and serves the author’s daily needs, though it does not have every feature of older tools.

In the comments, people praise the learning value and the focus on Unicode and terminal quirks. Some remind us that many pagers already exist, and that building your own can miss small but important features. There’s discussion about programming languages for such tools, and suggestions to add more features, like mouse support. Many say they’ll stick with `less` unless there is a big reason to switch, but like seeing new ideas for terminal tools.

Let’s turn to a different kind of story: using paper in a digital world. The writer dreams of handling emails, notes, and projects with pen and paper, letting computers help in the background by scanning or syncing. He says physical tools help him think and organize better, but digital tools are better for sharing and searching. He wishes for devices that do just one job at a time, like a real alarm clock, to help avoid distractions. The article suggests that maybe technology can help us work with less screen time and more focus.

Comments show many people like the idea of mixing paper and digital, but say it is hard to make real. Some use tablets or e-ink devices, but say it is not the same as paper. Others love the idea of single-purpose modes on devices, but worry it is hard to avoid all the distractions. Many agree that paper helps with deep thinking and focus, but digital tools are needed for syncing and search.

Now, let’s talk about Amazon and webcomic creators. Two creators, Sean Kleefeld and Tom Ray, had their Amazon accounts closed, likely by AI. Sean lost his digital content and services but did not lose income. Tom Ray lost his main way to sell comics and earn money. Amazon gave only a generic message about “violating terms,” with no way to appeal. The writer thinks Amazon uses AI to review and cancel accounts, with no human checking. He warns creators not to depend on one company and to have backup plans.

On Hacker News, many agree that trusting big platforms is risky, since accounts can be lost without warning. Some share their own stories of losing access. There is worry about more companies using AI to cut costs, which can hurt independent creators. Some hope for better laws to protect users, but most agree creators should not trust one company with their whole business.

Next, we have a story about privacy and Google. A Ph.D. student, Amandla Thomas-Johnson, says Google gave his personal data to U.S. immigration after he joined a peaceful protest. Google’s own policy promised to warn users before sharing their data, but Amandla was not warned. The data included IP addresses and login times, which can reveal a lot about a person. The EFF has asked for an investigation, saying Google’s actions are deceptive. The article warns that tech companies can easily share private data with the government, and laws make it hard for users to fight back.

In the comments, many are upset with Google for breaking its promise. Some say you cannot trust big tech with your data, as they will always help the government. Others point out that escaping this kind of surveillance is very hard. There’s talk about the chilling effect on protest and free speech, with some hoping for new laws or court cases, but many feel the system is against regular people.

Let’s move to open source and security. Cal.com, a popular scheduling software, announced they are moving from open source to closed source, because AI can now find bugs in public code much faster than before. They worry that open code makes it too easy for attackers, especially with new AI tools scanning everything. Cal.com will offer a hobby version under the MIT license, but their main codebase is now closed. They say this is to protect users, and hope to go open source again in the future.

Many in the comments are sad and disappointed. Some say open source is still the best way to find and fix bugs, because more people can help. Others understand the security worry, but feel closing the code is not a full solution, as attackers can still use leaked code or reverse engineering. Trust is a big topic—some trust Cal.com less now, while others think the company had no choice. The community seems split: some support the move for safety, others wish they had stayed open source.

Now, something lighter: a visit to a mineral exhibit called "Unearthed: Raw Beauty." The article is short and simple, just sharing photos and inviting readers to enjoy the beauty of minerals. The title, “God sleeps in the minerals,” is poetic and comes from a famous saying, though not explained deeply in the post. The comments are a mix of appreciation, confusion, humor, and debate about the meaning of the title. Some enjoy the poetry, others want more explanation, and a few take issue with the use of “God” in the title.

Next, we have news about Ludum Dare, a famous online game jam event, which is ending soon. The writer is sad about this and worries that the event’s creator, Mike, is not clear about the future. Ludum Dare was always about the community, and if many people start their own versions, the spirit may be lost. The article points out problems with the site and says Mike did not accept help from the community. Many in the comments share memories of Ludum Dare and wish the event could continue. Some defend Mike, saying running such an event is hard work. Others say it was a mistake not to share control. Many want the spirit of Ludum Dare to keep going, even if the name changes.

Finally, let’s look at a technical topic: making C interpreters faster with JIT using yk. The yk project lets you add a JIT compiler to C interpreters with very little extra code. This means you can get faster programs while keeping full compatibility. The article explains how yk uses meta-tracing to record hot loops and turn them into fast machine code using LLVM. This is much easier than writing a JIT by hand and makes it simple to keep up with language changes. The performance boost is about 2x on average, and the system is still early-stage.

Commenters are excited about this idea, saying it can help smaller languages and make JITs easier to maintain. Some remember past problems with meta-tracing, like Mozilla’s TraceMonkey, and wonder about the limits of this approach. There are questions about using yk with other interpreters and on other platforms. Many thank the author for a clear write-up and see yk as a good balance between performance and easy maintenance.
