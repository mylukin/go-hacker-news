Next, there’s news about new AI security models, like Anthropic’s Mythos, which are very good at finding security problems in code. Mythos is so strong that only big software companies can use it for now, to fix weaknesses before attackers can do harm. The main idea is that security now feels like a “proof of work” problem, similar to crypto mining. To protect a system, defenders must spend more computer power—and money—scanning for bugs than attackers spend trying to find them. For example, one Mythos run costs $12,500, and the more tokens you use, the better the results.

This changes how we think about cybersecurity. It’s not just about being clever, but about who can spend more money on AI runs. This could help open source projects, since many companies might work together to scan and protect the same code. But it also means popular open source tools are bigger targets, as hackers may spend more to attack them.

As for developers, their work may now include a “hardening phase,” where AI is used to look for problems until the budget is used up. Human skill is still needed to write and improve code, but money now limits how much you can test and secure it. The result is that making code secure will become much more expensive, and the main cost is now AI tokens, not just time.

Hacker News comments show mixed feelings. Some agree this is now an arms race—big companies have an advantage, and small teams may fall behind. Others worry attackers can share their tricks, so defenders need to spend even more. Some hope open source communities will work together, but others think high costs might push projects to close their code or even stop. Many say this “proof of work” model is wasteful, and wish for smarter ways to improve security. There’s also debate about software dependencies: should we use fewer to lower risk, or trust big open source projects because they get more protection? Some hope costs drop as AI gets cheaper, but others say attackers only need to get lucky once, so it’s always hard for defenders. The discussion is lively, with both hope and worry about the future.

Next, let’s look at PiCore, a special version of Tiny Core Linux made for Raspberry Pi. PiCore is very small and runs fully in memory (RAM). After boot, it does not use the SD card much, so every restart gives you a clean system. By default, apps and files are only in RAM and are lost after reboot. If you want to keep changes, you can use "Mounted Mode" and save things on a second partition. PiCore comes in different images, some with SSH or a graphical desktop.

To use PiCore, you download a ZIP image and write it to the SD card. You may need to make and resize partitions if you want to save changes. PiCore uses compressed swap in RAM by default, but you can add more swap space if needed. There are special boot codes, and the default user is “tc” with no password, except for SSH images. There’s a community forum and a free book called “Into the Core.”

In the comments, many like that PiCore is lightweight and fast, good for old or simple Raspberry Pi models. Running in RAM means the system is always clean and safe from SD card corruption. Some worry the setup is confusing, especially for beginners making partitions. Some compare PiCore to other small Linux systems, saying it’s one of the smallest. Users share stories about using PiCore for network tools or simple servers. Some want easier software install or backups, while others like how manual and simple it is. Some say PiCore helped them learn more about Linux internals.

Now, let’s talk about Cal.com. This well-known scheduling tool will stop being open source and move to closed source, because of new AI-powered security risks. The Cal.com team says AI tools can now easily scan open source code and find ways to attack, making it hard to keep user data safe. They say that in the past, hackers needed time and skill to find problems, but now AI makes it fast and easy, especially for open source projects.

Cal.com’s main worry is that open source is now like giving attackers a map to your secrets. They decided to close their code to better protect user privacy, even though they still believe in open source. They will still release a Cal.diy version under the MIT license, for hobbyists and developers, but it is different from their main product. The team says the main code has changed a lot, especially in authentication and data handling, to be more secure.

They give an example: AI found a very old bug in the BSD kernel and made working attacks in just hours. Cal.com says this shows how dangerous AI-powered attacks can be, even for strong open source systems.

Others understand Cal.com’s fear, since AI makes attacks easier and protecting user data is important. Some argue that closing the code will not stop skilled attackers, and customers may lose trust in Cal.com for leaving open source. Some think this is just an excuse, and the real reason is business, not security. Others say this is part of a bigger trend, as more companies close their code because of AI risks. Some developers feel let down, especially if they helped Cal.com when it was open source. Others think the open Cal.diy version is a good compromise, while some worry it will become old and unsupported. The community is split—some see the move as smart, others think it goes against open source values.

Now, a very practical tech problem: a monitor that blinks or goes black when the user moves in their office chair, because of static electricity. The author works from home with a MacBook, a 4K monitor, and an Ikea Markus chair. When they move or stand up, the monitor sometimes blinks, turns off, or fails to reconnect. The problem comes from static electricity building on the chair. When the user touches metal, they get a shock and sometimes the monitor blinks. Even without touching, gas lift chairs can create electromagnetic interference that messes up video cables.

The problem is worse with DisplayPort, USB-C to DP, or Thunderbolt cables. Many people online have the same issue. The author fixed the static problem by grounding the chair with a metal chain on the floor, and added ferrite rings to the cables to cut EMI. This helped, but did not fully solve it. Standing up slowly also helps.

In the comments, many share similar stories, some with plastic chairs or in dry air. Some say grounding the chair or using a humidifier helped. Others fixed it by using better cables. Some are surprised that office chairs can cause computer problems. One joked about “dangerous chairs” in tech jobs. Some warn that static can damage electronics, not just monitors. Others wanted more technical details about EMI and ferrite rings. Some liked the simple solution with a metal chain. A few never had this problem and think it depends on climate or chair type. Most agree it is a strange but real problem, with several ways to make it better.

Finally, we have a story about learning to write compilers. Many say that most compiler books are too big and complex for beginners. The author suggests starting with two things: Jack Crenshaw’s “Let’s Build a Compiler!” series, which uses simple steps and examples, and the “A Nanopass Framework for Compiler Education,” which shows how to split a compiler into many small, clear steps. Crenshaw’s guide skips internal trees (ASTs), but modern languages make them easy. The main message: you do not need big books to start. Try these two resources and build a simple compiler. Only after that should you read big theory books like the “Dragon Book.”

In the comments, many agree that big books can scare beginners. Some learned a lot from Crenshaw’s tutorials and say building a small compiler helps you understand programming better. Others like the Nanopass approach, saying small steps keep things clear. Some warn that real compilers are complex, but small examples are helpful. There’s debate about whether learning compilers is needed—some say yes, some say no. Others share more simple compiler tutorials or debate the value of the Dragon Book. Most agree: start small and hands-on, not with big theory books.
