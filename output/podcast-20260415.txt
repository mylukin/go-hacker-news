First, IPv6 has finally passed an important milestone: it now carries more than half of all internet traffic. This is the first time IPv6 has crossed the 50% mark, which means most online data now uses this newer system instead of the old IPv4. IPv6 was built because IPv4 was running out of addresses—IPv4 can only support about 4 billion devices, but the world now has many more online. IPv6 supports a much larger number of addresses, so every device can have its own. The switch to IPv6 has taken over 20 years, with many networks and websites only recently making the change. Big companies like Google and Facebook helped by moving their own services to IPv6, and now many phones and home networks use it by default. This is a big step for the internet, promising to connect more people and devices, and offering better security and easier management. However, not every country or company has made the switch, and some still have trouble with old systems.

In the comments, people were happy to see this progress, saying it’s a good sign for the internet’s future. Some were surprised it took so long, and a few still have trouble with IPv6 at work or at home because their providers or company networks don’t support it. Others noted that the move was hard, especially for big companies, and that some websites still only use IPv4, so both systems will be needed for now. While some think IPv6 is more secure, others warn that network security is still important. Many hope this milestone will speed up the move to IPv6 everywhere, but agree there’s still a lot of work before IPv4 can be turned off for good.

Next, let’s talk about Darkbloom, a new system that lets people use their idle Apple Silicon Macs to run AI tasks for others, while keeping data private. The idea is to connect unused Mac computers with people who need AI power, cutting out big tech companies in the middle. Most Macs are idle for about 18 hours a day, even though they have strong AI hardware. Today, using AI services is expensive because many companies take a share along the way. Darkbloom lets Mac owners earn money by sharing unused compute time, and users pay less because there’s no middleman. The API is compatible with OpenAI, making it easy to switch—just change the base URL. Security is a focus: all requests are encrypted end-to-end, and Apple’s secure hardware means the Mac owner can’t see user data. There are four security layers, including hardware keys and signed results.

Darkbloom says prices are about half of what you’d pay for similar services, and Mac owners keep all the revenue minus electricity costs. Setup is simple—just run a command in Terminal. The company also talks about how they keep data private and transparent.

Many commenters are excited about using idle Macs for AI, saying it’s a clever way to use existing hardware and earn money. Others like the privacy features and easy API switch. However, some are skeptical, asking if there will be enough demand for Mac owners to earn much, and if attackers could still find ways to see user data. Some wonder about possible wear-and-tear on their machines and if the electricity costs are really as low as promised. Developers like the OpenAI-compatible API, but some worry if Apple will allow this kind of use, or if future updates could break things. Overall, there is a mix of excitement, curiosity, and some healthy skepticism, with people waiting to see how it works in real life.

Let’s move on to a serious Windows security bug called the “Red Sun” vulnerability, found on Windows 10, 11, and Server with the April 2026 update. The problem is that Windows Defender, the built-in antivirus, can be tricked into putting a dangerous file back into the system instead of deleting it, if the file has a special cloud tag. The researcher who found it made a proof-of-concept program to show how it works. The bug lets attackers overwrite important system files and gain admin access. The code and details are open source on GitHub.

Many commenters are shocked that such a simple trick works on a modern system. Some say antivirus software has caused problems like this before, and making perfect security tools is very hard. There are jokes about not relying on Windows Defender alone, and some ask if Microsoft has patched the bug. System admins are worried, and people share ideas for temporary fixes. Some explain how similar bugs happened in the past, and a few say this shows why open source security tools can be safer. Overall, people find the bug both funny and worrying.

Now, let’s talk about NaviDial, a phone service in Japan using 0570 numbers. Many companies and public services use these, but they are not free and can be expensive, especially from a mobile phone—about 33 yen per minute, and the price will rise soon. Even unlimited calling plans do not cover NaviDial. This is a problem for people with low income or those needing help from public services, as they may stay on hold for a long time, making the call very expensive.

NaviDial started in the 1990s so businesses could reduce costs. Before that, FreeDial (0120 numbers) let people call for free, but FreeDial was expensive for businesses. NaviDial made callers pay, but kept smart features like call routing. Changing to a new system is hard in Japan—companies trust NTT, the provider, and moving would need money and technical skills. Other countries, like the US and UK, use more toll-free numbers for important services, and in the UK, paid customer service numbers were banned after complaints. In Japan, NaviDial remains because there hasn’t been enough public pressure.

Many commenters were surprised at how expensive NaviDial calls are, especially for public services. Some shared stories of paying a lot while waiting on hold. Others said the system makes life harder for poor or vulnerable people, and that other countries moved to cheaper or free systems after complaints. Some wondered why Japan doesn’t use more online options, but others replied that old habits and rules slow down change. Many agreed NaviDial is outdated and unfair, but it will likely stay until there’s real pressure to change.

Moving on, the XOR swap trick—a programming method to swap two variables without a temp variable—was discussed. XOR means “exclusive or,” and in code, XOR is `^`. The trick uses three lines:
a ^= b;
b ^= a;
a ^= b;
After this, `a` and `b` are swapped. The article explains why this works and tests if it’s useful today. Modern compilers turn both XOR and temp-variable swaps into the same machine code, so there’s no real benefit. For pointers, XOR swap uses more instructions and is slower, and can erase data if both pointers are the same. The addition swap trick can fail with overflow, so it’s unsafe.

Most readers agree the XOR swap trick is just a curiosity now. Some point out it can cause bugs, and modern compilers make it pointless. A few remember using it on old computers, but say those days are gone. Many say it’s still fun to explain in interviews, but not for real code. The main lesson is to trust the compiler and write clear code—swapping with a temp variable is easy to read and just as fast. Others note XOR is still useful for other algorithms, like checksums and cryptography.

Next, Airbnb shared their story of moving from an old StatsD metrics system to a modern pipeline using OpenTelemetry and Prometheus. They set up a system to send metrics to both StatsD and OTLP at the same time—called dual-write—so they could move slowly and safely. After switching, CPU use dropped, reliability improved, and they could use better features from Prometheus. For very busy services, memory use went up, so they switched to a “delta” mode to save memory, with small trade-offs.

To manage huge amounts of data, they used vmagent for routing and aggregating metrics, scaling up to handle over 100 million samples per second. After moving, they found a bug with counters in Prometheus, which they fixed using a “zero injection” trick—sending a fake zero to help Prometheus count correctly.

Commenters liked the smart dual-write migration and the technical details. Some warned that high-cardinality metrics can be expensive in Prometheus. Many liked vmagent for its scale and flexibility, and some wondered if Airbnb would open source their changes. Most praised Airbnb for sharing both their successes and the problems they faced.

Another article showed how to speed up finding similar words using Levenshtein distance and a trie data structure. The simple way is to compare every word in a dictionary with the target word, but this is slow for big dictionaries. Using a trie—a tree where each node is a letter—lets you check many words at once and skip repeated work. This makes the search much faster, over 300 times faster in tests. Memory use can be high, so more compact structures like DAWG are sometimes used. The method is used for RhymeBrain, a site that finds rhymes in a huge dictionary.

Other methods, like Peter Norvig’s, create possible misspellings and check a hash table, which is fast for small errors. Some use memoization, but this can get complex. Another method is the Levenshtein automaton, which is like a big regular expression but hard to code.

Commenters liked the trie approach and shared their own uses for similar algorithms, like spell checkers and DNA matching. Some noted that for small dictionaries or low error limits, hash-table methods are easier. Memory use was discussed, and some suggested more compact structures for big data. Others said Levenshtein distance isn’t always best, especially for other languages or fuzzy matching. Some debated language choice for speed, sharing code in C++ or Rust. Many liked that the article was simple and had working code, but some wanted more benchmarks. Most agree using tries for Levenshtein search is clever and practical, especially for big dictionaries or when you need to find all similar words fast.
