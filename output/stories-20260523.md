Instead of a normal desktop, they set up the computer to boot straight into a text-only terminal (tty), skipping all graphic interfaces. This helps break habits and keeps the device focused only on writing. They installed Debian with no desktop environment and made sure to set up their user for “sudo” access by skipping the root password when installing—something new Debian users often miss.

They customized tmux to show battery level, control brightness with keyboard shortcuts, and moved the status bar to the top of the screen with a green color. These tweaks made the device more comfortable and gave quick info without distractions.

They set up “vimwiki” for note taking and writing, and “syncthing” for syncing files to another server—important since the writerdeck is mostly offline. Because there’s no browser, they made syncthing’s web interface open to all addresses, though they know this isn’t ideal for security. For convenience, they enabled auto-login and set tmux and vimwiki to launch right away when the laptop starts, so they can begin writing immediately.

After a week of using the writerdeck, the author felt more productive and less distracted. They enjoyed having a device that only does one thing: writing. They suggest others might benefit from a similar setup if they want to focus better.

## .NET (OK, C#) gets union types

- 原文链接: [.NET (OK, C#) gets union types](https://andrewlock.net/exploring-the-dotnet-11-preview-2-dotnet-gets-union-types/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48234954)

.NET 11 (C# 15) now adds union types, a feature many developers have wanted for a long time. This lets you create a type that can hold one of several possible types, making your code clearer and safer.

Union types are common in languages like F#, TypeScript, and Rust. They let you define a type that could be, for example, either a success value or an error value—think of things like `Result<TSuccess, TError>` or `Option<T>`. In the past, C# developers had to work around this using base classes, `object`, or enums, but these methods had problems. Now, you can use the new `union` keyword to declare a union type, listing all the possible types it can hold. For example, you might make a `SupportedOS` union that can be either `Windows`, `Linux`, or `MacOS`.

To use a union, you just create it with one of the allowed types. You can also easily check which type it holds using a `switch` expression, and the compiler will warn you if you forget to handle a possible type. The union types are implemented as structs with one property (`Value`) that holds the current value, and they use special attributes and interfaces under the hood. If you want to avoid performance problems from boxing (putting value types on the heap), you can write your own custom union type using a `TryGetValue` pattern.

To try union types now, you need to use .NET 11 preview and turn on language preview features in your project. Some IDEs, like Visual Studio preview and VS Code, already support this feature, while tools like Rider are still catching up. The article also talks about possible future features, like closed enums and closed class hierarchies, which could make switch expressions even safer and more useful.

In the comments, many people are excited to see union types finally arrive in C#. Some point out that F# and other functional languages have had this for years, and they wonder why it took so long. Others worry that the current implementation, which uses boxing, could hurt performance in some cases. There are also questions about how union types will work with pattern matching and whether this will make C# more like functional languages.

Some developers share that they have been using libraries like OneOf to get similar features before this, and they hope the new built-in unions will make code simpler and safer. A few worry about compatibility with older .NET versions or with different IDEs. Others ask about real-world use cases, like error handling or API responses, and are happy to see more options for writing clear code.

Some comments discuss the differences between C# unions and those in languages like Rust or Swift, especially around exhaustiveness in switch statements. There's also talk about whether unions will push people to use more functional programming styles in C#. Finally, some are waiting to see how this feature will evolve, hoping for even better support in future releases.

## On The <dl> (2021)

- 原文链接: [On The <dl> (2021)](https://benmyers.dev/blog/on-the-dl/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48247325)

This article explains the HTML `<dl>` element, which is used for lists of name–value pairs, like a glossary or a list of product features. The author says `<dl>`, with its `<dt>` (term) and `<dd>` (detail) parts, is very useful but not used enough.

The article walks through how to use `<dl>` for things like book info: you put the whole list inside `<dl>`, then use `<dt>` for the name (like "Author") and `<dd>` for the value (like "Jeffrey Zeldman"). If you have more than one value for a name, you can use multiple `<dd>`s after one `<dt>`. You can also use `<div>` to wrap groups of `<dt>` and `<dd>` for styling, but only `<div>` is allowed for this.

The author says that people often use just `<div>`s for these lists, but this is less helpful for computers and users. Using `<dl>` gives more meaning (semantics) to the content, which helps devices and screen readers understand and present the information better. For example, a screen reader can tell the user how many items are in the list or let them skip the whole list if they want. These benefits are real in most browsers, but not all, so you might still need to use other elements in some cases.

The article gives a fun example: a Dungeons & Dragons monster stat block, which has many name–value pairs, showing how `<dl>` can be used for all kinds of lists, even in games.

The main takeaway is that using `<dl>`, `<dt>`, and `<dd>` helps make web pages more clear and accessible, not just for people but also for computers and screen readers.

In the top Hacker News comments, some people agree that `<dl>` is not used enough and praise its readability and structure. Others say that browser support, especially for accessibility, can be mixed, so they sometimes avoid `<dl>` in production. A few commenters mention they have seen `<dl>` misused for things that are not really name–value pairs, like navigation menus, which is not the intended use. Some like that `<dl>` helps with screen readers, but others point out that, for complex layouts, extra styling or JavaScript might be needed to get the look they want. There are also comments asking for more examples or showing creative uses, like using `<dl>` for API parameter lists or configuration files. Overall, people like the idea, but some want better support or clearer guidelines for when to use `<dl>`.

## Hengefinder: Finding When the Sun Aligns with Your Street

- 原文链接: [Hengefinder: Finding When the Sun Aligns with Your Street](https://victoriaritvo.com/blog/hengefinder/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48241335)

The article talks about Hengefinder, a tool that helps people find the exact dates when the sun sets in perfect alignment with city streets, like the famous Manhattanhenge in New York City. The author explains how they built this tool while at the Recurse Center, aiming to let anyone find these special "henge" moments anywhere, not just in Manhattan.

First, the article explains how to find the angle of a street, known as its "bearing," by using the latitude and longitude of two points on the street. At first, the author thought you could just use simple math, but then realized the Earth is not flat. Longitude lines get closer together near the poles, so you have to adjust the math by multiplying longitude by the cosine of the latitude. This makes the units match and gives the correct angle.

Next, the author needed to find the angle of the sun at sunset, called its "azimuth." They used a Python library called Astral, but found that its definition of sunset is a bit too late for a true henge moment. The real henge needs the sun to be just touching the horizon, not below it. To solve this, the author used a special kind of binary search to find the last minute when the sun is still visible above the horizon.

The last challenge was to find the date when the street bearing and the sun’s azimuth are the same. Because the sun's path changes in a complicated way, the author used a two-step search. First, they checked every 30 days to find possible dates. Then, if a close match was found, they checked day by day in that time period to find the exact date. This method is faster and uses fewer API calls than checking every single day.

With all these steps, the author finished Hengefinder, a website and app that anyone can use. It lets you enter any street and see when the next henge will happen there. It also has fun features, like finding "Sauron henges," where the sun sits on top of a tall building.

## Reverse engineering circuitry in a Spacelab computer from 1980

- 原文链接: [Reverse engineering circuitry in a Spacelab computer from 1980](https://www.righto.com/2026/05/reverse-engineering-spacelab-computer.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48248954)

This article is about looking inside and understanding how the Spacelab computer, used in the Space Shuttle in the 1980s, was built and how its main parts worked. The author explains how the Spacelab computer, called the Mitra 125 MS, was not like today’s computers because it did not use a single microprocessor chip, but instead was made from many small logic chips placed on several boards.

The computer was made in France and was used to control the Spacelab laboratory, which was a special area for astronauts to do science experiments in space. The Mitra 125 MS had a 16-bit processor and used magnetic core memory, holding only 128 KB of data. It was built from military-grade logic chips called the 5400 series, which are similar to the 7400 series that many electronics hobbyists know. The most important chip in the computer was the 74181, which is an arithmetic/logic unit (ALU). The 74181 chip could do math and logic operations, but only for 4 bits at a time. To work with 16 or 32 bits, several 74181 chips were connected together.

The author looked closely at one of the ALU boards in the computer. The Spacelab computer used eight ALU chips to make a 32-bit adder, which was surprising since the computer was supposed to be 16-bit. This larger ALU helped with faster multiplication and floating-point math. The ALU board had many other chips too—multiplexers to choose inputs, registers to store data, and special chips to make math faster. The boards were packed with chips but not as tightly as American space computers of the same time. For example, IBM computers used smaller chips and could fit more on each board.

The article also tells the history of French computer companies and why France wanted its own computers, not just American ones. This was because of problems in the 1960s, when the US would not sell powerful computers to France, and France wanted to be independent, especially for military reasons. The Mitra computer line came out of this effort.

Later, the Spacelab computer was replaced by an IBM computer because the new IBM computers were faster, smaller, and had more memory. The switch was also because of politics between Europe and the US. The IBM replacement could even pretend to be the old French computer by changing its microcode, which is the low-level software inside a processor.

A few people remembered working with the 74181 chips and shared their own stories about using similar parts. Others were surprised that the Spacelab computer used normal “consumer” chip packages, not special space ones, and they wondered if this affected reliability in space.

There was interest in the French computer industry’s history. Some commenters talked about Plan Calcul, the French effort to build its own computers, and said it was hard for any country to beat IBM back then. A few people thought that even if the French computer industry failed, trying was important for national independence and learning. Some readers compared the Spacelab computer to other space computers they had seen, saying each country had its own way of solving problems.

## z386: An Open-Source 80386 Built Around Original Microcode

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48248014)

This article is about z386, an open-source CPU that copies the 80386 chip using the original Intel microcode. The project runs on FPGAs and can boot old software and games like DOS, Doom, and Cannon Fodder. z386 is not just an emulator; it tries to rebuild the real 386 hardware using recovered microcode. The author compares z386 to another FPGA CPU, ao486, showing that z386 uses fewer resources but has a bit lower performance in some tests. The machine runs at higher clock speeds than the original 386, but each instruction takes more cycles. z386 keeps many parts from the 386, like the paging TLB, a barrel shifter, and the microcode ROM, but adapts some things for FPGA. It has eight main units: prefetch, decoder, microcode sequencer, ALU, segmentation, protection, paging, and memory/cache system. The prefetcher and decoder work together to read and understand complex x86 instructions. The decoder uses tables (PLAs) to know what to do next and how to start the right microcode routine. The microcode sequencer uses the original 37-bit micro-instructions, which makes z386 behave like a real 386, including delay slots and special instruction handling. z386 adds a 16KB L1 cache to speed up memory access, making it act more like a high-end 386 with fast memory. Testing was hard, especially in protected mode, but the author used lots of test programs, including games and DOS tools, to find and fix problems. z386 is different from ao486 because it keeps the old-style design with big, separate units, while ao486 uses a modern pipeline. The project is still in progress but can already run real software on several FPGA boards.

## 80386 Microcode Disassembled

- 原文链接: [80386 Microcode Disassembled](https://www.reenigne.org/blog/80386-microcode-disassembled/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48247004)

This article is about disassembling the microcode from the Intel 80386 CPU, which is a complex and important part of computer history. The author got a high-resolution image of the chip’s microcode and, with help from others, managed to turn it into readable code.

At first, the job looked impossible because the 80386’s microcode is almost ten times bigger than the one for the 8086. There was also no guide or patent to help. But thanks to teamwork, they used image processing, neural networks, and manual checking to extract the microcode from the chip image. Disassembling it was still hard. They had to find patterns to break up the huge blob into smaller, understandable parts, like micro-operations (μ-ops) and their fields. They guessed that some fields in the microcode matched source and destination registers, and that some patterns showed where instructions ended. Ken Shirriff helped by tracing how the chip’s logic lines were connected. Bit by bit, they learned how the microcode linked to real instructions.

The 80386 is much faster than the 8086 because it uses more hardware to do things that were done in software before. This means most of its microcode just sets up these hardware helpers, like the math unit and shifter. The microcode has 215 entry points, a big jump from 60 in the older chip. Every instruction on the 80386 is run by microcode—unlike the 8086 and modern CPUs, which sometimes use hardware alone. There is some “unused” code, but it looks like it’s related to handling page faults or debugging, not real junk. The author may have found a bug in how the chip checks access to input/output ports, which could let a program reach hardware it shouldn’t, but it’s hard to say for sure without testing on real hardware. You can read the microcode yourself from a GitHub repository.

Others are surprised that the microcode for every instruction is present, noting how different this is from newer CPUs, which often use hardwired paths for speed.
