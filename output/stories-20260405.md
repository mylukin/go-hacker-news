## Eight years of wanting, three months of building with AI

- 原文链接: [Eight years of wanting, three months of building with AI](https://lalitm.com/post/building-syntaqlite-ai/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47648828)

This article is about someone who wanted good developer tools for SQLite for many years but only managed to build them in three months using AI. The author explains how AI tools made it possible to finally complete a project that always seemed too hard and too boring to do alone.

The main goal was to build tools like formatters, linters, and editor extensions for SQLite, which is a very popular database, but didn’t have great open-source developer tools. The hardest part was making a parser that works exactly like SQLite’s own parser, which has no formal documentation and is written in dense C code. This made building good tools very difficult and slow. The idea was put off for years because it was too much work for a side project.

When AI code assistants improved in late 2025, the author started the project. At first, AI wrote a lot of the code, but the result was messy and hard to understand. So, the author rewrote it, this time using AI more carefully—using it for code suggestions and repetitive work, but making all the important design decisions personally. The rewrite happened mostly in Rust, making the project more maintainable.

AI helped by making it easier to try ideas quickly, write standard code, generate tests, and learn new things (like pretty-printing and editor extension APIs). This let the author finish not just the core tools, but also “extra” features like editor plugins and detailed documentation—things usually skipped in side projects. AI also made it faster to refactor code and add tests, which kept the project in better shape.

But AI also caused problems. It was easy to become “addicted” to prompting AI over and over, sometimes staying up too late trying to get better results. When using AI too much, the author lost track of how the codebase really worked, making some problems hard to debug. AI also led to putting off big design decisions, since it felt cheap to change things later, but this only made the project confusing. And AI can’t understand the history or the design “taste” that comes from human experience, so it made mistakes in project architecture and public API design that a human expert wouldn’t.

The author found that AI is best when you already know what you want and can check the output easily. For tasks where the goals are unclear or need creativity, AI often makes things worse or just wastes time. The real lesson is that AI is great for writing and refactoring code, but it’s not good at making big, important design choices or understanding what users want.

In the comment section, many people praised the honesty of the article and related to the struggle of starting and finishing side projects. Some said they had similar ideas about missing good SQLite tools. Others agreed that AI helps with repetitive work, but warned about losing touch with the code and making codebases messy. A few pointed out that relying too much on AI for design is risky—it can speed up building something wrong. Some shared their own experiences: AI made them more productive, but they still needed to understand and guide the project. Others were skeptical, saying AI code often needs just as much fixing as writing from scratch. Some liked the point about AI helping with learning new tech quickly. A few wondered if AI will soon be able to do more of the “design” parts, but most felt that humans are still needed for taste and judgment. Overall, readers found the story useful and realistic, showing both the good and bad sides of modern AI coding tools.

## Running Gemma 4 locally with LM Studio's new headless CLI and Claude Code

- 原文链接: [Running Gemma 4 locally with LM Studio's new headless CLI and Claude Code](https://ai.georgeliu.com/p/running-google-gemma-4-locally-with)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47651540)

This article explains how to run Google’s Gemma 4 large language model (LLM) on your own computer using LM Studio’s new command-line interface (CLI) and how to connect it with Claude Code for coding help. It shows how you can avoid cloud AI problems like costs, privacy, and slow speeds by using a strong AI model locally.

Gemma 4 uses a special “mixture-of-experts” design: it has 26 billion parameters, but only 4 billion are active at a time. This means you get high-quality AI results, but you don’t need a huge, expensive computer. The author runs Gemma 4 on a MacBook Pro with 48 GB memory and gets about 51 tokens per second, which is fast for local use.

Google released several Gemma 4 models for different hardware. The 26B-A4B model is a good balance between performance and memory use. It does well on AI benchmarks, almost matching much larger models but using less memory and running faster. You also get features like vision (analyzing screenshots), long context windows (up to 256,000 tokens), and tool calling.

LM Studio’s version 0.4.0 brings a headless “daemon” so you can run everything from the terminal, without needing the desktop app. You install LM Studio, start the daemon, download Gemma 4, and chat with it using simple commands. The CLI shows stats like speed and memory use, and you can list or manage models easily.

You can check how much memory you’ll need before loading a model, and adjust the context window to fit your machine. The article gives scripts and tips for planning memory use, especially on Macs, where the CPU and GPU share memory. You can also set how long a model stays loaded, control how many tasks run in parallel, and monitor everything with logs.

There are some advanced features, like flash attention (which saves memory) and speculative decoding (which speeds up dense models), but speculative decoding doesn’t work well with Gemma 4’s MoE design yet. The desktop app is still good for monitoring or using vision features.

A highlight is the Anthropic-compatible API endpoint. This lets you use Claude Code (a coding assistant) with your local Gemma 4 model instead of Anthropic’s cloud. The article shows how to set up a shell function so Claude Code uses your local AI, keeping your code private and saving cloud costs. Some features are slower or missing compared to the cloud, but for most coding and review tasks it works well.

In the Hacker News comments, many people are excited about running big AI models locally. They like the privacy, no more API fees, and having full control. Some share tips about hardware—most agree you need at least 32–48 GB of memory for fast results, and Apple Silicon works well thanks to unified memory.

A few users discuss the mixture-of-experts model: they say it’s smart to use only a small part of the model for each answer, which saves memory and speeds things up. Some worry about swap usage and high memory pressure on their machines, saying it’s best not to run other heavy apps at the same time.

Others ask if local models can fully replace cloud AI yet. Some say they’re almost there for code and text, but for complex tasks and very large files, cloud models are still a bit better. People also like the headless CLI for servers and automation, and some share their own scripts and workflows.

A few comments mention that speculative decoding is tricky with MoE models, and some hope for better support in the future. There’s interest in the vision features, especially for developers who want to analyze screenshots or diagrams.

Finally, some users point out that the setup is still a bit technical—installing, managing memory, and tuning settings can be hard for beginners. But most agree that running local LLMs like Gemma 4 is getting easier, and the new LM Studio CLI is a big step forward.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47647455)

This article is about a tool called Caveman, a plugin for Claude Code and Codex that makes AI write in short, simple “caveman” style to save tokens. The main goal is to cut the number of words (tokens) in AI responses by up to 75% while keeping the important technical meaning.

The article shows side-by-side examples: normal AI answers are long and polite, but Caveman answers are short and direct. For example, instead of a full explanation, Caveman might say “New object ref each render. Inline object prop = new ref = re-render. Wrap in useMemo.” This style removes filler words, pleasantries, and extra language, but keeps technical terms and code the same. Benchmarks show big savings: some answers use 80% fewer tokens, and on average, there is a 65% reduction across many coding tasks. Caveman only changes the output, not how the AI thinks, so technical accuracy stays the same. The plugin is easy to install and works with simple commands like “/caveman” or “talk like caveman.” You can turn it off at any time.

The article also says that science supports this style. A research paper found that short, direct answers can even make AI more accurate on some tasks. Using Caveman is not just about saving money; short answers are easier and faster to read, and they make the AI feel more fun.

In the comments, many people find the idea funny and clever. Some love how it saves money and time, especially for people who pay for AI tokens. Others say it helps focus on the real answer, without all the fluff. A few users point out that this style is good for technical work, but maybe not for customer service or teaching, where more polite language is needed. ## A tail-call interpreter in (nightly) Rust

- 原文链接: [A tail-call interpreter in (nightly) Rust](https://www.mattkeeter.com/blog/2026-04-05-tailcall/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47650312)

This article explains how someone built a fast interpreter in Rust, using a new feature called the `become` keyword. The interpreter is for Uxn, a simple virtual CPU used by a small software ecosystem. The author wanted to make the interpreter as fast as possible, faster than both his first Rust version and even his own hand-written assembly code.

The Uxn CPU is basic: it has two small stacks, some RAM, and a program counter. The usual way to write an interpreter is to read each instruction from memory, then run code for that instruction, and repeat. In assembly, the author made a “threaded code” interpreter, where each instruction ends by jumping directly to the next one. This makes it fast, but the code is large and unsafe.

In Rust, the author tried to copy this idea, but at first the stack kept growing and would overflow, because each function call added to the stack. The new `become` keyword in nightly Rust solves this: it replaces the current function with the next, so the stack does not grow. The author built a macro to reduce the amount of repeated code needed for each instruction.

Performance was very good, especially on ARM (Apple M1): the tail-call interpreter was faster than assembly and much faster than the basic VM. On x86, the tail-call interpreter was still faster than the basic VM, but not as fast as the hand-written assembly. The author found that the generated machine code was not as good as the assembly, with unnecessary register saves and restores. He also tested the interpreter as WebAssembly, but the results were poor—tail calls were much slower than the normal VM in browsers and in wasmtime.

In the comments, some readers were excited about the new `become` keyword and its uses. Others said that writing interpreters in this way is an old idea, known from languages like Forth or from threaded code in general. Several commented on the macro, calling it “ugly but effective,” and asked if Rust could make this easier in the future. Some people were worried about using nightly Rust and unstable features in production. Others discussed the differences in performance between ARM and x86, suggesting the Rust compiler and LLVM backend might improve in time.

A few readers noted that WebAssembly is not yet good at tail calls and that JIT engines may not optimize this code style well. Some shared their experiences with similar interpreters in C or other languages, comparing performance and code complexity. There were questions about the safety of this approach, but the author made clear that the code is “safe Rust.” Others wondered if this style could help with other virtual machines or emulators.

People liked the deep benchmarks and code examples, and some asked for more details about the Uxn system or how to make the x86 version faster. The overall mood was curious and positive, with readers interested in how nightly Rust is evolving and what it can do for high-performance interpreters.

## Nanocode: The best Claude Code that $200 can buy in pure JAX on TPUs

- 原文链接: [Nanocode: The best Claude Code that $200 can buy in pure JAX on TPUs](https://github.com/salmanmohammadi/nanocode/discussions/1)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47649742)

This article explains how to build your own coding AI, called "nanocode," for about $200 using JAX and TPUs. Nanocode is inspired by Anthropic's Claude Code and Andrej Karpathy’s "nanochat," but focuses on agentic coding, meaning it can use tools and interact with files and code on your computer.

The author shows how to use open datasets, like The Stack and FineWeb, to train the model, and explains that nanocode is optimized for TPUs but should also run on NVIDIA GPUs. There are two main versions: a smaller one (477M parameters) and a bigger one (1.3B parameters), with training times and costs shared for each. Nanocode uses special tokens and templates so the model knows when a user, assistant, or tool is speaking, and can perform actions like reading, editing, or searching files (using "grep") by calling simple tools.

Much of the work is about teaching the AI to use these tools. The training process uses many coding examples, both real and synthetic, and follows the "Constitutional AI" method, which means it aligns the AI’s behavior with a written “SOUL” document (telling it to be friendly, only use lowercase, and follow given instructions). Training includes supervised fine-tuning with feedback loops and direct preference optimization to help the AI choose better answers. The model is tested using standard metrics and compared against GPT-2 and nanochat, showing that nanocode is strong at coding tasks, even if its general language skills are a bit lower. Finally, the author explains how to interact with nanocode using a command-line tool, and encourages readers to modify the codebase and try new experiments.

Some ask if nanocode can really compete with big commercial models, while others are excited that it’s so hackable and easy to train. There’s praise for using JAX and TPUs, but a few mention that TPU access can be tricky for some users. Several readers like the detailed explanation of data generation and tool templating, saying it helps demystify how agentic models work. Some wonder if the “SOUL” approach makes the AI too limited or quirky, while others love the idea of giving models a personality. There are questions about how well nanocode handles complex, real-world codebases, and if it can be used for bug fixing or large projects. In general, the reaction is positive, with many saying this is a great resource for learning about modern AI training, and an inspiring project for indie hackers or small teams.
