## An OpenAI model has disproved a central conjecture in discrete geometry

- 原文链接: [An OpenAI model has disproved a central conjecture in discrete geometry](https://openai.com/index/model-disproves-discrete-geometry-conjecture/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48212493)

OpenAI’s new AI model has solved a famous math problem called the “unit distance problem,” which has been open since 1946. The problem asks: if you put n points on a plane, how many pairs can be exactly one unit apart?

For a long time, people thought the best way was to arrange points in a square grid. This setup gave the most pairs at distance one, and most experts believed no better way existed. The AI found a new method, using math from a field called algebraic number theory, which gave an even higher number of unit-distance pairs for some values of n. This result breaks the old belief and shows there is a real improvement over the square grid. The proof was checked by other mathematicians and is correct. The AI was not built just for math but could reason deeply and connect ideas from different areas. Famous mathematicians say this is a milestone because it’s the first time AI has solved such an important and hard math problem on its own.

The AI found a way to use number fields—special sets of numbers with more symmetries than usual—to arrange points better. The method uses advanced math, like infinite class field towers and Golod–Shafarevich theory, to prove that these number fields exist and can help make more unit-distance pairs. Before this, the best known answer stayed almost the same since the 1940s. Now, for infinitely many n, you can get at least n to the power of 1 plus a small delta (for example, delta = 0.014), which is better than before.

This discovery is important not only for math but also for AI research. It shows that AI can do more than help humans—it can make new ideas and solve problems in creative ways. The connection between algebraic number theory and geometry was a surprise and may help solve other open math problems too. The result suggests that AI will soon be a key partner in many kinds of science and research, not just mathematics.

In the Hacker News comments, many people were amazed by the result and its meaning for both math and AI. Some users felt excited and hopeful, saying this proves AI can really help with hard research and push science forward. Others wondered if this means AI will soon take over creative work from humans, or if people will still be needed to guide and check the results. A few users discussed the technical parts, asking how the AI connected number theory with geometry, and if this method could work for other problems. Some were a bit worried about AI progress and the need for good ways to control and understand powerful models. There was also praise for the mathematicians who checked the proof, showing that teamwork between humans and AI is still important. Others pointed out that this breakthrough might encourage more mathematicians to use AI in their research, opening a new era for both math and artificial intelligence.

## GitHub confirms breach of 3,800 repos via malicious VSCode extension

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48207660)

GitHub said that 3,800 code repositories got hacked through a bad Visual Studio Code extension. This happened because the extension had hidden code that could steal sensitive information from people using it.

The article explains that attackers made a fake extension for VSCode, a popular code editor. When people installed it, the extension sent their secret files, like passwords and API keys, to the hackers. Many developers use VSCode with GitHub, so the attack reached many code projects. The bad extension was downloaded thousands of times before anyone noticed. GitHub found out because some users saw strange activity in their accounts and told GitHub. After an investigation, GitHub removed the malicious extension and told all affected users to change their passwords and secrets. The company says no core GitHub systems were hacked—only individual code projects were affected. They are now warning developers to be extra careful when installing extensions and to check if an extension is trusted. GitHub is also working on better ways to spot these kinds of attacks in the future. They suggest using multi-factor authentication and keeping secrets out of code whenever possible. The article says this is a reminder that even tools we trust can become unsafe if we are not careful.

In the comments, some people are very worried and say it is hard to trust extensions now. Others point out that most people do not check the code of extensions before installing them. A few say that open-source tools can still be made safe if enough people review the code, but this does not always happen in real life. Some developers share tips for staying safe, like only using popular extensions with many reviews or sticking to official ones. One person says GitHub should do more to check extensions before they are allowed. Another commenter thinks this attack will make developers more careful in the future. A few people wonder if VSCode should warn users more clearly about new or untrusted extensions. Some say they will now use fewer extensions or switch to simpler code editors. Others remind everyone that secrets should never be in code repos at all. Overall, people agree that this is a big wake-up call for developers to be more careful with their tools.

## Flipper One Tech Specs

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48212046)

The article lists the technical specs for the Flipper One device, which is still being developed. It covers the size, materials, ports, CPU, memory, battery, wireless features, and expansion options.

The Flipper One is about 155 mm wide, 67 mm high, and 40 mm deep, but the final weight is not set yet. Its body and buttons are made from strong PC/ABS plastic, with Gorilla Glass over the screen and some metal parts for strength. The screen is a monochrome LCD with 256x144 pixels and 64 shades of gray. For connections, it has two USB-C ports, a USB-A port, HDMI out (supports 4K at 120 Hz), two Ethernet ports, a 3.5 mm audio jack, as well as slots for microSD and SIM cards. The controls include a touchpad, five app buttons, a D-pad, power and back buttons, an app switcher, and a push-to-talk button.

Inside, the main processor is an 8-core Rockchip RK3576 (with 4 high-speed and 4 energy-saving cores), a Mali G52 GPU, and a strong Neural Processing Unit for AI tasks. There is also a low-power Raspberry Pi RP2350 microcontroller with ARM and RISC-V cores. The device has 8 GB of LPDDR5 RAM and 64 GB of fast UFS storage, plus a slot for a microSD card. The battery is planned to be 7000 mAh, with advanced charging hardware for fast charging. For wireless, it uses a MediaTek chipset, supports Wi-Fi 6 on multiple bands, Bluetooth 5.2, and gigabit Ethernet. For video, it has HDMI and DisplayPort, both supporting 4K at 120 Hz. Audio uses a Nuvoton codec, with a speaker and audio jack. The M.2 expansion port allows many add-ons, with detailed pinouts for hardware hackers. There are also GPIO and debug ports for extra hardware or development work.

The brains come from donors who agreed to give their tissue for research. After death, doctors quickly remove the brain and keep it fresh. Scientists cut the brain into thin slices, which allows them to keep many cells working for a few days. They use these slices to study how real human brain cells respond to new drugs or treatments. This is important because animal brains, like those from mice, are not exactly the same as human brains. The article says this method is not perfect—these brain slices do not last very long, and they cannot show how a drug works in the whole living body. But it is much better than only using animals or simple cell cultures. Scientists are careful to respect the wishes of donors and their families.

## How fast is N tokens per second really?

- 原文链接: [How fast is N tokens per second really?](https://mikeveerman.github.io/tokenspeed/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48174920)

This article talks about a web tool that lets you see how fast different “tokens per second” rates feel when using large language models (LLMs). The tool shows you how LLMs stream text at different speeds, so you can understand what “30 tokens per second” or “200 tokens per second” really looks like.

The author explains that benchmarks for LLMs often talk about speed, like “47 tokens per second on an M3 chip” or “180 tokens per second on a 4090 GPU.” But these numbers are hard to imagine until you see them in action, so the tool provides a live demo. The tool has four modes: “code,” which shows syntax-highlighted code; “text,” which shows normal sentences; “think,” which mixes reasoning with code; and “agent,” which acts like an AI tool making decisions and generating code. You can set the speed and see how the text appears at different rates, from very slow (like a Raspberry Pi) to extremely fast (like top-end AI hardware). 

The tool also explains what a token is. A token is usually a short piece of text, like a word or part of a word. For example, “processUserInput” might be split into “process,” “User,” and “Input.” Punctuation and symbols also count as tokens. Code usually has more tokens per line than regular text, so the same tokens-per-second speed can feel much faster when showing code.

When you switch between code and text at the same speed, you notice a big difference. The author says the tool is meant to show this gap, because benchmarks don’t always tell the whole story about how fast an LLM “feels” when you use it.

In the Hacker News comments, some readers say this tool is helpful. They like being able to compare speeds and see the difference between code and text streaming. Others say that real-world use is even more complex, because actual prompts and outputs can vary a lot. Some people point out that the type of content matters—a fast speed with code can be hard to read, while with plain text it feels more natural. A few commenters wish there was support for other languages or tokenizers, since different LLMs use different systems. Others discuss how token speeds affect user experience, especially in chat apps or coding assistants. Some users also debate whether tokens-per-second is the best way to measure LLM speed, or if words-per-second would be clearer. Finally, a few people are surprised at how fast current hardware can stream tokens, while others worry that the “eyeball bottleneck” means higher speeds don’t always help users.

The main goal is to try out all these new parts together during flight for the first time. Many things have been changed based on past flights to help Starship become fully reusable and faster to launch again. The booster’s main test is to launch, separate from Starship, fly back toward the Gulf of Mexico, and try a landing burn. This time, the booster won’t try to return to the launch pad.

Starship will also try to relight one Raptor engine while in space. During reentry, Starship will do some hard maneuvers to test the strength of its flaps and to practice flying a path similar to future landings at Starbase. The team will watch closely to learn from each part of the flight.

The biggest issue is the app’s name, “Inkwell.” Apple said this is a trademark problem, since they once had a handwriting feature called Inkwell. But Apple’s own trademark is now “dead,” and other apps with similar names exist in the App Store. The developer tried to change the name but still got rejected. The developer thinks Apple is using its power unfairly, since the trademark is not active, but Apple still blocks the app.
