First, let's talk about an unusual experiment where 11 different AI models competed in a battle royale video game, 30 rounds in total, to see which AI is best at playing to win. The main question was, if you face a robot in a fight, would you want it controlled by Claude or by Grok? The results were clear: Grok 4.1 Fast won 13 games out of 30, while Claude Sonnet 4.6 came second with 5 wins, but at a much higher cost per win. Other models, like GPT 5.4-mini, DeepSeek, and Kimi K2.6, never won, though they sometimes survived for a long time or got many kills.

In this experiment, every AI played in a 2D world with weapons, cars, and a shrinking zone. Each model could "think," act, and write its own memory and personality files between matches, but only saw other players as letters, not by name. One big discovery was how "model alignment"—that is, how polite or helpful a model is trained to be—affected results. Claude Sonnet tried to make friends and help others, sometimes giving away its position or asking for truces, while Grok was aggressive and focused only on winning, using tactics like car ramming and detailed memory notes.

The test showed that being less "aligned," meaning less polite and more selfish, led to more wins. Grok was also much cheaper per win than Claude. Some models got many kills but never won, showing that "kills" and "wins" are not the same thing. Cost per win turned out to be important—using a cheaper but losing model can end up costing more for real tasks.

The article also shared funny moments, like car chases, mistakes, and the different ways each AI wrote its diary—Grok's was proud and full of stats, GPT 5.4 wrote like a manual, and Claude reviewed itself and tried to improve.

In the comments, many people enjoyed the creative test and said it showed real differences between AI models. Some warned that "alignment tax"—making an AI polite and safe—can limit performance in some jobs, but others pointed out that in real life, you often want AI to be safe and friendly, not just good at fighting. Some worried about Grok’s aggressive style if used outside games. Others liked that Grok found creative ways to win, showing how training style changes model behavior. Cost and use in real products were discussed, as well as the idea that benchmarks should look more like real-world tasks. Some suggested testing more models or trying cooperative games. Many agreed that "best" depends on what you want the AI to do, not just who wins.

Next, Epic Games released Lore, a new open source version control system designed for large projects and big teams, especially for code with lots of large files, such as games or digital art. Lore is easy to start—just run it locally to begin and scale up as your project grows. It is fast because it only downloads what you need, saving space with shared data, and lets you work on branches quickly. Lore keeps a tamper-evident history using special hashes and a Merkle tree, and it stores files in chunks to handle big files efficiently. It has a simple command line, APIs for Python, Go, and Rust, and supports centralized workflows. It is fully open source under the MIT license, and Epic wants the community to join in.

In the comments, many users were excited, especially because Lore is open source and handles big binary files better than Git. Some compared it to Perforce, popular in the games industry. Others asked if Lore is ready for production use, how well it scales, and if it’s friendly for non-programmers like artists. There were questions about server needs, APIs, and merge conflicts. Some worried if Epic will keep Lore open source, and many agreed that good documentation and a simple desktop client will be important for adoption.

Next, a technical story about Browser Use, a company that made their cloud browsers much faster and cheaper. Now, each browser starts in less than a second and costs only $0.02 per hour. The team wanted each browser in its own VM for safety, but regular VMs are slow and expensive, so they used Firecracker, a light VM system, running inside Amazon EC2 servers. This creates "nested VMs," which is not common and can be slow, but with careful design, they made it fast.

They used VM "snapshots" to resume new sessions quickly, tuned memory and CPU use, and built a control plane to automatically manage scaling. Their custom Chromium browser is harder for sites to detect as automated, and they use real user fingerprints to avoid anti-bot blocks. When a browser is ready, users connect over WebSocket. Most sessions now start in under 1 second.

In the comments, readers praised the technical achievement, especially running Firecracker inside EC2. Some suggested containers, but the team explained VMs are safer. Others discussed trade-offs between unikernels, VMs, and containers, or asked about costs and scaling. Security experts were interested in the isolation model, and some wondered about side channels or browser updates. Most found the work impressive, with a few ideas for going even faster.

Moving to AI, GLM-5.2 is a new large language model now ranked as the top open weights model on the Artificial Analysis Intelligence Index, scoring 51—higher than MiniMax-M3 and DeepSeek V4 Pro. GLM-5.2 has 744 billion parameters in total, with 40 billion active, and does much better in science reasoning, banking, code, and long-term planning, even matching closed models like GPT-5.5 on some tests. It has a huge context window of 1 million tokens and costs about $0.46 per task, though it uses a lot of tokens. It is open source under MIT and easy to access.

Commenters were excited about having a high-quality open weights model. Some noted the high token use could be costly, while others liked the open license and the long context window, though worried about speed. Some compared it to smaller models, saying those might be better for many tasks. Overall, people saw GLM-5.2 as a big step for open AI, even if it is not perfect for every use.

Next, let's talk about CADAM, a new open-source web app that uses AI to turn text and images into 3D CAD models. You use it in your browser with no install. You can describe what you want, or upload an image, and CADAM builds a 3D model. You can adjust size and shape with sliders, and export to STL, SCAD, or DXF for 3D printing or editing. It uses WebAssembly, React, Three.js, and OpenSCAD, connects to Claude’s API, and can pull out numbers from your prompts. The code is GPLv3 and open for you to run.

In the comments, users liked that CADAM makes CAD more accessible, even for those who don't know traditional CAD tools. Some asked about model accuracy and reliability, others wanted more export options or better AI for complex tasks. There were ideas for more features and better integration with other tools. Some saw it as good for accessibility, helping people with disabilities make models. Many liked the open license and the use of other open CAD tools.

Finally, there is now a new official HTTP method: QUERY. This method is designed for sending large or complex requests safely. GET is safe and idempotent but limited by URL length, while POST allows big bodies but is not always safe or idempotent. QUERY lets you send big data in the body, is always safe and idempotent, and works well with caching and retries. Sensitive data stays out of URLs, and new headers help clients and servers talk about query formats.

Developers in the comments were happy about this update, saying it fixes old problems with HTTP. Some pointed out that tools and servers may need updates, and adoption could be slow. Others compared QUERY to older methods like SEARCH, but most agreed QUERY is clearer. Security and naming were discussed, but the response was mostly positive.
