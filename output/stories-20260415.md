## Cybersecurity looks like proof of work now

- 原文链接: [Cybersecurity looks like proof of work now](https://www.dbreunig.com/2026/04/14/cybersecurity-is-proof-of-work-now.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47769089)

The article talks about new AI models, like Anthropic’s Mythos, that are very good at finding security problems in computer systems. Mythos is so powerful that only important software companies can use it for now, to help them fix issues before attackers can use this AI.

The main point is that security now feels like a “proof of work” problem, similar to cryptocurrency mining. To protect a system, you need to spend more computer power (measured in tokens) looking for bugs than attackers do trying to break it. For example, Mythos runs cost $12,500 for each try, and the model got better results the more tokens it used. If attackers keep spending money to find new bugs, defenders have to spend even more to stay safe.

The article explains that this changes how we think about cybersecurity. It’s not just about being smart or clever anymore. It’s about who can spend more on AI runs to find weaknesses. This could make open source software safer, because many companies might spend money to test and protect the same code. But it also means that popular open source projects become bigger targets, since hackers may spend more to attack them, too.

The author also talks about how developers’ work changes. Instead of just coding and reviewing, there might be a new hardening phase, where AIs are used to search for problems until the budget runs out. Human skill matters most for creating and improving code, but money limits how much you can harden it.

In summary, making code secure could get much more expensive. The main cost is no longer just time or people, but how many AI tokens you can afford to spend.

Commenters on Hacker News have many views. Some agree that security is now about money and resources, not just skill. They worry this could create an arms race—big companies with deep pockets might have an advantage, leaving small teams and indie projects behind. Others point out that attackers can share successful attacks, so defenders might need to spend even more to keep up.

Some people hope open source communities will work together to protect shared code, but others are concerned that this high cost could push projects to close their code or stop developing. A few users say this “proof of work” model is wasteful, like crypto mining, and wish for smarter, not just more expensive, ways to improve security.

Another group wonders if this changes the value of software dependencies—should we reduce dependencies to lower risk, or trust big open source projects because they’ll get more protection? Some are optimistic that costs will drop as AI gets cheaper, but others think the cost of finding every bug will always be high, because attackers only need to get lucky once.

Overall, the discussion is lively, with both excitement and worry about the future of cybersecurity in an AI-driven world.

## PiCore - Raspberry Pi Port of Tiny Core Linux

- 原文链接: [PiCore - Raspberry Pi Port of Tiny Core Linux](http://tinycorelinux.net/5.x/armv6/releases/README)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47784244)

PiCore is a special version of Tiny Core Linux made for the Raspberry Pi. Tiny Core Linux is a very small and flexible Linux system that lets you build your own custom setup.

PiCore runs completely in the computer’s memory (RAM). After it starts, it does not use the SD card much, so the system is always clean after every restart. By default, apps and files are downloaded from the internet and are stored only in RAM. When you reboot, all changes are gone, and you start fresh every time. If you want to keep some changes, you can use "Mounted Mode." In this mode, you make a second partition on the SD card. Then, you can save apps and files, and choose what to back up. PiCore comes in different images, like a base system or with pre-installed apps such as SSH or X (for a graphical desktop).

To use PiCore, you download a ZIP file with an SD card image. You write this image to your SD card using tools like “dd” on Linux or “Win32 Disk Imager” on Windows. The SD card needs at least one partition for the system and, if you want to save changes, a second Linux partition. You can expand the size of this storage to fit more apps and files. PiCore uses compressed swap memory by default, but you can add more swap space if you need it, especially for big tasks.

There are special boot codes you can use to control how PiCore starts. The default user is called “tc,” and you do not need a password (except for the SSH version). The project is supported by a community forum, and there is a free book called “Into the Core” to help you learn more.

In the comment section, many people liked that PiCore is lightweight and fast. Some users said it is great for old or simple Raspberry Pi models because it uses little memory and storage. Others pointed out that running fully in RAM means the system is always clean and safe from SD card corruption, which can happen if power is lost suddenly. A few commenters worried that beginners might find the setup confusing, especially when making and resizing partitions. Some people compared PiCore with other small Linux systems for the Raspberry Pi, saying PiCore is one of the smallest. There were also stories from users who used PiCore for special projects, like building network tools or running simple servers. Some wished for easier ways to install software or make backups, while others liked how manual and simple it is. Finally, a few developers talked about how using PiCore helped them learn more about how Linux works under the hood.

## Cal.com is going closed source

- 原文链接: [Cal.com is going closed source](https://cal.com/blog/cal-com-goes-closed-source-why)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47780456)

Cal.com, a popular scheduling software, will stop being open source and is moving to closed source because of new security risks from AI. The company says AI tools can now easily scan open source code and find ways to attack, which puts user data at risk.

In the past, hackers had to spend a lot of time and skill to find problems in code. Now, AI makes it much faster and easier to find these problems, especially when the code is open and public. Cal.com’s team saw that new AI security startups are using these tools to quickly discover weaknesses in many apps. This means it is harder to keep things safe, and there is no single way to know if code is really secure anymore.

Cal.com’s main worry is that being open source is now like giving attackers a map to all your secrets. They think closing their code will make it harder for bad actors to break in and steal customer data. They are not happy about this choice because they believe in open source, but they feel they must protect user privacy first.

To help the community, Cal.com will still release a version called Cal.diy under the MIT license. This version is for hobbyists and developers who want to learn or experiment, but it is different from their main product. The company says their main code has changed a lot, especially in parts like authentication and data handling, to be more secure.

They also point out a recent example: AI found a very old bug in the BSD kernel, a system known for security, and made working attacks in just hours. Cal.com says this shows how dangerous AI-powered attacks can be, even for strong open source projects.

In the comments, many people are sad and disappointed by this move. Some say open source makes software better and safer because more people can check the code for mistakes. Others understand Cal.com’s fear, saying that AI is making attacks much easier, and companies have to protect user data at all costs.

A few commenters argue that closing the source will not stop skilled attackers, since they can still find bugs in closed code using other tools. Some users warn that customers may lose trust or stop using Cal.com because they liked it for being open source. Others think this is just an excuse, and the real reason might be business, not security.

There are also voices saying this is the start of a bigger trend, where more companies will close their code because of AI risks. Some developers feel let down, especially if they helped build or support Cal.com while it was open source. Others suggest that the open Cal.diy version is a good compromise, while some worry it will become old and unsupported.

Overall, the community is split. Some see the move as smart and needed, while others think it goes against the spirit of open source and may not really solve the problem.

## Fix monitor that goes black, off or blinks due to static electricity in chair

- 原文链接: [Fix monitor that goes black, off or blinks due to static electricity in chair](https://aalonso.dev/blog/2023/how-to-fix-monitor-that-goes-black-off-due-to-static-electricity-in-chair/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47738081)

This article is about a strange problem: a monitor that turns off, blinks, or goes black when the user moves in their office chair, because of static electricity. The author works from home, using a MacBook with a 4K monitor and an Ikea Markus chair, and noticed the problem happened often—especially when standing up or moving in the chair.

The main problem is static electricity building up on the chair. When the author touches something metal, like the MacBook, they get a shock, and sometimes the external monitor turns off, blinks, or doesn’t reconnect. The issue is not only static shocks—sometimes the problem happens even without touching anything. After some research, the author learned that gas lift office chairs can create electromagnetic interference (EMI) spikes when someone sits or stands. These EMI spikes mess up video cables, making the monitor blink or go black.

The problem is worse with DisplayPort cables, USB-C to DP adapters, or Thunderbolt cables. The author found that many people online have the same problem, and there are support pages and research papers that explain it. To fix the static electricity, the author grounded the chair with a metal chain touching the floor. This helped reduce the shocks a lot. For the EMI spikes, the author added ferrite rings to the video cables, as suggested by support sites. This helped reduce, but not completely stop, the monitor problems. Standing up slowly helps too.

In the Hacker News comments, some people shared similar experiences. A few said they had the same problem with plastic chairs or dry air. Others said grounding the chair worked for them, or that using a humidifier in the room helped stop static buildup. Some people found that using better-shielded cables fixed the issue. Others were surprised that office chairs could cause computer problems at all. One commenter joked about “dangerous chairs” in tech jobs. Another person warned that static can damage electronics, not just make monitors blink. Some users wanted more technical details about EMI and how ferrite rings work. Others liked the simple, practical solution with the metal chain. A few people said they never had this problem and guessed it depends on climate or chair type. Overall, the comments showed many people have faced this odd issue, but there are several ways to make it better.

## Want to write a compiler? Just read these two papers (2008)

- 原文链接: [Want to write a compiler? Just read these two papers (2008)](https://prog21.dadgum.com/30.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47776796)

The article says writing a compiler seems very hard because most books are too big and complex for beginners. It explains that these books cover too many details, which can make people feel lost and not know where to start.

The author suggests a better way is to read just two things. First, Jack Crenshaw’s “Let’s Build a Compiler!” series, which breaks compiler writing into small, simple steps. Crenshaw uses easy examples in Pascal, and there are also versions in C and Forth. His style is clear and helps you build a real, working compiler, even if it’s basic.

However, Crenshaw’s guide skips one important thing: it does not use an internal tree structure (called an Abstract Syntax Tree, or AST) to represent programs. This makes the compiler less flexible. But if you use a higher-level language like Python, Lisp, or Haskell, making and using trees is much easier.

The second paper suggested is “A Nanopass Framework for Compiler Education.” This paper explains that a compiler can be built as many simple steps, each changing the program a little. Each step, or “pass,” is kept small and easy to understand. The authors use Scheme, which is good for handling tree-like data.

The article’s main idea is you do not need to start with big, heavy books. With these two resources, you can make a simple compiler and understand the core ideas. Only after trying this should you read bigger books like the “Dragon Book.”

In the Hacker News comments, many people agree that compiler books are too big and can scare beginners. Some say they learned a lot from Crenshaw’s tutorials and that building a small compiler really helped them understand programming better. Others mention that modern languages make handling trees much easier, so the missing AST step is not a big problem anymore.

A few commenters prefer the “Nanopass” approach, saying splitting work into many simple passes keeps things clear and less confusing. Some warn that real-world compilers can get very complex, but learning through small examples is still useful.

There are also people who think learning compilers is not needed unless you want to become a language or tool developer. Another group says it’s valuable for understanding how computers work, even if you never build a real compiler.

Some share links to other simple compiler tutorials they like. Others debate whether the Dragon Book is still useful today, with mixed opinions. Overall, most agree that starting small and hands-on is better than reading big theory books.
