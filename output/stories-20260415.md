## Cybersecurity looks like proof of work now

- 原文链接: [Cybersecurity looks like proof of work now](https://www.dbreunig.com/2026/04/14/cybersecurity-is-proof-of-work-now.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47769089)

The article talks about a new AI model, Mythos, that is very good at finding security problems in computer systems. Anthropic made Mythos, but they did not release it to the public because it is too powerful. Instead, only important software companies can use it for now.

The AI Security Institute tested Mythos and said it is better than other AI models at breaking into networks. In their test, Mythos finished a hard 32-step attack that usually takes a human 20 hours. It did this in 3 out of 10 tries, using a lot of computer tokens each time. Each test cost about $12,500, and the more tokens the AI used, the more bugs it found. There was no sign it was running out of things to find, which is worrying.

The article says that security now feels like “proof of work,” a system from cryptocurrency where you succeed by spending more computer power or money than your competitor. For software to be safe, you must spend more tokens (money) checking for bugs than an attacker will spend trying to break in. It is not about being clever; it is about outspending the attacker.

The writer points out that open source software is still important. If a big company spends a lot of tokens checking an open source library, it might be safer than small private code. But, because open source is used by many people, attackers will also target it more, making it a big risk.

Another point is that coding with AI might soon have three steps: building the software, reviewing the code, and then “hardening” it by running these expensive security checks. Before, security reviews were rare. Now, with AI, you can do them all the time if you pay for it.

In the comments, many people say this is a big change for security. Some think it is bad because only rich companies can afford good security if it is just about who can spend more. Others ask if this will really work, or if attackers will always find new ways to get around defenses.

Some developers worry that this will make software more expensive, and small teams will be left behind. Others point out that making all software secure is almost impossible, and attackers only need to find one weakness.

A few people think this model is fair, because attackers also have to spend money. Some ask if there will be new business models, like “security as a service,” where you pay for ongoing checks. Others worry about the arms race: as attacks get smarter, companies must spend more and more. There is also talk about open source projects needing more funding to keep up with security costs.

Finally, some say that no matter how much money is spent, there will always be new bugs, and perfect security is not possible. Others hope that better AI will at least make it harder for attackers, even if the fight never ends.

## Keycard – inject API keys into subprocesses, never touch shell env

- 原文链接: [Keycard – inject API keys into subprocesses, never touch shell env](https://www.keycard.studio/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47787643)

Keycard is a new tool for developers to manage API keys safely and easily on their own computers. The main idea is to stop using .env files, password managers, or notes apps to handle secrets, and instead use one local, encrypted vault.

Keycard lets you save API keys quickly from your clipboard. It encrypts each key and stores them in a local database. You can group keys into profiles, like “dev” and “prod,” and pick which profile to use when you run a command. When you start a process with Keycard, it injects the right keys as environment variables only for that process. The keys never go into your main shell, so they disappear when the program exits. All data stays on your computer; nothing is sent to the cloud or any server. Security uses strong encryption (XChaCha20-Poly1305), and the vault is useless without your password. Keycard works from the command line, fits into developer workflows, and supports clipboard shortcuts for quick-saving secrets. There are features for syncing, snapshots, and team use planned for the future. The core tool is free and open source, but “Pro” and “Team” versions will have more features for a price.

In the comments, some people really like the idea of a local-first, CLI-friendly secrets manager. They say it fixes real problems with .env files and avoids mistakes like checking secrets into Git. Others are not sure about trusting a new tool for important keys, especially since it is only for macOS right now. Some users want Linux and Windows support soon. A few people point out that password managers are not great for developer secrets, so a tool built for code workflows is helpful. There is also debate about whether local-only is good or if cloud sync is needed; some like the safety of “no cloud,” others need syncing across devices. Security experts in the thread ask about how the encryption and backups work. Some comment that forgetting your master password could mean losing everything, but they like that this also keeps things safe from attackers. Overall, the commenters think Keycard is a promising tool, but they want to see how it works in practice, and hope for more platform support and maybe open standards for secret management.

## I made a terminal pager

- 原文链接: [I made a terminal pager](https://theleo.zone/posts/pager/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47786164)

This article is about someone who made their own terminal pager, a tool for viewing lots of text in the command line. The author explains why terminal pagers are important and how they work, especially for developers who often read big text files, logs, or command outputs.

Most terminals show text in a grid, and when there’s too much text to fit, a pager lets you scroll and search through it. Programs like `less` are the default pagers, but there are others like `bat` or `most`. You can set your favorite pager with the `PAGER` environment variable, so commands like `git` or `man` use it.

The author builds terminal user interfaces (TUIs), which are like full-screen apps inside the terminal. These apps often have parts that act like mini pagers, letting you scroll, search, and select items. To help with this, the author created a reusable `viewport` component in Go. This component can handle resizing, scrolling, searching, and even special text styles and Unicode characters.

Handling Unicode is tricky because some characters take up more space in the terminal. For example, the "✨" emoji is two cells wide, while normal letters are just one. The author explains how their component figures out the width of each character, even when characters combine, like “é”.

Searching is a big part of the tool. You can do exact matches, regex searches, or case-insensitive searches with simple shortcuts. You can also choose to see only the matching lines or the matches with some extra context. Selection is supported too: you can select an item and do things like zoom in for more details.

The author’s pager, called "lore," is built using the Bubble Tea framework and uses their `viewport` component. It doesn’t have every feature that `less` has, but it works well for the author’s daily needs and is easy to improve. They even set it as their main pager, so all programs use it.

In the comments, some people praise the project for being a good learning exercise and for sharing the details about handling Unicode and terminal quirks. Others talk about how many pager tools already exist, like `less`, `bat`, and even custom setups with `tmux` or editors like `vim`. Some users say that while building your own tools is fun, it’s easy to miss small features that make established tools powerful, like advanced search or mouse support.

Several developers mention that making your own pager helps you understand how terminals work, especially with tricky things like ANSI codes and Unicode. A few comment that the Go language is a good choice for this type of tool, while others suggest Rust or even C for better performance or lower memory use.

There are suggestions to add more features, such as better mouse support or smarter search, and some people mention problems they’ve had with existing pagers, like slow performance on large files. A few users say they’ll try "lore" out of curiosity, but most plan to stick with `less` unless they see a big reason to switch. Some also discuss the growing interest in custom TUIs and how making core components, like a viewport, can help build better and more personal tools.

## Retrofitting JIT Compilers into C Interpreters

- 原文链接: [Retrofitting JIT Compilers into C Interpreters](https://tratt.net/laurie/blog/2026/retrofitting_jit_compilers_into_c_interpreters.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47777897)

This article talks about how you can make C interpreters much faster by adding a JIT (Just-In-Time) compiler with very little extra code, using a system called yk. The yk project lets you take a C interpreter (like the ones for Lua or Python) and, with only small changes, turn it into a JIT VM that runs programs faster while keeping full compatibility.

The main point is that C interpreters are simple but slow. JIT compilers make things faster, but they are hard to write and even harder to keep up-to-date when the programming language changes. Usually, to get a JIT, you have to rebuild or rewrite the interpreter, which can break compatibility with the original language or libraries. This is why many JIT projects for popular languages fail or become outdated.

The yk system uses "meta-tracing", which means it can watch what the C interpreter does while running a program and then make fast machine code for those parts. It does this mostly automatically. The article gives an example with Lua: by changing and adding less than 500 lines, they turned the normal Lua interpreter into a JIT version that can be upgraded easily when Lua changes. The performance boost is about 2x on average, sometimes more, sometimes less.

The article explains technical details like tracing, meta-tracing, how to mark hot loops, and how yk uses LLVM (a compiler tool) to record and optimize traces. It talks about optimizations like "promotion" (making values constant in traces) and "idempotent" functions (which always return the same result), which help yk make better machine code. The system also uses "backwards code generation"—writing machine code from the end of a trace to the start—which helps with register use and removes useless code.

yk is still early-stage software, with some missing features and only works on x64 right now. But the main idea is new: you can add a JIT to almost any C interpreter fast, get better speed, and stay compatible with new language updates.

In the Hacker News comments, many people are excited about this idea. Some say it could help smaller or less popular languages get better performance without a huge team. Others like that yk makes it much easier to keep up with language changes—something hand-written JITs struggle with. People mention that while the speedup is not as big as with some older, hand-tuned JITs (like LuaJIT), having an easy, maintainable JIT is very valuable.

A few commenters point out that meta-tracing can have downsides. They recall how Mozilla’s TraceMonkey (a tracing JIT for JavaScript) was dropped because it was hard to maintain for all code patterns. Some wonder if there are limits to how fast yk can go, or if more complex languages will be harder to support.

Some developers ask about using yk with other interpreters, like CPython, and what needs to change in the interpreter code. Others discuss technical details, like how yk handles deoptimization (going back from JIT code to the interpreter when needed), and whether yk could work on other platforms beyond x64.

Finally, several readers thank the author for a clear write-up and for sharing both wins and challenges. They say the project fills an important gap: it trades a bit of speed for big wins in ease of use and long-term support, which is often what matters most for real-world language projects.
