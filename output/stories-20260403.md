## What changes when you turn a Linux box into a router

- 原文链接: [What changes when you turn a Linux box into a router](https://patrickmccanna.net/7-configuration-changes-that-turn-a-multi-homed-host-into-a-switch-router/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47586885)

This article explains how you can turn a regular Linux computer into a router or switch. It lists seven changes you need to make to do this.

First, you must enable IP forwarding in the Linux kernel. This lets the computer send network packets from one network to another. Next, you create a network bridge to connect wired and wireless interfaces, so both can talk as if they are on the same network. Then, you set up firewall rules using nftables, which control what traffic is allowed or blocked. You also enable conntrack, which tracks connections and helps the firewall understand which packets belong to which sessions. After that, you configure NAT (Network Address Translation) and masquerading so devices on your local network can access the internet using the computer’s public IP address. You need to run a DHCP and DNS server—usually with dnsmasq—so other devices can get IP addresses and find the router. Finally, you use hostapd to turn the wireless card into a WiFi access point, letting wireless devices connect.

The article describes how packets move through the Linux networking stack, and how different tools and kernel features work together. It explains the roles of hooks in the kernel, how nftables rules are attached to these hooks, and how stateful firewalling with conntrack allows simple rules to manage complex traffic. It also discusses why bridges work differently for wired and wireless interfaces, and how tools like dnsmasq and hostapd handle key services. The article shows that with these changes, a Linux system can act just like a normal home router, handling both wired and wireless clients.

In the Hacker News comments, people shared different views. Some said it’s fun and educational to build your own router using Linux, especially on small computers like the Raspberry Pi. Others pointed out that using a Linux router gives you more control and can be more secure than cheap consumer routers. Some commenters warned that managing your own router means you need to keep up with security updates and handle problems yourself. A few mentioned performance: a Linux-based router might not be as fast or efficient as special hardware, especially with lots of users or high network speeds. Others shared tips for simplifying the setup, like using distributions made for routing, such as OpenWRT or pfSense. Some people said they learned networking by building projects like this, while others prefer ready-made solutions because they “just work.” There were also discussions about the complexity of nftables versus iptables, and some suggested extra features like intrusion detection or advanced monitoring. Overall, commenters agreed that turning a Linux box into a router is a great learning experience, but it’s important to understand both the benefits and the challenges.

## We replaced RAG with a virtual filesystem for our AI documentation assistant

- 原文链接: [We replaced RAG with a virtual filesystem for our AI documentation assistant](https://www.mintlify.com/blog/how-we-built-a-virtual-filesystem-for-our-assistant)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47618223)

This article talks about how a team changed their AI documentation assistant. Before, they used RAG (Retrieval-Augmented Generation) to answer questions, but it had limits. RAG could only find small pieces of text, so answers that needed info from many places did not work well. Users often wanted exact code or details, but the system could not always find it. The team wanted the assistant to work more like a developer exploring files with commands like grep, ls, or cat.

They saw that others were starting to use filesystems as the main way for AI agents to find information. At first, they tried real filesystems in sandboxes, but those were slow and cost a lot of money. Starting a new session took about 46 seconds and would cost about $70,000 a year if used often. This was too much, so they looked for a faster and cheaper way.

Their solution was to make a virtual filesystem called ChromaFs. Instead of using real files, ChromaFs gives the feeling of a filesystem by using their existing Chroma database. This means UNIX-like commands are turned into database queries. Sessions now start in 100 milliseconds, and there is almost no extra cost.

ChromaFs uses just-bash, a TypeScript tool that lets you run bash commands in code. It has a way to plug in new filesystems, so ChromaFs connects bash commands to the database. When a session starts, the whole file tree is loaded as a gzipped JSON file. This lets commands like ls or cd work in memory, making them very fast.

The system also handles access control. Some files are only for certain users. The virtual filesystem checks the user's rights and only shows files they can see. This is easier than managing Linux permissions in real sandboxes.

When users read a file, ChromaFs puts together all the file chunks from the database in order. The result is cached for speed. Some files, like big API specs, are fetched only when needed. The system is read-only, so users cannot change anything—this keeps everything safe and simple.

For searching, ChromaFs changes grep commands into database searches first, then finishes the search in memory. This two-step process makes searches very fast. Now, their documentation assistant serves many users quickly, with almost no extra cost or new infrastructure.

In the Hacker News comments, many people liked the creative solution. Some said this is a smart way to use existing systems and make things very fast. Others thought it was interesting to see database queries used like a filesystem. A few worried about edge cases and if this would work for very big or changing docs.

Some commenters shared stories about trying similar ideas in the past, saying that virtual filesystems can help in many tools, not just AI assistants. Others asked about security and if users could break out of the virtual system, but the author said that read-only access keeps things safe. Some liked that caching makes repeated searches fast, while a few wanted to know how well it scales with more users.

There were also questions about how hard it would be to add write support or handle more complex file types. Some thought the system might inspire new ways to organize and search documentation in other apps. Overall, people thought this was a clever way to make an AI assistant faster and cheaper.

## Go on Embedded Systems and WebAssembly

- 原文链接: [Go on Embedded Systems and WebAssembly](https://tinygo.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47629116)

This article is about TinyGo, a special Go compiler that lets you use the Go programming language on small devices and in WebAssembly. TinyGo is made for running Go on things like microcontrollers and also for making small, fast WebAssembly programs.

The TinyGo compiler is built on top of LLVM, which helps it make very small and efficient code. It supports more than 100 kinds of microcontroller boards, including popular ones like the BBC micro:bit and Arduino Uno, as well as more advanced chips from companies like Nordic Semiconductor and ST Microelectronics. TinyGo can also create WebAssembly code, which works well for web browsers and for server or edge computing where size and speed are important. This makes it possible to use the Go language in places where regular Go would be too big or slow.

The article shows a simple example program that makes an LED blink using TinyGo. It includes code for configuring and turning the LED on and off, with sleep timers to make it blink. There are also power usage numbers for the Arduino Uno board when running this code, though these are only estimates. The TinyGo website offers a playground to try TinyGo online, so you can experiment with code in your browser without any setup. There are links to documentation, a tour, community channels, and the TinyGo source code on GitHub.

From the top comments, people are excited that Go can now run on embedded systems and WebAssembly, since Go is known for being easy to use and productive. Some users share that TinyGo lets them write code for hardware projects in a language they already know, instead of learning C or C++. Others point out that while TinyGo is not 100% compatible with all Go features, it works well for many small projects. A few commenters are happy about the WebAssembly support, saying it makes it easier to share code between servers and the browser. There are questions about how TinyGo compares to other tools like MicroPython or CircuitPython, with some saying Go feels faster or more robust for their needs. Some users mention that TinyGo’s smaller binary size is helpful for fitting code onto small chips. A few warn that not all Go libraries will work, and you may need to write simpler code. Some people ask about debugging and tool support, and others reply that it is improving but not as strong as for C. There is also discussion about the power usage numbers and how they are estimated. ## Samsung Magician disk utility takes 18 steps and two reboots to uninstall

- 原文链接: [Samsung Magician disk utility takes 18 steps and two reboots to uninstall](https://chalmovsky.com/2026/03/29/samsung-magician.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47567222)

Samsung Magician is a disk utility for managing Samsung SSDs, but uninstalling it from a Mac is very hard and frustrating. The writer needed the software to set a password on a portable drive, but when it did not work, they tried to remove it and found the process was a nightmare.

First, there is no simple uninstall option or drag-to-trash feature. The only way is to find a hidden script deep inside the app’s folders. Running this script causes hundreds of errors and does not remove anything. After that, the author tried to manually delete every folder and file related to Samsung Magician, using command-line tools like rm -rf and find. Still, many files stayed on the system, including protected kernel extensions that regular commands could not remove.

To fully remove these stubborn files, the author had to reboot into Mac Recovery Mode, disable System Integrity Protection (SIP), delete the files, and then reboot again to re-enable SIP. All together, it took 18 steps and two special reboots, just to uninstall a program that did not even work for its main purpose.

The article also points out many strange things inside Samsung Magician. For example, the app includes hundreds of PNG files just to animate a spinning circle, uses the big Electron framework just to show simple charts, has special fonts, many languages, and even advertises with banner images. This makes the app very large and complicated for a simple disk tool.

In the Hacker News comments, many people were shocked at how difficult the uninstall process was. Some shared their own stories about trouble removing other disk utilities or printer software, saying that “bloatware” is common with big companies. Others joked about the huge number of PNGs and how silly it is to use Electron for a utility app. A few developers explained that Electron and Squirrel make apps easier to build and update, even if they are heavy.

Some commenters pointed out that MacOS security, like SIP, is important but makes it hard to clean up after bad software. Others suggested that Samsung should provide an official uninstaller or use the Mac’s normal app system. Several people said this kind of experience makes them avoid vendor tools and prefer open-source or built-in utilities.

A few users warned that tools requiring kernel extensions are always risky and should be used with care.
