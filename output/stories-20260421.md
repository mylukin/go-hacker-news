## The Vercel breach: OAuth attack exposes risk in platform environment variables

- 原文链接: [The Vercel breach: OAuth attack exposes risk in platform environment variables](https://www.trendmicro.com/en_us/research/26/d/vercel-breach-oauth-supply-chain.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47851634)

The article explains a big security problem at Vercel, a company that helps people deploy web apps. Attackers broke into Vercel’s systems by using a third-party OAuth app that Vercel employees had trusted. Once inside, the attackers could read environment variables, which often store passwords and secret keys for many different services.

The main issue was how Vercel managed these secrets. If developers didn’t mark a variable as “sensitive,” it was not well protected. Many developers didn’t use the “sensitive” option, so lots of secrets were left unprotected. Attackers could see these secrets and possibly use them to access databases, cloud accounts, payments, and many other services. This problem is not new: other companies like Codecov and CircleCI had similar breaches where attackers stole secrets from platform environment variables.

The attack started in June 2024, but Vercel only told users about it in April 2026. That means the attackers had access for almost two years without being caught. One customer even got a warning about a leaked secret nine days before Vercel made the news public.

The article also talks about how OAuth, which is supposed to help apps talk to each other safely, can be risky. If attackers break into a trusted OAuth app, they can move around inside a company’s systems without needing passwords. This makes it hard for security teams to notice an attack.

Another big point is that attackers might be using AI tools to move faster and smarter than before. Vercel’s CEO believes this attack moved unusually quickly, possibly because of AI. This is a growing concern in the security world.

To fix these problems, the article suggests companies should stop storing important secrets as plain environment variables. Instead, use special tools called secret managers that keep secrets safe. Companies should also check which OAuth apps are trusted, rotate secrets often, and act fast when they get warnings about leaked credentials.

In the Hacker News comments, many users said this breach was not surprising. They pointed out that storing secrets as environment variables is common but risky, and that most developers forget to mark variables as sensitive. Some said Vercel’s default settings make it too easy to make mistakes. Others felt this is a problem across the whole industry, not just at Vercel. A few people praised the article for being clear and detailed about the timeline and technical steps.

There was debate about whether OAuth is too dangerous to use, or if the main problem is how companies manage trust and secrets. Some developers shared that their own companies also have weak controls on OAuth apps and secrets but it’s hard to fix because it slows down work. A few comments were worried about the long time (22 months) it took for Vercel to detect and disclose the breach, saying this shows how hard these attacks are to catch.

Some users thought that companies should take leaked-credential alerts much more seriously and act on them right away. Others felt that using AI to help attacks will make things worse in the future, so companies must change how they protect secrets now. Finally, several commenters hope that this breach will push more teams to use proper secret managers and review OAuth apps often.

## I don't want your PRs anymore

- 原文链接: [I don't want your PRs anymore](https://dpc.pw/posts/i-dont-want-your-prs-anymore/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47854051)

This article is about a software maintainer who says they do not want to accept pull requests (PRs) from others anymore. The main reason is that reviewing code from strangers is risky and takes more time than just making the changes themselves, especially now that AI tools can write code quickly.

The author says it is hard to trust code from people they do not know, because someone could hide bad code inside a PR. Also, everyone has their own style and way of writing code, so merging changes often leads to extra work fixing style or structure. There is also a lot of back-and-forth: reviews, fixing conflicts, and waiting for replies. In the past, getting a good PR was worth it because writing code took more effort. Now, with large language models (LLMs) helping, writing code is much faster and easier for maintainers. The LLM can follow the maintainer’s style and does not try to sneak in bad code. The maintainer can just ask the LLM to make changes, review the result, and move on, without waiting for someone else.

The author also says that writing code is no longer the hard part. The real challenges are understanding the project, designing good solutions, and reviewing code. PRs from others do not help with these tasks, so the maintainer prefers to do the coding themselves (with AI help).

Instead of sending PRs, the author suggests other ways to help: give feedback about what works or what is confusing, discuss ideas to find better solutions, report bugs clearly, or show small code examples or prompts that explain your idea. Reviewing code and pointing out mistakes is helpful too. If you want to change something big, just fork the code and do it yourself—there is no need to ask first. Maybe both versions will teach something new.

In the comments, some people agree that reviewing unknown code is hard and risky, especially with more tools to help write code faster. Others worry that this attitude could make open source less friendly or limit new contributors from learning. A few say that open source is about sharing and that forking is not always easy, especially for beginners. Some maintainers say they feel the same way—the time spent reviewing PRs is not worth it now. Others argue that PRs are still useful for growing a project and building a community. Some point out that AI tools are not perfect and still make mistakes, so human review is needed anyway. A few remind everyone that open source is a gift, and it is up to maintainers to decide how they want help. Overall, the discussion shows that people have many different feelings about how AI and new tools are changing how we work together on code.

## 10 years: Stephen's Sausage Roll still one of the most influential puzzle games

- 原文链接: [10 years: Stephen's Sausage Roll still one of the most influential puzzle games](https://thinkygames.com/features/10-years-of-grilling-stephens-sausage-roll-remains-one-of-the-most-influential-puzzle-games-ever-created/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47814874)

Stephen’s Sausage Roll is a puzzle game that came out 10 years ago, and it changed how people think about puzzle games. The article talks about why this game is so important and how it still influences game designers today.

The game looks simple: you use a big fork to roll sausages onto grills, but the puzzles are very hard from the start. It uses only a few things—player, fork, sausage, grill, block, and ladder—but these make for deep, surprising gameplay. The rules never change, but the game keeps finding new ways to challenge you. The difficulty is tough but always fair, which makes you want to keep trying.

Stephen Lavelle, who made the game, also created PuzzleScript, a tool for making grid-based puzzle games. This tool and Stephen’s Sausage Roll together have inspired many other puzzle games and developers. The article shares quotes from different game developers who say this game pushed them to make their own games, change their ideas, and see how much you can do with simple rules.

One developer said it was like “the Ramones in 1976” for puzzle games—simple, raw, and exciting. Some developers were not even puzzle fans before, but after playing this game, they wanted to make their own sokoban-style games (where you push things around on a grid). Many of today’s puzzle games, including “A Monster’s Expedition,” “Patrick’s Parabox,” “Kine,” and “Bonfire Peaks,” were inspired by Stephen’s Sausage Roll.

In the comments, some people say the game is too hard and they could not finish it, but they still respect it for its design. Others praise the game for being clever and fair, even if it made them frustrated sometimes. A few mention they did not like the graphics, but they admit the gameplay is top-level. Many say the game made them think differently about puzzles and inspired them to try other hard games. Some wish more puzzle games were this creative and strict with their rules. There are also people who thank the game for helping them get into making games, not just playing them. A few mention the price was high but felt it was worth it for the challenge and new ideas. Some share tips for new players, like not giving up and taking breaks when stuck. Overall, most agree that Stephen’s Sausage Roll is a special game that changed puzzle games for the better.

## Laws of Software Engineering

- 原文链接: [Laws of Software Engineering](https://lawsofsoftwareengineering.com)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47847179)

The article lists well-known “laws” and principles that guide software engineering. It puts together 56 ideas—like Conway’s Law, the Boy Scout Rule, and YAGNI—that help people build better software and manage teams. 

Each law covers a key topic. For teams, Conway’s Law says your product will look like your company’s structure. Brooks’s Law warns that adding people to a late project makes it later. The Bus Factor reminds us not to rely too much on a few people. The Peter Principle and Dilbert Principle deal with problems in company management. For planning, Parkinson’s Law says work fills the time you give it, and Hofstadter’s Law says everything takes longer than you think. Goodhart’s Law warns that targets can ruin useful measures. For coding, the Boy Scout Rule means always leave code better than you found it. The DRY Principle says don’t repeat yourself in code. KISS tells us to keep things simple, and YAGNI says don’t add features you don’t need. The Law of Leaky Abstractions notes that no abstraction can hide all details. CAP Theorem explains limits in distributed systems. The Sunk Cost Fallacy warns against continuing bad projects just because you already spent time or money. Linus’s Law says with enough eyes, bugs are shallow.

People in the comments liked having all these laws in one place. Some said this is a good cheat sheet for new developers. Others pointed out that not every law fits every situation—sometimes you need to break a rule. A few said some “laws” are really just opinions, not hard facts. There was debate over Conway’s Law and if changing company structure really changes software. Several commenters shared their own favorite laws, or wished more context was given for each one. Some said it’s easy to misuse these laws to argue for bad decisions. Others warned that knowing the law is not the same as using good judgment. Many agreed that these principles are helpful reminders, but real-life software is always more messy. Some liked the links for reading more about each law, and a few planned to share the site with their teams. Overall, people found the collection useful, but said it’s important to think, not just follow rules.

## Zindex – Diagram Infrastructure for Agents

- 原文链接: [Zindex – Diagram Infrastructure for Agents](https://zindex.ai/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47854116)

Zindex is a tool for making and managing diagrams, built mainly for software agents (like AI) to use. It helps agents create, update, check, and show diagrams in a way that keeps every change, like a database for diagrams.

The main idea is that diagrams are not just pictures but are stored as structured data. You can see every version and change over time. Agents define nodes and edges (points and connections), and Zindex figures out where everything goes—no need to drag things around by hand. It uses a special layout engine (Sugiyama-style) to arrange diagrams automatically.

You can make changes to just one part of a diagram without redoing the whole thing. Zindex gives each part a stable ID, so updates are simple and don’t break things. It works with different diagram styles (SVG, PNG) and themes (clean, dark, blueprint, sketch). The process is always the same: check the diagram, fix problems, arrange it, then show it. You always get the same result for the same input.

Zindex is made for teams or many agents working together. It supports revision history, diffs, and can be used for system architecture, workflows, database diagrams, org charts, and more. It is not just for drawing shapes; it understands the meaning of what you are drawing. It has built-in validation rules, supports secure storage (PostgreSQL), and can handle things like user accounts and rate limiting.

In the Hacker News comments, some people liked the idea of treating diagrams like code with version history and validation. They said this could make it easier to manage complex system diagrams over time. Others wondered how easy it is to use for humans, not just agents, and asked about the user interface. A few compared it to tools like PlantUML or Mermaid, and asked what makes Zindex different. Some liked that it supports collaboration and stable IDs for small edits. Others thought it might be too complex for simple uses, but could be great for big teams or AI workflows. A few users wanted to see more real-world examples or open-source options. Some were curious if it can import from or export to other diagram tools. Overall, people saw promise in the idea, especially for teams, but wanted to see how it works in practice.
