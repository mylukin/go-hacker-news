## Just Use Postgres for Durable Workflows

- 原文链接: [Just Use Postgres for Durable Workflows](https://www.dbos.dev/blog/postgres-is-all-you-need-for-durable-execution)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48313530)

This article says you do not need a special system to manage durable workflows—just use Postgres. Durable workflows are when your program saves its progress often, so if it crashes, it can start again from the last saved point. Usually, people use tools like Temporal or Airflow, which have a big central system (an orchestrator) to manage all of this.

The article argues that having a separate orchestrator is too complex. Since workflows are about saving steps in a database, why not let the database (like Postgres) handle everything? In this design, your application servers write workflow steps straight into a Postgres table. Servers check the table for new tasks, do the work, and save results. If a server crashes, another one can pick up where it stopped, using the saved steps in Postgres.

This approach means you do not need a central orchestrator. Servers work together using Postgres, making sure only one server works on a task at a time by using database locks. Postgres also helps avoid doing the same work twice by checking for duplicates. With this setup, you get fewer moving parts, and you can solve problems like scaling and backups with Postgres tools, which are well tested.

For scaling, you can add more worker servers or use bigger or distributed Postgres setups. For reliability, Postgres already has strong features like replication and failover. For checking on workflows ("observability"), you can use simple SQL queries to see what is happening or find errors—no need for extra dashboards or tools.

Security and reliability are also better because you only have to worry about Postgres, not a new orchestrator system. If your app already uses Postgres, you are not adding new risks or points of failure.

In the Hacker News comments, many people like the idea of keeping things simple. They say Postgres is fast, reliable, and easy to use, so it makes sense to use it for workflows. Some developers share stories of using the same approach in their own projects and say it works well, especially for small and medium workloads.

Others are more careful. They warn that Postgres can become a bottleneck if you have huge numbers of workflows. Some say tools like Temporal are better for very complex workflows with lots of dependencies, retries, or timeouts. A few people point out that distributed databases like CockroachDB can help with scaling, but add their own complexity.

Some commenters mention that using Postgres gives you strong guarantees and good backup options, but you need to be careful with locking and transaction design to avoid deadlocks. A few discuss cases where they had to move away from Postgres to a more advanced workflow engine when their needs grew. There is also a debate about "reinventing the wheel"—some think it is better to use a ready-made solution, while others prefer the control and simplicity of the Postgres approach.

Overall, readers see value in using Postgres for many workflow problems, but they remind everyone to look at their own needs and limits before choosing.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48308376)

The game puts you in a situation where you have only 60 seconds before your next meeting, and an AI helper (called Claude Code) is asking you to quickly approve or deny several commands.

The main idea in the game is "permission fatigue." This means if you get too many permission requests, you might stop reading them carefully and just click "approve" to save time. The game asks you to read each command and press 1 to approve or 2 to deny as fast as you can, showing how easy it is to make mistakes under pressure. The game tries to help you feel how tricky it can be when AI tools ask for lots of permissions and you don’t have much time.

The game warns that this is a real problem with AI systems today. If users are in a hurry, they may approve risky actions without thinking. The linked article explains attacks where attackers take advantage of busy users who get used to clicking "yes" on everything. For example, an AI code helper might ask for permission to run dangerous code, and the user might approve it just to get their work done faster. The lesson is that too many permission requests can make people careless, leading to real security risks.

In the comments, some people say the game is a good way to show this problem—it makes the risk feel real and urgent. Others point out that this is not just an AI problem; we already see it with software updates and cookie consent popups, where people click "accept" without reading. Some users talk about how hard it is to design a system that keeps users safe without annoying them with too many questions.

A few commentors worry that AI tools might get even more permissions in the future, making this problem worse. Others suggest possible fixes, like better user training, smarter permission systems, or "default deny" rules for risky actions. Some people think that AI helpers should be more careful about when they ask for approval, maybe only asking when something is truly important. There are also comments about the game itself, with some saying it’s fun and others finding it stressful, which is exactly the point. Overall, the discussion shows that permission fatigue is a real and growing problem for both AI and regular software.
