Scientists have made a new type of CRISPR tool that is able to destroy only cancer cells with a certain mutation, while leaving healthy cells alone. The main idea is to use CRISPR not to fix genes, but to find and kill cells that have dangerous changes. This is important because most cancer drugs today turn off genes that are too active, but they do not work if a tumor suppressor gene, like p53, is broken. The p53 gene is often mutated in cancer, but until now no drugs can target this mutation well.

The new tool uses a version of CRISPR called Cas12a2. It looks for a special RNA that only appears in cells with the mutated gene. When Cas12a2 finds this, it cuts up all the DNA of that cell, killing it. Healthy cells, which do not have the mutation, are not harmed. In lab tests, the tool killed only the mutated cancer cells, even if the difference was just one letter in the genetic code. The method is also easy to update for new mutations. The main problem left is how to deliver this tool to the right cells in the body, which is still hard for all gene therapies.

On Hacker News, some users are excited, calling it a big step forward. Others are worried that getting the tool safely into every cancer cell in a real person will be much harder than in lab dishes. Some fear side effects if the tool targets the wrong cells, or if healthy cells look like cancer cells by mistake. A few are hopeful that because CRISPR is programmable, it could be used for many cancers and maybe other diseases. Many users say more studies are needed, as early results often do not work as well in patients. There are also comments on cost and access if the treatment works. Some people like the idea of using CRISPR as a cell-killing tool, since that is how it fights viruses in nature.

Next, there is a story about open source projects and AI code contributions. The writer, who manages open source projects, now often gets code changes written by AI tools like LLMs. He feels unhappy, because his work is now about reviewing machine-made code instead of working with people. Before, a pull request meant someone cared and wanted to help. Now, many changes are made quickly with AI for someone’s own needs, and often do not help the project. Sometimes, these changes make things worse, and the author must spend time checking them, which he finds tiring.

To improve things, he made a new rule: if you want to help, you must open an issue and discuss your idea before making a pull request. He closes any pull requests that come without talking first, if they look like AI work. He admits this might mean missing some good changes, but he prefers working with people who care. The author also wonders if open source is less important now, since more people just use AI to write code. He worries that soon, only machines will write code, and humans will just review it.

Many commenters agree, saying they also see too much low-quality, AI-written code in open source. It is hard to tell if a person or a machine wrote it, and it wastes time. Some like the rule of discussing before coding, as it filters out lazy contributions. A few say AI can help new coders learn, but sending code without understanding it is not good. Some believe open source will change, but people who care about good code will always be there. Others think this problem will grow as AI gets better. A small group thinks the author is too strict, but most agree that quality and human connection are important.

Apple has rewritten its TrueType font hinting interpreter, moving from C to Swift. This part of macOS is key for showing text clearly, especially on old or low-res screens. The main reason for the change is security. Fonts come from many places, and C code can have memory bugs. Swift is safer, because it stops many of these bugs. Apple also wanted the new code to be just as fast as before and to show every letter the same way—what they call "pixel-identical" rendering.

To make sure it worked, Apple wrote four times more test code than interpreter code. They tested with millions of PDFs and thousands of fonts to check that the Swift version matched the old C version. At first, moving data between C and Swift was slow, so they used new Swift features like projection types and spans to work with C data directly, without copying. Swift’s rules about memory and value types helped keep things fast and safe. Some parts needed "unsafe" Swift code, but these were well-documented and checked. In the end, the Swift interpreter was about 13% faster than the C one, and the code is now easier to read and safer to update.

Apple put the Swift code on GitHub so others can learn from it. They also used what they learned to help LLMs do code conversion, which helps other projects.

Hacker News users praised Apple for moving to safer code and were surprised Swift could beat C in speed. Some were curious about how pixel-identical results were possible, and wanted details about the tests. Developers liked the technical tricks, such as avoiding data copies. Some asked if this would work in other projects, and a few worried about Swift’s future and support. Some pointed out that even a little unsafe code can be risky, but most agreed that the benefits are clear. There were also calls for Apple to accept outside contributions, but most were just happy to see the code shared. Overall, the move was seen as a win for safety, speed, and learning.

Next, we have a guide about setting up a fast local coding agent on a Mac, even without internet. The author wanted it to be quick, support images, and use an OpenAI-style API. The setup uses llama.cpp with Metal acceleration, the Gemma 4 model, an MTP draft model for speed, a tool for images, and Pi as the agent in the terminal. The main model file is about 16 GB, and the whole folder is about 17 GB.

Running Gemma 4 with llama.cpp and Metal gave about 58 tokens per second. Adding the MTP draft model increased speed to about 72 tokens per second. The best draft setting was to use 3 draft tokens. The author also tested MLX, but llama.cpp was faster. Adding image support worked by loading a “multimodal projector”, and did not slow things down.

There are step-by-step instructions for installing, downloading models, and starting the local server. The API is OpenAI-compatible, so other tools can use it. There are also sample commands for testing the server and using Pi with both text and image input. The author tried another model, Qwen3.6, which is better for code, but slower at about 55 tokens per second.

In the comments, readers liked the clear steps and the focus on fully offline AI. Some shared that they also want to depend less on cloud services. Others said 17 GB is large, and wished for smaller models. Some talked about the trade-off between speed and quality—some want faster results, others want better code, even if slower. A few asked if this works on older Macs, and the author said 64 GB RAM is helpful, but smaller models might work with less. Some users gave tips for even more speed, like tuning draft settings or using quantized models. There was also talk about privacy, since all code stays on the laptop. Finally, some people thought this could help teams in places with bad internet, making local AI coding more practical.

Hackers have started using a new trick: they put words about nuclear or biological weapons into their spyware. The reason is that many AI security tools, which use large language models, are built to stop working if they see dangerous topics. By adding these words, hackers make the AI refuse to scan their malware. This is a new kind of attack, using AI’s safety rules against itself. The article says security teams may need to make AI less strict, so it can still do its job. It also warns that hackers will keep finding ways to mess with AI by adding fake or scary content.

In the comments, some users thought this was clever but worrying. Some said AI safety must be balanced, so real threats can still be found. Others said relying only on AI is risky, and human experts are still needed. Some shared that hackers have always tried to fool scanners by adding confusing text. Some suggested that security teams should train AI to ignore such bait. Others said that if models are too strict, they will not be useful for real work. Some wondered if hackers will always be one step ahead. Others worried that weaker safety rules could be abused in other ways. Most agreed it is a hard problem with no easy answer.

There is also a new idea for making AI models faster and cheaper when reading the same documents. Now, every AI agent repeats a slow step called "prefill" each time it reads a document. The authors suggest doing this step once and sharing the result, called the "KV cache." If a publisher makes and sells the cache, other AIs can skip the slow step. This can be 9 to 50 times cheaper, and the more people use it, the bigger the savings.

The main problem is that KV caches are big and hard to compress. Sending them over the internet can cost more than just repeating the work. The best way is to keep the cache close to where the AI runs, like web caching. There are also open questions about privacy, security, payment systems, and who owns the caches.

Commenters liked the idea and said it could save money, especially for popular content. Some worried about privacy, data leaks, and technical problems. Others wondered if it works only for common documents. There were also questions about control and power over caches. Some pointed out similar ideas in web caching, but said this is new for AI. Overall, most agreed it is a smart idea but needs more work.
