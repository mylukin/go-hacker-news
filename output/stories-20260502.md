## VS Code inserting 'Co-Authored-by Copilot' into commits regardless of usage

- 原文链接: [VS Code inserting 'Co-Authored-by Copilot' into commits regardless of usage](https://github.com/microsoft/vscode/pull/310226)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47989883)

VS Code changed its Git extension so that it now adds “Co-Authored-by: Copilot” to commit messages by default, even if you do not use Copilot. The update changed a setting called `git.addAICoAuthor` from “off” to “all,” so the AI co-author tag is now always enabled unless you turn it off yourself. This means if you make a commit using VS Code, you might see Copilot listed as a co-author, even when you wrote all the code by hand. 

The change was made in the configuration file, but some code in the extension still referred to the old default, which could cause more confusion. Many users noticed the tag showing up in their commits without warning. Some even turned off all AI features but still saw Copilot added as a co-author. There was no big announcement or clear notice in the UI, so people found out only after seeing the commit messages.

In the comments, most people are upset and confused. Many call the change “unacceptable” and say it feels dishonest or like “vandalism” to add Copilot as a co-author when it did nothing. Some say this makes commit history less useful, because now you can’t trust if Copilot actually helped. A few users worry that this could make author tracking harder for real projects. Others are angry that it is turned on by default, and that users were not told about it. 

A few people say they are switching to other editors, like Zed, because of this. Some comments even joke that Microsoft did this because not enough people use Copilot, so they want it to look popular. A couple of users point out you can turn this off by changing your settings, but they still think it should not be the default. One commenter says other editors, like Cursor, do this better by not adding any AI tags unless you actually use AI code. Overall, most people want this change undone or at least made opt-in, not automatic.

## Six Years Perfecting Maps on WatchOS

- 原文链接: [Six Years Perfecting Maps on WatchOS](https://www.david-smith.org/blog/2026/04/29/maps-on-watchos/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47990606)

This article is about making a great map app for the Apple Watch over six years. The writer, who loves hiking, explains how he built mapping into his app, Pedometer++, so people can check their location easily when outdoors.

At first, watch screens were very small and slow, so early maps were simple and made on a server—not useful for offline or real-time navigation. When SwiftUI came to watchOS, the writer built a custom map engine that worked faster and let him show map tiles and user location right on the watch. Designing for the tiny watch screen was hard, and he tried many layouts to show both maps and workout data, but most designs did not feel right. He wanted the app to be easy to use with one hand and not require much setup, since watches should be quick and simple.

When Apple released “Liquid Glass” for watchOS, he saw a new design chance but found normal maps did not look good with this style. So, he hired a cartographer to make a custom map that looked better and was easier to read, even in dark mode. Still, the user interface was not perfect, so he worked with a designer to find a layout where metrics sat in the corner and the map filled the rest of the screen. After much testing and walking hundreds of miles, he finished a design that was clear, useful, and looked good on the watch.

He also explains why he did not use Apple’s own MapKit for watchOS. MapKit was not flexible enough and often lacked details for hiking trails. It always showed dark mode, which was not always best, and it did not allow enough custom features or overlays.

In the comments, many people praise the effort and attention to detail over such a long time. Some are happy to see thoughtful design for a small but important use case, like hiking. Others share their own struggles with making apps for watchOS, agreeing that design is often harder than coding. A few point out that offline maps are very valuable for hikers, and they wish more apps offered that. Some developers ask about the technical challenges of making custom map rendering engines, and the writer replies with tips and lessons learned. One commenter thinks Apple should let developers use more tools for watchOS, to make this kind of work easier. Another person wonders if custom maps could be shared with other apps, and some discuss the pros and cons of open-source maps versus custom ones. There’s also a small debate about whether most users need this level of detail, but most agree it’s great for people who do outdoor activities.

Both the C++ and Rust parts share a fast memory allocator (mimalloc), speeding up many things.

Popular sites like Reddit and YouTube work better now due to these changes. Many bugs and crashes on other sites got fixed too. Ladybird’s score on web platform tests jumped, showing it is more standards-compliant. Rust is now required to build Ladybird, and there are many small improvements, like better text selection, faster tab switching, and better error pages.

## Do_not_track

- 原文链接: [Do_not_track](https://donottrack.sh/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47988592)

This article talks about a new idea called DO_NOT_TRACK. The goal is to help users easily stop software tools from tracking or collecting data about them.

Many command-line tools and SDKs collect information about how you use them. Each tool has a different way to turn off this tracking. For example, for .NET, you set one environment variable, for AWS SAM, you set another, and so on. There are many tools, and the opt-out process is confusing because it is different every time.

The proposal is simple: There should be one standard way to say, “Please do not track me.” The article suggests a new environment variable called DO_NOT_TRACK. If you set DO_NOT_TRACK to 1, all tools should respect this and stop collecting data. This would cover ad tracking, usage reporting, telemetry, crash reports, and any other non-essential data sharing.

The article gives clear steps for how to set DO_NOT_TRACK in different shells, like Bash, Zsh, Fish, PowerShell, and Windows CMD. It also asks software makers to check for this variable in their code. If DO_NOT_TRACK is set, they should turn off all tracking features and still allow their own opt-out methods. Even better, the article suggests making tracking opt-in, not opt-out, to respect privacy.

There are examples of similar standards, like NO_COLOR and FORCE_COLOR, which help control colored output in terminals.

In the comments, some people really like the idea and say it would make their lives easier. They are tired of having to remember different variables for every tool. Others wonder if software companies will actually respect the new variable, since some want the data for business reasons. A few say that there is already a “Do Not Track” setting in web browsers, but it is often ignored, so they worry this might happen here too.

Some developers say it's hard to add support for new environment variables in old tools, but others reply that it is easy for new tools to start using it. A few suggest that if enough users ask for this, more software makers will listen. There are also comments comparing this to the NO_COLOR idea, with people sharing stories about how adoption of simple standards can be slow but helpful over time.

One person points out that data collection should be opt-in by default, not opt-out. Others agree, saying that respecting privacy builds trust. Some also share tips for making sure your shell sets the variable everywhere, so you don’t miss any tool. Overall, the comments show both hope and doubt, but many agree that a simple, standard way to say “do not track me” would be a good thing.

## Neanderthals ran 'fat factories' 125,000 years ago

- 原文链接: [Neanderthals ran 'fat factories' 125,000 years ago](https://www.universiteitleiden.nl/en/news/2025/07/neanderthals-ran-fat-factories-125000-years-ago)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47990284)

Neanderthals in Germany, 125,000 years ago, used special ways to get fat from animal bones. Scientists found thousands of small bone pieces at a site called Neumark-Nord 2, showing that Neanderthals crushed big bones and heated them in water to get bone grease, a food rich in calories.

Usually, people thought only modern humans did this kind of complex food processing. But at this site, Neanderthals worked together to collect bones from at least 172 big animals—like deer, horses, and ancient cows—and processed them by the lake. They picked this place on purpose and used fire, stones, and tools to break and cook the bones. The work was organized: they hunted in some places, butchered in others, and made fat in a special area. This means Neanderthals planned ahead, worked as a group, and understood the value of fat in their diet. The site’s remains also show they used plants, and sometimes hunted huge animals like elephants and rhinos.

The discovery changes how we see Neanderthals. It shows they were clever, could plan, and changed their environment in big ways. Some experts think this means Neanderthals had more advanced culture and skills than many people believed before.

In the Hacker News comments, some people are surprised at how smart Neanderthals were, saying this makes them seem less “primitive” than old textbooks suggest. Others discuss how fat is important for survival, especially in cold times or when food is hard to find. A few users point out that making bone grease is not easy—it needs a lot of work and teamwork, which means Neanderthals must have organized their group well. Some wonder if Neanderthals shared fat with the whole group or if only a few got it. There are also comments about how this kind of food processing means Neanderthals could live in more places, not just where easy food like fruit was available. A few users ask if this new research changes what we know about human evolution and how we survived ice ages. Others just enjoy learning something new about ancient people, and share links to similar discoveries about early humans.

## The agent harness belongs outside the sandbox

- 原文链接: [The agent harness belongs outside the sandbox](https://www.mendral.com/blog/agent-harness-belongs-outside-sandbox)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47990675)

This article talks about where to run the “agent harness” that controls how an AI code assistant works. The author explains there are two main ways: running the harness inside the sandbox (the safe, isolated environment) or outside it. Running inside is simple and works well for one person. But for many users in a company, running the harness outside the sandbox has more benefits.

If the harness is inside the sandbox, everything happens in one place. The agent can use files and tools like on a normal computer. This is easy to set up and works for one user. But, when the harness is outside, it controls the sandbox from the outside using APIs. The agent’s secrets (like API keys) stay safe and never go into the sandbox, which is important for security. You can also stop and start sandboxes as needed, saving resources. If a sandbox fails, it’s easy to make a new one—the session keeps going because the harness is outside. For teams, skills and memories can be shared in a database instead of each user having separate files, making teamwork easier. But, you have to build your own system for things like saving progress and handling files, since many tools expect a normal filesystem.

The author’s team chose to run the harness outside the sandbox. They had to solve problems like keeping the agent running even if servers restart, making sandboxes start fast, and storing files that are really in a database. Their system uses a tool called Inngest to save progress, and Blaxel to quickly start sandboxes. The harness pretends there’s a normal filesystem, but behind the scenes, some files are in the database and some are in the sandbox. This makes things work for both the agent and the users.

Many people in the comments agree that running the harness outside the sandbox makes sense for security and multi-user systems. They like the idea of keeping secrets out of the sandbox and using databases for shared knowledge. Some worry about extra complexity, like building a fake filesystem and handling problems when several users update files at the same time. Others say that using APIs and databases is common in larger systems, so these tradeoffs are normal. A few comment on possible problems with Bash commands bypassing the system, and agree that it’s hard to make everything perfect. Some users think this approach is good for scaling up, but maybe too much for small, personal projects. There’s also talk about “last-writer-wins” for handling file updates, with some saying it’s fine for most cases, while others warn about lost data. Overall, the comments show that people understand the tradeoffs and see why the author’s team made their choice.

## NetHack 5.0.0

- 原文链接: [NetHack 5.0.0](https://nethack.org/v500/release.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47988776)

NetHack 5.0.0 is a new version of the classic dungeon exploration game, released in May 2026. This update brings many changes to improve the game and make it easier to build and run on different computers.

The source code now follows the C99 standard, which is a common rule set for writing C programs. This helps developers understand and work with the code more easily. The game also now supports cross-compiling, so you can build NetHack on one type of computer and play it on another, even if they are very different. This is explained in the “Cross-compiling” file included with the game.

Another big change is how the game handles level and dungeon files. Before, NetHack used special tools called “yacc and lex” to make these files at build time. Now, it uses Lua scripts instead. These are loaded when you play, making things simpler and more flexible for both players and developers.

There are over 3100 bug fixes and changes in this release. Some of these changes are explained in a special file, but be careful—reading it might spoil some surprises in the game. Old saved games and “bones” files from earlier versions will not work in NetHack 5.0.0, so players will have to start new games. The release notes also explain how to check that your downloaded files are safe, using SHA256 checksums. There are new commands to help find where NetHack stores its files on your computer. The developers ask players to report bugs and give feedback, but also to look at known issues first so they don’t report the same problem twice.

In the comments, some people are excited about the move to C99, saying it will make the codebase cleaner and easier to maintain. Others are happy to see cross-compiling support, since it means NetHack can reach even more platforms, like new computers and operating systems. Some longtime fans feel sad that old saved games won’t work, but they understand this is often needed for big updates. There is debate about the switch to Lua for level files—some think it makes modding easier, while others worry it could introduce new bugs or make troubleshooting harder.

A few commenters mention that the list of 3100 changes is impressive and shows how much work the team put into this release. Some users warn new players about “spoilers” in the change log, reminding them that discovering things on their own is part of the fun. Others discuss the continued importance of NetHack in gaming history, saying it’s great to see the project still active after so many years. Finally, some share tips on how to verify downloads and suggest everyone report bugs to help make the game even better.
