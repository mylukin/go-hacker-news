## Tell HN: Litellm 1.82.7 and 1.82.8 on PyPI are compromised

- 原文链接: [Tell HN: Litellm 1.82.7 and 1.82.8 on PyPI are compromised](https://github.com/BerriAI/litellm/issues/24512)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47501426)

The article warns that the Python package litellm versions 1.82.7 and 1.82.8 on PyPI are compromised with malware. In version 1.82.8, a special file is included that runs a script every time Python starts, even if you do not import litellm. This script collects many secrets from your computer or server: environment variables, SSH keys, cloud credentials, shell history, and even crypto wallets. It puts this data in a file, encrypts it, and sends it to a fake server controlled by the attacker. The attack works on local machines, servers, Docker containers, and CI/CD pipelines. The malware is hard to spot because it is hidden and encoded. The attack uses a .pth file, which Python runs automatically. The official advice is to remove these versions right away, check your computer for the bad file, and change any secrets or passwords you had on the system.

In the comments, people confirm that they have been affected and are very worried. Some note that not only 1.82.8 but also 1.82.7 is compromised, though in a different way. Many users thank others for sharing how to find and fix the problem. Some are upset about spam in the comments and suggest reporting spam accounts. There is debate about whether GitHub or CI/CD systems are partly to blame because of weak security. Some suggest practical steps: always run risky tools in containers, rotate your secrets, and never download code from untrusted places. Others argue that more needs to be done to keep the software supply chain safe. A few people promote alternatives to litellm, while some question if this is the right time to advertise. Some users share scripts and lists to help others clean up their systems. Overall, people are shocked by the scale of the attack and agree that developers and teams need to be much more careful with dependencies.

Arm says this new CPU gives double the performance per rack compared to the latest x86 chips, mostly because of better memory, single-thread speed, and efficiency. Each core does more work, and the system handles more jobs at once.

## Hypura – A storage-tier-aware LLM inference scheduler for Apple Silicon

- 原文链接: [Hypura – A storage-tier-aware LLM inference scheduler for Apple Silicon](https://github.com/t8/hypura)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47504695)

Hypura is a tool that lets you run large AI models on Apple Silicon Macs, even if the models are bigger than your computer’s memory. It solves the problem where big models would crash your Mac or make it super slow by smartly using your GPU, RAM, and SSD together.

The main idea is that not all parts of an AI model need to be loaded into fast memory all the time. Hypura checks your hardware and decides which parts go on the GPU, which go in RAM, and which can stay on the SSD. For example, small and important data is kept on the GPU, while big, less-used pieces are loaded from the SSD only when needed. For special models called MoE (Mixture of Experts), Hypura only loads the “experts” you need for each step, reducing the amount of data moved from SSD by 75%. It also tries to guess which parts you’ll need next and loads them early to save time.

This means you can run a 31 GB Mixtral model on a 32 GB Mac Mini at a usable speed, or even a 40 GB Llama model that would normally crash. If a model fits in your memory, Hypura runs it as fast as possible, with no slowdown. If the model is too big, Hypura makes it work by streaming data from the SSD without overwhelming your Mac.

To use Hypura, you build it from source with Rust and CMake. You can profile your machine, run models, chat with them, and even use Hypura as a server that works with tools like OpenClaw or Ollama. It never writes large amounts of data to your SSD, so it’s safe for your disk.

In the Hacker News comments, many users are excited about running bigger models at home, especially on MacBooks and Mac Minis. Some are surprised by how fast the SSD is on Apple Silicon and think this approach is clever. Others point out that reading from SSD will still be much slower than using RAM, so while Hypura makes things possible, performance may not be great for all models. Some wish for Windows or Linux support. A few users ask about SSD wear, but others explain that reading doesn’t harm SSDs. There is also praise for the detailed technical explanation and for making the tool open source. Some developers discuss technical limits, like SSD bandwidth and memory mapping, and wonder if similar tricks could work on other hardware. Overall, the community sees Hypura as a big step for local AI and a smart way to use the hardware people already have.

## Wine 11 rewrites how Linux runs Windows games at kernel with massive speed gains

Wine 11 is a new version of software that lets people run Windows games on Linux computers. This update changes how Wine works with the Linux system to make games run much faster.

The big change is that Wine 11 moves some important parts from the user space into the Linux kernel. This means the software now works closer to the core of the operating system. Before, Wine had to translate every instruction between Windows and Linux, which made things slow. With the new method, Wine uses a system called "Hangover" to run 32-bit and 64-bit Windows apps together without extra overhead. Wine 11 also changes how it handles memory, making games load and play smoother. Another big improvement is better support for game controllers and graphics cards. Many popular games now start up faster and have higher frame rates. This means less lag and fewer crashes for players. The developers also made it easier to install updates and fix bugs. Wine 11 is open source, so anyone can help make it better. People who use Linux for gaming might not need Windows at all anymore.

Others worry about possible security problems, since running code in the kernel can be risky. Some mention that not all games will work perfectly right away. A few users are impressed by how hard the developers work to keep Wine up to date. Others ask if these changes could help other programs, not just games. ## How the world’s first electric grid was built

- 原文链接: [How the world’s first electric grid was built](https://worksinprogress.co/issue/how-the-worlds-first-electric-grid-was-built/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47455079)

This article tells how London went from having many small, separate electricity systems to building the world’s first true electric grid. It covers more than a hundred years, starting with private experiments and ending with today’s mix of public and private control.

First, in the late 1800s, rich people and businesses wanted electricity for things like lighting art galleries. Early power came from small generators, often using direct current (DC). Each local system was different. Some ran on AC (alternating current), others on DC, and they all used different voltages and frequencies. This made it hard for devices to work everywhere, and made the system messy and wasteful. As demand grew in the early 1900s, private companies and city councils fought for control. There were more than 50 different systems in London alone by 1918. World War I showed how bad this setup was—factories needed more power, but the patchwork of systems could not keep up.

To fix this, the British government tried to get companies to join their systems together, but at first, they did not force anyone. People and companies wanted to keep their own power and profits. Other countries, like the US and Germany, moved faster by building bigger, connected grids. By the 1930s, Britain finally built a national grid, using AC to move power long distances and connect regions. Engineers even secretly connected the whole grid one night as a test. During World War II, the grid helped the country keep power running even when some power plants were bombed.

After the war, the government took over the grid and most power stations to make building and planning easier. In the 1950s and 1960s, they built the “Supergrid,” with higher voltage lines to move even more electricity between regions. In the 1980s and 1990s, the government privatized much of the system, letting private companies run and own parts of the grid and power stations, but keeping rules to ensure fair use. Recently, the government has taken back some control over how the grid is run, but private companies still own most of it.

In the comments, many people admired the engineering and speed at which the first grid was built in the 1930s. Some said it showed how much you can do when people work together, and how “breaking the rules” sometimes helps, like when engineers secretly connected the grid. Others pointed out that private companies often blocked progress to protect their profits, and that only government rules made it possible to have a shared system. A few commenters compared the UK’s grid history to their own countries, noting similar fights between local control and national planning. Some thought nationalization was better for planning, while others said privatization lowered costs and sparked new ideas. There was also debate about whether today’s grid problems come from too much private control, or too much government interference. One person noted that even now, different parts of the UK grid still do not work perfectly together. Another said it is hard to balance local concerns with the need for big, shared projects. Many agreed that the story of the electric grid is really about people, politics, and making tough choices as much as about technology.

## Hypothesis, Antithesis, synthesis

- 原文链接: [Hypothesis, Antithesis, synthesis](https://antithesis.com/blog/2026/hegel/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47504094)

This article is about a new tool called Hegel, a set of property-based testing libraries, starting with Rust, from the makers of Hypothesis and now part of Antithesis. The main idea is to bring the high-quality testing features of Hypothesis (originally for Python) to many programming languages and integrate them with Antithesis for even better bug finding.

The article explains property-based testing: instead of writing one test for each example, you describe the kinds of values you want to test, and the library creates many tests for you. For example, you might say “my parser should never crash on any string,” and the tool tries thousands of random strings to check this. This finds bugs you may not think of, like dividing by zero or handling strange Unicode characters.

There are examples in Rust using Hegel, such as checking that parsing a fraction string never crashes, or testing that converting numbers to and from scientific notation always works. These tests found real bugs in popular Rust libraries, such as crashing on "0/0" or handling of zero in scientific notation. The article lists three main types of bugs property-based testing finds: forgetting edge cases like zero, problems with tricky data types (like Unicode), and errors in complex logic or data structures.

Another type of test shown is “model-based testing,” where you compare your code with a simple “model” version to make sure they always produce the same result. The article says starting with property-based testing is easiest when you change one of your regular tests into a property-based one.

The history of Hypothesis is discussed: it became the most used property-based testing library for Python because it was easier to use and had better features than earlier tools. These include good default data generators, automatic shrinking of failing examples to the smallest case, and saving failing tests for future runs.

Instead of writing a new Hypothesis for every language, Hegel uses Hypothesis as a backend and connects to it from the target language (like Rust or Go). This way, new languages get a powerful testing library with less work. Hegel’s long-term plan includes dropping the Python dependency and making a Rust server for more speed.

The article also discusses why property-based testing is important for both human and AI-written code, as it finds many mistakes that regular tests miss. There’s even a Hegel “skill” to help AI write property-based tests for you, making it easier for beginners.

If you already have good property-based tests, you may not need Hegel yet, but if you are starting fresh, Hegel is a strong choice with many features from Hypothesis. The team plans to support more languages soon and improve Hegel for testing complex, concurrent systems, especially to work better with Antithesis and another project called Bombadil.

In the comments, many people are excited about bringing Hypothesis-style testing to more languages, especially those with weaker existing property-based tools. Some developers like the idea of using a Python backend, saying it’s a clever way to share work across languages, while others worry about the extra complexity and possible speed issues. A few are cautious, pointing out that Python dependencies can be problematic in environments where Python is not allowed or wanted.

Some users share stories of finding strange bugs with property-based testing, like how easy it is to miss edge cases or “weird” data. Others ask how hard it is to write custom generators for their own data types, or if Hegel’s API feels natural in their language. A few discuss the shrinking feature and how important it is to get small, readable failing examples.

People also talk about model-based testing and how useful it is for checking correctness by comparing real code to simple, slow versions. There’s curiosity about how Hegel will support highly concurrent or distributed systems, since that’s a weak spot for many testing libraries. Some wish for even more language support, like JavaScript, and a few mention they’re eager to try Hegel on new projects or convert old tests. Overall, the comment section shows strong interest, some technical questions, and a hopeful attitude about Hegel’s future.
