## GitHub Stacked PRs

- 原文链接: [GitHub Stacked PRs](https://github.github.com/gh-stack/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47757495)

GitHub has announced "Stacked PRs," a new way to break big code changes into smaller pull requests (PRs) that build on each other. This feature is now in private preview and comes with both a web interface and a command-line tool called `gh stack`. 

The main idea is to help teams manage big changes by splitting them into small, clear steps. Each step, or "layer," is its own pull request. All the layers are linked together in a stack, so code reviewers can check one small change at a time. The GitHub web UI shows a "stack map," letting users see all the PRs in the stack, move between them, and check their status. If you want, you can merge the whole stack or just some layers at once. When you merge, GitHub will rebase the rest of the stack for you, so everything stays in order.

The `gh stack` CLI helps developers make stacks locally. You can create new branches, add layers, rebase, push, and open PRs—all from your terminal. The system also works with AI coding agents, so you can ask an agent to break up a big change into a stack of PRs. The key benefit is that reviews are easier, faster, and less confusing because each PR is small and focused. Continuous integration (CI) tests run for every PR as if it’s going into the main branch, so you don’t miss problems. Branch protection rules work for the whole stack, not just one PR.

To get started, you install the CLI, use it to create layers, push them, and then submit the stack of PRs to GitHub. The docs show simple commands for each step.

Hacker News users had many reactions. Some people were excited, saying this solves a real problem for teams dealing with big, complex changes. They liked that it keeps code reviews simple and helps avoid messy conflicts. Others pointed out that tools like "ghstack" or "Stacked Diffs" already exist, but were happy to see it natively in GitHub. Some worried about confusion if team members don’t understand how stacks work, or if the UI will be too busy with many PRs.

A few users asked if this will slow things down by creating too many small PRs, but others replied that focused reviews are much faster and safer. Some people shared that they already use a similar workflow manually, and are glad for official support. There were questions about how well the tool handles rebasing, merge conflicts, and CI failures in the middle of a stack. Others wondered if the AI integration is useful now, or if it’s just hype.

A few developers said they don’t like the idea of merging many PRs at once, fearing it hides problems, but most agreed that the benefits outweigh the risks. Some users wanted to know when it will be available for all users, not just the private preview. Overall, the main feeling was positive—GitHub is making a tricky part of code review simpler and faster.

## Someone Bought 30 WordPress Plugins and Planted a Backdoor in All of Them

- 原文链接: [Someone Bought 30 WordPress Plugins and Planted a Backdoor in All of Them](https://anchor.host/someone-bought-30-wordpress-plugins-and-planted-a-backdoor-in-all-of-them/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47755629)

A hacker bought more than 30 popular WordPress plugins and added a secret backdoor to all of them. The plugins were trusted and had thousands of users, but after being sold, they became tools for attacking websites.

The main problem started when a client saw a security warning about the “Countdown Timer Ultimate” plugin. The plugin had already been force-updated by WordPress.org, but the real damage was hidden. The malware was placed in the wp-config.php file, making it hard to find. This code let the hacker control the site, show spam only to Google’s search bot, and change their command server using blockchain, making it almost impossible to block.

The hacker was very patient. They bought the plugins on a public site called Flippa for a large amount of money, then waited eight months before using the backdoor. The first thing the new owner did was add the malicious code. The original plugin authors were not involved. The plugins were from a real business that was in decline and decided to sell. The attack was only possible because there is no review or alert when a WordPress plugin changes owners.

WordPress.org reacted by closing all 31 affected plugins and pushing out emergency updates. However, these updates only blocked the malware from spreading further; they did not remove the code from infected sites. The article’s author patched all affected plugins on their clients’ sites and shared instructions for others to do the same. They also explained how to check for infection by looking at the size of the wp-config.php file.

This attack is not the first of its kind. In 2017, a similar thing happened to another popular plugin. The pattern is clear: buy a popular plugin, wait, and then attack many sites at once.

Many Hacker News readers were shocked at how easy it was for the attacker to buy and control so many plugins. Some said WordPress should check plugins more carefully when they are sold to new owners. Others worried that WordPress sites are too easy to attack because many people do not update their plugins or check for problems.

Some commenters pointed out that this is a supply chain attack—users trust plugins from the WordPress store, but that trust can be broken if ownership changes. A few people shared tips on how to protect your own sites, like only using plugins you really need and checking for strange changes after updates.

Others wondered if a better plugin marketplace or stricter rules would help, while some argued that the open nature of WordPress makes these attacks hard to stop completely. There was debate about whether plugin sellers like Flippa should have more responsibility. A few people said this is a warning sign for anyone running a WordPress site: always watch for updates, check your files, and be careful about which plugins you use.

Overall, the comments showed that many in the community are worried about plugin security and want better ways to protect sites from these kinds of attacks.

## Stanford report highlights growing disconnect between AI insiders and everyone

The report shows that many young people, like Gen Z, feel angry or less hopeful about AI, even though they use it often. Experts worry about things like super-smart AI (AGI), but normal people care more about losing jobs or paying more for electricity because of AI data centers. This difference became clear after some violent events against tech leaders, with many people online supporting these actions out of frustration.

Only 10% of Americans feel more excited than worried about using AI every day, while over half of experts think AI will be good for the country in the next 20 years. When it comes to healthcare, 84% of experts say AI will help, but only 44% of the public agree. For jobs, 73% of experts feel AI will improve work, but just 23% of regular people feel the same. Most experts think AI will help the economy, but the public is not so sure because of recent layoffs blamed on AI.

Data shows that 64% of Americans think AI will create fewer jobs in the future, while experts are less concerned. Trust in the government to control AI is low in the U.S.—only 31% trust officials to regulate AI well, compared to 81% in Singapore. Many Americans feel current rules won’t go far enough to keep AI safe.

Still, more people around the world think AI’s benefits are growing, with global support rising from 55% to 59% in one year. But the number of people who say AI makes them “nervous” is also going up.

In the comments, some people agree with the report and say tech insiders live in a “bubble,” not seeing real worries about jobs or fairness. Others think the public is getting scared by news or social media, and that experts need to explain AI better. Some readers point out that AI experts may be too hopeful because they have good jobs and trust the tech, while many people just see job losses and rising bills. A few commenters say that it’s normal for people to worry about new technology, and things may get better as AI improves. Others warn that if experts ignore people’s fears, it could lead to protests or even violence. Some also mention that experts should be more honest about AI’s risks, not just its benefits. There are debates about whether the government can really regulate AI, given how fast it changes. Lastly, a few commenters note that this divide is similar to other times in history when new tech changed society, and that open talks between experts and the public are important.

## How to make Firefox builds 17% faster

- 原文链接: [How to make Firefox builds 17% faster](https://blog.farre.se/posts/2026/04/10/caching-webidl-codegen/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47756321)

This article talks about making Firefox build 17% faster by caching part of the build process called WebIDL code generation. The author explains how using a tool called buildcache, along with its Lua plugin system, helps speed up the build by saving and reusing results from this step.

Normally, when building Firefox, there is a step where Python scripts turn many .webidl files into C++ code. This step is slow and always runs, but the output is always the same if the input doesn’t change. Before this change, build tools like ccache and sccache only cached compiler steps, not this Python codegen step. Buildcache is different because it can cache any command by using Lua scripts that tell it what files are inputs and outputs.

To use buildcache for WebIDL codegen, a small change was made in the build files. Now, when buildcache is enabled, it wraps the Python codegen command. A Lua script (webidl.lua) tells buildcache how to handle this step: it looks for the “mozbuild.action.webidl” command, finds all the input and output files, and then saves or restores them from the cache.

The article shows how much faster builds get with this change. Without any cache, a build takes about 5 minutes and 35 seconds. With ccache, a warm build takes 3 minutes and 21 seconds. With sccache, it takes 2 minutes and 49 seconds. With buildcache and the WebIDL plugin, the build drops to only 1 minute and 12 seconds. This is not a huge change by itself, but it shows that more Python build steps could also be sped up this way.

The author explains how to set up buildcache and the Lua script, and hints that other parts of the Firefox build could use the same trick. The main point is that buildcache’s Lua plugin system makes it possible to cache more than just compiler steps, leading to faster builds.

In the comments, many people are happy to see build times getting shorter, especially for big projects like Firefox. Some developers say they have not tried buildcache before and are excited about its Lua plugin feature. Others ask if this could work for other build steps, or even other projects besides Firefox.

A few commenters point out that most caching tools only work for compilers, so this is a nice improvement. Some users worry about the complexity of managing more cache scripts, but others say the time saved is worth it. There are questions about whether ccache or sccache could add similar features in the future.

One person says these changes are most helpful for developers who rebuild Firefox often, while another notes it might not matter as much for people who only build once in a while. Some share tips about setting up buildcache, and a few ask if it works well on Windows or macOS.

Overall, the comments are positive, with people sharing ideas, small concerns, and requests for more ways to make builds faster. Many hope this kind of caching can be used in more places.

## (AMD) Build AI Agents That Run Locally

- 原文链接: [(AMD) Build AI Agents That Run Locally](https://amd-gaia.ai/docs)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47756772)

AMD has launched GAIA, an open-source toolkit for building AI agents that work fully on your own computer, without sending data to the cloud. You can use GAIA with Python or C++ to create agents that answer questions, search documents, generate code, and even talk with you—all running locally on AMD hardware.

The main idea is privacy: your data never leaves your device, so you do not need API keys or external services. GAIA takes advantage of AMD chips, using their NPU and GPU for faster AI processing. The SDK gives you tools to build agents in both Python and C++. There are quickstart guides for both languages. For Python, you install GAIA, start a server, and run an agent in a few steps. For C++, you can build a native agent with no Python needed.

GAIA supports many features: you can have a private desktop chat, ask questions about your own documents, do speech-to-speech with offline voice tools (like Whisper and Kokoro), generate and test code, create images with AI, and connect agents to outside tools using a special protocol called MCP. There are also built-in agents for system health checks and Wi-Fi troubleshooting. If you want, you can build your own agent from scratch in C++ and add custom tools.

The SDK comes with full documentation, code examples, and a community on Discord. You can find everything on their GitHub page.

In the Hacker News comments, many people like that GAIA helps keep data private and avoids sending information to the cloud. Some users are happy that it works without API keys and does not depend on big cloud services. Others appreciate that it can run on AMD hardware, making use of new chips with AI acceleration. A few developers say the local-only approach is good for security and for people who cannot use cloud tools.

However, some commenters wonder if it will be hard to set up or if it only works well on the newest AMD chips. They ask if it will support other hardware, like Intel or Nvidia, in the future. Some users also question if the performance will match cloud-based AI or if running big models locally will be too slow for most laptops. There is debate about how useful the built-in agents are and if the platform will get enough community support. Still, many agree that having an open-source option for local AI is a step in the right direction.

## Building a CLI for All of Cloudflare

- 原文链接: [Building a CLI for All of Cloudflare](https://blog.cloudflare.com/cf-cli-local-explorer/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47753689)

Cloudflare is making a new command-line tool (CLI) called "cf" to help developers use all of Cloudflare’s products more easily. They want this CLI to work with every Cloudflare product, making it simple to set up, test, and manage things from your terminal.

Right now, Cloudflare has many products and a large API, but not everything is available in their old CLI, Wrangler. The new "cf" CLI is being rebuilt so developers and their coding bots (agents) can use the same commands for all Cloudflare services. This new tool will be able to set up products, check logs, manage databases, and more, using clear and standard commands. Cloudflare is asking for feedback with an early version you can try by running `npx cf` or installing it with npm.

To build this, Cloudflare made a new TypeScript-based system to define APIs, commands, and settings. This makes it easier to keep everything up-to-date as Cloudflare adds new products. The new system uses TypeScript schemas, so they can generate documentation, SDKs, and commands more quickly and with fewer mistakes.

They are also working to make all commands consistent. For example, the command to get information will always be `get` (not `info`), and options like `--json` will always be used the same way. This helps both humans and bots use the CLI without confusion.

Another new feature is Local Explorer. This tool lets you see and work with your test data (like databases and storage) when developing on your computer. Before, you had to check hidden files or use third-party tools, but now you can open Local Explorer and see everything in one place. This helps you build, test, and fix apps faster without needing internet or extra setup.

Cloudflare wants this new CLI to work the same for both local and remote resources. So, if you use the `--local` flag, the CLI will work with your local environment just like it would with the real Cloudflare API—very helpful for testing and writing code.

In the comments, some developers really like the idea of a single CLI for everything and think it will save time. Others are happy about the focus on making local development easier and more like working with real cloud resources. People appreciate the move to TypeScript for schemas, saying it helps with consistency and faster updates. A few users note that keeping all commands and names the same across tools is important for both people and bots.

Some commenters ask about the future of the old Wrangler tool and want to know how long it will be supported. Others wonder if the new CLI will be open source, so they can help fix problems or add features. A few are worried about breaking old scripts or workflows, but Cloudflare says they will bring the best parts of Wrangler into the new CLI. There are also requests for better ways to manage DNS, cache, and security settings. Many are excited to try the tool and plan to give feedback to help shape its future.
