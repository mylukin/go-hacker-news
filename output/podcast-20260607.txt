Our first story is about why Linear, a project management tool, feels much faster than most web apps. The article explains that Linear makes your browser the main database. When you make changes, the app updates instantly without waiting for the server. The server gets your update in the background. This means there are no loading spinners for most actions. Linear uses IndexedDB and MobX for quick, detailed updates, and if the server rejects a change, the app can roll it back, though this does not happen often.

The frontend is built with React and TypeScript, and the backend uses Node.js and Postgres. They keep things simple, avoid complex frameworks, and have improved their build system many times to reduce the amount of JavaScript and CSS sent to users. They also drop support for old browsers, split their code, and preload important files. After the first load, a service worker caches the rest of the app, so you can even use Linear offline.

Linear also splits up vendor files, so updating one library does not force users to redownload everything. Fonts are loaded smartly to avoid flashes on the screen. When you start the app, the loading screen’s CSS is inlined, so you see something right away. Authentication is checked after showing the UI, and the app can render local data first, making things feel instant.

Their sync engine is built on three ideas: all data is in the browser when you start, changes are made locally first, and updates are very detailed, so only changed parts of the UI re-render. Design choices are also important—keyboard shortcuts and a fast command palette make the app feel even faster. Animations are short and only affect things that do not slow down the browser.

In the comments, many people praise Linear’s speed and attention to detail. Some say more apps should use the local-first model, but others warn that custom sync engines are hard to build and handle offline errors. There are worries about large JavaScript bundles, but code splitting and caching help. Some users prefer simpler tools for smaller apps. Many point out that keyboard shortcuts and design choices add a lot to the sense of speed. Some wish more apps would preload and cache assets like Linear. There are also concerns about browser storage limits and the work needed for good offline support. Overall, readers say Linear’s speed comes from many careful choices over years, not just one trick.

Now, let’s talk about a project that lets you view Microsoft Office files—Word, Excel, PowerPoint—directly in your web browser, showing them as they really look. What makes this special is that all the code was written by Anthropic’s AI, Claude, with no human code.

The tool uses Rust to parse files, then runs in your browser as WebAssembly. Drawing is done with TypeScript on the HTML Canvas, so you get accurate text and layout. You can use your own UI, not just the default viewer. It works with DOCX, XLSX, and PPTX files, and you can use only the parts you need to keep your app small. There’s an add-on for math equations that loads only if needed.

Each file is parsed in a web worker and sent to the main thread to render, keeping things fast. Features like formatting, tables, charts, and images are supported. The tool works with frameworks like React, Vue, Angular, Svelte, SolidJS, and Qwik. There are extra packages for converting files to Markdown, using in Node.js, and even a VS Code extension. The system focuses on security and privacy: it does not run scripts from documents and does not send data to the internet unless you choose to.

Some developers want to use this in their own projects, while others note that Office file standards are hard to work with. Now, let’s look back at the IBM 604, an early electronic calculator from 1948. It was not a full computer, but a programmable calculator using vacuum tubes, making it much faster and more powerful than earlier machines. The IBM 604 was about the size of two refrigerators, could do 60 math operations in under a second, and process about 100 punch cards per minute. People programmed it by plugging wires into plugboards, not by writing code.

A big innovation was the pluggable tube module. Each module had a vacuum tube and parts inside a small box with pins. If a module broke, you could pull it out and replace it easily. The article focuses on the thyratron tube, which acted as a high-power electronic switch, using xenon gas. When you sent a signal, the gas allowed a strong current to flow until the power was cut. The writer tested a thyratron module by using it to turn a light bulb on and off.

These modules made the IBM 604 reliable and easier to fix. Its success led to more advanced IBM computers in the 1950s, like the IBM 650.

In the comments, readers are impressed by the engineering of the IBM 604. Many enjoy seeing old technology still working. Some share their own stories with vacuum tube equipment. There are comparisons to modern electronics, and talk about how heavy and power-hungry these machines were. People discuss the IBM 604’s place in tech history, saying it was an important step between mechanical calculators and true computers. Many respect the careful work needed to keep old parts running and hope to see more about restoring other old computer modules in the future.

Our next story explains the `lost+found` folder at the root of Linux and Unix systems. This folder helps when the file system gets damaged, for example, after a crash or power cut. When you run `fsck` to fix file system problems, it may find lost files or data blocks that are not linked anywhere. These files are placed in `lost+found`. Usually, they have no names or places, but sometimes they have important information.

Each partition has its own `lost+found` folder. It is created with extra space for `fsck` to use. If you delete it by mistake, you should use `mklost+found` to make a new one. Most people never need to look in this folder, but after big problems, it is worth checking.

In the comments, people share different views. Some say the folder is usually empty and not needed. Others point out it only appears on some file systems, not all Unix systems. Some find it annoying because of permission errors, and others are thankful it can recover files after crashes. There is debate about which systems use it, but most agree it is a useful safety net.

Now, a story about building trust in AI-assisted software development. The author uses many automated checks to review every step. They lost trust in AI tools that tried to do too much, so now they use different “subagents,” each focused on part of the process. It starts with an AI-created plan for a new feature, then agents with names like “Assumption Excavator” and “Documentation Validator” check for missing parts or problems. More agents join for big projects, and each gives feedback.

Code writing follows the improved plan, but only one main AI writes code, while others review. After coding, agents like “Code Validator,” “Type Safety Validator,” and “Security Analyst” check the code for bugs and risks. This cycle repeats until the code is good. Before release, more agents check everything again, including an “Anxiety Reader” for risky parts.

The author says everyone has their own idea of what “good” means, so you have to keep checking and improving. But this process uses a lot of AI resources, so it may be too much for small projects.

In the comments, some people like this careful process, saying it helps catch mistakes early. Others worry about the cost and time, especially for small teams. Some just want simple checks and think this is too complex. A few ask for tips to keep things simple. Others say trust in AI will improve over time. Some share that feedback from AI can be too generic. Many are interested in the “Assumption Excavator,” saying hidden assumptions are a big problem. There are concerns about relying too much on AI and getting false confidence. Some wonder if these agents can work with other AI models. Several agree that “never finished, only released” is true in software.
