## Since Chronium 148, Math.tanh is now fingerprintable to link underlying OS

- 原文链接: [Since Chronium 148, Math.tanh is now fingerprintable to link underlying OS](https://scrapfly.dev/posts/browser-math-os-fingerprint/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48884853)

This article talks about how the Math.tanh function in Chrome now gives slightly different results depending on your operating system (Linux, macOS, or Windows). Since Chrome version 148, this small difference can be used to figure out which OS you are really using, even if you try to hide it.

The article explains that most browser fingerprinting looks at things like canvas, WebGL, or fonts. But now, even the last digit of a math function result can give away your OS. For example, when you run Math.tanh(0.8) in Chrome, you get a different result on Linux, macOS, and Windows, just by one or two digits at the end. This happens because Chrome’s JavaScript engine, V8, now uses the operating system’s own math library for Math.tanh, while other math functions (like Math.sin or Math.exp) still use a built-in version that gives the same result on all platforms.

The article gives a table showing which math functions leak information and which do not. For JavaScript, only Math.tanh leaks the OS. For CSS trig functions (like sin, cos, tan), all of them leak the OS because they use the system library. Web Audio functions also leak, but in different ways depending on the OS and the part of the audio stack.

The author explains that it is very hard to fake or spoof this behavior. You would need to copy the exact math formulas and even the tiny mistakes or rounding choices each OS makes. Sometimes, even different parts of macOS use different math libraries, and picking the wrong one will make your fake stand out. Also, the CPU type (ARM or x86) can make a small difference.

To fight fingerprinting, they suggest copying the exact math code and coefficients from the real OS libraries, and making sure the code runs the same on any machine. Sometimes, they even load the actual Windows library inside Linux and call its functions directly, to get the exact same bits.

Finally, the article shows how Scrapfly, a web scraping company, worked hard to make their browser match a real one down to every math bit, so they cannot be easily detected as fake traffic.

In the Hacker News comments, many people are surprised that such a tiny thing can be used for fingerprinting. Some think it is clever and shows how hard it is to hide from websites. Others note that this kind of difference is not new—floating-point math has always had small differences between platforms, but now it can be used as a tracking tool.

Some users worry this makes privacy even harder, as even simple math can now break your cover. A few people wish browsers would go back to using a bundled math library for all functions, to avoid leaking OS details. Others are impressed by Scrapfly’s effort to match every detail, calling it “next-level” for anti-bot tech.

There are also comments about the complexity of trying to spoof all these details, and how every new browser change can make it even tougher. Some suggest that the browser vendors should make results consistent to help privacy, while others argue that the cost and speed trade-offs make this difficult. A few see it as a cat-and-mouse game where anti-bot and privacy tools will always chase each other. Overall, the discussion shows both technical interest and concern about privacy and the future of browser fingerprinting.

## Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k

- 原文链接: [Claude Code sends 33k tokens before reading the prompt; OpenCode sends 7k](https://systima.ai/blog/claude-code-vs-opencode-token-overhead)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48883275)

This article compares two AI coding tools, Claude Code and OpenCode, to see which one uses more tokens and why that matters. The team tested both tools doing the same tasks and checked exactly what each one sent to the model.

Claude Code sends much bigger prompts than OpenCode. For a simple reply, Claude Code used about 33,000 tokens before the user even typed anything, while OpenCode used about 7,000. The difference got smaller with a newer model, but Claude Code still used over three times more tokens. Claude Code also rewrites its prompt a lot during a session, which makes caching less efficient and increases cost. OpenCode’s requests stay nearly the same, so it saves money using the cache.

Adding a big instruction file or more servers makes both tools use even more tokens, but Claude Code’s numbers grow faster. Using subagents with Claude Code can push token use very high—over four times more than running the same task directly. In one case, using subagents made a task jump from 121,000 to 513,000 tokens. OpenCode handles subagents with smaller prompts, so it should be cheaper, but the test could not measure its total exactly.

There was one situation where Claude Code was better: for a long, multi-step task, it finished with fewer total tokens than OpenCode because it completed more steps in one go, while OpenCode had to send its smaller prompt many times.

The article explains that if you have a lot of configuration—like big instruction files, many servers, or subagents—your token use and costs can go up a lot, no matter which tool you use. But Claude Code’s frequent changes to its prompt mean it writes a lot more to the cache, which is expensive. Both tools completed the tasks with the same quality, so the extra tokens did not give better answers in these tests.

In the comments, some people were surprised at how much bigger Claude Code’s prompts are and worried about higher costs for companies. Others pointed out that Claude Code has more features, so maybe the extra tokens are worth it for some users. A few argued that the real question should be about output quality, not just token use, and wanted to see benchmarks on harder problems. Some liked OpenCode’s simple, stable requests and saw it as better for businesses that care about price and predictability. Others mentioned that many people do not realize how much overhead these tools add until they check the logs, and that every configuration detail can make a big difference. Some users shared their own tricks for cutting costs, like using smaller instruction files and fewer tools. Others wondered if Claude Code’s design could be improved to reuse more of its prompt, making it cheaper. A few thought prompt caching is still too hard to use well, and wish the tools would handle it automatically. Several people agreed it is important to monitor what your AI sends and receives, to avoid surprise bills. Finally, some commenters said this kind of deep, data-driven comparison is very useful and want to see more tests like it.

## Can we understand how large language models reason?

- 原文链接: [Can we understand how large language models reason?](https://cacm.acm.org/news/can-we-understand-how-large-language-models-reason/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48883090)

This article looks at how researchers try to understand the way large language models (LLMs) like GPT or Llama actually “reason” when solving problems. While these models can write text, answer questions, and even do math, we still do not know exactly how they work inside, because they have billions of parameters that interact in complex ways.

A professor named Thomas Icard from Stanford is working to make these models more understandable, or “interpretable,” using ideas from logic and science about how humans think. One big question is: do LLMs just copy patterns from training data, or do they really build some kind of logic or reasoning inside? Icard and his team use “causal abstraction” to look at the models’ behaviors at a higher level, a bit like how physics can describe a gas by pressure and temperature instead of looking at every single molecule. This means they try to find simple explanations for what the big, complex model is doing.

For example, they found that a BERT model learned to handle logic words like “every,” “some,” and “not” by building a kind of logic system inside. In another study, a Llama-based model solved questions like “What month is six months after August?” by first turning the month into a number (August = 8), adding 6 to get 14, and then converting 14 back into a month (since there are only 12 months, 14 becomes February). The model uses the same trick for days of the week and time on a clock. This shows the model is applying a general strategy, not just copying answers.

Icard and others only get to work with open-source models like Llama, not giant commercial models, but they talk to teams at companies like Anthropic and Google DeepMind who are also trying to make LLMs more transparent. The goal is to make these models safer, less biased, and more reliable by understanding their inner workings. Still, it is hard to scale these methods to the largest models or to automate the process, because right now, it depends a lot on people looking inside and figuring things out by hand.

In the end, Icard says we may never be able to explain everything LLMs do with a few simple rules, but we can hope to slowly uncover more about how they “think.”

In the Hacker News comments, some people are excited about this research and think it is an important step for AI safety. Others are more skeptical and say that models might only look like they are reasoning, when really they are just very good at pattern matching. A few users point out that humans also work with abstractions and shortcuts, so maybe it is not strange that LLMs do the same. Some worry that, since commercial models are closed, we will always be a step behind in understanding the newest and biggest systems. Others are hopeful that open research and sharing methods will help the whole field improve. There are also comments about how much human work is still required to study these models, and some ask if real “understanding” is even possible. A few users suggest that, even if we never fully understand LLMs, making them a bit more interpretable will already make AI systems safer and more useful.

## Migrating a production AI agent to GPT-5.6: 2.2x faster, 27% cheaper

- 原文链接: [Migrating a production AI agent to GPT-5.6: 2.2x faster, 27% cheaper](https://ploy.ai/blog/migrating-a-production-ai-agent-to-gpt-5-6)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48882716)

This article talks about moving a real AI agent from using Claude Opus to the new GPT-5.6 Sol model. The AI agent builds and edits marketing websites, so the change needed to be a big improvement. After tests, GPT-5.6 Sol was faster and cheaper than Claude Opus, and made better results.

The team did many careful tests. They checked if the new model was really better, and found their testing tools were built for the old model. For example, GPT-5.6 worked in parallel and used features the old model didn’t, so old test settings gave unfair results. Once they fixed the tools, GPT-5.6 finished builds in about half the time, used fewer tokens, cost 27% less, and made designs that looked as good or better.

They also found GPT-5.6 made shorter, cleaner code than Claude Opus, but sometimes the designs looked too similar and generic. Their team had to adjust prompts and other settings to get more unique results.

There were technical problems too. GPT-5.6 would fill in all possible tool options, even if they weren’t needed, which broke some file reads. The fix was to change the code so the model could say “I’m not using this option” by sending "null" values, and then remove those before the code ran. This solved empty file reads and made the agent use fewer tool calls.

Prompt caching was another big issue. GPT-5.6 changed how caching worked. The team needed to set special keys so that repeated work didn’t get billed again. After they fixed this, their costs dropped a lot. They also found out that, on OpenAI, workspaces can’t share cached data, so each one has to pay a little extra sometimes.

Another problem was with how GPT-5.6 saved its reasoning. By default, it used server pointers, which sometimes broke. They fixed it by changing the settings so the reasoning was sent with each request.

In the end, the team says GPT-5.6 Sol is ready, is faster, cheaper, and makes good websites. They invite others to try it.

In the comments, many people like the detailed technical write-up and say it is helpful for anyone switching models. Some users are surprised by how much the test tools mattered and say it is easy to trust the wrong numbers if you don’t check how your tests work. Others point out that needing to change so much code for a new model shows how hard it is to make AI tools work in production.

A few people worry that OpenAI’s caching changes make things more complex and expensive, especially for startups. Some like that GPT-5.6 makes shorter code, but others wish it made more creative designs. There are comments about the tool-call bug, with developers saying they have seen similar issues when switching models. Some users ask if these problems are common for all LLMs or just GPT-5.6.

Others discuss the business side, wondering if being tied to one model provider is too risky. Some say that having to rework for each model is a big cost, but others reply that the speed and price improvements make it worth it. A few people ask for more details on how to make models follow unique brand styles. Finally, some users thank the team for sharing clear numbers and lessons learned.

## Irish datacenters now guzzle 23% of the country's electricity

- 原文链接: [Irish datacenters now guzzle 23% of the country's electricity](https://www.theregister.com/on-prem/2026/07/11/irish-datacenters-now-guzzle-23-of-the-countrys-electricity/5270013)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48884322)

Ireland’s datacenters now use 23% of the country’s electricity, which is almost a quarter of all power. This is a big rise from 14% in 2021 and just 5% in 2015. In 2025 alone, their energy use jumped by 10%, even though new datacenters were mostly blocked from connecting to the grid around Dublin. Datacenters now use more power than all city homes (18%) and over twice as much as rural homes (9%). The government had to set new rules: if a datacenter wants a big power connection, it must also have batteries or generators and be able to give power back to the grid if needed. Some big companies, like Microsoft, already do this. Ireland has more than 80 datacenters, which is a lot for a country with just over 5 million people. There have been protests about this, and worries that datacenters could take up a third of the country’s power soon. The rules to slow new datacenter growth were lifted at the end of 2025, but usage still rose fast.

In the comments, some people say datacenters bring money and jobs to Ireland, so the high power use is worth it. Others worry about the stress on the electric grid and the environment, especially as AI and cloud services grow. A few think datacenters should be built in places with extra green energy, not where the grid is already tight. Some users note that the new rules about batteries and feeding power back are smart, but wonder if they go far enough. There are jokes about “bit barns” and how Ireland has become a global server farm. A few people fear this might push up power prices for regular people. Others ask if this is really a problem if the datacenters help pay for greener power. Some point out that Ireland’s weather helps keep datacenters cool, which saves energy. A few worry about the risk if something goes wrong—so much of the internet depends on these centers now. ## The One-Step Trap (In AI Research)

- 原文链接: [The One-Step Trap (In AI Research)](http://incompleteideas.net/IncIdeas/OneStepTrap.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48883415)

Rich Sutton’s article talks about a mistake in AI research called the “one-step trap.” This is when people think an AI can learn just simple, one-step predictions and then use these to figure out what will happen far into the future.

Sutton explains that it sounds good: if your one-step predictions are always right, you could use them again and again to predict anything. But in real life, these little mistakes add up fast. If your AI guesses wrong at each step, those errors pile up, and your long-term predictions get worse and worse. 

Also, in the real world, the future isn’t just one straight path. There are many possible futures, like branches on a tree. To guess which one will happen, you’d need to look at every possibility and work out its chance—which is way too hard and slow for computers as things get more complicated. This makes using one-step predictions for big problems almost impossible.

Sutton says many people fall for this trap, even in big fields like control theory and AI research. They use one-step models because they’re simple and tempting, even though they don’t work well for longer-term thinking.

Instead, Sutton suggests using “temporally abstract models.” These let the AI think in bigger chunks of time, not just one step at a time. He gives some research papers where he and others show how this can work, using ideas like “options” and “GVFs” to help AIs learn better about the world.

In the Hacker News comments, some people agree and say they have seen this problem in their own work. They notice that errors in one-step models get worse over time, just like Sutton says. Others point out that using “options” or higher-level models can be hard to design or understand.

A few commenters say that one-step models are still useful for some simple tasks or as a starting point. Some think that with better data and better models, one-step predictions might work better in the future.

There are also comments about how this problem shows up in real-world things, like robots or games, where planning far ahead is tough. Some mention that even big AI companies struggle with this and still use simple models because they are easier to build.

A few technical readers ask questions about Sutton’s proposed solutions and want more examples. Others share links to related research or tools that try to fix this problem.

Overall, the discussion shows that many people know about the “one-step trap” and want better ways to help AI plan and predict into the future. But real solutions are still difficult, and there’s no perfect answer yet.
