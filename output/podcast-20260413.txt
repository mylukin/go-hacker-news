First, GitHub has announced a new feature called “Stacked PRs.” This helps developers break big code changes into smaller pull requests, or PRs, that build on each other. The idea is to split complex work into clear, simple steps. Every step is a PR, and they all connect in a stack. The GitHub website shows a stack map, so reviewers can see all PRs, move between them, and check their status. You can merge the whole stack or just some layers, and GitHub will rebase the rest for you to keep everything in order.

There is also a command-line tool, `gh stack`, that helps you make stacks locally. Developers can create new branches, add layers, rebase, push, and open PRs directly from the terminal. The system works with AI coding agents, too—an agent can split a big change into stacked PRs for you. The main benefits are simpler code reviews and fewer conflicts. Each PR runs its own CI tests, and branch protection rules work for the whole stack.

Getting started is simple: install the CLI, create layers, push, and submit your stack of PRs. Documentation gives clear steps.

Others noted that similar tools exist, but were happy to see a native GitHub solution. There were worries about confusion if team members don’t understand stacking, or if the UI will get too busy with many PRs. Some asked if too many small PRs could slow things down, but most replied that focused reviews are faster and safer. Users shared stories of doing this manually and are glad for official support. There were questions about how the tool handles rebasing, merge conflicts, and CI failures in the stack. Some wondered if the AI part is useful or just hype. A few developers said merging many PRs at once might hide problems, but most agreed the benefits are bigger. Many want to know when it’ll be public. Overall, people feel positive—this makes code review easier and faster.

Next up, a hacker bought over 30 popular WordPress plugins and added a secret backdoor to them. These plugins were trusted and had thousands of users. After the sale, they were turned into tools for attacking websites. The problem was found when a client got a warning about the “Countdown Timer Ultimate” plugin. WordPress.org had already pushed a force update, but the real damage was hidden. The malware was placed in the wp-config.php file, making it hard to spot. The code gave the hacker control of the site, let them show spam only to Google’s search bot, and used blockchain to change command servers, making it very hard to block.

The hacker was patient, buying plugins on Flippa and waiting eight months before using the backdoor. The original plugin authors were not part of the attack. The plugins were from a real business that was sold. The attack worked because there is no review or alert when a WordPress plugin changes owners.

WordPress.org reacted by closing all 31 affected plugins and pushing emergency updates. But these updates only stopped the malware from spreading—they did not remove it from already infected sites. The article’s author patched all affected plugins for their clients and shared instructions for others. They also explained how to check if your wp-config.php file is infected.

This is not the first time this happened. In 2017, a similar attack took place. The pattern is clear: buy a popular plugin, wait, and then attack many sites at once.

Hacker News readers were shocked at how easy it was for the attacker. Some said WordPress should check plugins more carefully when they are sold. Others worried that WordPress sites are too easy to attack because many people do not update plugins or check for problems. Some pointed out that this is a supply chain attack—users trust plugins, but that trust breaks when owners change. A few shared tips: only use plugins you really need and check for strange changes after updates. There were ideas about better markets or stricter rules, but some said WordPress’s open nature makes these attacks hard to stop. Others said sites like Flippa should be more careful. Many agree this is a warning for anyone running a WordPress site: always watch for updates, check your files, and be careful about the plugins you use.

Data shows that 64% of Americans think AI will lead to fewer jobs. Trust in the US government to control AI is low—only 31% trust officials to regulate AI well, compared to 81% in Singapore. Many Americans feel current rules will not keep AI safe enough.

Still, more people around the world think AI’s benefits are growing, but the number of people who say AI makes them nervous is also rising.

Next, let’s talk about Firefox builds. An article shared how to make Firefox build 17% faster by caching WebIDL code generation. Normally, Python scripts turn .webidl files into C++ code—a slow step. Before, build tools like ccache and sccache only cached compiler steps, not this Python codegen step. But buildcache, with its Lua plugin system, lets you cache any command.

With a small change, buildcache can now wrap the Python codegen command. A Lua script tells buildcache how to find the input and output files, then saves or restores them from the cache. The result: a warm build drops from over 5 minutes to just over 1 minute. This is not a huge change by itself, but it shows that caching more steps can make big projects like Firefox build much faster.

In the comments, people were happy to see builds getting faster. Some said they are excited about buildcache’s Lua plugin feature. Others asked if it could work for other build steps or other projects. Some worried about the complexity of managing more cache scripts, but others said the time saved is worth it. There were questions about ccache or sccache adding similar features. One person said this helps most for people who rebuild often. Some shared tips about setup, and a few asked if it works on Windows or macOS. Overall, the mood is positive, with people hoping for more ways to make builds faster.
