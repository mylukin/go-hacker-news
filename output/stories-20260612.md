## CRISPR tech selectively shreds cancer cells, including "undruggable" cancers

- 原文链接: [CRISPR tech selectively shreds cancer cells, including "undruggable" cancers](https://innovativegenomics.org/news/crispr-technique-selectively-shreds-cancer-cells/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48505231)

Scientists have made a new CRISPR tool that destroys only cancer cells with a common mutation, even those that are hard to treat with normal drugs. The main idea is to use CRISPR not to fix genes, but to find and kill the cells with dangerous changes, while leaving healthy cells alone.

Most cancer drugs today work by turning off genes that are too active, but this does not help when the problem is a tumor suppressor gene that is broken and cannot stop cancer from growing. One important tumor suppressor is called p53, which is often mutated in many cancers. So far, no drugs can target p53 mutations well.

The scientists used a version of CRISPR called Cas12a2. It looks for a special RNA made only by cells with the mutated gene. When CRISPR-Cas12a2 spots this mutation inside a cell, it cuts up all the DNA of that cell. This makes the cancer cell die, while healthy cells, which do not have the mutation, are not harmed.

The team tested this in lab dishes with both healthy and cancer cells. The CRISPR tool only killed the cells with the mutation, not the healthy ones. Even when the difference between healthy and cancer cells was just one letter in the genetic code, the tool was very accurate.

This new method is also fast to update. If a new cancer mutation appears, scientists can quickly make a new guide for CRISPR to target it. The main problem left is how to deliver the tool into the right cells in the body, which is a common challenge for gene editing therapies.

In the Hacker News comments, some people are excited about the new way to fight cancer, calling it a big step forward. Others warn that it will be hard to get this tool safely into every cancer cell in a real person, not just in lab dishes. A few users worry about possible side effects if the tool targets the wrong cells, or if healthy cells sometimes look like cancer cells by mistake. Some are hopeful that because CRISPR is programmable, it could be used for many different cancers and maybe even other diseases. Others remind everyone that many new cancer ideas look good in early tests but do not work as well in real patients, so more studies will be needed. There are also comments about the cost and who will have access to this kind of treatment if it works. Lastly, some users think it is clever to use CRISPR as a cell-killing tool, since that is what it does in nature to fight viruses.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48507282)

The article talks about how the author, who runs open source projects, now gets many code contributions written by AI tools like LLMs (Large Language Models). He feels unhappy that his work is becoming more about reviewing machine-generated code instead of working with real people.

He explains that before AI tools, getting a pull request from someone felt special. It meant someone cared about his project and wanted to help. Now, he says, many people use AI to quickly make changes for their own needs and send the code without much thought. These changes often do not improve the project and sometimes make it worse. The author has to spend time checking if the code is good, which he finds tiring and not enjoyable.

To deal with this, he changed his rules for contributions. Now, if someone wants to help, they must first open an issue and discuss the idea before making a pull request. This way, he knows they are a real person and really care. If someone sends a PR without talking first, he closes it right away if it looks like it came from an AI. He admits this means he might miss some good changes, but he prefers working with people who are truly interested in his projects.

The author also wonders if open source is less important now. He feels there is less interest in sharing code, and more people just use AI to write code for them. He worries that soon, only machines will write code, and humans will just review it. He wants to avoid this future and keep coding as a human activity.

In the comments, many people agree with the author. They also feel sad that open source projects now get lots of low-quality, AI-generated code. Some say it is hard to tell if a person or a machine wrote the code, and it wastes the maintainer’s time. Others think the new rule—asking for an issue before a pull request—is a good idea and helps filter out lazy contributions.

A few people say that AI tools can help new coders learn, but they agree that sending code without understanding it is not good. Some commenters suggest that open source will change, but there will always be people who care about good code. Others wonder if this problem will get worse as AI tools get better and more common. ## Swift at Apple: Migrating the TrueType Hinting Interpreter

- 原文链接: [Swift at Apple: Migrating the TrueType Hinting Interpreter](https://www.swift.org/blog/migrating-truetype-hinting-to-swift/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48508726)

Apple rewrote its TrueType font hinting interpreter from C to Swift to make it safer and faster. This interpreter is important for showing text clearly, especially on older and lower-resolution screens.

The main reason for the rewrite was security. Fonts come from many places, and the old C code could have memory bugs that hackers might use. Swift is a memory-safe language, which means it helps prevent these bugs. Apple also wanted to keep the interpreter just as fast, if not faster, than before, and make sure that every letter looked exactly the same as it did with the old code. They called this requirement “pixel-identical” rendering.

To make sure everything worked, Apple wrote a huge number of tests—four times more test code than the interpreter code itself. They used millions of PDF files and thousands of fonts to check that the new Swift code produced the same results as the old C version. They also cared a lot about speed. At first, moving data between C and Swift was slow, so they changed their approach to avoid copying data. Instead, they used new Swift features like projection types and spans to safely work with C data directly.

Swift’s design, such as using value types and special memory rules, helped keep things fast. Apple could use abstractions and still get good performance, because Swift’s compiler is smart about optimizing code. Some operations that needed “unsafe” Swift code were well-documented and carefully checked. In the end, the Swift interpreter was about 13% faster than the old C one. The code was also easier to read and safer to change in the future.

Apple shared its Swift interpreter code on GitHub for others to learn from. They also used what they learned to help large language models (LLMs) do code conversions better, which is helping with other projects.

People in the Hacker News comments had many thoughts. Some praised Apple for improving security by moving away from C. They liked that Swift proved you can write fast, safe systems code. Others were surprised that Swift could be faster than C, since C is known for speed. A few commenters wondered how “pixel-identical” was possible, and asked for details on the testing process.

Some developers were interested in the technical tricks Apple used, such as projection types and avoiding data copies. Others asked about how easy it was to work across C and Swift, and if this method would work in other projects. A few worried about Swift’s long-term stability and whether Apple would keep supporting these changes. Some pointed out that even a small amount of “unsafe” code can be risky, but most agreed that the benefits were clear.

There were also comments about the open source release. Some people wished Apple would take outside contributions, while others were happy just to see the code. Many agreed that sharing these results helps the whole software community. Overall, the move was seen as a big win for safety, performance, and learning.

## How to setup a local coding agent on macOS

- 原文链接: [How to setup a local coding agent on macOS](https://ikyle.me/blog/2026/how-to-setup-a-local-coding-agent-on-macos)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48507020)

This article explains how to set up a fast local coding agent on a Mac, even when the internet is down. The writer wanted the agent to work quickly, support images, and use an API that is compatible with OpenAI tools.

To build this setup, the author used llama.cpp (a program to run AI models) with Metal acceleration for Mac, the Gemma 4 model in a special format, a draft model for extra speed (called MTP), a tool for images, and Pi as the coding agent in the terminal. The main model file is about 16 GB, and with the other parts, the folder is about 17 GB.

First, running the Gemma 4 model with llama.cpp and Metal gave about 58 tokens per second. This is usable, but not very fast for coding help. Adding the MTP draft model increased the speed to about 72 tokens per second. Testing different draft settings showed that using 3 draft tokens was fastest on the writer’s Mac.

The author compared this to running the model with MLX (another Mac-optimized tool) but found llama.cpp with MTP was faster. Adding image support worked by loading a “multimodal projector” so the model could handle both text and images, and this did not slow things down.

The article then gives step-by-step instructions for installing llama.cpp, downloading the models, and starting the local server. The server uses an OpenAI-compatible API, so other tools can connect easily. There are sample commands for checking the server and for configuring Pi to use the local model with both text and images.

The author also tested another model, Qwen3.6, which is better at coding tasks but slower on their machine (about 55 tokens per second). All the setup steps and configuration files for this model are also explained.

In the Hacker News comments, many readers liked the clear instructions and the focus on running AI models completely offline. Some people shared that they also had internet problems and wanted to depend less on cloud services. Others pointed out that 17 GB for a model is large—one commenter wished for smaller models that are still useful for coding.

Several developers discussed the trade-off between model speed and quality. Some prefer faster but slightly less accurate models, while others want the best possible code help, even if it is slower. A few readers said they tried similar setups with Qwen or other models, and agreed that Qwen is smarter for code, but Gemma is faster on Mac.

A handful of users asked if this approach works on older Macs with less memory, and the author replied that 64 GB RAM is helpful for these large models, but smaller versions might work with less. Some readers liked that llama.cpp keeps getting better on Mac, even beating Apple’s own MLX in some cases.

Others gave tips for making things even faster, like tuning draft settings or using quantized models. There was also talk about using this setup for privacy, since no code leaves the laptop. Finally, a few people wondered if this could help teams in places with slow or unreliable internet, making local AI coding more practical for everyone.

The article explains that many AI models are designed to stop working when they see dangerous topics, like weapons or violence. The writer warns that if we make AI too quick to refuse, hackers will keep finding ways to hide their real goals. He says we’re just starting to see these tricks, and security teams might need to make AI less strict so it can still do its job. He also points out that when designing tools to check for malware, it’s important to think about how attackers might try to mess with the AI by adding fake or scary content.

In the comments, some users thought this was clever but also worrying. A few said this shows why AI safety features need to be better balanced, so real threats can still be found. Others pointed out that relying only on AI tools is risky, and human experts are still needed. Some shared that this is not a new trick—hackers have always tried to fool scanners by adding confusing or fake information. Others argued that security teams should expect these tricks and train their AI to ignore such bait. One person said that if models are too strict, they won’t be useful for real work. Another commenter wondered if this means hackers will always be one step ahead. Some worried that if AI gets too weak in its safety rules, it could be abused in other ways. ## Can I Buy Your KV Cache?

- 原文链接: [Can I Buy Your KV Cache?](https://arxiv.org/abs/2606.13361)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48508949)

This article talks about a new way for AI models to save time and money when reading the same documents. Right now, every AI agent must repeat a hard step called “prefill” every time it reads something, even if other agents already did the same thing before.

The authors suggest a simple idea: only do this heavy work once, then let others reuse the results. The important part that gets reused is called the “KV cache.” If a publisher makes the KV cache for a document ahead of time, other AI agents can just buy it and skip the slow step. This makes things much faster and cheaper. Tests show that using a pre-made KV cache is just as accurate as doing the work from scratch. For example, on a model called Qwen3-4B, reusing the cache is 9 to 50 times cheaper, and the savings get bigger for longer documents.

But there is a problem: KV caches are very big and hard to compress. If you try to send them over the internet every time, it actually costs more money than just letting each AI do the work again. The solution is to keep the KV cache close to where the AI runs, like how big web companies use “caches” to speed up websites. This way, using the cache is almost free.

The savings can be huge. The authors say that if 80 million AIs read a long document, redoing the work costs $1.5 million, but reusing the KV cache costs only about $30,000. So, companies could save millions of dollars, and users could pay much less. The article also mentions some open problems, like making KV caches smaller and building a system for people to pay each other for sharing these caches.

People in the comments had a lot to say. Some loved the idea and said it could really speed up AI and save money, especially for popular content. Others worried about privacy and security—if you share a cache, could you leak private data? Some said that technical problems, like the size of the KV cache and payment systems, are big hurdles. A few wondered if this would only work for very popular documents, not for rare or custom data. There were also questions about who would own and control these caches, and if it could make some companies too powerful. Some users pointed out that similar ideas are already used in other fields, like web caching or CDN systems, but this is new for AI. Overall, most people agreed it’s a smart idea, but some big problems still need to be solved.
