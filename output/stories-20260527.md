## YouTube to automatically label AI-generated videos

- 原文链接: [YouTube to automatically label AI-generated videos](https://blog.youtube/news-and-events/improving-ai-labels-viewers-creators/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48299753)

YouTube is making it easier for people to see when videos use AI. Now, YouTube will show a clear label if a video is photorealistic and AI-generated or strongly changed by AI.

For long videos, the label will appear just below the video player, above the video description. For YouTube Shorts, the label will be right on the video as an overlay. If the video is only a little changed or is animated, the AI label will be in the expanded description, not right on the video.

YouTube is also starting to use automatic AI detection. Before, creators had to say if they used AI. Now, if YouTube’s system thinks a video uses a lot of realistic AI, it will add the label even if the creator does not. If creators think the label is wrong, they can change it in YouTube Studio. But for videos made with YouTube’s own AI tools, or videos with special AI metadata, the label will stay and cannot be removed.

These new rules do not change how videos are shown to viewers or if they make money. YouTube wants viewers to know what is AI and wants creators to have some control too.

People in the comments had different thoughts. Some users liked the idea and think it helps viewers trust YouTube more. Others worry that automatic detection might make mistakes and label videos incorrectly. A few people say creators might try to avoid the label, so YouTube should keep improving its AI detection. Some think the labels will help people know what is real and what is fake, but others feel that most viewers will ignore the label.

One commenter pointed out that AI is now everywhere in video, so maybe labels will soon be on almost every video. Someone else asked if these labels will help with deepfakes or dangerous fake news. Another person said they wish the labels were even bigger, so nobody misses them. Some creators worry about losing control, but others feel it is fair if everyone follows the same rules. Overall, people agree that AI in video is growing fast, and YouTube is trying to keep up.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48297645)

This article is about getting SimCity 3000, an old city-building game, to run well in 4K resolution on a modern Windows 10 computer. The writer explains that the game is their favorite because of its detailed art, good balance, and great music, but says it has many problems when running on new hardware.

First, the game only supports low, old screen resolutions, has no widescreen, and is slow or buggy when scrolling or loading cities. The music is also missing unless you use the CD, and some graphics are broken. To fix these problems, the writer uses several patches and tricks. They start with a special EXE from GOG.com to add widescreen and skip needing the CD. Next, they change a setting in a config file to fix the mouse speed. Then, they install a D3D Wrapper, which lets the game use full 4K resolution and fixes other graphic issues. They also change the wrapper’s settings so the game runs in true fullscreen, not borderless, and turn on VSync for smoother play.

The writer adds a 4GB patch so the game can use more RAM, which helps with loading graphics. They disable the auto-updater, which was causing slowdowns because it tries to reach an old server. To get all the music tracks, they use a patch and copy music files from the CD to the hard drive. After all these steps, SimCity 3000 works in 4K on Windows 10, with good performance and all the music.

Others say they use similar tricks for other old games and like seeing classic games made to work with new technology. ## Rust (and Slint) on a Jailbroken Kindle

- 原文链接: [Rust (and Slint) on a Jailbroken Kindle](https://sverre.me/blog/rust-on-kindle/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48299623)

This article is about someone jailbreaking their Kindle Paperwhite and running Rust code with a Slint GUI on it. The author wanted to use the Kindle as a clock and ended up exploring how to make custom apps with Rust.

First, the author explains that to run Rust programs on a Kindle, you need to cross-compile for ARMv7 with musl libc. They use tools like Zig and cargo-zigbuild to help make this process simple. After building a small test app, they needed a way to actually run it on the Kindle, so they set up SSH access using the USBNetwork tool, which lets you connect to the Kindle by USB or WiFi. They had to manually copy their SSH key because the usual tool (ssh-copy-id) didn’t work.

Once they got a “Hello, World!” program to run, they wanted to make something that actually used the Kindle’s screen. The author chose Slint, a GUI library for Rust, because it can work on many devices, including ARM. To show graphics, they wrote code that takes image lines from Slint’s renderer and writes them directly to the Kindle’s screen framebuffer, which is just a file in Linux. To refresh the e-ink display, they used a function from libc to tell the system which part of the screen to update.

For touch input, the author read data from another file that represents the touchscreen. The Kindle uses a standard Linux multi-touch protocol, so the program just reads batches of events and converts them into Slint pointer actions, like pressed, moved, or released.

After a lot of debugging, they got a simple counter app with a button running on the Kindle’s screen. They mention that it works for their Kindle model but might need changes for others. They also published their Kindle backend code for Slint as an open-source crate, so others can use it too.

In the comments, many people are impressed by the technical achievement and happy to see old devices get new uses. Some suggest more ideas for e-ink hacks, like smart home dashboards or note-taking apps. Others ask about battery life and whether running custom apps drains the Kindle quickly. A few commenters discuss the challenges of jailbreaking and the risks, like bricking the device or losing Amazon’s support. There’s praise for using Linux device files to control hardware directly, and some share similar stories using Kindles for DIY projects. One person warns about screen burn-in if the display updates too often. Another asks if the Kindle’s hardware can handle more complex apps, and the author replies that performance is good for simple GUIs but could be slow for heavy graphics. Overall, readers are enthusiastic and share tips about Rust, cross-compiling, and e-ink device programming.

## What Apple and Google are doing to push notifications

- 原文链接: [What Apple and Google are doing to push notifications](https://www.jacquescorbytuech.com/writing/what-apple-and-google-are-doing-your-push-notifications)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48299220)

Apple and Google now control almost all push notifications on your phone, acting as gatekeepers between apps and users. The article explains how, over time, both companies have taken more control over what notifications get delivered, how they look, and when they appear.

Push notifications started as a way to save battery, so Apple and Google each built a single channel for apps to send alerts. At first, the companies did not change much, but now they use AI and rules to filter, group, and even rewrite notifications. Android and iOS now let users set different levels of importance for each app or type of message. Apple also has Focus modes and Notification Summaries, which bundle and shorten messages so users are not interrupted too often. In recent years, users have to give more permissions for push, and opt-in rates have dropped, especially for games and news apps.

This change means apps and marketers have less power. The platforms decide what is important, often using secret models. Apple and Google say this protects users from too many alerts, but it also keeps users happy with their phones and gives the companies more control. The article compares this to email, where providers like Google already filter and sort messages, making it harder for marketers to know if their emails are seen. But push is even more closed—there is no inbox for users to check old notifications, and senders cannot track what happens on the device.

On both iOS and Android, AI models now decide if and how a notification is shown. These models summarize or reword alerts, and can suppress notifications that seem unimportant or too frequent. Both Apple and Google have patents for these systems, showing this trend will continue. Developers and marketers can do very little to change or track these edits, and can only guess if their messages are seen, clicked, or ignored.

User research shows people get many notifications each day, but only care about a few—like messages from friends or important updates. Too many unhelpful alerts make people turn off notifications or even delete apps. Data from companies like Localytics and CleverTap show that personal, relevant messages work much better than generic ones, and that sending too many alerts leads users to opt out.

The article suggests that marketers should reserve push for important, time-sensitive updates, and use in-app messages for everything else. In-app messages are not filtered by Apple or Google, and are seen by users who are already engaged. But push is still needed to reach users who have not opened the app in a while.

Looking forward, Apple and Google plan to let their AI assistants act on notifications for users—like booking an appointment or replying to a message—without opening the app. This means senders will need to make notifications that are simple and useful for both people and machines.

In the Hacker News comments, some people agree that too many notifications are annoying and that strong controls are good for users. Others worry that Apple and Google have too much power and can block important messages by mistake. A few developers say it is now much harder to reach users or to measure if notifications work. Some commenters like the new AI summaries because they feel less distracted. Others think this change hurts smaller apps and businesses that cannot adapt quickly. There are also questions about user privacy and if Apple and Google are doing enough to explain these changes. Some users say they miss important updates because of these new filters, while others are happy to get fewer alerts. Many agree that personal, user-requested messages should always get through, but there is debate about where to draw the line.
