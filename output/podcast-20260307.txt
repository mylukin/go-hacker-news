Let’s start with CasNum, a Python library that lets you do math only with compass and straightedge, like the ancient Greeks. The project began as a fun way to build numbers and arithmetic by drawing lines and circles, but it grew into something bigger. Every number is a point on a plane, and all math is done by geometric constructions. For example, to add, you find a midpoint and double it; to multiply, you use triangles. Even logic like AND and OR is done with geometry. CasNum is very slow, but it works. You can plug it into a Game Boy emulator, so every calculation in the game is done using only compass and straightedge math. There’s an RSA encryption example too, where you can see each geometric step.

The code has some smart optimizations, but it’s still very slow—booting Pokémon Red takes 15 minutes, and after that, you might get one frame per second. The author jokes that this is how math should feel, and that we are spoiled by easy “a + b” code. CasNum is open source under the MIT license, using parts from other projects like PyBoy and a Game Boy ROM for 2048.

Next, we look back at ten years of Docker. Docker changed the way we build and run software by making it easy to package apps with everything they need, so they can run anywhere. Before Docker, running software on different computers was hard—you had to install many tools and fix conflicting files. Docker solved this with “containers” that hold the app and all its files together. With a Dockerfile, developers can write simple steps to build and share containers. Docker uses “namespaces” in Linux to keep each app separate, but lighter than virtual machines. Containers share the same Linux, but each has its own view of files and networks.

Docker’s main commands—build, push, run—have stayed simple. Images are shared on Docker Hub and use layers to make sharing and updating easy. Running a container is fast and uses little memory. Docker later made it possible to run Linux containers on Mac and Windows by adding a small Linux inside the desktop app. Networking and file sharing between the container and host were tricky, but Docker solved them with tools like SLIRP and vpnkit.

As more computers use different CPUs, like ARM, Docker now supports many CPU types. For security, Docker added ways to keep secrets safe, even using special hardware. For AI, Docker works to handle GPUs, which are harder to share than CPUs. Now, Docker is everywhere: in the cloud, on desktops, and even in space. Its goal is to stay simple and useful, letting developers focus on code.

On Hacker News, many people share how Docker made their lives easier—they no longer have to say “it works on my machine.” Some point out Docker is not perfect: containers can be big, or less secure than they want. Others remember the pain of using VMs for every app and praise Docker for fixing that. Some say tools like Podman and Kubernetes have gone further, but Docker made the first big step. There are comments about Docker’s smart but complex use of Linux namespaces, and some wish Windows had something similar. Some talk about problems with Docker on Mac, but see improvements. People like that Docker helps new developers get started quickly, but some worry about too much “magic” hiding how things work. There is always a security debate, with some saying containers are not as safe as VMs for running untrusted code. Others like that Docker makes it easy to try new software without breaking the system. Some sysadmins love and hate Docker, saying it makes things easier but also more complex. A few worry about “Docker fatigue”—too many tools and updates. Still, most agree Docker made sharing and running apps much simpler than before.

Now, let’s talk about saving old firmware. Someone backed up the original firmware from a LEGO Mindstorms NXT brick from 2006. The author wanted to save it because it was hard to find online and wanted to help the Pybricks project. They tried the official updater tool, but it erased part of the firmware instead of saving it. Using JTAG would mean opening the NXT and soldering, which is risky, so they looked for a software-only way.

Normal NXT programs run in a virtual machine that does not let you read memory outside a small area. The author studied the firmware code and found “IO-Maps,” which let you read and write data between the VM and the system. They discovered a function pointer that could be changed from the computer over USB. By changing this pointer, they could run their own code on the NXT. They used Python and PyUSB to send commands, filled a memory area with custom code, and changed the pointer to point there. This gave them full control, and they wrote a small assembly program to read memory and send it over USB. Now, they could dump the whole 256KB of flash to a file.

In the comments, people were impressed by the clever use of the function pointer and IO-Map. Some talked about how old hardware often had simple protections, making these exploits possible. Some felt nostalgic about hacking similar devices. A few worried about risks if such exploits are used for malware, but others said this work is important for saving history. Some suggested JTAG as an alternative, but agreed software-only methods are easier. People asked technical questions about the assembly and USB, and talked about how this research helps community projects and makes devices more secure. Many thanked the author for the detailed write-up and hope more old firmware can be saved.

Now to FLASH radiotherapy, a new way to treat cancer by giving one very strong, fast burst of radiation instead of many small doses over weeks. The French company Theryq and CERN are developing this method using advanced particle accelerators. Standard therapy can hurt healthy tissue because it takes many sessions. FLASH therapy uses a high dose in less than a tenth of a second, which studies show can kill tumors but spare healthy tissue. The idea came from a lab accident in the 1990s, where a quick, powerful blast did not scar mouse lungs. Later tests on animals and some humans showed the same effect.

Nobody fully understands why FLASH works, but one theory is that healthy and cancer cells react differently to fast, high-energy treatment. New, precise accelerators are being built to make FLASH work for deep tumors. Theryq already has machines for surface tumors and now works on a bigger system, FLASHDEEP, that can reach tumors deep in the body. They use special CT scans to aim the radiation exactly. Testing is done on fish and mice, and new sensors are being developed to measure the strong, quick doses.

FLASH therapy could make cancer treatment faster, cheaper, and safer. In poor countries, one session might be enough instead of many trips to the hospital. Even in rich countries, fewer sessions mean less cost and fewer side effects.

On Hacker News, some are very excited and think FLASH could change cancer treatment. Others are more careful, saying animal results do not always match humans, and worry about the cost and size of the machines. Some are surprised a strong, fast dose can be safe, and want more studies to understand the effect. There are questions about training doctors and building enough machines. Some hope smaller, cheaper machines will help close the gap in cancer care between rich and poor countries. Overall, comments show excitement, hope, and cautious waiting for more proof.

The goal is to make editing code less about fighting with text and more about shaping your ideas.

Next, let’s talk about injecting code into running programs on macOS. The author missed a tool called Live++, which allows hot-reloading of code, but it only works on Windows. So they tried to make a simple version for macOS. First, the author wrote a test program in C++ that prints a number every second. The goal was to attach to this running program, change the value of a variable, and even replace a function while the program is running. They set up the build system, permissions, and code signing so macOS would allow these changes. The test program writes its process ID and memory addresses to a file, making it easier to find the right spot.

The injection program reads this file, attaches to the process using Mach APIs, and suspends the process to safely read or write memory. The author shows how to read the value of a variable, change it, and resume the program. For replacing a function, the injection program copies new machine code into the target process, marks the memory as executable, and writes a “trampoline” at the start of the old function to jump to the new code. The article covers both Intel and Apple Silicon computers, and explains how to handle memory permissions and function size.

In the comments, some people are impressed with the clear example, saying it helps them understand code injection on macOS. Others point out that these tricks can also be used for malware or anti-cheat software, not just for helpful tools. Some worry about security risks, noting that giving the right entitlements could open up your system, and that macOS tries to stop this for safety. Some developers share tips about using similar methods on Linux or Windows, and mention other tools for code hot-patching. One commenter asks about the stability of injected code and warns that changing running programs can cause crashes. Another says this is a great learning project, but real tools like Live++ are much more complex. Some share stories about using code injection for debugging or testing, while some say Apple is making it harder to do this. Overall, readers like the clear, step-by-step guide and are inspired to try similar experiments.

Now we turn to compiling Prolog to Forth. Prolog is a logic programming language, while Forth is a stack-based language. Prolog uses facts and rules to solve problems, while Forth uses a stack to manage data. Compiling Prolog to Forth is hard because they are very different. The article explains how to represent Prolog’s features, like unification and backtracking, in Forth. Backtracking in Prolog is done by searching for answers and trying different options. The article shows how to use Forth’s stack to remember choices, so the program can go back if needed. There are examples of Prolog code and how it looks as Forth code.

The article also explains how to manage variables and data in Forth, since Prolog uses variables differently. The process includes building special data structures in Forth to store Prolog information. The author gives step-by-step examples, talks about performance, and compares the result to running Prolog directly.

Now let’s talk about Plasma Bigscreen, a new open-source TV interface for Linux, made by KDE. It turns your TV or set-top box into a smart, customizable device. The main idea is to bring the power and flexibility of the Linux desktop to your TV. Plasma Bigscreen lets you use your TV remote, game controller, keyboard, mouse, or phone to control everything. You can launch apps, change settings, and switch between programs easily from your couch. The interface is designed to be big and easy to read from far away.

Plasma Bigscreen supports many input methods, like HDMI-CEC for TV remotes, KDE Connect for using your phone, and normal PC inputs. You can install and run your favorite Linux apps like Kodi, Jellyfin, Steam, or YouTube. It works with many Linux distributions and uses open tools like KDE Plasma, KWin, Qt, and Flatpak. You can change how it looks and works, rearrange apps, pick wallpapers, set colors, and more. There’s a special settings app made for TV screens.

A big focus is on privacy and openness. Most smart TV software is closed and hard to trust. Plasma Bigscreen wants to give users control and keep things private. The project is built by volunteers, and anyone can help.
