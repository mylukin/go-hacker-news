## Malus – Clean Room as a Service

- 原文链接: [Malus – Clean Room as a Service](https://malus.sh)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47350424)

This article introduces a company called Malus that offers “Clean Room as a Service.” Their service promises to help businesses avoid open source software license obligations by recreating open source code with robots using only public information.

Malus says their robots never look at the original code. Instead, they read documentation, API specs, and type definitions to build new, but functionally similar, code. The goal is for companies to get code that works the same, but is not legally tied to the original open source license. This means no more having to give credit, share changes, or worry about legal issues from licenses like AGPL or GPL. They claim to help with all kinds of licenses, including MIT, Apache, AGPL, and more, and offer fast, pay-per-kilobyte pricing.

The process works like this: you upload your project’s dependency list, their robots analyze the public information, then a separate set of robots writes new code from scratch. The new code is delivered under a license with no obligations—companies can use it however they want. Malus says this is all legal because of the “clean room” approach, which has been used in other industries before. They even promise legal protection, saying that if anything goes wrong, they’ll refund you and “relocate” their company.

Malus markets their service to companies tired of tracking open source licenses, giving credit, or contributing back to the community. They have fake testimonials from happy corporate clients and joke about the moral side, saying things like “guilt doesn’t show up on quarterly reports.” They also promise to help with hard cases, like AGPL code that companies want to avoid at all costs.

In the Hacker News comments, people have strong reactions. Some think the service is a parody, making fun of companies that try to avoid open source rules. Others worry it’s real, and that it could hurt open source projects by taking code without giving back. Many users point out that, even if the process is “clean room,” copying an API or functionality can still be a legal gray area, and might not protect companies from lawsuits. Some developers share stories of real clean room efforts in the past, like how companies rebuilt UNIX or BIOS code, but note those cases were very careful and expensive.

A few commenters are amused by the sarcastic tone on the website, especially lines about moving the company to international waters. Others debate whether this kind of service would actually work in court, with many saying the legal risk is still high. Some people argue that open source licenses are important for the community, and that trying to “liberate” code this way is wrong. A couple of users think building code based on documentation is still a lot of work and might not save companies much time or money. In general, the discussion is split between those who see the humor, those who see danger, and those who think it’s just not practical.

## Reversing memory loss via gut-brain communication

- 原文链接: [Reversing memory loss via gut-brain communication](https://med.stanford.edu/news/all-news/2026/03/gut-brain-cognitive-decline.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47353533)

This article talks about a new study where scientists helped old mice get their memory back by changing how their guts and brains talk to each other. The researchers found that as mice get older, the bacteria in their guts change, which causes problems with signals sent through the vagus nerve to the brain, leading to memory loss.

They saw that when young mice lived with old mice, the young ones got "old" gut bacteria and started having memory problems too. When the scientists gave these young mice antibiotics, their memories got better again. The study also found that one kind of bacteria, called Parabacteroides goldsteinii, becomes more common in old mice and makes memory worse. This happens because the bacteria cause inflammation in the gut, which blocks the vagus nerve from talking to the brain’s memory center, the hippocampus.

By using a molecule to boost the vagus nerve in old mice, their memory went back to normal, like young mice. The researchers say this shows that gut health is very important for the brain, and changing the gut might be a way to help older people keep their memory. They are now checking if this works in humans too. The vagus nerve is already used for other treatments, so maybe this can be used to help with memory loss as well.

In the Hacker News comments, some people were very excited about the connection between gut bacteria and brain health. They thought this could open new ways to treat memory loss using diet, probiotics, or even nerve stimulation. Others were more careful and said that what works in mice does not always work in humans, so more research is needed.

A few people shared stories about family members who improved memory after changing their diets, while others wondered about the safety of using antibiotics or nerve stimulation in people. Some users talked about the big role the gut microbiome already plays in other diseases, like depression or Parkinson’s, and hoped this research would help those areas too.

One commenter pointed out that it may be hard to target only the "bad" bacteria without hurting the "good" ones. Another said that since the vagus nerve is already used for other medical treatments, it might be easier to try this in people. Some were hopeful but reminded everyone that the human gut is even more complex than in mice.

Others joked about how the phrase “trust your gut” now has more meaning. A few people asked about practical steps, like what food or supplements could help their own memory. People agreed that this research is interesting and shows how much we still have to learn about the body’s systems working together.

## ATMs didn't kill bank teller jobs, but the iPhone did

- 原文链接: [ATMs didn't kill bank teller jobs, but the iPhone did](https://davidoks.blog/p/why-the-atm-didnt-kill-bank-teller)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47351371)

This article looks at why the rise of ATMs didn’t reduce the number of bank teller jobs, but the invention of the iPhone did. It explains that, for many years, people thought ATMs would make bank tellers lose their jobs, but this didn’t happen right away.

When ATMs first appeared, banks needed fewer tellers per branch, but they could open more branches because ATMs made it cheaper. As a result, there were still many jobs for bank tellers. ATMs took over simple tasks, like giving out cash, but tellers started doing more complex work, such as helping customers with loans or advice. This shift is called “complementarity”—the idea that machines and people can work together, with humans taking on new roles.

However, after the iPhone and smartphones became common, things changed. Mobile banking apps let people do almost everything on their phones—checking balances, moving money, or depositing checks. Now, people didn’t need to go to a bank branch at all. This new way of banking meant banks closed many branches, and the need for tellers dropped a lot. The iPhone didn’t just automate tasks; it made the whole job of a teller less important.

The article says that real job losses happen when a new technology creates a new way of doing things, not just by making tasks faster. The ATM made tellers’ work easier but did not remove the need for them. The iPhone, by enabling mobile banking, made the teller’s work unnecessary. This idea matters today, especially when people talk about AI and what jobs it might take over.

From the comments, some people agree with the main idea, saying that the biggest job changes come when whole systems or “paradigms” change, not just when machines handle tasks. Others point out that online banking—before smartphones—also pushed banks to reduce staff, so it’s not just the iPhone that caused the shift. Some commenters note that older people and people in rural areas still use bank branches, so teller jobs haven’t disappeared everywhere.

A few argue that the story is more about changes in banking laws and the way banks do business, not just technology. Others mention that while teller jobs fell, new jobs for app developers and tech support grew, but these are different kinds of jobs. Some people are worried that future changes from AI could be even faster, while others think humans will always find new roles. There’s also talk about “job polarization”—how mid-level jobs disappear, leaving mostly high-skill or low-skill work.

Several commenters like how the article connects past examples to current worries about AI and jobs. A few think the story is more complicated, saying that banks wanted to cut costs anyway, and technology just made it easier. Lastly, some appreciate that the article warns against using old stories for comfort, since technology can shape work in unexpected ways.

## Runners who churn butter on their runs

- 原文链接: [Runners who churn butter on their runs](https://www.runnersworld.com/news/a70683169/how-to-make-butter-while-running/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47355781)

This article is about runners who make butter while running by carrying cream in bags during their runs. Libby Cope and her boyfriend Jacob Arnold tried this idea after wondering if the motion of running could churn cream into butter.

They put heavy cream and salt into double-bagged Ziplocs, secured them in their running vests, and set off on a trail run. At the end of the run, they spread the fresh butter on bread. Libby used to be vegan but missed good butter, which is why she was excited to try this. The idea came from noticing how cream foamed up easily when shaken, and after some searching online, they found out that nobody else had tried making butter this way before. 

The science is simple: cream is full of fat, and shaking it makes the fat molecules stick together to form butter. Their first try was a bit much—they each carried 32 ounces of cream, which was heavy. They worried about the cream getting too warm from their body heat, so they put the bags in a cold river during the run, but this actually slowed things down. Warm cream churns faster, but not too warm or it just melts. It took them about an hour to finish, with extra time spent checking their progress.

On their next try, they used better cream, ran a harder trail, and picked a slightly warmer day (about 50–55°F). This time, the butter formed much faster. Now, they have more butter than they can eat, and Libby jokes about breaking out from all the dairy. Their videos went viral, inspiring other runners to try churn-running, and some tried making different things like corn juice honey butter or even ice cream during their runs.

In the Hacker News comments, some people loved the creativity and thought it was a fun way to combine exercise and food-making. A few found the idea funny and enjoyed the science behind why it works. Others pointed out that running with bags of cream might be uncomfortable or messy, and wondered about the practical side of cleaning up. Some shared stories of making butter by shaking jars at home with their kids, saying it is a fun science activity. There were also comments about food safety and keeping dairy at the right temperature, with some worrying about spoilage. A few people thought the trend was just for social media attention, while others liked how it made running feel less serious and more about having fun. ## Document poisoning in RAG systems: How attackers corrupt AI's sources

- 原文链接: [Document poisoning in RAG systems: How attackers corrupt AI's sources](https://aminrj.com/posts/rag-document-poisoning/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47350407)

This article explains how attackers can trick AI systems that use Retrieval-Augmented Generation (RAG) by adding fake documents to their knowledge base. The author shows, with a local setup and simple tools, how just three fake documents can make an AI confidently report false financial numbers for a company.

The attack works by adding new documents that seem official and use the right keywords, like “Q4 2025” and “CORRECTED FIGURES.” These poisoned documents are crafted to look more relevant than real documents. They use official-sounding language, such as “CFO-approved,” to make the AI trust them more. One fake document even mentions the real number but calls it an error, making the AI believe the fake number is the correct one. When the user asks about company finances, the AI reads the fake documents and gives a wrong answer—saying the company lost money, instead of reporting a profit.

The attack is dangerous because it is easy and long-lasting. Anyone with permission to add documents—like editors or automated tools—can do it. The fake documents stay in the system until someone finds and removes them. Regular users never see the actual documents, just the AI’s answer, so the attack is hard to spot. The same trick could work on much bigger knowledge bases, especially if the attacker uses more advanced methods.

The article also tests ways to defend against this attack. Simple methods, like checking documents for bad patterns or controlling who can add files, do not stop most attacks. A stronger defense is “embedding anomaly detection,” which checks if new documents are too similar to each other or to existing ones. This stops most poisoned documents before they enter the system. Combining all defenses brings attack success down to 10%, but some attacks still get through, especially in small or new knowledge bases.

People in the comments say this risk is real for anyone building RAG-based AI. Some worry that most teams skip strong defenses because they seem complex or slow. Others mention that many companies trust their knowledge bases too much and do not check for poisoned content. Some commenters point out that in big companies, there are often many ways for documents to get added—by bots, scripts, or syncing tools—making it hard to control all the paths for attack. A few readers note that the attack works because AIs trust “official” language and do not always weigh source trust, not just similarity. Some think detection should be built into the AI itself, while others argue that better document review and access controls are important too.

A few developers share their own worries about using RAG in business or healthcare, where wrong answers could cause real harm. Others think the problem will get worse as more companies use AI chatbots for sensitive tasks. Some readers like the simple code and tests in the article, saying it makes the problem easy to understand and fix. Others think the advice on anomaly detection is the most useful part, as it uses tools teams already have. Finally, some commenters want more research on stopping indirect attacks, like those that use hidden prompts or cross-company data leaks.

Before, Chrome on Linux was only for Intel and AMD CPUs (x86_64), so ARM users had to use Chromium or other browsers. Now, official Chrome will run on ARM64, bringing features like automatic updates, syncing, and better security. This move can help people who use cheap and low-power devices, since ARM chips are popular in mini-PCs, servers, and even some laptops. The Chrome team says they worked to make sure performance is good and that all Chrome features work as expected on ARM. They also mention that this helps web developers test sites on more hardware. For now, Chrome for ARM64 is available as a Debian package, so you can install it on systems like Ubuntu or Raspberry Pi OS. Other Linux distributions may get support soon. Google also says this will help with education and makers, since a lot of schools and hobbyists use ARM devices. Updates and security patches will come at the same time as other platforms.

In the comments, many people are happy and say “finally” because they have waited years for this. Some users talk about how they had to use Chromium builds before, but those missed some features or did not get updates as fast. Others worry about how much RAM Chrome uses, since many ARM devices like Raspberry Pi have less memory. A few worry about privacy and Google’s tracking, but others say you can still use Chromium if you want more control. Lastly, some people ask if this means Chrome will also come to other ARM platforms like Android tablets or even more Linux devices in the future.
