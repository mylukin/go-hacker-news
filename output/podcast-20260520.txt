Let’s start with one of the biggest stories in mathematics and artificial intelligence. OpenAI’s new AI model has solved a famous math problem called the “unit distance problem,” which was open for 80 years. The problem asks: if you put n points on a flat plane, how many pairs can be exactly one unit apart? For a long time, people thought the best way was to place points in a square grid. Most experts believed there was no better way. But the AI found a new method using advanced math from algebraic number theory. This gave even more unit-distance pairs for some values of n. The proof was checked by other mathematicians and is correct. The AI was not built just for math but could reason deeply and connect ideas from different areas. Many top mathematicians say this is a milestone, since it’s the first time AI has solved such a big math problem on its own.

The key idea is that the AI used special number fields, which are sets of numbers with more symmetries than usual, to arrange points in a smarter way. It used advanced topics like infinite class field towers and Golod–Shafarevich theory to prove that these number fields exist and can help make more unit-distance pairs. This breaks the old record, which had not changed since the 1940s. Now, for many values of n, you can get at least n to the power of 1 plus a small delta, like 0.014, which is better than before.

This discovery is important not just for math but also for AI research. It shows that AI can make new ideas and solve problems in creative ways, not just help humans do the work. The connection between algebraic number theory and geometry was a big surprise. It may help solve other hard math problems in the future. The result suggests that AI will soon be a key partner in many kinds of science, not just math.

Next, let’s talk about a major security story. GitHub revealed that 3,800 code repositories were hacked through a bad Visual Studio Code extension. Attackers made a fake extension for VSCode, a popular code editor. When people installed it, the extension sent their secret files like passwords and API keys to the hackers. Many developers use VSCode with GitHub, so the attack reached many code projects. The bad extension was downloaded thousands of times before anyone noticed. Some users saw strange activity in their accounts and told GitHub, which started an investigation. GitHub removed the malicious extension and told all affected users to change their passwords and secrets. GitHub says no core systems were hacked—only individual code projects. They are warning developers to be extra careful when installing extensions and to check if an extension is trusted. GitHub is working on better ways to spot these kinds of attacks in the future, and they suggest using multi-factor authentication and keeping secrets out of code whenever possible.

In the comments, many people are worried and say it is hard to trust extensions now. Some point out that most users do not check extension code before installing it. A few say open-source tools can be safe if enough people review the code, but this does not always happen. Some developers share tips for staying safe, like only using popular or official extensions. One person says GitHub should check extensions more before allowing them. Others say this attack will make developers more careful in the future. Some people say they will now use fewer extensions or switch to simpler code editors. Many agree that this is a big wake-up call for developers to be more careful with their tools.

Now let’s talk about science. Scientists are using human brains, taken from people who have died, to test new drugs. These brains are not alive, but still have living cells and can react to medicine in a lab. The brains come from donors who agreed to give tissue for research. After death, doctors quickly remove the brain and keep it fresh. Scientists cut the brain into thin slices and keep many cells working for a few days. They use these slices to study how real human brain cells respond to new drugs or treatments. This is important because animal brains are not the same as human brains. By using human tissue, researchers hope to find safer and better drugs for brain diseases like epilepsy, Alzheimer’s, or depression. The method is not perfect—these slices do not last long, and cannot show how a drug works in a whole living body. But it is much better than only using animals. Scientists are careful to respect the wishes of donors and their families.

Now, here’s a quick look at a web tool for developers who use large language models. The tool lets you see how fast different “tokens per second” rates feel when using LLMs. It shows how LLMs stream text at different speeds, so you can understand what “30 tokens per second” or “200 tokens per second” really looks like. The tool has four modes: code, text, think, and agent. You can set the speed and see how the text appears at different rates, from very slow to extremely fast. The tool also explains what a token is: a short piece of text, like a word or part of a word. Code usually has more tokens per line than normal text, so the same speed can feel much faster when showing code. When you switch between code and text at the same speed, you notice a big difference. The author says the tool is meant to show this gap, because benchmarks don’t always tell the whole story about how fast an LLM “feels” in use.

In the comments, some readers say the tool is helpful. They like being able to compare speeds and see the difference between code and text streaming. Others say real-world use is even more complex, since prompts and outputs can vary a lot. Some people think fast speed with code can be hard to read, while with plain text it feels more natural. Some wish there was support for other languages or tokenizers, since different LLMs use different systems. Others discuss how token speeds affect user experience, especially in chat apps or coding assistants. Some debate whether tokens-per-second is the best way to measure LLM speed, or if words-per-second would be clearer. Some are surprised at how fast current hardware can stream tokens, while others worry that higher speeds do not always help users, since humans can only read so fast.

Next, let’s update on space. SpaceX is preparing for the twelfth test flight of Starship, set for May 21, 2026. This test will use a new version of Starship and the Super Heavy booster, with new engines and a new launch pad at Starbase. The main goal is to try out all these new parts together during flight for the first time. Many things have been changed based on past flights to help Starship become fully reusable and faster to launch again. The booster’s main test is to launch, separate from Starship, fly back toward the Gulf of Mexico, and try a landing burn. This time, the booster won’t try to return to the launch pad.

The Starship upper stage will also try many new things. It will release 20 Starlink simulators and 2 special Starlink satellites. These two special satellites will scan Starship’s heat shield and send pictures back, to help check the heat shield for future missions. Some of Starship’s heat shield tiles are painted white to act like missing tiles, so the scan system can be tested. One real tile is missing on purpose to see what happens to the tiles next to it.

Starship will also try to relight one Raptor engine while in space. During reentry, Starship will do hard maneuvers to test the flaps and to practice a path like future landings at Starbase. The team will watch closely to learn from each part of the flight. The article also gave a detailed countdown and timeline for each step, from fueling to landing. The launch will be streamed live, and schedules might change, so fans should watch for updates.

Next, let’s look at PopuLoRA, a new way to train language models. PopuLoRA aims to make LLMs better at reasoning by letting them “play” with each other. In this system, a group of LLMs learn together by testing and challenging each other. They ask questions, answer, and check each other's work. When one model makes a mistake, others help correct it. This teamwork helps every model in the group improve over time. The team believes PopuLoRA can make training more efficient and lead to better results. By using many LLMs together, each model gets stronger by learning from the group.

In the comments, some people are excited about LLMs learning from each other, saying it is like how people get smarter by working in teams. Others wonder if this just makes the models agree with each other or if they actually get better. Some worry that the models might all start making the same mistakes if they copy each other too much. A few ask about the cost and if this process uses more computer power. Some wonder how to make sure the models do not reinforce bad answers. One commenter thinks this could help LLMs become better problem solvers. Others want to see more real-world tests and open-source code. Overall, people agree it is an interesting way to push LLMs forward, but they want to see more proof that it works well.

Finally, let’s talk about a developer’s struggle with Apple’s App Store. The developer of Inkwell, an iOS app, has had trouble getting approved by Apple. The app has been stuck in review for almost a month, facing many rejections and confusing requests from Apple. First, Apple said the app needed ways to report or block users, even though it’s just an RSS reader. The developer added these features, plus more links to legal policies, even though it made the welcome screen crowded. Then, Apple said the “Sign in with Apple” button did not work right, so the developer changed how and when it shows up. Apple also wanted to know how Inkwell makes money, so the developer answered those questions. Then, Apple complained that the app lets users access Micro.blog content without using Apple’s in-app purchases, which would give Apple a share of sales. To fix this, the developer removed some features and limited the app’s availability to only the US, where rules are different. Apple also found problems with the app’s design, especially with how “Sign in with Apple” asked for too much information, so the developer made changes. There was another problem: users could not delete their Micro.blog account from the app, so a delete option was added.

The biggest issue is the app’s name, “Inkwell.” Apple said this is a trademark problem, since they once had a handwriting feature called Inkwell. But Apple’s own trademark is now “dead,” and other apps with similar names exist in the App Store. The developer tried to change the name but still got rejected. The developer thinks Apple is using its power unfairly, since the trademark is not active, but Apple still blocks the app.
