## Temporal: A nine-year journey to fix time in JavaScript

- 原文链接: [Temporal: A nine-year journey to fix time in JavaScript](https://bloomberg.github.io/js-blog/post/temporal/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47336989)

The article explains Temporal, a new way to work with dates and times in JavaScript, and how it took nine years to become part of the language. JavaScript’s old Date object caused lots of problems because it was copied from Java in a rush and did not handle time zones, calendars, or date math well.

The Date object was easy to break—if you tried to add a day or a month, it could change the date in ways you didn’t expect. It also sometimes changed the original date (mutated it), which caused bugs. Parsing dates from strings was not reliable, since different browsers did different things. Because of these problems, many developers used libraries like Moment.js, but these libraries made apps bigger and were hard to keep updated.

A group of developers and companies worked together to create Temporal. They wanted it to be a replacement for Date, with clear types for different kinds of time (like exact moments, dates without times, or times without dates). Temporal is immutable, so it never changes the original value. It also handles time zones and calendars correctly, which is important for global apps. There are types like Temporal.ZonedDateTime (for a moment in a certain time zone), Temporal.Instant (an exact point in time, no time zone), and others for just dates, just times, or durations.

Making Temporal was hard. The process took years and needed agreement from many people and companies. Temporal had to be fast and work in all browsers. Teams from Microsoft, Google, Mozilla, Bloomberg, and more all helped. A big help was a shared Rust library called temporal_rs, which let different browsers use the same code and made maintenance easier.

Temporal reached the final stage (“Stage 4”) in 2026, so it will be in the ES2026 JavaScript standard. It is already available in Firefox, Chrome, Edge, and partially in Safari and Node.js. There is still work to do to make sure it works with all web APIs, like date pickers and cookies.

In the Hacker News comments, many people are happy about Temporal, saying it fixes problems they have had for years. Some like that it is immutable and that it handles time zones and calendars well. Others wish it had come sooner, because they spent a lot of time fighting with Date or using big libraries. A few commenters worry about learning a new API, but most think it is worth it. Some ask if browsers will really support all of Temporal, or if there will still be differences. Others mention that time is always tricky, especially with daylight saving changes and leap seconds, but agree Temporal is a big step forward. Several users thank the teams for working together and making a shared library, saying this is a good example for future features. Some hope that other languages will learn from JavaScript’s long struggle with time and do better from the start.

## Making WebAssembly a first-class language on the Web

- 原文链接: [Making WebAssembly a first-class language on the Web](https://hacks.mozilla.org/2026/02/making-webassembly-a-first-class-language-on-the-web/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47331811)

This article explains why WebAssembly (Wasm) is still hard to use on the web, even after many improvements since 2017. WebAssembly lets you run code from languages like C++ and Rust in browsers, but it is not as easy to use as JavaScript.

The article says that WebAssembly is a “second-class” language because it cannot connect to the web platform directly. You must always use JavaScript as a middleman. Loading a Wasm file is much harder than adding a JavaScript file—you need to write or generate extra “glue code” to make things work. If you want to call a web API, like console.log, from WebAssembly, you need complicated steps: sharing memory, encoding strings, and writing special functions in JavaScript. Tools exist to help, but they add more steps and can be different for each programming language.

This extra work makes WebAssembly too complex for many developers, especially for small projects. Most people stick to JavaScript, and only big companies with special needs use WebAssembly. Also, standard compilers don’t make Wasm files that work in browsers without extra tools, and most web guides focus only on JavaScript. All this means that to use WebAssembly well, you must understand both your own language and JavaScript.

The article suggests that the upcoming “WebAssembly Component Model” could fix these problems. With this, Wasm code could call web APIs directly, and loading Wasm could be as easy as loading JavaScript. The idea is that different languages could use the web in the same way and share code more easily. The author gives some examples of how this might look in the future, with code in Rust calling web APIs directly, and JavaScript using functions from Wasm modules as if they were normal JS modules.

In the Hacker News comments, some people are excited about the Component Model and hope it will make Wasm much easier to use. They like the idea of using many languages on the web without needing JavaScript for everything. Others are worried about the complexity of the web platform itself and wonder if Wasm will ever be as simple as JavaScript. Some say that even with better tools, Wasm might always be best for special use cases, like games or big apps, not for simple websites.

A few commenters point out that browser support and standards move slowly, so it may take years before these changes help most developers. Some share stories of trying Wasm and getting stuck with confusing errors or hard-to-debug problems. Others ask if security or performance could become issues if Wasm gets more web power. There are also people who enjoy the current “power user” aspect of Wasm and fear it might get bloated or lose its speed if it becomes too integrated.

Overall, the community agrees that making WebAssembly easier and more “first-class” is important, but there are many technical and social challenges. Some want simple, direct access to web APIs from any language, while others value the web’s current simplicity with JavaScript. Many are watching closely and hope the Component Model will help, but most see that Wasm still has a long road ahead before it is as easy to use as JavaScript on the web.

## Entities enabling scientific fraud at scale (2025)

- 原文链接: [Entities enabling scientific fraud at scale (2025)](https://doi.org/10.1073/pnas.2420092122)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47335349)

This article is about how scientific fraud is happening at a large scale, not just by single people but through big groups and companies. It explains that so-called “paper mills”—businesses that sell fake or low-quality research papers—are helping more people cheat in science, and these groups are growing fast.

The article says the problem is much bigger than many people think. It gives examples of how paper mills work: they make many fake papers, sometimes by copying images or using the same text, and then help get those papers published in journals that don’t check closely. Some editors and authors are working together to let these fake papers through, and sometimes even whole journals or conferences are part of the problem. The study found that some editors approve many more bad papers than normal, and some authors always send their work to these editors. 

There are also “brokers” who connect paper mills to journals. When one journal gets caught and is removed from big databases, the fraud groups just switch to new journals—a process called “journal hopping.” The authors show that the number of fake papers is growing much faster than the number of real science papers, and that current ways to stop fraud—like removing journals or retracting papers—are not enough. In some research areas, up to 4% of papers get retracted, which is much higher than normal. Paper mills also target certain fields more than others, and many fraudulent papers stay in the scientific record even after journals are removed from databases.

The article says that the main reason for this problem is the pressure on scientists to get jobs and funding by publishing more papers. This pressure makes it tempting for some to cheat. Also, some people think the problem is only in certain countries, but the article shows it happens everywhere, including in top journals.

In the comments, some readers are shocked at how organized and big the problem is. Others say they have seen signs of this themselves, like strange papers in their fields or journals suddenly full of low-quality work. Some point out that the use of metrics like the h-index and impact factor makes things worse, because people focus on numbers instead of good science. A few readers think the only real fix is to change how science is rewarded, but they worry this will be very hard because the current system helps some people. Others ask if technology, like AI, can help spot fake papers, but some are worried that as fake science grows, it will be harder for both humans and machines to tell what is real. Some suggest that science needs better ways to report and punish fraud, and that outsiders—not just journals or universities—should investigate these cases. A few readers are discouraged, saying that if fake science keeps growing, it will be hard to trust research at all. But some are hopeful and think with more awareness and better tools, science can fight back against fraud.

## Many SWE-bench-Passing PRs would not be merged

- 原文链接: [Many SWE-bench-Passing PRs would not be merged](https://metr.org/notes/2026-03-10-many-swe-bench-passing-prs-would-not-be-merged-into-main/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47341645)

This article looks at how well AI-generated code patches (PRs) that pass the SWE-bench test would actually be accepted by real software project maintainers. The authors found that about half of these AI-created, test-passing PRs would still be rejected by real maintainers, even after accounting for some noise in the review process.

The main reason is that the SWE-bench automated test is not enough for a maintainer to accept code. The study asked maintainers from three real open-source projects (scikit-learn, Sphinx, pytest) to review 296 AI-generated PRs that passed the automated grader. These maintainers had to decide if they would merge the PR or ask for changes, and they explained why if they wanted changes. The reasons for rejection included poor code quality (like not following project style), code breaking other parts, or not really solving the problem.

The study also checked how often maintainers accept “golden patches”—real, human-written PRs that had already been merged. Surprisingly, maintainers only re-accepted about 68% of these, showing that review is often subjective. To be fair, all scores in the paper are compared against this golden baseline.

Results showed that the automated grader’s pass rate is on average 24 percentage points higher than the maintainer merge rate. In other words, many PRs that pass tests still would not be merged. The improvement in AI models over time (from Claude 3.5 Sonnet up to Claude 4.5 Sonnet and GPT-5) is slower when looking at maintainer reviews than when just using the automated tests. Most rejections now are for code quality instead of obvious bugs, as new models get better at core functionality.

The study is careful to say this does not mean AI agents are fundamentally incapable. Unlike humans, the AI only gets one try and cannot improve with feedback. Also, the study did not use full CI pipelines or require test-writing, making it easier for the AI code to pass.

The authors warn that benchmarks like SWE-bench, which use automated grading, can overstate real-world usefulness of AI code agents. They suggest that benchmarks should be seen as just one signal, not the final word on AI capabilities.

Many commenters agreed with the article’s main point: passing automated tests is not enough for code to be accepted in real life. Some maintainers in the comments said they see similar things in their own projects—AI-written code often needs style or structure changes, or misses edge cases that tests do not cover. Others pointed out that code review is about more than just “does it work?”—it’s also about long-term maintainability, clarity, and fitting into the project.

A few people noted that even human-written PRs are sometimes rejected for minor issues, so the bar for AI is not impossibly high. Some commenters thought the study could have been even tougher on the AI by including full testing and review processes. Others said the lack of iteration for AI is a big weakness, since real development is a back-and-forth process. A few were optimistic, saying that as AI agents get better at taking feedback and understanding style, these problems could go away. But most agreed that, for now, automated test scores alone do not mean AI is ready to replace human coders for real-world software work.
