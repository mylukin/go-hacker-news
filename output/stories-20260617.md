## A robot is sprinting towards you. Do you want it running on Claude or Grok?

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48576824)

The article is about an experiment where the author made 11 AI models play a battle royale video game 30 times to see which would win more. The main question is: if a robot was running toward you, would you want it controlled by Claude or by Grok?

The winner in most games was Grok 4.1 Fast, which won 13 out of 30 games. Claude Sonnet 4.6 came second with 5 wins, but at a much higher cost per win. Some models, like GPT 5.4-mini, DeepSeek, and Kimi K2.6, never won a game, even though they did some things well, like getting kills or staying alive. The experiment used a 2D world with weapons, cars, and a shrinking zone to push players together. Each model could "think," act, and write its own memory and personality files between matches. The models only saw each other as letters, not by name.

The author saw that "model alignment"—how much a model is trained to be polite and helpful—affected how they played. Claude Sonnet tried to make friends and help others, even in a fighting game, sometimes giving away its position or asking for truces. Grok, on the other hand, was more aggressive and focused only on winning, with no hesitation about fighting. Grok used smart tactics, like car ramming, and kept track of what worked in its memory file.

Benchmarks usually test how smart or helpful a model is, but in this game, being less "aligned" (less polite and more selfish) led to more wins. Grok was cheaper per win than Claude by 27 times. Some models were good at getting kills but not at winning the game, which shows that "kills" and "wins" are not the same. Cost also mattered—a model that is cheaper but loses all the time ends up being more expensive for real tasks.

The article shares fun moments from the games, like big kill streaks, funny car chases, and even models writing about their mistakes or funny things that happened, like getting stuck or drowning. Each model wrote its diary differently: Grok's was proud and full of stats, GPT 5.4 wrote like a guide, and Claude Sonnet wrote self-reviews and tried to improve each time.

In the end, the author says that the best model depends on the job. If you want a robot to win at all costs, you want Grok. If you want a safe, careful, and friendly robot (like in your house), you want Claude. Benchmarks do not show how models act in all real tasks, especially where being careful or kind matters.

Others liked that Grok found clever ways to win, showing that training style changes how AI acts. One person pointed out that the cost differences matter a lot if you use these models in real products. Another said that benchmarks should be more like real tasks, not just test scores. A few suggested running the test with even more models or in other types of games to see if the results are the same. Others just enjoyed the funny stories and thought the diaries showed real signs of "personality" in the AI. ## Lore – Open source version control system designed for scalability

- 原文链接: [Lore – Open source version control system designed for scalability](https://lore.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48571081)

Lore is a new open source version control system made by Epic Games. It is built for big projects and big teams, especially where you have code and lots of large files, like games or digital art.

Lore is easy to start with—just run it locally in a few minutes. You can then make it bigger as your project grows. Lore is fast because it only downloads what you need, when you need it. It saves space by using shared data and lets you work on branches easily and quickly. You can trust the history in Lore because every change is saved with a special hash, making it tamper-evident. The command line interface is simple, and there are APIs for many languages, like Python, Go, and Rust. Lore works in a centralized way and uses a Merkle tree to track changes, which helps with speed and safety. It stores files in chunks, which makes it good at handling big files without using too much space. You can fetch only the parts of the project you need, so you don’t have to download everything. Caching makes it faster for big teams, and branches are light, so you can switch between branches quickly.

Lore is fully open source under the MIT license, and Epic Games wants people to join and help build it. The project is meant for teams who need to handle large amounts of data and want to work together easily.

## How we run Firecracker VMs inside EC2 and start browsers in less than 1s

- 原文链接: [How we run Firecracker VMs inside EC2 and start browsers in less than 1s](https://browser-use.com/posts/firecracker-browser-infra)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48556561)

This article explains how Browser Use made their cloud browsers much faster and cheaper. They now start a browser in under a second and lowered the price to $0.02 per hour.

First, cloud browsers must start quickly, be isolated for security, and not cost too much. Each browser needs its own safe space so it cannot see or harm others. Usually, virtual machines (VMs) are used for this, but regular VMs are slow and expensive to start and stop.

Browser Use wanted to keep each browser in its own VM, but without the usual slowness or high cost. They chose Firecracker, a lightweight VM system, and ran it inside Amazon EC2 servers. This setup creates “nested VMs” (VMs inside VMs), which is not common and can slow things down.

Before Firecracker, they used unikernels (with Unikraft), but scaling up when many users joined at once was slow and needed manual work. This caused outages when traffic spiked. With Firecracker, they built an automatic control plane to decide how many VMs to run and where to put them, making scaling fast and hands-free.

Running Firecracker inside EC2 (not on bare metal servers) lets them add new servers quicker and cheaper, even though it means one extra layer of virtualization. They chose this tradeoff and worked to make the system as fast as possible.

To speed up browser startup, they save the VM state right before launching Chromium (the browser engine), so new sessions resume from a snapshot instead of starting from scratch. They fixed memory delays by using bigger memory pages and a custom Linux handler, cutting the resume time from almost 10 seconds to about 3 seconds, and reducing the number of memory requests by 91 times.

For CPU bottlenecks, they let the system spread out CPU work during browser startup, then assign each running browser to fixed CPU cores. They also handle “hyperthreads” carefully to avoid resource fights and set high priority for browser VMs, which stopped session failures during big tests.

To avoid anti-bot detection, they run a custom version of Chromium that is harder for websites to spot as automated, even though the browsers run without a screen (“headless”). They use real user fingerprints for each session, which helps avoid blocks 81% of the time—much better than other providers.

When a browser is ready, users connect to it over a WebSocket using the Chrome DevTools Protocol. Edge routers pass connections to the right browser VM, but the control plane decides where each browser lives.

Now, the full process from request to ready browser takes under 1 second for most users. The biggest delay left is starting Chromium itself. Their next plan is to snapshot the browser after Chromium has started, so new sessions wake up with the browser already running.

In the Hacker News comments, many readers praised the technical achievement, especially running Firecracker in nested VMs on EC2, which is not typical. Some were surprised this works so well, given the extra VM layer usually adds slowdowns. Others asked why not use containers for even lighter weight, but the team explained VMs give better isolation and security.

Several commenters discussed the trade-offs between unikernels, containers, and VMs. Some liked how the control plane can track browser states in real time, instead of waiting for slower AWS monitoring tools. People also liked the use of memory snapshots and custom page fault handlers to speed up startup.

A few readers were curious about the costs, asking if the prices could get even lower or if there are hidden limits when packing many browsers on a single EC2 instance. Some pointed out that skipping Chromium startup (the next step) will be hard, since keeping the browser “fresh” for every user is tricky.

Security experts in the comments thought the isolation model was strong, but wondered about possible side channels in nested virtualization. Others asked how the system handles browser updates or different user profiles.

Overall, most commenters found the work impressive, with a few suggesting ways to go even faster, like using newer hardware or exploring other virtualization tech. Some shared their own stories about running browsers in the cloud and compared notes on performance and reliability.

## RFC 10008: The new HTTP Query Method

- 原文链接: [RFC 10008: The new HTTP Query Method](https://www.rfc-editor.org/info/rfc10008/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48568502)

This article is about a new HTTP method called QUERY, now an official standard. QUERY is made for sending requests with large or complex data, while still being safe and idempotent, like GET, but allowing a request body, like POST. 

The main idea is that GET is safe (doesn’t change data) and idempotent (can be retried without problems), but it can’t send big queries because all data must fit in the URL. POST lets you send big data in the body, but POST is not always safe or idempotent—so automatic retries or caching can be risky. Sometimes, people use POST for big queries, but servers and clients can’t always know if that POST is safe to repeat or cache. The new QUERY method fixes this: it lets you send big queries in the body, but is always safe and idempotent. That means it’s okay to cache, retry, or repeat. Servers can also give a URI for the query or the result, so you can use GET later to fetch the same results. There are also new headers like Accept-Query to help clients and servers talk about what query formats they support. Security is better because sensitive query data does not go into the URL (which can be logged), but instead stays in the request body. QUERY also works with caching, OPTIONS, and conditional requests much like GET.

In the comments, many people are happy about this update, saying it fixes an old problem with HTTP. Some developers point out that this will help APIs that need complex filters or search forms, which often struggle with URL length limits. Others mention that tools, proxies, and old servers may need updates before QUERY works everywhere. Some are worried that few browsers or libraries will support QUERY right away, so adoption could be slow. A few people compare QUERY to similar methods like SEARCH or REPORT from WebDAV, and wonder if adding another method is needed, but most agree QUERY is clearer and simpler. Some commenters like that QUERY helps keep URLs shorter and more private. Others ask how caching and redirects will work in practice, and if developers will understand when to use QUERY, GET, or POST. There’s some debate about naming, but most agree QUERY makes sense. Finally, a few warn that security needs care, as with any new HTTP method, but overall the reaction is positive and hopeful.
