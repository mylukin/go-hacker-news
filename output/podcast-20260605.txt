Next, Microsoft has open sourced a tool called pg_durable, which brings durable workflow execution into PostgreSQL. With pg_durable, you can run long background jobs directly in your database, and if anything fails, the workflow can pick up where it left off. You write your workflow in SQL, and pg_durable saves each step, or checkpoint, so you never lose progress. Before, people had to use cron jobs, special workers, queues, or make extra tables to track jobs. Now, it can all be done inside Postgres. This is useful for processing large batches of data, running maintenance, or calling outside APIs in order. You can run steps one after another, or even in parallel. It works with Postgres 17 and 18, is written in Rust, and does not send any telemetry to Microsoft.

In the comments, many people like the idea of moving workflow logic into the database, saying it can make systems simpler. Some like that it uses only SQL and Postgres. Others worry that putting too much logic into the database could make it harder to manage or scale. A few note that if you can’t add extensions to your database, you can’t use this tool. Some compare pg_durable to other tools like Temporal, Airflow, or cron jobs, saying it is easier for some jobs but not as flexible for all needs. Many say it is a good tool if you already use Postgres and want simpler, more reliable background jobs.

Let’s move on to AI. Google has released new versions of its Gemma 4 AI models, which are smaller and faster, made for laptops and phones. These models use Quantization-Aware Training, or QAT, which lets the model keep its quality even when it is made smaller. QAT works better than the old way, Post-Training Quantization, because it lets the model plan for being smaller during training. Gemma 4 now comes in two new formats: one for regular computers and one for mobile devices. The smallest model, Gemma 4 E2B, needs only 1GB of memory. Google also used special tricks to make the models faster and use less memory on phones, like using 2-bit numbers for some parts, and shrinking the memory for storing past conversation.

Developers can download these models in many formats, use them with tools like llama.cpp, Ollama, or LM Studio, and even run them on the web or on Apple hardware. There is clear documentation, and the models can be fine-tuned for your own needs.

In the comments, many users are excited to run advanced AI on their phones without the cloud. Some like that Google gives open access and many formats. Others discuss how QAT is a step forward, since the model is trained to handle compression. Some users ask how much quality is lost at small sizes, and if the 2-bit trick makes answers worse. Others wonder if these methods can help other AI models or save power on servers. There are also privacy questions, and tips for running the models on different hardware. Overall, the community is happy and ready to try these new models.

Next, an article explores using AI agents to help write tests with Test-Driven Development, or TDD. The author says most AI agents today are not good at writing tests, because they learn from humans, whose tests are often not great either. The author uses Kent Beck’s “Canon TDD” as a guide and suggests telling agents to follow strong rules. They also use a process called “specify-encode-fulfill”—first decide what to build, then write tests, then write code to pass the tests. The author made a “Test Design Review” agent to check for mistakes in tests, and a “Software Design Review” agent for code quality. This way, the AI can pause and suggest cleaning up code before making more changes. The main idea is that combining AI with classic software rules gives better results.

In the comments, people agree that AI agents are bad at writing tests, but like the idea of giving them more guidance. Some say most human-written tests are bad too, so the AI is just copying what it sees. Others think “specify-encode-fulfill” is a clear way to do TDD, for both humans and AI. Some worry that using AI for tests will make people lazier about learning good skills, while others like the idea of a review agent to catch mistakes. Many think these tools could help teach beginners to write better tests.

Now, a new method from the University of Rochester can turn ocean water into drinking water without making waste. The method uses a special black metal panel that works with sunlight to clean the water and collect the leftover salts. The panel is made with lasers, soaking up light and pulling water across its surface. Sunlight heats the panel, turning water to steam and leaving salts behind. The salts move to the side, away from the main area, so the panel never clogs. The team tested the panel with water from different oceans and found it could clean itself and keep working well. The process collects almost all salts in solid form, including valuable ones like lithium. The team even separated lithium from other salts using tiny particles on the panel. The process uses only sunlight and needs no chemicals, making it cheap and clean.

In the comments, some users are excited by the way it makes fresh water without polluting the sea. Others ask if this system can work at large scale or in places with little sun. Some wonder about the cost and how long the panels last outdoors. A few point out that collecting lithium from water could help the environment, while others ask about safe storage of the salts. Many like that no chemicals are needed, but a few worry about cleaning or repairing the panels. Some say solar desalination is not new, but this panel solves old problems. Many want to see real-world tests, not just lab results. Overall, people think it’s a smart invention with good potential, but want more proof.

Now, let’s discuss why transformer models are so powerful for language tasks. The article explains that transformers can describe some languages in a much smaller and simpler way than older models like finite automata, RNNs, or logic formulas. For some problems, a small transformer can do the job, but other models would have to be huge to do the same thing. This is because transformers can use attention to count or track information very efficiently. But, this high “succinctness” makes transformers harder to analyze or verify. For example, checking if a transformer accepts any words at all is an EXPSPACE-complete problem, which is much harder than for automata.

In the comments, some are surprised that transformers are less expressive than RNNs, but can be much more compact. Some think this helps explain why transformers work so well in practice. Others worry that this compactness makes them hard to debug or check. Some are excited by the theory and think it could help future research or model design. Others wonder if we should compare models by succinctness, not just raw power. There are questions about whether the results hold for other types of transformers or lower precision, and how this could help with model compression or explainability.

Finally, the UK government has stopped using Stripe for payments on its GOV.UK Pay service and will use Adyen, a Dutch company, instead. This change affects many payments for local councils, police, and public groups, with a contract worth up to £25.3 million. GOV.UK Pay was made to make payments easier and cheaper for public groups, handling over 137 million payments worth about £9.2 billion since 2016.
