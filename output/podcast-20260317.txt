First up, Eric Lengyel shared big news about the Slug Algorithm, a way to render fonts directly from Bézier curves on the GPU. Eric first built this method ten years ago, and since then it has been used in many fields—from video games to medical tech—by well-known companies like Activision, Adobe, and Ubisoft.

The main point of Slug is to draw text and vector graphics straight from curves, not from images or textures. This gives you smooth, sharp text at any size or angle, and it works fast enough for games. The algorithm avoids the usual floating-point errors that often cause glitches or missing pixels. Some parts of the algorithm, like the root eligibility check and the winding number, have not changed since 2017. Other features, such as “band split optimization” and old supersampling tricks, were removed to keep the code simple and fast. Multi-color emoji are now handled by stacking glyphs, which is easier and faster.

The biggest new feature is “dynamic dilation.” Before, users had to guess how much to expand a glyph’s edge to avoid visual problems. Now, the algorithm does this job automatically for each glyph, making the output cleaner and saving GPU work. Eric even explained the math, showing how each vertex is moved just enough to cover half a pixel using a Jacobian matrix.

Eric also announced he is giving up his patent on Slug, making it free for everyone. He released reference code for the shaders on GitHub with the MIT license. In the comments, many praised Eric for sharing his work and helping open-source projects and small developers. Others were excited to try Slug in their own code, and there was much talk about the technical details. Some users hope this will lead others to release their patents too. People also had questions about performance on new GPUs, and some shared their own struggles with font rendering. Overall, the reaction was very positive, calling this a win for the graphics and developer world.

Next, we have news about Python 3.15. This new version will include a Just-In-Time compiler, or JIT, and it is finally showing real speed gains. After many years of slow progress and even losing funding, the Python team reached their goal early. Python JIT is now about 11-12% faster on macOS and 5-6% on Linux.

Earlier JIT versions in Python 3.13 and 3.14 were sometimes slower than the normal interpreter. The big change came from growing the team, splitting the work into smaller pieces, and letting more people help. A key technical change was using trace recording, meaning the JIT records and speeds up common code paths. This was actually found by accident, and led to another quick win called dual dispatch. Removing extra steps for reference counting also made things faster and easier to learn about the JIT. Daily performance tests quickly showed if new changes helped or hurt. The team also learned from PyPy contributors and others.

In the comments, many readers were glad to see true progress for Python’s JIT, though some are careful and want to see if the speedups help real apps, not just benchmarks. There were worries about the lack of free-threading. Some praised the plan to let new people join the work, saying it helps the whole Python world. Others discussed technical details, like dual dispatch and trace recording, and compared this JIT to PyPy. A few are concerned about keeping the JIT stable and easy to maintain. Several thanked the volunteers, especially since funding was lost. Most people are hopeful, but waiting to try the new JIT on big projects.

Our third story is about "Get Shit Done"—or GSD—a tool that helps developers use AI to write and manage code. GSD is for people who want to describe what they need and let AI do the building, without the complex planning you find in big companies.

GSD fights “context rot,” where the AI forgets or mixes up details as a project grows. It keeps things clear by using files like PROJECT.md and ROADMAP.md, and making sure each step has the right info. The process is broken into phases: describe the project, answer questions, approve the plan, and let the AI work step by step. Tasks are grouped so independent work can run in parallel.

You can use GSD in “quick mode” for small jobs, or do the full process for bigger features. Each task gets its own git commit for easy tracking. The system is modular—you can add or remove phases, start new goals, or change plans as you go. GSD works with many AI models and lets you control how much is automated. For safety, you can block AI from seeing secret files.

There are commands for starting projects, planning, working, and checking results. GSD helps with UI design, code mapping, debugging, and keeping notes. Settings let you pick AI models or how git branches are handled. Setup is simple and works on Mac, Windows, and Linux.

In the comments, users liked GSD for solving real problems with AI code, such as lost context and messy results. Some said GSD is easier than similar tools. Security was a concern, especially with the “skip permissions” features. Many praised the clear planning and git history. Some wondered how GSD will work for teams, since it is aimed more at solo developers. Others liked the modular design but said users still need to check AI results, because AI can make mistakes. The comments were mostly positive, but people are careful about the risks.

Next up, Edge.js is a new tool that lets you run Node.js apps inside a WebAssembly sandbox. This makes them safer and faster to start than using containers like Docker. Edge.js aims for full Node.js compatibility, but adds strong sandboxing for risky code using WebAssembly and WASIX.

Developers can run their existing Node.js projects—including native modules—without changing code. Only the unsafe parts are sandboxed, while JavaScript runs at full speed. Edge.js is almost fully compatible with Node.js v24, and works with popular frameworks like Next.js and Astro. The performance is about 5–20% slower than native Node.js, or 30% slower when fully sandboxed, but the team plans to improve this. The project was built quickly with help from AI tools like GPT and Codex.

In the comments, users were excited about secure Node.js without Docker and happy to see almost full compatibility. Some want to use it for running untrusted code. Others worry about the real security of WebAssembly sandboxes and the performance hit. There are comparisons with Bun and Deno, and requests for more documentation. Overall, people see Edge.js as a promising step for safe edge computing, but want to see more real-world use.

Now, let’s talk about why Node.js should have a virtual file system. The article explains that Node.js apps often need to work with files, but the current system is not flexible enough. A virtual file system would let you create files in memory, not just on disk. This helps with testing, security, and simulating files for tools like Webpack or Vite. Other systems, like .NET, already have this. The author says adding a VFS would not break old code, and could make Node.js better for testing and cloud work.

In the comments, some developers want this feature for testing and sandboxing. Others worry it would add too much complexity. Some point out that you can already use in-memory file system libraries, but having it built-in would be easier. There are stories about problems caused by weak virtual file support. The comment section shows real debate about whether Node.js should add this feature.
