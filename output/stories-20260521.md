## Flipper One – we need your help

- 原文链接: [Flipper One – we need your help](https://blog.flipper.net/flipper-one-we-need-your-help/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48220647)

Flipper Devices is building a new device called Flipper One, and they are asking the community for help. Flipper One is not an upgrade to Flipper Zero but a separate project: it's a fully open Linux computer for network and hardware hacking.

The team wants Flipper One to be the most open ARM device, with all hardware and software fully documented and supported by the mainline Linux kernel. They are working to remove all closed-source parts and vendor-specific code, but there is still one binary blob left for RAM initialization. The device uses a powerful main CPU together with a small microcontroller; both work together, letting you run some features even when Linux is off. Flipper One comes with many ports: two Gigabit Ethernet, Wi-Fi 6E, and an M.2 slot for things like 5G modems, SSDs, or software-defined radio modules. It also has GPIO pins for simple DIY add-ons, and the full hardware design is open, so anyone can download 3D models and make custom modules or cases.

Flipper OS, the system running on Flipper One, will support "profiles"—snapshots of the OS with different settings, which you can quickly switch between. This should make it easier to use Flipper One for many tasks without breaking the system. The team is also building a simple, menu-based UI called FlipCTL for small screens, so you don't have to use a full desktop environment on a tiny display. Flipper One can also act as a small desktop computer, media box, or network tool, and supports AI models running locally, even without internet. The whole development process is open: everyone can read, join, and edit the documentation and help with hardware, software, docs, or testing.

In the Hacker News comments, many users are excited about the open hardware and hope it will push the industry toward fewer closed drivers and binary blobs. Some people worry about the last closed source part (the DDR trainer) and wonder if it will ever be fully open. Others like the co-processor design, especially the ability to control the device with only the microcontroller. There are questions about security, especially since the device is powerful and open—some are concerned about how it could be used for both good and bad purposes.

A few users compare Flipper One to the Raspberry Pi and wish Pi had been this open from the start. Some praise the Flipper team's honesty about the project's risks and the difficulty of keeping everything open. Others discuss the challenges of supporting new hardware in the mainline Linux kernel and mention that sometimes upstreaming support can take years. People are also curious about the device's price and how Flipper Devices will handle the high costs of open development. Several commenters want to help, either by writing code, testing hardware, or spreading the word. Overall, the community seems hopeful but realistic about the challenges ahead, and many are cheering the team on for trying something this ambitious.

## Blog ran on Ubuntu 16.04 for 10 years. I migrated it to FreeBSD

- 原文链接: [Blog ran on Ubuntu 16.04 for 10 years. I migrated it to FreeBSD](https://crocidb.com/post/this-blog-ran-on-ubuntu-16-04-for-10-years-i-migrated-it-to-freebsd/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48227397)

This article is about moving a personal blog from an old Ubuntu 16.04 server to a new FreeBSD server. The author had the blog running for ten years on Digital Ocean, but the system was outdated and not secure anymore, so it was time to change.

The author chose a Hetzner VPS in Germany, which was cheaper and had better specs than the old one. The new server runs FreeBSD, a system known for its stability and security. Instead of using Docker, the author used FreeBSD Jails, which are like lightweight virtual machines that help keep each site separate and safe. There’s also a tool called Bastille that makes Jails easy to manage.

The setup involved creating one Jail for each website and another for the main web server (using Caddy instead of nginx). This structure means if one site is hacked, it can be fixed without touching the others. The author explained how to set up the network, configure the firewall, and mount site directories into each Jail. Deploying updates is done with simple scripts. 

The author did benchmarks to compare the old and new servers. The new FreeBSD server was much faster and could handle more requests, but some of that advantage came from better hardware and a better network location, not just the software. The author also found and fixed some tuning problems (like connection limits) to make the new server run even better.

In the comments, many people praised the clear description of FreeBSD Jails and Bastille. Some said they were inspired to try FreeBSD for their own projects. Others noted that FreeBSD is less common than Linux but can be a great choice for web hosting, especially for learning and security. A few commenters worried about keeping up with FreeBSD updates, since it’s less popular and sometimes has less third-party support.

Some readers said using Jails is “old school” but still very effective, and they liked seeing a real-world example. Others shared their own stories of running servers for many years without trouble, and some argued that keeping software updated is more important than switching systems. A few people suggested using managed hosting or static site services to avoid these challenges, but others replied that self-hosting is a good way to learn important skills.

There was debate about Caddy vs nginx. Some people liked Caddy for its easy SSL setup, while others preferred nginx for its flexibility. A few mentioned that FreeBSD’s documentation and tools are high-quality, making the switch easier than expected. Some readers wondered if the performance gains were mostly from better hardware, but most agreed the project was a fun and interesting learning experience.

Overall, the comments showed a mix of practical advice, personal stories, and encouragement for trying new systems. Many liked the author’s curiosity and hands-on approach, even if most web traffic is now from bots and crawlers.

## Show HN: Freenet, a peer-to-peer platform for decentralized apps

- 原文链接: [Show HN: Freenet, a peer-to-peer platform for decentralized apps](https://freenet.org/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48223362)

Freenet is a new peer-to-peer platform that lets people build and use apps without big tech or central servers. It aims to help users communicate, collaborate, and even trade online in a way that is hard to block or control.

Freenet connects computers in a network where each one helps share data. The network is organized like a ring, and messages travel quickly between computers, so it can support millions of users. You don’t have to trust a company or pay for servers; everything happens between people’s own devices. Apps on Freenet run in your web browser and look like normal websites. But they are different: no one can easily take them down, they don’t collect your data, and they work directly from user to user. If you are a developer, you can use popular programming languages like Rust and TypeScript to build apps for Freenet. You don’t need to worry about server costs or cloud rules. The project is made by a small team and gets money from donations and grants. They hope more people will support them to build a better, more private internet.

In the comment section, some people are excited about Freenet’s ideas. They like the plan to avoid big tech and protect privacy. A few users remember older versions of Freenet and are happy to see it being updated for today’s web. Some developers like that they can use modern tools and do not have to pay for servers. Others wonder about possible problems, like how to stop illegal content or spam on a system no one controls. A few worry that making things too “unstoppable” could be dangerous. There are questions about how easy Freenet is to use compared to other peer-to-peer tools, and some suggest it may be hard for normal people to get started. Some users point out that similar projects have struggled before, but hope that Freenet’s new design will work better. Others ask about the project’s funding and if it can last. Overall, people think Freenet is interesting, but they want to see how it works in the real world.

## Was my $48K GPU server worth it?

- 原文链接: [Was my $48K GPU server worth it?](https://rosmine.ai/2026/05/13/was-my-48k-gpu-worth-it/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48184402)

This article is about someone who quit their big tech job and spent $48,000 to build a six-GPU server at home, named “grumbl.” The main question is whether buying their own GPU server was better than renting GPUs from a cloud service.

The writer explains that the server was expensive, but it was still less costly than losing more months of income while waiting for experiments to finish on slower hardware. They chose RTX 6000 Ada GPUs instead of even more expensive H100s or A100s because of a better balance between price and performance, especially for machine learning tasks. Building this server in an apartment was hard because of power limits—one circuit was not enough, so they used two power supplies in two outlets. To be safe, they hired a pro to build it, though in the end, they moved the server to their parents’ house where power upgrades were easier.

To compare buying vs. renting, they tracked GPU usage and power use closely. They found that if the server was used at least 85% of the time, buying it made sense compared to cloud rental prices. Their actual usage was about 76% on average, and 85% after a certain date, which was a little less than expected but still high. Maintenance and downtime were stressful, and sometimes the server had issues, like failed risers. When they added up the costs, including $3,000 for electricity, they found they saved about $17,000 over renting cloud GPUs. Now, every day of use saves them more money.

But money was not the only reason for this project. The writer wanted to do interesting, risky research, and owning the server helped them try many things fast. They also had a big research breakthrough, with lots of interest after launch.

In the advice section, the writer warns that building a custom GPU server is risky and not for everyone. Mistakes can be very expensive. For most people, using cloud GPUs or smaller setups is fine. They also say building and maintaining this server took a lot of time, which is another hidden cost. Renting vs. owning changes your mindset: when renting, you think before running each experiment because of cost; when owning, you feel you should keep the GPUs busy all the time. Insurance was a problem—they needed business insurance, not just renter’s insurance. If they did it again, they would buy a standard server and put it in a data center, but they would miss having the server nearby.

In the Hacker News comments, some people said owning GPUs is good if you use them a lot, but for most, renting is safer and easier. Others liked the honesty about the real costs, especially time and stress. Some said the feeling of owning your hardware is great—you can experiment fast, and there are no waiting times or surprise cloud bills. But many warned about hidden costs: hardware can break, power and cooling are expensive, and moving big servers is tough. A few pointed out that cloud prices may drop over time, making ownership less attractive later. Some shared their own stories of building home labs or renting GPU time, with mixed results. There was also discussion about insurance problems and how hard it is to get coverage for expensive home equipment. Finally, some readers said having your own server is fun and gives you control, but you have to be ready for challenges and surprises.

## Seattle Shield, an intelligence-sharing network operated by the Seattle police

- 原文链接: [Seattle Shield, an intelligence-sharing network operated by the Seattle police](https://prismreports.org/2026/05/20/seattle-shield-private-companies-surveillance/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48226588)

Seattle Shield is a private intelligence-sharing network run by the Seattle police. Big companies like Amazon and Facebook, as well as law enforcement and ICE, are members and can share and receive information.

The network started in 2009 and is supposed to help stop terrorism by sharing information about suspicious activity between police and private groups. Reports show that most alerts sent in recent years were about protests and traffic, not terrorism. Members get details about local events, possible threats, and even protests against groups like ICE. Sometimes, photos of people or cars end up in reports, which are sent to hundreds of agencies and private security groups. There are also updates on dignitary visits, crime, and tips on how to keep property safe.

The article says there’s little public oversight or proof that the network really helps stop crime. Civil rights groups were not aware of the program, and it is unclear if anyone is checking to make sure the network is used fairly. Some members, like the Church of Scientology and the U.S. Navy, have left the group. The Seattle Shield is similar to other “Shield” programs across the U.S., based on a model from New York after 9/11, and is part of a larger Global Shield Network. 

The article shares concerns that protest activity, even legal speech, could be seen as a terrorism risk and that private companies are being asked to report on people. There is worry that these systems could be used to watch and control the public, not just stop real threats. Some police and network leaders say the system helps share information in emergencies, but others say there is not enough proof it works.

In the comments, some people say this network is dangerous. They worry it could be used to track innocent people, especially those who protest or disagree with the government. One top comment says it creates a “panopticon,” meaning people feel watched all the time. Others note that private companies working with police is risky, since business interests might not line up with public good. Some tech workers ask if their own employers are involved and say it is scary to think their company might help the police in this way. A few say that sharing protest information is wrong, and could hurt free speech. There are also comments saying these types of systems are everywhere now, not just in Seattle. Some ask why the public was not told about this network, and if there is any way to see what information is being shared about them. A few defend the idea, saying police need ways to stop threats and keep people safe, but most think the system needs more oversight and clear rules. Many agree that more light should be shined on these networks to protect privacy and rights.

## Indexing a year of video locally on a 2021 MacBook with Gemma4-31B (50GB swap)

- 原文链接: [Indexing a year of video locally on a 2021 MacBook with Gemma4-31B (50GB swap)](https://blog.simbastack.com/indexed-a-year-of-video-locally/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48222733)

This article talks about how someone used a five-year-old MacBook to index a year of video footage using a local AI model called Gemma 4 31B, instead of cloud tools. The writer splits their year between working in Africa and software development, and has a huge collection of unlabeled video clips from many devices.

They first tried using paid online tools for editing and organizing, but found problems: generative AI video isn’t right for real travel brands, and the cost didn’t make sense for their needs. Plus, most AI editors expect your clips are already labeled, which was not true here.

Instead, the main challenge was making an index—a way to search video clips in plain English. The author built a local system to do this. The process creates a “sidecar” text file for each video. This file has details like time of day, lighting, faces, location, keywords, transcript, and a description, all made by the AI looking at frames and audio. The tools used include ffprobe for video data, exiftool for GPS, ffmpeg for images, WhisperX for speech, and insightface for faces. The AI vision model then writes all the information into the sidecar.

The author ran this on their MacBook with 64GB RAM, pushing the laptop with 50GB of swap memory. They were surprised that an old laptop could run such a big AI model locally and well. Most of the code was written by Claude Code AI; the human focused on the system design and fixing bugs. Some bugs included API changes, permission errors, schema mistakes, and errors in filtering clips.

The biggest lessons were: use strict schemas for the AI, run big jobs local and only use cloud AI for the hardest cases, and that most AI video editors solve the wrong problem—they need to start with a searchable index. Now, with the index built, editing is easier. But the social media channels for the lodge are still quiet—the index is only half the solution; editing is next.

In the comments, people praised the technical depth and creativity of the solution. Many were impressed by running such a large AI model on a laptop and asked about performance and SSD health. Some worried about SSD wear from so much swap use, but others pointed out that short bursts are fine. Several readers liked the idea of sidecar files for easy portability and search. A few thought cloud-based services are still better for scale, while others preferred local processing for privacy and cost. Some mentioned that AI video tools are hyped but miss real-world needs, and agreed that indexing is the true first step. A couple of readers asked for the code, and the author later shared the GitHub link. Others pointed out that hiring a human editor might still be the best answer, especially for creative work. Overall, the project sparked a good discussion on practical AI, hardware longevity, and what really helps with big video archives.
