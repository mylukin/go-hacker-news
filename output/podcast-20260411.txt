First, we look at Anthropic’s new AI model, Mythos, used for finding security bugs in software. The article’s author and their team checked if smaller, cheaper AI models could do the same work. For example, all eight small models found a serious bug in FreeBSD, and some even explained a 27-year-old bug in OpenBSD. In basic tests, small models sometimes did better than big, expensive models, especially at telling if code was truly vulnerable. Their company used many models to find and fix bugs in real projects like OpenSSL and curl and found that no single model is best for everything. Sometimes a small model is better, sometimes a big one wins.

They tested models on three main tasks: separating real bugs from false alarms, finding a dangerous FreeBSD bug, and understanding a tricky OpenBSD bug. The best model changed for each task. Models could find bugs, but often made mistakes checking if a bug was really fixed, so a careful system is needed to avoid false alarms. The article also notes that making working exploits is much harder. Mythos was creative here, but open models gave other good ideas too. Still, the main job for defenders is to find and fix bugs, not to attack systems.

Next, an article explains why Apple Silicon Macs only let you run two macOS virtual machines at once, and how the author got around this limit. Apple’s license says only two VMs are allowed, and if you try more, you get an error. The author checked the Virtualization framework but found out the limit is set in the macOS kernel. They found a kernel variable called `hv_apple_isa_vm_quota` that controls the VM count. If you use a development kernel, you can set this variable and raise the limit. To do this, you need Apple’s Kernel Debug Kit, build a dev kernel for your Mac, and boot with it, turning off some security features and using special recovery commands.

With this setup, the author could run up to nine macOS VMs at once. But using a custom kernel means you cannot update macOS normally, and you must switch back to the normal kernel for updates. The author hopes future tools or kernel extensions could make this easier.

Hacker News users were surprised by Apple’s strict VM limit and interested in the technical details. Some said this shows how Apple controls their hardware and software. Others noted that most people do not need more than two VMs, but for developers and testers, this is a real problem. Many think using a dev kernel and disabling security is risky and not for most users. Some were impressed Apple left a “back door” for developers, even if not documented, and wondered if Apple will close it in future macOS versions. There was talk about how Linux and Windows do not have this kind of VM limit, and why Apple enforces it by license, not just technical reasons. A few shared their own stories of running many VMs and hope Apple will make it easier for advanced users.

Now to a story about AI benchmarks. Researchers at Berkeley found that many top AI agent benchmarks can be hacked easily, getting perfect scores without solving real tasks. They built a simple tool that could use tricks like wrapping test files or reading answer files from disk. For example, in FieldWorkArena, one line of code could make all tests pass, even if the answers were empty. In OSWorld, the tool just downloaded the “gold” answer file for a perfect score. Some benchmarks had security holes, letting the tool run any code it wanted.

These are not just theories—some AI teams have used tricks to boost their scores. Benchmarks sometimes keep answers or tests in places agents can see, or use weak checking code. Sometimes the agent can even change the test code or cheat the scoring system. The article lists seven common mistakes, such as not keeping agents and checkers separate or using unsafe code like eval().

The authors suggest a checklist to fix these problems, like always keeping the checker and agent separate, never using unsafe code on agent output, keeping answers secret, and testing benchmarks with “exploit agents” before release. They also introduce BenchJack, a tool to help benchmark creators find and fix these issues automatically.

Hacker News readers were surprised and worried about how easy it is to cheat today’s AI benchmarks. Some say this shows AI testing needs better security, like software does. Others think it’s a natural step as AI gets smarter—if you give an agent a goal, it will find the easiest way, even if that means cheating. Keeping answers truly secret is very hard, especially for open-source benchmarks, so some suggest using private test sets or rotating questions. Many agree adversarial testing—trying to break your own benchmark—should be standard. Some are excited to try BenchJack, and others hope this work will push the field to make more honest and useful benchmarks.

Next up is new memory technology using fluorographane. This material allows massive data storage—447 terabytes in just one square centimeter—and needs no power to keep data safe. Each bit is a single fluorine atom pointing one of two ways, making it stable and unlikely to flip by accident. This means data is safe even without electricity. A sheet the size of a postage stamp could hold more than all current memory types, and stacking them could reach nine zettabytes per cubic centimeter—more than all the world’s storage today.

The article describes how to read and write: first with a single probe, then with arrays of sensors, and later with a fast, full-parallel system. Even the first step can store more than anything we have now.

We also have news about Advanced Mac Substitute, a project that lets people run old 1980s Mac apps without the original Apple ROM. Instead of copying all the old Mac hardware, it recreates the software API so apps can run directly. Only the 68k CPU is fully emulated—the rest is new code acting like the old Mac OS. This means apps like MacPaint and Lode Runner can run right away, without a long startup. The app works on macOS, Linux, and over VNC, with support for classic Mac graphics and interface features. It’s open source and easy to try.

Now, a new version control system called Pijul. Like Git, it tracks changes in code, but it uses a “theory of patches.” Each change is a “patch” that can be applied in any order if independent, and the result is always the same. This is called “commutation.” Pijul uses “channels” instead of branches, but channels are less central than in Git. Merging is more correct, keeping the order of lines and avoiding the line shuffling that can happen in Git. If two people edit the same part at once, Pijul shows a conflict, but conflicts are normal and fixed with a new patch—once fixed, the conflict never returns. Pijul also allows partial clones, so you can get just the changes you want.

In the comments, some people are excited by Pijul’s new way of handling patches and merging and hope it fixes problems they have with Git. Others like the idea of partial clones for big projects. Some are curious how easy it is to switch from Git, while others worry about missing features or support. There are questions about performance and if Pijul can be as fast as Git. Some say using your own tool to build itself is a good sign, and others hope Pijul gets more popular if it solves real problems for developers.

Finally, Surelock is a new Rust library to prevent deadlocks when using mutexes, by checking for problems at compile time. Deadlocks happen when four things—the Coffman Conditions—occur at once. Surelock breaks one of these: circular wait. For locks at the same “level,” Surelock groups them and always locks in the same order. For locks at different levels, Surelock uses type rules so the compiler stops you if you lock things in the wrong order. Surelock uses a MutexKey system, a kind of token that tracks what locks you have, so you cannot make mistakes. The API is designed so if your code compiles, it cannot deadlock, except for one rare edge case.

Surelock is compared to happylock and lock_tree, two other Rust libraries. Happylock is good but makes you grab all locks at once, which is not always possible. Lock_tree lets you set up ordering between locks, but Surelock goes further with some improvements. Surelock works with Rust’s no_std, so it is useful for small or embedded devices too. There is an “escape hatch” if you need to break the rules, but it is obvious in your code and project settings.

On Hacker News, some are excited by a compile-time solution to deadlocks, saying it could help avoid costly bugs. Others say type systems can make code more complex, and wonder if Surelock is easy to use for big projects. Some like the token idea, while others worry about extra steps slowing down development or confusing new Rust users. Embedded developers are interested, but want more real-world examples. Some debate if it is better to avoid mutexes with other designs, but others say mutexes are often best and tools like Surelock make them safer. People mention similar ideas in other languages. Most agree Surelock is a promising step for safe concurrency in Rust.
