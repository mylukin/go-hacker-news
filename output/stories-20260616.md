## TIL: You can make HTTP requests without curl using Bash /dev/TCP

- 原文链接: [TIL: You can make HTTP requests without curl using Bash /dev/TCP](https://mareksuppa.com/til/bash-dev-tcp-http-without-curl/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48558018)

The article explains how you can make HTTP requests from Bash using `/dev/tcp` when tools like `curl` or `wget` are missing. This is useful in Docker containers that only have a basic shell.

The main idea is that Bash can open a TCP socket by redirecting to `/dev/tcp/host/port`. You can then write an HTTP request directly to this socket and read the response. For example, you use `exec 3<>/dev/tcp/service/8642` to open the connection, then `printf` to send the HTTP request, and finally `cat <&3` to read the response. You can add headers by including them in your `printf` line. It’s important to use the `Connection: close` header; otherwise, the server may keep the connection open and your command will hang. This method does not support HTTPS, so it only works with plain HTTP. `/dev/tcp` is not a real file, but a Bash feature that handles DNS lookup and socket opening. Not all shells support this—`bash` does if built with the right options, but `sh` and `zsh` usually do not. Also, this method is quite basic: it doesn’t handle redirects, compressed responses, retries, or security features like TLS. For more complex needs, you should still use `curl`.

In the comments, many people are surprised to learn about `/dev/tcp` in Bash. Some share that they have used this trick for quick debugging or in very small Docker images. Others warn that this isn't suitable for production scripts, since it can't handle HTTPS or complicated HTTP features. A few users point out that this Bash feature may not always be available, especially on older or custom systems. Some suggest using `nc` (netcat) or `telnet` for similar tasks, if available. Others mention that this method is great for simple health checks or connectivity tests without installing extra tools. There are also comments discussing how this trick can help understand the basics of networking and HTTP. Some users note that this method can be risky if you need to send sensitive information, since it isn’t secure. A few people express concern about relying on non-standard Bash features in scripts that might be run in different environments. Overall, most agree this is a handy trick for certain cases, but not a replacement for real HTTP clients.

## Mechanical Watch (2022)

- 原文链接: [Mechanical Watch (2022)](https://ciechanow.ski/mechanical-watch/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48553550)

This article explains how a mechanical watch works, showing each part and how they fit together. Mechanical watches work without batteries; instead, they use springs, gears, and clever engineering to keep time.

The core of a mechanical watch is the “movement,” which is the inside mechanism. Power comes from a special spring called the mainspring. When you wind the watch, you make the mainspring tight. As it slowly unwinds, it releases energy. This energy turns a set of gears called the gear train. These gears help change the fast energy from the spring into the slow, steady movement needed for the watch’s hands. There’s a special part called the escapement, made of an escape wheel and a pallet fork. The escapement lets the gears move forward step by step instead of spinning freely. Another important part is the balance wheel, which swings back and forth, making sure the watch ticks at a regular rate—this is what keeps time accurate. Jewels, such as synthetic rubies, are used in some places to reduce friction and wear. The watch has a mainplate to hold all the pieces, and shock protectors help prevent damage if the watch is dropped. To wind the watch, you turn the crown, which uses a “click” mechanism to keep the spring from unwinding by itself. Some watches can also wind automatically from the movement of your wrist, using a small spinning weight. There are extra gears to make the minute and hour hands turn at the right speed, and some watches have features like a date display. Setting the time or date is done by pulling and turning the crown, which moves different gears inside. All these parts are tiny and finely made, fitting into a space the size of a credit card.

Many appreciated the simple, clear explanations and the color-coding of parts. Some watch lovers praised the detail, especially about the gear shapes and the escapement. A few shared stories about fixing or building watches themselves, saying the article matched their experience. Others talked about how amazing it is that such tiny parts can work together so well. Some readers pointed out how mechanical watches are not as accurate as digital ones, but they are more interesting as machines. There were questions about why watchmakers use certain shapes for gears, and others explained that cycloidal gears are better for small, low-power devices. A few commenters mentioned the challenges of repairing old watches, especially finding spare parts. Some people said they now understood why mechanical watches are expensive—they are hard to build and fix. Others shared links to videos and books for learning more. A few said this article made them want to buy or fix a mechanical watch. There were also some who just liked seeing how things work, even if they never plan to own a watch. Finally, some pointed out that, even if mechanical watches are less practical today, they are a great example of human creativity and engineering.

This article is about Bill Watterson, the creator of Calvin and Hobbes, and how he fought to keep his comic true to his own values. It tells the story of how Watterson cared more about art and honesty than making money or becoming famous.

Watterson started drawing as a student and liked to work for his own satisfaction, not for rewards or attention. When Calvin and Hobbes became very popular, he could have made a lot of money by selling toys, t-shirts, and other products with his characters on them. But Watterson refused, because he thought selling these things would ruin the magic and meaning of his work. He believed comics should be made with care, not just for jokes or money. He also wanted control over every part of his strip—writing, drawing, and coloring everything himself. This made his work slow and hard, but he felt it was important.

Watterson often had to fight with his syndicate (the company that sold his comics to newspapers). They wanted to make more money by selling Calvin and Hobbes products. Watterson said no, even when it meant turning down millions of dollars. He also asked for more creative freedom, like making his Sunday comics bigger and taking breaks from deadlines to avoid burnout. These were unusual requests, but he stood firm. Sometimes, he got tired and needed time off, but he always returned to his strip with the same strong ideas.

In the end, Watterson chose to stop Calvin and Hobbes while it was still loved. He wanted to leave his work pure and special, not stretched out or cheapened by products. After he finished the strip, he stayed out of the public eye and didn’t talk much about Calvin and Hobbes again.

## Stop Using JWTs

- 原文链接: [Stop Using JWTs](https://gist.github.com/samsch/0d1f3d3b4745d778f78b230cf6061452)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48558147)

The article says you should stop using JWTs (JSON Web Tokens) for keeping users logged in. The main idea is that JWTs were not made for session management, and using them for this is not safe or smart.

The writer explains that JWTs are meant for very short-lived tokens, maybe five minutes or less. But when a user logs in, their session usually needs to last much longer. Some people like JWTs because they seem "stateless," meaning you don’t need to store data on the server. But the author says this is not really true—secure systems always need some state, like a database. If you already have a database for your users, you can just store session info there.

The article points out that using JWTs for sessions brings no real benefit. Regular session cookies do everything better—they are more flexible and easy to use. Also, security experts do not trust the JWT standard. In the past, the JWT spec let attackers make fake tokens, and there could be more hidden problems.

The article suggests using PASETO as a safer choice if you need short-lived tokens, but not for sessions. It also says not to store JWTs or session tokens in localStorage, because it is not safe.

For developers worried about setting up sessions, the writer says that almost all web frameworks make it easy. For Node.js, you can use express-session and a database connector. For other languages, sessions are often built-in. On mobile, cookies are just headers, so you can use the same session system as on web.

In the comments, many people thank the author and agree that sessions with secure, HTTP-only cookies are best. Some suggest using Redis or another cache for sessions in big apps, but the author replies that for most apps, this is not needed. Some readers are confused about how to set up sessions for mobile apps, but the author explains you can use the same tools as for web.

One commenter says JWTs make everything manual and hard, like managing access and refresh tokens, while sessions are simple. Another says JWTs are useful for single sign-on (SSO) across services, with short-lived tokens, not for sessions. Others say JWTs can help in microservices, where many internal servers need to trust each other fast, but regular apps do not need this.

Some people argue that JWTs are fine if you use them correctly, for example, just as an ID to look up real data in a cache. But the author insists there is no right way to use JWTs for sessions, and that cookies and sessions are tried and true. A few mention that security is about trade-offs, and sometimes scaling sessions is hard, but most agree that JWTs are not the answer for standard session use. Overall, the main advice is: use cookies and server sessions for login, and only use JWTs for special short-lived tasks, not for keeping users logged in.
