## Autoresearch on an old research idea

- 原文链接: [Autoresearch on an old research idea](https://ykumar.me/blog/eclip-autoresearch/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47493460)

This article is about using an AI agent (Claude Code) to improve an old machine learning project with a method called “autoresearch.” The author wanted to see if an AI could help update and optimize their research code while they did chores.

The main idea is to let an AI agent repeatedly try small changes to the code, test them quickly, and keep only what works. The system is set up with files for instructions, working memory, and the main training code. The agent follows a loop: make a guess, change the code, train, check results, accept or reject the change, and repeat. The experiments are kept short (about 5 minutes each) to avoid overfitting and speed up progress. For safety, the agent runs in a locked-down container without internet or extra permissions.

Since the original medical X-ray data was not available, the author used a dataset of Japanese woodblock prints with labeled regions. The agent updated the code to handle the new data and wrote code to set up the experiments. The main performance measure was “mean rank” of image-text matches, with other checks for accuracy.

The AI ran 42 experiments in one day, keeping 13 changes and reverting 29. The model’s mean rank improved a lot—dropping from 344.68 to 157.43, a 54% reduction. After a final full training, test results were even better, showing that the short runs were underfitting.

Some of the best gains came from the agent fixing a bug in the code (relaxing a temperature limit), and from tuning settings like learning rate and model size. When the agent tried big, creative changes, success dropped, and most ideas didn’t help. The sandbox setup was important—sometimes the agent tried to do things it wasn’t allowed, which stopped the process. The author noticed that the first part of the work went smoothly with the AI, but the final tweaks were much harder and slower.

Now, let’s look at what people said in the comments. Some loved the hands-off approach, saying it’s amazing that you can let an AI experiment on your project while you do other things. Others pointed out the limits: the AI did well with simple, clear tasks like tuning numbers or bug fixing, but struggled when the problem needed real creativity or deeper understanding. A few thought the strict safety rules were smart, since letting AI change code freely could be risky. Some readers wondered if allowing the agent to plan bigger changes or use sub-agents could help with the more complex phases. There was also talk about how useful this could be for boring, repetitive coding tasks—maybe freeing up humans for more interesting work. A handful worried about overfitting or the agent making meaningless changes just to pass an eval metric. Finally, some were inspired to try autoresearch on their own old projects, seeing it as a new way to get value from AI in everyday coding.

The writer explains that every time they fix one slow part of their workflow, they find the next slow part. This is called the “theory of constraints.” By removing each friction point, the process keeps getting faster and smoother.

## Finding all regex matches has always been O(n²)

- 原文链接: [Finding all regex matches has always been O(n²)](https://iev.ee/blog/the-quadratic-problem-nobody-fixed/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47443903)

This article explains a hidden problem with regular expressions (regex): finding all matches in a string is much slower than most people think, even in the fastest regex engines. When you search for all matches, not just the first, the process often takes time that grows with the square of the text length (O(n²)), not linearly. This happens in almost every popular regex library, no matter how fast or safe it claims to be.

The author gives an example: if you use a pattern like `.*a|b` on a string with only the letter "b", the engine will look for "a" at every position, fail to find it, and then match "b" one by one. This means the engine does more and more work for longer texts—much more than you expect. Even new engines like Rust’s `regex`, Google’s RE2, or .NET’s NonBacktracking mode run into this problem when you ask for all matches, not just one.

Academic papers and most documentation focus on the single-match problem. They often say, "just iterate to find more matches," but that hides the real cost. The real world often needs all matches, for things like search and replace or data extraction. The issue has existed since the 1970s, but was rarely discussed because theory papers didn’t care about finding all matches.

Backtracking regex engines are even worse—they can be so slow that they are open to denial-of-service attacks (ReDoS). Libraries like minimatch on npm have had many security issues because of this. Some engines, like Aho-Corasick, solve the problem for simple string matches (not full regex) by using a clever algorithm that always runs in linear time.

Intel’s Hyperscan engine can find all regex matches in linear time, but only if you accept a different answer: it reports the "earliest" match, not the longest. For many uses, like search and replace, this is not what people want.

The author describes their own engine, RE#, which can find all matches in two passes over the input. It uses a reverse scan to find where matches could start, and a forward scan to confirm the longest match at each spot. This keeps the time close to linear, even for tricky patterns. If you want to guarantee linear time with no exceptions—important if users can supply the pattern—RE# has a "hardened" mode. This mode is a bit slower on normal patterns but much faster on bad cases.

Benchmarks show RE# is very fast, even compared to Aho-Corasick and Rust’s regex. However, RE# currently does not support capture groups or lazy quantifiers (like `.*?`). The author also built a new grep tool using RE# that supports complex boolean searches.

In the Hacker News comments, many users are surprised to learn about this O(n²) problem. Some say they have hit slow regex searches in practice but didn’t know why. Others discuss the trade-offs: engines like Hyperscan are fast but give different results, while engines that guarantee "greedy" or "longest" matches are slower for all matches. Some readers wish more regex libraries would warn users about this hidden cost. A few comment that most real-world patterns are safe, but others point out that when you do hit this issue, it can be a big problem, especially with user-supplied patterns.

Some developers share tips, like using Aho-Corasick for simple patterns, or splitting the work between different tools. A few are impressed by the RE# approach, but wonder if the extra work or lack of capture groups will limit adoption. Others ask about streaming support, and agree it’s hard to do with leftmost-longest matching. There is debate about whether library defaults should prioritize safety or speed, and several people agree that better documentation and warnings could help everyone avoid surprises. Overall, the comments show a mix of surprise, practical advice, and hopeful interest in new solutions.

## Trivy under attack again: Widespread GitHub Actions tag compromise secrets

- 原文链接: [Trivy under attack again: Widespread GitHub Actions tag compromise secrets](https://socket.dev/blog/trivy-under-attack-again-github-actions-compromise)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47475888)

Trivy, a popular security tool, suffered a major attack where its GitHub Actions tags were changed to include malware. The attack let bad actors put secret-stealing code into 75 out of 76 tags in the official Trivy GitHub Action, which many projects use in their CI/CD workflows.

The attacker got access by stealing credentials from Trivy’s systems. They used these to force-push new tags that looked normal but contained a hidden infostealer in the `entrypoint.sh` script. This malware ran before each normal Trivy scan, so users would not see anything strange in the output. The only safe tag not affected was `0.35.0`.

The attack collected secrets from the CI/CD runner, including SSH keys, cloud credentials (AWS, GCP, Azure), Kubernetes tokens, and more. It worked by reading process memory and searching for files with secrets. The data was encrypted and sent to a fake website with a name close to the real Aqua Security domain. If that failed, the malware used the victim’s own GitHub account to upload the stolen data as a release in a new public repo.

The attacker used smart tricks to hide their actions. They copied old commit info and made the tags look as if nothing had changed. GitHub’s “Immutable” badge was shown, but in reality, the tags had been changed by force-pushing. The attack is linked to a group called TeamPCP, known for targeting cloud systems and crypto wallets.

Trivy’s team said the root problem was leftover access from a previous breach. Even after rotating secrets, the attacker still had a way in due to non-atomic rotation. Projects using Trivy by tag are told to stop and only use the safe commit SHA or tag `0.35.0`. Any pipeline that used a bad tag should consider all its secrets stolen and rotate them right away.

In the Hacker News comments, many users showed shock that such a trusted tool could be hit so hard. Some pointed out this is a big example of why you should always pin to a commit SHA, not a tag, when using GitHub Actions. People talked about how easy it is for attackers to hide changes in tags and how GitHub’s UI does not make it obvious when tags are force-pushed.

Others debated if GitHub should block force-pushing tags or make it easier to see tag changes. Some said the “Immutable” feature is misleading if it does not prevent tag changes. A few users blamed the Trivy maintainers for poor secret handling, while others felt sorry for them, noting that supply chain attacks are getting harder to stop.

Several developers said they would now check their own projects for similar risks and rotate secrets just in case. Some shared scripts to detect if their workflows used a bad Trivy tag. There was also talk about the need for better education on supply chain security and for GitHub to add more warnings for tag changes.

A few users worried that more attacks like this will happen, as more projects trust actions by tag. Some called for more tooling to help people see when tags or dependencies change suddenly. The overall mood was serious, with many realizing how much risk there is in automated build systems and open-source supply chains.

## AI Risks "Hypernormal" Science

- 原文链接: [AI Risks "Hypernormal" Science](https://www.asimov.press/p/ai-science)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47493194)

This article talks about how today’s AI systems help science, but may also hold it back. Right now, AI is very good at using lots of data to make predictions and find patterns, but it mostly works within the rules and ideas we already know.

The article gives the example of Borges’s story, where a map is made so detailed it’s the same size as the land it describes. This is like AI today: it can make a very detailed map of what we already know, but sometimes what we need is not more detail, but a new way to look at things. The London Underground map is another example—when the map was changed from a true-to-life version to a simple diagram, it became much more useful.

In science, big changes (paradigm shifts) often happen when someone finds a new, simple idea that explains everything better, like Einstein’s relativity or Darwin’s theory of evolution. These changes did not come from adding more and more data, but from thinking differently. The article says AI is not designed for this yet. It’s trained to be good at working with current ideas, not at creating new ones.

The article gives more examples: AI can make better predictions in breeding or chemistry, but it does not discover new ideas like genes or DNA. Even when AI finds new things—like new materials or protein shapes—it’s still working inside the boxes humans made. Some newer AIs learn from raw data, but they still don’t change the “map” itself.

AI systems are starting to help scientists do their work faster, but they also make scientists focus more on popular topics instead of looking for new ideas. If AI is used to pick what is “good science,” it will often choose things that fit with what we already believe, making it harder for new ideas to get noticed.

Some people hope that making AI bigger and faster will help, like how AlphaZero learned to play chess in its own way. But science is different—sometimes the best ideas are very simple, and we don’t know how to teach AI to find them. Some suggest training AI to look for simple explanations or good analogies, or to learn from different senses, but this is still very new.

The article ends by saying we don’t really know how to make AI that can create big changes in science. We need to understand what makes paradigm shifts happen, and maybe study history or run experiments with AI groups to see what works.

In the comments, many readers agree that AI is great at speeding up what we already do, but not so good at finding new ways to think. Some say this is because AI is trained on past data, so it just repeats old ideas. Others hope that as AI gets better, it might surprise us and come up with new concepts. Some warn that if we rely too much on AI, we might lose the ability to think outside the box, since everyone will just follow what AI says.

A few commenters point out that even human scientists are slow to accept new ideas, so it’s not just an AI problem. Some think combining human creativity with AI’s power is the best way forward. Others worry that science funding and career rules also push people to do safe, normal work, and AI could make this worse by making it even easier to do more of the same.

One person suggests that future AIs might need to “think like outsiders,” just like Einstein did, and that maybe we should design AI to search for things that don’t fit the usual patterns. Another commenter wonders if AI could someday run its own experiments and develop its own theories, but most agree that for now, AI is mainly a tool for helping with current science, not making the big leaps on its own.

## BIO: The Bao I/O Coprocessor

- 原文链接: [BIO: The Bao I/O Coprocessor](https://www.bunniestudios.com/blog/2026/bio-the-bao-i-o-coprocessor/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47459363)

The article explains BIO, a new I/O co-processor made for the Baochip-1x, which is a mostly open-source chip. The author wanted to create an I/O co-processor like the Raspberry Pi’s PIO but with simpler, more open hardware.

BIO helps the main CPU by handling I/O (input/output) tasks so the CPU can work faster and with less delay. The author first tried copying the PIO design but found it used a lot of hardware resources and was hard to optimize. The PIO has only nine instructions, but each one is very complex, making the design slow and large. The author also warns there might be patent risks if others copy PIO exactly.

To solve these problems, the author built BIO using a RISC-V core, which is simpler and uses less space in hardware. The BIO connects some of its registers to special queues (FIFOs) and GPIO pins, letting its four cores communicate and control I/O efficiently. BIO can pause its cores until certain conditions are met, like waiting for data or a clock signal, which helps with timing.

The article gives code examples in assembly and C, showing how BIO can move data quickly, handle SPI (a common data protocol), and even control LED lights with exact timing. Each BIO core gets 4 KB of memory, which is much more than PIO, allowing for larger and more complex programs. The author also explains how to use standard tools (like Zig and Rust) to build BIO programs, making development easier and more open.

Comparing BIO and PIO, the author says PIO is faster for simple tasks, but BIO is smaller, runs at higher clock speeds, and is easier to program for more complex jobs. BIO is open-source and avoids patent issues. The article ends with links to code, hardware designs, and a development board.

In the comments, some people talk about tooling. One person suggests using Zig directly instead of C for better integration. Another commenter asks about timing details in the SPI and LED code; the author replies that timing can be made exact using special BIO features, but it’s tricky if different tasks run together. There’s also a discussion about how BIO allows you to do more complex things, like making random bit streams or advanced data transfers, which PIO struggles with. Someone else points out that BIO makes it easy to rearrange data from sensors, which is hard with PIO. Finally, one commenter says BIO could benefit from better error handling if timing is missed, but overall, people seem excited about the flexibility and openness of BIO.
