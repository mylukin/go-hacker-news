## The peril of laziness lost

- 原文链接: [The peril of laziness lost](https://bcantrill.dtrace.org/2026/04/12/the-peril-of-laziness-lost/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47743628)

This article talks about how “laziness” was once seen as a good thing in programming, because it pushed people to create better, simpler tools and code. The author explains how real programmer laziness means working hard now to save time later, by building good abstractions and simple systems.

The article says that in the past, programmers tried to make things easy for themselves and others by making reusable code and clear designs. This “laziness” actually took work—it meant thinking carefully and planning so that future work would be easier. But now, more people write software, and many may not even think about these old programmer values.

The author points out that new tools like large language models (LLMs), such as ChatGPT, let people create code much faster. Some people, like Garry Tan, brag about making thousands of lines of code every day with LLMs. But the article warns that just making more code is not always good. Sometimes, LLMs create messy code with useless files and repeated parts, because they don’t care about saving time or making things simple.

He gives an example where someone used an LLM to make a new software project, but the code was full of mistakes and pointless parts. The real problem, the author says, is that LLMs are not “lazy” in the good way—they do not try to make things easier for people in the future. They just produce more and more, without thinking about quality or simplicity.

The article says that best engineering comes from real human limits, like not wanting to waste time. Humans work to make things simple because our time is valuable. LLMs do not have these limits, so they might make software more complex instead of better.

Still, the author thinks LLMs can be helpful, but only if we use them as tools. We should use LLMs to help with boring or hard parts, but we must still use our own good judgment to keep things simple and well-designed.

Many comments agree that the old idea of “laziness” in programming is important. Some say that LLMs make it too easy to skip thinking and just produce code, which can make software worse in the long run. Others point out that it is easy to measure how much code you write, but much harder to measure if the code is actually good or useful.

Some people think that LLMs could be helpful if used carefully, and that the real problem is how people use the tool—not the tool itself. A few comments remember times when “hustle culture” in tech was praised, but they agree that making more code is not always better.

A few developers are hopeful that LLMs might get better at making simple code in the future, but most agree that humans must still guide the work. Some worry that newer programmers may not learn good habits if they rely too much on LLMs. Others say that it is up to teams and leaders to keep quality high, no matter what tools are used.

In the end, most people agree: good software should be simple and easy to change, and human “laziness”—wanting to save time and effort—is a virtue we should not lose.

## Bring Back Idiomatic Design

- 原文链接: [Bring Back Idiomatic Design](https://essays.johnloeber.com/p/4-bring-back-idiomatic-design)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47738827)

This article talks about how, in the past, computer programs all used the same design patterns, making software easy to use and understand. The author misses this “idiomatic design,” where things like buttons, menus, and keyboard shortcuts worked the same way everywhere.

The article explains that in the old days, with desktop software like Windows 95 to Windows 7, interfaces were very consistent. For example, a checkbox always looked and worked the same, and menu items like “File” or “Edit” were always in the same place in every program. Keyboard shortcuts, like Command + C for copy, worked everywhere. This made it easy for users, because once you learned something, you could use that knowledge in every new program.

Today, most software lives in the browser, and it’s not like this anymore. Every web app has its own way of doing things. For example, picking dates, entering credit card info, or using keyboard shortcuts all work differently from site to site. Even apps made by the same company (like Google) can feel very different from each other. The author says this is confusing and makes work slower, because users spend time looking for things instead of just using the software.

The reason for this change is that web apps have to work on many devices, like phones and computers, and design patterns for mouse, keyboard, and touchscreens are not the same. Also, modern web development uses many frameworks and reusable components, but instead of creating clear standards, this has led to hundreds of different styles. Developers often focus on what’s possible, not on making things consistent and easy to use.

However, some companies, like Apple, do stick to strong design rules. Their apps and even many third-party apps on iOS look and feel similar, so users know what to expect. The author gives some advice: follow standard HTML and browser behaviors, make links and buttons clear, use words instead of unclear icons, and copy good designs from the past.

Now, looking at top comments from Hacker News, some people agree and also miss the old days of simple, consistent design. They say modern web apps often hide basic features, or make things hard to find. Others point out that the web is a bigger, more complex place than desktop software, so it’s harder to make rules that everyone follows. Some developers say they try to follow good patterns, but their users or managers push for “fancy” or different designs.

A few commenters mention that accessibility is also a problem—when developers don’t use standard HTML elements, screen readers can break, making sites hard to use for people with disabilities. Some believe the focus on mobile-first design has made desktop experiences worse. There are also comments about how business needs, branding, and the desire to be “unique” make companies ignore design standards. On the other hand, some say that variety can be good, allowing for innovation and new ideas.

One person notes that strong design systems, like Google’s Material Design, try to help, but aren’t always followed. Others share that they like tools or apps that still use classic design idioms, finding them faster and easier. In the end, many wish for more consistency, but accept that the web may never be as simple as the old desktop days.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47702887)

This article is about learning how to juggle, starting from the very basics and going up to advanced juggling ideas. The writer is an experienced juggler who often teaches people in public places and wants to help beginners get started.

First, the article explains that most people struggle even with just one ball. The guide starts with standing in a relaxed position, then practicing throwing one ball from hand to hand. The main point is to throw in a smooth arc and avoid reaching up or following the ball with your eyes too much. When moving to two balls, the article warns not to pass balls straight across, but to throw in arcs so catches and throws are easier later. Mistakes like throwing too early or at different heights are common, but practice helps fix them.

For three balls, the pattern is similar: start with two balls in one hand and one in the other, throwing from the hand with more balls. The key tip is to throw slowly and avoid tossing balls forward. Simple tricks and patterns come next, like outside throws or under-the-arm catches. The article then explains how to juggle more balls, with four balls taking a month to learn and five taking years. Fountain and cascade patterns are introduced for juggling even and odd numbers of balls.

A section on siteswap notation shows how juggling can be described with numbers and math, making patterns easier to talk about and learn. The writer also mentions passing balls between jugglers, juggling with clubs or rings, and other forms like bounce or contact juggling. They end by saying juggling is creative, and people often invent new forms.

In the comment section, many readers appreciate the clear step-by-step teaching, saying they want to try juggling again. Some share their own stories about learning to juggle, mentioning how hard it is at first but how practice helps. A few people say they never realized how much technique was involved just to control one ball well. Others discuss the siteswap notation and how it brings math and logic into juggling, which they find surprising and interesting.

Some commenters point out that learning one ball is a good way to train hand-eye coordination, even for non-jugglers. A few debate whether juggling is mostly about raw skill or about understanding the patterns and movements. One person says that watching jugglers in public inspired them to try, while another says they felt embarrassed to learn in front of others. Some think tricks and fancy patterns look more impressive than just juggling many balls, while others admire the patience needed to learn high numbers like five or seven balls.

A few readers note that juggling helped them with focus and relaxation. Others talk about how juggling has a friendly and sharing culture, with people quick to teach each other. There’s also a discussion about how dropping balls is normal, and that failing is part of learning. Some share tips about practicing over a bed or in front of a wall to make catching dropped balls easier. Finally, several commenters agree that anyone can get better at juggling with time, and that starting simple is the best way.

## Show HN: Claudraband – Claude Code for the Power User

- 原文链接: [Show HN: Claudraband – Claude Code for the Power User](https://github.com/halfwhey/claudraband)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47741889)

Claudraband is a tool that lets power users control Claude Code from the command line or programmatically. It helps you keep your Claude Code sessions alive, resume them later, and even control them remotely or through other tools.

The main features include resumable, non-interactive workflows—you can start a session, come back to it, and ask follow-up questions. There’s a local mode that uses tmux for session management, and a daemon mode that lets you control sessions over the network. Claudraband comes with a TypeScript library, so you can add these features to your own apps. It also supports ACP, which means other editors or frontends can use it to talk to Claude Code. You can list, close, or continue sessions, and even answer pending prompts. The tool doesn’t replace the Claude SDK; rather, it works on top of the Claude Code TUI, so you need to log in the normal way. It doesn’t handle OAuth—it expects you to be already authenticated with Claude Code. You can use it with Node.js or Bun, and there are command-line examples for different workflows. There are also experimental features, like using xterm.js instead of tmux, but this is slower and not recommended for long sessions. The documentation provides example scripts for code review, multi-session work, and journaling sessions.

In the comments, some users are excited about being able to automate Claude Code, especially for code review and editor integration. A few people say the tmux approach is clever, as it keeps sessions stable and easy to manage. Others point out that the tool is experimental, so there may be bugs as Claude Code changes. Some wish it worked more like a true API, not needing the TUI, but understand why it doesn’t. There’s interest in the ACP support, since that could let different editors or tools work with Claude Code. A few users mention security concerns, since the tool depends on your existing authentication and doesn’t handle OAuth. Some like the clear session control, while others hope for more features, like better error messages or headless operation. There are questions about performance, especially with the experimental xterm.js backend. Finally, some users suggest improvements to make it easier for teams, not just individuals.

## Show HN: Oberon System 3 runs natively on Raspberry Pi 3 (with ready SD card)

- 原文链接: [Show HN: Oberon System 3 runs natively on Raspberry Pi 3 (with ready SD card)](https://github.com/rochus-keller/OberonSystem3Native/releases)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47739174)

This project is about running Oberon System 3 directly on a Raspberry Pi 3b, with a ready-made SD card image to make setup easy. Oberon System 3 is a simple, fast, and old-school operating system from the 1990s, designed by Niklaus Wirth, and it’s now been ported to modern hardware like Raspberry Pi.

The release includes an image file you can flash onto an SD card using common tools like dd, Raspberry Pi Imager, or Etcher. Once flashed, you can boot the Pi 3b and use Oberon System 3 with a regular screen and a keyboard. The author tested the system with a Lenovo ThinkPad TrackPoint keyboard, which works well because Oberon uses three mouse buttons.

The project also supports Raspberry Pi 2b (version 1.2 or newer) and Pi Zero 2, because they share similar hardware and are still easy to find. The Pi 3b will be produced until at least 2028, and the Pi Zero 2 until at least 2030, so the system should work for many years. The developer says the system may also be moved to the Raspberry Pi 4 in the future.

You can also build the system yourself if you want. The author provides the needed boot files, a pre-compiled toolchain for Linux, and build scripts. Earlier versions of the system worked on QEMU emulation for ARM and on i386 PCs, and builds are very fast—less than a minute on a modern laptop.

The release page offers all files: the SD card image, boot files, toolchain, and source code. The project is open source and has a small but excited group of users.

In the Hacker News comments, some people are very happy to see Oberon running on modern, cheap hardware. They like the simple and clear design of Oberon. Others remember learning from Wirth’s books and how Oberon helped them understand computers better. A few users talk about the three-button mouse, saying it’s rare today, but the ThinkPad keyboard is a good option.

Some comments compare Oberon to other old systems, like Plan 9, and wish more operating systems were this simple. There is praise for the fast build times and the detailed documentation. A few people ask about network support or wish for versions on newer Raspberry Pi models. Others say they want to use Oberon as a teaching tool for new programmers. Some users are just happy to see new life for old ideas, while others ask technical questions about hardware support and performance. Overall, the mood is positive, with people sharing tips, history, and a bit of nostalgia.

## Show HN: boringBar – a taskbar-style dock replacement for macOS

- 原文链接: [Show HN: boringBar – a taskbar-style dock replacement for macOS](https://boringbar.app/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47742200)

boringBar is a new tool for macOS that replaces the Dock with a taskbar, similar to what you see on Windows or Linux. It helps people keep track of their open windows and desktops, especially when using many displays or virtual desktops.

The main idea is that the standard macOS Dock is focused on apps, not windows, making it hard to see what is open and where. boringBar fixes this by showing only the windows on your current desktop. You can quickly switch between desktops, see previews of windows, and pin your favorite apps for easy access. It also has a searchable app launcher, so you can find and start any app quickly. Notification badges and attention pulses help you notice updates or apps needing your attention. The bar’s size is adjustable, and you can group windows by app or see full window titles if you want. If you like a tidy screen, boringBar can hide the Dock while it runs and works well with multiple monitors. It requires some special permissions (Accessibility and Screen Recording), but the developer says it only uses these for needed features, like window previews.

boringBar is free to try for 14 days. After that, a personal license is a one-time $40 payment for two devices, with updates for two years. Business licenses are annual and get cheaper per user as your team grows.

In the comments, many users say they miss having a taskbar like Windows on macOS, and think boringBar fills this gap. Some are excited because they use many desktops and windows, and the standard Dock does not help them stay organized. A few people mention that hiding the Dock is a great feature, making their workspace cleaner. Others are happy that the tool works well with multiple monitors. Some users ask about privacy, especially around the Screen Recording permission, but the developer explains it is only for window previews. There are questions about macOS version support—right now, it needs macOS 14 (Sonoma) or newer, which some find limiting. A few people are unsure about the price, saying $40 is a bit high for a Dock replacement, while others say it is fair for a helpful tool. Some users suggest extra features, like more customizations or a Windows-like Start menu. Overall, people think boringBar is a good idea, especially for those who want a more Windows-style workflow on Mac.
