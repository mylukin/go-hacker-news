This app lives in the menu bar, does not need you to change security settings, and works by “faking” a super-fast swipe gesture. It also lets you jump to any space by number and can be used from the command line. The install steps are simple: clone the repo, run the build script, and you’re set. The writer says it only has one star on GitHub (their own) and wants more people to try it.

## Reverse engineering Gemini's SynthID detection

- 原文链接: [Reverse engineering Gemini's SynthID detection](https://github.com/aloshdenny/reverse-SynthID)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47709130)

This project is about reverse-engineering SynthID, which is Google Gemini’s invisible watermark for AI-generated images. The author explains how they figured out where and how this watermark is hidden, and they built tools to detect and even remove it.

First, they used signal processing and spectral analysis to find out how the watermark works. They learned that the watermark uses special frequencies, and these change based on the size (resolution) of the image. They collected many pure black and white images to build a “codebook” of watermark fingerprints for each image size. With this, they made a detector that finds SynthID watermarks with about 90% accuracy. They also created a removal tool, called a “bypass,” which can take out the watermark without hurting the image quality too much. The best bypass (V3) works across many image sizes, drops most of the watermark’s signal, and keeps the image looking almost the same. The author describes how the watermark is strongest in the green color channel, and that phase (timing) of the watermark is always the same for each model. They also show how standard tricks like JPEG compression or adding noise don’t remove the watermark well. Their method, which uses the codebook and works in the frequency domain, is much more precise and effective. The author hopes more people will join in by adding new image samples to make the codebook better.

In the comments, some people are impressed by the technical deep dive and call it a clever and important project. Others debate ethics—some say reverse engineering is good for research, but others worry people will misuse these tools to hide AI content. One user points out that watermarks like SynthID will always have weaknesses that people can exploit. Another suggests Google should expect researchers to try to break these systems and use this feedback to improve SynthID. Some developers talk about the limits of watermarking: if you want a watermark to survive even after editing, it’s harder to make it invisible; if it’s easy to remove, it’s less useful. A few comments mention this could lead to a “cat and mouse” game, where new watermarks and new bypasses keep coming out. There’s also praise for the open-source approach, but some note the risk of helping bad actors. Finally, several people stress that this work is a good reminder: you can’t fully trust digital watermarks to prove where an image comes from.

## PicoZ80 – Drop-In Z80 Replacement

- 原文链接: [PicoZ80 – Drop-In Z80 Replacement](https://eaw.app/picoz80/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47708041)

The article is about the picoZ80, a new board that replaces the old Z80 CPU in retro computers with modern hardware. This device fits into the same socket as the original Z80 and gives old computers more speed, memory, and new features like WiFi and virtual disks.

The picoZ80 uses a Raspberry Pi RP2350 chip, which is much faster than the Z80. It runs two cores: one core emulates the Z80 and handles memory, while the other manages things like USB, file I/O, and talks to an ESP32 chip. The ESP32 chip adds WiFi, Bluetooth, and a web interface for control. All the settings are stored in a simple config file on an SD card, so users can change how the board works without needing to rebuild the software.

This board gives the host computer exactly the same timing as a real Z80, so old programs work as expected. It has a lot of RAM and Flash storage: 8MB of RAM and 16MB of Flash. Memory is organized in banks, so you can have many ROMs and run different “personas” for different computers, like the Sharp MZ-700 or Amstrad PCW. The board can also emulate floppy disks and QuickDisks, letting you load disk images from the SD card.

The web interface is easy to use. You can update firmware, manage files, change settings, and switch between different machine types from your browser. There’s also support for over-the-air updates, so you don’t need hardware tools to upgrade the board. Debugging is possible through USB and special debug headers. The hardware is open source for personal use, but not for commercial sale unless you get permission.

In the comments, some users are excited because this brings new life to old Z80 computers, making them more usable and fun. Many like the web interface and easy configuration. People also like that the timing is “cycle-accurate,” so software works just like on the original chip. Some say this is better than simple emulators because it works in the real computer, not just on a PC.

A few users worry about hardware compatibility. They ask if the board really works with all Z80 machines, since some systems use special timing or extra features. Others wonder about the long-term future of these kinds of projects and if the chips used will stay available.

Some people ask about using the board for other CPUs, like the 6502, and if similar upgrades are possible. There are questions about licensing, since commercial use is not allowed without permission. A few users discuss the technical details, like how the PIO state machines work and how debugging is done.

Overall, the community thinks the picoZ80 is a very clever project that mixes new and old technology. Many hope to see more boards like this for other retro computers. Some users say they plan to build one for their own machines, while others are happy to watch the progress and see old computers get modern features.

## Hegel, a universal property-based testing protocol and family of PBT libraries

- 原文链接: [Hegel, a universal property-based testing protocol and family of PBT libraries](https://hegel.dev)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47707762)

Hegel is a tool and protocol for property-based testing (PBT), which helps developers find bugs in their code by checking if certain rules always hold true. It builds on top of Hypothesis, a popular PBT library, and offers libraries for different programming languages like Rust and Go.

The main idea of property-based testing is to describe how your code should behave, then let the tool generate many random test cases to try to break your code. If it finds a problem, it gives you an example that fails, so you can fix your code. Hegel provides a standard way (a protocol) to do this, so you can use the same tests across different languages. This means you can write a test once and run it on both your Rust and Go projects, for example. Hegel also comes with guides, references, and explanations to help new users get started. It aims to make testing more powerful and flexible by letting more people share and reuse tests. Hegel is open source and has its code on GitHub, so anyone can look at it or help improve it. There are links to guides for getting started, understanding how Hegel works, and seeing which languages are supported.

In the comment section, some people are excited about having a universal protocol for property-based testing, saying it could make tests more reusable and powerful, especially for teams working in more than one language. Others ask if it is easy to use and how much setup is needed. Some users discuss the strengths of Hypothesis and hope Hegel will keep things simple and not add too much extra work. A few commenters worry about compatibility with older test suites or languages not yet supported. There is also talk about the value of property-based testing itself, with some users sharing that it helped them find serious bugs, while others feel it can be hard to figure out which properties to test. One commenter points out that having good documentation and guides is important for adoption. Another says open sourcing is a good move, as it lets the community help improve the tool. Overall, the response is mostly positive, but some people wait to see how it works in real projects.

## Research-Driven Agents: What Happens When Your Agent Reads Before It Codes

- 原文链接: [Research-Driven Agents: What Happens When Your Agent Reads Before It Codes](https://blog.skypilot.co/research-driven-agents/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47706141)

This article talks about coding agents—AI tools that can change and improve code by themselves. It explains that these agents work better if they first read research papers and check other projects before trying to make any code changes.

The main idea is: giving agents a “research phase” helps them find better ways to optimize code, especially for tricky problems. The authors tested this idea on llama.cpp, a program for running language models on CPUs. They set up four cloud computers and let the agent read about similar projects and special research papers. Then, in about three hours, the agent made five optimizations that made text generation 15% faster on Intel/AMD and 5% faster on ARM chips.

The article explains that if you only let the agent look at the code, it can miss bigger, smarter changes. For example, code-only agents often try small changes, like making a loop faster, but this does not always help. Sometimes, what makes the code slow is not visible in the code itself but is due to things like how memory is used, or tricks used in other projects.

To solve this, the team had the agent read research papers and look at similar or competing projects. The agent found ideas by studying forks like ik_llama.cpp and by reading papers about operator fusion (combining steps to save memory and time). The best results came from looking at other projects’ code, not just research papers.

Some of the optimizations included combining several steps into one to reduce extra work, and changing how the code handles memory. For example, instead of copying data three times in different loops, they did it all in one loop. Another change was fusing operations like RMS norm and multiplication into a single step, which other versions (like CUDA) already did.

The experiments ran on both Intel and ARM cloud computers, and the improvements were checked carefully. The team notes there was some noise in the results (because cloud computers can be busy with other work), but the speed-up was clear.

Most experiments did not work—only 5 out of more than 30 made things better. Many ideas failed because the computer or compiler was already doing that optimization, or because the change did not fit the way the code or hardware worked.

In the comment section, many readers liked the idea of adding a research phase for coding agents. Some said it makes the agents more like real engineers, who always check what others have done before starting work. Others pointed out that reading papers is good, but looking at real code from other projects is even better. A few readers warned that not all codebases will benefit—older, very optimized projects have fewer easy wins left.

Some people talked about how hard it is to measure small improvements on cloud computers, because results can change if other users are busy. They suggested ways to get better test results, like repeating tests or using private machines.

There were also comments about the limits of AI agents: right now, they still need help understanding complex domain knowledge or special hardware details. Some thought that the biggest wins come from mixing AI with good human review—using agents for ideas, but letting people check and finish the work.

Other readers were excited about using this setup on their own projects, especially machine learning code. A few shared their own stories about failed “optimizations” that didn’t help, showing that even smart tools need good direction.

Overall, readers liked the practical approach of mixing research, real code examples, and testing, and many want to try this method with their own coding agents.

## How Do You Find an Illegal Image Without Looking at It?

- 原文链接: [How Do You Find an Illegal Image Without Looking at It?](https://mahmoud-salem.net/the-invisible-shield)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47668818)

This article talks about how technology helps find illegal images of child abuse (called CSAM) without people actually looking at the pictures. The problem is huge: in just one year, there were over 61 million such files reported, and many are now made by AI.

The main idea is privacy and safety. Most pictures online are innocent, so the system must protect everyone’s privacy while still catching bad images. The key tool is called perceptual hashing. This turns a picture into a short code (a “fingerprint”) that shows what it looks like, but you cannot turn this code back into the original image. When someone uploads a photo, the system makes a hash and checks it against a list of known illegal image hashes. If they match, action is taken—often without any human ever seeing the picture.

Hashing works well for images already found and reported (known CSAM). It is fast and private but cannot catch new or AI-generated images. For new material, machine learning classifiers are used. These try to judge if an unknown image could be illegal, but are less accurate and raise privacy questions because they make guesses, not exact matches.

The article explains how hashing works by shrinking, blurring, and analyzing an image’s structure. Even if you change the image a little—like making it smaller or adding a watermark—the hash stays almost the same. For videos, a similar method is used, but it’s more complex because videos have thousands of frames. Systems like TMK+PDQF create fingerprints for entire videos, making it possible to find copies even if details change.

If an image has never been seen before, only classifiers can try to catch it. But these tools are not perfect—they can make mistakes, sometimes flagging innocent images or missing real abuse. The article also lists open-source and free tools that platforms can use to catch CSAM, making it clear that there is no excuse for not using these protections.

The comment section on Hacker News showed strong interest and concern. Some readers praised the technical details and the privacy-preserving design of perceptual hashing. Many felt it was impressive that systems can work at such a large scale without humans looking at private photos. Others pointed out the challenges with AI-generated images, worrying that these will flood the system and make it harder to find real victims. 

A few commenters stressed that false positives are a big problem—innocent people could get flagged, leading to stress or even legal trouble. Some readers asked about the risk of attackers creating images that purposely match known hashes (“hash collisions”) to trick the system. Others wondered about the ethics of using classifiers that might scan everyone’s photos and make mistakes.

There were concerns about the future: as AI gets better at making fake images, it may be harder for both machines and humans to tell what is real. Some readers shared ideas for improving tools, like better age detection or smarter filters to help human investigators focus on urgent cases. Overall, most agreed that privacy and child safety must both be protected, and that technology is important—but not a complete solution.
