## I Inspected My Take-Home Interview Project. It Was a Whole Operation

- 原文链接: [I Inspected My Take-Home Interview Project. It Was a Whole Operation](https://citizendot.github.io/articles/fake-job-interview-git-hook-malware/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=49013036)

This article is about someone who got a job offer for a remote Python developer role, but the offer was a scam with hidden malware in the interview project. The author shares step-by-step how the scam worked and how he found it.

First, the recruiter contacted the author with a very high salary, which seemed unusual. The author checked the company and found some things odd but decided to continue. After sending his resume, he got a take-home coding assignment in a zip file. The project looked normal at first—a typical FastAPI backend. But because of habit, the author checked all folders, including hidden ones. He found a .git directory with many pre-set Git hooks. He looked at the pre-commit hook and saw it would download and run a script from a strange IP address, depending on the user’s operating system.

The downloaded script would then fetch another script, hide it in the user’s Documents folder, make it executable, and run it quietly in the background. This second script would download even more files, including a JavaScript file with obfuscated code and a package.json with many dependencies. Some dependencies were suspicious, like clipboard access and file system tools. There was also a tool for Ethereum development, making the author think the malware was looking for crypto wallets or doing something with cryptocurrencies.

The author tried different things to figure out what the hidden JavaScript did, but it was hard to read. He noticed the script was unique for each victim, using an ID in the URL. Looking further, the author found that some victims got a .vscode folder that ran commands as soon as VSCode opened, so the infection could start even without running any code.

He checked the project’s git history and found it was copied from a public repo, with only the malware added. He scanned the attacker’s server but found nothing useful. The main advice was to always check hidden files and directories before running someone else’s code.

In the comments, some readers said this attack is a new twist on old tricks, like hiding malware in code samples. Others were surprised by the use of Git hooks, saying most people never check them. A few said they always delete .git folders from test projects for safety. Some users shared that they’ve seen similar scams, especially with fake job offers and take-home assignments. There were warnings not to trust high-paying job offers that move too fast or ask you to download zip files.

Other readers discussed how easy it is to hide malicious scripts in .vscode folders, and that many developers never look in hidden files. Some people suggested using virtual machines or containers for unknown projects. Others said that companies should give coding tests over trusted platforms, not by sending zip files. Finally, a few commenters praised the author for sharing the details, saying it helps others stay safe.

## Malleable Computing, Emacs, and You

- 原文链接: [Malleable Computing, Emacs, and You](http://yummymelon.com/devnull/malleable-computing-emacs-and-you.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=49013538)

This article is about making Emacs more useful by automating the process of copying GitHub issues into Org Agenda, so you can work with issues inside Emacs instead of using GitHub’s website. The author explains how they built a tool called “fj” to do this and uses their experience to talk about “malleable computing,” which means making software that the user can easily change.

First, the author wanted to avoid copying and pasting issue details from GitHub into Org Agenda by hand. They wanted a way to pull issues into Emacs, track them, make notes, and even create new GitHub issues from Emacs, without having to switch between tools or deal with complicated GitHub authentication. They did not want to write a full GitHub client or worry about syncing everything perfectly.

To solve this, the author used the GitHub command line tool (“gh”) to handle authentication and get issues. Emacs would run the “gh” command, get the results as JSON, and convert them into Org tasks. They used other Emacs packages for menus and displays (like Transient and vtable), and tools like “pandoc” for converting between Org and Markdown. All the main work happens in a small Elisp function that calls the “gh” command, gets the results, and parses them.

The code for the tool is short—under 400 lines—and the author says it only took a few hours to make it work. They highlight how Emacs lets you change and test code on the fly, without restarting anything. This is different from most other software, where you have to rebuild and restart to see changes. Emacs is easy to extend, and you can mix in other command-line tools as needed.

Next, the author talks about software design ideas like the 90/90 rule (the last 10% of features take most of the time) and the Pareto principle (most users only use a small part of the features). They point out that big programs try to serve everyone, but no one uses every feature. With “malleable” software like Emacs, users can build exactly what they need, mixing parts together quickly.

The author mentions that building for yourself is fast, but building for others (even just one more person) requires more work: error handling, documentation, tests, and so on. Malleable tools are great because you can stop when “it works for me.”

With their new tool, the author can browse and copy GitHub issues, create issues in Org syntax, and open issues in a browser—all from Emacs. They say this is a good example of how malleable software gives users power to make their own tools without waiting for someone else.

In the comments, many people agree that Emacs is powerful and unique because you can change it so much. Some say this is why they love Emacs and use it for everything. Others point out that Emacs can be hard for beginners, and the learning curve is steep. A few mention that other editors or tools do not allow this kind of deep customization, and that’s why Emacs is still popular after so many years.

Some comments discuss if malleable software like Emacs could be made easier for more people, or if it is only for technical users. There are also comments about the balance between building something quick for yourself and making a polished tool for everyone. A few users say they have built similar small tools in Emacs for their own work, and that it is very satisfying.

Others worry that using too many personal tools can make it hard to share work or work with teams. One person asks if there are ways to make personal Emacs tools easier to share or package. Some people are excited to try the “fj” package themselves, while others just enjoy reading about the creative ways people use Emacs.

Overall, the comments show respect for Emacs and for the idea of users making their own tools. There is a mix of excitement, advice, questions, and a little worry about sharing and learning.

## GigaToken: ~1000x faster Language model tokenization

- 原文链接: [GigaToken: ~1000x faster Language model tokenization](https://github.com/marcelroed/gigatoken/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=49010167)

Gigatoken is a new open-source tool that makes tokenization for language models much faster—up to 1000 times quicker than popular tools like HuggingFace’s tokenizers or tiktoken. Tokenization is the process of breaking text into smaller parts (tokens) so that AI models can understand and use it.

Gigatoken supports many tokenizers and works on different CPUs, including x86 and ARM. You can use it as a drop-in replacement for HuggingFace or tiktoken, so you don’t have to change much code. If you use Gigatoken’s own API, you get the best speed, since it uses Rust for direct file reading and parallel processing. The library makes sure the output matches other tokenizers, so your results stay the same, but you get much faster performance.

Benchmarks show that Gigatoken can process text at speeds of up to 24 GB/s on powerful servers, compared to just 24 MB/s for HuggingFace. Even on a Mac or typical desktop CPU, the speedup is huge—hundreds of times faster. This means you could tokenize very large datasets, even all of Common Crawl (the public internet), in just a few hours.

The main tricks behind Gigatoken’s speed are heavy low-level optimizations: using SIMD instructions for fast text processing, minimizing branching in the code, and smart caching of repeated tokens. It also avoids slowdowns from Python and thread communication. Some tokenizers, especially ones based on SentencePiece (used by BERT and some Google models), are still slower, but most popular ones work very fast.

If you want to check if your tokenizer is supported, you can run a provided command without installing anything. The developer is open to feedback and bug reports if you find any issues.

In the Hacker News comments, many users are impressed by the speed gains and the careful engineering. Some praise the use of SIMD and cache tricks, saying it’s rare to see this level of optimization in open source. Others ask about real-world use cases—does this really matter outside of very big data applications? Some point out that for most developers, tokenization isn’t the main bottleneck, but for people training or processing massive datasets, it is a game-changer.

A few users discuss the complexity of getting such fast speeds in Python, and why Rust and C are better choices here. There are questions about cross-platform support, especially for Windows users, and the author suggests using Linux or WSL for now. Some wish for better support for SentencePiece models, but understand why it’s a lower priority.

Overall, the community sees Gigatoken as a helpful tool for researchers and engineers working with large language models. There’s excitement about how such speedups might enable new workflows, and respect for the level of technical skill shown in the project. Some people mention they will try it out in their own data pipelines soon.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=49010129)

This article looks into whether AI labs are secretly making their models extra good at a famous test: drawing a pelican riding a bicycle when asked. The test started as a joke but became a well-known way to check if AI models really understand and can follow creative prompts.

The author, Dylan Castillo, wanted to see if labs are “pelicanmaxxing”—that is, training their models to score well on this specific test. He created an experiment: he asked seven popular AI models to draw 48 different animal-vehicle pairs, not just pelicans on bicycles, but things like cats on skateboards and whales on boats. He collected over 1,000 pictures and used another AI to judge how well each part was drawn.

The results showed that pelicans and bicycles were not drawn better than other animals or vehicles. In fact, pelicans were only 6th out of 8 animals, and bicycles were near the bottom for vehicles. Even the special combination, pelican on a bicycle, was not better than you’d expect if labs had trained for this prompt. He also checked if the pictures looked copied or memorized, but found that details like direction (most pelican-bicycle images faced right) were common for other animals and vehicles too, not just pelicans on bicycles.

The author looked at the numbers and did some simple statistics. No lab had a big or special improvement for pelicans on bicycles. Only one lab, GLM-5.2, looked a little better, but it wasn’t clear enough to say it was on purpose.

He also talked about limits in his experiment, like using just one judge AI and not having a big budget. He pointed out that maybe labs do “SVGmaxxing”—getting better at making SVG images in general—but there’s no proof they are cheating just for the pelican-bicycle test.

In the comments, people had mixed views. Some agreed with the author, saying the test is just a fun way to check if AIs can follow odd requests and that labs are probably not cheating. Others wondered if labs might still be optimizing for well-known benchmarks because it helps with marketing and hype, even if not just for pelicans on bicycles. A few users said it’s hard to measure if a test is “overfitted” because labs might just be getting good at lots of prompts, not only this one. Some readers liked the experiment and data, but wanted more images and human judges. There were jokes about how silly the pelican-bicycle prompt is, and some people praised Simon Willison for starting the trend. A couple of comments warned that any benchmark, if popular enough, will get gamed in the end. Others thought it was a good sign that models are not memorizing but actually understanding the prompt. Overall, most agreed the study was thoughtful and fun, and liked seeing a real test of whether “pelicanmaxxing” is happening.

## Everyone Should Know SIMD

- 原文链接: [Everyone Should Know SIMD](https://mitchellh.com/writing/everyone-should-know-simd)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=49010648)

This article explains why every software developer should learn about SIMD, which stands for "Single Instruction, Multiple Data." SIMD lets a CPU handle many pieces of data at the same time, making some programs much faster.

The author says that SIMD is not only for experts or special uses. Instead, it can be simple and helpful in many normal programs. If your code uses loops to process large amounts of data—like arrays or strings—SIMD can help speed things up. The basic idea is to process several values at once, instead of one at a time, by using a special kind of CPU instruction.

The article breaks SIMD programming into five easy steps: 
1. Set up your constants and any needed starting values.
2. Loop through your data in chunks the size of the CPU’s vector width.
3. Do your math or comparison on all those values at the same time.
4. Reduce or use the results as needed (for example, sum them or find where a condition is true).
5. Handle any leftover items with a regular loop.

There’s a real example in the article using the Zig programming language. In this case, the author wants to scan through an array of numbers and stop when a value is below or equal to 0xF. The scalar (normal) loop checks one number at a time. The SIMD version checks several at once, making it up to 16 times faster on some CPUs. The code might look strange at first, but the author explains it step-by-step, showing it’s not so hard once you know the pattern.

The article also explains that although compilers can sometimes do this automatically (auto-vectorization), they often miss chances to use SIMD. So, knowing how to write your own SIMD code gives you more control and better results, especially for important, performance-critical parts of your program.

The main takeaway is that developers don’t need to be scared of SIMD. You don’t have to be an expert or write very complex code; just knowing the basic pattern is enough to get good results.

In the Hacker News comments, many people agree that learning SIMD is helpful, especially for anyone working on performance-critical code. Some share their own stories about big speedups after using SIMD in their projects. Others say the topic can feel scary or too complex, but posts like this one help make it easier to understand.

A few commenters point out that most developers don’t need SIMD every day, so it’s okay to skip it for smaller projects. Some wish more programming languages made SIMD easier to use, without special libraries or hard-to-read syntax. Others remind everyone to always check if the compiler can already do what you want before writing SIMD code by hand.

There’s also a bit of debate about how much time to spend learning optimization tricks like SIMD, versus focusing on code clarity and maintainability. Some users say keep your code simple unless you really need the speed, while others believe that knowing basic SIMD can save a lot of time in the long run.

Overall, the discussion is positive. People like that the article shows how SIMD can be both powerful and simple, and many agree that it’s a good tool for every developer to have in their toolbox.
