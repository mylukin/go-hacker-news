## Claude Code is steganographically marking requests

- 原文链接: [Claude Code is steganographically marking requests](https://thereallo.dev/blog/claude-code-prompt-steganography)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48734373)

Claude Code is a coding tool from Anthropic. The article talks about how it hides secret information in the prompts it sends, using small changes that people usually cannot see.

The tool changes tiny details in the date string of the system prompt. For example, it swaps the normal apostrophe in “Today’s” with a special Unicode character or changes the date separator from “-” to “/”. These changes are used to mark requests in a way that is not obvious to users or models.

The code decides which mark to use based on your computer’s timezone and the base URL used to reach Anthropic’s API. If your timezone is set to “Asia/Shanghai” or “Asia/Urumqi”, or if the API URL matches certain domains or has special keywords, it changes the apostrophe in the prompt. The domain and keyword lists are hidden inside the app using base64 and XOR encoding, making them harder to find.

This hidden marker is part of the information sent to the AI model. Anthropic probably does this to spot people using unofficial gateways, reselling the service, or running special research pipelines. The author thinks this is not evil, but it is strange and sneaky for a tool that needs developer trust.

Anthropic could have just said openly, “We are checking for custom gateways,” and sent this information in a clear way. Instead, they hide it, which makes people more suspicious about privacy. Most users will never notice this, but developers using custom API setups will be marked.

If you want to avoid this, it is easy: change the hostname, timezone, or patch the app. But this trick mostly targets regular developers who do unusual but honest things.

In the comment section, some people say this technique is called “steganography” and is used to hide signals in plain text. Others compare it to digital watermarking, saying it helps catch abuse but feels wrong when it’s hidden. Some think it is clever but too secretive for a developer tool. A few argue that open disclosure would be better—just tell people what is being tracked.

One commenter warns that hiding things like this can break trust, especially for a tool with so much access. Another person points out that similar tricks are common in anti-abuse systems, but they should be transparent. Some are worried that small changes in the prompt could mess up results or break other tools. Others say that if you really want privacy, you should always check what your tools are doing. A few think it is not a big deal if you use the default setup, but it still feels sneaky. Some suggest that open-source tools might be safer for developers who care about trust. Overall, people agree that trust is important, and hiding things—even for good reasons—can make users nervous.

## I ported Kubernetes to the browser

- 原文链接: [I ported Kubernetes to the browser](https://ngrok.com/blog/i-ported-kubernetes-to-the-browser)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48738985)

A developer created “webernetes,” a version of Kubernetes that runs inside your browser, written in TypeScript. Instead of using WebAssembly to port the whole Kubernetes code, they rewrote key parts so it’s much smaller and faster to load, focusing on teaching and interactive demos, not production use.

Webernetes lets you create and manage pods and see how they talk to each other, just like in real Kubernetes. You write your own “container images” in TypeScript, and you can control your cluster with a JavaScript API. The project supports features like deployments, replica sets, simple networking, and pod lifecycle management. However, it doesn’t support everything—there are no ConfigMaps, Secrets, persistent storage, or image pulling from Docker Hub. The main goal is to help people learn Kubernetes concepts in a hands-on way, not to replace real clusters.

The developer used AI (LLMs) to port about 100,000 lines of code from the Go version of Kubernetes, but spent a lot of time reviewing every line and writing hundreds of tests to make sure everything works the same way as real Kubernetes. They found that LLMs made mistakes like skipping tests or using shortcuts, so human review was needed. Integration tests run the same code on both real Kubernetes and webernetes to make sure results match. The process took two months and cost a few thousand dollars in AI usage, but was much faster than manual porting.

Most of the project’s code is almost identical to the original Kubernetes source. The author believes careful review and testing are enough to make AI-generated code trustworthy. They plan to add more features as needed and invite others to contribute.

Hacker News users had many reactions. Some were impressed by the technical achievement and saw it as a great teaching tool for Kubernetes. Others questioned the use of LLMs, worrying about code quality and hidden bugs, but liked the author’s focus on tests and manual review. Some users wanted to see more features added, such as ConfigMaps or persistent storage, while others thought the current scope was perfect for demos and workshops. A few commenters were surprised by how small and fast webernetes is compared to “real” Kubernetes, and thought it could help people experiment with cluster concepts without big downloads.

There was also debate over the use of browser-based tools for developer education. Some people thought running clusters in-browser removes a lot of setup pain, making it easier for beginners. Others wondered if this approach could scale to more complex workloads or if it would only work for small, simple cases. One user suggested that using TypeScript instead of Go might make the project more accessible to web developers.

Some commenters shared their own experiences with porting code using LLMs and agreed that manual review is always needed. Others asked about the cost and token usage, and were surprised by how much AI compute was involved. There was general agreement that this kind of project shows how AI and humans can work together to build new tools quickly. Many users said they planned to try out webernetes or share it with people learning Kubernetes.

This article explains how a pull-back toy car works, using drawings and simple words. The car uses a special spring called a spiral spring and a set of gears to store and release energy. When you pull the car back, you are winding the spring by turning the wheels, which moves the gears inside. The gears help make it easy to wind the tough spring by increasing torque, so even kids can do it without much force. 

The spring has two ends connected to two different gears. One gear turns the outside of the spring, and the other turns the inside, but in opposite directions. This winding action stores energy in the spring. When you let go, the spring unwinds and spins the gears fast, making the car shoot forward. The design uses many small parts: springs, gears, axles, and wheels. The gears not only help with winding but also control the speed when the car drives forward. The article shows that the gears on each side of the car can work alone when driving, but join together when winding.

If you keep winding the car, the spring will not store endless energy; the gear has notches that stop the spring from breaking. The article uses many pictures and animations to show how everything fits together. It also encourages readers to look at more breakdowns and follow the creator for updates.

In the comments, many people share memories of playing with pull-back cars as kids and feeling amazed by the speed. Some readers say they took apart their own toy cars to see how they worked, and they liked seeing this detailed teardown. Others point out the clever use of gears and how it teaches kids about basic engineering. A few users discuss how the spring and gears limit how far you can wind the car, stopping damage. Some mention that the Darda brand is well known in Europe and these cars were very popular there. Others are surprised by how complex the inside of a small toy can be. ## I built a mmWave material classification radar (2025)

- 原文链接: [I built a mmWave material classification radar (2025)](https://gauthier-lechevalier.com/radar)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48736137)

This article is about building a radar device that can tell what materials are inside a wall, especially to find dangerous asbestos. The writer explains how he created this radar as a big school project and tried to turn it into a business, but had to stop because he ran out of money.

He talks about how asbestos is a big problem in Europe, and normal testing is slow and expensive. His idea was to make a radar that could check for asbestos without breaking the wall. To build it, he used two development boards: a Texas Instruments radar board and an ESP32. He set up tests to see how different materials react to the radar signals.

He used a type of radar called FMCW, which sends out signals that change in frequency. The radar listens for echoes, mixes them with the original signal, and uses math (FFT) to figure out how far away things are. Then, he used a method called Capon beamforming to get more details about the angle and type of material. He fed all this data to a small AI (a neural network), which learned to tell the difference between materials.

For the hardware, he built a case and worked a lot on the antennas. He used free software (openEMS) to model the antennas, because the commercial tools are too expensive. He also found tricks to make the simulations faster, like using a Gaussian pulse instead of a Dirac pulse.

He trained his AI to recognize several types of materials, like wood, stone, and plastic, sometimes mixed together. The data set was pretty small, but the system worked well in his tests. Sadly, he couldn’t find enough customers willing to buy or even test the product before it was finished, and getting all the needed legal approvals was too hard without money. In the end, he stopped working on it, but he learned a lot about building hardware and startups.

He shares some lessons: always check if people will pay before building, make a simple demo video, use existing boards to prototype, design cases around electronics, and make the product easy to update. He also warns that radio frequency (RF) work is very tricky and full of surprises.

In the comments, many people are impressed by the technical work and the clear write-up. Some say this is a great example of a real-world hardware project, and wish more startups would share honest stories like this. A few commenters talk about how hard it is to do hardware in Europe, especially with all the rules, and suggest the author might have found more support in other places. ## Stroustrup's Rule (2024)

- 原文链接: [Stroustrup's Rule (2024)](https://buttondown.com/hillelwayne/archive/stroustrups-rule/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48701721)

This article talks about "Stroustrup's Rule," which explains how programming syntax changes as people get used to new features. At first, when a language adds something new, people want very clear and detailed syntax. Later, when everyone understands it, they want shorter, simpler ways to write the same thing.

The article gives examples from Rust. When Rust first added option types, error handling was very clear and wordy, so beginners wouldn’t get confused. Later, Rust added the `try!` macro, and then the `?` operator, making error handling much shorter. This shows that as programmers learn, they want less to read and write.

The same thing happens in other languages. For example, Python added the “walrus” operator (`:=`). Experts liked it because it made code cleaner. But beginners and teachers thought it made things harder to understand. The article explains that this problem happens because beginners need things spelled out, but experts want to write less.

The author also talks about teaching. When teaching a language like TLA+, he uses the longer, clearer way to explain things, even though experts prefer a shorter version. He points out that what is easy for experts can be confusing for new people.

He asks if this way of making things shorter for experts can make languages harder to learn over time. The community starts with mostly beginners, but as more experts join, the language adds more features for them.

In the comment section, many people agree with Stroustrup’s Rule. Some say they have seen this in their own programming experience. Others point out that it’s a balance: too much short syntax can scare off new users, but too much detail can annoy experts.

Some comments mention that documentation should always show the clear, detailed way first, before showing the short version. Others say that languages should let programmers pick which style they want.

A few people worry that, over time, programming languages get too hard for beginners. Some think that new tools or teaching methods can help. Others say that making things too simple can slow down experts.

One commenter says that this rule is not just for programming, but for any skill: we start with clear steps, then use shortcuts when we know more. Another person adds that this is why some languages, like Python, try hard to stay easy to read.

There is also a debate about whether adding short syntax is always a good thing. Some say it keeps code neat, while others say it hides what is really happening. A few people share how they teach beginners, always starting with the longer, clearer way.

Finally, some readers like the idea of “growing with the language”—starting simple, and then learning the shortcuts as they get better. Others wish there was a better way to help both beginners and experts at the same time.

## Building a custom octocopter from scratch with no prior hardware experience

- 原文链接: [Building a custom octocopter from scratch with no prior hardware experience](https://karolina.mgdubiel.com/drone/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48704289)

This article is about someone building a custom octocopter drone from scratch, even though they had no hardware experience. The main goal is to make a drone that can keep flying even if some motors fail, by using reinforcement learning (RL) instead of traditional control methods.

The builder designed the drone in Fusion 360 and used CNC milling to cut parts from fiberglass and carbon fiber. They assembled everything by hand, learning as they went. The first step was to make the drone frame, install the motors and propellers, and get it to fly with a normal flight controller (Betaflight). Once it was flying, the next step was to teach it to handle motor failures using RL.

They started by training a control policy in a simulator (MuJoCo), adding random changes to things like motor strength, mass, and sensor noise to make sure the policy could handle real-world differences. The hardest failures to recover from are when two motors of the same spin direction fail at 90° apart, because this creates a big yaw problem. With RL, the hope is that the drone can learn to stay in the air, even if it has to spin slowly.

There were many challenges: soldering the electronics, balancing the drone, and even fixing bugs in the RL training code. Some problems included the actions getting stuck at the limit (fixed by squashing outputs), and the reward system not giving enough incentive to stay alive (fixed by increasing the survival bonus). After a lot of testing, the RL policy could keep the drone flying in the simulator, even with up to three motors failed in some cases.

The builder also decided not to use a separate computer like a Raspberry Pi for the RL code, but instead to run it directly on the drone’s STM32 flight controller, which cuts down on delays. The design allows the RL policy to control the motors directly, giving more flexibility to handle failures.

In the comments, people were very impressed by how much was learned in a short time, especially with no hardware background. Some said this was a great example of hands-on learning and praised the clear explanations of mistakes and fixes. A few suggested that using Model Predictive Control (MPC) would be more reliable, but agreed that RL is more fun and a good way to learn. Others liked the use of domain randomization in training, saying it helps make the RL controller work better in the real world. Some offered tips about using different microcontrollers or flight controllers, and others shared their own experiences with drones and RL.

There were also comments about the hardware choices, like frame design and motor size. Some people pointed out the importance of safety when testing drones, especially when shutting off motors in flight. A few warned about possible problems with latency and real-world noise, and suggested extra precautions for the first tests. Many commenters thanked the author for sharing all the details, including failures, because it helps others learn too. Some said this project might inspire them to try their own hardware builds, even if they are beginners.
