First, Anthropic and its partners have announced Project Glasswing, a new project that uses AI to find security problems in important software before attackers can use advanced AI to do harm. In just one month, the team and about 50 partners used the Claude Mythos Preview model to find over 10,000 serious bugs in key software systems. This is much faster than before. The main problem now is not finding bugs, but checking, reporting, and fixing all the issues.

By tradition, companies wait 90 days after finding a bug before telling the public, so Anthropic can only share some results. Still, companies like Cloudflare found thousands of bugs, many of them very serious. Groups like the UK’s AI Security Institute and Mozilla also tested Mythos Preview and found it very strong. For example, Mozilla found over 10 times more bugs in Firefox 150 than in previous versions. 

Open-source projects are also affected. Mythos Preview scanned over 1,000 open-source projects and found thousands of problems, including some in a popular security library called wolfSSL. Human experts must review the AI’s results, but this is slow and hard work. Open-source maintainers are struggling to keep up with all the reports. Anthropic has released tools like Claude Security to help companies scan and fix their own code, and they are working with the Open Source Security Foundation to support open-source maintainers. They stress that companies and users should patch software faster, as attacks could come quicker with AI.

Next, we look at why Japanese companies often do many different things, even in very different markets. The article gives examples like Toto, a company famous for toilets but now making most of its profit from a computer chip part. Kyocera makes ceramics, kitchen knives, and solar panels. Yamaha makes pianos, motorcycles, and robots. Even Japanese paper companies might run hotels or make diapers.

This is not common in other rich countries like the US, where companies usually focus on one thing. In Japan, big companies do many things, and do them well. This comes from how Japanese companies are built: they offer jobs for life, promote by seniority, and have company-based unions. Most shares are owned by other Japanese companies, and profits are invested back into the company. 

Japanese companies use a “J-firm” bundle of business practices that values stability and survival over making the most money. They diversify into new areas to use extra workers and reduce risk. This model started during World War II, when the government pushed for production and employee stability. It helped Japanese companies become leaders in cars, electronics, and other fields by making steady improvements. But it is not good at big, new innovations that need bold leadership.

When Japan tried to change to a more American system in the 1990s, it did not work well. Paying workers based on performance broke teamwork and made things worse. Japanese companies are still strong in technical areas, but less good at creating big new products. The article says the Japanese and American systems each have strengths and weaknesses.

In the comments, many people are surprised by how much Japanese companies diversify and how stable this model is. Some say it gives workers security and is good for society. Others point out that lifetime jobs and strong company loyalty can make it hard to change quickly or reward the best workers. Some share stories about working with Japanese firms, saying they value relationships and are slow to fire people. Others ask if this model can survive as Japan’s population gets older. A few note that Korean companies are also very diverse, but usually because of government support. There is also discussion about whether this approach would work in other countries or if it needs Japan’s special culture and history.

Next up, KanBots is an open source desktop Kanban app that lets you run AI agents on every card. This helps developers automate and manage tasks using AI. You can drop a folder to create a board, then assign AI agents to work on different cards. Each card becomes its own work area and updates live as agents make progress. There is an "autopilot" mode, where personas like product manager, engineer, or tester split work into subtasks, run in parallel, and even review each other’s output.

KanBots is local-first, so nothing leaves your computer. It works on Mac, Windows, and Linux, and is free and open source. You can connect it to GitHub, Sentry, and more, and use your own AI provider and custom personas. Other features include tracking the cost of AI runs, setting budget caps, and pausing for human input when needed.

On Hacker News, many people are excited about combining Kanban boards with AI agents. Some think it is a unique and useful way to manage projects, and others are curious about technical details, like how parallel agents work or how the app keeps everything local. Some worry about trusting AI agents with code changes, but like the human review option. Others discuss using KanBots for non-coding tasks or with different team workflows. Many like the cost tracking and the open source model. Some suggest more integrations and features, and others wonder if it could help solo developers or indie hackers. Overall, the response is positive, but with some healthy skepticism.

In other news, a new Minecraft mod called "waylandcraft" lets players run a full Wayland compositor inside Minecraft. This means you can launch Linux desktop apps and open windows right in your Minecraft world. The mod only works on Linux and is for Minecraft Java Edition. You can drag and drop between windows, play videos, and move app windows around inside Minecraft. The mod is open source and uses the Fabric mod loader.

Wayland is a system for managing windows and displays on Linux, so this mod turns Minecraft into a kind of desktop environment. You can pin a video player to your heads-up display or open other apps inside the game. The mod is not for Mac or Windows users.

On Hacker News, many people find the project wild and fun. Some are impressed that Wayland can run inside a game. Others are confused about why anyone would do this, but most agree it’s a cool hack. Moving on, Models.dev is an open-source project that keeps a database of AI models, including specs, prices, and features. The goal is to have one place for up-to-date info on AI models from many providers. Data is stored in simple TOML files, and you can access it through an API or the website. Each model entry lists provider, features, price per million tokens, input/output limits, release dates, and supported data types. You can add models or providers by following clear steps, and GitHub actions check your files for errors.

Anyone can help update the database by submitting pull requests. The website code is open-source, and the project is used by opencode.ai. The maintainers encourage questions and community help.

On Hacker News, many people like the idea of a central place for AI model info. Some say it solves the problem of scattered, out-of-date spreadsheets. Now, Deno 2.8 is out, bringing many new features and big improvements to the JavaScript/TypeScript runtime. This release is all about making Deno faster, easier to use, and more compatible with Node.js and web APIs. New commands include `deno audit fix` for fixing security problems in npm packages, `deno bump-version` to update version numbers, `deno ci` for repeatable installs, `deno pack` to package projects, and `deno transpile` to turn TypeScript into JavaScript. There is also `deno why` to explain why a package is installed.

Deno now treats unprefixed package names as npm packages, so `deno add express` works like `npm install express`. Node.js compatibility has jumped from 42% to over 76%, and Deno is now faster in real-world tasks like package installs and running servers. There are better debugging tools, a built-in CPU profiler, and new options for tracing and logging. The test runner now has per-test timeouts and better coverage reports. Deno supports more browser APIs and has better stream and crypto support. Workspace and package management are improved, and Deno can now use hoisted `node_modules` for legacy Node projects.

On Hacker News, many developers are impressed by the progress and new features. Some are happy that Deno is easier to use with Node projects. Others like the speed improvements. Some are excited about the new debugging tools and better testing. But some users still feel the biggest challenge is the size and history of the Node ecosystem, and moving big projects from Node to Deno is hard. Some worry that too many new features could make Deno more complex. Others point out that Deno is great for new projects, but it is tough to switch over existing Node code. There is also debate about the new `import defer` feature, which delays running code until it is needed. Improved TypeScript support is praised by TypeScript users. Overall, the community feels positive about Deno 2.8.

Next, the article tests different AI coding tools to see how well they can build a 3D model of the Pantheon in OpenSCAD, a code-based modeling tool. Six tools were tested, including Google Antigravity 2.0, Codex 5.5, Claude Code 2.1, Cursor Composer 2.5, and a human-assisted run with ModelRift. The hardest part was making the model look right. Antigravity 2.0 did the best job on its own, using real Pantheon measurements and adding the special ceiling pattern. Codex had good details but some problems in the final file. Claude Sonnet made a balanced model but was slow. Cursor was fastest but weakest. The human-assisted run did better, because a person could point out mistakes and help the AI improve.

The results showed that OpenSCAD is good for AI code generation, but making high-quality models still needs human help. Speed did not mean better results, and previews could be misleading. None of the models were perfect, but all looked like the Pantheon.

On Hacker News, some people praised OpenSCAD as a good tool for AI-driven modeling. Others noted that building something like the Pantheon is much harder than making simple shapes. Some liked that Antigravity 2.0 used real measurements. Some said the best results still needed human help, showing that fully automatic 3D modeling by AI is not ready yet. Others wondered if other CAD tools would work better. Finally, yt-dlp announced that support for Bun as a JavaScript runtime is now limited and will soon be removed. This affects people using yt-dlp with Bun, especially with its ejs system. The main reason is security: older Bun versions ignore lockfiles, which is risky after recent npm supply chain attacks. The team is also worried about Bun’s recent rewrite in Rust using AI and feel the project is moving in a direction they do not trust. The new minimum supported Bun version is 1.2.11 and the maximum is 1.3.14. If yt-dlp drops support for Bun later, they suggest plugins could add back support.

On Hacker News, some people agree with this cautious move, saying it is better to avoid trouble. Others are unhappy, saying it is inconvenient to install another JS runtime just for yt-dlp. Some want a warning instead of a block. There is debate about whether most web developers really have only Bun and not Node.js. Some joke that installing another runtime is not a big deal. One Deno maintainer explains that Deno uses AI for bug fixes but always reviews the code. Some suggest people could write plugins themselves if they want Bun support. Overall, opinions are mixed, with some supporting the move and others saying it limits user choice.
