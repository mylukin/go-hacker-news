First, we look at how an old laptop was turned into a “writerdeck”—a simple, distraction-free writing machine. The author wanted to focus on writing, away from the noise of the internet. They used a six-year-old System76 laptop with Debian Linux, skipping the desktop environment and booting straight into a text-only terminal. This breaks old habits and keeps the device focused only on writing.

The writer installed neovim as their editor and tmux for splitting windows and adding a status bar. They also used kmscon for a modern, scalable terminal, and customized tmux to show battery level, adjust brightness, and put the status bar at the top in green. For notes and writing, they set up vimwiki, and for syncing files, they used syncthing—even though the web interface is open to all addresses, which is not very secure. The laptop auto-logs in and launches tmux and vimwiki right away, so writing can start immediately.

After a week, the author felt more productive and less distracted, enjoying a device that does only one thing: writing. Next, .NET 11 and C# 15 have added union types, a feature long requested by developers. With union types, a variable can hold one of several different types, making code clearer and safer. This is common in languages like F#, TypeScript, and Rust. Before now, C# developers had to use base classes or objects, which was not ideal.

The new union types use the union keyword and let you easily check which type is held with a switch expression; the compiler warns you if you forget to handle a case. Unions are implemented as structs with a Value property, and you can avoid some performance issues by writing custom union types. To try union types, you need the .NET 11 preview and to turn on language preview features. Some IDEs already support this, while others are still adding support.

Many on Hacker News are excited to finally see union types in C#. Some point out that F# and functional languages have had this for years. Others are concerned about performance, especially with boxing, and ask how unions will work with pattern matching. Some mention using libraries like OneOf before this, and hope the built-in version will make code simpler. There’s also interest in real-world use cases, like error handling. A few compare C#’s unions with those in Rust or Swift, noting differences in how switch statements work. Many are waiting to see how this feature will evolve in future releases.

Now, let’s look at computer history. One article shows how the Spacelab computer from the 1980s was built. The Mitra 125 MS, made in France, used many small logic chips instead of a single microprocessor. It had a 16-bit processor, magnetic core memory, and was packed with chips like the 74181 ALU, famous among electronics hobbyists. The article also covers French efforts to build their own computers for independence.

Hacker News readers enjoyed the technical details and stories about the 74181 chip. Some talked about France’s Plan Calcul and the challenge of competing with IBM. Many said learning about these old systems helps you understand computers better. There was interest in the photos, the use of "bodge wires" to fix mistakes, and the special ways boards were keyed. People thanked the author for sharing this important piece of history.

On a related note, another article discusses z386, an open-source CPU that recreates the Intel 80386 using its original microcode and runs on FPGAs. It can boot DOS, Doom, and more, and aims to be a true 386 hardware copy, not just an emulator. z386 keeps many classic parts like the paging TLB and microcode ROM, and uses eight main units to handle x86 instructions. It adds a 16KB cache for speed and is tested with many old programs.

Many on Hacker News are impressed by the project. Some ask about the legal side of using Intel’s microcode, and others wonder if z386 can run Windows or Linux. There’s discussion about the challenges of x86 decoding and how open-source CPUs are good for learning. Some compare z386 to other FPGA CPUs, like ao486, and share tips or memories of old DOS games. Many thank the author and want to follow the project’s progress.

Finally, we have a deep dive into the disassembly of the Intel 80386’s microcode. The author and their team used image processing, neural networks, and manual labor to extract and read the microcode from a chip photo. The 386 uses microcode for all instructions, unlike some older or newer CPUs, and has many entry points and helper hardware units. The team found patterns, possible bugs, and shared the microcode on GitHub.

Hacker News readers are amazed by the teamwork and technical skill. Some discuss the possible security bug found in the microcode, while others praise the use of modern tools like neural networks to study old computers. There’s interest in doing similar work for other CPUs, but newer chips are much more complex. Many share stories about programming on the 386, and there’s respect for the original chip designers. Some ask technical questions and get helpful replies from the author and others.
