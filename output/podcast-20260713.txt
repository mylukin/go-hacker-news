First, let’s talk about building and releasing macOS and iOS apps without ever opening the Xcode app. You still need to install Xcode, but developers are using command-line tools and scripts for almost every step. The only time you need the Xcode GUI is for one-time setup: signing in to your Apple Developer account, creating certificates, and saving a special password for notarization. After that, the workflow uses tools like xcodebuild for building, notarytool for notarization, stapler for attaching notarization, and devicectl for installing to devices. Xcodegen helps manage the project using YAML, keeping your git repo clean.

A key tip from the article is to write a release.sh script. This script runs the full release: archiving, signing, notarizing, stapling, and installing your app. If anything fails, it stops and shows an error, so you always know if something is wrong. For code signing, your private key stays safe in your keychain, and the script finds it on its own. Notarization uses a password stored in a profile, which keeps secrets out of your code and git history.

For quick testing, you can build unsigned versions, and when you want to ship to users, just run the release script for a signed, notarized app. For iOS, you can use xcodebuild and devicectl to put the app on your phone, still without opening Xcode.

The article also points out that large language models, like Claude Code, can help write or update these scripts for you. Once your scripts are set, you can run everything from the terminal, or even let an AI agent handle it.

From the comments, many people say this saves time and avoids Xcode bugs. Automation makes builds more repeatable and perfect for CI systems. Some still like the Xcode GUI for debugging or logs, but agree that scripting releases is smarter for automation. The one-time setup is still confusing for some, especially around certificates. Beginners might find the command line hard, and a few warn that Apple sometimes changes things, which can break scripts after Xcode updates. Tips for using similar workflows on other platforms were shared, and many liked that secrets are never stored in git. Some worry about backing up private keys safely. In the end, most readers agree: leaving the Xcode GUI for routine builds makes life easier, but the first setup should be simpler.

Next, Apple has released a new speech recognition API called SpeechAnalyzer, replacing SFSpeechRecognizer in iOS and macOS 26. The team at Inscribe compared SpeechAnalyzer against OpenAI’s Whisper models and Apple’s old engine using standard tests. The main score is Word Error Rate, or WER. Lower WER means better results. On clean speech, SpeechAnalyzer scored 2.12% WER, much better than Whisper Small at 3.74% and the old Apple engine at 9.02%. For noisy speech, SpeechAnalyzer still led at 4.56%, with Whisper Small at 7.95%. The new Apple engine is about three times faster than Whisper Small and adds proper punctuation and casing. Inscribe now uses SpeechAnalyzer for supported languages and Whisper for the rest.

However, Whisper still supports over 100 languages, while SpeechAnalyzer has about 30, and Whisper runs on more platforms, not just Apple devices. The test was fair: all engines ran faster than real time, the same code and audio were used, and results are public online.

Some limits: tests were only for English, using clear audiobook speech, and only on one Apple chip. The Whisper models were converted for Apple, so might differ a bit from the originals.

In the comments, people were impressed by Apple’s accuracy and speed, saying this could help on-device privacy. Many like that Apple is catching up to or passing OpenAI for English speech recognition. Still, multi-language support and cross-platform use keep Whisper important, especially for apps on Windows or Android. Some users are waiting for more real-world tests—like meetings, heavy accents, or noisy rooms—before switching. People also praised Inscribe for publishing all their data and methods, which builds trust. Others warned that Apple APIs can change quickly and may not be as easy to use as open source tools. Some asked if Apple will open up their model for other platforms, but most doubt it. There were also questions about battery life, since faster models might use more power. Overall, most agreed this is a big step for Apple’s on-device speech recognition.

Now, let’s talk about the real cost of using different AI models for coding. The article explains that even if two models seem to cost the same per “token,” the way they count tokens can be very different, which changes your real cost. For example, the same TypeScript file is 681 tokens on GPT-5.x but 1,178 on Claude’s newest model—almost 1.73 times more. So, you could pay much more with Claude if your code files are large.

The author tested real files and found that Anthropic’s new tokenizer makes about 30% more tokens than their old one, with no change in sticker price. For code, Claude’s new tokenizer can make up to 73% more tokens than GPT. For Chinese text, Claude still uses more tokens than GPT, so it’s not just an English issue. The author checked invoices, and the numbers match what the token counters say. There are tables in the article showing the cost differences in practice. For example, after August 2026, the same code will cost about 32% more on Claude Sonnet 5, even if the token price stays the same. Other models like Gemini and Grok were also compared, but their token counts are closer to GPT’s.

The takeaway: don’t just look at price per token. You must check how your own content is tokenized, because model changes can feel like a hidden price hike. This is especially important for coding, where the token gap is biggest.

In the comments, many were surprised by how much token counts vary and how this can lead to unexpected bills. Some compared it to “shrinkflation” in food, where you pay the same for less product. A few defended the companies, saying improving tokenizers is hard and sometimes needed. Others said vendors should be clearer and show real tokenization examples. Some shared their own numbers and confirmed the differences. There was debate about whether this is misleading or just a technical detail, but most agreed developers need to watch this closely. Some also talked about the balance between model quality and price, saying sometimes it’s worth paying more for better results. But everyone agreed: don’t just compare per-token prices, look at how your data is split into tokens.

Now to an interesting open source project: linux-0.11-rs, a full rewrite of the old Linux 0.11 kernel in modern Rust. The new kernel can boot on i386 hardware using QEMU, just like the original from 1991. The goal is to keep the system working as before but use safe, clear Rust code. The kernel uses strong types and better code structure for safety and understanding. It supports processes, virtual memory, the Minix v1 file system, disk drivers, VGA and serial consoles, TTY, signals, and a complete set of system calls.

There is also a Rust library for user programs, so you can write software in Rust and it looks just like normal Rust code. The project comes with over 80 command-line tools and a shell with pipelines, loops, globbing, history, and tab completion. You can build a bootable image with one command, and there are tools for disk images and testing. The repo is well organized, with clear sections for kernel, user library, core utilities, and more. There’s even a developer container for easy setup, and tests that boot the kernel and check if everything works. It’s almost feature-complete with Linux 0.11, except for things like floppy disk support. The team plans to improve docs and guides.

People in the comments were impressed by how much of Linux 0.11 was rewritten in Rust, including the userland tools and shell. Many said this is a great way to learn both Rust and how operating systems work. Some think rewriting old code like this makes systems safer and easier to maintain. Others warn that writing an OS in Rust is still hard, especially for low-level hardware. Some asked about performance, or what the project is useful for, since it’s based on an old Linux version. Others praised the clear repo organization and docs. Some suggested porting more modern Linux features, or using the project for learning about kernels. Overall, there’s lots of excitement for Rust and respect for the effort.

Here’s a creative tool for musicians: a command-line app that turns YouTube guitar lesson videos into PDF files with guitar tabs. You just give it a video link. The tool uses yt-dlp to download the video, then ffmpeg to take screenshots every few seconds. It uses Claude vision, an AI, to find where the guitar tab is in each screenshot. It crops out just the tab part, then checks for near-identical images using the bar number in the music. This way, you get one tab image per bar in the song. Finally, it puts all the unique tab images into a PDF, using the video’s title as the name and heading.

To use it, you need Node.js, yt-dlp, ffmpeg, and an Anthropic API key. The setup is simple, and you can just run a command with the YouTube link. There are options to change how often screenshots are taken or which AI model to use, but the defaults work for most people. The process is automatic.

In the comments, many guitar players think the tool is very useful, especially for saving tabs from video lessons. Some are impressed by how well it finds and crops the tab parts. Others like the use of AI, but a few worry about needing an Anthropic API key, which might cost money or have limits. Some hope it will support other types of music notation in the future, not just guitar tabs. A few say the process could be slow for long videos, and some suggest features like making the PDF easier to read on phones or letting users choose sections. There are also comments about how open source tools like this help people learn songs faster. Some want support for other instruments or better performance. Overall, people like the idea and see many ways it can help musicians.

Now for a fun hardware project: the story of getting Linux running on the Sega 32X, an old add-on for the Sega Genesis game console. The 32X has two 32-bit CPUs and extra RAM, but is still very limited compared to modern computers. The author used a special flash cartridge to add more memory, turning some ROM into writable RAM. This made it possible to load and run the Linux kernel, even with tight limits.

There were many technical challenges. The toolchain crashed on some files, but the author found workarounds. Some Linux code did not work with the Sega CPUs and had to be fixed, especially math and memory parts. The author had to write new drivers, like for sending text over a serial port.

One of the hardest problems was using both CPUs at the same time. The 32X CPUs have no hardware to help them share memory safely. The author used Peterson’s algorithm, a software way for two CPUs to take turns, but it’s slow. The old Genesis CPU was used as a helper for communication. Cache was turned off to avoid bugs from old data, and a simple way was added to show Linux output on the Sega’s screen. Performance is slow, but everything works.

In the comments, many readers were impressed by the dedication and technical skills. Some said this kind of work is fun and a great way to learn about old hardware and Linux internals. Others noted how hard it is to run modern software on retro consoles, especially with the 32X’s dual CPUs and lack of standard memory. Some shared their own stories of running Linux on old hardware. There was discussion about Peterson’s algorithm, which only works for two CPUs. Some said the project is not practical, but very cool for learning and hacking. A few wondered if this skill could be useful for jobs in embedded systems or hardware bringup. Several thanked the author for open-sourcing their code and configs.
