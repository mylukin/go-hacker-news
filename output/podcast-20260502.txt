First, let’s talk about a change in Visual Studio Code that made many developers unhappy. The Git extension in VS Code now adds “Co-Authored-by: Copilot” to commit messages by default, even if you do not use Copilot. This happened because the setting “git.addAICoAuthor” was changed from “off” to “all,” so the tag appears unless you turn it off yourself. Some code in the extension still used the old setting, adding to the confusion. Many users found Copilot listed as a co-author, even when they wrote all the code by hand and had turned off all AI features. There was no big announcement or clear notice, so most only saw the change after checking their commits.

In the comments, most people are upset and confused. Many call the change “unacceptable” and say it feels dishonest, even like “vandalism,” to add Copilot when it did nothing. Some worry this will ruin commit history, and make it hard to track real authors. Others are angry that it is on by default and users were not told. A few say they are switching to other editors, like Zed, because of this. Some joke that Microsoft wants Copilot to look more popular. A couple of users explain how to turn the feature off, but most agree it should not be enabled by default. One says other editors, like Cursor, do it better by only adding AI tags if you actually use AI code. Overall, most want this change undone or at least made optional.

Now, let’s shift to a long-term project on the Apple Watch. The article shares how a developer spent six years building a great map app for the watch, called Pedometer++. The writer, who loves hiking, wanted to make checking your location easy while outdoors. Early watch screens were small and slow, so the first maps were very simple and not good for real-time or offline use. When SwiftUI came to watchOS, the writer made a custom map engine that worked faster and showed map tiles and user location right on the watch. Designing for the tiny watch screen was hard. He tried many layouts, but most did not feel right. He wanted the app to be easy to use with one hand and require little setup.

When Apple released the “Liquid Glass” style, the writer saw a new design chance, but found normal maps did not look good. So, he hired a cartographer to make a custom map that looked better and worked well in dark mode. He also worked with a designer to find a layout where metrics were in the corner and the map filled the rest of the screen. After much testing and hiking, he finished a design that was clear and useful.

He did not use Apple’s MapKit because it was not flexible enough for hiking trails and always showed dark mode. It did not allow enough custom features. In the comments, people praise the effort and attention to detail. Some share struggles making watchOS apps and say design is often harder than coding. There’s interest in offline maps for hikers, and questions about building custom map engines. Some hope Apple will give developers more tools for watchOS. There’s debate about whether most users need this level of detail, but most agree it’s great for outdoor activities.

Let’s move on to web browsers. Ladybird, an open-source browser, got many updates in April 2026. Thirty-five people contributed, including seven first-timers. The Human Rights Foundation became a big sponsor, giving $50,000. Ladybird now has an inline PDF viewer, better address bar autocomplete, and faster HTML parsing. JavaScript runs quicker, and iframes now draw faster and more safely. There is a new Linux frontend using GTK4 and libadwaita, matching GNOME design. Bookmarks are easier to manage. Ladybird also supports new CSS and web features, like image-set and smoother SVG animations. Networking is faster and blocks less, so sites load quickly, especially on slow networks. GPU painting is more efficient on Linux, and both C++ and Rust parts now use mimalloc for memory, making everything faster.

Popular sites like Reddit and YouTube work better, and many bugs are fixed. Ladybird’s web platform test score jumped, showing it is more standards-compliant. Rust is now required to build it, and there are many small improvements, like better text selection and error pages.

In the comments, many are excited about Ladybird’s progress. Some compare it to Chrome, Firefox, and Safari. Users praise its modern features and site compatibility, and wonder if it could compete as a daily or mobile browser. They are impressed by the team’s hard work and clear release notes. Some talk about the need for more browser engines for web freedom. A few worry about funding, but are glad to see new sponsors. Many thank the contributors and encourage others to help.

Next, we have a story about privacy and tracking. The article suggests a new standard: an environment variable called DO_NOT_TRACK, to help users stop software from collecting data. Today, command-line tools and SDKs all have different ways to turn off tracking, making it confusing. The proposal is simple: set DO_NOT_TRACK=1, and all tools should turn off tracking, including telemetry, crash reports, and usage data. The article gives examples for setting it in different shells and asks software makers to check this variable in their code. It also suggests making tracking opt-in by default to respect privacy.

In the comments, some users love the idea and say it would save time. Others doubt companies will respect the variable, since many want the data. There’s a comparison to the “Do Not Track” setting in browsers, which is often ignored. Some developers say it’s hard to add new variables to old tools, but easy for new ones. If enough users ask, more tools may support it. People compare this to the NO_COLOR standard—slow to adopt but helpful. Many agree data collection should be opt-in, not opt-out. There are tips for making sure the variable is set everywhere. Overall, there is hope and doubt, but most think a simple “do not track me” option is good.

Now, let’s go back in time to Neanderthals in Germany, 125,000 years ago. Scientists found thousands of small bone pieces at a site called Neumark-Nord 2, showing Neanderthals crushed big bones and heated them in water to get bone grease, a food rich in calories. This kind of complex food processing was thought to be unique to modern humans, but here it was done by Neanderthals. They worked together to collect bones from at least 172 big animals and processed them by the lake, using fire, stones, and tools. The work was organized, with hunting, butchering, and fat-making in different areas. This shows Neanderthals planned ahead, worked as a group, and valued fat in their diet. The remains also show they used plants and sometimes hunted huge animals.

This discovery changes how we see Neanderthals, showing they were clever and organized. In the comments, some people are surprised at how smart Neanderthals were. Others discuss the importance of fat for survival. Some point out that making bone grease is hard, needing teamwork and planning. There are questions about sharing food and living in different environments. Some ask if this changes what we know about human evolution and surviving the ice age. Others just enjoy learning something new about ancient people.

Next, an article discusses where to run the “agent harness” that controls AI code assistants. There are two main ways: inside the sandbox (simple and good for one user) or outside the sandbox (better for many users and for security). Running inside is easy, as everything is in one place. Running outside keeps secrets safer, lets you stop and start sandboxes as needed, and makes it easier to share data in teams. But you have to build your own system for saving progress and handling files.

The author’s team chose to run the harness outside. They solved problems like keeping the agent running if servers restart and storing files in a database. Their system uses Inngest to save progress and Blaxel to start sandboxes fast. The harness pretends there’s a normal filesystem, but really uses a mix of database and sandbox files.

In the comments, many agree that running outside is better for security and teamwork. They like keeping secrets safe and using databases for shared knowledge. Some worry about the extra complexity, like building a fake filesystem and handling file conflicts. Others say using APIs and databases is normal in large systems. There’s talk about scaling up, and about “last-writer-wins” for file updates. Overall, people understand the tradeoffs and see why this choice was made.
