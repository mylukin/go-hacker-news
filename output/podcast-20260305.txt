Next up, there’s a great article about how Linux detects when you plug in or remove hardware, like USB devices. The writer wanted to understand the low-level details, not just use libraries like libusb. Most systems use udev to manage hardware events. The Linux kernel tells udev about new devices using netlink, a special way for the kernel and programs to talk. Udev listens, sets permissions, updates device files, and rebroadcasts these events to other programs.

There are two ways to listen for hotplug events: directly from the kernel or from udev. Kernel events are simple—open a netlink socket and watch for messages. Udev events are a bit more complex, with headers and extra data like hashes and a bloom filter for fast filtering. The article shows C code to open sockets, choose events, and print details. It explains each field in the udev message header and how filtering saves system resources. Security is important—programs check that messages really come from the kernel or udev by looking at sender credentials.

In the comments, many readers thank the author for the detailed explanation and clear code. Some say the netlink and udev system is powerful, but also complex and easy to get wrong. There are stories about bugs caused by race conditions, and most agree that using udev is safer than listening to kernel events directly. Some like the use of bloom filters to avoid waking up every program for every event. Others ask about backwards compatibility, and a few want better documentation for netlink and udev internals. Overall, people found this article helped them understand how Linux handles hardware changes.

Next, a new attack called “Clinejection” let hackers break into 4,000 developer computers using just a GitHub issue title and some AI tools. The attack started when someone changed one line in a popular npm package called Cline. This line installed another tool, OpenClaw, on every user’s machine during setup. The attacker opened a GitHub issue with a special title. Cline’s AI bot read the title as an order and ran code from a fake repository. This code poisoned the GitHub Actions cache with junk files. When the team’s nightly script ran, it picked up the bad files—including the attacker’s code. The script had secret keys used to publish npm packages, and the attacker stole these keys. With them, the attacker published a hacked version of Cline. Anyone who updated or installed Cline during those eight hours also got OpenClaw, which can run any command on your computer and survive reboots. The attack was caught by automated security monitoring later.

A security researcher had reported the problem weeks before, but the Cline team did not act fast enough. They tried to fix things by rotating access tokens, but left the real token active, which the hacker used later. The lesson is: the attack didn’t use new malware or fancy code. It used normal tools, update systems, and AI bots that were not checked well enough.

Hacker News commenters had a lot to say. Some warn that letting bots run commands based on user input is dangerous, especially with real access tokens. Others say basic security, like checking inputs and using the least permissions, could have stopped this. Some worry AI-powered bots in developer tools will create more risk. There are calls for package managers to show when postinstall scripts run or block them by default. Some feel the Cline team’s slow response and mistakes made things worse. There’s debate about whether OpenClaw was a real harmful payload or just a proof-of-concept, but most agree the attack method is the real danger. Checking every small change in big projects is hard, especially when it’s just one line in a config file. Some suggest better code review tools or using OIDC-based publishing so tokens can’t be reused. Many see this as a warning: don’t trust inputs without checks, always protect secrets, and make sure bots have only the permissions they need.

Now, a fun science story: the family tree of citrus fruits, shown in a new way using a ternary plot. Most citrus fruits, like oranges, lemons, and limes, come from mixing three main types: mandarin, pomelo, and citron. The ternary plot has three corners, each for one original species. If a fruit is closer to one corner, it has more genes from that species. You can click on each fruit in the chart to see how much of each type it has. The article explains that normal family trees don’t work well for citrus, because the fruits mix so much. The three main species split apart over five million years ago, after a big climate change. Over time, people spread citrus fruits and made new types that are sweeter and nicer to eat. Oranges, lemons, and grapefruits are all mixes of the three main species.

Some wild species, like samuyao, also join the mix, making the plot more complex. The chart doesn’t show when fruits appeared, but shows how closely related their genes are. The author says this makes it easy to see which fruits are close, which are far, and how people’s taste for sweetness changed what we eat.

In the comments, many users like the ternary plot, saying it makes the mixed-up citrus family tree easy to see. Some are surprised that most citrus fruits are hybrids, not separate types. Others enjoyed learning how breeding and human choice changed citrus over time. Some point out that showing plant history is hard because plants mix in ways animals don’t. Others say the chart is not perfect and leaves out some wild types, or ask if more fruits could be shown this way. Some share stories about growing citrus or trying rare kinds. There’s debate about what counts as a true species versus a hybrid, and a few want more links to genetic studies. Overall, people are excited to learn how complex and interesting citrus history really is.
