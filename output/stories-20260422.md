- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47868867)

Apple released a fix for a bug that let police use special tools to recover deleted chat messages from iPhones and iPads. The bug kept copies of message notifications, even when users deleted the messages in apps like Signal.

The problem was that when someone got a message, the content often showed as a notification. Even after deleting the message in the app, a copy of the notification stayed in the phone’s database for up to a month. This allowed law enforcement, using forensic tools, to read messages that were supposed to be gone. The issue became public after a report showed the FBI used this method to get deleted Signal messages from a suspect’s iPhone. Signal’s president then asked Apple to fix the problem, saying deleted message notifications should not stay on the device. Apple’s new update removes these leftover notifications and backports the fix to older iOS versions too. Apple did not explain why the notification content was kept in the first place, but now it should be gone for good. Privacy experts were worried because many people use message deletion to stay safe if their phone is taken. Apps like Signal and WhatsApp let users set messages to disappear, but this bug made those features less safe.

In the Hacker News comments, many people said this is a big privacy issue and were surprised Apple had this bug for so long. Some users discussed how hard it is to make sure deleted data is really gone, especially on complex devices like phones. Others pointed out that users often think deleting a message means it’s fully erased, but often copies remain somewhere. A few commenters explained that notification systems are tricky, since different apps and the operating system itself might keep records. Some worried about what other kinds of data might linger after “deletion.” Others felt Apple should have caught the problem earlier, especially given their focus on privacy. Some people said this shows why open-source software can be useful—it’s easier to check how deletion works. There were comments about how law enforcement will always look for ways around security, so users need to be careful. A few thought Apple did the right thing by fixing it quickly, but others questioned if enough was done to warn users. Finally, some said this story is a good reminder to keep devices updated and not trust that any deletion is perfect.

## Over-editing refers to a model modifying code beyond what is necessary

- 原文链接: [Over-editing refers to a model modifying code beyond what is necessary](https://nrehiew.github.io/blog/minimal_editing/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47866913)

This article looks at how AI coding tools often change more code than needed when fixing bugs. The author calls this "over-editing," which means the model rewrites or adds things that were not part of the original problem.

Most AI coding assistants, like Copilot, Claude, or GPT, sometimes fix a small bug but then also rename variables, add new checks, or reorganize code. This makes the changes very big, even if only one line really needed to change. For people doing code reviews, this is a problem, because it’s hard to see what was actually fixed and why. Too many changes can also make the code harder to read and maintain.

The article gives an example where the bug is just an off-by-one mistake in a `range()` call, but GPT-5.4 rewrites the whole function, adding checks and changing logic that wasn’t needed. This is especially bad in real-world projects, where small, clear fixes are best.

To study this, the author makes a dataset of code with simple bugs, like a wrong operator or a boolean flipped. The idea is that the smallest fix is clear: just undo the bug. The author measures how much the AI changes the code using token-level Levenshtein distance (which counts how many code tokens change), and cognitive complexity (which measures how hard the code is to understand).

The results show that all tested models, even the newest ones, over-edit a lot. For example, GPT-5.4 changes much more code than needed, while Claude Opus 4.6 is more careful and makes smaller, focused changes. The author also finds that telling the model directly to "make as few changes as possible" helps a lot—almost all models do better with this simple instruction.

The article then looks at how to train models to avoid over-editing. Regular supervised fine-tuning works only if the training and test bugs are similar, otherwise the model just memorizes. Reinforcement learning, where the model gets rewarded for small edits that fix the bug, works better and generalizes to new kinds of bugs, without forgetting how to code in general.

Some other findings: using LoRA (a lightweight tuning method) is almost as good as full fine-tuning for this task; and these results hold for both small and large models. The author notes that GPT-5.4 is bad at minimal editing by default, but improves a lot with a clear prompt. Claude Opus 4.6 is good even without extra instructions.

In the Hacker News comments, many people say they have seen this problem in their own work. Some think over-editing happens because the models try to show off or "improve" the code, even when not asked. Others point out that AI models are trained on lots of code, so they may not know when to stop at the minimal fix. Some users like the idea of measuring and rewarding minimal editing, while others worry that sometimes a bigger rewrite really is better for code quality.

Several commenters suggest that better prompts can help, but that tools should make minimal edits by default, unless asked to refactor. Some people worry that junior developers may accept big changes from AI without understanding them, leading to problems in the codebase. Others note that as models get better at following instructions, this problem might go away, but only if we train and prompt them carefully.

A few users say that over-editing can hide bugs or introduce new ones, making code reviews harder. Some suggest adding tools that show exactly what changed and why, or even having a "minimal edit mode" in AI coding assistants. There’s also discussion about whether models should be able to explain their changes, not just make them.

Overall, the comments agree that over-editing is real and annoying. Many are hopeful that with better prompts, training, and tooling, AI models can learn to make smaller, safer code changes.

## We found a stable Firefox identifier linking all your private Tor identities

- 原文链接: [We found a stable Firefox identifier linking all your private Tor identities](https://fingerprint.com/blog/firefox-tor-indexeddb-privacy-vulnerability/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47866697)

This article talks about a privacy problem in Firefox and Tor Browser. Websites could use a hidden browser feature to track users, even when people thought they were private.

The problem comes from how Firefox handles something called IndexedDB. IndexedDB is a tool websites use to store data in your browser. There is a command, indexedDB.databases(), that lists these databases. In Firefox and Tor Browser, this list was ordered in a way that was not random or neutral. Instead, the order depended on hidden details inside the browser, and this order stayed the same as long as the browser was open.

Because of this, any website could create and list several databases, then look at the order. That order worked like a fingerprint. Even if you were using private browsing or Tor Browser’s “New Identity” button, the fingerprint did not change unless you completely closed and restarted the browser. This meant websites could track you across private sessions and even across different websites. This is a big deal for privacy, especially for Tor users who want extra protection.

The article explains that the problem happened because Firefox used a special way to name databases in private mode. These names were saved in a table that lasted as long as the browser was open. When websites asked for a list of databases, the browser gave them in the order from this table, which did not change unless you restarted the browser.

If a website made, for example, 16 databases, there were many possible orders (over 44 bits of information). That is enough to tell different users apart, or link visits together, even without cookies.

The fix was easy: just sort the list before giving it to websites. That way, the order is always the same, and does not leak private details. Mozilla has fixed this problem in Firefox 150 and in the latest version of ESR. The Tor Project is also aware and needs to update their browser.

People on Hacker News had a lot to say. Some were surprised that such a small detail could break privacy so badly. They pointed out that browser privacy is very hard, because even things that look safe can leak details. Others talked about how hard it is for browser makers to think of every way that features could be misused.

A few users worried that this kind of problem might exist in other browsers too, not just Firefox. Some said this is why they do not fully trust private browsing modes. Others praised Mozilla for fixing the problem quickly. Some comments reminded everyone that Tor Browser is only as strong as the browser it is built on, so bugs in Firefox can hurt Tor too.

One person wondered if other APIs might leak similar information. Another user said this shows why open-source browsers are important: people can find and report bugs quickly. There was also talk about how hard it is for developers to balance useful features and privacy.

A few users joked that privacy is always an illusion on the web. Others said that while the fix is simple, the lesson is important—privacy needs careful design, and even small leaks can matter a lot. Several people agreed that browser makers must test for tiny leaks, not just big ones like cookies.

Some users said they would update Firefox right away, while others asked for better tools to test browser privacy. Overall, the comments were a mix of worry, praise for the fix, and reminders that privacy is always a moving target.

## 5x5 Pixel font for tiny screens

- 原文链接: [5x5 Pixel font for tiny screens](https://maurycyz.com/projects/mcufont/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47824943)

This article is about making a 5x5 pixel font for very small screens, such as those found on microcontrollers. The author explains why 5x5 is the smallest size that still lets you read letters clearly.

The article looks at even smaller fonts—like 3x5, 3x4, and down to 2x2—but shows these are too hard to read or lose important details in letters. The 5x5 font fits all characters into a small, square grid, and it’s easy to use in code because every letter is the same width. This makes string length simple to calculate for display on tiny screens, and there’s no risk that some numbers or letters will be much wider than others.

The whole font uses just 350 bytes of memory, which is very good for small microcontrollers with little RAM. These chips usually have to work with tiny, cheap screens—like 128x64 pixel OLEDs. Such displays need special, space-saving fonts to look good. The author says vector fonts (the kind on your computer) use much more memory and look worse at this size.

On real screens, pixels are not perfect squares, so the font may look a bit different than a “perfect” version. Some effects, like fake shadows, can even make it look nicer on color screens. The author also shows what happens to readability as the font gets smaller—at 3x3, most numbers are lost; at 2x2, the font is basically useless for normal reading.

The article links to the C header file for the font and shows how you can preview or use it in your own projects. It also lists related fonts and resources for people interested in tiny pixel art fonts.

In the comments, many readers praise the effort and usefulness, saying 5x5 is a good size for both clarity and code simplicity. Some people share memories of working with similar fonts on old computers or calculators. Others suggest that even smaller fonts might work for very limited cases, like simple icons or for just numbers, but agree full text is tough to read. A few comment that using a fixed-width font is smart for easy programming and layout.

One commenter asks about adding more languages, like Greek or Cyrillic, and wonders how well those alphabets would fit in 5x5. Another person discusses possible tricks to make certain letters more clear, like changing how “M” or “W” are drawn. Some users are surprised at how well the font works on real screens, saying it looks better than they expected. A few share links to their own tiny font experiments, or to games and art made with small pixel fonts. There’s talk about the limits of legibility, with most people agreeing 5x5 is the best balance. Finally, some readers thank the author for sharing the font and the clear write-up, saying it will help with their own hardware projects.

## The Illuminated Man: an unconventional portrait of JG Ballard

- 原文链接: [The Illuminated Man: an unconventional portrait of JG Ballard](https://www.theguardian.com/books/2026/apr/20/the-illuminated-man-by-christopher-priest-and-nina-allan-review-an-unconventional-portrait-of-jg-ballard)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47868832)

This article reviews "The Illuminated Man," a new biography of JG Ballard, written first by Christopher Priest and finished by Nina Allan after Priest became ill and died. The book is not a normal biography; instead, it mixes Ballard’s life with the story of Priest’s sickness and Allan’s grief.

Ballard was a famous writer shaped by his childhood in Shanghai, living through a war camp, and losing his wife young. Much of his work shows strange, strong images from his past. Ballard did not like others writing about him and made his own short, careful memoir. The new biography tries to show how Ballard’s choice to write science fiction kept him from being fully accepted as a great writer, even though his ideas were unique. Instead of writing about space, Ballard wrote about the mind, calling it “inner space.” His stories almost always have the same themes: empty cities, abandoned malls, and strange, quiet people who just accept their fate. Many fans and critics say his work predicts the world we live in now, full of chaos and confusion.

When Ballard wrote "Crash," a book about sex and car accidents, people were shocked. Some called him a genius, others said he was too strange or even disturbing. Even though he became more known after "Empire of the Sun," he never became a part of the usual literary world. Ballard lived a quiet life, wrote by hand, and didn’t care much about money. Priest, who started the biography, argues that after "Empire of the Sun," Ballard lost some of his creative energy. Sadly, Priest died before finishing the book, so Allan completed it, mainly by including interviews and her own memories. This shift makes the book less focused on Ballard and more about Priest’s last days, which some readers might find confusing.

From the Hacker News comments, many readers respect Ballard for changing science fiction and making it deeper. Some say his books are hard to read but worth it, while others think his style is too strange and dark. People debate whether it’s fair to call him a “prophet” of today’s problems, or if his books just show the world as always a little broken. A few worry that the book’s focus on the authors’ personal lives takes away from Ballard’s story. Others like this new approach, saying it fits Ballard’s own style of mixing reality and imagination. Comments also discuss how Ballard avoided computers and technology, which feels odd for a sci-fi writer. Some wish the biography had more details about Ballard’s less-known stories. Overall, readers seem divided but agree Ballard’s work is powerful, strange, and impossible to copy.

## Website streamed live directly from a model

- 原文链接: [Website streamed live directly from a model](https://flipbook.page/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47867048)

Flipbook is a new website that creates every page you see as a fresh image, built in real time by an AI model. You don’t get normal web pages with text and links—instead, the whole web is just made of pictures that you can click on to explore something deeper.

All the words you see are part of the images, not real text. So sometimes, the letters look strange or are not in the right place. The creators say this will get better as the technology improves. The information shown on each page comes from both web searches and the AI’s own knowledge. The facts might not always be perfect, but the creators think it’s about as accurate as popular AI chatbots like ChatGPT or Gemini.

The team behind Flipbook says they started this because they wanted a different way to use computers. Most websites are just text or simple boxes, but they wanted to use pictures and images that can show ideas better. With Flipbook, anything you see could be a photo, drawing, word, or animation—whatever the AI thinks is best for the idea.

There’s also an experimental live video stream feature. Instead of jumping from one image to the next, you can turn on a mode that makes the pages animate and flow together like a video. This needs a lot of computer power, so it isn’t always on, and sometimes it acts strangely.

Flipbook is still an experiment. In the future, it might become more interactive, let you do real tasks, and even remember what you’ve done. The team imagines a world where you could research, plan, and book a whole trip inside Flipbook, with no need for other apps or websites. They hope this could make the web as visual and rich as the real world.

A few commenters were curious about the technology behind it. They wondered how expensive it is to generate all these images and videos on demand. Others pointed out that the images are not always accurate or readable, especially when it comes to text. Some people said this style might be fun for browsing and exploring, but not for serious work or reading long articles.

There were also comments about the future. Some hoped the technology would improve so the images and animations are smoother and more useful. Others thought this could be a new way to make creative websites or art, but maybe not replace normal web pages.
