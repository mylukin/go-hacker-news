## Building and Shipping Mac and iOS Apps Without Ever Opening Xcode

- 原文链接: [Building and Shipping Mac and iOS Apps Without Ever Opening Xcode](https://scottwillsey.com/building-and-shipping-mac-and-ios-apps-without-ever-opening-xcode/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48896665)

This article explains how you can build and ship macOS and iOS apps without ever opening the Xcode app. You still need to install Xcode, but you can use command-line tools and scripts for all the main tasks.

First, you set up Xcode and do some one-time steps in the GUI, like signing in to your Apple Developer account, creating certificates, and storing a special password for notarization. After this, you do not need the Xcode app for regular development or releases. The key command-line tools are `xcodebuild` (for building), `notarytool` (for notarization), `stapler` (for attaching notarization), and `devicectl` (for installing to devices). You also use `xcodegen` to make and manage the Xcode project from a YAML file, which helps keep your git repo clean.

The article suggests writing a `release.sh` script. This script handles the full release process: archiving, signing, notarizing, stapling, and installing your app. If any step fails, the script stops and shows an error. This way, you always know if something is wrong.

For code signing, the private key sits safely in your login keychain, and the script finds it automatically. Notarization is a separate step, and your password for it is stored in a profile, not in your code. This keeps secrets out of your git history.

To test your app fast, you can build unsigned versions with `CODE_SIGNING_ALLOWED=NO`. For shipping to users, you run the release script, which does everything needed for a real, signed, notarized app.

For iOS, you use `xcodebuild` and `devicectl` to put the app on your iPhone, also without opening Xcode. The only thing you need the GUI for is the first-time setup of credentials.

The article also shows how large language models like Claude Code can help by writing or updating the scripts for you. Once your scripts are ready, you can run everything from the terminal, and even let an AI agent handle it for you.

In the comments, some users say this workflow saves time and helps avoid Xcode bugs or crashes. Others like that it makes builds more repeatable and easier to automate in CI systems. A few say they prefer the Xcode GUI for debugging or viewing logs, but agree that scripting releases is better for automation. Some mention that the one-time setup is still confusing, especially around certificates and provisioning profiles. People also discuss that while this method is great for advanced users, beginners might find the command line steps hard. A few warn that Apple sometimes changes tools or requirements, so scripts may break after Xcode updates. Others share tips about using similar workflows for other platforms, or suggest even more automation with additional scripts or tools. One commenter likes that secrets are never put in git, while another worries about backing up the private key safely. Overall, readers agree that moving away from the Xcode GUI for routine builds makes life easier, but some think the first setup could be made even simpler.

## Apple's new SpeechAnalyzer API, benchmarked against Whisper and its predecessor

- 原文链接: [Apple's new SpeechAnalyzer API, benchmarked against Whisper and its predecessor](https://get-inscribe.com/blog/apple-speech-api-benchmark.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48894752)

Apple has released a new speech recognition API called SpeechAnalyzer, replacing its older SFSpeechRecognizer in iOS/macOS 26. The team at Inscribe compared this new engine against different versions of OpenAI’s Whisper model and Apple’s previous engine, using standard test data.

The main score used is Word Error Rate (WER), which measures how many words are wrong in the transcript. Lower WER means better transcription. On clean speech, SpeechAnalyzer scored a WER of 2.12%, beating Whisper Small (3.74%), Whisper Base (5.42%), Whisper Tiny (7.88%), and the old Apple engine SFSpeechRecognizer (9.02%). On noisier speech, SpeechAnalyzer also led with 4.56%, while Whisper Small had 7.95%, and SFSpeechRecognizer had 16.25%. The new engine was about three times faster than Whisper Small, and it gave transcripts with proper punctuation and casing. The team recommends everyone using SFSpeechRecognizer to switch to SpeechAnalyzer because it is much more accurate and reliable, especially for longer transcriptions like meetings.

Whisper, however, still has two big advantages: it works in more than 100 languages (while SpeechAnalyzer supports about 30), and it can run on more platforms, not just Apple devices. For English on current Macs and iPhones, SpeechAnalyzer is now the best option for on-device transcription. The test was done using Inscribe’s own product, which now uses SpeechAnalyzer for supported languages and Whisper for everything else.

All engines ran faster than real time, so you can transcribe an hour of audio in just a few minutes. The accuracy comparison was fair: they used the same code, the same audio, and the same way to judge results. The raw transcripts and evaluation process are public, so anyone can check or repeat the tests.

Some limits of the benchmark are that it only measures English, only uses clear audiobook speech, and only runs on one Apple chip (M2 Pro). The Whisper models tested were converted for Apple devices, which may be a bit different from the original versions.

From the Hacker News comments, many people were impressed by Apple’s accuracy and speed, with some saying this could make on-device privacy much easier. Some are happy that Apple is catching up or passing OpenAI in speech recognition for English. Others pointed out that Whisper’s multi-language support and cross-platform use are still very important, especially for apps that need to run on Windows, Linux, or Android.

A few users wondered about real-world performance, like in meetings with overlapping speakers or heavy accents, since the test only used audiobook speech. Some developers appreciated that Inscribe published all their data and methods, making the benchmark more trustworthy. Others warned that Apple’s APIs sometimes change quickly or are not as easy to use as open-source tools like Whisper.

Some people asked if Apple will open up its model to other platforms, but most agreed this is unlikely. There were also questions about battery life and resource usage, since faster models can sometimes use more power. Finally, some commenters said they will wait for more tests on conversational or noisy audio before switching engines, but most agreed this is a big step forward for on-device speech recognition on Apple hardware.

## Telegram's t.me domain has been suspended

- 原文链接: [Telegram's t.me domain has been suspended](https://www.whois.com/whois/t.me)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48897878)

Telegram’s short link domain, t.me, has been suspended and is now showing a “serverHold” status on its Whois record. This means the domain is not working on the internet, so Telegram links using t.me are broken for now.

The Whois data shows that t.me is registered with GoDaddy, and the “serverHold” status is set by the registry. This usually happens when there’s a legal issue, a policy problem, or a request from authorities. Other statuses like “clientDeleteProhibited” and “serverTransferProhibited” mean the domain cannot be deleted or moved while this is happening. The domain is not expired; it’s paid up until 2035. The actual owner’s info is hidden behind a privacy service, which is normal for big domains.

Many users on Hacker News are surprised that a big service like Telegram can have its main link domain taken down so suddenly. Some people worry about how much power registrars and governments have over important websites. Others say this shows why decentralized systems are needed, so no one group can take down a whole service.

One commenter points out that millions of links are now broken, which hurts both Telegram and its users. Another says companies should use their own main domains, not short links on country-code domains like .me, which can be risky. Some people ask if Telegram did something wrong, like breaking laws or rules, but no one seems sure yet.

A few users think this is a lesson for developers: don’t depend on a single domain, and always have backups. Some believe the domain will come back soon, while others worry it could be permanent. There are also concerns about how this affects people in countries where Telegram is one of the few free ways to talk. Overall, the comments show people are worried about internet control, link stability, and what this means for Telegram’s future.

But running the new site is hard. It depends on donations, not government money, to stay online. The article says this is not ideal, since taxes should support important public resources like this. Still, the team’s work helps everyone make good decisions about climate, and the site is like a form of journalism itself.

## The real prices of frontier models. Tokens * Price, right?

- 原文链接: [The real prices of frontier models. Tokens * Price, right?](https://playcode.io/blog/real-price-of-frontier-models)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48896800)

This article looks at the real cost of using different AI models, like GPT from OpenAI and Claude from Anthropic, when you use them for coding tasks. It explains that even if two models say they cost the same per "token," the way they count tokens can be very different, which changes what you actually pay.

The main point is: a "token" is not the same size across models. Each model has its own way to split up your text into tokens, and you pay for each token. For example, the same TypeScript file becomes 681 tokens on GPT-5.x but 1,178 tokens on Claude’s newest model—almost 1.73 times more. So, even if the price per token is the same, you could pay much more with Claude if you use a lot of code.

The article shows this is not just a guess: the author tested real files (like HTML, Python, and JSON) on each model’s official token counter. They found that Anthropic’s new tokenizer produces about 30% more tokens than their old one, even though the sticker price did not change. This is like a hidden price increase.

On code, the gap is biggest. Claude’s new tokenizer can make 1.5 to 1.73 times the tokens for the same code compared to GPT. For TypeScript, it is the worst: 1.73 times more tokens. That means if you use Claude for coding, you could pay up to 73% more than with GPT, just because of how it counts tokens.

The article also explains that Claude’s tokenizer did not get better for Chinese text; it still uses more tokens than GPT. But for code and English, the new tokenizer means higher bills. The author checked actual invoices and found the numbers match what the token counters say.

There are tables showing how much more you pay in practice. For example, after August 2026, the same code will cost about 32% more on Claude Sonnet 5 than before, even at the same list price. Other models like Gemini and Grok are compared too, but their tokenizers are closer to GPT’s.

The article says you should not just look at the sticker price per token, but check how your own content gets counted. If a model changes its tokenizer, it can feel like a surprise price hike. For coding agents (AI that writes code), this matters a lot, because code is where the difference is biggest.

From the comments, many people were surprised to learn that token counts can vary so much and that the real cost depends on what you feed the model. Some users said they had been caught off guard by higher-than-expected bills, especially after a model update. Others pointed out this is similar to “shrinkflation” in food, where you pay the same for less product.

A few commenters defended the companies, saying that improving tokenizers is hard and sometimes needed for better performance. Some think vendors should be clearer and publish real tokenization examples on their pricing pages. Others suggested always testing with real data before picking a model.

Some users said they now understand why their costs changed, and a few shared their own numbers, confirming the big differences. There was debate about whether this practice is misleading or just a technical detail, but most agreed it’s something developers need to watch closely.

Others talked about the trade-off between model quality and price, saying sometimes it’s worth paying more for better results. But everyone agreed that comparing models by just the price per token is not enough—you have to look at the full picture, including how your content is split into tokens.

## Linux 0.11 rewritten in idiomatic Rust, boots in QEMU

- 原文链接: [Linux 0.11 rewritten in idiomatic Rust, boots in QEMU](https://github.com/Poseidon-fan/linux-0.11-rs)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48898134)

This project is about rewriting the old Linux 0.11 kernel in modern Rust. The new version is called linux-0.11-rs and it can boot on i386 hardware in QEMU, just like the original from 1991.

The main goal is to keep the old Linux system working as before but rewrite it using safe and clear Rust code. The new kernel uses strong types and better code structure, making it easier to understand and safer to use. It supports features like processes, virtual memory, the Minix v1 file system, disk drivers, VGA and serial console, TTY, signals, and a full set of system calls.

There is also a Rust library for user programs, so you can write user software in Rust and it looks just like normal Rust code. The project comes with over 80 command-line tools and a shell that supports things like pipelines, loops, globbing, history, and tab completion. You can build a ready-to-boot image with one command, and there are tools to help with disk images and testing. The repo is organized into clear parts: kernel, user library, core utilities, test runner, and more. There’s even a developer container setup for easy development, and tests that boot the kernel and check if everything works.

The project is almost feature-complete compared to Linux 0.11, except for things like floppy disk support. The team plans to improve documentation and add more guides in the future.

Commenters on Hacker News had many thoughts. Some people are very impressed by how much of Linux 0.11 was rewritten in Rust and how it all works, including the userland tools and shell. Many say this is a great way to learn both Rust and how operating systems work. A few think rewriting old code like this helps make systems safer and easier to maintain. Others warn that writing an OS in Rust is still hard, especially with low-level hardware parts, and not all bugs are solved by using Rust.

Some users ask about performance and if the new kernel is as fast as the original. A few are curious about the real-world uses, since this is an old version of Linux. Others mention that the project can be useful for education and for people who want to see how a real OS works from the inside. There are also thanks and praise for the clear organization of the repo and the good documentation. Some people suggest trying to port more modern Linux features or using the project as a base for learning about kernels. Overall, the comments show both excitement for the use of Rust and respect for the effort involved.

## Show HN: YouTube Guitar Tab Parser

- 原文链接: [Show HN: YouTube Guitar Tab Parser](https://github.com/marcelpanse/youtube-guitar-tab-parser)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48898154)

This article talks about a tool that changes YouTube guitar lesson videos into PDF files with guitar tabs. The tool is a command-line app, and you only need the video link to use it.

First, the tool downloads the video using yt-dlp. Then, it takes screenshots from the video every few seconds with ffmpeg. It uses Claude vision, an AI, to find where the guitar tab is in each screenshot. The tool crops the images, so only the tab part stays. It checks these crops to remove near-identical ones, using a bar number printed in the music. This way, it only keeps one tab image for each bar in the song. The tool then puts all the unique tab images together into a PDF file. The PDF gets the video’s title as its name and heading. 

To use the tool, you need Node.js, yt-dlp, ffmpeg, and an Anthropic API key. The setup steps are simple, and you can just run a command with the YouTube link. There are some options, like changing how often screenshots are taken or which AI model to use, but the defaults work for most people. The process is automatic, and you don’t need to change any settings unless you want to.

In the comments, many people think this tool is very useful, especially for guitar players who want to save tabs from video lessons. Some users are impressed by how well the tool finds and cuts out the tab parts without extra work. Others like that it uses AI, but a few worry about needing an Anthropic API key, which might cost money or have limits. Some say it would be nice to support other types of music notation, not just guitar tabs. A few mention that the process could be slow, especially for long videos or if the AI is busy. Others suggest adding features, like making the PDF easier to read on phones or letting users pick which sections to save. There are also comments about open-source tools and how this one fits well for people who want to learn songs faster. Some users hope for future updates, like support for other instruments or better performance. Overall, people like the idea and see lots of ways it can help musicians.

## Linux on the Sega 32X. Who needs hardware synchronization primitives anyway?

- 原文链接: [Linux on the Sega 32X. Who needs hardware synchronization primitives anyway?](https://cakehonolulu.github.io/linux-on-32x/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48896600)

This article tells how the author got Linux running on the Sega 32X, an old add-on for the Sega Genesis game console. The project is part of the author’s journey to learn more about bringing up hardware boards and running modern software on old, unusual hardware.

The author explains that the Sega 32X is much more powerful than the Genesis, with two 32-bit CPUs and extra RAM. Still, it’s very limited compared to modern computers. To get Linux running, the author used a special flash cartridge that lets the console use more memory, turning some ROM into writable RAM. This made it possible to load and run the Linux kernel, even with big memory limits.

There were many technical problems. The toolchain (the program that builds Linux for the Sega CPUs) crashed on some files, but the author found workarounds. Some parts of Linux’s code didn’t work with the Sega’s CPUs and had to be fixed—especially parts that managed math and memory. Other problems came from missing hardware support, so the author wrote new drivers to make things work, like sending text over a serial port.

One of the hardest parts was getting Linux to use both CPUs at the same time (SMP). The Sega 32X’s CPUs have no hardware to help them share memory safely. The author solved this with a software method called Peterson’s algorithm, which lets two CPUs take turns safely, but is slow. The author also had to make the old Genesis CPU act as a helper for communication between the two newer CPUs. Cache (fast memory) had to be turned off to stop bugs from using old data. Finally, the author added a simple way to see Linux’s output on the Sega’s screen. Performance is slow, but everything works.

In the comment section, many readers were impressed with the dedication and technical skills needed for such a project. Some said this kind of work is both fun and a great way to learn about old hardware and Linux internals. Others pointed out just how hard it is to run modern software on old game consoles, especially with strange hardware like the 32X’s dual CPUs and lack of standard memory features.

A few commenters remembered their own tries to run Linux or other operating systems on retro consoles. They shared similar stories about fighting with odd toolchains, missing documentation, and tricky hardware bugs. Some were surprised that software-only synchronization (like Peterson’s algorithm) could actually work, even if it’s slow. One person asked if the method would work with more than two CPUs—others explained that Peterson’s only works for two, and more CPUs would need something else.

Another topic was the usefulness of projects like this. Some thought it’s not practical, but very cool for learning and hacking. A few wondered if the author’s knowledge could be used for jobs in embedded systems or hardware bringup, which is a rare skill. Lastly, several people thanked the author for open-sourcing their code and sharing all the configs, making it easier for others to try running Linux on their own Sega 32X systems.
