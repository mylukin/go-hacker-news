First, let’s talk about a big update for the Elixir programming language. Elixir just released version 1.20, and now it can check types in your code step by step, even if you don’t write any type annotations. This means Elixir can find more bugs and dead code before you ever run your program. The new type system is “gradual” and uses a special type called dynamic. This lets Elixir check types without being too strict. For example, if a variable could be an integer or a string, Elixir won’t complain unless you do something impossible, like treating both as a map. Elixir only warns you if it is sure something will crash at runtime.

This type checking is based on set theory, using things like unions, intersections, and negations. It understands guards and pattern matching, so it can catch mistakes such as missing fields in maps or using the wrong data shape. The dynamic type is not like any in other languages; it does not just say “anything goes.” Instead, Elixir tries to follow how the type changes as you use it, helping to find real bugs but not flood you with false alarms.

This release also made Elixir faster to compile, especially on computers with many cores, and added options to help big projects build even quicker. For the future, Elixir wants to add advanced types, like recursive and parametric types, but only if they are fast and easy to use.

From the Hacker News comments, many people are excited that Elixir is getting types, saying it helps catch bugs early and makes code safer. Some users are surprised by the lack of extra annotations, and they like that Elixir only warns about real, confirmed bugs. Others are worried it might make Elixir less simple or slower, but the update actually speeds up compile times. A few note that gradual typing is tricky and can sometimes miss bugs, but most agree that reporting only “verified bugs” is a good balance. Many compare Elixir’s system to TypeScript or Python, and they appreciate its precision and low rate of false positives. Some want to write their own type annotations, others are happy it works automatically. There are questions about how this affects existing code, and people are glad the changes don’t break old projects. Most comments are positive, calling this a big step for Elixir and saying it might attract more developers.

Next, Google DeepMind has released Gemma 4 12B, a new AI model that understands images, audio, and text, and can run on a normal laptop. The special part is that it does not use separate encoders for vision or audio—everything goes straight into the main language model. This mid-size model is strong but uses less memory, and you can run it on a laptop with just 16GB of RAM. It is the first Gemma model of this size to handle audio directly, making it useful for voice and sound tasks.

Gemma 4 12B’s architecture is simple: images are quickly transformed into data for the model, and audio is put into the model as if it were text, skipping extra encoders. This design makes it faster and more efficient. The model can do hard reasoning almost as well as Google’s much larger 26B model and can predict several words at once, making answers faster. Gemma 4 12B is open source, free to use, and works with many developer tools. There are guides, sample apps, and a “Skills Repository” to help build AI agents.

In the comments, people praise the encoder-free design for saving memory and making the model easy to run on a laptop. Some wonder if skipping encoders might miss details in images or sounds. Others compare Gemma 4 12B to Meta’s Llama or OpenAI’s models, asking which is better for real use. Developers like the open source license and easy tool support, while some worry about quality in vision and audio tasks. The Apache 2.0 license is popular for allowing commercial use. Some have already tried the model on their laptops and are happy with the results, but want more real-world benchmarks. Overall, there’s excitement and some careful questions about how this new design works in practice.

Next up, DaVinci Resolve 21 has been released. This is a popular video and photo editing software, now with even more AI tools and features. The biggest new thing is a Photo page, letting users edit still photos with the same strong color tools used in movies. There are new AI tools for searching people or objects, changing age or shape of faces, removing skin blemishes, sharpening images, and removing motion blur. AI can also read clapperboards to help organize files, and there is a speech generator to create voiceovers, even copying your own voice with a short sample.

Video editing is easier, with improved keyframing, support for more graphic formats, and simple ways to animate graphics. Lottie and HTML graphics are now supported, and text tools are better with spell check, emojis, and new styles. Color grading is stronger, with options for HDR and SDR, and new tools for complicated workflows. For special effects, Fusion adds over 70 graphics and faster macro building. Audio editing in Fairlight is also improved, with better management and grouping.

DaVinci Resolve 21 adds VR support, new upload options, and works well on Apple M-series and other modern chips, supporting 8K real-time editing. Hardware panels are available for professionals, but the main software remains free for most users, with the Studio version adding more AI tools.

In the comments, users are impressed by the free features, especially compared to Adobe or Final Cut. Many like the new AI tools, but some worry about deepfake features being misused. Some say the software is getting complex, making it harder for beginners, but power-users are happy with more options. DaVinci Resolve is now seen as a strong choice for both video and photo editing. Some users wish for better Linux support and more open formats, but most agree Blackmagic is pushing the industry forward.

Next, Uber is now limiting how much employees can spend on AI code tools like Claude Code, capping it at $1,500 per month per tool. The article’s author thinks this limit is smart for controlling costs. Another story is about how someone hacked a Creative Sound Blaster Katana V2X speaker using only Bluetooth, without touching the device. The speaker is usually connected to a PC over USB but can also be controlled by phone over Bluetooth. The author found that anyone nearby can connect to the speaker by Bluetooth and send commands, even upload new firmware. This means an attacker could make the speaker act as a keyboard and type commands on your PC, just like a “Rubber Ducky” attack. Creative, the company, did not see this as a security problem and has not fixed it. The author made their own patch, but it breaks the mobile app.

Comments are full of surprise and worry, as many did not expect a speaker could be used to attack a PC. Some note that Bluetooth devices often have weak security, and others are frustrated that Creative does not care about the risk. There are jokes about “smart” devices making new problems, and some people ask if similar attacks could work on other devices. Many thank the author for sharing technical details and making a patch.

Now let’s look at Gooey, a new UI framework for building fast, GPU-powered apps in Zig. Gooey works on macOS and Linux, with the web version planned for the future. It is both immediate and retained mode, so you can easily update the UI and keep data in sync. The API is declarative, and it comes with many built-in components. You get themes, accessibility, and support for international text. Gooey uses Zig’s async features to run background tasks without blocking the UI.

On Linux, it uses Wayland and Vulkan for speed, and on macOS, you get native effects. The web version is waiting for Zig support to return. In the comments, users are excited that Zig now has a modern UI toolkit, and they praise the careful design and documentation. Some compare it to other frameworks in Rust, while others like the zero-dependency approach and accessibility features. There are questions about performance, memory use, and future Windows support. Many thank the author for open-sourcing the project and hope to see it grow.

JPEG XL is our next story—a new image format built from many open-source experiments. For years, JPEG worked well, but new screens and cameras need better quality and smaller files. The team first worked to improve older formats with projects like WebP Lossless, Brotli, Guetzli, and Brunsli. They learned from these, then combined their best tools with another project to make JPEG XL. JPEG XL can compress images more and keep high quality, using tricks like VarDCT. It is used in digital cameras, medical images, and documents. Big companies support JPEG XL, and some browsers like Safari already use it, while others are still testing.

In the comments, people are excited about JPEG XL’s quality and small size, and like that it’s open-source. Some are worried about slow adoption in browsers like Chrome and Firefox. Others say it can be risky to switch image formats and hope for long-term support. There is discussion about JPEG XL versus AVIF, with some saying JPEG XL is faster and better quality, but AVIF is supported in more browsers. Many hope JPEG XL will become popular, but worry about slow adoption.

Finally, Ableton has released a new Extensions SDK for Live 12.4.5. This lets users build their own tools inside Ableton Live, using JavaScript and Node.js. Extensions can automate tasks, edit tracks and MIDI, connect to outside services, or even play games. Extensions are different from Max for Live—they focus on workflow and data, not sound. You need the Live 12 Suite Beta and Node.js to use them, and Extensions run once and then stop. You don’t need to be a professional developer; even AI tools can help you build Extensions.

In the comments, people are excited about customizing Ableton Live, and like that Extensions use JavaScript and common web tools. Some compare Extensions to Max for Live and wonder if they will work together or compete. A few mention that needing the Suite Beta blocks some users for now. Others ask about security, since running custom code could cause bugs. Many hope Extensions will let people share tools more easily, and wish the SDK was available for all Live versions. Most agree this is a big step for making Ableton more open and flexible.
