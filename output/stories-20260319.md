- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47442690)

Google is adding a 24-hour waiting period for people who want to sideload unverified Android apps. This new rule is meant to stop malware and scams, and it will start in some countries in September 2026. To install apps from unverified developers, you must turn on a special setting deep inside developer options, confirm you are not being forced, unlock your phone, restart, and then wait 24 hours before you can continue. After the wait, you can choose to allow unverified apps for a week or forever, but you must accept more warnings about the risks.

Google says this delay is because scammers often pressure people to install bad apps quickly. By making you wait, Google hopes people will have time to notice the scam. The company says it is not checking the content of apps, only who made them. Developers who want to be verified need to show ID, give signing keys, and pay a $25 fee. If a verified developer does bad things, they can lose their verified status. Google also says that if you want to keep sideloading freely, you only need to set up the “allow indefinitely” option once per device.

Some people worry that this system could hurt independent developers, or that it might be hard for people in some countries to pay the fee. There are also questions about privacy, since Google will have a list of developer identities. Google says it will protect this data, but did not give full details.

In the Hacker News comments, some users say the new process is very hard to find and takes too long, comparing it to hiding important information in a locked filing cabinet. Others feel the steps are annoying and will stop regular users from installing what they want. Some developers are worried that this is another step toward making Android more closed, like Apple’s iOS. A few people support the change, saying it will protect less technical users from scams and malware. There are concerns about how this will affect people in countries with fewer resources, since paying even a small fee can be hard. Some users point out that power users will still be able to sideload if they really want, but the process is now much less friendly. Other comments ask if this is just Google trying to control more of the Android world, not just helping security. Some people appreciate that Google is reacting to real problems, but many wish there was a simpler way to stay safe without so many limits.

## Noq: n0's new QUIC implementation in Rust

- 原文链接: [Noq: n0's new QUIC implementation in Rust](https://www.iroh.computer/blog/noq-announcement)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47443588)

noq is a new open-source QUIC implementation written in Rust, created by the n0 team. It was made after the team realized they needed more control than what the popular Quinn library could give for their iroh project, especially for features like multipath and NAT traversal.

The article explains how the team first tried to extend Quinn, but their needs grew too different. They wanted QUIC to handle things like using many network paths (for example, direct or relay connections) and NAT traversal directly, not as hacks. So, they started their own codebase, but still want to work with Quinn where possible.

noq’s main feature is full support for QUIC Multipath, which lets one connection use several network paths at the same time. This helps improve speed and reliability, because if one path is slow or broken, others can take over. noq also supports QUIC NAT traversal, making it easier for devices behind firewalls or routers to connect. Their NAT work is based on a new draft spec and is already tested on many devices.

Another feature is QUIC Address Discovery (QAD), which helps devices learn their public IP addresses using QUIC, not older tools like STUN. This is more private and secure. noq also adds better support for Qlog, which is a tool for logging and visualizing QUIC network activity. This helps developers debug connections, especially when using multiple paths.

For developers, noq introduces a WeakConnectionHandle type, which helps manage active connections more efficiently in code. noq has already been used in production as part of iroh since version 0.96 and has passed tests with other QUIC implementations, showing it's stable.

From the comments, many developers are excited to see a strong, open-source QUIC implementation in Rust, especially with multipath and NAT traversal. Some people are curious how noq compares to Quinn, and appreciate that the teams still talk and share ideas. A few ask about performance, memory use, and if noq could replace Quinn for general use. Others wonder about security and how well noq handles real-world network problems.

Some commenters like that noq is tested in production and with reference implementations, which gives them more trust. There are also questions about when the QUIC Multipath spec will be fully finished and how often noq will update as the spec changes. A few people point out that NAT traversal is very hard to get right, so they hope the team shares their learnings. Others are glad to see more tools for debugging QUIC connections, like the improved Qlog support.

Finally, some developers ask about using noq for peer-to-peer projects or in their own custom network stacks. The overall feeling is positive, with people keen to see how noq grows and hoping for more documentation and examples.

## From Oscilloscope to Wireshark: A UDP Story (2022)

- 原文链接: [From Oscilloscope to Wireshark: A UDP Story (2022)](https://www.mattkeeter.com/blog/2022-08-11-udp/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47444460)

This article is about looking at network data all the way from raw electrical signals (using an oscilloscope) up to seeing real UDP packets (using Wireshark). The author works at Oxide Computer Company and needed to debug a problem with their rack-scale computer’s management network.

To start, the author connects an oscilloscope to the network switch’s hardware. This lets them see the voltage changes on the wire. They record this raw data and move it to their computer. The data is huge, so they only capture a tiny slice, expecting to catch a few UDP packets. To read the oscilloscope’s file, they write a small program that extracts the important waveform data.

Next, the author explains the kind of network link they are looking at: QSGMII. This link combines four network channels into one, using a fast serial protocol. The signals are encoded using 8b/10b encoding, which helps keep the signal reliable. To decode the data, the author searches for special patterns called “comma characters” to find the start of each group of bits. They use both Rust and Python to process the data, finding where the bits change, and lining up the code-groups.

After decoding, they separate out the four channels. Each channel can then be turned into a stream of network packets. The author explains how to find the start and end of packets and how some bytes are repeated to match the slower network’s speed. When they remove the repeats, they see the familiar ethernet frame, starting with the “preamble.” They then extract the real data, like the MAC addresses and IPv6 headers.

Finally, the author writes the packets to a file format called pcap, so they can open it in Wireshark. They confirm that they are really seeing the UDP packets they expected. The whole process—from voltage waves to actual network packets—takes less than a second on their computer.

In the comments, many people are impressed by how deep the author went, from hardware signals all the way to network packets. Some say this is a great example of debugging and understanding every layer. Others point out that most engineers never need to go this deep, but it’s helpful for hardware or firmware developers. A few comment on the clever tricks used for decoding, especially handling the clock mismatches and finding the repeating patterns. Some share their own stories of using oscilloscopes or logic analyzers to debug network problems. Others suggest using existing tools, but agree that writing your own can teach you a lot. ## NanoGPT Slowrun: 10x Data Efficiency with Infinite Compute

- 原文链接: [NanoGPT Slowrun: 10x Data Efficiency with Infinite Compute](https://qlabs.sh/10x)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47444072)

This article talks about making language models much more data-efficient using NanoGPT Slowrun. The team trained a group of large models ("ensemble") and got the same results with only 100 million data points that usually need 1 billion.

They used several tricks: heavy regularization, looping layers in the model, new activation functions, and joining models together. Ensembling means training several models at once and combining their answers. This helps models learn better from the same data, especially when you train each one for many rounds (epochs). They also used "chain distillation"—each model in the group learns from the last one, which helps the whole group perform better.

Regularization (like strong weight decay and dropout) stops the models from memorizing the data and instead helps them generalize. They used much stronger regularization than normal because their models were much bigger than usual for the data size.

Looping means some layers in the transformer run several times before finishing, which helps the model think harder about each input. They also made architecture changes: removing some attention parts, using U-Net connections, changing activation functions to SwiGLU, and changing how embeddings work. These tweaks all gave small but important improvements.

The main point: these changes together made the model use data 10 times more efficiently than normal, which is a huge jump. The team thinks this approach can go even further, maybe 100 times, with more work.

In the comments, people are excited about the 10x efficiency but ask if this could work on bigger, real-world datasets. Some wonder if the heavy regularization and looping would help or hurt in other settings. Others are interested in ensembling and chain distillation, saying these tricks are not used enough in big AI research.

A few readers point out that using many models at once (ensembles) is expensive during usage, not just training. Some wish the paper compared their results directly with other common tricks, like data augmentation. There is also talk about whether this makes models more “robust” or just better on test scores.

Some commenters feel the paper is a bit hard to follow, with lots of small changes instead of one big idea. Others say it’s exciting to see people try new things instead of just making models bigger. Finally, many agree this could help smaller research groups, since it means you can do more with less data.

## EsoLang-Bench: Evaluating Genuine Reasoning in LLMs via Esoteric Languages

- 原文链接: [EsoLang-Bench: Evaluating Genuine Reasoning in LLMs via Esoteric Languages](https://esolang-bench.vercel.app/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47446021)

This article talks about testing AI coding models on strange, rare languages called “esolangs,” like Brainfuck and Whitespace, instead of popular ones like Python. The authors made a new benchmark with 80 problems in five esoteric languages, where there is almost no training data for the models.

The main point is that models do very well on normal languages because they have seen lots of those examples during training. But when tested on these weird languages, their performance drops from about 90% to less than 4%. In fact, the best model—GPT-5.2—only gets 4.2% right, and nobody can solve problems in Whitespace at all. If the problem is harder than “easy,” all models fail with a score of zero. Giving the AI more hints or examples (few-shot prompting) does not help. Even letting the model see its own mistakes and try again (self-reflection) does not work. The only thing that helps a bit is letting the AI call an interpreter tool to test its code after each try, but even then, the scores stay very low.

Each language causes different kinds of mistakes: for example, in Brainfuck, the code runs but gives the wrong answer; in Unlambda, the code often does not even compile. Models seem to know the basic commands, but cannot reason through complex problems. The dataset has problems from very simple (“Hello World”) to extra hard, but only the easiest ones see any success.

In the comment section, some people say this is a very fair way to test if AIs really understand programming, instead of just copying patterns from their training data. Others point out that almost nobody uses these languages, so maybe it’s not practical to expect models to do well on them. A few commenters say that if a model can solve esolang problems, it would be strong proof of real reasoning skills. Some worry that AI companies will just start adding esolang code to their training data to get better scores. Others think that if the AI can learn to use tools (like interpreters) better, it might bridge the gap. There are also jokes about how hard and strange these languages are for humans too, and some users share stories about trying to write Brainfuck or Whitespace code themselves. Finally, a few say this shows that AI is still far from true understanding, especially when faced with truly new or weird problems.
