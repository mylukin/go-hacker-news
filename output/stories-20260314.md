## Show HN: Han – A Korean programming language written in Rust

- 原文链接: [Show HN: Han – A Korean programming language written in Rust](https://github.com/xodn348/han)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47381382)

This project is about "Han," a new compiled programming language that uses Korean (Hangul) keywords, and it is written in Rust. Han is made for people who want to write code using the Korean language, and it can run programs either by compiling to a native binary or directly using an interpreter.

Han uses Korean words for common programming ideas like function (`함수`), variable (`변수`), loop (`반복`), conditional (`만약`), and print (`출력`). The language supports five basic types: integer, float, string, boolean, and void. You can use arrays, structs, pattern matching, closures, error handling, file input/output, and more. The language has both a compiler (making fast native programs) and an interpreter (for quick testing). There is also a REPL for interactive coding and an LSP server for editor help.

The documentation gives many simple examples, like counting words in a string, a small calculator, managing a to-do list with structs, and counting lines in a file. You install Han by cloning the GitHub repo and building it with Rust. Han can run your code with `hgl interpret`, compile to binary with `hgl build`, and has other common command-line tools.

Han is not just a technical project; it is also about culture and language. The creator wanted to show that programming can use any language, not just English, and wanted to use Hangul because it is a clear and logical writing system. This is also helpful for people learning Korean, as they can practice by coding.

Some features are still missing or only partly work—for example, there are no enums, no tuple types, no async or network code, and some things like garbage collection are very basic. Still, Han is already useful for many tasks.

Commenters on Hacker News had mixed opinions. Some people thought Han was a fun and creative idea, and they liked seeing programming made more accessible for Korean speakers and learners. Others pointed out that code is often shared worldwide, and using a non-English language could make it harder for people to collaborate or find help. A few said it might be great for teaching, especially for kids or beginners who know Korean better than English.

There was discussion about the technical choices, like using Rust to build the compiler and generating LLVM IR as plain text instead of using the LLVM library. Some liked this approach because it keeps things simple, while others wondered about the limits and possible bugs. People also talked about whether using Korean keywords would really help learners or just make things more confusing when they later use other programming languages.

Some shared stories about other programming languages in native scripts (like Arabic, Chinese, or Hindi) and how they are used mostly for fun or education. A few people commented on the beauty and design of Hangul, agreeing it is a great script to use for something new like this. In general, most commenters respected the project for its creativity and effort, even if they would not use it themselves.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47380512)

This article talks about running Fedora 44 on the new Raspberry Pi 5. The writer made new Fedora images for the Pi and shared what works and what still needs fixing.

Right now, Fedora 44 works on all Raspberry Pi 5B models, including those with 1 to 16 GB of RAM. The system supports the serial console, micro SD slot (which is the only way to run the operating system for now), HDMI with fast graphics, wired and wireless internet, USB ports (but not for running the OS), and both KDE and GNOME desktop images. The writer says these features have been tested and work fine, making the Pi quite usable as a little computer. Some extra things are working too.

To get the graphics and other features working, users have to add a line (`cma=256M@0M-1024M`) to the kernel command manually during setup. If users forget this, graphics will not run well. The author wants to make this step automatic in the future. Users should also turn off auto-suspend on desktop images to avoid sleep problems.

The article gives download links for three kinds of Fedora 44 images: Minimal, KDE, and GNOME. The writer is happy to share these on Pi Day and promises more updates soon.

## MCP is dead; long live MCP

- 原文链接: [MCP is dead; long live MCP](https://chrlschn.dev/blog/2026/03/mcp-is-dead-long-live-mcp/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47380270)

This article talks about the Model Context Protocol (MCP), which was very popular in tech just a few months ago but is now being criticized, with many people saying Command Line Interfaces (CLIs) are better. The author explains that the hype around MCP was mostly driven by influencers and that both MCP and CLIs have their own strengths and weaknesses.

The main idea is that using CLIs can save "tokens" (which means less information needs to be sent to AI models), but only if you use popular tools that the AI already knows. For new or custom CLI tools, you still have to explain everything to the AI, which makes things complicated. The author shows that both MCP and CLIs require some setup, and the supposed "savings" from using CLIs are often not as big as people think.

The article says that MCP becomes much more useful for big companies and teams, especially when it is used over the internet (HTTP) instead of just on one computer. With MCP on a server, you get benefits like easier sharing of tools, better security (you can control who has access), and better tracking of which tools are used and how. This makes it easier for companies to keep everything organized and up to date. MCP also lets companies send the latest instructions and documents to everyone without having to update each computer one by one.

The author argues that most of the negative talk about MCP comes from people who are only thinking about small projects or personal use. For bigger organizations, MCP solves many problems, especially around security, updates, and making sure everyone has the right information. The article ends by saying that even though MCP is not perfect for every use case, it is still very important for teams that want to move from chaotic, unorganized coding to a more structured way of building with AI agents.

In the comments, some people agree with the author and say that the hype cycle in tech is real—first everyone loves a tool, then everyone hates it, but the truth is usually in the middle. Others point out that for small teams or solo developers, MCP feels like too much work, and simple CLIs are easier to use. Some readers mention that good documentation and clear instructions are important no matter which tool you use. There are also comments about security—many agree that centralizing secrets and access is safer, but it can also become a single point of failure if not managed well.

A few commenters debate whether the extra features of MCP (like telemetry and live updates) are really needed for most projects. Some are worried about vendor lock-in or having to trust a third-party server with sensitive data. Others like the idea of having up-to-date documentation always available, especially in large companies with many teams. In general, the discussion shows that while MCP has strong points for big organizations, using simple CLIs still makes sense for smaller projects or quick tasks. Many suggest that the best choice depends on the size of your team, your security needs, and how you like to work.

## A Recursive Algorithm to Render Signed Distance Fields

- 原文链接: [A Recursive Algorithm to Render Signed Distance Fields](https://pointersgonewild.com/2026-03-06-a-recursive-algorithm-to-render-signed-distance-fields/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47335796)

This article is about a new recursive algorithm for rendering 3D objects called Signed Distance Fields (SDFs). SDFs are a way to describe shapes using math functions, not polygons, and they're very popular for creating complex, flexible graphics.

The author explains that SDFs let you build objects by combining functions. For example, you can make shapes, twist them, or cut them out of each other with simple code. This is hard to do with normal polygon graphics. SDFs are used in the demoscene, where small programs create amazing graphics, like city scenes, with just a bit of code.

Usually, SDFs are rendered using a method called raymarching or sphere tracing. This is simpler to code than polygon rendering, but it's slow, especially if your scene is complex. It takes a lot of computer power because every pixel needs to check the SDF many times. This is why SDF rendering is mostly done on powerful GPUs.

The author tried to make SDF rendering faster on a regular CPU by using a recursive divide-and-conquer method. The idea is to split the view into smaller and smaller parts (quads). For each part, you see how far you can move forward before hitting something. If you get close to an object, you split that section again. This method can skip empty space more quickly.

The author made a test program in Rust and found that this method uses fewer rays per pixel and makes the frame rate three times faster. They also improved it by shading whole patches of pixels at once using interpolation. If a patch is flat and inside a shape, you can guess the colors for all pixels instead of computing each one. This makes things even faster, sometimes using less than one sample per pixel.

The author learned that a similar idea exists on the GPU, called "cone marching." This uses multiple passes to speed up SDF rendering, and it's not widely known but has been used in some demos and games.

With these methods, the author can render simple SDF scenes at 50 to 100 frames per second on one CPU core. This suggests that, with good code and a modern CPU, you could make simple 3D games with SDFs on the CPU alone, not just on a GPU.

In the comments, many readers are excited about the new recursive method and its speed. Some people share links to similar ideas, like cone marching and other fast SDF rendering tricks. Others ask if this could be used for more complex scenes, or if it only works well when there is lots of empty space.

A few commenters point out that GPU methods are still much faster for big, busy scenes. Some people compare this to other new graphics techniques and wonder if SDFs will become more common in games. Others share their own projects or experiences with SDFs, saying they love how simple and creative it feels.

One person worries about the limits—like what happens with lots of overlapping shapes or curved surfaces. Another asks if this recursive method could work with animation or dynamic scenes. A few users think this is a great way to teach graphics programming, because it’s easy to understand.

Some readers are curious if this could be used in web browsers or on mobile devices. Others discuss how hard it is to write fast CPU code for graphics, but are happy to see progress. Overall, most people are impressed by the clever use of recursion and hope to see more examples and open-source projects using these ideas.

## Baochip-1x: What it is, why I'm doing it now and how it came about

- 原文链接: [Baochip-1x: What it is, why I'm doing it now and how it came about](https://www.crowdsupply.com/baochip/dabao/updates/what-it-is-why-im-doing-it-now-and-how-it-came-about)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47339219)

Baochip-1x is a new chip built to support secure, high-assurance software, and it’s different because it includes a Memory Management Unit (MMU)—a feature not found in most small microcontrollers. The creator, bunnie, explains he’s making this chip now because open-source hardware is becoming more possible, thanks to RISC-V, and because he wants developers to have more control and security.

The MMU is important because it lets each app run in its own protected space, the way your phone or computer works, but most small chips don’t have this feature. In the past, small chips skipped MMUs to save space and cost, since old chips had very little memory. ARM, a big chip company, kept MMUs out of their low-cost chips on purpose so they could charge more for their high-end ones. Over time, everyone got used to this, even though new chips are much more powerful.

Now, with open-source chip designs like RISC-V, it’s possible to add an MMU to even small chips. This means you can run real operating systems like Linux or Rust-based Xous on them. Bunnie believes waiting for perfect, fully open-source chips would take too long, so he chose to make a “partially open” chip now—most important parts are open, but some parts like USB and analog bits are still closed.

Baochip-1x happened because bunnie was able to “hitchhike” on another company’s chip project—he added his CPU into unused space on a chip being made by Crossbar, Inc. This made it affordable, even without outside investors. The chip uses RRAM, a kind of memory that’s fast and keeps data when power is off.

Security certification is tricky for open-source chips. Oddly, closed chips get better “security” scores because it’s harder for testers to find problems, even though open chips are more transparent. Banks and governments—big buyers—prefer closed chips because of these certifications.

Hundreds of early Baochips are already made and given to developers. Soon, more will be ready, and bunnie hopes to build a community to help test, write documentation, and support more programming languages—starting with his favorite, Rust.

In the Hacker News comments, many people are excited about seeing a real MMU in a small, affordable chip. Some think it’s about time, since hardware has improved a lot but chip makers haven’t changed old habits. Others are happy this chip uses open-source ideas and hope it will break ARM’s lock on the market. A few users discuss the technical details of the MMU and why it matters for running secure apps or modern operating systems.

Some commenters mention the fact that the chip isn’t fully open and wish the analog parts could be open too, but most agree that “partly open” is a big step forward. There’s debate about how hard it is to get open chips certified for security, and some say it’s a real problem that should be fixed. Several users praise the way bunnie got the chip made by sharing space on someone else’s project—calling it clever and resourceful.

A few people question whether developers will adopt it quickly, since tools and libraries will need to catch up, but others think having real hardware to hack on will attract open-source fans and drive progress. Some see this as a way to help move away from closed, locked-in hardware, and are excited to try running Linux or Rust OSes on a small, cheap chip. Overall, there’s a mix of hope, technical curiosity, and support for pushing open hardware forward, even if it isn’t perfect yet.
