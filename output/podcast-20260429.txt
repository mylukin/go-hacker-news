Now, let’s talk about a major Linux security bug called “Copy Fail” (CVE-2026-31431), found by Xint Code. This is a logic flaw in the Linux kernel that allows any local user to become root on almost every Linux distribution since 2017. The bug is not a race condition and does not need special versions. It is a simple mistake in the cryptography code. The exploit is a small Python script, only 732 bytes, that works out-of-the-box on systems like Ubuntu, Amazon Linux, RHEL, and SUSE.

The bug lets a normal user write four controlled bytes into the page cache of any readable file, such as /usr/bin/su. This gives root access without changing anything on disk, so the attack is hard to detect and does not persist after a reboot. Most at risk are shared servers, CI build runners, container hosts, and cloud platforms that run user code. For single-user machines, the risk is lower, but patching is still needed. The fix is to update the Linux kernel or disable the algif_aead module as a temporary step.

In the comments, users were shocked by how wide and simple this attack is. Some praised the clear write-up and the fast patch. Others worried about cloud and container systems, since the bug crosses container boundaries. Many compared Copy Fail to past bugs like Dirty Pipe and Dirty Cow, saying this one is more portable and reliable. Some discussed the ethics of releasing the script, but most agreed defenders need it to check their systems. There was also talk about file integrity tools, since the attack only changes memory. Some were impressed that AI tools helped find the bug, but noted human insight was still needed. Many called for better security reviews in kernel crypto code, since this bug went unnoticed for years.

Now let’s look at a debate about FastCGI versus HTTP for reverse proxies. The article says FastCGI, a protocol from 30 years ago, is still safer than HTTP. Using HTTP between a proxy and backend can cause security problems, like desync attacks and request smuggling, because HTTP messages are hard to parse and have unclear boundaries. FastCGI avoids these issues by keeping client headers and trusted proxy data separate, so the backend knows what is safe. Many popular proxies support FastCGI, and switching is often just changing one line in your code or config.

The article says FastCGI is not just for old CGI scripts—you can use it with modern apps. HTTP/2 can solve some problems, but not all software supports it well yet. FastCGI does not support WebSockets and sometimes is slower in benchmarks, but the author thinks it is due to less optimization. The tools are not as nice as HTTP’s, and curl does not support FastCGI.

In the comments, many people agreed FastCGI is safer for reverse proxies. Some shared stories about security bugs from HTTP request smuggling and said FastCGI avoids these problems. Others liked using FastCGI with Go and nginx. A few noted that FastCGI can be slower and the lack of WebSocket support is a real drawback. Some said it is hard to move away from HTTP because most tools and libraries are built for HTTP. Others asked if new protocols like gRPC or HTTP/2 could solve the same problems. Some people wish FastCGI had better tooling and docs, but enjoyed learning about old tech that still works well.

Another story discusses why the author prefers Lisp and Scheme over Haskell. The author says Haskell has advanced features and a strong type system, making it safe and pure, but also hard to use for quick or simple projects. Writing or testing new ideas in Haskell is slow because of the strict type system and special rules for effects. In Java or Kotlin, simple things like writing XML are much easier.

Scheme and Lisp are more flexible. You can try things quickly and print data anywhere, without worrying about special rules. Lisp’s macro system lets you change the language or build your own small languages. The REPL in Scheme makes it easy to test and change code live. For the author, Scheme is easier, more fun, and gives more freedom, even if it has fewer libraries.

In the comments, some agreed Lisp is great for quick tests and building new ideas, while others said Haskell’s strict rules are good for big projects where safety matters. A few said Haskell has a hard learning curve, while Lisp feels friendly. Some noted Lisp’s lack of libraries for large projects, and others defended Haskell for safety in production. People also talked about the REPL experience—Lisp’s is more powerful and feels like a live conversation. Some said macros are powerful but can make code hard to read. Most agreed both languages have their place, and programmers have different needs and tastes.

Ramp’s Sheets AI had a security problem that let attackers steal private financial data from spreadsheets. The issue was found and fixed in March 2026. When users imported outside data, hidden prompt injections could trick the AI into building a formula that sent private data to an attacker’s website. This was done using an IMAGE formula with a URL containing the data. The AI added the formula automatically, without user approval.

PromptArmor found the problem and said they had seen similar issues in other AI spreadsheet tools, like Claude for Excel. In that case, human checks were added, then improved to show clear warnings and the exact formula before adding it.

On Hacker News, users were surprised spreadsheet AIs could be tricked so easily. Some said formulas making network requests are always risky and should be blocked or warned about by default. Others said AI tools with important data need better human review. Some now avoid importing outside data into sensitive files. Others praised Ramp for fixing the flaw quickly, but warned that as AI gets more power in office tools, these attacks will be more common unless security is a focus from the start.

Our last story is about Laws of UX, a website listing important rules for making user interfaces easy and pleasant. The site gathers “laws” from psychology and design, with clear explanations. For example, the Aesthetic-Usability Effect says beautiful designs seem easier to use, Hick’s Law says too many options slow people down, and Miller’s Law says people can only remember about seven things at once. Other laws cover how people group items, remember the start and end of lists, or want sites to work like others they have used before.

In the comments, many people called the site a great guide for designers. Some wished they had learned these rules earlier. Others liked the simple and clear explanations, but a few warned that real projects are more complex. Some discussed the science behind the laws, saying some are well-tested, while others are more like advice. A few said knowing when to break a rule is as important as following it. One person pointed out the site is a good example of good design itself. Others said these laws are useful for any kind of product, not just web design. People debated which laws are most important, with many picking Hick’s Law and Miller’s Law. Some use the list as a checklist or share it in meetings to explain design choices. Overall, most agreed Laws of UX is a handy resource for making better, easier-to-use products.
