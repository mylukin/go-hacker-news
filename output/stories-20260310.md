## RISC-V Is Sloooow

- 原文链接: [RISC-V Is Sloooow](https://marcin.juszkiewicz.com.pl/2026/03/10/risc-v-is-sloooow/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47328214)

This article talks about using RISC-V hardware to build Fedora Linux and how slow it is compared to other computer architectures. The writer has been helping to fix and build packages for Fedora on RISC-V and has sent many updates to make things work better.

The main problem is that RISC-V computers are much slower than others like x86_64, ARM, or PowerPC when building software. For example, building a big package called “binutils” takes 143 minutes on RISC-V, but only 29 minutes on x86_64 or even less on some others. RISC-V machines also have less memory, which means some build features like LTO (Link Time Optimization) have to be turned off to avoid running out of memory. The writer compares the RISC-V cores to the slowest ARM cores, which shows RISC-V is still behind in speed.

There is hope for better hardware soon, like the UltraRISC UR-DP1000 and SpacemiT K3, which can have more memory and should be faster. But even these will not fix everything. Until there are fast, rackable, easy-to-manage RISC-V servers, RISC-V cannot become a main Fedora Linux architecture. Because real hardware is so slow, the writer often uses QEMU, a software emulator, with 80 virtual cores to build packages. Even in QEMU, building big packages like “llvm15” takes hours, but it is still faster than some real RISC-V boards. The team plans to keep working on Fedora for RISC-V and hopes to use newer, faster hardware soon.

In the comments, many people agree that RISC-V hardware is still slow and not ready for heavy server use. Some say this is normal because RISC-V is new and needs more time to catch up. Others mention that RISC-V is open and simple, which is good for learning and research, even if it is not fast yet. A few point out that software and hardware will both need many improvements before RISC-V can compete with x86 or ARM. There are also comments about how building big software projects always needs a lot of speed and memory, and this is a problem for all new computer architectures. Some users hope that companies will invest in faster RISC-V chips soon. Others suggest using cloud or emulation for now, until better RISC-V hardware is available. A few are excited for the future and believe RISC-V will get there, but it will take years. Some are worried that if RISC-V stays slow, developers may lose interest. Others say that speed is less important for some uses, like education or special projects, where RISC-V can still be useful today.

## Agents that run while I sleep

- 原文链接: [Agents that run while I sleep](https://www.claudecodecamp.com/p/i-m-building-agents-that-run-while-i-sleep)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47327559)

The article is about building AI agents that write and test code while the author sleeps, focusing on how to trust code that is created automatically. The main problem is that these agents can make a lot of changes quickly, but it’s hard to know if the code is really correct because people can’t review every change.

The author tried using tools like Claude, which can write code and tests, but found that when the same AI writes both, it just checks its own work. This means mistakes or misunderstandings can go unnoticed. Hiring more reviewers doesn’t help much, because you can’t scale up human review fast enough, and making skilled engineers read endless AI-generated code is not a good use of time.

The author points out that Test-Driven Development (TDD) got one thing right: you should write the tests or acceptance criteria first, before writing code. With AI, you can write what you want the feature to do in plain English, and let the agent build and check it. For example, you could describe how a login page should work, what error messages to show, and what should happen on success or failure.

The practical workflow is: write detailed acceptance criteria for each feature, let the AI build the feature, then use browser agents like Playwright to test if each criterion is met. For backend changes, use API checks. The system then creates a report showing which criteria passed or failed, often with screenshots.

This process doesn’t catch misunderstandings in the original spec—if you write the wrong acceptance criteria, the tests might all pass but the feature could still be wrong. Still, it’s better at catching real-world issues like bugs and failed integrations than just reviewing code.

To make this work, the author built a Claude Skill called “verify,” which runs several steps: checking setup, planning the tests, running browser agents in parallel for each acceptance criterion, and finally having the AI judge the results. It’s easy to add this to your workflow, and you can adjust or extend each stage.

The key idea is that you must define what “done” looks like before the agent starts. This is harder up front, but it ensures the code is checked properly, instead of just hoping it's right after it’s done.

In the comments, some people are excited by the idea of agents working overnight, saying it could boost productivity and let engineers focus on bigger problems. Others worry that relying too much on AI for both writing and checking code can hide mistakes, since the AI might miss the same issues a human would.

A few users point out that writing good acceptance criteria is hard and time-consuming, so teams might skip it or do it poorly. Some think this is similar to old TDD debates, where people resist writing tests first because it feels slower. There are also concerns that if the initial requirements are wrong, no amount of testing will help.

Some commenters like that the system uses real browser agents to check features, saying this catches bugs normal code review might miss. Others wonder if this approach will really scale for big teams or complex products. A few mention that this workflow might lead to less understanding of the code, since humans only review failures and not the code itself.

Overall, the community finds the idea interesting and promising, but they see risks in trusting AI too much and in skipping human understanding or review.

## FFmpeg-over-IP – Connect to remote FFmpeg servers

- 原文链接: [FFmpeg-over-IP – Connect to remote FFmpeg servers](https://github.com/steelbrain/ffmpeg-over-ip)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47327015)

This project is called ffmpeg-over-ip, and it lets you run GPU-accelerated ffmpeg on a remote server, even if your local computer has no GPU. It solves the problem of using a GPU for video tasks without needing shared filesystems or complex GPU passthrough setups in Docker, VMs, or other remote situations.

The way it works is simple: you run a server on the machine with the GPU, and a client on the machine with your video files. The client acts just like ffmpeg, but sends your commands to the server over a single TCP port. The server runs a special version of ffmpeg that sends all file operations back to the client, so files never touch the server disk. This means you don’t need NFS, SMB, or any network drives.

You don’t have to install ffmpeg yourself; the project gives you ready-built ffmpeg and ffprobe with support for many hardware accelerations (like NVENC, QSV, VAAPI, and more). The client and server work on Linux (x86_64, arm64), macOS (both Intel and Apple Silicon), and Windows. Security is handled using HMAC-SHA256, and only the server needs to open a port.

To use it, you just point your programs (like a media server) at the client binary instead of the normal ffmpeg. The client connects to the server, authenticates, and starts the job. You can have multiple clients at the same time, and each gets its own ffmpeg process on the server. There are guides for setup, Docker integration, and troubleshooting, and the project is open-source with a split GPL/MIT license.

From the comments, many people say this is a clever solution for video servers and media centers running in containers or VMs. Some users are excited because it removes the pain of mounting network drives and messing with GPU drivers for Docker. Others point out that patching ffmpeg for tunneling file I/O is a smart move, and are impressed by the idea that files stay only on the client.

A few commenters are curious about performance, wondering if tunneling file operations over TCP adds much delay, especially for big files. Some ask if this setup works well for real-time video encoding, or if it’s best for batch jobs. There are questions about security, like what happens if the server is exposed to the internet, and people suggest using firewall rules or VPNs for safety.

One person worries about edge cases, such as what happens if the client disconnects during a long job, or if the network is slow. Others wish there was support for more platforms, like ARM Windows or FreeBSD. Some mention that having ready-to-use Docker support is a big plus, and think more projects should do something similar.

Overall, people like that ffmpeg-over-ip keeps setup simple and makes GPU transcoding available without complex tricks. Some see it as a good building block for cloud media servers, and others already want to try it in their home labs.

## Billion-Parameter Theories

- 原文链接: [Billion-Parameter Theories](https://www.worldgov.org/complexity.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47326555)

This article talks about how simple theories, like E=mc², were enough to explain many things in science before. These short equations worked because people had to write them, remember them, and use them in their heads or on paper. For complicated systems, like machines or orbits, breaking things into small parts and understanding each one was possible.

But some problems are not just complicated—they are complex. Things like poverty, climate change, and the stock market have many parts that interact in changing ways. You can't just break them apart and understand everything. Sometimes, trying to study or change these systems even changes the system itself. Earlier, scientists tried to use simple models for these problems, but often failed or only partly succeeded.

The article says that, in the past, people often learned how to do things (like building cathedrals or farming) before they understood the science behind them. Now, with AI and big neural networks, we have new tools to build models of complex systems, even if we don't fully understand why they work. These models, like large language models, can make good predictions because they have learned from huge amounts of data. But their "theories" are so big—billions of numbers—that no human can remember or write them down.

Some people, like David Deutsch, say a good theory should be small, simple, and work everywhere. Big neural networks are not small or simple. But the article points out that while the details of each model are huge, the basic structure (like the transformer architecture) is simple and can be used in many different areas. So, maybe the "theory" is the model structure, and the details are just how it fits each problem.

Scientists are now trying to understand how these big models work inside. By studying the model, we might learn new things about the complex systems themselves. Instead of starting with a simple idea and building up, we start with a big working model and try to find the simple ideas inside it.

In the comments, some people agree that complex problems need complex models. They say simple rules won't work for everything. Others worry that if we can't understand these big models, we can't trust their answers, especially for important decisions. Some point out that AI is like blacksmithing before science: we can do useful things, but we don’t really know why they work.

A few argue that just because a model is big doesn’t mean it’s the best way to understand something. They want science to keep looking for simple explanations. Others think that studying these big models might help us find new simple rules in the future. There are also comments about how AI models can make mistakes in surprising ways, so we have to be careful. Some people are excited that we can finally make progress on hard problems, even if our theories are not elegant. Others remind that history shows practice can come before theory, but theory is still important for real understanding.
