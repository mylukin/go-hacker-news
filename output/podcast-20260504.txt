First, OpenAI explained how they built their voice AI feature to be fast and smooth for people all over the world. The main challenge was making conversations feel natural, with no awkward silences or delays, even when millions use the system at the same time. OpenAI uses WebRTC, a standard for sending audio and video between computers in real time. WebRTC handles tricky things like firewalls and encryption, so OpenAI can focus on the AI itself.

Most sessions are just one user talking to the AI, not group calls, so OpenAI made their own design. They use a special service called a “transceiver” to manage each user’s connection. Because WebRTC usually needs lots of open network ports, which is hard to secure at large scale, OpenAI added a “relay” server. The relay simply forwards audio to the right place using a small identifier, so only one port is needed. Relays are placed all over the world, sending user audio to the nearest relay for speed and stability. The relay is written in Go and is kept simple for both speed and security.

Hacker News comments praised OpenAI’s clear technical write-up and smart use of existing standards. Some were surprised at how much engineering is needed to make voice AI feel natural, and others were glad OpenAI kept security in mind. There was also a discussion about using Go for the relay, whether even faster languages might help, and if this system could be used for video or other apps. Overall, people agreed that OpenAI found a good balance of speed, security, and scale.

Next, we look at Bun, a fast JavaScript runtime, and worries after it was bought by Anthropic, an AI company. The author of the article really likes Bun as a Node.js alternative, but is now nervous about its future. After Anthropic bought Bun, they promised to keep it open source and use it for their coding tool, Claude Code. But over time, Claude Code became more limited, with confusing billing and bugs. Some features even started charging extra or blocking requests if certain words appeared in the project history.

The author calls this “enshittification”—when a product gets worse because of business decisions, not technical problems. Now, the author is switching back to pnpm for new projects, just to be safe. In the comments, many agree that Bun is excellent but share the same concerns. Some think big companies buying open source projects can be a bad sign, while others say Anthropic still needs Bun for their own tools, so it might stay strong. People also note that because Bun is open source, it can be forked if things go badly. Some say it’s wise to be cautious and keep an eye on company actions.

Now, a serious security story. A company called Strix found a big bug in the API of Schemata, a military training platform for the U.S. Department of Defense. This bug allowed anyone with a normal account to see private data from all users and organizations—not just their own. Strix found that there were no checks to stop users from viewing or changing other people’s data, including names, emails, military base assignments, and even confidential training manuals.

Strix tried to report the problem privately, but Schemata was slow to respond and only fixed the bug after Strix threatened to go public. Hacker News readers were shocked that a contractor with the government could miss such a basic security check. Some worried that other defense companies might have similar issues, and many praised Strix for being patient and responsible. There was also debate about whether companies should be punished more for slow security responses and if there should be better ways to report security bugs.

Moving on, Stripe wanted to autoformat all 25 million lines of their Ruby code in one night using a tool called rubyfmt. Before this, there was no fast, reliable autoformatter for Ruby. Rubyfmt was created to be fast, need no configuration, and make style decisions for everyone—like gofmt does for Go. Making rubyfmt fast was hard, so they wrote parts in C and Rust, and later switched to a new Ruby parser called Prism for even better speed and simplicity.

Stripe let teams opt-in slowly, tested a lot, and then formatted almost all files at once to avoid merge conflicts. Now, rubyfmt runs on all Stripe code, and engineers don’t argue about style anymore. The tool is open source, and many commenters praised Stripe’s technical work and focus on developer productivity. Some wondered about edge cases and the effect on code history, but most agreed that autoformatters save time and help new hires get up to speed faster.

GameStop’s CEO, Ryan Cohen, says he can make eBay better and help it compete with Amazon. If eBay’s board rejects the offer, he plans to appeal directly to shareholders. Experts are not sure the deal will happen, since GameStop is much smaller than eBay and would need to borrow $20 billion. After the news, eBay’s stock went up while GameStop’s went down.

Others pointed out that the real winner might be Amazon, since their big online rival would be distracted. Next, a security issue with Microsoft Edge, the web browser. It turns out Edge stores all your saved passwords in computer memory as clear text—even if you’re not using them. Anyone with access to the computer’s memory can see your passwords, making it easy for malware or hackers to steal them. This is a problem for people using shared or public computers, and it’s different from some other browsers that protect passwords better.

Comments were mixed. Some people are very worried and say this is a big mistake by Microsoft. Others note that if someone has memory access, you may already have big problems, but most agree Edge should do better. Some suggest using a separate password manager instead of saving passwords in the browser, and many hope Microsoft will fix this soon.

Now, a study about work and aging. The research looks at whether having a job can help slow down mental decline as people get older. The authors found that when there are fewer jobs in an area, people’s thinking skills go down faster, especially for men aged 51 to 64. They used special methods to show that losing jobs actually causes faster decline, not just by chance. The effect was less clear for women and older men. The study says working longer might help keep your mind sharp, but the main point is that there are many factors at play.

In the comments, some people shared stories about older relatives who stayed sharp by working, while others pointed out that not all jobs are good for your mind. Some noted that people who keep working might already be healthier. There was also discussion about other ways to keep your mind active, like learning, socializing, or doing puzzles, and some warned against using this research to push people to work longer if they don’t want to.

Next, a technical story about Redis. The author spent four months adding a new array data type to Redis, using both his own skills and help from AI tools like GPT. He started with a detailed plan and used AI to try different designs and test code. The new array type splits data into slices to handle large arrays without using too much memory. He also added a new command for searching arrays with regular expressions.

The community is excited about this new feature and thinks it will help many developers. Some worry that adding complex features could make Redis harder to use, but most are impressed with the careful planning and AI help. People are looking forward to trying the new array type and seeing how it performs in real projects.

Finally, an article about Large Language Models, or LLMs, and their real effect on programming. The author says calling LLMs “AI” is confusing and that most excitement is about writing code faster. But, he points out, writing code is only a small part of software development. Studies show LLMs can speed up code writing, but often cause more mistakes and unstable projects. The real gains still come from strong teamwork, clear planning, and good practices.

Some commenters agree and share stories about LLMs making mistakes, while others are hopeful that LLMs will get better soon. Many say the real problems in software come from unclear requirements, not from slow coding. There are worries that companies will push LLMs too much, hoping for big savings, but may end up with broken projects. Most agree that LLMs are just a tool—strong fundamentals still matter most.
