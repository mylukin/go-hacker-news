First, a team has made history by reading an entire Herculaneum scroll from 79 AD—without opening it. They used X-ray scanning and machine learning to “unwrap” the fragile papyrus virtually. The scroll, called PHerc. 1667, was too delicate to open by hand, and past attempts destroyed the outer layers. Using high-resolution X-rays, the team mapped the shape of the rolled-up sheets. With AI, they enhanced the faint ink and revealed about 1.4 meters of ancient Greek text, in 22 columns.

The text is a philosophical work on ethics, likely written by a Stoic philosopher in the 2nd century BC. It discusses human nature, wisdom, and happiness, and even has the name Aristocreon, linking it to the famous Stoic Chrysippus. While some parts are damaged, many lines are readable for the first time in almost 2,000 years.

The same methods were used on other scrolls, too. On one, higher-resolution scans made the ink directly visible. On another, they found the author and title: Philodemus, On Gods, Book 8. All the data and code are open, thanks to the Vesuvius Challenge, a public project that brought together scientists and hobbyists worldwide. Many team members started as contest participants.

Next, let’s talk about Un-0, a new AI image generator that uses coupled oscillators—like metronomes—instead of deep neural networks. The creators think this could make AI much more energy efficient, maybe even 1,000 times better than today’s systems. In Un-0, each oscillator has a phase and interacts with others. The system learns the strength of these connections and their speeds. To make an image, the oscillators start in random positions, then run for a while. Their final positions are used as a hidden code, which a decoder turns into an image.

Un-0 was tested on datasets like CIFAR-10 and ImageNet 64x64. The best model got a FID score of 6.74 on ImageNet 64x64—similar to early GANs and diffusion models. Most of the model’s size comes from the oscillators, not the decoder. Experiments show that removing the oscillator part makes image quality worse, so they really do useful work. The idea is inspired by the brain, where oscillations help with tasks. The team hopes to build hardware for this in the future, using the physics directly. All code and model weights are open.

IBM is in the news with a new chip technology that is smaller than 1 nanometer, thanks to a 3D “nanostack” design. This chip can fit around 100 billion transistors on a chip the size of your fingernail—almost double their old 2 nm chip. Instead of laying out transistors flat, IBM stacks them in layers, like a tall building. This saves space, makes chips faster, and uses less energy. IBM says the chip is up to 50% faster or 70% more energy efficient than their 2 nm chips. It also allows different materials in each layer and improves memory storage by 40%.

The chip uses new building methods, including high-precision EUV lithography, and IBM worked with partners to make it real. They say this will help the chip industry keep shrinking and improving chips for at least another decade.

Now, some news for Zig programmers. Zig has changed how its `@bitCast` function works and improved its LLVM backend. Before, Zig stored small custom integer types in memory exactly like LLVM, which caused missed optimizations and bugs. Now, these types are stored as standard sizes in memory, using custom sizes only when needed, just like C. This helps the optimizer and matches C’s style.

The `@bitCast` function now works based on logical bit order, not how the computer stores bytes. For example, casting an array of two u8s to a u16 always puts the first element in the least-significant bits, no matter the CPU’s endianness. This makes `@bitCast` work the same everywhere, avoiding undefined or dangerous behavior.

The compiler can now use a “Legalize” pass to make hard `@bitCast` operations easy and reliable. The new rules allow interesting tricks, like turning arrays of small integers into bit vectors. The update also blocks unsafe casts to and from vectors of pointers and allows casting enums using `@bitCast`. These changes will be explained in the next Zig release notes. Performance has improved too—some code runs about 5% faster, even in the Zig compiler.

On Hacker News, most are happy with these changes. People think the new semantics are easier and safer. Some worry about updating old code, especially if it depends on the old behavior. Many are glad that `@bitCast` now works the same everywhere, so no more endianness bugs. There’s praise for performance gains and matching C’s way of handling types. Some users ask for more documentation and migration tips. Overall, the reaction is positive, with excitement for writing safer, more portable Zig code.

Let’s look into the world of “Bank Python.” The article explores a special form of Python used in big investment banks, focusing on a fictional system called Minerva. In banks, Python is very different from the open-source version. Minerva is built around a big database called Barbara, which stores almost everything as Python objects—trades, market data, even code. There’s no normal file system. Everything is stored and read from the database.

The Dagger system keeps track of how the value of financial products depends on each other, updating related values when one changes. Walpole is the single job runner for the bank, making it easy to schedule jobs by editing a config file, much simpler than cloud systems. For data tables, they use MnTable, a home-grown library for “medium” sized data. Minerva is closed, with little code shared outside. It even has its own IDE and web framework. Over time, Minerva and public Python have grown apart.

Hacker News readers find this world fascinating. Many are surprised by how far “Bank Python” has moved from open-source Python. Some wonder if this hurts the developers’ skills and makes it hard to leave banking for other jobs. Others see similarities to other big companies with their own tools. Some think it is smart to keep everything in one place, for control and reliability; others see risks if only a few people understand the system. There are discussions about storing code in databases, the use of floats for models, and how much banks still rely on Excel. Some wish banks would share more with the world, while others think these systems are too tied to banking. Many agree that learning from these unique, long-lived systems helps all developers.

Let’s move on to parallel algorithms. The article explains how to check if parentheses in a string are balanced, first with a stack, then with parallel methods. For the simple case, you map ‘(’ to +1 and ‘)’ to -1, then do a prefix sum. If the sum goes negative or is not zero at the end, the string is not balanced. For more kinds of brackets, the problem is harder. The article explains methods involving nesting depth, stable sort, and reduction trees for efficiency.

On Hacker News, people say the stack method is simple and good for most real code, but parallel methods are interesting for teaching and big data. There are questions about handling many types of brackets and if this is useful for compilers or editors. Some like the map-reduce style for thinking differently, while others find the reduction tree method clever but hard to understand. Some share links to similar algorithms, and others debate if sorting-based methods are practical. There are worries about edge cases, like integer overflow or Unicode brackets. Most readers like learning about parallel algorithms, but think simple solutions are best for most tasks.
