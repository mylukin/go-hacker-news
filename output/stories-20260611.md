## Show HN: Homebrew 6.0.0

- 原文链接: [Show HN: Homebrew 6.0.0](https://brew.sh/2026/06/11/homebrew-6.0.0/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48490024)

Homebrew 6.0.0 is a new release of the popular package manager for macOS and Linux, bringing many improvements and changes. The main updates include a new security feature for trusting third-party taps, a faster internal JSON API, better sandboxing on Linux, improved performance, and first support for macOS 27 (Golden Gate).

A major focus in this release is security. Now, if you want to use software from third-party sources (called “taps”), you must mark them as trusted before Homebrew runs their code. This helps protect your computer from risky or unknown code. There are also new commands to help manage tap trust, and clearer documentation about how this works.

The new internal JSON API makes Homebrew faster by combining all package information into a single download. This means Homebrew will update quicker and use less internet data. The Linux version now runs builds in a sandbox, just like on macOS, which adds another layer of safety.

User experience is better too. After a user survey, Homebrew added features like an “ask” mode, which shows you what will change before you install or upgrade. There are improvements to “brew bundle”, allowing you to install many packages in parallel and manage extra systems like npm or winget (on Windows).

Performance has been boosted across the board, with faster startup, quicker commands, and smarter downloads. Homebrew is ready for the next macOS version and warns that old Intel Macs will soon lose support.

On the security side, several problems were found and fixed, including issues with downloads and installer scripts. There are also more ways to check for vulnerabilities, better handling of secrets, and new options for package checks.

Homebrew now has more features: you can pin casks, use AppImage on Linux, and enjoy smarter upgrades. There is better support for different operating systems and CPUs, and new commands like “brew exec” and “brew as-console-user”.

The release includes many smaller changes, like clearer information when you check a package, new download features, and better developer tools. The documentation is also improved, and Homebrew asks for donations to keep the project running.

In the comments, many users praise the focus on security, especially the tap trust system. They like that Homebrew is getting faster and more reliable. Some are happy about Linux improvements and the new internal API, saying updates now feel much quicker.

A few users worry about the upcoming end of support for Intel Macs, but most understand that it is needed to keep moving forward. Some mention that the long list of changes shows Homebrew is active and well-maintained.

People also talk about the move away from Ruby to Rust and back, discussing the real-world performance results and the honesty from the Homebrew team. Some wish for a stable graphical interface, while others are excited about the new BrewUI project.

There is debate about Homebrew’s role on Windows, with some saying it is useful, while others feel native Windows tools are better. Comments also mention the importance of good documentation and the helpful new guides.

Overall, the community is positive about the release, happy with the direction, and thankful for the volunteers behind Homebrew. Some users say it is their favorite tool for managing software, while others note that Homebrew’s focus on security and cross-platform support makes it stand out from other package managers.

## Petition to Withdraw Canada's Bill C-22

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48491830)

This petition asks the Canadian government to stop Bill C-22, a law about collecting and keeping people’s online data. The petition says Bill C-22 lets the government make internet companies save information about everyone, even if they are not under suspicion, for up to a year.

The law would let the Minister of Public Safety order any online service—like chat apps, email, VPNs, or even banks—to help gather this data. The kind of information collected can show where people go, who they talk to, their health, religion, and politics. The petition worries that this is a big privacy risk. It also says the law could force companies to weaken encryption, making online systems less safe and easier for criminals or foreign attackers to hack, like what happened in the 2024 Salt Typhoon attack in the US.

The petition argues that keeping everyone’s data without reason is likely against the Canadian Charter of Rights and Freedoms, which protects people from unfair searches. It also says the government could change important terms in the law, such as what “encryption” means, without asking Parliament again. Because of this, the privacy rules in the bill may not be strong.

The people signing the petition want the government to stop Bill C-22. They ask for no more laws that require data to be saved without suspicion, and they want laws to clearly say encryption should not be weakened.

In the comments, many people agree with the petition and say Bill C-22 is dangerous for privacy and freedom. Some users worry that this kind of law is becoming common in many countries, not just Canada. Others point out that when the government can change definitions, it is hard to trust any protection for privacy. Some think that forcing companies to weaken encryption will only make systems weaker for everyone, not just criminals. A few people say metadata can reveal almost as much as the actual messages, so collecting it is still a big problem.

Some comments wonder if these laws ever really stop crime or just make it easier for the government to watch citizens. Others believe that if the government wants people’s data, it should have to get a court order each time, not just collect it all the time. There are also people who feel that most people do not know about these laws and do not understand how they can affect them. A few users are not surprised, saying governments often want more control and surveillance. ## The RCE that AMD wouldn't fix

- 原文链接: [The RCE that AMD wouldn't fix](https://mrbruh.com/amd2/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48492215)

This article is about a security problem in AMD’s AutoUpdate software that lets attackers run code on your computer if they are on the same network. The writer found this problem by accident while fixing an annoying console window on his new gaming PC.

He discovered that AMD’s updater gets its update list over HTTPS, but the actual software downloads happen over HTTP, which is not secure. This means someone can trick the updater into downloading and running malicious programs without any warning. The updater also does not check if the downloaded files are signed or safe—it just runs whatever it gets from the internet.

He reported this to AMD, but their bug bounty program said this kind of attack (called a man-in-the-middle attack) was "out of scope" and closed the report. After people on Hacker News started talking about it, AMD said they would review the problem. They asked the writer to remove his blog post while they worked on a fix, and he agreed, but later felt this was a mistake.

AMD decided to give the issue a CVE (an official security number), fix the problem, and mention the writer’s name, but they would not pay a reward because the updater is "optional" and the attack needs special network access. The writer waited much longer than the usual 90 days for the fix, and AMD was slow to communicate. In the end, it took 124 days to add 'https' to the URLs and fix the updater.

Later, he learned that the updater was already broken for other reasons: it cannot handle a web link redirect, so it crashes before reaching the vulnerable code. This means the security issue could not be used in practice, but it also makes it hard to update the updater itself.

Finally, AMD did fix the updater. Now it uses HTTPS for downloads, and they say all updates are checked for signatures. But the writer found that the check is only a CRC-32, which is not secure.

In the Hacker News comments, many people were shocked that such a simple but dangerous bug was missed. Some said using HTTP for software downloads is a big mistake in 2026. Others blamed AMD for treating the reporter badly, not paying a bounty, and asking him to remove his post. A few said that man-in-the-middle attacks are common and should not be "out of scope" for bug bounties, especially for tools that millions use. Some users pointed out that many updaters have similar issues, and companies should do better. Others joked that the updater being broken was a "feature" because it made the bug useless. Some people advised uninstalling AMD’s software and only downloading updates manually. A few also discussed how bug bounty programs often have tricky rules that let companies avoid paying for real problems. Overall, most commenters agreed that AMD was too slow and not transparent, and the writer did the right thing by sharing his story.

## Ear Training Practice Exercises

- 原文链接: [Ear Training Practice Exercises](https://tonedear.com/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48447598)

This article is about online exercises that help people train their ears for music. The website, tonedear.com, gives different ways to practice listening and understanding music.

It has several exercises. One is about intervals, where you listen to two notes and guess how far apart they are. Another is about chords—you hear a chord and try to name what kind it is. There are also exercises for scales, where you listen and pick which scale is playing. You can practice hearing chord progressions, which means listening to a group of chords and naming each one. There’s a perfect pitch exercise, where you guess the note after hearing just one sound. You can also try scale degrees, which means hearing a chord progression and then a note, and saying what that note is in the key. Another exercise mixes intervals and scale degrees—you hear two notes after a chord progression and have to say both their place in the scale and the distance between them. The last exercise is melodic dictation, where you hear a short melody after some chords and say the scale degree for each note.

For teachers, there is a special version of the site. Teachers can give their students assignments, check scores, and use more music theory exercises like building chords or identifying key signatures.

People on Hacker News had mixed thoughts about these tools. Some think ear training is very useful for musicians, helping them play by ear and understand music better. A few users said they improved a lot by practicing a little every day, as the article suggests. Others mentioned that learning music by ear can be hard at first but gets easier with regular practice. Some people prefer using apps or in-person lessons, while others like the freedom of online tools. One person said perfect pitch is rare and most people should focus on relative pitch, which these exercises teach well. Another user warned not to get frustrated if progress feels slow—improvement happens over time. A few teachers liked the classroom features, but some wanted even more options for tracking student progress. There was also some discussion about how these tools could be useful for people who make electronic music, not just traditional musicians. Lastly, a few users pointed out that ear training is just one part of learning music, and you should also practice playing and theory.

## Software Is Made Between Commits

- 原文链接: [Software Is Made Between Commits](https://zed.dev/blog/introducing-deltadb)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48492533)

This article talks about how making software is more about the conversations between developers than about the commits you see in Git. The writer explains that the usual way of sharing code with pull requests and comments doesn't really help teams work well together, especially as more people use AI agents to help write code.

The team behind Zed wanted a better way to collaborate, so they built a new tool called DeltaDB. Unlike Git, which only saves code at each commit, DeltaDB saves every small change, called a “delta.” Each delta keeps its own identity, so you can look at your code at any point in its history—not just at each commit. DeltaDB also saves the conversation that led to each change, so you can always see what was discussed and why a change was made.

With DeltaDB, you don’t need to wait to commit code before you can talk about it. Teammates and agents can work on the same files at the same time, even from different computers. If you want, you can also use your own tools because the files are real and can be mounted to your disk.

Another big idea is that DeltaDB connects the code directly to the conversation that created it. If you’re looking at a line of code, you can jump back to the discussion that made it, or see how it has changed over time. This helps both humans and AI agents understand the reasons behind every piece of code.

The goal is to make collaboration smoother. By putting the code and the discussion in the same place, you don’t need complicated pull requests or review threads anymore. Git and CI tools still have their place, but they don’t have to be the main way people work together on code.

In the Hacker News comments, some people liked the idea of DeltaDB and said it might solve real problems with traditional version control. They were excited to see how connecting code and discussion could help teams, especially as AI gets more involved in coding. Some users were interested in how DeltaDB would handle performance and scaling with big projects.

Others were more careful. They said that Git works well for many people and is simple to use. They worried that saving every tiny change could make things confusing or hard to manage. Some asked if DeltaDB would work with existing tools or if it would lock users into a new system.

A few commenters pointed out that conversations can be messy or off-topic, and they wondered how useful it would be to link every bit of talk to the code. Some people liked that DeltaDB could help teams remember why code was written a certain way, especially when old team members leave.

Overall, many were curious to try DeltaDB, but wanted to see it in action first. They agreed that better tools for collaboration could help, but wanted to know if DeltaDB would be easy to use and really improve their workflow. Some also mentioned that new ideas like this often take time to catch on in the developer community.
