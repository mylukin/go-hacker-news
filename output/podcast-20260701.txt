ZCode works closely with the GLM-5.2 language model, supporting planning, coding, review, and deployment—all in one place. The tool is easy to install on macOS, Windows, and Linux, with downloads for both ARM and Intel chips. It focuses on speed and simplicity, letting you break big projects into smaller tasks, track progress, and use AI agents at every step.

One example in the article shows ZCode building a Gomoku game from scratch. The AI agent planned the work, wrote the HTML, CSS, and JavaScript files, and checked the logic. It handled game rules, made smart moves, judged wins or losses, and adjusted the code for mobile screens. The code was updated for local use, so it did not need internet fonts. This shows ZCode’s AI can help with real coding from start to finish.

ZCode also lets you control bots from your phone using WeChat or Telegram. There are different price plans for small projects or big teams. The tool is often updated and helps manage long, complex projects by tracking goals and tasks.

Next, scientists have built a cell from nonliving parts that can grow and divide—a big first. The cell is not truly alive, but it acts more like life than anything made in a lab before. The team built the cell with natural biological molecules inside a membrane. It has a small genome, tools to copy DNA, and a way to make proteins. It cannot make its own food or energy, so researchers feed it using “feeder” bubbles. To help it divide, they added proteins to the membrane, making it pinch into two cells.

This process took many tries. When the cell finally grew, copied its DNA, and split, it was a big success. The team calls these “spudcells” as a joke. They look like small blobs under a microscope, but they are special to the scientists. The team tried changing the DNA to see if the cells could evolve. Some cells grew bigger and divided faster, but it was not real evolution—the changes were made by the researchers, not random mutation.

Experts call this work amazing, showing that it is possible to build many parts of a living cell from scratch. But they warn not to call it real life yet, since the cell cannot make everything it needs or live alone. The next steps are to help the cells make their own ribosomes and maybe add a cytoskeleton.

In the comments, many readers are excited by the breakthrough. Some see it as a step toward understanding the start of life and predict it could help make new drugs or materials. Others note that the cell is not truly alive and depends on human help. A few worry about risks if these cells get more advanced, but most agree they are still safe. Some discuss the difficulty of getting evolution in these simple cells, and others wonder if this work will help answer big questions about life and its start on other planets. Overall, people agree it’s a big scientific step, with more work ahead.

Next, we have a guide on becoming a real-time graphics programmer. The writer says graphics programming is like two jobs: one is learning the CPU side—using APIs like DirectX12, Vulkan, or Metal, and handling assets; the other is the GPU side—making images look good and run fast with lighting, shading, and effects. You can focus on one side first, using simple tools for the CPU if you want to learn GPU.

For the GPU side, you should learn path tracing—used to make movie-like images—and Physically Based Rendering (PBR) for better assets. The article shares links to learn more, like "Ray Tracing in One Weekend." On the math side, you need linear algebra, basic trigonometry, and some calculus. C++ is the main language, but some use Rust or JavaScript with WebGPU. For shaders, HLSL and GLSL are common.

The article says machine learning is not very useful for graphics programming yet, but might help in the future. In the comments, many agree that learning both CPU and GPU sides is hard. Some suggest starting with simple projects, like drawing a triangle. Others say math is key, and not to fear it. Some talk about learning paths—starting with OpenGL, then moving to new APIs. There are tips on understanding hardware, reading open source code, and building a portfolio with finished projects. Most agree C++ is still the main language, but Rust and WebGPU are good for learning. Some worry about the future with more AI tools, but most believe learning the basics is the right way.

Now, FFmpeg 9.1 has a fully rewritten AAC audio encoder. The developer rebuilt every part, focusing on better rate control and smarter use of coding tools. Tests show this new encoder beats other popular AAC encoders, like qaac and fdk-aac, in quality and listening tests. It works best at a constant bitrate, mainly for 48kHz audio, but also supports 44.1kHz and 96kHz. The encoder avoids wasting bits on sounds you can’t hear, and turns off extra tools if not needed. There’s also a fix for a known bug in stereo PNS decoding. The developer tested it on 3,000 music tracks and added stat printing for bug reports.

In the comments, some users note most audio is still 44.1kHz, but the developer says resampling is easy and 48kHz is recommended. People hope this new encoder fixes the poor quality of the old one. Testers found big improvements, especially at 100kbps and above, though some still see issues at low bitrates or with tricky sounds. Some report small artifacts, like ticking, and suggest fixes. Others share command-line tips and discuss audio quality settings. Most are excited about the improvements, but some want more fine-tuning for special cases.

Next, Box3D is a new open source 3D physics engine, made as an upgrade to Box2D for use in modern 3D games. The author built it for his game, The Legend of California, after problems with Unreal’s built-in engine. Box3D keeps the core ideas of Box2D, but adds features for big 3D worlds, like triangle mesh collision and height-field support. It has a C API, uses C17, supports multi-threading, and aims for cross-platform determinism. The author wanted more control and speed, especially for handling many objects and tricky cases like falling trees on voxel terrain.

Box3D is open source and already used in games like s&box and Esoterica. The code is on GitHub with samples and a Discord community. It’s still early and will keep improving, with plans for more features. In the comments, many are excited to see a new open source 3D physics engine from the creator of Box2D. Some praise the C API and openness. There are questions about performance, real-world use, and comparisons to engines like Jolt or Bullet. Some share their own struggles with game physics and say good docs and examples are key. Others talk about the pros and cons of making your own engine for big projects. Many thank the author for sharing his work.

Now, an article explains how an internal combustion engine works, from simple ideas to a real car engine. It starts with a crank, which turns a push into spinning movement, then explains how a cannon works, and how a piston replaces the cannonball to turn explosions into motion. The engine uses the four-stroke cycle: intake, compression, power, and exhaust. More cylinders are used to run smoother.

The article explains important parts: engine block, crankshaft, pistons, piston rings, valves, and how oil and flywheels help the engine run better. Modern engines use computers for control. Timing belts keep everything in sync. The article also explains why engines have several cylinders, and how pressure and speed change inside the engine. Cooling and oil systems, manifolds, and sometimes turbochargers help the engine work better and last longer.

Many people praise the article’s clear animations and simple explanations. Some wish they had this resource when first learning about engines. Others say visualization is very helpful. There’s talk about electric vehicles replacing combustion engines, but some say learning about classic engines is still useful. Some discuss engine repair and building, and others add more details or share links to other resources. Overall, the article is seen as a great way to start learning about engines.
