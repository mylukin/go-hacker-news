The article gives examples of privacy and security problems with cars, such as hackers unlocking cars remotely, insurance companies raising rates based on driving data, and car companies sharing private videos or information. The writer wanted to stop this tracking, so they decided to physically remove the parts that let the car send data: the modem (called the DCM) and the GPS.

This can be tricky, but the guide includes steps for doing it safely.

One important warning: if you use Bluetooth to connect your phone, the car can still send data using your phone’s internet. Using a wired USB connection for CarPlay avoids this. The writer shares tips for which cables and adapters to use.

The article lists all the tools needed for the job and gives step-by-step instructions—using trim tools, removing panels and bolts, taking out the modem, installing the bypass kit, and unplugging the GPS wire. After reassembling everything, you can check if the removal worked by looking for a “no connection” icon and testing the microphone.

The writer says they’re happy with the result, but worries manufacturers may soon make it harder to remove these parts, or pass laws against it. They end by hoping for stronger privacy laws so people don’t need to do this kind of work.

A security research team found the first public macOS kernel memory corruption exploit on Apple’s new M5 chip, even though this chip uses strong protections called Memory Integrity Enforcement (MIE). The team went to Apple’s main office to report the bug in person and shared their story online. They say memory corruption bugs are still a big problem on all systems, including macOS, and companies try to stop them with new hardware and software defenses. Apple’s MIE is a special system built into the latest chips that should stop many memory attacks. Apple spent years and lots of money on MIE, and it has stopped many known hacks before.

The researchers found this bug by accident and used both human experts and an AI tool called Mythos Preview to build their attack. The exploit starts with a normal user and ends with full system control (root shell), even with MIE turned on. They worked fast—finding the bug in late April and making the exploit by May 1st. The attack uses two different bugs and clever techniques, but they will only share full technical details after Apple fixes the problem. The team says this is the first time anyone has shown a public exploit that works on MIE hardware for macOS. They also point out that while MIE is strong, it is not perfect—if you find the right type of bug, you can still get through. They believe AI will help find more of these bugs in the future, and that security teams need to prepare for this.

Some say this shows how strong attackers can become when they use both AI and human skills together. Others worry that if small groups can do this, it might become much harder for companies to keep systems safe in the future. A few commenters discuss whether Apple’s approach—putting so many protections in hardware—is really the best way, or if it just makes things slower for normal users. ## RTX 5090 and M4 MacBook Air: Can It Game?

- 原文链接: [RTX 5090 and M4 MacBook Air: Can It Game?](https://scottjg.com/posts/2026-05-05-egpu-mac-gaming/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48137145)

This article is about connecting a powerful desktop GPU, the RTX 5090, to an M4 MacBook Air to see if it can run modern games and AI tasks well. The author uses a special setup involving Thunderbolt eGPU docks, Linux virtual machines, and some custom software tricks.

The MacBook Air doesn’t support NVIDIA GPUs natively on macOS, so the author runs Linux in a virtual machine and passes the GPU through to it. There are technical problems, like mapping the GPU’s memory into the VM and handling direct memory access (DMA). The author creates a new virtual PCI device and guest driver to work around the Mac’s hardware limits, such as a 1.5GB DMA mapping cap. They also patch the Linux NVIDIA driver using a tool called kprobes to fix alignment errors.

Performance is a big concern. Running Windows games on the MacBook Air needs several layers: Linux VM, FEX emulator for x86 code, and Proton for Windows compatibility. This costs about 50% of the CPU speed compared to running native code. The Thunderbolt connection itself also makes the GPU run about 20% slower compared to a direct PCIe slot on a desktop.

Game benchmarks show mixed results. At low resolutions like 720p, the Mac’s own GPU is sometimes faster than the eGPU setup, because the overhead from emulation and virtualization is too high. At higher resolutions like 4K, the eGPU helps a lot—the MacBook Air goes from “unplayable” to “playable” frame rates in games like Cyberpunk 2077 and Shadow of the Tomb Raider. Still, a normal gaming PC with the same GPU is about 2–4 times faster.

AI inference works much better with the eGPU. The RTX 5090 gives huge speedups for local LLM (large language model) use, with much faster token generation and better concurrency than the Mac’s own GPU. However, there are still some software and hardware limits, and the setup is not easy to install or stable for everyone.

In the Hacker News comments, many people are impressed by the technical depth and creativity of the project. Some think it’s a fun “because I can” experiment, not a practical solution for most people. Others point out the high cost, complexity, and fragility of the setup, saying that buying a gaming PC or using cloud GPUs is easier. A few users explain that Apple’s hardware and software choices make eGPU support hard, and they hope future Macs will have better native support for gaming and AI.

Some commenters discuss the technical details, such as why DMA and BAR mapping are tricky on Apple Silicon, and how tools like FEX and Proton work. Others share their own eGPU experiences, noting similar problems with drivers and performance. A few are hopeful that with more ARM-native games and software, Macs could become better gaming and AI machines in the future.

Overall, readers agree this is a cool demonstration of what’s possible with enough effort, but not something most users should try—at least not yet. The main value is showing how far you can push Apple hardware, and maybe inspiring improvements in future Mac support for external GPUs and gaming.

## New Nginx Exploit

- 原文链接: [New Nginx Exploit](https://github.com/DepthFirstDisclosures/Nginx-Rift)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48138268)

A new exploit for Nginx, called “Nginx-Rift,” targets a serious bug (CVE-2026-42945) that lets attackers run code on web servers from far away. This bug is in Nginx’s rewrite module and has existed since 2008, putting many servers at risk if they use certain settings.

The problem comes from how Nginx handles URLs when using “rewrite” and “set” commands. Nginx tries to figure out how much memory it needs for a web request. But because of a flag mix-up, it asks for too little memory. Later, when it copies the data, it writes too much and overflows the memory buffer. An attacker can use this to control nearby memory, break a pointer, and finally run any command they want on the server. The exploit even gives an example that opens a shell on the target machine.

The bug affects Nginx open source versions from 0.6.27 to 1.30.0, and Nginx Plus versions R32 to R36. It was found by an automated security tool from a group called DepthFirst. They also give clear steps to reproduce the exploit using Docker and Python scripts.

Many people in the Hacker News comment section are worried because Nginx is so popular and runs on many websites. Some users are surprised that the bug was missed for 18 years and only now found by a tool, not a person. Others ask if most sites are truly at risk, since not all use the “rewrite” and “set” commands in the same way. A few point out that patching is easy if admins stay up to date, but many real-world servers run old software and might not get fixed soon. Some wonder if automated tools will soon find many more old bugs in big projects like Nginx and Apache. Others debate if open source software is safer or not, since anyone can read the code but bugs can still hide for years. Finally, some users thank the researchers for sharing the details and helping the community stay safe.

## Tesla Wall Connector bootloader bypasses the firmware downgrade ratchet

- 原文链接: [Tesla Wall Connector bootloader bypasses the firmware downgrade ratchet](https://www.synacktiv.com/en/publications/exploiting-the-tesla-wall-connector-from-its-charge-port-connector-part-2-bypassing)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48140953)

This article explains how security experts found a way to bypass Tesla Wall Connector’s firmware downgrade protection. Tesla uses two firmware slots in its charger: one active (running now), one passive (used for updates). Normally, when updating, the system checks the new firmware’s version and a “ratchet” number to stop users from downgrading to older, possibly unsafe versions.

In newer firmware (24.44.3), Tesla added a stronger check. Now, when trying to install old firmware, the system sees the lower ratchet number and deletes the firmware, blocking the downgrade. This check happens in the update process, not in the bootloader (the part that starts up the device).

However, the bootloader only checks if the firmware is signed and not changed. It does not look at the ratchet number. The trick is: after updating to the new firmware and setting the partition table (which tells the charger which slot is active), you can erase that slot and write an older, signed firmware into it—without running the version check again. On reboot, the bootloader sees a valid signature and runs the old firmware, because it does not know about the ratchet rule.

The exploit works by sending the new firmware first, updating the partition table, then erasing the slot and writing the old firmware. The device then reboots into the old version, bypassing Tesla’s protection. This process takes about 30 minutes.

The article suggests better ways to protect against this, like having the bootloader check the ratchet, or making the updater reboot right after a successful update, so the trick cannot work. Tesla was told about this problem and fixed it in a later update. Because Tesla updates chargers automatically, most devices are now safe.

In the comments, some people praise the deep technical analysis and the clever attack. Others worry about the danger: if someone hacks a wall charger, they could use it to attack other devices on the home or business network. A few readers point out this is a common problem when only part of the system (here, the updater) checks for security, and the bootloader does not. Some developers say the fix is simple—just check the version in the bootloader. Others ask if similar flaws could be present in other devices, not just Tesla’s. One commenter notes that automatic updates are good here, because they quickly fix problems. Another person says this shows why physical access is always a security risk. A few people discuss how hard or easy it would be for regular users to try such an attack, with most agreeing it is not simple, but possible with the right tools. Some also talk about how the update and boot process could be designed better to avoid this kind of bug in the future.

## RISC-V Router

- 原文链接: [RISC-V Router](https://router.start9.com/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48140541)

The article is about a new router called the RISC-V Router from Start9, aimed at making advanced networking and security easy for everyone. It stands out because it uses a RISC-V processor and has a very open-source design, which is rare for home routers.

Most routers on the market use closed hardware and software. The Start9 router uses open-source parts like the RISC-V instruction set, OpenSBI, U-Boot, Linux, and shares its board schematics. The only closed parts are the WiFi firmware (which is always closed on modern WiFi cards) and two early boot binaries, though the team is working to open-source those too. The router’s software, StartWRT, is a fork of OpenWrt with a modern, user-friendly GUI. This makes it easier for non-technical people to use features that are usually only for experts.

A big feature is Security Profiles. Each device on the network gets a profile that controls what it can do—like which devices it can see, which DNS it uses, and what kind of internet access it has (full, none, whitelisted, blacklisted, or through VPN). You can set these profiles differently depending on how devices join: through Ethernet, WiFi (each password gives a different profile), or VPN. For WiFi, instead of just “main” and “guest” networks, you can have many passwords, each tied to different permissions.

You can run as many inbound VPN servers as you need for remote access, and connect to multiple outbound VPNs, even chaining them together for better privacy. For example, if you use two VPNs in a chain, neither provider can know your full internet activity unless they work together. Other features include scheduling when WiFi is on or off, one-click dynamic DNS setup, and a “Help Mode” for explanations inside the interface.

People like the idea of Security Profiles and multi-password WiFi, since it makes it easy to share your network safely with guests or kids. A few comments ask about software updates and support, and others hope that Start9 will make it easy for users to contribute to the project.
