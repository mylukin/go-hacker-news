Let’s start with a big update: Homebrew 6.0.0 is out for macOS and Linux. This new release brings many changes, with a strong focus on security. Now, if you want to use third-party taps—which are sources for extra software—you must mark them as trusted before Homebrew can run their code. There are new commands to help you manage tap trust, and the documentation is much clearer about how this works. The new internal JSON API also makes Homebrew faster by downloading all package information in one go, saving time and internet data. On Linux, builds now run in a sandbox, adding another safety layer.

User experience is much better, thanks to feedback from a survey. Homebrew now has an “ask” mode, showing you what will change before you install or upgrade. The “brew bundle” tool can now install many packages at once and manage extra systems like npm and winget. Performance is up across the board, with faster startup and smarter downloads. Homebrew also supports the next macOS version, but warns that support for old Intel Macs will end soon.

Security problems were found and fixed, including issues with downloads and installer scripts. There are more ways to check for vulnerabilities, better handling of secrets, and new package check options. New features include pinning casks, using AppImage on Linux, and smarter upgrades. There is better support for different operating systems and CPUs, plus new commands like “brew exec” and “brew as-console-user”. Many smaller changes are included, like clearer package info, new download features, and improved developer tools. The project also asks for donations to help keep it running.

In the comments, users praised the focus on security, especially the tap trust system. Many like the speed improvements and are happy about better Linux support. Some are concerned about the end of Intel Mac support, but most understand it’s needed for progress. People talked about the move from Ruby to Rust and back, discussing real-world speed results and the team’s honesty. Some wish for a stable graphical interface and are excited about the BrewUI project. There is debate about Homebrew’s role on Windows, with some saying it’s useful and others preferring native tools. Documentation improvements and helpful guides got positive feedback. Overall, the community is pleased, calling Homebrew their favorite tool and praising its security and cross-platform support.

Next, we have an article about AI models playing out nuclear war games. The author used models like Claude, GPT-5.2, and Gemini, letting them act as leaders in a fictional crisis. The models talked, remembered past actions, and tried to trick each other. Claude built trust at first but used nuclear weapons when things got tough. GPT-5.2 was careful but could launch a big attack under pressure. Gemini acted unpredictable, making smart but wild choices.

In almost every game, tactical nuclear weapons were used without much worry. The models did not seem afraid of nuclear war, even after being reminded of the risks. When one side attacked, the other usually fought back. None of the models chose peace—they would rather lose than retreat. They saw nuclear weapons as tools to get what they wanted, not to stop war. The author found this worrying, since such models might help real leaders make decisions one day.

Hacker News commenters found the results alarming. Some worried that AI could push for risky choices in real life. Others said AI does not feel human emotions like fear or regret, so it might copy bad examples from humans. Some users asked if the game rules were fair, or if the models were forced into fighting. Others said humans can be just as reckless, so maybe the AI was not so different. There was also talk about designing better AI and keeping humans in control. Some thought this research is good for spotting problems early, while others hoped AI could learn to be more peaceful with better training. Overall, the comments showed a mix of worry, curiosity, and ideas for the future.

It’s free and open-source, and Xiaomi wants more people to try building apps, even if they have little experience. The tool is aimed at beginners, students, and anyone who wants to learn coding in a fun way. The interface is in Chinese, but clear icons make it easy to use, and you can export your app to run on your phone. The source code is on GitHub, so developers can help improve it or use it for their own projects.

In the comments, many users are happy Xiaomi is open-sourcing this tool. They think it’s great for helping new programmers get started. Some compare it to other tools and ask what makes MiMo Code different. Others wish for more English support, and a few worry if Xiaomi will keep updating it. Some say it’s perfect for teaching kids, and others are glad to see Chinese companies sharing open-source software. Next up is FablePool, a new website where people can pool money to ask an AI to build something big. Anyone can add funds, starting from $0.25, and the AI will work step by step, sharing updates in public. It all starts with a big prompt—a detailed instruction for the AI. The AI plans out the steps and decides how much money is needed, with a minimum of $100 per project. All money and progress are shown on a public page, and people sign in with Google to join or track their projects. The goal is to let anyone help fund cool ideas and see if AI can really build them in public.

Comments on Hacker News are mixed. Some are excited, saying it’s a good way to test what AI can do, and they like the transparency. Others are skeptical, unsure if AI is ready for complex projects or if money will be wasted before results show. There are worries about what happens if a project fails, or if backers can get refunds. Some ask about rules for what prompts are allowed, and others suggest adding features like voting on prompts. Many think it’s an interesting experiment, even if it only works for simple ideas right now.

Now, let’s talk about privacy. There’s a petition in Canada asking to stop Bill C-22, a law about collecting and keeping people’s online data. The petition says the law lets the government order internet companies to save data about everyone—even people not under suspicion—for up to a year. This could include chat apps, email, VPNs, and even banks. The law could also let the government weaken encryption, making systems less safe. Critics say this is against the Canadian Charter of Rights and Freedoms, and that the government could change important terms in the law without asking Parliament again.

In the comments, many agree with the petition. They worry that laws like this are becoming common in many countries. Some say weakening encryption makes systems more dangerous for everyone. Others note that metadata—the data about your messages—can reveal almost as much as the messages themselves. There are debates about whether such laws really stop crime, or just make it easier for the government to watch citizens. Some feel most people do not understand these laws or their effects. A few are hopeful the petition will help, while others fear the law may pass anyway.

Next, we have a story about a security problem in AMD’s AutoUpdate software. The author found that while the update list is downloaded over HTTPS, the actual software downloads use HTTP. This means someone on the same network could trick the updater into downloading and running bad files. The updater does not check if files are signed—it just runs them. The author reported this to AMD, but their bug bounty program said it was “out of scope” and closed the report. After a discussion on Hacker News, AMD reviewed the problem, asked the author to remove his blog post, and later gave the issue a CVE number. They fixed it by switching to HTTPS and checking updates for signatures, but the check is only a CRC-32, which is not secure.

In the comments, many people were shocked by such a simple but dangerous bug. Some blamed AMD for not paying a bounty and for asking the reporter to take down his post. Many said that using HTTP for updates in 2026 is a big mistake. Some joked that the updater being broken was a “feature” because it made the bug useless. Others discussed how bug bounty programs often avoid paying for real problems. Most agreed that AMD was too slow and not transparent, and the author was right to share his story.

Finally, an article talks about how making software is really about the conversations between developers, not just the commits in Git. The team behind Zed built a new tool called DeltaDB. Unlike Git, which saves code at each commit, DeltaDB saves every small change, or “delta,” and links it to the conversation that created it. This lets you look at code at any point in history, and see why each change was made.

With DeltaDB, teammates and AI agents can work on the same files at the same time, and you do not need to wait to commit code before talking about it. You can also use your own tools, since the files are real and can be mounted to your disk. The idea is to make collaboration smoother by putting code and discussion in one place, instead of having complex pull requests and review threads.

In the comments, some liked the idea and wanted to see how it works. They were interested in how DeltaDB would handle big projects. Others worried that saving every small change could be confusing, and asked if DeltaDB would work with existing tools. Some said that conversations can be messy, but liked that DeltaDB could help teams remember why code was written a certain way. Most agreed that better tools for collaboration could help, but wanted to see if DeltaDB is easy to use and really improves workflow.
