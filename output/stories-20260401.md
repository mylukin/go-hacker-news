## Signing data structures the wrong way

- 原文链接: [Signing data structures the wrong way](https://blog.foks.pub/posts/domain-separation-in-idl/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47605677)

This article talks about a common problem in cryptography: how to safely sign or hash data structures so attackers can't trick systems by swapping data types. It explains that just encoding and signing the data isn't enough—different types of messages can look the same in bytes, which can lead to real attacks.

The main issue is called "domain separation." This means making sure that when you sign a message, the signature can only be used for that exact message type, not for some other type that happens to have the same fields and values. The article gives examples, like "TreeRoot" and "KeyRevoke" messages, that could be encoded into the same byte stream, leading to confusion and possible attacks. This risk has caused problems in systems like Bitcoin, Ethereum, TLS, JWTs, and AWS.

The author says that current solutions are not systematic—they use tricks like adding method names or context strings, but these can be forgotten or misused. The article introduces a new idea, used in a system called Snowpack: every data structure that needs signing gets a random, unique number (a domain separator) added to its definition. When signing or verifying, the code always includes this number, so signatures can't be reused for a different type.

The compiler and runtime check that these domain separators are unique, and the system only allows signing if the type has a separator. The article also explains how this works in code: each struct has a method that returns its unique ID, and only those structs can be signed or verified.

The system also makes sure data is always encoded in a standard, predictable way (canonical encoding) using Msgpack, which prevents confusion or attacks based on different ways to write the same data. Developers are encouraged to generate these unique numbers randomly and keep them fixed, even if the structure changes.

In the comments, some readers agree this is a real and important problem, sharing experiences with bugs caused by missing domain separation. Others note that many developers don't think about this issue until something breaks, so having it enforced by the type system is a good idea. Some commenters point out that similar ideas have been used before, but not as systematically.

There is discussion about whether random numbers for domain separators are the best choice—some think hashes of type names or fields would be better for tracking, while others worry that could lead to collisions or accidental reuse. A few people wonder about the risks of AI code tools copying these numbers, and the article addresses this by saying the compiler checks for uniqueness.

Some readers are excited to see more languages supported and hope this becomes standard in cryptographic libraries. Others are more cautious, noting that enforcing such rules might make code harder to write in some cases, but agree the security is worth it. Overall, the comment section shows a mix of concern for real-world risks, interest in the technical solution, and curiosity about how it could be improved or adopted more widely.

## Show HN: Git bayesect – Bayesian Git bisection for non-deterministic bugs

- 原文链接: [Show HN: Git bayesect – Bayesian Git bisection for non-deterministic bugs](https://github.com/hauntsaninja/git_bayesect)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47557921)

This article is about a tool called “git bayesect,” which helps find which Git commit caused a bug when test results are sometimes random (flaky). The tool uses Bayesian inference, a math method that updates guesses based on new information, to find the commit where something changed—like when a test started failing more often.

With git bayesect, you do not need to know the exact chance of failure. You only need to notice that the behavior changed at some point. You start a “bayesian bisection” from an old commit to a new one. As you test each commit, you record if it passes or fails. The tool uses this information to decide which commit to check next. You can set “priors” to give extra weight to commits that look suspicious, like those with certain filenames or messages. There are commands to undo actions, reset, or run tests automatically. There’s even a demo that makes a fake repo with a flaky script to help you try it out.

This tool is useful when normal git bisect does not work well because the bug does not always appear. It’s written in Python and is easy to install with pip.

In the comments, some people are excited and say this tool fills a real need, especially for teams that deal with flaky tests. Others point out that Bayesian methods have been used in debugging before, but it’s still rare to see a simple tool like this. A few users share stories about spending hours or days tracking down random bugs, saying this would have saved them time. Some ask about how well it works with big projects or if it can handle very rare failures. There are questions about setting priors and if the tool can guess which files are riskier. A few users wonder if it could connect with CI systems or be used with other version control tools. One person worries that using Bayesian math might confuse developers who aren’t familiar with it. But most people agree this is a smart idea and hope it gets more attention. Some even ask for more documentation or examples to help them get started.

## The revenge of the data scientist

- 原文链接: [The revenge of the data scientist](https://hamel.dev/blog/posts/revenge/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47557385)

This article asks if data science is still important now that large language models (LLMs) are easy to use through APIs. Before, data scientists and machine learning engineers were key to building AI. Now, some people think their jobs are less valuable because teams can add AI to products without them. The author disagrees and says the real value of data science is not just training models. Most of the job is about testing, debugging, and measuring how AI works—things that still matter even with LLMs.

The article explains that good AI needs a “harness”—a system of tests, logs, and metrics to check if the AI is working well. Data scientists are important because they know how to build this harness. They focus on understanding data, designing the right metrics, and analyzing failures. The author lists five common problems (“pitfalls”) teams face when they skip data science basics:

1. Using generic metrics that do not fit the real problem.
2. Trusting LLMs to judge outputs without checking if the “judge” is reliable.
3. Building bad experiments, like using fake test data instead of real examples from users.
4. Not caring enough about data and labels, or letting unskilled people label data.
5. Trying to automate everything, when some things need careful human work.

The author says teams need to “look at the data” often, read logs, and do error analysis. Simple metrics like accuracy can hide important mistakes—so it’s better to use precision, recall, and custom checks. Data scientists also make sure test sets are real and cover edge cases. They get experts to label data and help teams decide what really matters for their product. The article ends by saying all these tasks are old data science skills, even if the tools and names change.

In the comments, some people agree that data science is still important. They say that plugging in an API is not enough if you want quality results. Others think the role will change, but not disappear. They point out that business leaders may not understand why these steps matter until problems show up in production. Some readers think automation will take over more parts of the job, but agree that human judgment is needed for the hardest problems. A few commenters worry that companies want to cut costs and may skip careful data work, leading to worse products. Another group says the best data scientists are good at both technical work and understanding business needs. Some share stories about teams failing because they trusted metrics that did not fit their real goals. Others say that data science is becoming more about communication and less about technical tricks. Overall, most agree that “looking at the data” and asking good questions will always be needed, even as AI tools get better.

## Scientists crack a 20-year nuclear mystery behind the creation of gold

- 原文链接: [Scientists crack a 20-year nuclear mystery behind the creation of gold](https://www.sciencedaily.com/releases/2026/03/260313002633.htm)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47605374)

Scientists have solved a big question about how gold forms in the universe, a problem that lasted for 20 years. The study explains new details about how heavy elements like gold are made when stars explode or crash into each other.

The team found three important things. First, they measured the energy of two neutrons leaving an unstable atom, something never done before. They worked with a rare type of atom called indium-134, made at CERN, and watched it break down into tin atoms. When indium-134 decays, it can release one or two neutrons, but it was hard to tell before because neutrons are tricky to track. Using new detectors, they saw how two neutrons come out and measured their energy. This helps us understand the “rapid neutron-capture process,” or r-process, which makes heavy elements in space.

Second, they observed a special state in tin-133 that scientists had predicted but never seen. Usually, when the tin atom cools down, it loses its memory of how it was created. But the team saw that the tin “remembers” its past, which means current models don’t tell the full story. This new state is a missing piece in the puzzle of how atoms behave after a star explosion.

Third, they noticed that the way the tin-133 state is filled during decay doesn’t follow normal patterns. This was unexpected and suggests current theories might not work for strange, rare atoms. The results mean that scientists must rethink how they model nuclear changes in extreme parts of the universe.

Early-career scientists, like Peter Dyszel, played a big role in the experiments. He helped build detectors, set up the tests, and analyze the results. His story shows how curiosity and teamwork are important in science.

Hacker News commenters were excited by the discovery and said it is a big step for nuclear physics and understanding the universe. Some pointed out that the experiment was very technical and praised the use of new detectors and teamwork at CERN. A few users explained how the r-process is important for the gold in everyone’s jewelry, making the science feel personal. Others asked if these results could help us make gold on Earth, but most agreed it is not practical—these cosmic events are too extreme to copy.

Some people wondered what this means for physics models and if we need to update textbooks. Others discussed the long search for the missing neutron state in tin-133 and how it shows science is sometimes slow but worth it. A few mentioned how the work inspires young scientists and that real progress takes years of careful study. There were also comments about how studying rare atoms helps us understand not just gold, but many heavy elements in the universe. Finally, some users joked that now we know “where gold comes from,” but agreed this knowledge is still amazing and important.

## Jax's true calling: Ray-Marching renderers on WebGL

- 原文链接: [Jax's true calling: Ray-Marching renderers on WebGL](https://benoit.paris/posts/jax-ray-marcher/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47605494)

This article talks about using JAX, a Python library, to make fast 3D graphics in the browser with WebGL. The author shows how you can use JAX and math functions to create a simple ray-marching renderer that draws shapes like spheres and cubes.

Instead of drawing shapes with polygons, the method uses Signed Distance Functions (SDFs). An SDF is a math function that tells you how far a point is from the surface of a shape. If the value is negative, the point is inside; if positive, it’s outside. This lets you build shapes easily with functions, and you can combine shapes by using min and max functions. For example, to join two shapes, you take the minimum of their SDFs.

With JAX, you can run these calculations fast, because it can work on the GPU and can process all pixels at once. The code is short—about 100 lines of Python. JAX can also export code to run in the browser using WebGL, thanks to tools like TensorFlow.js.

A big benefit is that you can get the surface normal (the direction the surface points) easily by using the gradient of the SDF. This helps with lighting and reflections in the renderer. The author likes how using JAX makes the code look a lot like math, and most of the code is just math functions.

At the end, the author lists other ideas, like trying WebGPU with jax-js, using geometric algebra, or exploring the connection between neural networks and SDFs. They also mention using advanced features in JAX to make even faster or smarter renderers.

In the comments, some people are impressed that JAX can be used for graphics, not just for machine learning. Others like seeing math and graphics come together in code that is simple and easy to understand. A few users point out that JAX’s ability to export to the browser is very cool, though they wonder about performance limits compared to real shader code.

Some commenters discuss SDFs and how they make building complex scenes easier than using polygons. There’s excitement about using gradients for lighting. A few people share their own projects with SDFs and ray-marching, and some post links to tools like ShaderToy.

Others are curious if JAX is the best tool for this or if using GLSL (the shader language) would be faster. Some warn that using Python in the browser can be slow for big scenes, but agree it’s great for learning.

## EmDash – a spiritual successor to WordPress that solves plugin security

- 原文链接: [EmDash – a spiritual successor to WordPress that solves plugin security](https://blog.cloudflare.com/emdash-wordpress/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47602832)

EmDash is a new open-source content management system (CMS) built by Cloudflare, aiming to replace WordPress and fix its plugin security problems. The article explains that WordPress is old, powers a big part of the web, and has security issues—especially with plugins.

EmDash is built with TypeScript and is serverless, meaning you do not need to manage servers. You can run it almost anywhere, including your own servers or Cloudflare’s platform. The main change is how EmDash handles plugins. In WordPress, plugins can access everything and often cause security problems. In EmDash, each plugin runs in a safe, separate sandbox. Plugins have to say what they want to do in a manifest file, and they can only do those things. For example, if a plugin wants to send an email, it must ask for permission to do only that, and nothing else.

This model also means that plugin authors can pick any license they want, unlike WordPress, which often requires the GPL license. Also, plugins do not have to go through a slow, central review process, since the security model makes them safer by design. EmDash supports charging for content using the x402 standard, so site owners can get paid for their work without setting up subscriptions. It scales to zero—meaning it only uses resources when people visit the site—which saves money.

Themes in EmDash use Astro, a modern frontend framework. This makes it easier for web developers to create themes and keeps the sites safe, since themes do not have access to databases or server code. EmDash is also “AI native:” it has tools for AI agents to help with tasks like moving content or setting up new sites. Authentication uses passkeys by default, making it safer than passwords. You can move your old WordPress site into EmDash, including custom content types and media.

In the Hacker News comments, some users are excited about the strong plugin security model, saying it could fix one of WordPress’s biggest problems. Others like that EmDash is serverless and open source, making it easy to try and cheap to run. Some developers are happy about using TypeScript and Astro, as these are modern and popular tools. A few people are interested in the x402 payment system, but wonder if it will catch on with real users.

However, some are cautious. They say WordPress’s success comes from its huge library of plugins and themes, and EmDash will need time to build a similar ecosystem. Others note that moving big, old WordPress sites might be hard, especially if they use many custom plugins. Some point out that serverless may not work for everyone, especially in countries with slow Internet or for sites with special needs. A few worry about relying on Cloudflare, but the open-source license helps.

There are also questions about the real security of sandboxed plugins, and if clever attackers might find holes. Some ask if plugin developers will really follow the rules, or if there will still be risky plugins. People like the idea of breaking marketplace lock-in, but wonder if users will trust plugins that are not reviewed by a central team. Finally, some are just glad to see new ideas in the CMS space, even if they are not ready to switch yet.
