First, we look at Kage, a new tool that lets you save any website for offline viewing by removing all JavaScript. With Kage, you can copy a website as it looks after loading, keeping CSS, images, and fonts, but deleting scripts so nothing runs in the background. Kage works by using Chrome or Chromium to load the page, waits until everything settles, then takes a snapshot of what you see. It deletes all scripts and saves local copies of assets, so you can open the saved pages later, even if the website changes or disappears. The saved files are just plain HTML and assets, so you don’t need to worry about trackers or broken links.

To use Kage, you can install it with Go, download a binary, or run it in a container. The main command is “kage clone <url>”, which saves the site. You can preview the copy offline with “kage serve”, or use “kage pack” to make a single file archive (ZIM) or a self-contained executable. Kage also has options for saving more pages, following links, or including subdomains, and it politely reads robots.txt. The ZIM format is popular for offline content, and you can use your saved file in apps like Kiwix. If you build Kage with a special option, the saved site can even open in its own window.

Many like that Kage is open source and hope it keeps being updated. There are technical suggestions too, like supporting more browsers. Overall, the reaction is very positive.

Next, let’s talk about Zinnia, a new operating system kernel, like Linux, but written almost fully in Rust. The main goal is to make a safe, modern, and modular Unix-like system that uses as little unsafe code as possible. Zinnia supports many POSIX system calls, so it can run a lot of software that works on Linux or BSD. It also includes modern features like epoll and timerfd. Zinnia can run a desktop using Wayland or X11, so you get a familiar graphical interface.

Device drivers in Zinnia are loaded as separate modules written in Rust, using the ELF file format. Zinnia uses the Limine bootloader and works on any UEFI computer, which is most new computers. The project started in 2024 as a way for the creator to learn more about operating systems. Now, Zinnia can run on many real x86_64 computers, with plans for ARM and RISC-V in the future. The creator welcomes help from other developers.

In the comments, users are excited to see a kernel written in Rust, since Rust can prevent many bugs common in C code. Others point out that making a new OS is very hard, and it will take a long time to catch up to Linux or BSD. Some ask about performance, since Rust’s safety checks can slow things down, but others say it’s worth it for safer code. Many like the choice to load drivers as modules, making the system flexible. Some wonder how much software will run on Zinnia, since many programs expect Linux features. There are comments about needing good documentation and ways to contribute. Some people offer to test on different hardware. Overall, the community is supportive and interested to see how Zinnia will grow.

Now, let’s look at yserver, a new X11 server for Linux, written in Rust. The goal is to make a modern, practical X11 server that works with today’s desktops and apps, but without supporting old features most people no longer use. Yserver drops things like support for multiple screens, non-TrueColor visuals, indirect GLX, and the old DDX driver ABI. It still runs full desktop environments like MATE, XFCE, Cinnamon, and window managers like FVWM3 and Window Maker. The server supports many X11 extensions, including GLX, Composite, RandR, and XInput. GLX_EXT_texture_from_pixmap works on AMD, Intel, Asahi, and Qualcomm GPUs, but not with Nvidia’s proprietary driver.

Yserver has been tested on many types of hardware and works with modern Linux graphics and seat management. You can build and run yserver with Rust and some system libraries, and there are install instructions for major Linux distributions. Yserver can run as the display server for LightDM or be started from the console, with handy shortcuts for screenshots. The project checks its protocol support with the X.Org test suite and is open source under the MIT license.

In the comments, many users are excited to see a new X11 server in Rust, hoping it will make X11 safer by avoiding memory bugs. Others like the modern codebase without old features. Some worry that X11’s future is limited because Wayland is becoming more popular, and ask if it makes sense to put effort into X11 instead of Wayland. There are concerns about driver support, especially for Nvidia, and about breaking compatibility with some old apps. But many agree that dropping unused code makes maintenance easier. There are also suggestions for testing and contributing, and some users share their own experiences with X11. Overall, the community shows a mix of excitement, curiosity, and cautious optimism.

Now, let’s talk about the history of networks. Chaosnet was a local area network built at MIT’s AI Lab to connect computers in the same building. There was no central controller—each computer could talk to others directly over a shared cable. The system was designed for Lisp machines, with every user having their own computer, but sharing files and printers over the network. Chaosnet used coaxial TV cable as its main wire, and each computer had special hardware to connect. Computers sent packets to each other, with sender, receiver, and check values for errors. If two computers sent at the same time, their messages could crash into each other—a “collision.” The system used tricks to avoid or recover from collisions.

The software handled how programs on different computers talked, making sure messages arrived correctly and in order. It had simple ways to start and stop conversations, keep track of who’s talking, and make sure packets didn’t get lost or doubled. Chaosnet could connect many types of computers, using bridges to link cables and routing tables to find the best path. The design avoided long-distance links or complex security, focusing on speed, reliability, and easy repairs. Programs could ask for services from any computer, using a system of requests and answers like today’s client-server setups.

In the comments, many people praise Chaosnet’s simplicity and its influence on later networks like Ethernet and TCP/IP. Some remember using Chaosnet in the 1980s and say it was quick and easy to use. Others compare it to modern protocols, noting that Chaosnet avoided complexity by focusing on local connections. Some say the lack of security would not work today, but it helped keep things fast then. Commenters share stories about debugging with network lights and test buttons, or how Chaosnet could recover from problems without crashing everything. There’s debate about whether its collision-avoidance ideas could work in bigger networks, and some say Chaosnet’s design is a reminder that simple tools can be best for small problems.

Now, we have Trace, a new Mac app that records and transcribes your meetings directly on your computer. The main idea is privacy: it never uploads audio or transcripts to the cloud—everything stays on your Mac. The app works with any meeting software, capturing both your microphone and system audio. Transcription happens locally, and you get a clean markdown transcript, which is easy to use in Notion, Obsidian, or with AI tools like ChatGPT.

During a call, you can press a shortcut to flag important moments and add notes. These notes appear in your transcript with the right timestamp. You can quickly view the transcript of the last two minutes, including flagged moments. Trace lives in the menu bar, with all actions available by shortcuts. Each meeting session is saved as audio and markdown files in Finder, and you control your data. The app can connect to your calendar to name recordings and remind you before meetings. It identifies different speakers and marks who said what. It asks for audio recording permissions, but never accesses your screen or sends data anywhere.

Let’s move to segmented type. This is a style of writing numbers and letters using straight lines, like you see on digital clocks or train station signs. Segmented type is made by lighting up certain lines to form each character. It’s popular in digital watches, calculators, and display boards because it’s easy for machines to show and read, even from far away or in bad lighting. Each number or letter is broken into simple segments. For example, “8” uses all the segments, while “1” uses just two. This style is not the most beautiful, but it is practical and lasts a long time in busy places.

In the comments, some people feel nostalgic, remembering old electronics from childhood. Others note that segmented type is simple, but can be hard for all letters, so it works better for numbers. Some enjoy the design, saying it gives a retro look to signs and devices. Some note that today’s screens can show much more detailed fonts, so segmented type is used less now, but it’s still best for some tools because it’s cheap and strong. Some like to make their own segmented displays as a hobby. A few call it “ugly” or too simple, but admit it works well. Overall, people have mixed feelings—some love the style, others see it as only useful.

Finally, let’s look at zeroserve, a fast HTTPS server that now supports Caddy configuration files. Zeroserve reads a Caddyfile, turns it into eBPF code, then into machine code, and runs it in a special fast event loop. The main idea is that zeroserve can now work like Caddy, but it is much faster. In benchmarks, zeroserve with clang reaches almost 39,000 requests per second, while Caddy handles about 12,500. Zeroserve’s median response time is 1.45ms, much faster than Caddy’s 4.74ms, and it uses less memory.

You can try zeroserve by downloading it and running it with your own Caddyfile. Zeroserve can run custom eBPF code, so you can do advanced things like signing requests for AWS S3. There is an example Caddyfile for using zeroserve to reverse proxy an S3 bucket with authentication.

In the comments, people are impressed by the performance and technical work, and think using eBPF in userspace is clever. Some point out that for small websites, these speed gains may not matter much. Some worry about security, since running custom eBPF could add risks. Others ask if zeroserve will support more Caddy features, or how easy it is to use. Some developers are excited about using zeroserve for custom middleware or high-traffic APIs. There are questions about stability and memory use. Some want to see benchmarks on more hardware or with more complex setups. Overall, many are curious and want to try zeroserve, but some will wait for it to mature.
