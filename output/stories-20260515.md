## WinCE64 – Windows CE 2.11 for N64

- 原文链接: [WinCE64 – Windows CE 2.11 for N64](https://github.com/ThroatyMumbo/WinCE64)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48152729)

This project shows how someone got Windows CE 2.11 to run on a real Nintendo 64 game console. The work is a hobby project, not something official from Microsoft or Nintendo, and it uses a lot of custom code to make it all work.

The main idea is to take the Windows CE kernel and get it working with the N64’s MIPS processor. The project uses a special piece of hardware called EverDrive-64 X7 to load files from an SD card. The N64 controller acts like a mouse, and you can see a desktop, open windows, and launch programs—just like a simple Windows computer. The system can play sound through the N64’s audio chip, and you can even run some third-party Windows CE apps straight from the SD card.

To do this, the creator had to write many custom drivers for things like the display, SD card, mouse/keyboard, and sound. There is even a special 3D demo that draws shapes using the N64’s graphics chip. The project does not include any Microsoft or Nintendo files—users must find these themselves. To build everything, you need old Windows CE tools, a special homebrew toolkit, and some open-source libraries. The steps to build and run the system are described, but you need real N64 hardware to use it fully.

The FAQ explains that there is no ready-to-use ROM to download because of licensing rules. The project is just for fun, inspired by another old device that used a similar CPU. The desktop shell (the main desktop, taskbar, and file browser) is all original code, since the Windows CE SDK did not provide one. Most of the system beneath the Windows API is also custom-built for the N64.

In the Hacker News comments, many people say this is a fun and impressive project, even if it has no practical use. Some remember using Windows CE devices long ago and find it cool to see it on a game console. Others talk about the technical challenges, like writing device drivers and handling the N64’s hardware quirks. A few users wish Microsoft would open-source old software like Windows CE, so projects like this would be easier or legal to share. Some wonder why anyone would do this, but most understand it is for the challenge and learning. A few point out that the N64’s hardware is quite limited, so getting a desktop system running is quite a feat. Some users ask about emulation, but the author says it works best on real hardware, not in emulators. Finally, there is praise for the detailed documentation and for keeping the code open (as much as possible) under the MIT license.

Mitchell Hashimoto says that many companies are acting strangely about AI, almost like they are not thinking clearly. He worries that it is very hard to talk honestly about AI risks with these companies, even when his friends are involved.

He remembers when tech teams argued over how to build strong systems—should you avoid all failures, or just fix failures very quickly? Now, he sees that same argument coming back, but with AI. Some people say it is fine if software has bugs, because AI tools can fix them faster than humans ever could. Mitchell says this is a mistake. In the past, when building cloud systems, people learned that fixing problems quickly is good—but you still need to build systems that do not break all the time. If you only focus on fixing things fast, you might miss deeper problems.

He is worried because some teams point to things like "full test coverage" or "fewer bug reports" as proof that everything is fine. But these numbers can hide bigger, hidden risks. Sometimes, systems look healthy on the surface, but are actually getting worse underneath. Test coverage can go up, but the real understanding of the system can go down. Bugs might seem to disappear, but the risk is just not visible yet. If changes happen too fast, the whole structure can fall apart before anyone notices.

In the Hacker News comments, some people agree with Mitchell. They say that companies are rushing to use AI everywhere, sometimes without thinking about the long-term risks. They worry that too much trust in AI could lead to big mistakes later.

Others say that every new technology causes this kind of excitement and fear. They think that people will learn over time, just like when cloud computing started. Some believe that focusing on fixing problems quickly (MTTR) is not so bad, as long as you still pay attention to deeper system health.

A few commenters point out that AI tools really do help find and fix bugs faster, and that this is a good thing. They say that teams should use these new tools, but stay careful and watch for hidden risks. Some people share stories about how their company is pushing AI too hard, and it feels uncomfortable. Others say that change is always hard, but it is not always bad.

Some comments warn that relying too much on testing numbers or bug counts can be dangerous, because those numbers do not always show the real picture. But a few people say that this is just part of progress, and mistakes will help everyone learn. Most agree that it is important to talk about these risks openly, even if it is difficult.

## The Zulip Foundation

The founder expects work might slow down for a short time as the team adjusts but says Zulip is more stable than ever.

It also opens new ways to raise money, like grants and donations, that were not possible for a regular company. The founder says this change means Zulip’s users can trust the service will not sell their data or add unwanted ads in the future.

Tim Abbott explains he is leaving because he wants to work on AI safety at Anthropic but made sure Zulip can do well without him. Most of the Zulip team will stay and keep building the product. They are hiring to replace the people leaving, and the founder will still help sometimes as an advisor.

The Zulip team invites users to ask questions and join a live Q&A chat about the changes. They want to be open about the transition and are happy to talk with anyone interested in helping or learning more.

A few people wondered if the leadership change might slow development or cause problems, but others pointed out that Zulip’s team is experienced and has kept things running smoothly during past challenges. Some users asked about the long-term funding and if the foundation could survive on donations and grants alone. Others suggested this move could make Zulip more attractive to big organizations and open-source projects looking for a stable, open chat platform.

There were also technical comments praising Zulip’s unique threading system and its usefulness for big communities. Some shared their positive experiences using Zulip, while a few offered advice about running a nonprofit and keeping the community strong. Overall, most commenters were hopeful for Zulip’s future and thankful for its focus on values and open development.

## A 0-click exploit chain for the Pixel 10

- 原文链接: [A 0-click exploit chain for the Pixel 10](https://projectzero.google/2026/05/pixel-10-exploit.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48148460)

This article explains how researchers found a new way to hack the Pixel 10 phone without any user action—just like they did with the previous Pixel 9. They first updated their old exploit for a Dolby audio bug to work on Pixel 10, changing some technical details because the phone’s software had changed a bit. But the old method for getting full control (using the BigWave driver) didn’t work on Pixel 10, because that driver was gone. Instead, the researchers found a new driver called VPU, used for video processing.

When they looked at VPU’s code, they found a simple but very serious bug. This bug lets any app map (or copy) big parts of the phone’s memory into its own space, including areas where the operating system (the kernel) lives. Because the memory addresses are always the same, anyone using this bug could quickly find and change important system code. This could let them fully control the phone. The exploit for this bug was very easy to write—just five lines of code could give an attacker full access.

The good news is that the bug was reported quickly, rated as a high severity issue, and patched within 71 days—much faster than similar bugs in the past. The article says this is a sign that Android is getting better at fixing security problems fast. But, it also shows that simple, dangerous bugs still make it into important parts of the phone’s software, and that companies need to work harder to check their code and prevent mistakes like this before phones are sold.

In the comments, some people are shocked by how simple the bug is and wonder how it got missed during code reviews. Others say that even though the bug is bad, it’s good to see Google patching it so quickly this time. A few users talk about the risks of hardware drivers in phones, saying that they are often rushed and not checked enough for safety. There are also people who point out that the same group of developers made both the old and new buggy drivers, suggesting that better training or rules are needed for these teams.

Some commenters argue that the way Android allows apps to interact with hardware is risky and needs a redesign. Others defend the Android team, saying that with such complex software, some bugs are always going to get through, but fast patching helps. A few give technical tips on how to spot this kind of bug, while others share worries about how easy it is for attackers to use this exploit. Overall, the main feeling is that while progress is being made, there is still a lot of work to do to make phone software safe for everyone.

## California bill would require patches or refunds when online games shut down

- 原文链接: [California bill would require patches or refunds when online games shut down](https://arstechnica.com/gaming/2026/05/bill-to-keep-online-games-playable-clears-key-hurdle-in-california/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48152994)

California is pushing a new law that would make game publishers refund customers or give them a way to keep playing if an online game shuts down. The bill, called the Protect Our Games Act, just passed a key vote and could soon be decided by the full California Assembly.

If the law passes, game publishers must either offer a patch to allow single-player or local play or pay players back when they stop running game servers. They must also warn players 60 days before ending these online services. The rule would not apply to free games or games only available through a subscription. Only games sold in California after January 1, 2027, would be affected.

The bill comes after players got upset when Ubisoft shut down the game “The Crew” in 2024, making it unplayable. A group called Stop Killing Games, which formed after that event, helped write the bill. They argue that people pay real money for games, so it’s unfair when companies take games away without warning or a backup plan. They say no other product is sold this way.

On the other side, the Entertainment Software Association (ESA), which speaks for big game companies, says the law misunderstands how game sales work. When you buy a game, you usually get a license to use it, not full ownership. The ESA also says it's normal for old games to close, especially those needing lots of online support. They argue that forcing publishers to keep games alive or give refunds could be impossible, especially if music or other parts of the game are only licensed for a limited time.

The bill still has a long way to go before it becomes law. It needs to pass more votes and get the governor’s signature. But supporters are excited, especially since similar efforts in the UK have stalled.

In the Hacker News comments, some people feel this law is needed to protect consumers. They are tired of paying for games that can vanish at any time. Others think the law might be too strict and could hurt small game studios, which may not have the money or rights to keep games running forever. Some worry about what happens if a company goes out of business and can’t offer refunds or patches. A few think the law could push publishers to make more games fully offline, which could be good for preservation. Others believe that better licensing deals could be made so games can live longer, but that could raise costs. Some commenters say players should know the risks when buying online-only games, while others argue that companies should be more honest about these risks. There’s also talk about whether this law would even work outside California, since games are sold everywhere. Finally, some hope this bill will make companies think twice before shutting down games and might lead to more respect for digital ownership.

## Microscale Thermite Reaction

- 原文链接: [Microscale Thermite Reaction](https://sciencedemonstrations.fas.harvard.edu/presentations/microscale-thermite-reaction)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48153142)

This article explains a small and safe way to show the thermite reaction, using two rusty iron balls, with one wrapped in aluminum foil. The reaction happens when you hit the balls together, making sparks and loud noises.

The thermite reaction is a process where aluminum reacts with iron oxide (rust) to make aluminum oxide, iron, and a lot of heat. The chemical formula is 2Al + Fe₂O₃ → Al₂O₃ + 2Fe + heat. In this demo, the energy from hitting the balls starts the reaction. The reaction gives off so much heat that the iron melts, and this makes bright yellow sparks fly out. The yellow color comes from hot iron pieces burning in the air. The noise is not just from the impact, but also from the reaction itself.

This reaction is very hot—about 2200°C, which is enough to melt iron. The classic thermite reaction is used to weld train tracks and even for underwater welding. The small version in the demo is much safer, but you still need to wear safety goggles and a lab coat, because the sparks can go far. The balls can be reused, and if the rust is gone, you can make more by leaving the balls in salty water.

You need to use a hard, glancing blow to get the best reaction. Only the instructor should do this demo, because it can hurt your arms and shoulders if you do it too much. You can even put a piece of paper between the balls to burn holes in the paper with the sparks.

In the comments, some people are amazed by how a simple setup can show such a powerful reaction. Others remember doing similar experiments in school, but note that safety is very important. A few say that even though this is safer than the big thermite reaction, you still need to be careful with hot sparks. Some users suggest other metal combinations for similar reactions, while others share tips for getting the rust back on the balls if it wears off. There are also jokes about using this to light a barbecue, but most agree it is best kept as a science demo. A few commenters explain more about the chemistry, saying the reaction is fast because the balls are hit hard, and the energy from the hit is important. Some worry that people might try this at home without enough safety, but most like how this shows real science in a simple way.

## Waymo updates 3,800 robotaxis after they 'drive into standing water'

- 原文链接: [Waymo updates 3,800 robotaxis after they 'drive into standing water'](https://www.cnbc.com/2026/05/12/waymo-recalls-3800-robotaxis-after-able-drive-into-standing-water.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48151767)

Waymo had to update and recall about 3,800 of its robotaxis because some cars were driving into flooded roads. This recall happened after several Waymo cars in Austin, Texas, and other places were seen going into standing water and sometimes getting stuck.

The problem was with Waymo’s software in their fifth and sixth generation self-driving systems. The cars did not always see flooded roads as a danger, so they would keep driving, even if it wasn’t safe. In one case, a Waymo car in San Antonio went into deep water and was carried away by a creek. All the cars in this recall are being updated so the software will notice flooded roads and avoid them.

Waymo said it already made some changes to the software and is working on more fixes. The company also decided to stop its robotaxi service in San Antonio for now until things are safe again. For now, Waymo cars will not drive in places where heavy rain or flooding is expected, and they will avoid flooded areas.

Waymo says safety is the most important thing. The company gives more than half a million rides every week in busy cities like San Francisco, Los Angeles, Phoenix, Austin, and Miami. Some people can use the service in other cities, but only if invited.

Some people on Hacker News are glad Waymo found the problem and is fixing it fast. They think software updates for self-driving cars are a good way to keep people safe, and they like how problems can be fixed over the air, without needing to take the car to a shop. Others, however, worry about these kinds of mistakes, especially if robotaxis become more common. They ask why the cars did not see the water or know to stop, since even regular drivers avoid big puddles.

A few commenters say driving into standing water is dangerous for any car, not just self-driving ones, but a human would probably be more careful. Some people are surprised this kind of mistake still happens after so many years of testing. But others point out that even human drivers make mistakes, and self-driving cars are still learning to handle rare problems like sudden floods.

Some think Waymo should test even more in bad weather and unusual situations. A few are concerned that too many mistakes could slow down the use of robotaxis everywhere. Others believe each fix makes the whole system better and safer for the future. Overall, there is a mix of hope and worry about the future of self-driving cars.
