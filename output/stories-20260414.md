## Claude Code Routines

- 原文链接: [Claude Code Routines](https://code.claude.com/docs/en/routines)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47768133)

Claude Code Routines let you save tasks that run in the cloud, so they keep working even if your laptop is off. You set up a routine with a prompt, code repositories, and connectors to other tools, and then choose when or how it should run—on a schedule, by API call, or when something happens in GitHub.

The article explains that routines help you automate work like labeling GitHub issues, triaging alerts, reviewing code, checking deployments, updating docs, or porting code between libraries. For example, you could set a routine to scan new issues every night, tag them, assign people, and post a summary to Slack. Or you could have a routine that checks alerts from your monitoring tool, finds the cause, and even opens a pull request with a possible fix.

You can start a routine in three ways: on a schedule (like every night), by sending an API call (for example, from a script or alert system), or by linking it to GitHub events (like when a pull request opens). You can use more than one trigger for each routine.

To set up a routine, you can use the web, the desktop app, or the command-line interface. All methods connect to your account in the cloud. The web and desktop app let you add all trigger types, but the CLI can only set up scheduled routines. You can pick which triggers, repositories, and connectors to use, and routines always run as your user account—they are not shared with teammates.

Triggers are easy to set up. A scheduled trigger runs at times you pick. An API trigger gives you a link you can call with a token to start the task. A GitHub trigger starts the routine when a chosen event happens in your repo, like a PR or push. You can filter which PRs trigger actions, for example by author, label, branch, or draft state.

When a routine runs, you can check what it did, review changes, or continue working with it. You can edit, pause, or delete routines from the detail page. You control which branches routines can push to, and you can add or remove connectors for things like Slack or Linear. Each routine runs in a cloud environment, which you can configure to allow or limit network access and set variables.

Routines count toward your usage limits, and there’s a daily cap on runs, unless you pay for extra usage. You can see your usage in your account settings.

In the Hacker News comments, some users liked how routines make automation easier, especially for regular code reviews or alert handling. They saw value in running tasks even when their own computer is off. Others wondered about security, since routines can push code or send messages as your user—if something goes wrong, it could cause trouble. A few users wished routines could be shared across a team, not just per user, to help with collaboration. Some were excited about using routines for DevOps or to keep documentation up to date. Others asked how pricing and usage limits work, and if routines could be triggered fast enough for real-time needs. A few people compared this to GitHub Actions or Zapier, but said Claude routines are easier for some common coding tasks. Some users warned to be careful with permissions, especially when routines can make real changes to code or send alerts. There was also talk about how easy it is to set up routines from the CLI, but some want more features there. Overall, people saw routines as powerful, but said you should start small and be careful with what you let them do.

Jacobs knew his tapes might break down over time, so he agreed to let the Internet Archive digitize them. So far, 2,500 tapes are online, including some unique recordings like a Nirvana concert from 1989, before they became famous. There are also shows from artists like Phish, Liz Phair, Pavement, Neutral Milk Hotel, and many punk bands that are hard to find anywhere else.

The tapes were not recorded with fancy gear, but volunteers at the Internet Archive worked hard to make the music sound good. One volunteer, Brian Emerick, drives to Jacobs’ house every month to pick up more tapes. He uses old cassette players to copy the music into digital files. Other helpers clean up the sound, fix the song names, and organize everything online.

The project is a big effort by fans and volunteers. Some recordings even include hidden gems, like a Tracy Chapman show from 1988. The goal is to save music history and make it easy for anyone to listen.

People on Hacker News had many views about this. Some are excited and thankful that these concerts are now saved and shared for free. They say it is important to save music that could be lost forever. Others share stories about finding rare music online and how it helps them remember good times.

Some worry about copyright and if sharing these tapes is legal. A few say that artists should be asked before their music is posted. But others point out that many bands, especially in the punk scene, liked people recording shows.

There is talk about the hard work of digitizing old tapes and how it takes time and special equipment. Some users explain how they have tried to save old tapes themselves, and how tricky it can be to get good sound. A few people remember using the Internet Archive before and are glad to see it growing.

Some users hope more people will share their own collections of old music, movies, or software. They say the Internet Archive is a great place to keep things safe for the future. Others just say they are happy to listen to old concerts online and discover bands they never heard before.

## The Orange Pi 6 Plus

- 原文链接: [The Orange Pi 6 Plus](https://taoofmac.com/space/reviews/2026/04/11/1900)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47732535)

The article talks about the Orange Pi 6 Plus, a new single-board computer (SBC) with strong specs. The writer tested it for a few months, trying to see if it could be a good machine for AI, server tasks, or homelab use.

The Orange Pi 6 Plus uses a CIX P1 chip with 12 ARM CPU cores, a Mali GPU, a special NPU for AI, 16GB of RAM, and two fast network ports. The hardware looks very good, but using it was not simple. The writer did not use the vendor’s ready-made system image; instead, he built his own Debian Linux image. This took time, as there were problems with the boot process, with resizing the system partition, and with getting the GPU and NPU (the AI chip) to work with Linux.

After fixing these issues, the writer moved the system from an SD card to a fast NVMe drive. He had to install special drivers and tools to get the GPU and NPU working. The GPU worked well after some work, but the NPU software was messy and hard to install. For AI tasks, the board could run small models and simple tasks locally, but not very big models. The writer tested different AI runtimes and found that using Vulkan on the GPU for “llama.cpp” was the fastest and most stable for his needs.

He also found the board’s fan was loud, with only three settings (off, normal, loud). The board stayed cool but used more power than other SBCs, even when idle—about 15-16W, which is high for this type of device. Performance is good: the fast CPU cluster is quick, memory bandwidth is solid, and the board handled real developer work well, like porting an emulator and running AI tests.

In the end, the Orange Pi 6 Plus is not perfect. The hardware is strong, but the software side still needs work. It is best for developers who like to tinker, need fast network ports, or want to run local AI or edge tasks—not for beginners or as a desktop. The writer thinks it is better than similar Rockchip boards, but not as power-efficient as a Raspberry Pi.

In the comments, some users are excited by the specs and price, calling it a possible “Raspberry Pi killer.” Others warn that software on these boards is often a problem, saying driver support and updates are not as good as on Raspberry Pi. People like the dual 5GbE ports for networking projects but worry about power use and fan noise in home labs. A few users share tips for building custom images or using other operating systems, while others say they will wait for better software support. Some mention the NPU is only useful for vision tasks, not for big language models. Overall, the community is split: some are eager to try the board, while others are cautious and want to see if software support improves.

## 5NF and Database Design

- 原文链接: [5NF and Database Design](https://kb.databasedesignbook.com/posts/5nf/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47767676)

This article talks about the fifth normal form (5NF) in database design. The writer says that 5NF is often taught in a confusing way, but it does not have to be.

The main points are about how to design database tables so there is no extra or missing data. The article shows that many examples used to explain 5NF, like the one on Wikipedia, are not clear or realistic. The Wikipedia example uses salesmen, brands, and product types, but the rules do not make sense for normal business situations. The author suggests starting with real business needs and building a logical model first. 

Two main patterns are used to design these tables: the AB-BC-AC triangle (like the “ice cream” example) and the ABC+D star pattern (like the “musicians” example). In the triangle pattern, there are three groups (for example, friends, ice cream brands, and flavors). Each friend likes some brands and some flavors, and brands make some flavors. The database has three link tables to connect these. In the star pattern, like for concerts, musicians, and instruments, there is a new table called “performance” that links the three together. You can use a composite key (using all three columns) or a synthetic key (an extra ID column) for uniqueness.

The article says you do not need to split tables or think too much about 5NF if you start from business requirements and use a clear design method. Both patterns (triangle and star) can be used together, depending on what the business needs. The author also explains that sometimes you need extra tables for special cases, like when a friend likes only one flavor from one brand.

The article gives links to more reading about 5NF and normal forms, and says that higher normal forms are often made to look more mysterious than they really are. The key idea is to focus on what the business needs, not just on following textbook examples.

In the comments, many users agree that the Wikipedia example is confusing and does not help real database design. Some people say that in practice, most databases do not need to worry about 5NF, because most problems are solved by 3NF or 4NF. A few users share their own methods, saying that starting from business requirements is always best. Some mention that higher normal forms are not needed unless the data is very complex, like in big companies or science data. 

Others point out that teaching normal forms by just splitting tables makes it harder for beginners to understand why you need them. One commenter says that examples about music and ice cream are easier to follow than the salesman example. Another user reminds that sometimes, breaking normal forms on purpose is useful for performance reasons, but you need to know the risks.

Finally, a few users discuss primary keys, with some preferring composite keys and others using synthetic IDs for simplicity. There is also a short debate about how much normalization is really needed, and if denormalization is sometimes better for speed. Overall, most people agree that clear, real examples and focusing on business needs is the best way to design databases and teach normal forms.

## Let's talk space toilets

- 原文链接: [Let's talk space toilets](https://mceglowski.substack.com/p/lets-talk-space-toilets)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47758870)

This article looks at the history and challenges of toilets in space. It explains how astronauts have tried to avoid using toilets on early missions, and how difficult it is to manage waste without gravity.

At first, astronauts ate food to make bathroom visits less frequent. Early toilets were very basic—just bags and tubes—and using them could take a long time and get messy. On Apollo missions, the process was slow and smelly, and astronauts had to mix powder into bags to control germs. Gravity on Earth helps keep waste away and stops smells, but in space, special systems are needed. Astronauts use air suction to pull waste away, which means toilet seats are small, and toilets are loud. Urine is stored in tanks, and feces are kept in bags that get compressed and stored in sealed cylinders, but odors are still a problem.

There have been many funny and gross stories, like frozen urine sticking to the outside of the Space Shuttle or toilets spraying dried fecal dust in the cabin. Space toilets have slowly improved, but even today, astronauts still use bags for solid waste. The International Space Station recycles most water from urine, but fecal collection is almost the same as it was in the 1970s.

Looking ahead to Mars missions, toilets need to work for much longer and stay clean while not in use, which is a big challenge. Mars has a little gravity, which makes toilet design tricky. Waste must be stored safely for years, and NASA is testing ways to roast it, turning it into odorless blocks that could even help shield astronauts from radiation.

In the Hacker News comments, some readers laugh about how space toilets are still a problem after all these years. Others suggest that the difficulties show how hard real engineering is, even for simple things. A few share more stories from NASA reports or suggest new ways to handle waste, like burning it or using it for fuel. Some people are surprised by how much effort goes into toilet design compared to other space technology. There are jokes about how this topic is always funny, but also serious talk about the health risks of bad waste management. A few say that these toilet problems are a good reminder that space travel is not glamorous. Others wish more people knew how much work goes into making space life possible—even for things as basic as using the bathroom.

## Show HN: Plain – The full-stack Python framework designed for humans and agents

- 原文链接: [Show HN: Plain – The full-stack Python framework designed for humans and agents](https://github.com/dropseed/plain)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47768750)

Plain is a new full-stack Python web framework, based on Django, made for both people and AI agents. It aims to be simple, clear, and easy to use, with strong typing, good docs, and tools for both humans and code-writing bots.

The framework uses Python 3.13+, PostgreSQL as the database, Jinja2 for templates, htmx and Tailwind CSS for the frontend, and lots of modern Python and JavaScript tools. The code examples show how to define models, views, and URLs in a way that looks familiar to Django users but is more strict, typed, and explicit. Everything is organized so that both people and AI tools can understand and work with the code easily.

Plain comes with more than 30 built-in packages for things like authentication, jobs, email, cache, admin UI, feature flags, analytics, and more. It has a command-line tool (“plain”) that helps you start servers, run tests, check code, format code, and see docs. There are special features for AI agents: project “rules” to avoid common mistakes, skills you trigger with slash commands (like upgrading packages or reporting bugs), and docs you can search from the terminal.

The project is a fork of Django, but it is changing fast to fit today’s needs, especially for teams that use AI code assistants. It’s open source and BSD-licensed.

In the Hacker News comments, some people like that Plain brings fresh ideas to Django and is built for AI agents. They find the strict typing and rules a good match for automated tools. Others are unsure why we need another framework since Django is already powerful and mature. Some users are worried about it dropping support for MySQL and SQLite, making it Postgres-only, which could be a problem for some projects. A few people like the idea of having one tool with so many built-in packages, while others wonder if it might become too complex or “opinionated.” There’s also interest in how well it works with existing Django code, and if it’s easy to migrate. Some ask how stable it is and if it will be maintained long-term. Lastly, several developers are curious to try Plain out, especially for new apps where agent support and clear code matter most.
