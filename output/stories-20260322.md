## The future of version control

- 原文链接: [The future of version control](https://bramcohen.com/p/manyana)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47478401)

This article talks about a new project called Manyana, which tries to make version control better by using CRDTs (Conflict-Free Replicated Data Types). The author explains that CRDTs are good because they always merge changes without errors, but they have not been used in version control much because of user experience problems.

With Manyana, when two people change the same part of a file, the system shows exactly what each person did, not just a confusing mix of changes. For example, if one person deletes a function and another edits it, Manyana tells you who did what, instead of showing you two big blocks of code you have to compare yourself. This makes it easier to understand conflicts.

CRDTs make sure that merges never fail, and the final result is always the same, no matter the order of merges. This helps when many people work on the same file at once. When two people add code in the same place, the system picks an order and keeps it forever, so there is less confusion. Conflicts do not stop the merge; they are just shown for review, and the system always gives you a result.

Another idea is that the whole history is saved in the file structure itself, so you do not need to look for a common ancestor when merging. Rebasing can keep all history, not just rewrite it, which is better for tracking changes.

Manyana is only a demo for now. It is written in Python, works on single files, and does not yet include some features like cherry-picking or local undo. The code is public and open for anyone to use or build on.

In the comments on Hacker News, some people are excited about CRDTs in version control. They say this could fix old problems with Git, like hard-to-understand conflicts and confusing merges. Others point out that CRDTs have been tried before but did not catch on, maybe because CRDTs are hard to understand, or because developers are used to how Git works.

A few users worry about performance with big projects or binary files, while others argue that the better conflict view is worth it. Some like that history is kept in the file and not just a list of commits. There are questions about how this system would scale for big teams and complex projects.

One person says the demo is simple but proves the main idea works. Another suggests that better user tools are needed before people will switch from Git. Some users hope Manyana will push existing version control tools to improve. Others are unsure if companies will want to change their process for a new system.

Overall, people think it is a cool experiment, with some excited for the future and others cautious about replacing old tools. Many agree that version control needs better ways to show and solve conflicts, and Manyana is a step in that direction.

## The gold standard of optimization: A look under the hood of RollerCoaster Tycoon

- 原文链接: [The gold standard of optimization: A look under the hood of RollerCoaster Tycoon](https://larstofus.com/2026/03/22/the-gold-standard-of-optimization-a-look-under-the-hood-of-rollercoaster-tycoon/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47480886)

This article explains how RollerCoaster Tycoon became famous for being a very fast and smooth game, even on old computers from 1999. The main reason is that Chris Sawyer, the creator, wrote almost all of the game in Assembly, which is a low-level language that gives the programmer a lot of control over how code runs.

The game’s code was carefully optimized in every way. Different types of money values in the game use different data sizes to save memory and speed up calculations. For example, the whole park’s value uses four bytes, but a small value like the price of a shop item uses only one byte. The code also uses bit shifting instead of normal multiplication and division, which is much faster for computers. For example, multiplying a number by 4 is done by shifting its bits to the left, which is quicker than using a multiply operation.

Chris Sawyer also designed the game itself to match what computers could do quickly. One big example is how guests move around the park. Instead of each guest thinking about where they want to go and finding the best path (which is slow), guests just walk randomly and only use pathfinding when they really need it—like when they want to leave the park. To keep the game running smoothly, the pathfinding system has limits, so if a guest can’t find a way out quickly, they just give up and complain, which players can see in the game as “can’t find exit” messages.

Another smart trick is how the game handles crowds. Guests don’t really “see” each other and can all stand on the same path tile, so there’s no complicated collision system. But guests still get unhappy if there are too many people nearby, which makes the player want to build bigger paths—so the gameplay still works. All these tricks mean the game can handle thousands of moving guests without slowing down.

Interestingly, many of these optimizations were possible because Chris Sawyer was both the main programmer and the designer. He could change the game’s design to help the code run faster, something that is hard today because teams are bigger and roles are separate.

In the comments, many people praise Chris Sawyer’s skill and say this level of optimization is rare today. Some share stories about how well RollerCoaster Tycoon ran on their old computers, while newer games often struggle. A few people point out that writing in Assembly is very hard and time-consuming, and most modern games use high-level languages because computers are much faster now. Others discuss how modern compilers have improved and can do many optimizations automatically, but there’s still value in designing games with performance in mind from the start. Some mention OpenRCT2, a fan-made project that re-creates the game and gives more insight into how the original worked. A few commenters wonder if this kind of deep optimization is even needed today, but most agree that it’s inspiring to see what’s possible when someone really understands both code and game design.

This article is about someone at Microsoft who started and ran a systems reading group over five years, beginning with database topics and expanding to wider system subjects. The author explains how the group began soon after they joined Microsoft, wanting to share and learn more about databases and storage systems with curious coworkers. 

The group started with simple meetings, where everyone read a paper and then talked about it together. They covered important topics like B-trees, LSM-trees, and reviewed classic research papers about storage and databases. Over time, members suggested new papers, sometimes from other fields, and shared interesting blog posts or talks in a chat channel. This informal sharing was as useful as the main meetings.

The group didn't stay focused only on databases. As discussions grew, they explored related areas like memory, networking, and consensus algorithms. In later years, they changed from reading random papers to following a series on one big topic, like reading sections from famous books or focusing on datacenter foundations. This helped everyone build deeper understanding and have better discussions.

The author shares advice about running such a group: start small, stick to a regular meeting schedule, and let the topics grow naturally based on what people want to learn. Having a co-organizer helps keep things going when one person gets busy. It’s important to make meetings friendly, even for people who didn’t finish the reading, by giving a short summary at the start. The author says you don’t need to be an expert—just be willing to learn together.

The main benefits were learning new things and meeting other curious people across the company. These connections sometimes helped with real work and made the author feel proud to be in a company full of interested engineers. The author encourages others to start a reading group without worrying too much about getting it perfect from the start.

Some commenters discussed how hard it can be to keep the group going if the organizer is busy, so they liked the idea of having a co-organizer. Others said that the best groups are open to new topics and let members suggest what to read next, rather than sticking to a strict plan. A few people shared tips about using online tools to organize meetings and share links, saying this helps more people join. Some wished their company had something similar and felt inspired to start their own group. Others pointed out that even if only a few people join, the connections and new ideas make the effort worthwhile. A few mentioned that joining a reading group helped them with job interviews by making them more confident about topics outside their daily work. Overall, readers thought the article gave good, practical advice for anyone who wants to start a learning group at work.

- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47476821)

The main idea is to bring together resources like Wikipedia, medical guides, and Khan Academy, and make them work without an internet connection. You can also run AI models locally, so you can chat with an assistant, write, or code offline. NOMAD uses strong hardware, so you get fast AI and big collections of information. Unlike other offline products that are expensive or locked to certain hardware like Raspberry Pi, NOMAD gives you full control and works on more powerful computers. You can choose what content to download, upgrade your setup, and make your own offline server. The project is funded by the community and has no paywalls, so anyone can use it.

## Flash-MoE: Running a 397B Parameter Model on a Laptop

- 原文链接: [Flash-MoE: Running a 397B Parameter Model on a Laptop](https://github.com/danveloper/flash-moe)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47476422)

This project is about running a huge AI model (397 billion parameters) on a normal MacBook Pro laptop using a tool called Flash-MoE. The main idea is to make very large language models work fast and well on small machines, without needing a big server.

The model used is Qwen3.5-397B-A17B, which is very large and would normally need a lot of memory and a strong computer. Flash-MoE makes it possible by streaming the model’s weights (data) from the SSD only when needed, instead of loading everything into RAM. It uses C and Apple’s Metal (a GPU system) for speed, not Python or big frameworks. The software can run at over 4 tokens per second, which is fast enough for real use, including tool calling and chat.

To do this, only a small part of the model (the “experts” needed for the current task) is loaded at a time. The OS handles caching, so there is no need for a custom memory cache. The project uses special GPU code (Metal shaders) for the math, like matrix multiplies and activations, and clever tricks like reordering math to use fused multiply-add instructions for more speed. The pipeline is designed to use the SSD and GPU efficiently, avoiding slowdowns when both want memory at the same time. Many experiments were tried: some ideas made things faster (like trusting the OS page cache and using BLAS for some math); others made things slower and were dropped (like compression, custom caching, and speculative routing).

The main hardware tested is a MacBook Pro with an Apple M3 Max chip and 48GB memory. Only about 6GB of RAM is used for code and data, and the rest is left for the OS to manage SSD caching. The whole model is 209GB on disk at 4-bit (compressed) weight size. The software is open source, written mostly in Objective-C and C, with GPU code in Metal.

In the comments, many people are excited to see such a large model running on a laptop, calling it impressive and a big step for local AI use. Some ask about running it on other systems besides Mac, but the current version is tied to Apple hardware and Metal. Others point out that this shows how much hardware and software have improved: running something this large on a laptop was unthinkable a few years ago. There are questions about real-world use, like how much slower it gets on less powerful laptops, or if this can work with other models. Some users discuss the “trust the OS” idea, saying it is smart to let the operating system handle caching instead of writing extra code. A few people worry about SSD wear, since it streams lots of data, but others reply that modern SSDs are very durable. Some wonder about battery life, since the GPU works hard, while others are just happy to see open source code written in plain C and Metal, not just Python. Overall, people see this as a clever project that pushes the limits of what laptops can do with big AI models.

## LLMs Predict My Coffee

- 原文链接: [LLMs Predict My Coffee](https://dynomight.net/coffee/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47425874)

This article is about testing if large language models (LLMs) can predict how fast coffee cools in a mug, using real math and a home experiment. The author asks several LLMs for an equation to estimate coffee temperature over time, after pouring boiling water into a mug at room temperature.

The problem is complex because many things affect how coffee cools: heat moves between the water, cup, air, and table; water evaporates; the mug material matters; and more. The author notes that it’s hard for even humans to guess what matters most or make a perfect prediction. None of the LLMs can know every detail, so they all have to make assumptions.

The LLMs gave equations using exponential decay, showing temperature dropping quickly at first, then more slowly. Each model had slightly different numbers, and some used two rates (a “fast” and “slow” process), while others used just one. The author then did a real experiment, carefully measuring the coffee’s temperature every few seconds after pouring.

Comparing the LLM predictions and the real data, the author found that the models were all “OK” but not perfect. The best model (Claude 4.6 Opus) was close, but still not fully correct. The real coffee cooled even faster at the start and slower at the end than the models guessed. The author jokes that their own intuition was worse than the LLMs, so maybe the models are still useful.

In the end, the LLMs all used the same kind of math (exponential decay), but with different details. Some other LLMs failed to give any answer at all, but still charged money. The article suggests that while LLMs can do a decent job with physics problems, real experiments are still needed to check their answers.

In the comments, some people said it’s impressive that LLMs can do this much, given the lack of details. Others pointed out that these problems are hard even for scientists, and the models’ use of exponential decay is the typical approach. A few readers thought the LLMs did better than many humans would, and liked that the author actually tested their predictions in a real experiment.

Some commenters wondered if giving the models more details (like the mug’s material or the air’s humidity) would help, or if LLMs could ever really replace experiments in science. Others said the cost of using LLMs for this kind of problem is still high, especially when some models charge money but don’t answer. A few people joked about the “middle-school science project” tone, saying it’s a good reminder that real-world physics is messy.

One reader mentioned that LLMs’ answers are limited by their training data and might not improve unless they “see” more real experiments. Another person suggested that combining LLMs with traditional physics simulation tools could give better results. Overall, the comments show people are curious, impressed, and a bit skeptical about using AI to model simple but real-world science problems.

## MAUI Is Coming to Linux

- 原文链接: [MAUI Is Coming to Linux](https://avaloniaui.net/blog/maui-avalonia-preview-1)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47478687)

MAUI, a framework for building cross-platform apps with .NET, can now run on Linux thanks to a new Avalonia backend. The Avalonia team released a preview that lets developers use Avalonia to bring .NET MAUI apps not only to Linux, but also to WebAssembly (for web apps).

The article explains how to get started with four easy steps: create a MAUI app, add the Avalonia.Controls.Maui.Desktop NuGet package, add the net11.0 target, and update your MauiBuilder. This setup means you don’t need a lot of custom code to run MAUI apps on Linux. Avalonia handles the platform-specific work, and you can even change or extend the setup if you want more control.

The team improved Avalonia itself as part of this project, adding new navigation APIs and controls in Avalonia 12. These upgrades help close the gap between MAUI’s and Avalonia’s features, so you don’t need to write extra code for each platform. Because Avalonia draws its own UI, your app will look the same everywhere.

To test everything, the team ported several apps, like MauiPlanets, a 2048 game, and larger sample apps such as AlohaAI and MyConference. They made small fixes for themes and other features, but most code worked as expected. They also ported other sample apps like WeatherTwentyOne, using Avalonia’s new open-source WebView for web content.

If your app uses MAUI’s GraphicsView or SkiaSharp, it should work with Avalonia MAUI without changes. The team even wrapped SkiaSharp.Views.Maui so libraries depending on it would work for Linux and the web too. They’re working on more features, like support for MAUI Essentials (for device APIs) and better WinUI interoperability. The goal is to give developers choices: you can use native controls that match each platform, or Avalonia’s drawn controls for a consistent look everywhere.

In the Hacker News comments, many users are excited to see MAUI running on Linux, as this was often missing from cross-platform frameworks. Some think Avalonia is now the top option for building desktop apps with .NET, since it brings real cross-platform support. Others still prefer native toolkits for the best user experience, but agree it’s good to have choices.

A few developers shared worries about performance and memory usage for drawn UIs, especially compared to truly native apps. Some mention that Avalonia’s look is very consistent, but might not always fit in with each operating system’s theme. Others point out that for business apps, having the same UI across platforms is often more important than looking native.

There are also questions about how well more complex apps will work, like those with deep OS integration or special hardware needs. Some users are happy about the open-source direction and the easy migration path from MAUI to Linux. Others say that while it’s promising, they hope the project keeps improving documentation and support.

A few comments bring up the long history of cross-platform .NET UI attempts, and hope this will be the version that finally “sticks.” Some are excited about the WebAssembly support, since it opens new doors for using .NET apps in browsers. Overall, the mood is positive, with most people agreeing that Avalonia MAUI is a big step forward for cross-platform .NET development.

## Teaching Claude to QA a mobile app

- 原文链接: [Teaching Claude to QA a mobile app](https://christophermeiklejohn.com/ai/zabriskie/development/android/ios/2026/03/22/teaching-claude-to-qa-a-mobile-app.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47480900)

This article is about using Claude, an AI tool, to help test (QA) a mobile app called Zabriskie, which one developer built alone. The main problem was testing the same app across web, Android, and iOS, without writing different code for each platform or doing all the manual work alone.

The developer used Capacitor, a tool that wraps a web app into a mobile app, so the same code runs everywhere. This makes building and updating the app fast, but it also makes testing hard. Usual testing tools for web (like Playwright) don’t work, and native mobile testing tools can’t see inside the web part of the app. So, the developer taught Claude to do mobile QA by driving the apps, taking screenshots, checking for problems, and creating bug reports.

On Android, this was easy. The developer could connect to the Android WebView using Chrome DevTools Protocol (CDP), which is the same system tools like Puppeteer use. With CDP, Claude could log in, move around the app, take screenshots, and spot issues very quickly. The whole Android setup took only 90 minutes.

On iOS, the process was much harder and slower (over 6 hours). The iOS Simulator blocks many ways of controlling the app. Typing special characters (like “@”) for emails didn't work well. Native iOS dialogs (like notification permissions) were hard to dismiss automatically. Tapping on buttons by guessing their screen coordinates often failed because of scaling and layout differences. The developer had to write backend changes, dig into system databases, and use special tools to finally get reliable, automated navigation and testing working.

The article also tells a story about a mistake with git, where Claude made changes in the wrong place, mixing unrelated code and breaking tests. This caused extra work fixing and cleaning things up.

In the end, both Android and iOS can now be tested automatically each day, with screenshots and bug reports. The key lessons are: use system protocols (like CDP) when you can, measure and check before acting, keep code changes isolated, and always run tests before pushing code.

Commenters on Hacker News had different reactions. Some praised the creative solutions, especially for iOS, saying the debugging stories were both funny and painful. Others pointed out that Apple’s closed approach makes automation and testing much harder than on Android, and wished Apple would offer better tools like CDP. Some readers liked the lesson about not mixing code changes and the reminder to always run tests before pushing. A few suggested that using cross-platform tools always brings pain, but agreed it is often the only way for solo developers. There were questions about whether these methods would work on real devices, not just emulators or simulators. Others wondered if Apple would ever make testing easier for developers or if third-party tools might fill the gap. Some people said this article showed why indie developers need to be creative and flexible, especially when tools don’t do what they want. Overall, readers agreed that mobile testing is hard, and this story showed both the challenges and the smart workarounds possible with enough patience.
