## CasNum

- 原文链接: [CasNum](https://github.com/0x0mer/CasNum)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47291292)

CasNum is a Python library that lets you do math using only compass and straightedge, like ancient Greek mathematicians. The project started as a fun way to build numbers and arithmetic from just drawing lines and circles, and then grew into something bigger.

The core idea is that all numbers are points on a plane, and all math is done by geometric constructions. For example, adding two numbers means finding the midpoint and doubling it. Multiplication uses triangles, and even logical operations like AND or OR are built using geometric tricks. The library is very slow, but it does work—you can even plug it into a Game Boy emulator so that every calculation in the game happens with compass and straightedge math. There’s an RSA example too, and you can see the geometric steps in a viewer.

The code is optimized in smart ways—like making multiplication by 2 faster, or improving modulo calculation. Still, running real programs is so slow that booting Pokémon Red takes about 15 minutes. After that, the emulator might reach 1 frame per second if you’re lucky. The author jokes that this is how math "should" feel, and that modern programmers are spoiled by simple “a + b” code. The project is open source and uses the MIT license, with some parts from other projects like PyBoy and a Game Boy ROM for 2048.

In the Hacker News comments, many people enjoy the playful and absurd nature of the project. Some say it’s a great example of building something just because it’s fun, not for speed or usefulness. A few users are amazed by the idea of running a Game Boy this way, calling it “gloriously pointless” or “an art project.” Others point out that the code is a good lesson in how math can be built up from simple rules. Some ask how practical it is, and wonder about performance or memory use; others answer that it’s not meant to be practical at all. One comment jokes that “now we just need a CPU built of actual rulers and compasses.” A few users also talk about the history of compass and straightedge problems, like squaring the circle, and enjoy that you can almost play Pokémon with ancient math tools. Overall, most comments show respect for the creativity and humor behind CasNum, and some even want to try it just for fun.

## A decade of Docker containers

- 原文链接: [A decade of Docker containers](https://cacm.acm.org/research/a-decade-of-docker-containers/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47289311)

This article looks back at 10 years of Docker and explains how it changed the way we build and run software. Docker made it easy for developers to package apps with everything they need, so they can run anywhere, not just on the developer’s own machine.

At first, running software on different computers was a big pain—people had to install many tools and fix conflicting files. Docker fixed this by using “containers” that hold the app, its code, and all its files together. By using something called a Dockerfile, developers can write simple steps to build and share these containers. Docker uses special features in the Linux system called “namespaces” to keep each app separate, but not as heavy as running a whole virtual machine. Instead, containers share the same Linux but have their own view of files and networks, so apps do not mess with each other.

The article shows how Docker’s main commands—build, push, and run—have stayed simple. Images (the files for containers) are shared on Docker Hub, and many people use and reuse them. Docker uses a layered way to store these images, which makes them easy to share and update. Running a container is fast and does not use lots of memory. Later, Docker made it possible to run Linux containers on Mac and Windows by putting a small Linux inside the desktop app, so developers could work the same way on any computer. Networking and file sharing between the container and the host computer were tricky problems, but Docker solved them with clever tricks, like using old tools (SLIRP) and new ones (vpnkit, virtio-fs).

As more computers started using different CPUs (like ARM, not just Intel), Docker added support for building and running containers for many CPU types. For security, Docker added ways to keep secrets safe inside containers, even using special hardware. For AI, Docker had to handle GPUs, which are much harder to share than CPUs, but they keep working on this. The article says Docker is now everywhere: in the cloud, on desktops, and even in space. Its goal is to stay simple and useful, letting developers focus on building and shipping software.

In the Hacker News comments, some people share how Docker made their lives much easier—they no longer have to say “it works on my machine” when something fails elsewhere. Others point out that Docker is not perfect: sometimes containers are too big, or security is not as strong as they want. A few users remember when setting up a VM for every app was slow and painful, and they praise Docker for fixing that. Some say tools like Podman or Kubernetes have taken things further, but Docker made the first big step.

There are also comments about how Docker’s use of Linux namespaces is both smart and complicated, and some wish Windows had something similar. A few people talk about problems with Docker on Mac, like slow file sharing, but they see the improvements over time. Some discuss how Docker helps new developers get started quickly, while others worry about too much “magic” hiding how things work. Security is a common concern, with some saying that containers are not as safe as real VMs for running untrusted code.

Other users like that Docker made it easy to try out new software without breaking their main system. Some old-school sysadmins share their love-hate relationship with Docker, saying it has made things easier but also more complex in some ways. A few worry about “Docker fatigue”—too many tools and updates to keep up with. Still, most agree that Docker changed software development for the better, making sharing and running apps much simpler than before.

## Dumping Lego NXT firmware off of an existing brick

- 原文链接: [Dumping Lego NXT firmware off of an existing brick](https://arcanenibble.github.io/dumping-lego-nxt-firmware-off-of-an-existing-brick.html)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47271988)

This article explains how someone backed up the old firmware from a Lego Mindstorms NXT brick from 2006. The author wanted to save this original firmware because it was hard to find online, and also wanted to help the Pybricks project.

First, the author checked if anyone else had saved this firmware but could not find a copy. They tried to use the official updater tool, but it erased part of the firmware instead of saving it. Next, they thought about using the JTAG hardware interface, but this would mean opening the NXT and soldering, which is hard and risky. The author wanted a software-only way.

They looked into writing a normal NXT program to dump the firmware, but normal programs run inside a limited virtual machine. This VM does not let you read or write memory outside a small area, so this idea would not work.

Then, the author studied the NXT firmware’s source code and communication protocol. They found something called IO-Maps, which let you read and write data between the VM and the rest of the system. While looking at the VM’s IO-Map, they discovered a function pointer that could be changed from the computer over USB. If you change this pointer, you can control what code runs for certain commands.

The author used Python and PyUSB to send commands to the NXT and read memory. By filling a memory area with “nop” instructions and some custom code, then changing the function pointer to point there, they got the NXT to run their own code. This gave them full control, so now they could read any part of the device’s memory, including the firmware.

The author wrote a small assembly program to read memory and send it back over USB. With this, they made a script to dump the whole 256KB of internal flash to a file. They plan to clean this file before sharing, because it also contains user data.

In the comments, some people were impressed by the clever use of the function pointer and IO-Map. Others talked about how old hardware like this often had simple protections, making these kinds of exploits possible. A few remembered hacking similar toys or devices and felt nostalgic. Some worried about the risk of releasing exploits that could let people write malware for toys. Others thought this work was important for preserving history, since old firmware is often lost. A few suggested alternative ways to dump firmware, like using JTAG, but agreed that software-only methods are easier for most people. Some pointed out that this kind of research helps make future devices more secure. Others asked technical questions about the assembly code or how USB communication worked. There was also talk about open hardware and how community projects like Pybricks benefit from this kind of hacking. Finally, some readers thanked the author for the detailed write-up and hoped more old firmware could be saved in the future.

## FLASH radiotherapy's bold approach to cancer treatment

- 原文链接: [FLASH radiotherapy's bold approach to cancer treatment](https://spectrum.ieee.org/flash-radiotherapy)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=47288533)

This article talks about FLASH radiotherapy, a new way to treat cancer by giving one very strong, fast burst of radiation instead of many small doses over weeks. The method is being developed by the French company Theryq and CERN, using advanced particle accelerators.

Standard radiation therapy can hurt healthy tissue around a tumor because it needs many sessions at lower doses. FLASH therapy uses a very high dose in less than a tenth of a second, which studies show can kill tumors but cause much less harm to healthy tissue. The idea came from an accident in a lab in the 1990s, where scientists noticed that mouse lungs did not scar after a quick, powerful blast of radiation. Later, tests on different animals and some humans showed the same effect: tumors die, healthy parts stay mostly safe.

Nobody fully understands why FLASH works this way, but one theory is that healthy and cancer cells react differently to the fast, high-energy treatment. Scientists are still researching this.

To make FLASH therapy possible for deep tumors, researchers are building new, more powerful and precise accelerators. These machines, first made for physics, can now send strong electron beams deep into the body in a split second. The machines are getting smaller and better, but it’s still a challenge to make them cheap and easy to use in hospitals.

Theryq has already built machines for surface tumors and is now working on a bigger system, FLASHDEEP, that can reach tumors up to 20 centimeters inside the body. They also use special CT scans to make sure the radiation hits the tumor exactly.

Testing is done first on fish and mice at special labs, since not all research centers allow animal tests. The experiments use very precise equipment to control the timing and power of the radiation. Measuring these strong, quick doses is hard, so new sensors are being developed.

FLASH therapy could make cancer treatment faster, cheaper, and safer. In poor countries, it could help more people get care, since one session may be enough instead of many trips to the hospital. Even in rich countries, fewer sessions mean less cost and fewer side effects.

In the Hacker News comments, some people are very excited and think FLASH could change cancer treatment a lot. They like the idea of fewer hospital visits and less pain for patients. Others are more careful, saying that results in animals don’t always match what happens in humans. Some worry about the cost and size of the new machines, asking if hospitals can really use them soon.

A few commenters discuss the science and are surprised that such a fast, strong dose can be safe. Some think the real reason for the FLASH effect is not clear yet and want more studies. There are also questions about how easy it will be to train doctors and build enough machines for everyone.

Some people point out that if the machines become smaller and cheaper, even small clinics might offer FLASH therapy. Others hope this will help close the gap in cancer care between rich and poor countries. There are also comments about possible new uses for the technology, like learning more about how cancer and healthy cells are different.

Overall, the comment section shows a mix of excitement, hope, and cautious waiting for more proof before FLASH is used everywhere.

Ki Editor is a new code editor that works directly with the structure of your code, not just the text. It lets you interact with the code’s Abstract Syntax Tree (AST), which means you can edit code in a smarter way.

The main idea is that, instead of only moving your cursor by letters or lines, you can select and change parts of the code based on their meaning, like whole functions, variables, or conditions. You do not have to fight with the keyboard or mouse to select code—just pick the piece you want by its role in the code. The editor supports multiple cursors, so you can work on several code parts at once. This helps a lot with big changes, like renaming a variable everywhere or refactoring code in many places. Ki Editor also changes how you move around code. There are different modes for moving by word, line, or by syntax node, so you can work faster and more consistently. The goal is to make editing code less about fighting with text and more about shaping your ideas.

They say working with the AST is the future, and it could solve problems that happen with simple text-based editors, like hard-to-find bugs from missed replacements. The goal is to attach to this running program, change the value of a variable, and even replace one of its functions with a new one—all while the program is running. To do this, the author explains how to set up the build system (using CMake), set the right permissions (entitlements), and sign the code so macOS allows these changes. The test program writes its own process ID and memory addresses to a file, making it easier to find the right spot to poke in memory.

The injection program reads this file, attaches to the running process using Mach APIs, and suspends the process to safely read or write its memory. The author shows how to read the value of the variable, change it, and resume the program so it keeps running with the new value. For replacing a function, the injection program copies the machine code of a new function into the target process, marks that memory as executable, and then writes a “trampoline”—a short jump instruction—at the start of the old function so it jumps to the new code. The article covers both Intel and Apple Silicon (ARM) computers, and explains how to handle problems like memory permissions and function size.
