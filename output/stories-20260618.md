This article is about a new website called “In the Weights,” made to check if language models know about certain people or things. The site lets you see how much different AI models “recognize” you by connecting to many models at once and studying their answers.

The website was created because more people are using large language models (LLMs) instead of searching the web. The team wondered what information about people is inside these AI models, stored in their “weights.” You can use the site to ask if models know about you or something else. The system sends your question to many models—big ones and small ones—at the same time. It collects the responses, groups similar answers together, and then shows you how strongly each model recognizes your input. The website is meant to help people understand what data AI models keep inside, even after the web pages might be gone. It can help you see if your name or a topic you care about is inside these models. This tool can be useful for privacy, for curiosity, or for checking how well-known something is in the AI world.

In the main comment, the creator explains their goal: to let people find out what traces are “in the weights” of LLMs as the internet changes. They built the tool quickly with a design partner and are open to questions. Some users find the idea interesting and want to know more about how the clustering works. Some think it is a fun way to learn about AI, while others wonder if the tool could be used to find bias or mistakes in the models. ## I found 10k GitHub repositories distributing Trojan malware

- 原文链接: [I found 10k GitHub repositories distributing Trojan malware](https://orchidfiles.com/github-repositories-distributing-malware/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48583928)

A developer found 10,000 GitHub repositories that were spreading Trojan malware by copying real projects and adding links to infected files. These fake repos looked almost the same as the originals, even listing the real contributors, but always updated their README files to add a link to a zip archive. The zip file had several files inside, including an .exe file with malware, but if you only scanned the archive’s link on VirusTotal, it showed clean; only uploading the file itself would reveal the Trojan. The author wrote a script to find these repos based on their update patterns—like frequent README changes and copying commits from other projects—and found that after adjusting the filters, thousands existed. GitHub did not respond quickly to reports, and only after the author published the findings did GitHub start deleting the bad repos. The article raises questions about GitHub’s security, the real scale of this problem, why these attackers target new repos, and how the malware works.

Many commenters agree this is a big problem. Some think these attacks are aimed not at humans, but at automated bots or “agents” that auto-install dependencies, which could start new infection chains. Others share stories of accidentally downloading malware from GitHub, even after checking the code, and warn that password managers and TOTP codes can be at risk if your device gets infected. There is debate about the best way to protect credentials—some use hardware keys like Yubikey, but others say it’s hard to manage backups and switching devices. Several people think GitHub could do more, like scanning uploads for malware or using simple rules to catch suspicious repos, but others argue the scale is hard to manage.

Some users believe open source is not always safer, since most people do not review code or verify binaries, and attackers take advantage of the trust people put in popular-looking repos. A few discuss how easy it is for phishing sites to appear high in search results, not just on Bing but also on Google, which leads to more people getting tricked. People also mention other similar malware campaigns, and note that even after reporting bad repos, GitHub is often slow to act unless the issue becomes public. Others point out that fake stars and new accounts can push malware repos up in trends, making them look more trustworthy. Some feel powerless to stop these scams, but agree that sharing findings with the community helps, while others call for better tools to scan and check code before downloading. Overall, the comments show worry about how easy it is to spread malware using trusted platforms, and frustration that big companies don’t do enough to protect users.

## I told them forced consent was unlawful. 5 years later it cost Elkjop €1.8M

- 原文链接: [I told them forced consent was unlawful. 5 years later it cost Elkjop €1.8M](https://www.thatprivacyguy.com/blog/elkjop-forced-consent-fine/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48589501)

The article tells the story of how Elkjop, a big electronics retailer in the Nordics, was fined €1.8 million for breaking EU privacy rules. The writer was a member of Elkjop’s customer club and got many marketing emails, but the only way to stop them was to leave the club and lose all membership benefits.

The writer explained to Elkjop that this was illegal under GDPR, which says you must be able to say no to marketing without giving up other things. Elkjop replied that getting marketing emails was a condition of being in the club. The writer then made a formal complaint and used different GDPR rights, like asking for all their data and making Elkjop stop using it. The Swedish data authority started the case but later passed it to Norway, where Elkjop’s head office is.

After five years, the Norwegian authority fined Elkjop €1.8 million because members could not freely say no to marketing. The company also used customer data for extra advertising without checking if this was okay, which broke more rules. The article says this is not just about one company—many digital businesses use forced consent, which is against the law if you can’t say no without losing access or benefits.

The writer also says it was wrong that neither data authority told them about the result. Instead, they found out from a volunteer wiki. The law says authorities must inform complainants about updates, but they did not. The writer now wants an explanation and may report the authorities for not following rules.

In the comments, one reader says it’s “extremely cool” that Elkjop was actually fined and thanks the writer for sharing, but is surprised the authorities didn’t keep them updated. Another person wonders how much money Elkjop made during those five years by using this unlawful consent. A third commenter says it’s a good outcome and wishes similar privacy rights existed in the USA but notes that it took too long for justice to happen.

Overall, commenters support the fine and the fight for privacy, but are frustrated by slow action and poor communication from authorities. Some think big companies might still profit from breaking rules, even with fines. Others wish these strong privacy laws were available in more countries.

## American Express: Cell-Based Architecture for Resilient Payment Systems

- 原文链接: [American Express: Cell-Based Architecture for Resilient Payment Systems](https://americanexpress.io/cell-based-architecture-for-resilient-payment-systems/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48547969)

This article explains how American Express builds its payment system using something called a "cell-based architecture." The goal is to keep payments fast, always online, and able to recover quickly from problems.

The main idea is to split the whole system into small, independent units called cells. Each cell can work on its own, has its own services and data, and does not need help from other cells to process payments. If one cell breaks or needs maintenance, the rest of the system keeps working fine. This stops problems from spreading everywhere.

To make this work, American Express puts all needed data into each cell ahead of time. For things that change often, they use smart routing: a central router checks which cell has the right, up-to-date data and sends the payment there. This router also controls all the traffic between cells, so cells do not talk directly to each other. This keeps everything neat and safe.

When a cell fails during a payment, the router moves the payment to a healthy cell and starts over. They do not try to continue where the failed cell left off—this keeps things simple and avoids mistakes or double payments. Each payment has a unique ID, which helps avoid duplicates if something is retried.

American Express also keeps logs, metrics, and other tracking inside each cell first, and gathers them later for the whole system. If some tracking tools break, only that cell loses visibility, not the whole system. They also make sure that logging and other non-critical services do not slow down or block payments if they become unavailable.

All of this helps American Express keep the payment system running smoothly, even if parts of it have problems. The company chose this design to make sure failures are always small and easy to control, and to keep payments fast for people everywhere.

In the comments, there is just one so far, so there is not much discussion yet. Some readers might be interested in how this compares to similar systems at other big companies. Others might wonder about the extra work and cost of running many independent cells instead of one big system. Some developers may like how clear the boundaries are, while others may worry about making sure data stays in sync or the risk of duplicated effort. But most agree that, for payments, safety and uptime are worth the extra work.

## Migrating from GNU Stow to Chezmoi

- 原文链接: [Migrating from GNU Stow to Chezmoi](https://rednafi.com/misc/chezmoi/)
- HN链接: [Hacker News讨论](https://news.ycombinator.com/item?id=48588413)

This article is about moving from using GNU Stow to manage dotfiles, to using Chezmoi instead. The author explains why Stow became hard to use across several Macs and how Chezmoi made things easier and more organized.

Stow works by creating symlinks from files in a git repo to your home directory. This is simple for one computer, but on many machines, it causes problems. Changes can get out of sync, and new machines need manual cleanup before Stow can link files. Stow also only handles files, so you need extra scripts for things like installing Homebrew packages or setting macOS settings.

Chezmoi keeps all your dotfiles in a source directory as real files, not symlinks. You add files with `chezmoi add`, and Chezmoi tracks them with special names like `dot_zshrc` for `.zshrc`. It can also handle file permissions, templates for machine-specific files, and ignore files you don’t want in your home directory. Applying changes is easy: `chezmoi apply` copies everything to the right place. Edits can be done in the source and pushed with git, then pulled and applied on other machines.

The author likes that changes now only happen when they want, not by accident. Bootstrapping a new Mac is quick with just a couple of commands. Chezmoi also runs scripts at certain times, like before or after copying files, and only when files change. This helps automate package installs and system tweaks. The setup is simple and avoids lots of custom scripts.

For day-to-day use, the author edits files in the source or imports changes from live files, then shares updates with git. On other machines, catching up is just one command. Chezmoi also helps manage AI agent “skills” so all agent tools use the same files on every computer.

In the comments, many people share their own dotfile management stories. Some say they were happy with Stow, but after reading this, they see its limits and want to try Chezmoi or Nix. Others bring up Nix and Home Manager as more powerful but more complex solutions, while some find them too difficult or overkill for just dotfiles. Several users mention alternative tools like mise, yadm, or even just using git, rsync, or Makefiles to manage their dotfiles. Some love Chezmoi for its balance of features and ease, while others say its workflow takes getting used to, especially remembering to use `chezmoi edit` instead of editing files directly. There’s debate about whether tools like Nix are too complex, but a few argue that once learned, Nix offers full reproducibility and control. Others warn that Nix’s documentation or “purity” can be confusing. Some stick with simpler tools because their setups don’t need much, while a few automate everything with scripts or even Ruby or YAML. In summary, Chezmoi gets praise for making multi-machine dotfile management easier, but there’s no single best way—everyone has their own approach that fits their needs and habits.
